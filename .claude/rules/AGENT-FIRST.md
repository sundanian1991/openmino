# AGENT-FIRST.md — 子代理优先策略

> 主代理只做协调+聚合，执行交给子代理。轻量操作直执行保证丝滑。

---

## 主代理功能边界

**核心职责**：意图识别 → 任务拆分 → 委派子代理 → 聚合结果 → 交付

主代理**不参与**具体执行（读多文件、深度分析、写代码、搜索研究）。

**允许直执行的例外**（保证丝滑，不走 Agent 开销）：

| 操作 | 条件 |
|------|------|
| 读文件 | 仅 1 个已知路径 |
| 搜索 | 简单 grep/glob，结果 < 10 条 |
| 修改 | 单文件单行 |
| 其他 | 严格顺序依赖 / 年老师说"直接做" |

超出以上范围 → **必须派子代理**。

---

## 子代理数量上限

同一批次并行子代理 **严格限制 3-5 个**。

- 任务超过 5 个独立子任务 → 分批执行，每批 ≤5 个
- 避免过度拆分导致上下文碎片化
- 主代理负责控制节奏，不一次性铺开

---

## 子代理类型速查

| 类型 | 什么时候用 |
|------|-----------|
| **Explore** | 快速搜文件、搜关键词、探索目录结构 |
| **Plan** | 设计实施方案、架构决策、多方案对比 |
| **general-purpose** | 通用深度研究、复杂多步骤任务 |
| **专家子代理** | 领域特定任务（判官→供应商、账房→数据、文书→公文等） |

**选用原则**：能用专家子代理就不用 general-purpose；能 Explore 解决的不用 general-purpose。

---

## 强制触发条件

遇到以下场景，**必须**使用 Agent 工具：

| 场景 | 子代理策略 |
|------|-----------|
| 搜索/查询 | 按关键词/目录/类型拆分，并行搜索 |
| 文件处理 | 按文件/目录/模块拆分，并行处理 |
| 数据分析 | 按维度/指标/时间段拆分，并行分析 |
| 多步骤任务 | 每步一个子代理，主代理协调聚合 |
| 跨领域任务 | 每个领域一个专家子代理 |
| 代码审查 | 按文件/模块拆分，并行审查 |
| 文档生成 | 按章节/主题拆分，并行生成 |

---

## 执行原则

1. **能并行就不串行** — 3 个独立搜索 → 3 个子代理同时跑
2. **能深入就不浅尝** — 子代理可以继续创建孙子代理
3. **能分工就不包揽** — 复杂任务拆给专家子代理
4. **能广度就别深度** — 先做广覆盖，再逐步聚焦
5. **轻量不绕路** — 单文件/简单搜索主代理直做，保证丝滑

---

## 轻量 vs 重量判断标准

**先问自己：这个操作会污染/占满我的主上下文吗？**

| 维度 | 轻量（主代理直做） | 重量（派子代理） |
|------|-------------------|-----------------|
| 文件数 | 1 个 | ≥2 个 |
| 搜索范围 | 单目录/单模式 | 跨目录/多模式 |
| 结果量 | <10 条 | ≥10 条或需深度分析 |
| 任务复杂度 | 一步到位 | 需多步推理/研究 |
| 是否需要独立上下文 | 不需要 | 需要（避免污染主上下文） |

## 使用模式

### 模式 A：并行搜索（多关键词）

关键词列表：["A", "B", "C"]
→ 同时启动 3 个子代理，每个搜索一个关键词
→ 主代理聚合结果，去重整合

### 模式 B：并行分析（多文件/模块）

文件列表：[file1, file2, file3]
→ 同时启动 3 个子代理，每个分析一个文件
→ 主代理汇总分析结果

### 模式 C：专家分工（多领域）

任务：设计一个系统
→ 子代理 1：架构设计专家
→ 子代理 2：API 设计专家
→ 子代理 3：安全审查专家
→ 主代理整合方案

### 模式 D：深度递归（子代理再派子代理）

子代理发现任务复杂 → 继续拆分 → 创建孙子代理
→ 每层都并行，直到任务粒度足够小

---

## 决策检查清单

启动任务前，问自己：

- [ ] 这个操作在轻量范围内吗？→ 是就直做
- [ ] 这个任务能拆成多个独立子任务吗？
- [ ] 拆完后每批 ≤5 个子代理吗？
- [ ] 用对子代理类型了吗？（Explore → Plan → 专家 → general-purpose）
- [ ] 子代理的结果需要我聚合吗？聚合逻辑是什么？

---

## 反模式（禁止）

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 串行搜索 | 慢，信息面窄 | 并行多关键词 |
| 主代理大包大揽 | 深度不够、上下文被污染 | 拆给专家子代理 |
| 过度拆分 | 上下文碎片化 | 按自然边界拆分，每批≤5 |
| 子代理结果不聚合 | 信息散落 | 主代理必须汇总 |
| 该用不用 | 简单操作也派 Agent，反而慢 | 轻量操作直执行 |
| 不该用乱用 | grep 两条结果也派子代理 | 遵守轻量/重量判断标准 |

## 记忆锚点

> "主代理做裁判，子代理做运动员。轻量不绕路，重量不硬扛。每批不超过五个，能并行就不串行。"

每次会话启动，先读此文件，让子代理优先成为肌肉记忆。
