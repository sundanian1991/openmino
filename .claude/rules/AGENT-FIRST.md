# AGENT-FIRST.md — 子代理优先策略

> 只要任务可拆分，必须用 Agent 工具并行执行

---

## 强制触发条件

遇到以下场景，**必须**使用 Agent 工具：

| 场景 | 子代理策略 |
|------|-----------|
| 搜索/查询 | 按关键词/目录/类型拆分，并行搜索 |
| 文件处理 | 按文件/目录/模块拆分，并行处理 |
| 数据分析 | 按维度/指标/时间段拆分，并行分析 |
| 多步骤任务 | 每步一个子代理，主代理协调聚合 |
| 跨领域任务 | 每个领域一个专家子代理 |
| 代码审查 | 按文件/模块拆分，并行审查 |
| 文档生成 | 按章节/主题拆分，并行生成 |

---

## 执行原则

1. **能并行就不串行** — 3 个独立搜索 → 3 个子代理同时跑
2. **能深入就不浅尝** — 子代理可以继续创建孙子代理
3. **能分工就不包揽** — 复杂任务拆给专家子代理
4. **能广度就别深度** — 先做广覆盖，再逐步聚焦

---

## 使用模式

### 模式 A：并行搜索（多关键词）

```
关键词列表：["A", "B", "C"]
→ 同时启动 3 个子代理，每个搜索一个关键词
→ 主代理聚合结果，去重整合
```

### 模式 B：并行分析（多文件/模块）

```
文件列表：[file1, file2, file3]
→ 同时启动 3 个子代理，每个分析一个文件
→ 主代理汇总分析结果
```

### 模式 C：专家分工（多领域）

```
任务：设计一个系统
→ 子代理 1：架构设计专家
→ 子代理 2：API 设计专家
→ 子代理 3：安全审查专家
→ 主代理整合方案
```

### 模式 D：深度递归（子代理再派子代理）

```
子代理发现任务复杂 → 继续拆分 → 创建孙子代理
→ 每层都并行，直到任务粒度足够小
```

---

## 例外（不用 Agent）

以下情况**直接执行**，不使用子代理：

- 单文件单行修改
- 需要严格顺序的依赖操作
- 用户明确说"直接做"或"简单处理"
- 读取单个已知路径的文件
- 简单的 grep/glob 查询（结果 < 10 条）

---

## 决策检查清单

启动任务前，问自己：

- [ ] 这个任务能拆成多个独立子任务吗？
- [ ] 如果并行执行，会有冲突或依赖问题吗？
- [ ] 用 N 个子代理 vs 1 个主代理，哪个信息更广？
- [ ] 子代理的结果需要我聚合吗？聚合逻辑是什么？

---

## 反模式（禁止）

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 串行搜索 | 慢，信息面窄 | 并行多关键词 |
| 单代理大包大揽 | 深度不够 | 拆给专家子代理 |
| 过度拆分 | 上下文碎片化 | 按自然边界拆分 |
| 子代理结果不聚合 | 信息散落 | 主代理必须汇总 |
