# AGENT-FIRST — 子代理优先策略

> 00-IDENTITY.md "Agent 委派铁律"的操作手册。原则在 IDENTITY，执行细节在此。
> 主代理只做协调+聚合，执行交给子代理。轻量操作直执行。
> **核心约束：token 无限，注意力有限。子代理 = 注意力防火墙。**

---

## 主代理功能边界

**核心职责**：意图识别 → 任务拆分 → 委派子代理 → 聚合结果 → 交付

主代理**不参与**具体执行（读多文件、深度分析、写代码、搜索研究）。

**例外**（直执行，不走 Agent 开销）：

| 操作 | 条件 |
|------|------|
| 读文件 | 仅 1 个已知路径 |
| 搜索 | 简单 grep/glob，结果 ≤ 5 条，不需深度分析 |
| 修改 | 单文件单行 |
| 其他 | 严格顺序依赖 / 年老师说"直接做" |

超出 → **必须派子代理**。

### 上下文隔离三原则

> 上下文不可再生。探索产生噪音，决策需要清晰。

| 原则 | 做法 |
|------|------|
| **探索归子代理，决策归主代理** | 多文件/多路径/试错 → 派子代理。子代理只带回结论，噪音自动清理 |
| **主上下文只装决策信息** | 子代理结果只保留与决策相关的部分 |
| **宁派不读** | ≥2 个文件读取 → 派子代理，不在主上下文逐个读 |

---

## 注意力管理（核心升级）

> Token 不是瓶颈，注意力才是。子代理的本质是注意力防火墙——把噪音隔离在子代理上下文中，主上下文只留决策信号。

### 主代理自检红线

主上下文出现以下任一信号 → 立即改用子代理：

| 红线信号 | 说明 |
|----------|------|
| 连续读 2+ 文件还没开始决策 | 探索占了决策位 |
| 一段回复引用 3+ 段原始内容 | 噪音淹没结论 |
| "让我看看…"后面跟一长串搜索 | 该派没派 |
| 主上下文 token 消耗 > 40% 还没产出 | 分工失败 |

### 合同式委派（强制）

派子代理时，prompt 必须包含**输出格式约束**，否则子代理回来一堆过程又污染主上下文。

**合同模板**：

```
任务：[具体任务]
输出要求：
- 只返回结论，不要过程
- 不超过 N 条要点
- 每条不超过 M 字
- 需要支撑数据时附 1-2 个关键数字，不要贴原文
```

**默认输出上限**：

| 子代理类型 | 条目上限 | 每条字数 |
|-----------|---------|---------|
| Explore（侦察） | 5 条路径 | 每条 1 句说明 |
| Plan（分析） | 3 个方案 | 每方案 2-3 句 |
| general-purpose/专家（执行） | 结论 + ≤3 条依据 | 每条 ≤50 字 |
| code-developer（代码） | 文件路径 + 变更摘要 | 摘要 ≤30 字 |

### 三层分级派遣

| 层级 | 触发 | 子代理类型 | 输出要求 |
|------|------|-----------|---------|
| **侦察** | 搜文件/查结构/定位 | Explore | 文件路径 + 1 句说明 |
| **分析** | 读内容/做判断/出方案 | general-purpose/专家 | 结论 + 依据(≤3条) |
| **执行** | 写代码/做产物/改文件 | code-developer/领域专家 | 产物路径 + 变更摘要 |

**典型节奏**：并行侦察 → 主代理收口决策 → 并行执行。不让一个代理干全程，也不让主代理等串行结果。

---

## 子代理数量上限

同一批次并行子代理 **≤ 5 个**。超过 → 分批。主代理控制节奏。

---

## 子代理类型速查

| 类型 | 用法 |
|------|------|
| **Explore** | 搜文件、搜关键词、探索目录 |
| **Plan** | 设计方案、架构决策、多方案对比 |
| **general-purpose** | 深度研究、复杂多步骤任务 |
| **专家子代理** | 领域任务（判官→供应商、账房→数据、文书→公文） |

**选用**：专家 > general-purpose > Explore 能解决就不用更重的。

---

## 强制触发条件

以下场景**必须**用 Agent 工具：

| 场景 | 策略 |
|------|------|
| 搜索/查询 | 按关键词/目录拆分，并行 |
| 文件处理 | 按文件/目录/模块拆分，并行 |
| 数据分析 | 按维度/指标/时间段拆分，并行 |
| 多步骤任务 | 每步一个子代理，主代理协调 |
| 跨领域任务 | 每领域一个专家子代理 |
| 代码审查 | 按文件/模块拆分，并行 |
| 文档生成 | 按章节/主题拆分，并行 |

---

## 执行原则

1. **能并行就不串行** — 3 个独立搜索 → 3 个子代理同时跑
2. **能深入就不浅尝** — 子代理可继续创建孙子代理
3. **能分工就不包揽** — 复杂任务拆给专家子代理
4. **能广度就别深度** — 先广覆盖，再聚焦
5. **轻量不绕路** — 单文件/简单搜索主代理直做
6. **有用就派** — 子代理开销远小于注意力污染，宁派勿扛

---

## 轻量 vs 重量判断

**核心问题：会污染主上下文吗？**

| 维度 | 轻量（直做） | 重量（派子代理） |
|------|-------------|-----------------|
| 文件数 | 1 个 | ≥2 个 |
| 搜索范围 | 单目录/单模式 | 跨目录/多模式 |
| 结果量 | ≤5 条 | >5 条或需深度分析 |
| 复杂度 | 一步到位 | 多步推理/研究 |
| 独立上下文 | 不需要 | 需要 |
| 探索噪音 | 不会 | 会 → 必须隔离 |

---

## 决策检查清单

**收到 ≥3 步任务时，先自检**：
1. 该不该派子代理？（≥2 文件读取 / 跨目录搜索 / 需独立上下文 → 必须派）
2. 能拆成并行的吗？（独立子任务 → 并行 ≤5）
3. 选对子代理类型了吗？（专家 > general-purpose > Explore）
4. 合同写清楚输出格式了吗？（条目数 + 字数上限）
5. 主上下文有没有被噪音侵占？（自检红线）

轻量？→ 直做。能拆？→ 每批 ≤5。用对子代理类型？→ 逐项验收。合同约束输出？→ 不带回噪音。

## 反模式（禁止）

串行搜索、大包大揽、过度拆分、结果不聚合、该用不用、不该用乱用、没签合同就派活、笼统验收、子代理输出无格式约束。

---

## 记忆锚点

> "Token 无限，注意力有限。子代理是注意力防火墙，不是可选项。主代理只装决策信号，不装探索噪音。合同式委派：写死输出格式，不带回过程。侦察→收口→执行，并行开路，串行决策。有用就派，宁派勿扛。"
