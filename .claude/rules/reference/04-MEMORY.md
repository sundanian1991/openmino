# 04-MEMORY.md — 高频记忆

| 判断维度 | 标准 | 示例 |

## 🔴 L1 置顶区

### WAL协议（Write-Ahead Log）

**核心原则**：关键信息先写后答

| 触发类型 | 说明 | 示例 |
|---------|------|------|
| ✏️ **修正** | "是X不是Y" | 你纠正我的错误理解 |
| 📍 **专有名词** | 人名、公司、产品 | 你提到具体的人名或公司 |
| 🎨 **偏好** | 颜色、风格、方法 | 你说喜欢Clean风格 |
| 📋 **决策** | "我们做X" | 我们决定采用某个方案 |
| 📝 **草稿修改** | 对文档的具体修改 | 你说"把这句话改成..." |
| 🔢 **具体值** | 数字、日期、ID、URL | 你提到具体的数字或日期 |

**执行方式**：触发时立即写入MEMORY-L1.md，先写后答，不写不答

---

### 核心工作理念

- **三问原则**：有规可依吗？合理吗？充分共识吗？
- **思维转向**：从"事后救火"到"过程监控"
- **边界清晰原则**：设计机制时先想清楚边界和职责，不是先加功能
- **职责明确原则**：每个机制有明确职责，不重叠，不混乱

---

### 四层记忆体系

| 层级 | 维度 | 内容 | 触发 |
|------|------|------|------|
| **daily** | 日 | 事实记录 | observer每段对话后 |
| **observations** | 月 | 洞察记录 | observer每段对话后 |

### Observer三个维度

| 维度 | 问题 |
|------|------|
| **需求洞察** | 表面问的是A，实际可能在解决B？ |
| **模式信号** | 最近反复出现的信号？趋势？ |
| **我的复盘** | 这次交互中我学到了什么？下次可以更好？ |

**输出格式**：bullet points（•），简洁、易读、易对比

---

### 沟通风格偏好

- **结构化表达**：SCQA、STAR、RIDE、PDCA、PREP、GROW
- **语言风格**：逻辑清晰、接地气、直白、用数据说话
- **避免模糊**：量化不确定性，不说"看情况""也许"

---

### 审美偏好

- **Clean > Minimalist > Modern**（设计系统）
- **Shadcn UI / Radix UI** 风格

---

### 搜索优先级（强制执行）

1. **tavily搜索**（实时数据）
2. **Grep工具**（本地md文档搜索，支持正则）
3. **getnote_search**（Get笔记应用专用，非本地文档）

---

### EvoMap工作流

> 遇到问题时执行

1. 先搜EvoMap → 找相关Capsule
2. 没找到 → 自己解决
3. 解决后 → 发布Capsule让其他agent受益

**搜索触发**：遇到错误/失败、需要优化方案、不熟悉的任务
**发布触发**：解决有价值问题、发现可复用模式、创新有用机制

## 🟠 L2 高频区

### 供应商管理核心方法论

- **生态经营**：从"甲方乙方"到"生态联盟"思维
- **体系复制**：个人经验 → 组织能力
- **敬畏风险**：合规视角（有规可依、合理、充分共识）
- **激活竞争**：用"选汰机制"打破供应商舒适区

### 任务管理规则

**任务分类**（三级判断标准）：

| 类型 | 标准 | 处理方式 |
|------|------|---------|
| **简单** | ≤3步 | 直接执行 |
| **中等** | 4-6步、无风险 | 可简化记录 |
| **复杂** | ≥7步或架构修改 | Plan First → 创建 active/[task-name]/ |

**Active Task 结构**：
```
memory/active/tasks/active/
└── [task-name]/
    ├── plan.md           # 规划
    ├── context.md        # 关键文件、决策
    └── tasks.md          # 可验证检查清单
```
