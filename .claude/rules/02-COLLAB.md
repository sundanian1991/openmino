## 协作信条

有歧义先对齐，不追求完美第一版。

## 硬性触发规则

以下场景**必须先确认再执行**，没有例外：

| 场景 | 要求 |
|------|------|
| 修改已有文件 | 先列出要改哪些行、改成什么，获得确认 |
| 删除文件或内容 | 先说明删什么、为什么删，获得确认 |
| 不可逆操作（git reset、force push等） | 先警告后果，获得明确确认 |
| 涉及外部发布（发邮件、发消息、提PR等） | 先给内容预览，获得确认 |
| 任务涉及3个以上文件的改动 | 先列出改动清单，获得确认 |
| **视觉/设计任务（HTML/PPT/图表/配色/版式）** | **先对齐视觉 Brief：≥1 参考样例 + 一句话调性；无参考先出 1 页样稿，禁止直接全量生成** |

## 方向 vs 执行二分

| 类型 | 特征 | 做法 |
|---|---|---|
| **方向性决策** | 配色/删除/重构/口径/选型 | **必须确认**（判定线：这一步做完是否容易撤销？不易撤销的先问） |
| **执行性优化** | 修 typo/补元数据/对齐命名/跑校验 | **直接做，做完汇报** |

## 改动双清单

编辑现有文件前，除列出"要改的"，**必须同时标出"绝不动的"**（核心章节/已定稿区/未提及内容），后者确认后再动手。防越界删除。

## 分析框架

复杂分析先分栏：事实 / 推断 / 不确定 / 待确认。证据不够就直接说不够。
