# 公文起草技能

> 专业的公文/邮件/审批材料起草工具

---

## 用途

**当用户需要**：
- 起草审批邮件、决策请示
- 写工作汇报、周报月报
- 写通知、公告、协作邮件
- 写供应商管理相关公文（寻源、评估、清退、结算等）
- 写内部协调文档

**不是用于**：
- 发送邮件（用 email skill）
- 个人日常非正式沟通

---

## 输出格式

**Markdown 格式**，支持：
- 直接在回复中预览（渲染效果）
- 复制到 Markdown 编辑器
- 转换为 HTML 后用 email skill 发送

---

## 工作流

```
用户说："写一个 XXX 审批"
  ↓
调用 document-writer 技能
  ↓
输出 Markdown 格式文档
  ↓
用户预览 → 提出修改 → 调整内容
  ↓
用户确认 → 转换为 HTML → 调用 email skill 发送
```

## 文档规范

### 核心原则

**段落逻辑优先，视觉装饰为零，适度留白舒适**

| 正确做法 | 错误做法 |
|----------|----------|
| 段落开始用**加粗关键词**区分 | 用分割线/边框等装饰 |
| 表格用于数据展示 | 用表格做视觉分隔 |
| 段落靠内在逻辑推进 | 依赖分段/标题堆叠 |

### 标准格式结构

```
1. 开头：称呼 + 目的
2. 事实部分：列表/表格呈现（时间、人员、数据等）
3. 过渡：段落间用"基于上述..."、"综合评估..."等连接
4. 分析判断："我们认为..."、"这说明..."等加入观点
5. 结尾：明确请求（"请审阅"、"如有问题请反馈"）
```

### 6 条具体规则

1. **开头称呼**：称呼单行，空一行后接主体
2. **关键词标识**：每个段落开始用`<strong>关键词：</strong>`标识
3. **禁止装饰**：不使用分割线、边框等视觉装饰
4. **数据结构化**：时间、人员、数据等用列表/表格清晰呈现
5. **留白适度**：段落之间空一行，保持阅读节奏
6. **结尾行动**：明确的行动请求或请示

## 使用方式

### 方式 1：直接指定模板

用户说："写一个审批邮件" / "写一个结算通知"
→ 根据用途选择对应模板
→ 按模板格式起草
→ **输出 Markdown 格式，可预览可编辑**

### 方式 2：用户描述需求

用户说："我要写一个 xxx，告知 yyy，请求 zzz"
→ 分析需求
→ 匹配最接近的模板
→ 按模板格式起草
→ **输出 Markdown 格式**

### 方式 3：用户提供内容要点

用户说："帮我写个邮件，要点是..."
→ 提炼结构（背景/事实/分析/建议）
→ 匹配模板格式
→ 组织语言
→ **输出 Markdown 格式**

## 预览和编辑

**输出 Markdown 格式后，你可以**：

1. **直接在回复中预览**：我会用渲染格式展示
2. **复制到 Markdown 编辑器**：Typora、Obsidian、VS Code 等
3. **提出修改要求**：调整内容、格式、措辞
4. **转换为 HTML**：用 Pandoc 或在线工具转换
5. **用 email skill 发送**：确认无误后，我转为 HTML 格式发送

## 质量检查清单

### 场景适配性检查（强制执行）

**起草前必读**：`CHECKLIST.md` — 完整的场景适配性规则

**快速检查流程**：
1. 确定场景（向上审批/向上汇报/供应商管理/内部协调/整改通知）
2. 对照 `CHECKLIST.md` 中的禁用词检查
3. 数据化表达（用数字代替模糊描述）
4. 风险对应方案（每个风险都有应对措施）
5. 姿态检查（向上谦逊、向下专业、平级协作）

### 基础格式检查
- [ ] 有明确的称呼吗？
- [ ] 段落开头用了 `**关键词：**` 吗？
- [ ] 数据用表格/列表呈现了吗？
- [ ] 段落之间有空行吗？
- [ ] 结尾有明确的行动请求吗？
- [ ] 没有用分割线等装饰吗？
