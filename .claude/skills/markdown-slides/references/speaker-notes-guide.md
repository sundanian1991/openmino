# Speaker Notes Guide

演讲者注释转换指南 — 什么内容应该转为注释，什么应该保留在幻灯片上。

## Format Syntax

### Deckset (Default)

```markdown
幻灯片内容。

^ 这是演讲者注释。可以是多句话，提供额外上下文。

^ 这是第二段注释。每段注释独立成行。
```

### Marp

```markdown
幻灯片内容。

<!-- 这是演讲者注释。Marp 使用 HTML 注释格式。 -->
```

---

## When to Convert to Speaker Notes

### ✅ Convert (应该转换)

| 类型 | 示例 | 原因 |
|------|------|------|
| **多句解释段落** | "这个流程包含三个阶段，每个阶段都有其特定的目标和交付物..." | 信息量大，适合作为口述内容 |
| **背景上下文** | "这是在 2025 年 Q4 试点项目中验证的方法，当时我们在三家供应商进行了测试..." | 提供额外背景， clutter 页面 |
| **技术细节** | "底层实现使用了异步队列机制，确保高并发场景下的稳定性..." | 技术深度，听众不需要看到 |
| **个人经验** | "我在实际操作中发现，第三步最容易被忽视，建议设置检查点..." | 个人洞察，适合口述 |
| **数据解读** | "注意这里的增长率是环比，不是同比。Q3 的异常值是因为..." | 帮助演讲者解读数据 |
| **类比说明** | "这就像开车时的导航系统，实时调整路线以避开拥堵..." | 帮助理解，不需要显示 |

### ❌ Don't Convert (不转换)

| 类型 | 示例 | 原因 |
|------|------|------|
| **单句段落** | "下一步是测试。" | 可能是幻灯片核心内容 |
| **Bullet lists** | "- 第一步<br>- 第二步<br>- 第三步" | 标准幻灯片内容格式 |
| **Numbered lists** | "1. 分析<br>2. 设计<br>3. 实现" | 流程/步骤展示 |
| **引用块** | "> 创新不是偶然，是系统设计的结果。" | 强调的重点内容 |
| **冒号结尾句** | "关键因素包括：" | 引入列表，属于内容 |
| **标题** | `## 核心发现` | 结构元素 |
| **表格** | `\| 指标 \| 值 \|` | 数据展示 |
| **代码块** | `\`\`\`python ... \`\`\`` | 技术内容展示 |

## Decision Framework

```
看到一段内容
    ↓
是标题/列表/表格/代码吗？
  ├─ YES → 不转换，保留在幻灯片上
  └─ NO ↓

是单句段落吗？
  ├─ YES → 不转换 (可能是核心内容)
  └─ NO ↓

以冒号结尾吗？
  ├─ YES → 不转换 (在引入列表)
  └─ NO ↓

是多句解释性段落吗？
  ├─ YES → 转换为演讲者注释 ✅
  └─ NO ↓

提供额外背景/上下文吗？
  ├─ YES → 转换为演讲者注释 ✅
  └─ NO → 保留在幻灯片上
```

## Best Practices

### 1. One Slide, One Note Paragraph

```markdown
## Good ✅

幻灯片要点。

^ 第一段注释，解释这个要点的背景。可以 2-3 句话。

## Bad ❌

幻灯片要点。

^ 第一段。
^ 第二段。
^ 第三段。
```

### 2. Notes Should Support, Not Repeat

```markdown
## Good ✅

幻灯片：增长率 35%

^ 这是环比增长率，主要 driven by 新供应商引入和现有供应商效率提升。

## Bad ❌

幻灯片：增长率 35%

^ 增长率是 35%。
```

### 3. Use Notes for Timing Cues

```markdown
## 项目时间线

- Q1: 规划
- Q2: 执行
- Q3: 评估

^ Q1 规划阶段需要 4-6 周，重点是需求收集和供应商筛选。Q2 执行阶段是核心，建议每周站会追踪进度。Q3 评估阶段产出完整 ROI 分析。
```

### 4. Add Transition Hints

```markdown
## 当前方案

- 优点：成熟、稳定
- 缺点：扩展性有限

^ 讲完当前方案后，可以问"那有没有更好的方案？"然后自然过渡到下一页的新方案。
```

## Length Guidelines

| 幻灯片类型 | 建议注释长度 | 示例 |
|-----------|-------------|------|
| **章节 intro** | 1-2 段，50-100 词 | 介绍本章目标和主要内容 |
| **数据幻灯片** | 1 段，30-50 词 | 解读关键数据点和异常值 |
| **流程幻灯片** | 1 段，40-60 词 | 说明各阶段重点和注意事项 |
| **对比幻灯片** | 1-2 段，50-80 词 | 解释对比维度和推荐理由 |

## Platform Differences

### Deckset

- 支持多段注释，每段独立
- Presenter View 中清晰显示
- 支持 Markdown 格式 (bold, italic)

```markdown
^ **重点**：这个数据是环比。

^ 第二段的 *补充说明*。
```

### Marp

- 使用 HTML 注释 `<!-- -->`
- 支持 `<!-- pause -->` 控制节奏
- 注释内可用简单 HTML

```markdown
<!-- **重点**：这个数据是环比。 -->
<!-- pause -->
<!-- 第二段的说明。 -->
```

## Common Mistakes

| 错误 | 问题 | 修正 |
|------|------|------|
| **过短** | "^ 好。" | 扩展为有意义的注释 |
| **过长** | "^ 300 词的段落..." | 拆分到多页或简化 |
| **重复幻灯片** | "^ 如你所见，增长率是 35%..." | 添加解读而非重复 |
| **遗漏关键解释** | 只有数据，没有解读注释 | 补充数据背后的含义 |
| **格式错误** | `^注释` (无空格) | `^ 注释` (有空格) |

## Quick Checklist

- [ ] 多句解释段落已转换
- [ ] 列表/表格/代码未转换
- [ ] 单句段落未转换
- [ ] 冒号结尾句未转换
- [ ] 注释提供额外价值 (不重复)
- [ ] 长度适当 (30-80 词/页)
- [ ] 格式正确 (`^ ` + 内容)
- [ ] 平台正确 (Deckset: `^ `, Marp: `<!-- -->`)

## Examples

### Full Slide Example

```markdown
## 供应商绩效评估 📊

- 质量得分：4.2/5.0
- 交付准时率：94%
- 成本竞争力：+12% YoY

^ 质量得分较上季度提升 0.3 分，主要得益于新引入的言犀和零犀两家供应商。交付准时率 94% 略低于行业标杆 96%，这是下季度重点改进项。成本竞争力的 12% 增长来自赛马机制带来的价格竞争。

^ 注意：这个数据是 Q4 数据，已剔除春节假期影响。下一页会详细展示各供应商的具体表现对比。
```

### Section Intro Example

```markdown
# 2. 供应商生命周期管理 🔄

^ 这一部分我们会完整 walkthrough 供应商从引入到退出的全生命周期。重点讲解六个关键阶段：储备、引入、运营、评估、清退、结算。每个阶段我都会分享实际操作中的注意事项和常见陷阱。整个过程预计 15 分钟。
```
