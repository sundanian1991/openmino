# Spec-Driven Development (SDD) 技能

> **核心理念**：Spec 是源代码，代码是生成物。

---

## 何时激活

- ✅ 新功能开发（多个组件、复杂交互）
- ✅ 业务规则实现（供应商评估、风控逻辑）
- ✅ 数据处理/报表生成（Dashboard、KPI 计算）
- ✅ 代码重构/框架迁移
- ⚠️ 简单修改/BUG 修复 → 用 `tdd-agentic` 即可

---

## 工作流总览

```
┌─────────────────┐
│  1. 写 Spec      │ → 定义功能目标、输入输出、业务规则
└────────┬────────┘
         ↓
┌─────────────────┐
│  2. 拆解测试     │ → 每条规则一个测试用例
└────────┬────────┘
         ↓
┌─────────────────┐
│  3. AI 循环实现   │ → 编码 → 跑测试 → 失败 → 重编码 → 通过
└────────┬────────┘
         ↓
┌─────────────────┐
│  4. 人类验收     │ → 业务验证、真实数据跑
└─────────────────┘
```

## Step 1: 写 Spec

### Spec 模板

```markdown
# [功能名称] Spec

## 功能目标
[一句话描述：这个功能解决什么问题]

## 输入
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 字段 1 | string | 是 | 描述 |
| 字段 2 | number | 否 | 描述 |

## 输出
| 字段 | 类型 | 说明 |
|------|------|------|
| 字段 1 | type | 描述 |

## 业务规则
1. **规则 1**: 条件 A → 结果 B
2. **规则 2**: 条件 C + D → 结果 E
3. **例外**: 特殊情况处理

## 边界条件
- 边界 1: [描述，如"输入为空时如何处理"]
- 边界 2: [描述，如"输入超出范围时如何处理"]

## 性能要求（可选）
- 响应时间：< X ms
- 并发支持：> Y QPS

## 验收标准
- [ ] 测试用例 1 通过
- [ ] 测试用例 2 通过
- [ ] 边界条件测试通过
- [ ] 真实数据验证通过
```

### 写 Spec 的原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **可验证** | 每条规则都能写成测试 | ❌ "用户体验好" → ✅ "响应时间<200ms" |
| **无歧义** | 不用"可能"、"也许" | ❌ "一般情况下返回 A" → ✅ "条件 X 时返回 A" |
| **完整** | 覆盖正常 + 异常流程 | 正常流程 + 错误处理 |
| **可执行** | AI 能看懂并实现 | 用结构化语言，不用散文 |

## Step 2: 拆解测试用例

### 测试用例模板

```markdown
## 测试用例 [编号]: [名称]

**对应规则**: 规则 X

**输入**:
```
{具体输入数据}
```

**预期输出**:
```
{预期输出数据}
```

**执行方式**:
- [ ] 自动化测试（代码）
- [ ] 人工验证（步骤说明）

**状态**:
- [ ] 未执行
- [ ] 已执行 - 通过
- [ ] 已执行 - 失败
```

### 测试分类

| 类型 | 用途 | 示例 |
|------|------|------|
| **正常流程** | 验证主功能 | 输入有效数据 → 输出正确结果 |
| **边界条件** | 验证边界处理 | 输入 0、空值、最大值 |
| **错误处理** | 验证异常处理 | 输入非法数据 → 抛出错误 |
| **组合场景** | 验证多规则交互 | 多个条件同时满足 |

### 测试拆解原则

**一规则一测试**：
```
❌ 坏：一个测试验证所有规则
✅ 好：每条规则一个独立测试
```

**可独立执行**：
```
❌ 坏：测试 B 依赖测试 A 的结果
✅ 好：每个测试都能单独跑
```

## Step 3: AI 循环实现

### 循环流程

```
┌──────────────┐
│ AI 编码        │ → 基于 Spec 写实现
└──────┬───────┘
       ↓
┌──────────────┐
│ 跑测试        │ → 执行测试用例
└──────┬───────┘
       ↓
    通过？
   ╱     ╲
  是      否
  ↓       ↓
 完成    AI 修复
        （给错误信息）
          ↓
       回到编码
```

### 给 AI 的指令

```markdown
## 任务

基于以下 Spec 和测试用例，实现功能：

**Spec**: [链接/内容]
**测试用例**: [链接/内容]

## 要求

1. 先写一个最小实现，让第一个测试通过
2. 跑测试，看结果
3. 失败 → 根据错误信息修复 → 再跑
4. 重复直到所有测试通过

## 当前状态

- 已完成：0/N 测试
- 当前测试：[测试名称]
- 错误信息：[如果有]
```

### 循环中的角色

| 角色 | 职责 |
|------|------|
| **AI** | 编码、跑测试、修复 |
| **你** | 看进度、解答疑问、最终验收 |

## Step 4: 人类验收

### 验收清单

- [ ] **功能正确**：所有测试用例通过
- [ ] **业务正确**：用真实数据跑，结果符合预期
- [ ] **边界处理**：边缘情况有合理处理
- [ ] **代码质量**：可读、可维护、无过度工程
- [ ] **文档完整**：使用说明、注意事项

### 验收流程

```
1. 看测试报告 → 确认全部通过
2. 用真实数据跑 → 确认业务正确
3. 抽查代码 → 确认无过度工程
4. 写使用文档 → 方便后续使用
```

## 案例：供应商评分计算

### Spec 示例

```markdown
# 供应商评分计算 Spec

## 功能目标
根据供应商 KPI、合规记录、交付及时率计算综合评分

## 输入
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| kpi_score | number | 是 | KPI 得分 (0-100) |
| is_compliant | boolean | 是 | 是否合规 |
| ontime_rate | number | 是 | 交付及时率 (0-1) |

## 输出
| 字段 | 类型 | 说明 |
|------|------|------|
| grade | string | 等级 (A/B/C/D) |
| risk | string | 风险标识 (正常/预警/高危) |
| score | number | 综合得分 |

## 业务规则
1. **合规一票否决**: is_compliant=false → risk=高危，grade=D
2. **KPI 阈值**: kpi_score<60 → grade≤C
3. **及时率惩罚**: ontime_rate<0.8 → 降级处理
4. **综合得分**: score = kpi_score * ontime_rate * 权重

## 边界条件
- 输入为空 → 抛出错误
- kpi_score>100 或<0 → 抛出错误
- ontime_rate>1 或<0 → 抛出错误

## 验收标准
- [ ] 测试用例 1: 合规 + 高 KPI → A/正常
- [ ] 测试用例 2: 不合规 → D/高危
- [ ] 测试用例 3: KPI<60 → C 或以下
- [ ] 测试用例 4: 及时率<0.8 → 降级
```

### 测试用例示例

```markdown
## 测试用例 1: 合规 + 高 KPI

**对应规则**: 规则 1, 4

**输入**:
```
{
  kpi_score: 90,
  is_compliant: true,
  ontime_rate: 0.95
}
```

**预期输出**:
```
{
  grade: 'A',
  risk: '正常',
  score: 85.5
}
```

## 测试用例 2: 不合规一票否决

**对应规则**: 规则 1

**输入**:
```
{
  kpi_score: 95,
  is_compliant: false,
  ontime_rate: 0.99
}
```

**预期输出**:
```
{
  grade: 'D',
  risk: '高危',
  score: 0
}
```
```

## 常见问题

### Q: Spec 写多细？

**A**: 细到 AI 能看懂并实现，不细到限制实现方式。

### Q: 测试用例太多怎么办？

**A**: 先写核心规则测试，边缘测试后续补充。

### Q: AI 循环很多次都不过？

**A**: 可能是 Spec 有歧义或测试有误，先检查 Spec。
