# 述标评估

> 供应商述标的完整评估工作流，四阶段渐进式推进。

---

## 触发条件

用户提供供应商名单并启动述标评估时触发。

---

## 启动流程

技能启动时，**必须先问**：

1. 本次述标项目名称是什么？
2. 有哪些供应商参加？请按述标顺序排列。

收到回答后：
1. 在 `projects/` 下创建项目文件夹：`供应商述标评估-YYYYMMDD-项目名/`
2. 创建 `00-项目概览与对比表.md`（供应商名单 + 状态跟踪）
3. 按供应商顺序，从第一家开始阶段1

---

## 四阶段流程

### 阶段1：初步调查（AnySearch）

**触发**：年老师给出公司名

**执行**：
1. 用 AnySearch 搜索以下维度（参考 [references/评估逻辑.md](references/评估逻辑.md)）
2. 结果写入 `{序号}-{公司名}-初步调查.md`
3. 输出初步判断：风险预警 + 需关注的点

**搜索维度**：
- 工商信息（注册资本、成立时间、法人、经营范围）
- 司法风险（被执行人、失信记录、劳动仲裁）
- 经营异常（行政处罚、经营异常名录）
- 行业口碑（招聘评价、员工评价、行业新闻）

### 阶段2：材料评估（PDF/PPT）

**触发**：年老师给出 PDF 或 PPT 文件

**执行**：
1. PDF → 用 Read 工具读取，或用 pdfplumber 提取表格
2. PPT → 用 `markitdown` 转 markdown 提取文字
3. 提取关键信息（参考 [references/评估逻辑.md](references/评估逻辑.md)）
4. 与阶段1搜索结果交叉核对
5. 追加到 `{序号}-{公司名}-初步调查.md` 的"材料评估"章节

**提取内容**：
- 案例数据（项目类型、转化率、合作甲方）
- 团队配置（项目经理、核心成员、经验背景）
- 方案架构（业务策略、资源保障、风控机制）
- 资质证书（截图/文字，与搜索结果比对）

### 阶段3：述标问题生成

**触发**：材料评估完成后

**执行**：
1. 基于阶段1+2的信息，生成针对性追问清单
2. 写入 `{序号}-{公司名}-述标问题.md`
3. 问题分三类：必问 / 建议问 / 风险验证

**问题来源**（参考 [references/评估逻辑.md](references/评估逻辑.md)）：
- 材料中数据模糊或缺失的点
- 搜索结果与述标声称矛盾的点
- 方案中需要验证可行性的点

### 阶段4：最终报告（转文字）

**触发**：年老师给出述标转文字内容

**执行**：
1. 分析转文字内容（沟通逻辑、应变能力、配合度）
2. 交叉验证：说的 vs 材料 vs 搜索结果
3. 按评分细则打分（参考 [references/评分细则.md](references/评分细则.md)）
4. 检查一票否决项（参考 [references/一票否决.md](references/一票否决.md)）
5. 生成完整评估报告：`{序号}-{公司名}-评估报告.md`
6. 更新 `00-项目概览与对比表.md` 的对比表

## 文件结构

```
projects/供应商述标评估-YYYYMMDD-项目名/
├── 00-项目概览与对比表.md
├── 01-{公司A}-初步调查.md
├── 01-{公司A}-述标问题.md
├── 01-{公司A}-评估报告.md
├── 02-{公司B}-初步调查.md
├── 02-{公司B}-述标问题.md
├── 02-{公司B}-评估报告.md
└── ...
```

**命名规则**：`序号-公司名-{阶段}.md`
- 初步调查（阶段1+2合并）
- 述标问题（阶段3）
- 评估报告（阶段4，最终交付物）

## 评估分析模型

> 详细内容见 [references/评估逻辑.md](references/评估逻辑.md)

### 正向指标（什么是好供应商）
- 案例数据可验证（有具体数字、有甲方背书）
- 团队配置合理（不是挂名，有实操经验）
- 方案有针对性（不是通用模板，针对我司场景定制）
- 风控意识前置（主动提合规，不等问题暴露）
- 沟通配合务实（不夸大、不回避、能接住追问）

### 负向信号（什么是差供应商）
- 数据模糊（"转化率很高"但不说具体数字）
- 方案通用（换个甲方名字也能用）
- 团队包装（头衔多但没有实操细节）
- 回避风险（不提合规、不提投诉处理）
- 承诺过度（什么都能做到，没有取舍）

### 权重逻辑
- 团队经验 > 公司规模（人对了事才对）
- 方案针对性 > 方案完整性（有重点比面面俱到重要）
- 风控意识 > 资源投入（出事了再多人也没用）

## 评分规则

1. 严格按评分细则的分值范围打分（参考 [references/评分细则.md](references/评分细则.md)）
2. 每个建议分必须有对应依据（从材料/转文字/搜索结果中提取）
3. 综合实力部分标记为"人工填写"
4. 依据需具体引用来源（转文字/PPT/PDF/搜索）
5. 一票否决项不通过则直接标注，不进入正常评分流程

## 依赖自检

技能启动时，按顺序检查以下能力是否可用，缺失则自动安装：

### 1. AnySearch（公司信息搜索）
- **检查**：调用 `mcp__anysearch__list_domains` 是否返回结果
- **缺失时**：提示用户需要配置 AnySearch MCP 服务，引导安装

### 2. PDF 读取
- **检查**：尝试用 Read 工具读取一个 PDF 文件
- **缺失时**：自动安装 Python 依赖
  ```bash
  pip install pdfplumber pypdf
  ```

### 3. PPT 读取
- **检查**：尝试运行 `python -m markitdown --help`
- **缺失时**：自动安装
  ```bash
  pip install "markitdown[pptx]"
  ```

### 4. 文件系统
- **检查**：`projects/` 目录是否存在
- **缺失时**：自动创建
  ```bash
  mkdir -p projects
  ```

---

## 关键原则

- **先搜后评**：每家公司必须先做 AnySearch 交叉核对
- **证据优先**：打分必须有依据，不凭印象
- **交叉验证**：搜索结果 vs 材料 vs 转文字，三方比对
- **问题导向**：述标问题必须针对薄弱点，不问废话
- **独立报告**：每家公司独立成文，方便提交审批
