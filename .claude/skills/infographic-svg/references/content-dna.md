# 内容 DNA — 从文字到视觉的转换方法论

> **用途**：拿到一段文字内容后，如何像那 6 张图一样，逻辑清晰地呈现出来。
>
> **触发**：每次需要将文本内容转化为 SVG 信息图时读取。

---

## 一、先判断：内容在表达什么关系？

那 6 张图之所以好，不是因为"好看"，是因为**每一张都在清晰表达一种逻辑关系**。

| 关系类型 | 判断信号词 | 对应视觉模板 |
|---------|-----------|------------|
| **组成部分** | "包括"、"包含"、"由...组成" | 模板 1：时间线/容量轴 |
| **程度变化** | "从...到..."、"越来越"、"逐渐" | 模板 2：渐变衰减 |
| **选项排列** | "有以下几种"、"可以选择"、"策略" | 模板 3：光谱排列 |
| **筛选过滤** | "只看到"、"实际接收"、"被过滤" | 模板 4：左右对比 |
| **方案比较** | "A 更好"、"B 不行"、"对比" | 模板 5：方案对比 |
| **传递返回** | "传给"、"返回"、"留在" | 模板 6：父子容器 |
| **完整 vs 精简** | "全部记录"、"实际收到" | 模板 7：信息过滤 |

**核心原则**：不要从"好看"出发，从"这个内容在表达什么关系"出发。关系确定了，视觉模板自然就选出来了。

## 二、切割文本：切成标准"卡片单元"

### 切割规则

每段文字切完后，每个单元应该是这样的结构：

```
[标题] ≤ 5 词，名词短语
[说明] ≤ 2 行，主谓宾完整
[标注] ≤ 1 行，灰色小字（可选）
```

### 切割示例

**原始内容**：
> 上下文窗口是模型在生成下一个回复时能看到的所有内容，包括系统提示词、CLAUDE.md 文件、对话历史、工具调用和输出、读取的文件，以及剩余的自由空间。最多 100 万 tokens，到达上限会被硬截断。

**切割后**：

| 单元 | 标题 | 说明 | 标注 |
|------|------|------|------|
| 标题 | The context window | Everything the model can see when it generates the next response | — |
| 卡片 1 | system prompt | — | — |
| 卡片 2 | CLAUDE.md | — | — |
| 卡片 3 | conversation so far | — | — |
| 卡片 4 | tool calls + outputs | — | — |
| 卡片 5 | files read | — | — |
| 卡片 6 | free | — | — |
| 底部标注 | — | — | 0 / 1,000,000 tokens, hard cutoff |

### 切割铁律

1. **不创造原文没有的信息** — 只整理，不创造
2. **修饰词去掉** — "非常重要的系统提示词" → "system prompt"
3. **合并同类项** — "工具调用"和"工具输出" → "tool calls + outputs"
4. **保留英文术语** — 不要翻译成中文，保持原文的技术术语

## 三、排优先级：P0-P3

不是所有信息同等重要。强制分级：

| 级别 | 含义 | 视觉表达 | 每图配额 |
|------|------|---------|---------|
| **P0** | 这图想让读者记住的 | 标题 + 最大视觉元素 | 最多 1 个 |
| **P1** | 支撑 P0 的关键证据 | 卡片 + 正常大小 | 3-7 个 |
| **P2** | 辅助理解的背景 | 灰色小字标注 | 按需 |
| **P3** | 删掉不影响理解 | **删掉** | 0 |

### P0 判断方法

| 信号 | 说明 |
|------|------|
| 内容的标题/第一句话 | 通常就是 P0 |
| "所以""因此""结论是"后面的内容 | 结论性陈述 |
| 如果删掉这个信息，整个内容失去意义 | 不可删除 = 核心 |
| 数字中最大/最小/异常的值 | 数据型内容的核心焦点 |

### 常见错误

- ❌ 把所有数据都标 P0 → 没有焦点，全是重点等于没有重点
- ❌ 保留所有原始文字 → 图片拥挤，信息密度失控
- ❌ 自己脑补了原文里没有的洞察 → 只整理，不创造

## 四、视觉隐喻选择指南

### 4.1 时间线 / 容量轴

**什么时候用**：
- 内容在说"这个东西由哪些部分组成"
- 组成部分有顺序关系
- 有"从左到右"的容量或进程概念

**布局公式**：
```
标题 + 副标题
[ 大容器 ]
  [卡片1][卡片2][卡片3][卡片4][卡片5][虚线卡片]
[ 底部轴线 ]—— 0 ———— 最大值 ———
```

**尺寸参数**：
- 大容器：x=40, width=600
- 卡片高度：28-40px
- 卡片间距：6-8px
- 底部标注：距容器底部 10px

### 4.2 渐变衰减

**什么时候用**：
- 内容在说"越远的越模糊"、"时间久了就忘了"
- 有"sharp → rotting → gone"的连续变化
- 需要展示"质量递减"的过程

**布局公式**：
```
标题 + 副标题
[ 渐变条 600px 宽，从深到浅 ]
[解释卡片1][解释卡片2][解释卡片3]
[ summary 卡片 ]
底部斜体标注
```

**尺寸参数**：
- 渐变条：width=600, height=24px
- 解释卡片：等宽，间距 8px
- summary 卡片：width=600，居中

### 4.3 光谱排列

**什么时候用**：
- 内容在说"有这几种策略/方法/选项"
- 选项之间有"从简单到复杂"的递进
- 需要展示"强度等级"

**布局公式**：
```
标题 + 副标题
轴 ——→
[卡片1][卡片2][卡片3][卡片4][卡片5]
```

**尺寸参数**：
- 卡片宽度：100-116px（统一）
- 卡片高度：80-100px
- 间距：8px
- 右侧状态竖线：stroke-width=3, 高度对齐卡片

### 4.4 左右对比

**什么时候用**：
- 内容在说"全部 vs 实际"、"完整 vs 精简"
- 需要展示"过滤"或"筛选"过程
- 两个集合有包含关系

**布局公式**：
```
标题 + 副标题
[ 容器 A: 全部内容 ]  →箭头→  [ 容器 B: 实际看到 ]
  密集列表卡片                    精简卡片
底部标注
```

**尺寸参数**：
- 两个容器等宽：280px
- 间距：20px
- 箭头：居中，带文字标注

### 4.5 方案对比

**什么时候用**：
- 内容在说"A 方法好，B 方法不行"
- 需要展示两种方案的**效果差异**
- 有"正确做法 vs 错误做法"

**布局公式**：
```
标题 + 副标题
[ 方案 A 卡片 ]          [ 方案 B 卡片 ]
  [mini 进度条/条状图]     [mini 进度条/条状图]
  说明文字                 说明文字
底部使用建议
```

**尺寸参数**：
- 卡片宽度：280-300px
- mini 条状图：width 根据百分比计算
- 间距：20px

### 4.6 父子容器

**什么时候用**：
- 内容在说"父进程 spawn 子进程"
- 需要展示"什么传过去了，什么留下来了"
- 有 spawn/return 的信息流动

**布局公式**：
```
标题 + 副标题
[ 父容器 ]
  [ 子容器 ]
  spawn 曲线箭头 →
  子容器内部卡片（工具调用、文件读取）
  result 曲线箭头 ←
  底部斜体标注
```

**尺寸参数**：
- 父容器：width=600, rx=12
- 子容器：width=190-280px, rx=10
- 曲线箭头：Q 贝塞尔曲线

### 4.7 信息过滤

**什么时候用**：
- 内容在说"完整的记录"vs"模型实际接收"
- 左边密集、右边精简
- 有明确的"过滤"动作

**布局公式**：
```
标题 + 副标题
[ 左: Session History ]  [ 右: What Claude Sees ]
  密集列表卡片              精简卡片
  [ 中间过滤箭头 ]
底部斜体标注
```

## 五、颜色语义分配

拿到内容后，按以下步骤决定颜色：

```
Step 1：内容需要语义区分吗？
  → 不需要 → 只用结构色
  → 需要 → Step 2

Step 2：需要几种语义色？
  → 1 种 → 选最匹配的一种
  → 2 种 → Step 3

Step 3：两种语义色的关系？
  → 对比（好 vs 坏） → 绿色 + 橙色/蓝色
  → 递进（低 vs 高） → 浅橙 + 深橙
  → 分类（类型 A vs 类型 B） → 橙色 + 蓝色
```

**语义色速查**：

| 语义关系 | 颜色组合 | 示例 |
|---------|---------|------|
| 成功 vs 失败 | `#8f9b7f` + `#d4845a` | Correcting vs Rewinding |
| 用户 vs 系统 | `#6a8fb5` + `#1a1a1a` | 用户消息 vs 系统响应 |
| 警告 vs 正常 | `#d4845a` + `#6b6b6b` | 警告块 vs 普通说明 |

## 六、文案精简检查

生成 SVG 之前，对每条文字做精简检查：

| 检查项 | 标准 | 示例 |
|--------|------|------|
| **标题 ≤ 5 词** | 名词短语，无动词 | ❌ "This is how the model works" → ✅ "The context window" |
| **说明 ≤ 2 行** | 主谓宾完整 | ❌ 长段落 → ✅ 一行话 |
| **标注 ≤ 1 行** | 灰色小字 | ❌ 完整句子 → ✅ "hard cutoff" |
| **无修饰词** | 去掉形容词副词 | ❌ "very important system prompt" → ✅ "system prompt" |
| **无重复** | 合并同类项 | ❌ "tool calls" + "tool outputs" → ✅ "tool calls + outputs" |
