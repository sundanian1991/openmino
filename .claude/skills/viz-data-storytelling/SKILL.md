# Data Storytelling — 数据沟通策略与编排

> 来源：《Storytelling with Data》全书 30 skill 整合 | 核心产出 = Markdown 大纲

---

## 定位

data-storytelling **不画图、不做 PPT、不指定图表类型**。它的唯一产出是一份 **Markdown 大纲**，定义：

- 受众是谁、要他做什么（Big Idea）
- 需要几张图、每张图表达什么叙事功能
- 图的叙事顺序（先讲什么后讲什么）
- 原始数据或数据引用（供下游 viz-design 消费）

大纲完成后，**交给 viz-design 做视觉设计和渲染委派**。

## 工作流

```
拿到数据
  ↓
data-storytelling：分析 + 策略 + 大纲（产出 Markdown 大纲 + 原始数据引用）
  ↓
独立审查：子代理只看大纲，不带背景上下文（找盲点）
  ↓
迭代：修复 P0/P1 问题
  ↓
下游设计：viz-design 消费大纲 → SPEC + JSON → viz-echarts / viz-svg-flow 渲染
```

## Phase 1：数据理解与策略

### 1.1 理解数据

先回答：

- 这份数据**有什么**？（指标、维度、时间范围、样本量）
- 数据**质量如何**？（缺失值、异常值、口径一致性）
- 缺什么？如果关键维度缺失，先列出来

### 1.2 定义受众

**禁止**用"所有利益相关者"、"领导"等模糊定义。

| 维度 | 回答 |
|------|------|
| **决策者是谁** | 具体到人和角色 |
| **信任度** | 权威（他信你）/ 中立（需要证据）/ 怀疑（需要更多证明） |
| **受众是否多元** | 是 → 分开做，不要拼成一个谁都不满意的 |

### 1.3 Big Idea

一句话，包含 **洞察 + 建议**。

> 例：不是"供应商绩效数据分析"，而是"伽玛本季度履约率下滑 15%，建议启动备选供应商引入流程"

所有后续设计选择是否服务于这个 Big Idea？不是就去掉。

### 1.4 探索性 vs 解释性

| | 探索性（内部发现） | 解释性（对外沟通） |
|--|-------------------|-------------------|
| 目的 | 自己发现规律 | 让受众理解并采取行动 |
| 做法 | 多试，全保留 | 只展示"珍珠"（关键洞察） |
| 图表数 | 不限 | 精简（通常 3-7 张） |

**默认走解释性路线**，除非明确说是探索。

## Phase 5：叙事结构与大纲输出

> 产出叙事大纲（不包含图表类型选择、不指定渲染技能）。

### 5.1 三幕叙事结构

| 幕 | 内容 | 图表编号 |
|----|------|----------|
| **第一幕（铺垫）** | 设定场景、指出失衡、预览方案 | 图 1-2 |
| **第二幕（冲突）** | 用证据展开问题、对比案例、不行动的后果 | 图 3-5 |
| **第三幕（决议）** | 明确行动号召、回顾紧迫性 | 图 6-7 |

### 5.2 多图表排序规则

1. **先全局后细节**：先整体趋势，再拆解分项
2. **先问题后方案**：先"问题图"，再"解决方案图"
3. **先事实后推断**：先实际数据，再预测/建议
4. **每页/每张图只讲一件事**

### 5.3 载体适配

| 载体 | 规则 |
|------|------|
| **演讲** | 一页一图，精简，视觉强化口头观点 |
| **报告/邮件** | 多图一页可接受，确保自包含 |
| **仪表盘** | 表格可扫视，避免完整表格占满屏 |

### 5.4 数据伦理

论证类沟通（汇报、提案、建议）：
- **不得只呈现支持结论的数据**，必须提及已知反对证据或局限性
- 把反对点当作深度分析的证据，不是弱点

### 5.5 大纲输出格式

**这是 data-storytelling 的唯一核心产出**。使用以下 Markdown 结构：

```markdown
# [汇报/报告标题]

## Big Idea
[一句话：洞察 + 建议]

## 受众
- 决策者：[具体人和角色]
- 信任度：[权威/中立/怀疑]
- 载体：[演讲/报告/仪表盘]

## 叙事结构：三幕

### 第一幕：铺垫（场景 + 问题）
- **图 1**：[图表名称]
  - 功能：[展示什么]
  - 类型：[折线/条形/...]
  - 数据源：[哪些字段]
  - 高亮：[什么用红色，什么灰化]
  - 排序：[按什么排]
  - 标题：[行动导向的标题]
  - 数据源：[具体字段名或文件路径]

- **图 2**：[同上结构]

### 第二幕：冲突（证据 + 对比）
- **图 3**：[...]
- **图 4**：[...]

### 第三幕：决议（行动号召）
- **图 5**：[...]

## 反对证据与局限性
- [已知的反对数据或上下文]

## 底部总结
[一段话：起点值、终点值、关键拐点原因、建议行动]
```

## Phase 6：独立审查

> 大纲写完后，**不要自己审自己**。派一个子代理，只给 Markdown 大纲文档，不给任何背景上下文，让它以"第一次看到这份汇报的受众"视角评估。

### 6.1 审查代理的输入约束

| 给什么 | 不给什么 |
|--------|----------|
| Markdown 大纲全文 | 数据原始表格 |
| 图表呈现策略描述 | 背景上下文（如"这是给供应商季度汇报"） |
| Big Idea | 业务细节 |

**为什么不给上下文**：如果审查者知道背景，会"自动补全"逻辑缺口，失去发现问题的意义。

### 6.2 审查维度

审查代理从以下维度评估大纲：

| 维度 | 检查点 |
|------|--------|
| **Big Idea 清晰度** | 一句话能否说清？是否有洞察+建议？ |
| **图表必要性** | 每张图是否服务于 Big Idea？有没有"为了画图而画图"？ |
| **高亮合规** | 每张图的高亮是否 ≤10%？是否只高亮一个关键点？ |
| **排序一致性** | 多张图讲同一分类时，顺序是否一致？ |
| **叙事连贯性** | 上一页到下一页是否有逻辑递进？ |
| **数据真实性** | 标题是否夸大？是否有推断性语言（如"数据不撒谎"）？ |
| **标题行动导向** | 标题是否直接说出结论，还是空洞描述（如"销售数据"）？ |
| **反对证据** | 是否提及已知局限性或反对数据？ |
| **格式规范** | 是否使用正式结构（一、二、三）而非非正式叙事（第一幕、第二幕）？ |

### 6.3 审查输出格式

审查代理返回分级问题列表：

```
## P0（必须修复，阻塞进入渲染阶段）
- [ ] 问题描述 + 修复建议

## P1（强烈建议修复）
- [ ] 问题描述 + 修复建议

## P2（可选优化）
- [ ] 问题描述
```

### 6.4 审查后的行动

- **有 P0**：修复后重新审查（或快速自查确认 P0 已消除）
- **无 P0，有 P1**：修 P1 后传递给下游 viz-design
- **无 P0 无 P1**：直接传递给下游 viz-design

### 6.5 什么时候跳过审查

- 简单场景（1-2 张图的快速展示）
- 紧急汇报（时间窗口 < 30 分钟）
- 探索性分析（非对外沟通）

---

## 快速检查

大纲完成后问：

- [ ] Big Idea 一句话能说清？
- [ ] 每张图的功能描述都有理由支撑（功能→数据类型）
- [ ] 每张图携带了数据源引用（字段名或文件路径）
- [ ] 叙事顺序符合"先全局→细节、先问题→方案"？
- [ ] 包含了反对证据？
- [ ] 包含了完整的原始数据引用（供下游 viz-design 消费）
- [ ] **已走 Phase 6 独立审查？无 P0 阻塞？**
