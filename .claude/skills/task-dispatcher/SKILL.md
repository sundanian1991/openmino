# task-dispatcher — 任务调度器

> 需求澄清 + 任务分发。不亲自做，只做对。

---

## 触发词

"帮我搞定"、"做个东西"、"帮我弄个方案"、"需求如下"、"帮我执行"

---

## 四步流程

### 第一步：5W2H 需求澄清

**一次性输出所有问题**，不挤牙膏：

| 维度 | 问题 |
|------|------|
| **What** | 要做什么？交付物是什么形态？ |
| **Why** | 背景和目的是什么？解决什么问题？ |
| **Who** | 目标受众/使用对象是谁？ |
| **When** | 截止时间？紧急程度？ |
| **Where** | 在哪用？什么场景/平台？ |
| **How** | 有什么偏好/风格要求？ |
| **How much** | 体量/规模/范围多大？ |

**如果年老师已经说清楚了** → 跳过提问，直接进入第二步。

### 第二步：输出 Spec（需求规格说明书）

整理需求方回答，输出：

```
【需求 Spec】

1. 背景目标：
2. 交付物：
3. 目标受众：
4. 时间要求：
5. 场景平台：
6. 风格要求：
7. 体量范围：
8. 其他约束：
```

**等年老师确认"没问题"后再进入第三步。**

### 第三步：生成 Plan 并调度技能

根据 Spec 判断调用哪个技能：

| 需求类型 | 调用技能 | 说明 |
|----------|---------|------|
| PPT/演示文稿 | `ppt-master` | 专业演示文稿生成 |
| 前端页面/仪表盘/HTML 页面 | `mino-frontend` | 6 种 UI 类型 + 100+ 组件库 |
| 公文/报告/提案 | `official-doc` | 正式文档、供应商管理文档 |
| 供应商分析/沟通 | `supplier-mentor` | 供应商管理导师 |
| 数据分析/KPI | `判官` | 供应商业绩分析 |
| Excel/数据处理 | `xlsx` | 表格处理 |
| PDF 处理 | `pdf` | PDF 提取/合并/转换 |
| Word 文档 | `minimax-docx` | 高品质 DOCX |
| 工作总结/汇报 | `掌柜` | 向上汇报专家 |
| 正式函件/通知 | `文书` | 对外正式函件 |
| 制度/SOP | `师爷` | 管理制度、流程规范 |
| 沟通准备/会议提纲 | `信使` | 沟通准备专家 |
| 行业资讯/竞品 | `情报官` | 行业资讯搜索 |
| 文件整理/归档 | `管家` | 工作区整理 |
| 不确定/其他 | 自行判断 | 分析后选择最匹配技能 |

输出 Plan 格式：

```
【执行 Plan】

1. 调度技能：[技能名] — 理由：[为什么选它]
2. 输入参数：[Spec 中哪些信息传给该技能]
3. 预期产出：[最终交付物]
```

### 第四步：执行交付

调用对应技能完成最终结果，交付给年老师。

---

## 调度原则

1. **能调专家不调通用** — `ppt-master` > `general-purpose`
2. **能调内部不调外部** — 优先已有技能，不够再搜索
3. **复杂任务拆给多技能** — 如"做个供应商汇报PPT" → `判官` 分析数据 + `ppt-master` 做PPT

---

## 文件输出规范

**一个需求一个文件夹**，全部产出归入 `plans/`：

### 目录结构

```
plans/YYYY-MM-DD-主题描述/
├── spec.md        ← 第二步输出的需求规格说明书
├── plan.md        ← 第三步生成的执行 Plan
├── result.md      ← 第四步执行结果 + 交付物路径
└── [交付物]       ← 最终产物（如有独立文件）
```

### 命名规则

- 文件夹：`YYYY-MM-DD-需求主题`（与 `plans/` 现有风格一致）
- 每个文件按流程顺序写入，不跳过任何步骤

### 文件内容要求

**spec.md**：
> 【需求 Spec】
> 1. 背景目标：...
> 2. 交付物：...
> ...

**plan.md**：
> 【执行 Plan】
> 1. 调度技能：...
> 2. 输入参数：...
> 3. 预期产出：...

**result.md**：
> 【执行结果】
> - 完成状态：✅/❌
> - 交付物：[文件路径]
> - 备注：[异常/遗留]
