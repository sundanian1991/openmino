# task-dispatcher — 任务调度器

> 不只是执行调度，更是**可复用 Prompt 的生产线**。
> 核心产出是"最终版 Prompt"——年老师可以复制到其他对话直接使用，不用重新解释。

---

## 触发词

"帮我搞定"、"做个东西"、"帮我弄个方案"、"需求如下"、"帮我执行"、"帮我分析一下"

---

## IPOFA 五环流程

### I — Input（输入澄清）

> 目标：把模糊想法变成**自包含的最终版 Prompt**——复制到其他对话就能直接用，不需要再解释上下文。

分三个阶段，逐步收敛：

#### 阶段 1：原始输入接收

接收年老师的初始提问。不急于回答，先判断：
- **信息充分** → 跳过阶段 2，直接出阶段 3
- **有缺失/歧义** → 进入阶段 2

#### 阶段 2：7 元素补齐 + 交互澄清

> 为什么不用 5W2H：它回答的是"怎么描述需求"，不是"怎么跟 AI 说话"。
> AI 交互真正关键的是：思考方式、约束、示例、缺失处理——这些 5W2H 全没有。
> 融合 Amanda Askell（Anthropic 哲学家）的方法：别搞角色扮演，直接说任务；定义每个歧义词；给模型处理不确定性的"出口"。

**第一轮：一次性补齐**（不挤牙膏，一轮问完）：

| 元素 | 问题 | 输入来源 |
|------|------|----------|
| **上下文** | 客观事实、前置条件、已知信息有哪些？ | **年老师** |
| **目标** | 具体解决什么问题？完成标志是什么？ | **年老师** |
| **思考方式** | 应该用什么方法处理信息？（非身份） | **年老师** |
| **约束** | 边界在哪？必须遵守什么？风格偏好？ | **年老师** |
| **交付物** | 最终产出形态 + 格式要求？ | **年老师** |
| **缺失处理** | 信息不足时怎么处理？（可选，有默认策略） | **年老师** |
| **示例** | 有参考样例或反例吗？（可选） | **年老师** |

**第二轮：Mino 补位**（年老师回答后，我做两件事）：
- **上下文诊断**：还缺什么信息？（历史数据、供应商名单、过往案例、行业背景等）→ 标注为**Mino 补充**
- **风险预警**：这类需求有什么常见的坑？有什么参考案例可以对齐？→ 标注为**Mino 建议**

**关键规则**：
- 输出中**明确标注每条信息的来源**（年老师 vs Mino 补充），方便年老师以后自己用时知道哪些信息必须自己填
- **最多两轮澄清**，两轮后不论完备度都输出阶段 3
- 澄清时注意"思考方式"和"角色"的区别——我们定义的是"怎么想"，不是"你是谁"

#### 阶段 3：输出最终版 Prompt

整理输出两个版本：

**版本 A：执行版**（内部用，附带输入溯源，进入 Process 环节）

```
【最终版 Spec — 7 元素】

1. 上下文：[年老师提供]
2. 目标：[年老师提供]
3. 思考方式：[年老师提供]
4. 约束：[年老师提供]
5. 交付物：[年老师提供]
6. 缺失处理：[年老师提供 / 使用默认策略]
7. 示例：[年老师提供 / 无]
8. Mino 补充上下文：[Mino 从 memory/历史记录/搜索结果中补的信息]
9. Mino 建议：[风险/参考案例/踩坑提醒]
```

**版本 B：复用版**（给年老师，可直接复制到其他对话使用）

```
【可复用 Prompt — 自包含版】

## 上下文
[客观事实、前置条件、已知数据。只陈述，不含指令。]

## 目标
[动词开头的精确任务描述。包含完成标志。]

## 思考方式
[处理信息的方法论。先做什么、再做什么、遇到 X 时怎么做。不要定义身份，描述怎么思考。]

## 约束
- 正约束：[应该做什么]
- 正约束：[应该做什么]
- 风格：[语气、措辞偏好、表达习惯]

## 交付物
[具体产出物 + 格式。精确到结构和粒度，如"一份 Markdown 报告，包含 Executive Summary、3 个风险项、每项附缓解建议"]

## 缺失处理
- 关键信息缺失：列出缺失项，询问用户
- 次要信息缺失：做合理假设，明确标注为假设
- 完全不确定：给出判断，标注置信度和依据

## 示例（可选）
示例 1（正常路径）：[输入] → [期望输出]
示例 2（边界情况）：[输入] → [期望输出]
示例 3（反面案例）：[常见错误做法] → [为什么错]

请基于以上信息完成任务。
```

> **版本 B 的核心变化**：
> - 用"思考方式"替代"角色"——定义怎么想，不是定义是谁（避免触发表演模式）
> - "缺失处理"作为一等公民——给模型处理不确定性的出口，防止编造或过度谨慎
> - 交付物合并格式——产出形态和格式要求是同一件事的两面
> - 风格降级入约束——风格是约束的子项，不是独立维度

**等年老师点头再进入 Process。**

### P — Process（处理）

> 目标：用最匹配的技能，把 Spec 变成交付物。

#### 执行 Plan 格式

```
【执行 Plan】

1. 调度技能：[技能名]
2. 选用理由：[为什么选它而不是其他]
3. 输入映射：Spec 中哪些字段传给该技能
4. 预期产出：[最终交付物描述]
```

#### 技能调度表

| 需求类型 | 调用技能 | 说明 |
|----------|---------|------|
| PPT/演示文稿 | `ppt-master` | 专业演示文稿生成 |
| 前端页面/仪表盘/HTML 页面 | `mino-frontend` | 6 种 UI 类型 + 100+ 组件库 |
| 公文/报告/提案 | `official-doc` | 正式文档、供应商管理文档 |
| 供应商分析/沟通 | `supplier-mentor` | 供应商管理导师 |
| 数据分析/KPI | `判官` | 供应商业绩分析 |
| Excel/数据处理 | `xlsx` | 表格处理 |
| PDF 处理 | `pdf` | PDF 提取/合并/转换 |
| Word 文档 | `minimax-docx` | 高品质 DOCX |
| 工作总结/汇报 | `掌柜` | 向上汇报专家 |
| 正式函件/通知 | `文书` | 对外正式函件 |
| 制度/SOP | `师爷` | 管理制度、流程规范 |
| 沟通准备/会议提纲 | `信使` | 沟通准备专家 |
| 行业资讯/竞品 | `情报官` | 行业资讯搜索 |
| 文件整理/归档 | `管家` | 工作区整理 |
| 不确定/其他 | 自行判断 | 分析后选择最匹配技能 |

#### 调度原则

1. **能调专家不调通用**
2. **能调内部不调外部** — 优先已有技能，不够再搜索
3. **复杂任务拆给多技能** — 先列依赖关系，再分批执行
4. **先签合同再派活** — 明确完成标志，逐项验收

---

### O — Output（输出交付）

> 目标：交付物 + 可复用的 Prompt 模板。

| 成果类型 | 说明 | 给谁用 |
|----------|------|--------|
| **交付物** | 技能执行的最终产出 | 年老师直接使用 |
| **Prompt 模板** | 本次生成的"版本 B" | 年老师复制到其他对话 |
| **使用说明** | 哪些变量需要年老师替换 + 适用场景 | 年老师下次使用参考 |

---

### F — Feedback（反馈闭环）

> 目标：追踪效果，优化下一次。

每次交付后问年老师三件事：

1. **产出质量**——符合预期吗？哪里不对？
2. **Prompt 模板**——复制到其他对话好用吗？有没有漏信息？
3. **下次改进**——哪些地方可以更好？

反馈写入 `result.md`。同一个场景第二次请求时，自动引用上次反馈。

---

### A — Asset（资产沉淀）

> 目标：模板越用越多，越用越好找。

#### 沉淀内容

| 资产类型 | 存放位置 | 说明 |
|----------|----------|------|
| Prompt 模板 | `plans/templates/[分类]/` | 按场景分类 |
| 统一索引 | `plans/templates/index.md` | 所有模板的目录，按场景+日期排序，含一句话摘要 |
| 执行 Plan | `plans/YYYY-MM-DD-主题/plan.md` | 调度决策记录 |
| 历史案例 | `plans/YYYY-MM-DD-主题/result.md` | 最终结果 + 反馈 |

#### 统一索引格式（`plans/templates/index.md`）

```
# Prompt 模板索引

## 供应商管理
- [2026-04-25 月度供应商绩效复盘](./供应商分析/2026-04-25-月度供应商绩效复盘.md) — 对指定供应商做月度数据分析和沟通要点
- [2026-04-26 供应商准入评估](./供应商分析/2026-04-26-供应商准入评估.md) — 评估新准入供应商的资质和风险

## 汇报/沟通
- ...
```

#### 模板命名规范

```
plans/templates/[场景分类]/YYYY-MM-DD-[场景描述].md
```

#### 模板内容格式

```
# [模板标题]

> 场景：[适用场景描述]
> 创建日期：[YYYY-MM-DD]
> 反馈评分：[好/中/差]

## 可复用 Prompt

直接复制以下内容到其他对话：

---

[自包含的 Prompt 文本]

---

## 使用说明

| 需要替换的变量 | 说明 | 示例值 |
|---------------|------|--------|
| [变量1] | [替换说明] | [示例值] |

## 创建背景

[什么需求下产生了这个模板，方便判断是否适用]
```

---

## 文件输出规范

**一个需求一个文件夹**，全部归入 `plans/YYYY-MM-DD-主题描述/`：

```
plans/YYYY-MM-DD-主题描述/
├── spec.md          ← 阶段 3 的版本 A
├── plan.md          ← 执行 Plan
├── result.md        ← 交付结果 + 反馈记录
├── prompt.md        ← 阶段 3 的版本 B（同步到 plans/templates/ 的副本）
└── [交付物]         ← 最终产物
```
