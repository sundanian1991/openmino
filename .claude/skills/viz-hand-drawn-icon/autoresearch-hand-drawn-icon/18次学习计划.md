# 18 次 API 学习计划 — 掌握手绘 SVG 风格

**目标**：通过 18 次 Quiver API 调用，掌握手绘 SVG 的设计决策模式，达到 90% 独立完成能力。

**原则**：API 是为了学习，不是为了产出。每次调用都要提炼可复用的设计模式。

---

## Phase 1: 复杂场景深度学习（2-3 次）

### 目标
建立"复杂场景设计模式"，掌握多元素组合、层次控制、繁而不乱的平衡。

### R1: 东京猫（61-88 path）- 已完成 ✅
- **学习重点**：复杂生物的结构分解（头/身/四肢/尾巴/装饰）
- **提炼模式**：5 个主要部分 → 每部分 10-20 path → 总计 25-30 组件
- **关键洞察**：复杂度不是 88 个独立 path，而是可拆解的组件组合

### R2: 东京猫完整版（88 path）
- **目标**：验证 R1 学到的"复杂模式"是否适用于更细节的版本
- **学习重点**：细节如何增加但不破坏整体感
- **预期提炼**：细节添加的"边界原则"（何时该停）

### R3: 复杂器物（如：电扇/台灯）
- **目标**：验证"生物类模式"是否可迁移到"器物类"
- **学习重点**：功能性结构（可动部件）如何表达
- **预期提炼**：器物类的结构分解模式

---

## Phase 3: 边界测试 + 补漏（3-4 次）

### 目标
测试极端案例，补充遗漏的设计决策场景。

### R15: 极简图标（1-5 path）
- **测试**：超简单场景是否需要 API
- **预期**：可能不需要，建立代码片段库

### R16: 极繁场景（100+ path）
- **测试**：复杂度上限在哪里
- **预期**：提炼"繁而不乱"的终极原则

### R17: 特殊视角（如：仰视/俯视）
- **测试**：非常规视角如何处理
- **预期**：补充视角选择的决策规则

### R18: 自定义挑战（根据前 17 次的薄弱环节）
- **目标**：针对性弥补
- **示例**：如果发现"阴影处理"薄弱，就专门测试阴影
