# 编排模式详解

> 三种 Agent 编排模式的详细指南


## 模式对比

| 维度 | 串行编排 | 并行编排 | 混合编排 |
|------|---------|---------|---------|
| **执行时间** | 长 | 短 | 中等 |
| **质量保证** | 高 | 中 | 高 |
| **适用场景** | 强依赖任务 | 独立子任务 | 大多数复杂任务 |
| **复杂度** | 低 | 中 | 中高 |


## 模式 1：串行编排

### 流程图

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│   Spec   │ -> │   Test   │ -> │   Code   │ -> │  Review  │
│  Agent   │    │  Agent   │    │  Agent   │    │  Agent   │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                        ↓
                                               ┌──────────┐
                                               │  Human   │
                                               │  验收    │
                                               └──────────┘
```

### 适用场景

- 任务步骤有强依赖关系
- 前一步输出 = 后一步输入
- 需要严格质量保证
- 时间要求不紧迫

### 优点

- 每步有验证，方向正确
- 问题早发现，返工成本低
- 流程清晰，易于追踪

### 缺点

- 执行时间长
- 资源利用率低（单 Agent 工作）

### 执行模板

```markdown
【串行任务】[任务名]

→ Spec Agent: 编写 Spec 文档
→ 等待完成后继续
→ Test Agent: 拆解测试用例
→ 等待完成后继续
→ Code Agent: 实现功能
→ 等待完成后继续
→ Review Agent: 代码审查
→ 等待完成后继续
→ Human: 最终验收
```


## 模式 2：并行编排

### 流程图

```
                    ┌──────────────┐
                    │   Spec Agent │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ↓            ↓            ↓
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │ Agent A  │ │ Agent B  │ │ Agent C  │
       │ (模块 A)  │ │ (模块 B)  │ │ (模块 C)  │
       └────┬─────┘ └────┬─────┘ └────┬─────┘
              │            │            │
              └────────────┼────────────┘
                           ↓
                    ┌──────────┐
                    │Integration│
                    └────┬─────┘
                         ↓
                    ┌──────────┐
                    │  Human   │
                    └──────────┘
```

### 适用场景

- 任务可拆分为独立子任务
- 多模块可并行开发
- 时间敏感，需要快速交付

### 优点

- 执行时间短
- 资源利用率高（多 Agent 同时工作）

### 缺点

- 需要集成测试
- 模块间可能有隐性依赖
- 问题发现晚，返工成本高

### 执行模板

```markdown
【并行任务】[任务名]

**Spec 阶段**
→ Spec Agent: 编写整体 Spec

**并行执行**
→ Agent A: [模块 A 职责] — 预期 [输出 A]
→ Agent B: [模块 B 职责] — 预期 [输出 B]
→ Agent C: [模块 C 职责] — 预期 [输出 C]

→ 并行执行中...（预计 X 分钟）

**集成阶段**
→ Integration: 模块集成 + 集成测试
→ Human: 最终验收
```


## 模式 3：混合编排（推荐）

### 流程图

```
┌──────────┐    ┌──────────┐
│   Spec   │ -> │   Test   │
│  Agent   │    │  Agent   │
└──────────┘    └──────────┘
                        ↓
         ┌──────────────┼──────────────┐
         ↓              ↓              ↓
  ┌──────────┐   ┌──────────┐   ┌──────────┐
  │ Code A   │   │ Code B   │   │ Code C   │
  └────┬─────┘   └────┬─────┘   └────┬─────┘
         │              │              │
         └──────────────┼──────────────┘
                        ↓
                 ┌──────────┐
                 │  Review  │
                 │  Agent   │
                 └────┬─────┘
                      ↓
                 ┌──────────┐
                 │  Human   │
                 └──────────┘
```

### 适用场景

- 大多数复杂任务（推荐默认使用）
- 需要前期方向正确 + 中期效率 + 后期质量

### 优点

- 前期保证方向正确（Spec + Test 串行）
- 中期提高效率（Code 多 Agent 并行）
- 后期保证质量（Review 串行）

### 缺点

- 需要检查点管理
- 复杂度较高

### 执行模板

```markdown
【混合编排】[任务名]

**阶段 1：Spec + Test（串行）**
→ Spec Agent: 编写 Spec 文档
→ 检查点 1：确认 Spec 完整无误
→ Test Agent: 拆解测试用例
→ 检查点 2：确认测试覆盖核心规则

**阶段 2：编码（并行）**
→ Code Agent A: [职责] — 预期 [输出]
→ Code Agent B: [职责] — 预期 [输出]
→ 并行执行中...

**阶段 3：审查 + 验收（串行）**
→ Review Agent: 代码审查
→ 检查点 3：确认无重大问题
→ Human: 最终验收
```


## 选择决策树

```
收到任务
    ↓
任务有强依赖吗？
    ├─ 是 → 串行编排
    └─ 否 ↓
任务可拆分为独立子任务吗？
    ├─ 是 → 并行编排
    └─ 否 ↓
需要平衡效率和质量吗？
    └─ 是 → 混合编排（推荐）
```


## 检查点管理

混合编排需要在关键位置设置检查点：

| 检查点 | 位置 | 检查内容 |
|--------|------|---------|
| **检查点 1** | Spec 完成后 | Spec 是否清晰完整？ |
| **检查点 2** | Test 完成后 | 测试是否覆盖核心规则？ |
| **检查点 3** | Code 完成后 | 测试是否全部通过？ |
| **检查点 4** | Review 完成后 | 审查意见是否全部修复？ |


*选择合适的编排模式，平衡效率与质量。*
