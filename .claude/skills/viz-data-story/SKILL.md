# Data Storytelling — 数据沟通策略与编排

> 来源：《Storytelling with Data》全书 30 skill 整合 | 核心产出 = Markdown 大纲

---

## 定位

data-storytelling **不画图、不做 PPT**。它的唯一产出是一份 **Markdown 大纲**，定义：

- 受众是谁、要他做什么（Big Idea）
- 需要几张图、每张图什么类型、为什么选它
- 每张图怎么呈现（高亮什么、排序规则、标注什么）
- 图的叙事顺序（先讲什么后讲什么）

大纲完成后，**交给渲染技能去画和组装**。

## 工作流

```
拿到数据
  ↓
data-storytelling：分析 + 策略 + 大纲（产出 Markdown）
  ↓
独立审查：子代理只看大纲，不带背景上下文（找盲点）
  ↓
迭代：修复 P0/P1 问题
  ↓
渲染技能：画图（chart-visualization / echarts-visualization）
  ↓
组装技能：做 PPT（html-ppt-skill / presentation-skill）
```

**简单场景**（只画一两张图）：跳过大纲，直接走 Phase 1→2→3，给出图表选择建议和渲染指令。
**正式场景**（汇报/提案/多图表）：走完整流程，产出大纲。

## Phase 1：数据理解与策略

### 1.1 理解数据

先回答：

- 这份数据**有什么**？（指标、维度、时间范围、样本量）
- 数据**质量如何**？（缺失值、异常值、口径一致性）
- 缺什么？如果关键维度缺失，先列出来

### 1.2 定义受众

**禁止**用"所有利益相关者"、"领导"等模糊定义。

| 维度 | 回答 |
|------|------|
| **决策者是谁** | 具体到人和角色 |
| **信任度** | 权威（他信你）/ 中立（需要证据）/ 怀疑（需要更多证明） |
| **受众是否多元** | 是 → 分开做，不要拼成一个谁都不满意的 |

### 1.3 Big Idea

一句话，包含 **洞察 + 建议**。

> 例：不是"供应商绩效数据分析"，而是"伽玛本季度履约率下滑 15%，建议启动备选供应商引入流程"

所有后续设计选择是否服务于这个 Big Idea？不是就去掉。

### 1.4 探索性 vs 解释性

| | 探索性（内部发现） | 解释性（对外沟通） |
|--|-------------------|-------------------|
| 目的 | 自己发现规律 | 让受众理解并采取行动 |
| 做法 | 多试，全保留 | 只展示"珍珠"（关键洞察） |
| 图表数 | 不限 | 精简（通常 3-7 张） |

**默认走解释性路线**，除非明确说是探索。

## Phase 2：图表清单

> 每张图的"选什么、为什么、怎么呈现"。

### 2.1 功能决定形式

先回答："受众需要**做什么**或**理解什么**？"再选图。

| 功能目标 | 图表形式 |
|----------|----------|
| 看时间趋势 | 折线图 |
| 分类对比 | 条形图（水平优先） |
| 看相关性 | 散点图 |
| 看构成/占比 | 堆叠条形图（禁止饼图） |
| 1~3 个关键数字 | **大字展示，不是图** |
| 表格数据看模式 | 热力图 |
| 看流入流出+缺口 | 正负堆叠条形图 |
| 调研排名数据 | 水平堆叠条（排名分段） |
| 精确数值查找 | 极简表格（不同单位时） |

### 2.2 禁止项

| 禁止 | 替代方案 |
|------|----------|
| 饼图/环形图 | 水平条形图 / 100% 堆叠条 / 大数字 |
| 3D 效果 | 永远不用 |
| 双 Y 轴 | 直接标值 / 上下分图共享 X 轴 |
| 截断条形图轴 | 强制零基线 |
| 默认多色轮播 | 3 色：红 `#C13531` / 深蓝 `#293C54` / 灰 `#CDCECD` |
| 渐变/阴影/发光 | 纯色 + 透明度 |

### 2.3 比例变化的处理

对比"之前 vs 之后"的占比，**绝对不要用两个饼图**：

| 场景 | 方案 |
|------|------|
| 只关心一个核心指标 | 大数字 + 样本量 + 基线对比 |
| 全分布对比 + 保留顺序 | 分组条形图 |
| 关心两端对比 | 100% 水平堆叠条 |
| 关心变化幅度和排名 | **坡度图** |

## Phase 3：每张图的呈现策略

> 决定了"用什么图"之后，决定"这张图怎么让人一眼看懂"。

### 3.1 高亮规则（≤10% 法则）

- 只高亮 ≤10% 的元素（关键数据点/线），其余灰化 `#CDCECD`
- 关键数据用 `#C13531`（红色）
- 永远不要同时高亮多个不相关的点

### 3.2 排序规则

| 数据类型 | 排序方式 |
|----------|----------|
| 分类（无内在顺序） | 按意图排序（最优/最差/未使用），不按字母 |
| 分类（有内在顺序） | 保持自然顺序（年龄段、学历、绩效等级） |
| 多张图讲同一分类 | 保持顺序不变，用颜色/标注突出次要洞察 |
| 堆叠条形图 | 核心关注段放固定位置（底部或顶部），跨图一致 |

### 3.3 时间序列轴标签

- 日线数据 → 每 5 或 7 天标一次，不要每个点都标
- 可选加"第 1 周""第 2 周"超级分类
- Y 轴金额：千万级保留完整数字，亿级改用"（百万元）"

### 3.4 预测与实际区分

| 元素 | 实际 | 预测 |
|------|------|------|
| 线条 | 粗实线 | 细虚线，50% 透明度 |
| 背景 | 正常 | 淡色阴影隔离 |
| 数值 | 只标关键点 | 每点都标 |
| 锚点 | 最后实际点：白色填充 + 红边框 + 粗体数字 | — |
| X 轴 | ACTUAL | FORECAST |

**不能混成一条无区分的线。**

### 3.5 极简表格

- 去重边框，留白分隔
- 数字右对齐，文本左对齐
- 报告保留完整表格，演讲提取关键洞察、表格放附录

### 3.6 热力图

- 单调渐变（低→深 / 高→浅），加图例
- 保留单元格内数字（模式识别 + 精确读数）

### 3.7 认知负荷检查

每个元素问："它的信息价值值得认知成本吗？"
- 去掉：装饰边框、多余颜色、冗余标签
- 标题要行动导向（不是"销售数据"，是"Q3 销售低于目标 12%"）
- **10 秒测试**：非专业人士能否 10 秒内理解要点？

## Phase 4：复杂图表模板

### 4.1 多线图解构（>4 条且重叠）

| 策略 | 做法 | 适用 |
|------|------|------|
| 逐个高亮 | 其他灰化，目标加粗+色 | 演讲 |
| 垂直小多图 | 上下排列，共享 X 轴，相同 Y 范围 | 关心各自趋势 |
| 水平小多图 | 左右排列，共享 Y 轴 | 特定时间点对比 |
| 组合 | 小多图 + 每面板高亮一条 | 报告 |

**强制**：所有面板轴范围相同。

### 4.2 正负堆叠条形图

- 零基线 = 当前状态
- 正贡献堆上方，负贡献堆下方
- **缺口用空心条**（最关键信息）
- 视觉路径：标题 → 缺口数字 → 描述 → 支撑数据

### 4.3 调研排名堆叠条

- 总 % 降序排列（最高在上）
- 水平放置，类别名在左
- 排名从左到右：第 1 名（最深）→ 第 2 名 → 第 3 名（最浅）
- 最多 4 个排名级别

## Phase 5：叙事结构与大纲输出

> 把前面所有决策整合成一份 Markdown 大纲。

### 5.1 三幕叙事结构

| 幕 | 内容 | 图表编号 |
|----|------|----------|
| **第一幕（铺垫）** | 设定场景、指出失衡、预览方案 | 图 1-2 |
| **第二幕（冲突）** | 用证据展开问题、对比案例、不行动的后果 | 图 3-5 |
| **第三幕（决议）** | 明确行动号召、回顾紧迫性 | 图 6-7 |

### 5.2 多图表排序规则

1. **先全局后细节**：先整体趋势，再拆解分项
2. **先问题后方案**：先"问题图"，再"解决方案图"
3. **先事实后推断**：先实际数据，再预测/建议
4. **每页/每张图只讲一件事**

### 5.3 载体适配

| 载体 | 规则 |
|------|------|
| **演讲** | 一页一图，精简，视觉强化口头观点 |
| **报告/邮件** | 多图一页可接受，确保自包含 |
| **仪表盘** | 表格可扫视，避免完整表格占满屏 |

### 5.4 数据伦理

论证类沟通（汇报、提案、建议）：
- **不得只呈现支持结论的数据**，必须提及已知反对证据或局限性
- 把反对点当作深度分析的证据，不是弱点

### 5.5 大纲输出格式

**这是 data-storytelling 的唯一核心产出**。使用以下 Markdown 结构：

```markdown
# [汇报/报告标题]

## Big Idea
[一句话：洞察 + 建议]

## 受众
- 决策者：[具体人和角色]
- 信任度：[权威/中立/怀疑]
- 载体：[演讲/报告/仪表盘]

## 叙事结构：三幕

### 第一幕：铺垫（场景 + 问题）
- **图 1**：[图表名称]
  - 功能：[展示什么]
  - 类型：[折线/条形/...]
  - 数据源：[哪些字段]
  - 高亮：[什么用红色，什么灰化]
  - 排序：[按什么排]
  - 标题：[行动导向的标题]
  - 渲染技能：[chart-visualization / echarts-visualization]
  - 渲染指令：[传递给渲染技能的具体参数]

- **图 2**：[同上结构]

### 第二幕：冲突（证据 + 对比）
- **图 3**：[...]
- **图 4**：[...]

### 第三幕：决议（行动号召）
- **图 5**：[...]

## 反对证据与局限性
- [已知的反对数据或上下文]

## 底部总结
[一段话：起点值、终点值、关键拐点原因、建议行动]
```

大纲完成后，按每张图的"渲染技能"和"渲染指令"，**依次调用对应的渲染技能完成图表生成和组装**。

## Phase 6：独立审查

> 大纲写完后，**不要自己审自己**。派一个子代理，只给 Markdown 大纲文档，不给任何背景上下文，让它以"第一次看到这份汇报的受众"视角评估。

### 6.1 审查代理的输入约束

| 给什么 | 不给什么 |
|--------|----------|
| Markdown 大纲全文 | 数据原始表格 |
| 图表呈现策略描述 | 背景上下文（如"这是给供应商季度汇报"） |
| Big Idea | 业务细节 |

**为什么不给上下文**：如果审查者知道背景，会"自动补全"逻辑缺口，失去发现问题的意义。

### 6.2 审查维度

审查代理从以下维度评估大纲：

| 维度 | 检查点 |
|------|--------|
| **Big Idea 清晰度** | 一句话能否说清？是否有洞察+建议？ |
| **图表必要性** | 每张图是否服务于 Big Idea？有没有"为了画图而画图"？ |
| **高亮合规** | 每张图的高亮是否 ≤10%？是否只高亮一个关键点？ |
| **排序一致性** | 多张图讲同一分类时，顺序是否一致？ |
| **叙事连贯性** | 上一页到下一页是否有逻辑递进？ |
| **数据真实性** | 标题是否夸大？是否有推断性语言（如"数据不撒谎"）？ |
| **标题行动导向** | 标题是否直接说出结论，还是空洞描述（如"销售数据"）？ |
| **反对证据** | 是否提及已知局限性或反对数据？ |
| **格式规范** | 是否使用正式结构（一、二、三）而非非正式叙事（第一幕、第二幕）？ |

### 6.3 审查输出格式

审查代理返回分级问题列表：

```
## P0（必须修复，阻塞进入渲染阶段）
- [ ] 问题描述 + 修复建议

## P1（强烈建议修复）
- [ ] 问题描述 + 修复建议

## P2（可选优化）
- [ ] 问题描述
```

### 6.4 审查后的行动

- **有 P0**：修复后重新审查（或快速自查确认 P0 已消除）
- **无 P0，有 P1**：修 P1 后直接进入渲染阶段
- **无 P0 无 P1**：直接进入渲染阶段

### 6.5 什么时候跳过审查

- 只画 1-2 张图的简单场景
- 紧急汇报（时间窗口 < 30 分钟）
- 探索性分析（非对外沟通）

## 渲染技能路由

data-storytelling 产出大纲后，按每张图的"渲染技能"字段，路由到对应技能。

### 图表渲染层

| 图表类型 | 首选技能 | 备选 | 输出 | 为什么 |
|----------|----------|------|------|--------|
| 标准数据图（柱/线/散点/堆叠/饼/漏斗） | `echarts-visualization` | `chart-visualization` | HTML / PNG | ECharts 支持交互（tooltip、缩放、筛选），优先选 |
| 特殊图表（桑基/地图/热力图/矩形树图/关系图/流向图/雷达图/日历图/河流图） | `echarts-visualization` | — | HTML | chart-visualization 不支持这些类型 |
| 信息图（结构化信息+视觉符号，压缩复杂信息为直观符号） | `infographic-creator` | — | HTML | 用 AntV Infographic，适合知识普及/概念解释 |
| 技术图（流程图/架构图/时序图/泳道图/四象限/嵌套/树图/金字塔） | `diagram-design` | — | SVG HTML | 专业技术图表，inline SVG |

### 演示组装层

| 场景 | 首选技能 | 适用条件 | 输出 |
|------|----------|----------|------|
| **重要演示/需要引人入胜** | `cinematic-ui` | 有详细大纲，需要独特视觉风格、强叙事感、电影级页面呈现。**只要有一份好的 Markdown 大纲，它会自动找到风格和呈现形式** | HTML |
| **品牌级汇报/高设计品质+交互+幻灯片** | `huashu-design` | 需要品牌识别度（Logo+产品图+品牌色）、反 AI slop、高保真幻灯片、可导出 PDF/PPTX。**出稿慢但品质高，流程：事实验证→资产协议→Junior pass→Full pass** | HTML / PDF / PPTX |
| **杂志风/电子墨水/发布会风格** | `magazine-web-ppt` | 需要高级设计感，WebGL 流体背景，衬线+非衬线混排，横向翻页 | HTML |
| **通用数据汇报/快速出稿** | `html-ppt-skill` | 多风格可选，静态 HTML PPT，键盘导航，快速生成 | HTML |
| **动画丰富/全屏不滚动/演讲为主** | `frontend-slides` | 重动画，创意视觉，适合 Talk/Pitch，每页严格 100vh | HTML |
| **演讲稿 + PPT 一键生成** | `presentation-skill` | 快速生成，62 种风格，含演讲稿 | HTML |
| **数据仪表盘/复杂前端页面** | `mino-frontend` | 需要高保真、复杂交互、仪表盘布局 | HTML |

### 不使用

| 技能 | 原因 |
|------|------|
| `chart-visualization` | echarts-visualization 已覆盖所有图表类型且输出交互 HTML，仅做备选 |

### 演示技能选择建议

| 维度 | cinematic-ui | huashu-design | magazine-web-ppt | html-ppt-skill | frontend-slides |
|------|-------------|---------------|-----------------|---------------|-----------------|
| **视觉独特性** | ★★★★★（每次保证不重复） | ★★★★★（反 AI slop 体系） | ★★★★ | ★★★ | ★★★★ |
| **叙事感** | ★★★★★（storyboard-first） | ★★★★（位置四问+叙事角色） | ★★★★ | ★★ | ★★★ |
| **品牌识别度** | ★★★ | ★★★★★（核心资产协议） | ★★★ | ★★ | ★★★ |
| **出稿速度** | 慢（decisions→storyboard→spec） | 慢（事实验证→资产→Junior→Full） | 中 | 快 | 中 |
| **需要的输入** | 详细大纲 | 品牌信息/参考 | 主题+结构 | 主题即可 | 主题+结构 |
| **适合场景** | 述职、提案、重要汇报 | 品牌汇报、发布会、高保真原型 | 分享、杂志风发布 | 日常快速汇报 | Talk/Pitch |
| **可导出** | HTML | HTML + PDF + PPTX | HTML | HTML | HTML |

### 演示技能快速决策

```
这次汇报重要吗？
├── 非常重要（述职/提案/给大领导）
│   ├── 有详细大纲 → cinematic-ui
│   └── 有品牌/产品要展示 → huashu-design
│
├── 一般重要（周会/月会/内部汇报）
│   ├── 要高级设计感 → magazine-web-ppt
│   └── 快速出稿 → html-ppt-skill
│
└── 演讲为主（Talk/分享会）
    ├── 重动画 → frontend-slides
    └── 需要演讲稿 → presentation-skill
```

### 路由决策树

```
要什么？
├── 单张图表 → 选"图表渲染层"
│   ├── 柱/线/散点/堆叠/饼/漏斗 → echarts-visualization
│   ├── 桑基/地图/热力图/雷达/特殊 → echarts-visualization
│   ├── 信息图（概念解释/知识普及） → infographic-creator
│   └── 流程图/架构图/技术图 → diagram-design
│
└── 演示文稿（多张图表+文字） → 先画图 → 再选"演示组装层"
    ├── 重要汇报+有详细大纲 → cinematic-ui
    ├── 品牌汇报+高保真+要导出PPTX → huashu-design
```
