# 通用可视化工作流

> 当输入为文字/数据/想法时，走此工作流。
> 不限领域——商业分析、个人规划、学习笔记、生活决策均可使用。
> 核心原则：先出 SPEC 供确认，不直接渲染。

---

## 完整工作流（5 Phase）

```
输入文字/数据
  ↓
Phase 1: 解析 — 识别逻辑关系、提取数据、识别 So What
  ↓
Phase 2: 匹配 — 匹配咨询原型 C01-C16、判定风格
  ↓
Phase 3: 构建 — 按 12+4 项编写 SPEC
  ↓
Phase 4: 门控 — 运行 12 问质量检查
  ↓
Phase 5: 输出 — 输出 SPEC 供确认，确认后委托渲染
```

---

## Phase 1: 解析

**目标**：从输入中提取结构化信息。

### 1.1 识别逻辑关系

判断输入在表达什么类型的逻辑关系：

| 关系类型 | 识别标志 | 走向 |
|---------|---------|------|
| 分解/MECE | "由哪些部分组成"、"拆解"、"分类" | → C01 |
| 变化驱动 | "什么导致"、"同比/环比"、"增量拆解" | → C02/C12 |
| 战略定位 | "优先级"、"定位"、"四象限"、"重要紧急" | → C03 |
| 时间叙事 | "历程"、"节点"、"何时发生" | → C04 |
| 评估对比 | "对比"、"排名"、"哪个好" | → C05/C13 |
| 瓶颈识别 | "卡在哪"、"转化"、"流失"、"衰减" | → C06 |
| 趋势拐点 | "趋势"、"为什么涨/跌"、"变化" | → C07 |
| 构成演变 | "组成"、"占比变化"、"结构" | → C08 |
| 差距目标 | "差多少"、"目标"、"进度"、"KPI" | → C09/C14 |
| 相关洞察 | "关系"、"规律"、"两个指标" | → C10 |
| 层级分解 | "架构"、"汇报"、"分类体系" | → C11 |
| 根因分析 | "根因"、"问题在哪"、"假设" | → C16 |
| 流向追踪 | "去了哪里"、"资金流向"、"客户来源" | → C15 |

### 1.2 提取关键数据

从输入中提取：
- **数值**：所有数字及其单位
- **标签**：每个数值对应的维度/类别名
- **时间**：时间范围、时间点
- **比较基准**：均值、目标值、去年同期、行业基准

### 1.3 识别 So What

从输入文字中找出或推断：
- 作者暗示了什么结论？
- 这张图要支持什么论点？
- 读者看完应该采取什么行动？

如果输入中没有暗示，标注"未明确，建议"。

---

## Phase 2: 匹配

**目标**：选择最佳图表原型和视觉风格。

### 2.1 图表原型匹配

1. 根据 Phase 1.1 的逻辑关系 → 初步匹配 1-2 个 C 系列原型
2. 同时参考 viz-pattern 72 模式，看是否有更匹配的非咨询模式
3. 选最佳匹配，记录选择理由

**匹配冲突处理**：
- 如果 C 系列和 72 模式都适用 → 优先 C 系列（咨询场景更专业）
- 如果多个 C 系列都适用 → 选最窄匹配（最具体的那个）
- 例：对比场景，C05 比 C13 更具体 → 选 C05

### 2.2 风格自动判定

按 `consulting-style-guide.md` 的判定规则：

```
扫描输入关键词和内容调性
  ├─ 分析/评估/排名/对比/业绩/汇报 → 正式（默认）
  ├─ 增长/策略/建议/方向/创新/规划 → 前瞻
  ├─ 观点/争议/风险/立场/质疑 → 鲜明
  ├─ 笔记/学习/读书/旅行/生活/兴趣/总结 → 轻松
  ├─ 手绘/草图/涂鸦/随性 → 手绘
  ├─ 用户显式指定 → 遵循指定
  └─ 以上皆无 → 正式（默认）
```

记录选定风格。

---

## Phase 3: 构建 SPEC

**目标**：编写完整的咨询级 SPEC 文档（12+4 项）。

### 3.1 基础 12 项（遵循 spec-template.md）

```
## 【可视化目的】
## 【想传达什么】
## 【结论】
## 【思路】
## 【模式选择】
## 【标题】
## 【视觉编码】
## 【数据组织】
## 【标注策略】
## 【图例与辅助】
## 【布局】
```

### 3.2 咨询扩展 4 项

在基础 12 项之后，追加：

```
## 【Exhibit 编号】
- 图 X / Exhibit X（用于多图文档中的引用，从 1 开始编号）

## 【Source 来源行】
- 数据来源 + 样本量 + 时间范围
- 格式："Source: [系统/报告名], N=[样本量], [起止时间]"
- 如输入未提供 → 标注"待补充"

## 【So What 洞察框】
- 1-2 句话，不是数据摘要，是分析结论
- 例："方案确认环节 54% 转化率低于均值 65%，
       说明问题在流程设计而非供应商能力。"

## 【Action 行动暗示】
- 读者看完这张图应该采取什么行动
- 例："重新设计方案确认流程，而非增加供应商培训投入"
```

### 3.3 风格注入

根据 Phase 2.2 判定的风格，在 SPEC 中注入：

- 颜色 Token（从 consulting-style-guide.md 取对应风格）
- 排版规格（字号、字重）
- 布局规格（如 McKinsey 的 Exhibit 框架）
- 标题语气（冷静/前瞻/鲜明）

---

## Phase 4: 质量门控

**目标**：确保 SPEC 达到咨询级质量标准。

按 `consulting-quality-gates.md` 执行 12 问检查：

1. 逐项检查 Q1-Q12
2. P0 不通过 → 标注失败项，说明原因，**打回 Phase 3 重修**
3. P1 不通过 → 标注建议优化项，记录到 SPEC 末尾的"待优化"段
4. 全部通过 → 进入 Phase 5

输出格式见 `consulting-quality-gates.md` 的"检查输出格式"。

---

## Phase 5: 输出 SPEC 供确认

**目标**：输出完整 SPEC 文档，等待年老师确认。

### 输出格式

```
---

## 可视化 SPEC — [标题]

**模式**：[C 编号]-[名称]
**风格**：[McKinsey/BCG/Economist]
**渲染目标**：[viz-echarts / viz-svg-flow / viz-data-viz]

### 第一部分：叙事意图

【可视化目的】...
【想传达什么】...
【结论】...
【思路】...

### 第二部分：视觉执行

【模式选择】...
【标题】...
【视觉编码】...
【数据组织】...
【标注策略】...
【图例与辅助】...
【布局】...

### 第三部分：咨询扩展

【Exhibit 编号】...
【Source 来源行】...
【So What 洞察框】...
【Action 行动暗示】...

---

**质量门控**：P0 [X/5] 通过 | P1 [Y/7] 通过 | 结论：[通过/打回]

---

年老师，SPEC 如上。请确认：
1. 图表类型是否合适？
2. 标题结论是否准确？
3. 高亮和标注是否突出你想强调的点？

确认后我委托渲染技能出图。如需调整，告诉我改哪里。
```

### 确认后委托渲染

年老师确认后，按以下格式委托：

```
用 [渲染技能] 渲染以下 SPEC：
- 模式：[C 编号]-[名称]
- 风格：[风格名]
- [SPEC 全文]
请遵循 consulting-style-guide.md 的 [风格名] 配色规范。
```

## 与 viz-pattern 现有工作流的关系

本工作流是 viz-pattern 现有 6 步工作流的**专业化分支**：

| viz-pattern 通用流程 | 咨询工作流 | 关系 |
|---------------------|-----------|------|
| Step 1: 意图识别 | Phase 1: 解析 | 通用版更细化逻辑关系识别 |
| Step 2: 模式匹配 | Phase 2: 匹配 | 增加 C01-C16 原型 + 5 种风格判定 |
| Step 3: 编写 SPEC | Phase 3: 构建 | 增加 4 项扩展字段 |
| Step 4: 委托渲染 | Phase 5: 确认后委托 | 增加 Phase 4 质量门控 |
| Step 5: 自检 | Phase 4: 门控 | 用 12 问替代通用自检 |
| Step 6: 验收 | Phase 5: 确认 | 输出 SPEC 供人工确认 |

**关键区别**：咨询工作流在渲染前增加一道人工确认环节（Phase 5），确保 SPEC 正确后再出图。
