# viz-pattern — 视觉模式翻译器

> 核心定位：不是画图工具，是**视觉翻译引擎**。
> 输入：抽象概念、非结构化数据、文字描述、复杂关系
> 输出：完整的可视化 SPEC（模式选择 + 标题 + 数据组织 + 标注策略 + 叙事结构）
> 执行：委托 viz-echarts / viz-chart / viz-svg-flow 等渲染技能画出来

---

## 一、核心原理

**图表思维是认知压缩。** 文字是线性的，图像是并行的。一张好图让大脑跳过解码阶段，直达模式识别。

**选图本身就是洞察。** 选漏斗说明你看到了衰减关系，选齿轮说明你看到了联动机制。图的选择先于图的绘制。

**88 种模式 = 88 种"概念→图形"的翻译模板。** 不需要从零想，按需调用。

---

## 二、技能边界

| 本技能负责 | 不负责 |
|-----------|--------|
| 概念→模式匹配（选什么图） | 实际画图（交给 viz-echarts / viz-chart） |
| SPEC 编写（标题/标注/布局/叙事） | 配色设计（各渲染技能自带配色体系） |
| 视觉叙事策略（先讲什么后讲什么） | 数据清洗（交给数据技能） |

## 四、三层架构

**渐进加载原则**：先读 `references/library-index.md` 决定下一步加载什么，不要一次性加载所有参考文件。

```
Layer 1  意图识别 + 模式匹配    ← 概念→图形翻译（本文件）
Layer 2  叙事意图 + SPEC 编写   ← references/spec-template.md
         自动选择器              ← references/auto-selector.md
Layer 3  视觉执行 + 渲染委托    ← references/execution-steps.md
                              ← references/visual-guidance-elements.md
                              ← references/anti-patterns.md
                              ← references/verification-standards.md
                              ← references/decision-rules.md
                              ← references/concept-pattern-map.md
```

```
收到输入（概念/数据/文字/问题）
  ↓

  ↓

  ↓

  ↓

  ↓

  ↓

```

## 五、工作流（旧版，保留为快速参考）

```
收到输入（概念/数据/文字/问题）
  ↓
Step 1: 意图识别 — 你要表达什么？
  ↓
Step 2: 模式匹配 — 从 72 种中选最佳
  ↓
Step 3: 编写 SPEC — 标题/数据组织/标注/叙事/布局
  ↓
Step 4: 委托渲染 — 交给 viz-echarts / viz-chart / viz-svg-flow 执行
  ↓
Step 5: 验收 — 对照 SPEC 检查产出
```

### Step 1: 意图识别

先判断输入类型，再判断核心诉求：

| 输入类型 | 判断问题 | 走向 |
|---------|---------|------|
| 结构化数据 | 数据有几个维度？要表达趋势/对比/占比/关系？ | → 分类 A（数据图表） |
| 抽象概念 | 这个概念的本质是什么？层级/流程/平衡/增长/风险？ | → 分类 B/C（类比/模拟） |
| 文字描述 | 这段文字在讲什么关系？因果/递进/并列/转化？ | → 分类 B/C/D |
| 复杂问题 | 需要拆解成几个子问题？每个子问题用什么图？ | → 多图组合 |

### Step 2: 模式匹配

**决策树：**

```
有精确数据？
├─ 是 → 数据有几个维度？
│  ├─ 1 维（单指标）→ 柱状图(13)、折线图(07)、面积图(08)
│  ├─ 2 维（x/y 关系）→ 散点图(09)、气泡图(05)
│  ├─ 3 维（x/y/大小）→ 气泡图(05)
│  ├─ 占比 → 饼图(01，≤3 类)、矩形树图(16)、堆叠条(14)
│  ├─ 趋势 → 折线图(07)、面积图(08)
│  ├─ 多层级 → 旭日图(10)、矩形树图(16)、抽象树(27)
│  ├─ 流程/转化 → 桑基图(20)、漏斗(32)、甘特图(15)
│  ├─ 多维对比 → 雷达图(22)、并列比较(26)
│  └─ 网络/关系 → 弦线图(21)、弧线图(19)、桑基图(20)
│
└─ 否 → 要表达的关系本质是什么？
   ├─ 层级/优先级 → 金字塔(31)、同心图(03)、层次(55)
   ├─ 衰减/筛选 → 漏斗(32)、冰山(49)
   ├─ 循环/闭环 → 循环轮(34)、循环系统(60)
   ├─ 联动/依赖 → 齿轮(39)、桑基图(20)
   ├─ 平衡/权衡 → 天平(42)、杠杆(41)
   ├─ 中心辐射 → 辐轮(33)、宇宙(53)

需要表达逻辑关系？（分解/变化/对比/因果/层级/流向）

```

### Step 3: 编写 SPEC

> 详细模板见 `references/spec-template.md`。SPEC 是**视觉叙事剧本**，不是渲染指令。
必须包含两部分共 12 项：

**第一部分：叙事意图（SPEC 的核心，必须先写）**
```
【可视化目的】为什么要做这张图？认知缺口是什么？
【想传达什么】核心信息（只有一句话）
【结论】读者应该得出什么判断？
【思路】为什么选这个图？视觉叙事路径是什么？
```

**第二部分：视觉执行（渲染指令）**
```
┌─────────────────────────────────────────┐
│ 【模式选择】                            │
│ - 模式编号 + 名称                       │
│ - 为什么选这个（1 句话说明匹配理由）    │
│                                         │
│ 【标题】                                │
│ - 主标题：结论性标题（不是描述性）      │
│ - 副标题：时间范围 + 对象               │
│                                         │
│ 【视觉编码】                            │
│ - X/Y 轴编码什么                        │
│ - 颜色编码什么                          │
│ - 大小/形状编码什么                     │
│                                         │
│ 【数据组织】                            │
│ - 数据字段清单                          │
│ - 排序规则                              │
│ - 聚合规则                              │
│                                         │
│ 【标注策略】                            │
│ - 高亮 ≤10%                             │
│ - 标注内容（写原因+幅度，不写数字）     │
│                                         │
│ 【图例/辅助】                           │
│ - 图例/脚注/特殊说明                    │
│                                         │
│ 【布局】                                │
│ - 画布尺寸/标题位置/留白/配色           │
└─────────────────────────────────────────┘
```

### Step 4: 委托渲染

> 执行细节见 `references/execution-steps.md` + `references/visual-guidance-elements.md`。

根据模式类型选择渲染技能：

| 模式类型 | 渲染技能 | 说明 |
|---------|---------|------|
| 数据图表（01-30） | viz-echarts | ECharts 支持所有标准图表 |
| 流程/架构 | viz-svg-flow | SVG 流程图、架构图 |
| 抽象类比（31-48） | viz-svg-flow 或 viz-chart | 复杂结构用 SVG，简单用 chart |
| 模拟类比（49-60） | viz-svg-flow | 需要用 SVG 手绘形态 |
| 寓言（61-72） | viz-svg-flow | 场景类用 SVG |

委托时传入完整 SPEC，渲染技能按 SPEC 执行。

### Step 5: 出图前自检

> 见 `references/anti-patterns.md` — 不同 AI 常见错误模式。

### Step 6: 验收

> 见 `references/verification-standards.md` — 麦肯锡/贝恩级质量标准。

对照 SPEC 逐项检查：

| 检查项 | 标准 |
|-------|------|
| **叙事意图** | 四要素齐全：目的/传达/结论/思路，缺一不可 |
| **目的** | 写了"认知缺口"——读者看图前后应该有什么变化 |
| **传达** | 核心信息只有一句话，没有多条 |
| **结论** | 是判断不是描述，有数据/证据支撑 |
| **思路** | 解释了"为什么选这个图型"，不是"选了什么图型" |
| 模式匹配 | 选的图确实匹配要表达的概念 |
| 标题 | 是结论不是描述 |
| 高亮 | ≤10%，只突出 1 个关键点 |
| 数据 | 字段齐全、排序正确 |
| 标注 | 写了原因+幅度 |
| 配色 | ≤2 ramp，遵循 13-VISUALIZATION.md |

## 七、技能生态联动

> viz-pattern 不是孤立的，是可视化技能链路的**上游决策层**。
> 必须与以下文件/技能配合使用。

### 与 13-VISUALIZATION.md 的关系

**13-VISUALIZATION.md 是视觉执行的宪法**，viz-pattern 的 SPEC 必须符合 13 的硬规则：

| 13 的规则 | viz-pattern 的 SPEC 中如何体现 |
|----------|-------------------------------|
| 标题写结论，不写描述 | SPEC 的【标题】项必须是结论性的 |
| 颜色是信号不是装饰 | SPEC 的【视觉编码】必须说明颜色编码什么 |
| 高亮 ≤10% | SPEC 的【标注策略】限定高亮范围 |
| 默认 Warm，≤2 ramp | SPEC 中注明色系选择理由 |
| 一个图一个核心洞察 | SPEC 的【想传达什么】只写一句话 |
| 标题 15-18px/600/Warm800 | SPEC 的【布局】引用 13 的字号层级 |
| 间距/边框/圆角 | SPEC 的【布局】引用 13 的间距标准 |
| 禁止渐变/阴影/emoji | 决策规则第 3 条"可画性检查"过滤掉 |

**流程**：viz-pattern 选模式 + 写 SPEC → 13-VISUALIZATION.md 校验 SPEC 合规 → 委托渲染。

### 与其他可视化技能的分工

```
viz-data-story       ← 数据讲故事的策略层（大纲 → 几张图 → 叙事顺序）
  ↓
viz-pattern          ← 视觉模式翻译层（每张图 → 选什么模式 → 写 SPEC）
  ↓
viz-echarts          ← 数据图表渲染层（ECharts HTML）
viz-chart            ← 通用图表渲染层（Chart.js）
viz-svg-flow         ← SVG 手绘渲染层（流程图/架构图/抽象类比）
viz-infographic      ← 信息图渲染层（长图/海报）
viz-data-viz         ← Python matplotlib 渲染层
viz-editorial        ← 编辑式图文渲染层
```

**调用时机**：

| 场景 | 先走哪个技能 | 为什么 |
|------|------------|--------|
| "拿这批数据做个汇报" | viz-data-story → viz-pattern → viz-echarts | 先定叙事大纲，再定每张图的视觉方案 |
| "这个概念怎么可视化" | viz-pattern → viz-echarts/svg-flow | 直接翻译概念为 SPEC |
| "帮我画一张趋势图" | viz-pattern → viz-echarts | 简单场景跳过 data-story |
| "做个信息图海报" | viz-pattern → viz-infographic | 抽象概念→视觉方案→信息图渲染 |

### 与 viz-echarts 的交接

委托 viz-echarts 时，传入 SPEC 并附加以下参数：

```
委托指令示例：
"用 viz-echarts 渲染以下 SPEC：
- 模式：22-雷达图
- 标题：毅航健康度全面领先，3 家低于基准线
- 数据字段：supplier_name, achievement_rate, peak_ratio, relative_efficiency, fci_score, turnover_rate, compliance_rate
- 高亮：只高亮毅航（Warm 500），其余灰化（Stone 300）
- 标注：毅航顶点标'综合得分 92，超基准 15 分'
- 基准线：灰色虚线圆圈，标'行业基准线'
- 画布：800×550
请生成 ECharts HTML，遵循 13-VISUALIZATION.md 配色规范。"
```

### 与 viz-svg-flow 的交接

委托 viz-svg-flow 时，需额外说明手绘风格细节：

```
委托指令示例：
"用 viz-svg-flow 渲染以下 SPEC：
- 模式：49-冰山
- 标题：客诉问题冰山：表面 12% 可量化，88% 需深挖
- 手绘要素：水线以上小三角（可见问题），水线以下大三角（隐藏根因）
- 标注：水线标'可量化指标'，底部标'需深挖的根因'
- 画布：600×500
请生成 SVG，遵循 Warm 色系，线性克制风格。"
```

### 与 viz-data-story 的配合

viz-data-story 产出大纲后，viz-pattern 负责将大纲中的"需要一张 XX 类型的图"翻译为具体模式：

```
viz-data-story 大纲说："需要一张图展示供应商分层"
  ↓ viz-pattern 翻译：
  - 概念 = 分层/层级
  - 模式选择 = 31-金字塔（首选）或 03-同心图（备选）
  - 理由 = 金字塔天然表达自下而上的层级关系
  - SPEC = 完整编写
  ↓ 委托 viz-echarts 或 viz-svg-flow 执行
```

## 八、使用示例

**示例 1：年老师问"怎么可视化供应商健康度？"**

→ Step 1: 意图识别 — 多维评估，无精确数据，需要整体态势感知
→ Step 2: 模式匹配 — 雷达图(22) 适合多维能力画像
→ Step 3: 自动选择器 — hash("供应商健康度") → 麦肯锡风格 + 指标阵列开场 + 全景式弧线
→ Step 4: 编写 SPEC（12 项完整，含自动选择记录）
→ Step 5: 委托 viz-echarts 执行
→ Step 6: 验收：对照 SPEC 逐项检查

**示例 2：年老师说"把供应商转化的问题可视化"**

→ Step 1: 意图识别 — 流程+转化，有阶段数据
→ Step 2: 模式匹配 — 漏斗(32) 表达递减转化
→ Step 3: 自动选择器 — hash("供应商转化") → 贝恩风格 + 漏斗开场 + 诊断式弧线
→ Step 4: SPEC → 瓶颈环节用 Coral 色标注
→ Step 5: 委托 viz-echarts（漏斗图原生支持）
→ Step 6: 验收

**示例 3：年老师说"我想表达我们供应商管理是系统工程"**

→ Step 1: 意图识别 — 抽象概念，无数据，表达"系统运转"
→ Step 2: 模式匹配 — 72-机器（表达系统化运作）或 39-齿轮（表达联动机制）
→ Step 3: 自动选择器 — hash("系统工程") → IBM Carbon 风格 + 中心辐射构图
→ Step 4: SPEC → 选择 72-机器，画布 600×500，SVG 手绘
→ Step 5: 委托 viz-svg-flow 执行
→ Step 6: 验收
