ljg-rank

给一个领域，找出背后真正撑着它的几根独立的力。十几个现象砍到不可再少的生成器——砍完能把现象一个个生回来，才算数。

类别 方法论框架
行数 43
复杂度 极简
生成方式 手工编写

它是什么

一句话：你给它一个领域（比如"教育"、"电商"、"恋爱"），它帮你找出这个领域背后真正在起作用的几根独立的力。

打个比方

想象你面前有一团乱麻——一个领域里有几十个现象、几十个说法。ljg-rank 做的事情就像是：帮你找到这团乱麻里真正的几根线头。只要抓住这几根线头，整团麻都能理顺。

关键是：它不是让你"总结要点"或"列出关键因素"。它要求更严格——找到的东西必须能反过来生成所有现象。能生回来，才算找对了。

基本信息

有多长 43 行。整个 skill 只有 43 行，比很多人写的周报都短。
有多复杂 极简。没有分阶段、没有 Agent、没有联网搜索。就是一条直线：输入 → 思考 → 写出来。
谁造的 手工编写，不是自动生成的。
输出什么 一篇散文（不是表格！），存成 org 文件。

它怎么运转

从头到尾就 5 步，简单得像一条直线。没有分支、没有循环、没有"如果……就……"。

1
你给它一个领域

比如"短视频"、"婚姻"、"SaaS"。就一个词或短语，不需要更多解释。

入口极简——降低使用门槛，不让你在"该输入什么"上纠结。
2
它先告诉你"秩不是什么"

不是关键要素、不是核心原则、不是总结要点。先把你脑子里的错误预期清掉。

就像老师上课前先说"今天不是考试"——让你放下紧张，进入正确的心态。
3
用四个判据来验证（核心步骤）

找到候选的"生成器"之后，必须过四道关卡：能生回来吗？少一个行不行？它们真的独立吗？能预测新东西吗？全过才算数，任何一道没过就推倒重来。

这是整个 skill 的心脏。43 行里有一半的力量都在这四道关卡上。
4
写成一篇散文

不是填表，不是列清单。要求：想一口气读完、记得住、有从混沌到极简的落差感。验证过程也要融进故事里。

就像一部好电影——高潮不是另外加的彩蛋，而是剧情本身的一部分。
5
存成文件

自动存到 ~/Documents/notes/ 目录，带时间戳，org 格式。

自动归档——你不用操心存哪里，以后想找也知道去哪找。

它的巧妙之处

这个 skill 只有 43 行，但藏了好几个聪明的设计。不是拿清单去套的——是读完之后真觉得"这个想法挺妙"的地方。

手法一：四道越来越严的关卡

找到候选的"生成器"之后，不是拍脑袋说"感觉对"就完了。要过四道关：

①
能生回来吗？

用你找到的生成器，能不能把每个现象都解释回来？一个都不能漏。

就像拼图——你说这几块是关键，那就用它们把整幅图拼出来看看。

②
少一个行不行？

随便拿掉一个生成器，看看有没有现象解释不了了。如果拿掉了也没事，说明它是多余的。

就像桌子腿——四条腿的桌子，拿掉一条就歪了，说明每条都有用。如果拿掉一条还稳，那这条腿就是多余的。

③
它们真的是不同的力吗？

每两个生成器之间，能不能找到一个真实案例：一个变了，另一个没变？

就像"身高"和"体重"——有人高但轻，有人矮但重。如果两个指标永远一起变，那它们可能就是同一个东西换了个名字。

④
能预测新东西吗？

用这些生成器，能不能推出你一开始没想到的现象？而且这个现象在现实中确实存在？

这是最厉害的一关。前三关只能证明"说得通"，这一关才能证明"真的对"。就像一个理论不光能解释已知的事，还能预测未知的事——那才是真本事。

妙在哪：这四道关卡的顺序是精心设计的——从"能解释"到"不多余"到"真独立"到"能预测"，一道比一道严。不是拍脑袋判断好不好，而是有客观标准可以检验。

手法二：先说"不是什么"

skill 一上来就说："秩不是关键要素，不是核心原则，不是总结要点。"

为什么这么做？因为大多数人听到"找出一个领域的核心"，脑子里第一反应就是"总结几个要点"。如果不先把这个误解清掉，后面说什么都白搭——AI 会按照"总结要点"的思路去做，而不是真正去找生成器。

妙在哪：用否定句比正面定义更快。与其花 200 字解释"秩到底是什么"，不如先用 3 个"不是"把最常见的误解堵死。读完这三个"不是"，你就知道该往哪个方向想了。

手法三：把验证变成故事的一部分

很多工具会把"验证"放在最后当附录——"请检查以下几项"。ljg-rank 不这么干。它明确说："验证本身就是叙事的一部分。"

"拆掉一个生成器，世界会变成什么样"——这本身就是一个好故事。"用生成器推出一个意想不到的现象"——这本身就是一个好结尾。

妙在哪：如果验证是单独的附录，读者会跳过它。但如果验证就是故事的高潮，读者不但不会跳过，还会觉得这是最精彩的部分。把"必须做的苦差事"变成了"最好看的段落"。

手法四：只定终点不定路线

"想一口气读完、记得住、有落差"——这三条要求没有告诉 AI 具体怎么写，但清楚地定义了什么算写得好。没有规定章节、没有规定格式、没有规定小标题。

为什么不规定？因为这个 skill 的难点不在"怎么做"，在"做得好不好"。写再多流程指令也不能让散文写得更好。

妙在哪：好的约束不是告诉你"走哪条路"，而是告诉你"终点在哪"。路怎么走，你自己找。这给了 AI 最大的创作空间，同时又不会跑偏——因为终点是明确的。

看不见的选择

这些选择没有写在 skill 里面，但它们决定了这个 skill 为什么好使。每个选择背后都有道理。

为什么写散文，不写表格？

作者选了

散文。没有固定章节、没有规定格式、没有小标题。怎么写由 AI 自己决定。

也可以选

结构化模板，像很多 skill 那样有严格的 Phase 流程和固定输出格式。

为什么选这个

降秩的输出不是一份报告，是一次认知体验。读者需要感受从"这个领域好乱"到"原来就这两三根线"的落差。这种落差感，表格给不了，只有故事能给。

输出是体验就用散文，输出是系统就用模板。

为什么没有"中间检查点"？

作者选了

一口气跑完，中间不停。

也可以选

做到一半先暂停让用户检查，确认没问题再继续。很多复杂 skill 都这么做。

为什么选这个

降秩是一次连贯的思考过程，中间打断会破坏思路。而且它有内置的质量控制——四个判据本身就是检查机制。不需要额外的"停下来检查"，因为整个过程就是在检查。

流程越长、环节越多，越需要中间检查。流程短且自带验证的，就不需要。

为什么只有 43 行？

作者选了

极简。只写最核心的东西，其余全部留给 AI 自由发挥。

也可以选

写 200+ 行，详细规定每一步怎么做、输出什么格式、遇到什么情况怎么处理。

为什么选这个

这个 skill 的难点不在流程，在输出质量。写再多流程指令也不能让散文写得更好。反而是那三条写作要求（"想一口气读完、记得住、有落差"）比任何详细流程都管用——它们定义了终点，但不规定路线。

如果 skill 的难点在"怎么做"，就写详细流程。如果难点在"做得好不好"，就写清楚标准，让 AI 自己找路。

为什么用 org 格式而不是 markdown？

作者选了

org 格式。Emacs 生态的标准格式。

也可以选

markdown（.md），更通用，几乎所有编辑器都支持。

为什么选这个

这是一个纯粹的个人偏好选择——作者用 Emacs，org 格式是他的主力笔记格式。对 skill 的功能没有任何影响，换成 markdown 也完全一样。

输出格式是最容易改的部分。如果你不用 Emacs，换成 .md 就行，不影响任何核心逻辑。

可以搬走用的

"四道关卡"的筛选逻辑

如果你的 skill 需要"从一堆东西里挑出真正重要的几个"，这四道关卡（能生成 → 不多余 → 真独立 → 能预测）可以直接搬过去。比如：筛选核心指标、提炼关键原则、精简功能列表。关键是递进的严格程度——每一道都比上一道更难过。

"先说不是什么"的开场方式

如果你的 skill 处理的概念容易被误解，开头先用几个"不是"把最常见的误解堵死。比正面解释快得多，而且能防止 AI 一开始就跑偏。

"定义终点不规定路线"的写法

"想一口气读完、记得住、有落差"——这三条没有告诉 AI 具体怎么写，但清楚地定义了什么算写得好。当 skill 的难点在输出质量而非流程时，这种写法比详细步骤更有效。

别照搬的

极简结构

ljg-rank 能只写 43 行，是因为它不需要联网、不需要多个 Agent、不需要处理各种意外情况。如果你的 skill 需要搜索信息、协调多个步骤、或者处理不同类型的输入，43 行肯定不够。极简不是目标，够用才是。

没有中间检查

只有当你的 skill 是"一口气完成的单次思考"并且自带验证机制时，才能省掉中间检查。多步骤的 skill 如果不设检查点，前面错了后面全白做。

散文输出

散文适合"认知体验"类的输出。如果你的 skill 输出的是需要被其他工具引用的结构化数据，或者需要用户逐项检查的清单，散文反而会添乱。

可以改进的地方

加一段"什么时候别用这个工具"

有些领域可能不适合降秩（比如高度混沌、没有规律的系统，或者领域太小没什么可降的）。加一段说明，帮用户判断什么时候该用、什么时候别用。

给一个示例片段

"想一口气读完"到底是什么感觉？给一段 200 字的示例散文，让 AI 有个具体的参照物。光说"写得好"太抽象，看一个例子胜过一百句描述。
