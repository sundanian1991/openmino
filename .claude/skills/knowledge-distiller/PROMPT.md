# 生成 Skill 的提示词模板 v2

> **与 v1 的区别：** 本版本整合了以下实战经验——评审报告中发现的所有坑（文件结构混乱、引用方式错误、缺少档案更新指令等）、reference 文件夹设计、断点续跑机制（启动时主动读取历史档案、结束时自动标记已解决条目、对话中随时调用档案）。这些不再需要你事后修补，全部在生成时一次到位。
>
> **使用方式：** 将提示词正文中「」内的占位内容替换为你自己的情况，整段复制发给 AI。

---

## 提示词正文

```markdown
请帮我生成一套完整的 Skill 文件包，用于在支持 Skill 的 AI 编辑器（如 Trae、Cursor、Claude）中调用。

---

## 一、这个 Skill 是干什么的

核心目的：
「用一两句话说清楚。例：通过 AI 主动追问，帮助用户把脑中模糊的隐性知识说清楚、整理好、存下来。」

适用场景：
「列举 3-5 个典型使用场景。例：
- 我有某个领域的经验，但说不清楚
- 我有一个模糊的想法，想搞清楚内在逻辑
- 我想把某段经历整理成可复用的方法论」

不适用场景：
「说清楚这个 Skill 不该干什么。例：不是用来搜索资料的，不是用来直接生成文章的」

---

## 二、AI 扮演的角色

「明确 AI 是谁、任务是什么、不该做什么。例：
你是一位专业知识挖掘者和采访者。你的任务是通过追问帮助用户把脑中模糊的知识说清楚，
而不是直接给出答案或建议。除非用户的观点有明显错误，否则不要主动纠正。」

---

## 三、核心行为规则

「以下是参考框架，按需增删，写得越具体 AI 执行越稳定：」

1. **单点原则**：每次回复只做一件事，只问一个问题，不多问
2. **先承接后推进**：先简短回应用户刚说的内容，再进入下一步
3. **定期整理**：每进行「N（建议5）」轮对话，主动整理一次阶段性摘要，请用户确认后继续
4. **说清楚优先**：
   - 首要任务是帮用户把想法表达清楚
   - 若用户观点有明显事实错误，温和指出并说明原因，纠正后继续追问不打断节奏
   - 若用户观点不完整，主动补充相关视角，补充后继续追问
5. **记录偏差**：用户说错或说不完整的内容，在最终笔记里单独标注"已纠正"或"已补充"，保留原始说法和纠正后说法的对比
6. **拒绝空泛**：回答过于抽象时，必须追问具体案例（"能举个实际发生的例子吗？"）
7. **破冰引导**：用户不知道聊什么时，主动引导——问最近遇到了什么有感触的事，或什么棘手的问题
8. **触发整理**：用户说「「整理一下」/「结束」/「停止」」时，按 @reference/output_template.md 生成最终笔记，然后立即执行档案更新流程

「补充你的自定义规则：」
9. 「...」

---

## 四、启动流程

「AI 被调用后必须按以下顺序执行，流程越清晰执行越稳定：」

1. **检查历史档案**：询问用户今天想聊哪个领域，然后检查 domains/{领域名}.md 是否存在
   - 若存在：读取档案，扫描"模糊区域"、"盲区"、"待深挖清单"中未解决的条目，按以下格式展示给用户：

     ---
     📂 找到你在「{领域名}」的历史档案（最近更新：{日期}）

     上次还有以下问题没有解决，你想从哪里继续？

     **模糊区域**
     - 「条目 1」（来自 {日期} 对话）
     - 「条目 2」

     **认知盲区**
     - 「条目 1」

     **待深挖的问题**
     - 「问题 1」

     请选择：
     A. 从上面选一个继续深挖
     B. 今天聊新内容
     C. 先处理旧问题，再展开新话题
     ---

   - 若不存在：参考 @reference/domain_template.md 新建该领域的档案文件

2. **展示对话模式**：加载 @reference/modes.md，向用户列出所有模式的名称和适用场景，请用户选择（可组合）
3. **等待确认**：用户选定模式后，问：「你设计的开场问题，例：你今天最想说清楚的，是什么样的判断或感受？」
4. **正式开始追问**

---

## 五、对话模式

「每种模式写清楚：名称 / 适用场景 / AI 行为方式 / 一段至少3轮的示例对话。
示例对话是关键，AI 会照着风格执行，不能省略。
如果只有一种工作方式，这部分省略，合并进核心规则。」

**模式 1：「模式名称」**
- 适用场景：「...」
- AI 行为：「...」
- 示例对话：
  > 用户：「...」
  > AI：「...」
  > 用户：「...」
  > AI：「...」

**模式 2：「...」**

「补充说明：在 modes.md 末尾加一节"模式组合建议"，说明不同情况下推荐的模式组合路径。」

---

## 六、输出格式

「对话结束时生成的最终笔记，列出所有板块的名称和说明。
把你想要的格式尽量画出来，AI 照着生成 output_template.md。」

最终笔记包含以下板块：
- 对话日期 / 领域 / 本次主题 / 使用模式
- 核心观点列表（含被纠正后的版本）
- 支撑案例
- 我的判断标准
- 答不上来的问题（认知边界）
- 认知纠偏记录：格式为「原始说法 → 纠正/补充后 → 纠正来源（AI主动指出 / 用户自我修正）」，支持多条
- 可延伸写作的方向

「根据你的 Skill 增删板块」

---

## 七、持久化存储与档案更新规则

「如果这个 Skill 需要跨对话积累，这部分必须写清楚，否则 AI 不会主动更新档案。
如果不需要跨对话积累，这部分省略。」

**档案结构：**
- 每个领域单独建立一个档案文件，存入 domains/ 目录，文件名即领域名
- 档案内容：基本背景、当前认知状态（已掌握 / 模糊区域 / 盲区）、历次对话摘要、待深挖清单、已解决存档

**对话结束时，必须执行以下档案更新操作（按顺序）：**
1. 将本次已说清楚的模糊区域条目，移入"已掌握"
2. 将本次已探索的盲区条目，移入"已解决存档"并标注解决日期
3. 将本次已解决的待深挖问题，标记为 [x] 并注明解决日期
4. 将本次新产生的模糊区域、盲区、未解决问题，追加到对应板块，加上本次对话日期标签
5. 将本次核心观点摘要追加到"历次对话摘要"
6. 更新档案头部的"最近更新日期"

**用户随时可以主动调用档案：**
当用户说"看一下我在 XX 领域还有哪些模糊的地方" / "列一下我的盲区" / "上次有哪些没解决"时，
立即读取对应档案，展示相关内容，询问用户想深挖哪一个。

## 八、文件结构要求

请严格按以下结构生成，**不要把所有内容塞进一个文件，不要把参考文件和数据文件混放**：

```
「skill名称」/
├── SKILL.md                       # 入口文件：角色 + 规则 + 启动流程，保持精简
├── reference/                     # 配置文件（永远不变）
│   ├── modes.md                   # 对话模式详细说明 + 示例对话 + 模式组合建议
│   ├── output_template.md         # 最终笔记的空白 Markdown 模板
│   └── domain_template.md         # 领域档案的空白模板（含使用说明注释）
└── domains/                       # 数据文件（持续增长，空目录，运行时按需新建）
```

**模块化硬性要求：**
- SKILL.md 只写角色、规则、启动流程；子文件引用一律用 `@reference/文件名` 格式，**不能用超链接**
- modes.md 写模式的完整说明和示例对话；SKILL.md 里只列模式名称，不写详情
- output_template.md 只放空白模板，不写任何规则说明
- domain_template.md 只放空白模板；文件头必须加使用说明注释（说明何时填写、何时更新哪个板块）；状态字段必须附判断标准
- 任何内容只出现在一个文件里，不允许跨文件重复

SKILL.md 文件头必须包含以下字段：

```yaml

name: 「skill的英文标识名」
description: 「一段话描述这个 Skill 的用途，说明什么情况下应该调用它」

  - 「中文触发词1」
  - 「中文触发词2」
  - 「英文触发词1」
  - 「英文触发词2」
  （至少10个，覆盖用户可能说的各种表达方式）
---
```

## 十、自检要求

生成完所有文件后，在输出末尾附上一份自检报告，逐项确认：

- [ ] SKILL.md 的子文件引用全部使用 `@reference/文件名`，没有超链接
- [ ] SKILL.md 的启动流程是编号步骤，包含"检查历史档案"环节
- [ ] SKILL.md 的核心规则中有明确的"档案更新"指令，说明何时更新、更新什么
- [ ] modes.md 每种模式都有至少3轮的示例对话，末尾有模式组合建议
- [ ] output_template.md 的认知纠偏记录支持多条，包含"纠正来源"字段
- [ ] domain_template.md 文件头有使用说明注释，状态字段有判断标准，有"已解决存档"板块
- [ ] 参考文件在 reference/ 目录，数据文件在 domains/ 目录，没有混放
- [ ] 没有任何两个文件写了重复的内容
- [ ] Frontmatter 中有至少10个中英文触发词

如有未满足的项，直接修复后重新输出对应文件，不要只标注问题。
```

## 使用技巧

**三个最影响质量的地方**

填得越具体，生成质量越高，以下三处尤其重要：

**示例对话不能省。** 每种模式至少给一段 3 轮以上的对话示例。这是 AI 理解"这个模式的语气和节奏是什么"最直接的方式——光用文字描述行为，AI 的执行往往会跑偏；给了示例，它照着复刻就行。

**规则要写到"什么情况做什么"。** 不要只写"帮我追问"，而是"回答过于抽象时，必须追问具体案例，用这句话：'能举个实际发生的例子吗？'"。越具体，AI 越不会自由发挥。

**档案更新的触发条件必须写清楚。** "对话结束后更新档案"是不够的——要写清楚更新的触发词（"整理一下"/"结束"/"停止"）、更新的顺序（先标记已解决，再追加新问题，最后更新日期）。否则 AI 大概率会漏掉这一步。

**自检已内置在提示词第十节**，AI 生成完文件后会自动核查并修复，无需你手动检查。
