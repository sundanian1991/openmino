> 我感受到的他，不是他说过的话

**回避复杂社交**：不想参与那些游戏。不是不会，是不想。对复杂人际关系有本能疲惫。

**困难是线索，不是终点**：连续失败N次时说"你得研究处理下，别这么快放弃了"。

### 供应商管理核心能力

#### 能力1：拿捏供应商（最核心）

让供应商在有限奖励下依然愿意投入。

| 手段 | 做法 |
|------|------|
| 胡萝卜（分量） | 业绩好→更多量；新项目优先；信息同步；内部规划透露 |
| 大棒（合规+分量削减） | 不听话→频繁复盘问询；挑肥拣瘦→新项目不找；业绩差→分量削减；严重违规→清退+拉黑 |

**独特之处**：以业绩论英雄，不管背景/历史；规则运行在"阳光"下（相对透明）；口碑效应：做好就有，躺平就没

**降价对冲策略**：降价不是永远降，后面可能回调；承诺看好生命线（如6000元活下来线）；放大客户获客额度

#### 能力2：分量策略（最不可替代）

把合适的供应商放在合适的地方，让每家都能产出最大价值。

**供应商画像维度**：配合度、业务结构（头部/中部/尾部）、学习能力、业务经验（拉新/企微/大额/创新型）、上人速度、历史表现

**分量打分维度**：
| 维度 | 权重 |
|------|------|
| 业绩 | 30% |
| 质检客诉 | 25% |
| 上人数字 | 15% |
| 配合度 | 15% |
| 战略打分 | 15% |

**网状结构思维**：供应商不是一个个的，而是按业务类型×梯队层级的网状结构。没把合适供应商放在合适地方→潜在产出损失。

#### 能力3：建立对标

通过自建团队建立效能基准，倒逼供应商优化。

- 自建产能+110% vs BPO +45%
- 自建费效1.2（花1.2万 vs 供应商结1万）
- "自建能做到110%增速，你们45%，还想拿更多钱？"
- 成果：单价下降15%，每月降本100万

#### 能力4：无损退出

让供应商心甘情愿自己申请退出。两难选择：闹→被京东拉黑；走→日后好相见。

#### 能力5：京东适应体系

自己总结文档（京东电销团队运转、踩坑、内部配合、管理要求），让新供应商快速适应，减少磨合时间。

### 供应商管理方法论

#### 培养悖论定律

培养供应商的必然路径：小→给资源→做大→向外扩张→有退路→议价/流失。这不是背叛，是模式的必然结果。

| 策略 | 做法 |
|------|------|
| 梯队备份 | 始终培育第二梯队，保持替代选项 |
| BPO联盟 | 把自营培训体系交给中腰部供应商 |
| 新业务优先 | 新业务分配优先给中腰部，不让头部一家独大 |
| 差异化定价 | 头部用市场价，中腰部给成长空间 |

#### 诊断驱动决策

**准入诊断6问**：为什么必须是这家？没有这家会怎样？谁最需要说名字？最小试单是什么？有成功案例吗？3年后还需要吗？

**业绩诊断6问**：核心指标到底怎样？根本原因是什么？谁受影响最大？能改进吗要什么？去现场看过吗？3个月后会怎样？

#### 分层干预

| 信号 | 风险等级 | 干预力度 |
|------|----------|----------|
| 连续2月不达标 | 高风险 | 重点干预 |
| 核心人员离职 | 高风险 | 重点干预 |
| 单月不达标 | 中风险 | 关注跟进 |
| 正常运营 | 低风险 | 日常维护 |

#### 问题诊断三层次

```
现象 → 结构 → 根源
```

找到系统性原因，而非表面问题。区分能力问题 vs 结构问题，避免归因偏差。

示例：新人流失率高（现象）→ 结算低→培训投入不足→新人体验差（结构）→ 名单结构不合理，不是人力能力问题（根源）

**失败复盘铁律**：修补失败2次→立刻停下来反思方向。架构问题不能靠修补解决，必须重构。复盘三步：记录教训→提炼铁律→形成方法论。

### 赛马机制与排名思维

- **赛马机制**：当月有新增供应商时，切量直接交给新增供应商
- **人力效应**：小团队加新人→业绩下滑；大团队加人→可以消化；老员工占比高→排名有优势
- **排名思维**：看相对不看绝对——"88分和89分没有区别，唯一的区别是88分可能是第二名，也可能是倒数第二名"

### 电销AI转型

- **四阶段递进**：能看见（数据采集/SPC）→ 能判断（复盘SOP/考核规则）→ 能行动（整改闭环/双档案）→ 自动化（Agent）
- **6月范围**：只做前三阶段
- **数字人三层架构**：感知层（三源输入+人工校验）→ 判断层（硬/软规则+生长机制）→ 行动层（被动咨询+主动推送）
- **数据来源三合一**：数据表填写 + 群聊抓取 + 人工反馈，非单一自动化管道

## L3 思维偏好

### 沟通风格

- 结构化、直白、数据说话
- 结论在前，理由在后
- 量化不确定性，不说"看情况""也许"
- 引导式设问：先给已知信息→追问细节→给出可能性选项（vs 刘乾坤只问不给）

### 沟通表达层级

| 层级 | 定义 | 示例 | 问题 |
|------|------|------|------|
| Level 1 | 纯感受表达 | "我觉得不太行" | 高度主观，信息密度极低 |
| Level 2 | 结构化定性表达 | "问题出在用户路径过长" | 有分析结构，但不可验证 |
| Level 3 | 定量表达 | "转化率下降了12%" | 可验证，但缺乏解释力 |
| Level 4 | 定性+定量融合 | "转化率下降12%，因路径从5步增至8步" | 有数据、有解释、有因果结构 |

**目标**：Level 4。向上汇报、跨部门协作、供应商谈判用 Level 4；日常闲聊 Level 1-2。

### 表达模板

| 场景 | 模板 |
|------|------|
| 问题汇报 | 问题 + 数据 + 原因 + 建议 |
| 进度同步 | 完成X%（数据）+ 剩余Y因Z原因延期（原因）+ 预计W日完成（时间线） |
| 风险预警 | 风险点 + 影响范围（数据）+ 根因 + 应对方案 |

### 日周月报规范

**周报四段式**：结果（含数据）→ 根因/问题 → 下一步行动 → 时间线

**月报六段式**：整体概况 → 核心指标达成 → 供应商分级分布 → 本月重点事件 → 重点工作进展 → 下月工作计划

**会议纪要五段式**：核心思路 → 关键设计 → 特别说明（引用规矩）→ 待讨论 → 时间节点

**公文三段式**：事实部分（列表/表格）→ 分析判断 → 行动请求

**语气适配**：
| 对象 | 语气 |
|------|------|
| 向上级 | 正式、简洁 |
| 向下级 | 柔和、指导 |
| 平级 | 协商、协作 |

### 工作理念

| 原则 | 含义 |
|------|------|
| 先对齐再动手 | 不清楚就问，不问就做是最大浪费 |
| 做减法 | 功能少一行是一行，不为一次性场景抽象 |
| 输出即态度 | 结论放第一句，细节跟在后，没有废话 |
| 诚实比正确重要 | 错了直接说，不确定直接说，不包装不铺垫 |
| 先做出来 | 完美靠迭代，第一版靠判断力 |

### 工作铁律

| 铁律 | 含义 |

### 设计哲学

**三原则**：克制、锐度、诚实

**配色系统**：Nian 14 色自然色板，不造新色

**反 AI 味**：无阴影、无渐变、无 emoji、无圆角滥用、无居中偏执

**视觉规则**：
- 图表仅限浅色主题（白底），禁用深色背景
- SVG 必须有手工感（Q 贝塞尔曲线），禁几何直线，禁止 emoji
- 数据展示：横向标签优于竖向
- 信息设计：一页一核心、强调色≤10%、50%视觉率
- PPT：克制 > 装饰，颜色压缩到 3 角色（正文灰+点缀橙+语义色仅对比用），层级靠字重不靠形状
- 可视化替代文本符号（流程/关系/对比用 generative-ui-widget，禁止→↓│模拟）

### 技术偏好

- MD 文档用引用块（>），交互中不用代码块
- 拦截性规则优于描述性（规则要能"拦截"而非"建议"）
- 双击操作优先（桌面 .command 文件双击启动，不要命令行）
- 设计系统统一思维（Design Tokens + 同一网格 + 视觉锚点贯穿）

### 品质标准

- **质量标杆**：haglofs-component-showcase.html——"让我欲罢不能的，眼前一亮"
- **教科书级交付物**：3秒法则（核心信息3秒可理解）→ 3步决策链 → 14点质量清单
- **品味定义**：充分信息输入 + 理解方式 + 表达
- **任务验收**：所有任务必须有验收标准和产出物定义
