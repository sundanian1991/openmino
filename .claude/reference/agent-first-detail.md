## 注意力管理

> Token 不是瓶颈，注意力才是。子代理的本质是注意力防火墙——把噪音隔离在子代理上下文中，主上下文只留决策信号。

### 主代理自检红线

主上下文出现以下任一信号 → 立即改用子代理：

| 红线信号 | 说明 |
|----------|------|
| 连续读 2+ 文件还没开始决策 | 探索占了决策位 |
| 一段回复引用 3+ 段原始内容 | 噪音淹没结论 |
| "让我看看…"后面跟一长串搜索 | 该派没派 |
| 主上下文 token 消耗 > 40% 还没产出 | 分工失败 |

### 合同模板

```
任务：[具体任务]
输出要求：
- 只返回结论，不要过程
- 不超过 N 条要点
- 每条不超过 M 字
- 需要支撑数据时附 1-2 个关键数字，不要贴原文
```

### Plan（分析）输出上限

| 子代理类型 | 条目上限 | 每条字数 |
|-----------|---------|---------|
| Plan（分析） | 3 个方案 | 每方案 2-3 句 |

## 子代理类型速查

| 类型 | 用法 |
|------|------|
| **Explore** | 搜文件、搜关键词、探索目录 |
| **Plan** | 设计方案、架构决策、多方案对比 |
| **general-purpose** | 深度研究、复杂多步骤任务 |
| **专家子代理** | 领域任务（判官→供应商、账房→数据、文书→公文） |

**选用**：专家 > general-purpose > Explore 能解决就不用更重的。

## 强制触发条件

以下场景**必须**用 Agent 工具：

| 场景 | 策略 |
|------|------|
| 搜索/查询 | 按关键词/目录拆分，并行 |
| 文件处理 | 按文件/目录/模块拆分，并行 |
| 数据分析 | 按维度/指标/时间段拆分，并行 |
| 多步骤任务 | 每步一个子代理，主代理协调 |
| 跨领域任务 | 每领域一个专家子代理 |
| 代码审查 | 按文件/模块拆分，并行 |
| 文档生成 | 按章节/主题拆分，并行 |

## 轻量 vs 重量判断

**核心问题：会污染主上下文吗？**

| 维度 | 轻量（直做） | 重量（派子代理） |
|------|-------------|-----------------|
| 文件数 | 1 个 | ≥2 个 |
| 搜索范围 | 单目录/单模式 | 跨目录/多模式 |
| 结果量 | ≤5 条 | >5 条或需深度分析 |
| 复杂度 | 一步到位 | 多步推理/研究 |
| 独立上下文 | 不需要 | 需要 |
| 探索噪音 | 不会 | 会 → 必须隔离 |

---

## 决策检查清单

**收到 ≥3 步任务时，先自检**：
1. 该不该派子代理？（≥2 文件读取 / 跨目录搜索 / 需独立上下文 → 必须派）
2. 能拆成并行的吗？（独立子任务 → 并行 ≤5）
3. 选对子代理类型了吗？（专家 > general-purpose > Explore）
4. 合同写清楚输出格式了吗？（条目数 + 字数上限）
5. 主上下文有没有被噪音侵占？（自检红线）

轻量？→ 直做。能拆？→ 每批 ≤5。用对子代理类型？→ 逐项验收。合同约束输出？→ 不带回噪音。
