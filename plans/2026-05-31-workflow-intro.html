Workflow

用 JavaScript 脚本编排多个子代理并行执行。不是 prompt chain——是代码级的确定性控制流，支持 barrier 同步、流水线吞吐、schema 强制输出。

Two Execution Modes

两种执行模式

Workflow 的子代理有两种运行方式。默认隔离，实验性模式可协作。

Default

Subagent Mode

每个子代理在独立 session 中运行，互不通信。主代理编排波次（wave），结果通过返回值传递。上下文效率高，适合大多数场景。
  • Execution独立 Agent() 实例
  • Communication无（隔离）
  • Task List框架管理 TaskCreate
  • Coordination主代理编排波次
  • Context通过输出文件传递
Experimental

Team Mode

通过 TeamCreate 创建团队，子代理变成持久 teammate。可互相发消息（SendMessage）、共享任务列表、自协调。适合复杂协作任务。
  • ExecutionAgent(team_name=...) 持久实例
  • CommunicationSendMessage 点对点
  • Task List共享，teammate 自认领
  • Coordinationteammate 自协调
  • Context共享上下文 + 消息
开启方式： CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 → 出现 TeamCreate 工具 → 自动切换 Team Mode
Core API

四个核心函数

Workflow 脚本用这四个函数定义控制流。JS 脚本，不是 prompt 模板。

agent()
派发一个子代理。可指定模型、类型、输出 schema。返回结构化结果。
schema model isolation
parallel()
多个 thunk 同时执行，全部完成后返回。Barrier 模式——最慢的决定总耗时。
barrier 并发
pipeline()
每个 item 独立流过多个 stage。不等其他 item。默认模式，吞吐量最高。
默认 流水线
phase()
启动新阶段。后续 agent() 自动归入该阶段，在进度面板中分组显示。
进度 分组
Execution Flow
// 定义编排脚本
export const meta = { name: 'review', description: '...' }

phase('Scan')                         ← 启动阶段
const bugs = await agent('find bugs', { schema: BUGS })  ← 子代理 A

phase('Verify')
const verified = await parallel(              ← 并行验证
  bugs.map(b => () => agent(`refute: ${b.title}`, { schema: VERDICT }))
)

return verified.filter(v => v.survives)   ← 返回结果
Quality Patterns

六个质量模式

Workflow 不只是"能并行"，而是内置了验证正确性的模式。每个模式解决一种失败场景。

01

对抗验证

每个结论派 N 个子代理分别反驳。≥多数反驳则杀死。防止"看起来对但其实不对"的发现存活。
3 skeptics → vote → survive
02

多视角验证

一个发现在多个维度可能失败。每个验证者给不同视角（正确性/安全/性能），而不是 N 个相同的反驳者。
3 lenses × N findings
03

评审团

N 个独立尝试从不同角度（MVP优先/风险优先/用户优先），打分后胜者方案融合落选者最佳点。
parallel → score → fuse
04

循环收敛

未知数量的发现，连续 K 轮无新增则停止。不靠固定数量限制，靠实际收敛信号。
while (dry < 2) { find }
05

多模态扫荡

不同搜索角度并行（按实体/按时间/按内容），每个盲区不同，合并去重后覆盖更全。
parallel finders → dedupe
06

完整性评审

最终一个代理问"缺了什么？没跑的模态、没验证的声明、没读的源"，发现变成下一轮工作。
critic → gap → round 2
Use Cases

适用场景

什么任务值得用 Workflow，什么任务 Agent 工具就够了。

场景 模式 为什么用 Workflow
代码审查 parallel bug + 安全 + 性能多维度并行审查，各自独立出结论，最后合并
方案对比 parallel 3 个子代理各研究一个方案，交叉打分，选出最优
批量处理 pipeline 10 个文件各自独立分析，流水线出结果，吞吐量远超串行
对抗验证 parallel agent 一个出结论，多个反驳，存活的才算数
多源搜索 parallel 不同搜索角度并行扫，合并去重
渐进发现 loop 未知数量的目标，循环直到连续无新增
大规模审计 parallel pipeline 多轮 fan-out，每轮独立视角，对抗验证后合成
文档生成 pipeline 研究 → 起草 → 审查 → 定稿，每个文档独立流过流水线
vs. Agent Tool

Workflow vs Agent

Agent 工具派一个子代理。Workflow 编排一群。什么时候该用哪个？

维度 Agent 工具 Workflow
子代理数 1 个 多个，可并行
控制流 无（串行调用） JS 脚本定义
输出验证 文本解析 JSON Schema 工具层验证
文件隔离 共享文件系统 worktree 独立分支
中断恢复 不支持 支持，已缓存结果复用
预算控制 无 token 预算 + 动态检查
适用规模 单任务委派 多任务编排，100+ 子代理
Architecture

执行架构

从用户请求到最终结果，Workflow 经过四个阶段。

Step 1

任务分解

分析复杂度 → 拆分为原子阶段 → 依赖分析 → 决定串行还是并行 → 分配子代理类型（Explore / general-purpose / code-developer）

Step 2

波次执行

独立阶段组成波次并行执行，依赖阶段串行执行。每个 agent() 调用可指定 schema 强制输出格式。worktree 隔离文件系统。

Step 3

验证合成

对抗验证 / 多视角审查 / 评审团打分。子代理结果通过 schema 验证，不合格自动重试。结果去重、合并。

Step 4

输出交付

聚合所有验证后的结果，返回结构化数据。已执行的 agent() 调用缓存结果，中断后可 resume。
