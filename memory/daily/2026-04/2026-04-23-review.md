## 🛠 使用洞察

### 模型使用
- 多模型混用：主供应商项目使用 qwen3.6-plus，夜间 cron 使用 glm-4.7
- 多轮迭代场景下建议保持同一模型，避免风格漂移

### Token 效率
- 供应商项目 session 仅 7 条记录（3.4MB）→ 子代理输出大量写入文件而非返回主上下文，效率较好
- C2材料 session 10 条记录（125KB）→ 三次迭代符合"需求-调整-确认"最小闭环

### 工具使用
- context-mode 工具（ctx_batch_execute、ctx_execute_file、ctx_search）高频使用 → 适合文件处理
- 子代理并行化运用合理 → 供应商项目拆成探索+分析+写作多个子代理
- **潜在浪费**：配置清理 session 中探索过多非必要信息（GSD workflow、MCP 服务器等），偏离核心目标

### 工作模式建议
- 配置类任务容易"越查越多"，建议事先限定范围（"只看 hooks 和 context-mode"）
- 月会材料迭代三次才到位 → 下次可先用一句话确认受众和目的，减少往返

## 💻 代码/技术建议

### hooks 配置管理
- 当前 hooks 配置散落在 settings.json 和独立文件中，建议收敛到单一配置源
- 删除的钩子引用应及时清理，避免残留配置指向不存在的文件

### session JSONL 存储
- 大 session（3.4MB）仅 7 条记录 → 说明子代理输出直接写入文件，不在 JSONL 中记录详情
- 这导致无法从 JSONL 分析子代理的执行细节和 token 消耗 → 如需精细分析，建议记录子代理的 summary 到主 session
