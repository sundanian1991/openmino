# CLAUDE.md — 数据可视化知识库（卡帕西式三层架构）

> 这是知识库的 schema 层。你（AI agent）进入这个目录时，按本文件的规则读、写、巡检。
> 三层结构：`raw/`(原文) → `distill/`(提炼) → `notes/`(健康判断)。本文件 = 第三层 schema。
> 与 `../wiki/`（供应商管理）平级独立，编码体系互不干扰。

---

## 这是什么

数据可视化领域的卡帕西式 wiki。收录：图表设计原则、叙事弧线、模式库、图表百科、麦肯锡复刻指南、HTML 样板。

**一句话定位**：当你（AI 或人）要做一张图、讲一个数据故事、选图表类型、定视觉规范时，先来这里找依据。

**不收录**：品牌设计系统（→ 以后建 wiki-design）、供应商业务（→ ../wiki/）、纯前端组件实现（→ reference/）。

---

## 三层结构（不可混淆职责）

| 层 | 目录 | 职责 | 可变性 | 你能做什么 |
|----|------|------|--------|-----------|
| 原文层 | `raw/` | 完整原文归档，source of truth | **只读，永不修改** | 只能读、只能往里加新文件，不能改已有文件内容 |
| 提炼层 | `distill/` | 结构化知识，AI 直接调用 | 可重新编译 | 读、写、改、重写——但每段必须标 `[来源]` |
| 健康判断层 | `notes/` | 冲突、待验证、已过时 | append 为主 | 读、追加、标记解决；不能删历史，只能移到「已过时」 |

> **铁律**：`distill/` 的每一条关键规则（阈值、比例、选型规则、配色值）必须能追溯到 `raw/` 里的某个文件。追溯不上的，写进 `notes/待验证假设.md`，不要硬塞进 distill。

## 编码体系

| 前缀 | 含义 | 典型内容 | 示例 |
|------|------|---------|------|
| `VZ-2026-xxx` | 可视化原则/规范（Vision rule） | 原则、构图、配色、叙事 | VZ-2026-001 可视化原则总纲 |
| `VZ-2026-1xx` | 模式/类型（100段） | 图表模式库、类型选型 | VZ-2026-100 图表模式库 |
| `VZ-2026-2xx` | 叙事/叙事弧（200段） | 叙事结构、视觉节拍 | VZ-2026-200 叙事弧线模板 |
| `VZ-2026-3xx` | 复刻/样板（300段） | 麦肯锡复刻、HTML样板 | VZ-2026-300 麦肯锡图表复刻指南 |
| `VZ-2026-9xx` | 百科/参考（900段） | 图表百科、速查 | VZ-2026-900 60种图表全览 |

> raw/ 命名：`VZ-2026-xxx_主题.md`。distill/ 分册命名：`NN_主题.md`（NN=01-09）。

## 三个操作（每次只做其中一类）

### Op 1 — Ingest 摄取（写入）

当用户给你一份新文档，或 `raw/` 里出现新文件：

1. **归档原文**：转成 markdown 原样放进 `raw/`，命名 `VZ-2026-xxx_主题.md`。**不改一个字。** HTML 样板保留 `.html` 原文件到 `raw/originals/`，转出一份 `.md` 描述版进 raw。
2. **读全文**：识别里面的实体（图表类型、配色值）、概念（原则、模式）、规则（阈值、比例）。
3. **写/更新 wiki 页**：新概念→新增段落；已有概念更新→**改旧段落，不新建重复页**（合并优于追加）。每个新/改段落立刻补 `[来源：raw/文件名]`。
4. **更新交叉链接**：相关页之间用 `[文本](文件名.md)` 互相链接。
5. **更新 `distill/00_索引与导航.md`**：新页登记到索引。
6. **记日志**：`notes/ingest日志.md` 追加一行。

### Op 2 — Query 查询（读取）

1. **先读 `distill/00_索引与导航.md`** 定位答案在哪个分册。
2. **读对应分册**，必要时顺着 `[来源]` 回 `raw/` 看原文。
3. **回答时引用来源**：每条结论标注来自哪个 distill 段落 + 哪个 raw 文件。
4. **发现知识缺口**：记进 `notes/待验证假设.md`，回答末尾告诉用户。

### Op 3 — Lint 健康检查（巡检）

1. **跑脚本**：`cd wiki-viz/scripts && python3 wiki_health_check.py --wiki-dir ../`
2. **按报告修**：来源覆盖率<95%补标注；缺失raw补文件或修引用；冲突逐个核对。
3. **语义深查**：矛盾（两分册对同一规则说法不同）、孤儿页、过期声明、缺交叉引用。

## 与 ../wiki/（供应商管理）的协作

两座 wiki **独立 schema、独立编码、独立巡检**，但内容上有交叉：

| 交叉点 | 处理方式 |
|--------|---------|
| 供应商可视化规范（本库 06）引用供应商业务规则 | 链接到 `../../wiki/distill/0X_xxx.md`，不复制内容 |
| 供应商 wiki 需要画图 | 反向链接到本库相关分册 |
| 同一个 HTML 样板被两边引用 | 原文只存本库 raw/，供应商 wiki 用链接引用 |

> 原则：**原文归一处，链接连两边**。避免同一文件存两份导致版本漂移。

## 自动化（scripts/）

| 脚本 | 作用 | 何时跑 |
|------|------|--------|
| `scripts/wiki_health_check.py` | 健康度巡检 | 每周或重大更新后 |

```bash
cd wiki-viz/scripts && python3 wiki_health_check.py --wiki-dir ../ --output health_report.json
```

## 反模式（绝对不做）

1. **改 raw/**：原文是 source of truth，永远不改。
2. **distill 不标来源**：没来源的规则 = 不可信。宁可写进待验证，也别硬塞。
3. **只追加不合并**：同一概念有新版，改旧段落，不要新建重复页。
4. **跳过 index 更新**：新页不登记到 `00_索引与导航.md`，等于不存在。
5. **删 notes 历史**：冲突解决了移到「已过时」，不要直接删。
6. **把供应商业务内容搬进本库**：本库只收可视化，业务规则归 ../wiki/。

## 快速入门（给第一次进来的 agent）

1. 读本文件（你正在读）。
2. 读 `distill/00_索引与导航.md` —— 知道有什么。
3. 根据任务选 Op：给新文档 → Op1；问问题 → Op2；维护 → Op3。
4. 动手前确认：这一步会动 `raw/` 吗？会的话，停。
