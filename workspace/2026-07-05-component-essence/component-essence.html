Haglöfs 范式 · 组件认知

组件不是
UI 部件

组件是信息本质到视觉形态的翻译器。选组件不是选"好看的图表"，是识别"这块信息本质上是什么"。

认知 01 · 本质

每个组件翻译一种信息本质。

在 Haglöfs 范式里，55 个组件不是 55 个"长得不同的卡片"。它们是 55 种信息本质到视觉形态的映射——每种信息本质（二元对立 / 成组数字 / 单句宣言 / 时间序列 / 层级构成）都有对应的组件来翻译它。

选组件的第一步不是翻组件库找"哪个好看"，是问：这块信息本质上是什么？是对比？是趋势？是构成？是流程？是声明？——回答了这个问题，组件自己就浮出来了。

— 数据组件
信息层
展示事实、数据、规格。是页面的"骨"——承载论证的证据。
16 个 · Tension Grid / Stat Grid / Elevation / Seam / Ring / Layer / Pipeline ...
— 品牌叙事组件
节奏断点
制造呼吸、停顿、收束。是页面的"气口"——在信息密集段之间插留白和升华。
12 个 · Manifesto / Statement Quote / Principle Cards / Footer ...
— editorial 组件
阅读体例
承载长篇阅读的结构和体验。是页面的"脉络"——阅读宽度、进度、首字装饰、引文抽取。
11 个 · Article Header / Drop Cap / Pull Quote / Reading Progress ...
◆ 分类原理

三类不是按"长得像"分，是按在叙事里干什么分——数据做"信息层"，品牌做"节奏断点"，editorial 做"阅读体例"。三类在同一页面混用：数据组件铺信息，品牌组件做呼吸谷和收束，editorial 组件管阅读宽度。它们是叙事的三个声部，不是三种卡片样式。

认知 02 · 设计宪法

4 条宪法不是 4 条规则，是同一个原理。

组件设计宪法的本质是减法优先——每加一个元素问"删了它损失多大"。4 条宪法是这个原理从 4 个角度的切面，不是 4 件独立的事。它们回答的是同一个问题的 4 种问法：这个元素该不该存在。

01

信息本质驱动形态

量化事实 → 数字最大、标签最小。对比多项 → 结构最清晰、边框最少。声明主张 → 文字最少、留白最大。违反 = "为好看加东西"。

先问"这是什么"，再问"长什么样"。

02

每个元素挣到自己的像素

每加一个元素问：删了它，信息传达损失多少？损失 ≤5% 就删。表格删外框、斑马纹、粗边框 → 只留发丝线。产品卡删价格、评分 → 只留图+名+一句话。

留白也是产出。不干活的元素不配占像素。

03

字体角色是信息性质的翻译器

Playfair = 郑重/温度（标题、主张、权威数字）。Mono = 精确/刻度（数据、标签、编号）。Inter = 功能/清晰（正文、描述）。表格数字 Mono、列头 Mono 全大写、总计行 Playfair——每种字承担一种认知角色。

字体不是风格选择，是信息的性质分类。

04

层次靠留白 + 色调 + 1px 线

section 之间 96-120px 留白，不加分隔条。表格行之间 1px 发丝线，不用斑马纹。卡片之间 cream 底差异 + 间距，不用阴影。层次不是"加东西"造的，是"减东西"露出来的。

哑光克制的本质：深度来自克制，不来自装饰。

认知 03 · editorial 即视觉化

大字数字 + 发丝线 = 高强度编码。

"70 vs 20"用 Stat Grid 两个大数字并列比用饼图更有力——大字 Playfair 数字本身是高强度的视觉编码，饼图反而稀释了对比。在选 chart 类组件之前，先问 editorial 组件能不能完成同样的视觉化任务。

— 饼图

70% vs 30% 分工

饼图把对比分散到圆形面积里，读者要估算扇形角度才能感知比例差距。在浅底 offwhite 上对比更弱。

— Stat Grid
70%
20%

70% vs 20% 分工

大字 Playfair 数字并列 + 1px 发丝线切分。对比是 0 秒发生的——视觉系统直接处理"大数字 vs 小数字"，不需要估算。

◆ 编码强度原理

在 Haglöfs 的浅底克制语境里，editorial 组件（大字数字、网格并列、发丝线切分）经常比 chart 类组件（折线、饼图、雷达图）有更高的编码强度——因为大字 + 留白对视觉系统的刺激比细线条 + 色块面积更直接。这是范式 DNA 决定的：Haglöfs 是 editorial 不是 dashboard。

认知 04 · radius 三级

圆角不是审美选择，是角色硬度。

Haglöfs 范式的 radius 三级（0 / 4 / 8px）不是"圆多少好看"的审美问题，是组件在结构里扮演什么角色的视觉表达——结构性越强越硬（锐角），交互性越强越软（圆角）。读者感知到的不是"这个圆角几像素"，是"这个组件在结构里有多硬"。

— 结构件
0px
Tension Grid / Data Table / Pipeline / Grid Matrix。结构靠对齐和发丝线，圆角破坏秩序感。
— 内容卡
4px
Swatch / Product Card / Code Block / Ring / Stat Grid。极轻微圆角暗示"这是一个物体"，但不柔软。
— 交互控件
8px
Button / Input / Toggle。交互控件可稍软，暗示"可操作"。pill 50% 仅限圆点。
◆ 为什么不用大圆角

大圆角（>8px）是 Material / 电商的语言——它暗示"柔软、友好、可触摸"。Haglöfs 的哑光克制恰好相反：结构要硬朗、editorial 要锐利、克制要有边界感。圆角越大，品牌温度越向 SaaS 滑。radius 是品牌 DNA 的视觉度量，不是细节装饰。

认知 05 · 深底反相

同一组件，两种环境。

浅底 DNA 的组件能适应深底环境——不靠每个组件各写一套 inverse 规则，靠 .section--dark 父级作用域统一继承。组件只需声明"我是浅底通用类"，放到深段时自动反相：主文字 offwhite、次要文字 50% 白、强调色 forest→moss（深底提亮）。

这背后的认知：组件的环境适应性 > 组件的独立样式。一个组件不应该"只会在浅底工作"——它应该像一件好的户外装备，在不同光线下都保持识别度。

— 浅底 offwhite
70%
主文字 charcoal · 强调 forest-soft · 底 offwhite
— 深底 charcoal
70%
主文字 offwhite · 强调 moss（提亮）· 底 charcoal
◆ 三类深底角色

原生深底（Tension Grid / Manifesto / Grid Matrix）：天然为深底设计，骨架即深底。浅底通用（Data Table / Ring / Swatch / Stat Grid）：放深段时继承 `.section--dark` 统一反相。仅浅底（Drop Cap / Pull Quote / Testimonial）：深底会破坏其 DNA——Drop Cap 的 forest-soft 浮字在深底显脏。

认知 06 · 选型

选组件不是查清单，是翻译。

组件选型决策树的本质不是"信息类型 → 组件名"的查找表，是信息本质 → 视觉形态的翻译过程。决策树只是一个加速器——当你真正理解了"这块信息要求读者做什么运算"，组件自己就浮现了。

真正会选组件的人，不记 55 个组件名。他记的是每种信息本质该用什么编码强度——对比用并列大数字，趋势用面积/箭头，分布用条形阵列，层级用堆叠条，流程用步骤编号。组件名只是这些编码方式的命名。

◆ 选型的 3 个问题

每次选组件，问这三个问题。

1. 这块信息本质上是什么？（对比 / 趋势 / 构成 / 流程 / 声明 / 阅读流）
2. 读者需要在脑子里运算吗？（要 → 画出来；不要 → 用文字/大字）
3. editorial 组件能完成吗？（能 → 不用 chart；不能 → 才用 chart 类）

◆ 组件不是终点

组件是"信息→视觉"翻译的已验证方案——55 个组件是 55 种已经被证明有效的翻译。但它们不是唯一的翻译方式。当你理解了翻译的原理（信息本质 → 编码强度 → 视觉形态），你可以在没有现成组件时，自己推导出合适的视觉形态——这才是组件系统的真正价值：不是让你记住 55 个答案，是让你学会怎么到达答案。

"组件不是 UI 部件，是信息本质的翻译器。选组件不是选好看的图表，是识别这块信息本质上是什么。"
