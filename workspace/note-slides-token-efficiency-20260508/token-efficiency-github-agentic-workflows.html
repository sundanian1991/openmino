发现一
07 / 20

最常见的低效：未使用的 MCP 工具注册，每轮对话携带 10-15 KB 无用 schema

问题机理

LLM API 是无状态的，agent 运行时每次请求都附带 MCP 工具的函数名和 JSON schema。GitHub MCP server 有 40 个工具，每个工具加上 schema 总共 10-15 KB。如果 agent 只用了其中 2 个，剩下的 38 个就是纯浪费

优化方式

Optimizer 交叉比对工具清单和实际调用记录，识别从未用到的工具并建议从配置中移除。在 smoke-test workflow 中，这一改动每轮减少 8-12 KB 上下文，节省数千 token，行为完全不变

来源：Eliminating unused MCP tools
07 / 20
发现二
08 / 20

更大的结构性机会：把数据获取从 LLM 推理循环中移出去

MCP 的代价

一次 MCP 调用=一次完整 LLM 往返：决定调用、构造参数、接收响应，每一步都消耗 token。gh pr diff 则是一次确定性的 HTTP 请求

策略一：预下载

agent 启动前，setup 步骤用 gh 命令把 PR diff、文件列表等数据写到工作区文件，agent 直接读取，零 LLM 开销

策略二：运行时代理

agent 在运行中需要获取什么时，通过透明 HTTP proxy 调用 gh CLI 获取结构化数据，同样不暴露 token 给 agent

来源：Replacing GitHub MCP with GitHub CLI

测量问题
09 / 20

Not all tokens are created equal

ET = m × (I + 0.1C + 4.0O)

m = 模型成本系数（Haiku 0.25x, Sonnet 1.0x, Opus 5.0x）

I = 新处理输入 tokens, C = 缓存读取 tokens (0.1x), O = 输出 tokens (4.0x)

输出 tokens 权重重是因为它是最贵的 token 类型；缓存读取权重低是因为缓存价格只有新鲜输入的几分之一

来源：Measuring efficiency gains

测量难点
10 / 20

三个混杂因子让"优化是否有效"的判定不直接

模型差异

相同 token 数量下 Haiku 成本只有 Sonnet 的 1/4。原始 token 数不变不代表成本不变。ET 公式通过模型系数解决这个问题

负载波动

同一个 workflow 这次处理 5 行 fix，下次处理 200 行 PR。token 数下降可能是运气（任务变轻了），不是效率提升。需同时追踪 LLM 调用轮次

质量不可知

没有 ground-truth correctness。过程信号（输出 token/call、轮次/run）可近似质量，但无法直接观测到优化是否伤害了输出质量

模式三
15 / 20

A single misconfigured rule can cause runaway loops

问题

Daily Syntax Error Quality 是优化前 token 消耗最高的 workflow。根因：workflow 把测试文件拷贝到 /tmp/ 后调用 gh aw compile *，但 sandbox 的 bash allowlist 只允许相对路径 glob 模式

后果

每次编译尝试都被拒绝。agent 无法使用需要的工具，掉入 64 轮 fallback 循环，手动读源代码去猜测 compiler 会报什么

修复

一行修复：扩大允许的 bash patterns 后循环消失。这是一行配置错的代价：64 轮无意义的 LLM 调用。

来源：Take aways · 模式3

优化后 ET 上升 5% 不等于优化失败，工作负载偏移可以掩盖效率提升

数据

Contribution Check workflow 优化前：41% 小型 PR（ET <100K），39% 大型 PR（ET >300K）。优化后遭遇开发活动高峰：9% 小型 PR，65% 大型 PR。

分析

输出 tokens（ET 公式中权重 4x）上升 14%，因为 agent 需要审查更大的 diff。每轮效率实际改善了，但负载变化在聚合数据中掩盖了这一改善。始终需要结合 LLM 轮次来判断。

Episode 分析

同一个 workflow 内部，哪些阶段值得做、哪些阶段是浪费

一次 workflow 运行不是平坦的 API 序列，而是一串 episode：上下文收集、读取 artifact、失败重试、合成最终答案。能看清 episode 就能回答：哪段 episode 真正导致了高成本？哪些应该变成确定性预步骤？

Portfolio 分析

跨 workflow 的重复读取和重叠判断

一个 repo 运行的不只是单个 workflow，而是一整支自动化舰队。它们经常触发在相同事件上、读取相同的 diff 和日志、产出相邻的判断。下一个问题是：哪些工作流在重复读数据？哪些应该合并？

核心总结
19 / 20
01

GitHub 自用数百个 agentic workflow 做 CI 和维护。Agentic workflow 自动调度触发，成本低调积累不易被察觉。优化它的优势在于行为完全由 YAML 定义，可重复、可衡量。

02

API proxy（原本用于安全隔离）意外成为 unified logging 的切入点。每个 workflow 产出 token-usage.jsonl，含 input/output/cache tokens、模型和时间戳。有了数据才能优化。

03

两个 agentic workflow 构成自愈循环：Auditor 聚合报告标记异常，Optimizer 分析源码创建 Issue 给出具体优化建议。它们自己的 token 也出现在每日报告中，形成良性循环。

04

两个最常见的低效模式：① 未使用的 MCP 工具，40 个工具的 schema 每轮浪费 10-15 KB 上下文；② 数据获取经过 LLM，用 gh CLI 替代 MCP 把确定性的 HTTP 请求移出推理循环。一个 search_repositories 工具单轮被调 342 次。

核心总结
20 / 20
05

测量效率提升有三个混杂因素：模型差异（Haiku 比 Sonnet 便宜 4x）、工作负载波动（5 行 fix vs 200 行 PR）、质量不可知（没有 ground-truth correctness）。Effective Tokens 公式用模型系数和 token 类型权重归一化成本。

06

初步结果：12 个生产 workflow 中 9 个接受优化建议。Auto-Triage Issues −62%（109 个修复后运行），Security Guard −43%，Smoke Claude −59%。运行频率和单次节省同样重要，Auto-Triage 日均 6.8 次，累计节省 780 万 ET。

07

模式总结：① 许多 agent 轮次是确定性的数据获取，"最便宜的 LLM 调用是不需要的那一次"；② 未使用的工具成本很高，SR 工具单轮调用 342 次全是浪费；③ 一行配置错误可以导致 64 轮回退循环。

08

下一步方向：从 workflow 级优化升级到系统级，episode 分析（区分必要阶段和浪费阶段）和 portfolio 分析（跨 workflow 去重和缓存共享）。但核心建议一样：加 API proxy、开日志、让数据说话。
