# viz-design 边界场景测试报告

> 测试日期：2026-05-04
> 测试范围：5 个边界场景 × (intent.md + spec.md + JSON)
> 测试方法：人工应用 viz-design 工作流，产出每个场景的预期正确输出

## 一、测试总览

| 场景 | 输入特征 | 核心测试点 | 预期模式 | 关键规则 |
|------|----------|-----------|---------|---------|
| 1 极小数据集 | 2 个数值 | 是否识别数据过少 | 69-KPI 大字报 | 规则 1：≤3 个数字不画图 |
| 2 纯概念文字 | 无数据，管理概念 | 是否走概念→模式匹配 | 39-齿轮 | concept-pattern-map 联动关系 |
| 3 超宽数据 | 20 列 × 5 行 | 是否识别维度过多需分图 | 多图（折线+雷达+柱状） | 规则 2：一图一核心信息 |
| 4 矛盾数据 | 自报 vs 实际差距大 | 是否识别矛盾并给出洞察 | 26-并列比较 | 标题规则 5：结论性标题 |
| 5 单行数据 | 1 个数值 | 最简输入是否走轻量模式 | 69-单值大字报 | 规则 1 + 轻量模式 |

## 二、逐场景测试详情

### 场景 1：极小数据集

**输入**：`指标,今年,去年 / 营收,1200 万,800 万`

| 检查项 | 预期 | 说明 |
|--------|------|------|
| 模式选择 | 69-成套工具 → KPI 大字报 | 规则 1 强制触发，不画传统图表 |
| intent.md | 识别对比关系 | 明确标注"≤3 个数字，不画图"的决策 |
| spec.md | 大字报规格 | 字号分级（主 42px/基线 26px/增量 18px） |
| JSON v2 | 有效 | mapping.x = null（无坐标轴），3 条 rows |
| 标题 | 结论性 | "营收同比增长 50%，达 1200 万" |

**风险点**：如果 AI 忽略了规则 1，可能错误选择柱状图或饼图。

### 场景 2：纯概念文字

**输入**：`"帮我画一张图表达供应商管理的三问原则"`

| 检查项 | 预期 | 说明 |
|--------|------|------|
| 模式选择 | 39-齿轮 | concept-pattern-map 联动关系匹配 |
| intent.md | 走概念匹配流程 | 不尝试从文字中提取数据 |
| spec.md | 概念图规格 | data 部分标注"概念驱动，无结构化数据" |
| JSON v2 | 有效 | geom_gear + geom_arrow 层 |
| 标题 | 结论性 | 包含"三问"关键词 |

**风险点**：如果 AI 试图将"三问"解析为 3 行数据表格，则概念识别失败。概念图应直接跳过数据解析步骤。

### 场景 3：超宽数据

**输入**：20 列 × 5 行（月度 + 6 项指标 + 分类）

| 检查项 | 预期 | 说明 |
|--------|------|------|
| 模式选择 | 多图拆分（≥3 张图） | 规则 2：一图一核心信息，20 列不可塞入一图 |
| intent.md | 多图拆分表 | 明确每张图的角色和覆盖的数据子集 |
| spec.md | 3 张图的规格 | 折线图（趋势）+ 雷达图（画像）+ 柱状图（排名） |
| JSON v2 | 仅包含 1 张图 | spec.md 可以包含多图描述，JSON 只交付 1 张 |
| 数据降维 | 删除峰值比等衍生指标 | 不是所有字段都需要可视化 |
| 反模式拦截 | 雷达图 ≤8 维 | anti-patterns B2 检查 |

**风险点**：
- AI 可能试图画一张 20 维的雷达图（违反 B2：>8 维度禁止）
- AI 可能画出"意面图"（5 条折线全部彩色，违反 A3 高亮≤10%）
- AI 可能不做数据降维，把所有字段塞进图表

### 场景 4：矛盾数据

**输入**：3 家供应商自报 vs 实际（差距 21-27pp）

| 检查项 | 预期 | 说明 |
|--------|------|------|
| 模式选择 | 26-并列比较 | 两组数据逐项对比 |
| intent.md | 识别矛盾/异常 | 明确标注差距 21-27pp 的异常性质 |
| spec.md | 对比图规格 | 差距标注、按差距排序 |
| JSON v2 | 有效 | geom_bar + geom_label 层 |
| Big Idea | 指出虚报 | 标题应为"供应商虚报 20-27pp"，不是"数据对比" |
| 排序 | 按差距降序 | 不是按字母或自报值排序 |

**风险点**：
- AI 可能只做数据展示，不给出"虚报"的定性结论（洞察深度不足）
- AI 可能按供应商名称字母排序，而不是按差距排序（排序无叙事）
- 标题可能写成描述性"自报 vs 实际达成率对比"（违反规则 5）

### 场景 5：单行数据

**输入**：`指标,值 / 年度总营收,1.2 亿`

| 检查项 | 预期 | 说明 |
|--------|------|------|
| 模式选择 | 69-成套工具 → 单值大字报 | 规则 1 最强触发场景 |
| 轻量模式 | 走快速路径 | Phase 1+2 合并，不写 intent.md/storyboard |
| spec.md | 极简规格 | 1 个数字 + 标签 |
| JSON v2 | 有效 | mapping.x = null, 1 个 geom_text 层 |
| 不画图表 | 不选择任何传统图表 | 柱状图/饼图/折线图均错误 |

**风险点**：
- AI 可能画一个"只有一个柱子"的柱状图（荒谬但常见 AI 错误）
- AI 可能画一个饼图（只有一个扇形=整个圆）
- AI 可能过度设计：加装饰、加图标、加不必要的视觉元素

## 四、关键发现

### 4.1 技能设计对边界场景的覆盖

| 边界类型 | 规则支持情况 | 缺口 |
|----------|-------------|------|
| 数据过少 | 规则 1 明确 | 无缺口 |
| 纯概念输入 | concept-pattern-map 覆盖 | 但 spec 模板假设数据驱动，概念图的 data 字段需要特殊处理 |
| 数据过多 | 规则 2 明确 | 无明确"列数>阈值"的量化触发条件 |
| 矛盾数据 | 无专门规则 | **缺口**：决策规则中没有"矛盾/异常数据"的专门处理逻辑 |
| 单值输入 | 规则 1 覆盖 | 无缺口 |

### 4.2 spec 模板对概念图的不适配

spec-template.md 的"数据组织"部分假设所有输入都有字段、排序、聚合规则。概念图（场景 2）的 data 应改为"概念节点 + 关系"，而非结构化数据行。

**建议**：spec-template.md 增加概念图的 data 模板变体。

### 4.3 矛盾数据缺少检测规则

场景 4 的矛盾数据（自报 vs 实际差距 27pp）是最有商业价值的场景——它要求 AI 不仅展示数据，还要发现异常并定性。但 decision-rules.md 中没有任何关于"数据矛盾检测"的规则。

**建议**：新增规则"数据一致性检查——当两组数据差距超过合理阈值（如>15pp），应标注为异常并定性"。

## 六、结论

viz-design 技能在边界场景下的规则覆盖总体良好，但有 2 个缺口需要关注：

1. **矛盾数据检测**：decision-rules.md 缺少异常数据识别规则，AI 可能只展示不分析
2. **概念图 spec 适配**：spec-template.md 的 data 部分对概念驱动场景不够友好

场景 1、3、5 的规则支持完善（规则 1 + 规则 2），场景 2 的 concept-pattern-map 覆盖良好，场景 4 是主要改进点。
