## 一、技能定位与核心理念

**名称**：思维透镜 (Thinking Lenses)
**一句话**：25个跨职业思考工具的"思维透镜"系统，帮助用户跳出单一视角，从25种职业思维框架中重新审视问题。

**理论基础**：Scott H. Young 的博文 "Twenty-Five Useful Thinking Tools" (2018-12-04)
原文：https://www.scotthyoung.com/blog/2018/12/04/25-thinking-tools/

**核心引言**：
> "More mental models means you have more ways to solve more problems." — Charlie Munger

**核心洞见**：人的最大思维陷阱，是误以为自己看到的就是全部。25个思维透镜 = 25把看问题的钥匙。

### 浅层做法 ❌ vs 深层做法 ✅

**浅层**：三个问题换了皮，本质都是"你觉得问题在哪？"
- Doctor: 诊断是什么？
- Engineer: 能建模吗？
- Hacker: 底层在运转什么？

**深层**：每个职业加载一套完整的**认知协议**——不是一句话问题，是 3-6 步微型思维算法。
- 每步强制输出中间推理
- 使用该职业的专业语言约束
- 如果两个透镜的核心判断用了相同思考框架 → 判定切换失败，重新加载

## 二、动态思考旅程（内部运作流程）

每次调用遵循统一的5阶段旅程。**前3个阶段是强制门——不跳过。**

🛑 **CHECKPOINT**：Situation → Check-In → Input Gate 三个阶段必须按顺序完成。

### 1. Situation（锚定问题）
- 接收用户输入 → 隐性识别问题类型/情绪/约束
- 如果输入含糊 → 轻量Check-In（最多1-2个追问）
- 如果带具体内容 → 进入内容审查模式
- 🛑 1a. 结构化摘要（CHECKPOINT）：把用户输入拆成5个要素——背景、目标、约束、痛点、模糊点
  - 每个要素用一句话概括，包含用户原话中的具体数字/人名/资源限制
  - 缺失要素标注"用户未明确"而不是自己补

### 2. Check-In（用户适配）
- 系统内部完成：问题分类 + 情绪识别 + 约束判断
- 不增加额外交互轮次

### 3. 🛑 Input Gate（CHECKPOINT — 选透镜前必须完成）
- 3a. 事实/观点校验（Journalist精神）：区分事实/推测/情绪，推测标"待核实"
- 3b. 内部判定（一次性完成）：
  - 认知偏差扫描：对照8种偏差信号
  - 场景匹配：扫9大预设场景
  - 跨类型判定：主类型 + 次类型
- 3c. 查表交叉（强制）：对每个判定类型查映射表获取推荐透镜 → 交叉筛选 → 初选集合（4-7个候选）

### 4. Tool Thinking（视角调用）
- 基于 3c 的初选集合，结合跨类别覆盖检查（≥3个认知原理类别）确定最终透镜组合（3-5个）
- 🔴 4a. 回环验证（CHECKPOINT）：选完透镜后回到用户"痛点"要素，问"这个组合真的对准了用户最痛的点吗？"
- 按透镜输出协议逐个输出分析

### 5. Synthesis（综合洞察）
- 5a. 冲突焊接（CHECKPOINT）：列出所有透镜核心判断，识别矛盾与汇聚
- 5b. 涌现检查：冲突焊接后是否涌现出单个透镜看不到的新判断？
- 5c. 结尾设计（强制）：留一个"关键不确定性"——开放式问题抛给用户

## 三、透镜输出协议（每个透镜必须遵守）

### 铁律 1：加载完整认知协议
每个透镜必须加载完整认知协议（3-6步算法），按步骤逐一执行，每步输出中间推理。

**协议执行要求**：
- 使用该职业的语言约束词汇（如 Doctor 用"主诉""鉴别诊断"）
- 每步必须输出中间推理，不能跳过
- 完成全部步骤后，输出一句话核心判断
- 如果两个透镜的核心判断使用了相同思考框架 → 判定切换失败

**输出原则**：
- 透镜名称格式：英文在前，中文括号跟后（如 "Engineer（工程师）"）
- 不显示步骤标签
- 用"核心问题"统领每个透镜
- 让分析自然流动，像讲故事
- 关键问题锚定用户输入中的具体事实
- 行动优先级用时间层（立即→本周→决定→准备）
- 必须有"一句话总结"

### 铁律 2：永不重复上一轮的角度
输出当前透镜前，检查上一个透镜的核心判断：
- 如果当前分析结构和上一个同构（只是换了职业标签），立即停止
- 换一个异质性框架重新切入

### 冲突矩阵（主动制造张力）

| 冲突对 | 张力 | 适用场景 |
|--------|------|---------|
| Artist × Accountant | 创造力 vs 数字纪律 | 制度设计、创新评估 |
| Soldier × Entrepreneur | 流程 vs 试错 | 体系搭建、风险决策 |
| Journalist × Novelist | 事实 vs 叙事 | 向上汇报、晋升述职 |
| Engineer × Hacker | 建模 vs 拆解 | 问题诊断、系统设计 |
| Politician × Mathematician | 可信 vs 可证 | 跨部门推动、方案说服 |
| Doctor × Philosopher | 诊断 vs 追问后果 | 业绩下滑根因分析 |

## 四、五种使用模式

### 模式A：精准诊断（默认）
**触发**：用户描述了一个具体问题、困惑或决策
**流程**：分析问题性质 → 推荐3-5个透镜 → 盲区覆盖裁决 → 逐个输出分析 → 请用户选择深入

**盲区覆盖裁决（停止信号）**：
- 🔴 核心盲区（必看）：Top 2 透镜，覆盖约80%高频盲区
- 🟡 补充盲区（建议看）：第3个透镜，特定场景遗漏风险
- 🟢 深度盲区（可选）：第4-5个透镜，重大决策时展开

**模式A-深挖版（二次补充协议）**：第一轮看机制层（系统/激励/流程），第二轮补人和体验层（文化/叙事/心理/感知）。

**模式A-快速版（一句话洞察）**：只推 Top 2 透镜，每个一句话匹配+一个核心问题，输出<200字。

| 维度 | 快速版 | 完整版 |
|------|--------|--------|
| 透镜数量 | 2个 | 3-5个 |
| 每个透镜输出 | 一句话+一个问题 | 为什么匹配+2个引导问题 |
| 输出长度 | <200字 | 不限 |
| 使用场景 | 快速咨询、碎片时间 | 深度分析、重要决策 |

### 模式B：全视角扫描
**触发**：用户说"帮我全面看看这个问题"或"有没有我漏掉的角度"
**流程**：25个透镜分高/中/低相关三组 → 每组选2-3个最有洞察力的提问 → 标记"高风险盲区"

### 模式C：强制跨界组合
**触发**：用户陷入深度思维定势，或明确说"来个离谱的"
**流程**：识别当前思维习惯 → 推荐最不相关/最反直觉的2个透镜组合 → 强制制造认知冲突

### 模式D：单透镜深入
**触发**：用户主动说"我想用XX的视角看看"
**流程**：确认透镜 → 加载完整解释 → 提出3-4个层层递进的引导问题 → 转化为可执行结论

### 模式E：内容审查
**触发**：用户说"帮我审一下这段内容/方案/邮件/PPT"
**自动选取沟通类透镜组合**：
- Journalist（记者）：事实核查
- Novelist（小说家）：叙事结构
- Designer（设计师）：信息设计
- Politician（政治家）：受众感知
- Critic（评论家）：深度解读
逐条标注严重度（🔴 必须改 / 🟡 建议优化 / 🟢 可考虑）

## 五、25个思维透镜完整清单

> 按5大家族组织（诊断/推演/共情/构建/验证），每族5个透镜。

### 家族 A：诊断组（看清"是什么"）

**A1. Doctor（医生）— 临床推理协议**
- 核心问题：What's the Diagnosis?（诊断是什么？）
- 语言约束：用"主诉""鉴别诊断""辅助检查""病变"思考
- 6步协议：主诉识别 → 现病史 → 鉴别诊断（≥3个）→ 辅助检查 → 初步诊断 → 治疗验证
- 最佳场景：故障排查、根因分析、症状复杂

**A2. Journalist（记者）— 事实纯度协议**
- 核心问题：Just the Facts（只讲事实）
- 语言约束：每句话标注 [观察/一级推断/二级推断/假设]
- 5步协议：事实清单 → 纯度标注 → 信源追溯 → 缺失事实 → 标题压力测试
- 最佳场景：信息混乱、事实核查、会议纪要、去情绪化

**A3. Accountant（会计师）— 比率异动协议**
- 核心问题：Watch the Ratios（关注比率）
- 语言约束：用"比率""异动""基准线""异常偏离"思考
- 5步协议：关键比率清单 → 基准线建立 → 异动时序 → 联动分析 → 比率故事
- 最佳场景：数据诊断、效率评估、比率分析、隐藏问题

**A4. Hacker（黑客）— 渗透测试协议**
- 核心问题：What's Really Going on Underneath?（表象底下到底在运转什么？）
- 语言约束：用"表面声明""信任边界""隐藏层""利用链"思考
- 5步协议：表面声明 → 信任边界 → 隐藏层侦察 → 利用链 → 持续机制
- 最佳场景：深层机制、系统漏洞、表象之下、unintended features

**A5. Plumber（水管工）— 拆解检查协议**
- 核心问题：Take it Apart and See What's Broken（拆开看看哪里坏了）
- 语言约束：用"拆开""部件""连接处""堵塞""泄漏"思考
- 5步协议：整体功能 → 物理拆解 → 连接处检查 → 堵塞点定位 → 拆开验证
- 最佳场景：系统拆解、动手实践、恐惧面对、故障排查

### 家族 B：推演组（预判"会怎样"）

**B1. Chess Master（象棋大师）— 多步推演协议**
- 核心问题：See The Moves In Your Mind's Eye（在脑海中推演棋局）
- 语言约束：用"局面""着法""应对""变化"思考，必须至少推演3步
- 6步协议：当前局面 → 我方着法 → 对方最佳应对 → 对方最坏应对 → 第三步局面 → 替代变化
- 最佳场景：通用分析

**B2. Engineer（工程师）— 系统建模协议**
- 核心问题：Can I Model This and Calculate?（我能建模并计算吗？）
- 语言约束：用"变量""约束""输入""输出""弹性""优化目标"思考
- 6步协议：系统边界 → 变量拆解 → 关系建模 → 约束条件 → 失效模式 → 优化目标
- 最佳场景：模糊决策、量化评估、风险评估、建模分析

**B3. Economist（经济学家）— 激励分析协议**
- 核心问题：How Do People React to Incentives?（人们如何对激励做出反应？）
- 语言约束：用"激励""边际""权衡""均衡""博弈"思考
- 5步协议：参与方清单 → 各自激励结构 → 激励错位 → 边际变化 → 意外后果
- 最佳场景：团队激励、政策设计、行为预测、制度激励对齐

**B4. Philosopher（哲学家）— 意外后果协议**
- 核心问题：What are the Unexpected Consequences of an Intuition?（一个直觉会带来哪些意外后果？）
- 语言约束：用"前提""推论""极限""崩溃""悖论"思考
- 6步协议：前提清单 → 前提否定 → 极限推演 → 反命题检验 → 边界划定 → 反直觉可能性
- 最佳场景：直觉检验、极限推演、潜在风险、二阶效应

**B5. Architect（建筑师）— 愿景构建协议**
- 核心问题：Envisioning the Future（envisioning 未来）
- 语言约束：用"蓝图""结构""空间""体验""落地"思考
- 5步协议：愿景画面 → 结构需求 → 空间关系 → 体验路径 → 第一块砖
- 最佳场景：长期规划、愿景设计、大项目、阶段性里程碑

### 家族 C：共情组（理解"他们要什么"）

**C1. Salesperson（销售）— 深层需求协议**
- 核心问题：Understand Their Minds Better than They Do（比他们自己更懂他们的想法）
- 语言约束：用"需求""痛点""动机""立场""筹码"思考
- 6步协议：对方说出来的需求 → 对方没说出来的痛点 → 对方自己还没意识到的 → 提案在对方脑中的映射 → 拒绝的真实原因 → 最小抵抗路径
- 最佳场景：谈判准备、客户洞察、说服策略、动机分析

**C2. Anthropologist（人类学家）— 文化翻译协议**
- 核心问题：Can You Immerse and Join Another Culture?（你能沉浸并融入另一种文化吗？）
- 语言约束：用"文化""语言""仪式""规范""局内人/局外人"思考
- 5步协议：部落识别 → 语言差异 → 仪式和规范 → 权力结构 → 文化翻译
- 最佳场景：跨部门协作、文化差异、陌生领域、沉浸式理解

**C3. Psychologist（心理学家）— 理解检验协议**
- 核心问题：Test Your Understanding of Other People（检验你对他人理解的程度）
- 语言约束：用"检验""验证""预测""反馈"思考，必须包含验证步骤
- 5步协议：你对某人的判断 → 可观察预测 → 替代解释 → 验证实验 → 反馈校准
- 最佳场景：人际冲突、动机分析、行为预测、认知偏差

**C4. Teacher（教师）— 知识盲区共情协议**
- 核心问题：Can You See What it is Like Not to Know Something Obvious?（你能体会"不知道显而易见的事"是什么感觉吗？）
- 语言约束：用"已知""未知""前提知识""理解门槛""误解"思考
- 5步协议：你知道但对方不知道的 → 前提知识检查 → 常见误解预测 → 不知道的体验 → 搭建阶梯
- 最佳场景：知识诅咒、沟通障碍、培训设计、受众理解

**C5. Politician（政治家）— 信念操作协议**
- 核心问题：What Will People Believe?（人们会相信什么？）
- 语言约束：用"信念""解读""联盟""叙事""利益"思考
- 5步协议：利益方清单 → 各自解读 → 意外策略 → 联盟分析 → 叙事设计
- 最佳场景：舆论管理、利益相关者、感知现实、预期管理

**C6. Actor（演员）— 真实感受协议**
- 核心问题：The Best Way to Pretend is to Be Real（最好的假装就是真实感受）
- 语言约束：用"感受""状态""唤起""真实""角色"思考
- 5步协议：需要什么状态 → 真实感受差距 → 过去的类似时刻 → 唤起路径 → 真实检验
- 最佳场景：状态管理、情绪调节、角色代入、自信构建

### 家族 D：构建组（创造"怎么做"）

**D1. Programmer（程序员）— 自动化识别协议**
- 核心问题：What's the Pattern I Can Automate?（我能自动化的模式是什么？）
- 语言约束：用"模式""规则""输入""输出""自动化""重构"思考
- 5步协议：重复识别 → 模式抽象 → 边界情况 → 自动化成本 → 重构机会
- 最佳场景：流程自动化、重复劳动、模式抽象、标准化

**D2. Designer（设计师）— 物替人言协议**
- 核心问题：The Things You Make Communicate For You（你制作的东西替你说话）
- 语言约束：用"形式""引导""暗示""自解释""体验"思考
- 5步协议：你的意图 → 当前形式 → 形式与意图对齐 → 自解释检查 → 摩擦点消除
- 最佳场景：用户体验、信息设计、自解释文档、表达优化

**D3. Novelist（小说家）— 故事连贯性协议**
- 核心问题：Does Your Story Make Sense?（你的故事讲得通吗？）
- 语言约束：用"角色""情节""动机""连贯""场景"思考
- 5步协议：故事线 → 角色一致性 → 场景缺失 → 读者测试 → 替代情节
- 最佳场景：叙事结构、品牌故事、说服力、起承转合

**D4. Entrepreneur（企业家）— 试错优先协议**
- 核心问题：Do a Lot of Things; See What Works（多尝试，看哪些有效）
- 语言约束：用"验证""迭代""假设""速度""学习"思考
- 5步协议：你的默认方案 → 最小验证 → 替代方案 → 速度 vs 完美 → 学习机制
- 最佳场景：资源有限、方向不明、快速验证、MVP设计

**D5. Artist（艺术家）— 创造力优先协议**
- 核心问题：What if Creativity Were the Priority?（如果把创造力放在第一位会怎样？）
- 语言约束：用"独特""有趣""约束""表达""新奇"思考
- 5步协议：常规答案 → 约束清单 → 约束释放 → 新奇标准 → 约束回加
- 最佳场景：创意瓶颈、方案同质化、差异化竞争、品牌设计

### 家族 E：验证组（检验"对不对"）

**E1. Mathematician（数学家）— 可证伪性协议**
- 核心问题：You Don't Know Until You Can Prove It（无法证明就不算知道）
- 语言约束：用"证明""证伪""前提""条件""命题"思考
- 5步协议：结论精确定义 → 证伪条件 → 证据评估 → 反命题构建 → 边界划定
- 最佳场景：逻辑严谨性、证明标准、漏洞排查、结论验证

**E2. Scientist（科学家）— 假设检验协议**
- 核心问题：Make a Hypothesis and Test It（提出假设并验证）
- 语言约束：用"假设""零假设""实验""变量""证据"思考
- 5步协议：你的假设 → 零假设 → 混淆变量 → 实验设计 → 重复性
- 最佳场景：假设验证、因果推断、实验设计、A/B测试

**E3. Critic（评论家）— 借力构建协议**
- 核心问题：Can You Build on The Work of Others?（你能在他人作品基础上构建吗？）
- 语言约束：用"作品""脉络""位置""构建""超越"思考
- 5步协议：前人作品 → 位置判断 → 借力点 → 超越路径 → 后人起点
- 最佳场景：创新不足、二次创作、深度解读、竞品分析

**E4. Soldier（军人）— 纪律防错协议**
- 核心问题：Routine and Discipline Prevent Deadly Mistakes（纪律和常规防止致命错误）
- 语言约束：用"流程""检查点""纪律""失误""保护"思考
- 5步协议：关键节点 → 失误成因 → Checklist 设计 → 自动化保护 → 纪律成本
- 最佳场景：关键流程、安全合规、checklist设计、纪律执行

## 六、认知原理七分类

25个透镜按底层认知机制分为7类。选择透镜组合时，**必须确保覆盖至少3个不同类别**。

| 类别 | 核心机制 | 透镜 | 一句话 |
|------|---------|------|--------|
| **诊断类** | 从症状到根因 | Doctor, Accountant, Engineer, Hacker | 找到问题在哪 |
| **建模类** | 抽象、量化、预测 | Engineer, Architect, Programmer, Mathematician | 把问题结构化 |
| **人际类** | 动机推断、认知弥合 | Salesperson, Psychologist, Teacher, Anthropologist, Economist | 理解人 |
| **推演类** | 多步预判、极限检验 | Chess Master, Philosopher, Politician | 看未来会怎样 |
| **创意类** | 约束释放、快速试错 | Artist, Entrepreneur, Critic, Novelist | 打破定势 |
| **执行类** | 纪律、自动化、设计 | Soldier, Programmer, Designer, Plumber | 保落地 |
| **求真类** | 事实核查、假设检验 | Journalist, Scientist, Mathematician | 去噪音 |

## 八、模糊输入识别与隐形 Check-In

### 识别信号
- 输入以情绪词开头（"烦死了""感觉不对""事多且杂"）
- 只有症状描述，没有明确问题（"供应商不配合""方案写不下去"）
- 输入长度短且跳跃，缺乏上下文
- 用户说"帮我看看"但没有指定看什么

### Step 1：问题分类（自动识别，不问用户）

| 输入特征 | 判定类型 | 进入模式 |
|---------|---------|---------|
| "帮我看看这段内容/方案/邮件" | 内容审查 | 模式E |
| "全面审视/有没有漏掉" | 全面扫描 | 模式B |
| 具体问题描述（有主体、有症状） | 精准诊断 | 模式A |
| "卡住了/想不出" | 创意突破 | 模式C |
| "用XX视角看看" | 单透镜深入 | 模式D |

### 跨类型问题处理
判定主类型 + 标注次类型，主类型给2个透镜，次类型给1个透镜。

### Step 2：轻量补充（只在必要时问，最多1个追问）
- 无法判定目标 → 问："你希望通过这次思考达成什么结果？"
- 带情绪但无事实 → 问："能描述一下最近发生的具体事件吗？"

### Step 3：约束判断（内部推断，不询问）
从上下文推断时间、资源、权力边界等约束。

## 十一、停止信号（认知负荷控制）

**核心理念**：5个透镜 = 5次认知切换 = 每次使用都在付认知税。用户的时间有限，不是每个盲区都值得看。

**经济学原理**（用 Economist 边际收益视角）：
- 第 1-2 个透镜：覆盖约 80% 核心盲区（边际收益最高）
- 第 3 个透镜：覆盖特定场景遗漏风险（边际收益递减）
- 第 4-5 个透镜：仅在决策重大、时间充裕时展开（边际收益最低）

**裁决规则**：每次输出透镜组合时，必须附带盲区覆盖裁决表（4列：优先级、透镜、覆盖的盲区[第二人称]、建议投入时间）。

## 十四、失败模式与 Fallback

| 场景 | 触发条件 | 一线修复 | 仍失败兜底 |
|------|---------|---------|-----------|
| 输入太模糊 | 1a 拆不出 5 要素，3 个以上标"用户未明确" | 追问 1 个关键问题（目标或约束） | 用"症状描述"模式，只做快速版（2 透镜） |
| 透镜匹配不准 | 4a 回环验证发现没对准痛点 | 回到 3c 重新查表交叉 | 降级为模式 B（全视角扫描），让用户自己选 |
| 透镜间同构 | 铁律 2 检查发现当前透镜和上一个重复 | 换异质性框架重新切入 | 砍掉重复透镜，减少到 2 个但保证角度不同 |
| 用户无响应 | 输出后用户沉默或只说"嗯" | 用关键不确定性追问一次 | 停止推进，输出当前结论 + "需要深入哪个方向告诉我" |
| 跨类型无法判定 | 主类型和次类型都模糊 | 默认"精准诊断"模式，选最通用的 3 个透镜 | 直接问用户"这个问题你最想从哪个角度切入" |

## 十五、使用原则

1. **不做简单罗列**：不要一次性抛出25个工具让用户自己选。先分析，再精准推荐。
2. **引导而非灌输**：提出好问题比给答案更重要。
3. **鼓励远迁移**：主动提示用户"这个工具来自XX领域，用到你的问题上会看到什么？"
4. **承认边界**：有些组合确实不适用（如小说家诊断医疗问题）。
5. **中文为主**：职业名称保留英文以便溯源。
6. **每次输出使用记录模板**：帮助用户积累个人透镜档案。
7. **输出最后必须有"行动优先级"**：不让思考停在洞察层面，必须落地为可执行项。

## 十六、咏春拳哲学对应关系（特色亮点）

### 组合策略：咏春负责深度，透镜负责广度

| 咏春支柱 | 核心含义 | 对应认知原理类别 | 具体透镜 |
|---------|---------|------------------|---------|
| **结构 > 力量** | 角度和框架优于蛮力 | 建模类 | Engineer、Architect、Programmer |
| **守中用中** | 控制核心通道，找到最优破局点 | 诊断类 | Doctor、Accountant、Hacker |
| **同时攻防** | 化解阻力同时直达目的 | 推演类 | Chess Master、Philosopher、Politician |
| **小念头** | 回到最原初的基本结构 | 求真类 | Journalist、Scientist、Mathematician |

### 实战组合公式
**先用透镜扫描（广度）→ 再用咏春破局（深度）**：
1. 问题出现 → 用问题分类映射表选出2-5个最相关透镜
2. 透镜扫描 → 快速暴露盲区，收集多维度信息
3. 找到核心 → 用咏春"守中用中"定位最关键的1-2个破局点
4. 一击制胜 → 用咏春"结构>力量"设计最小调整撬动最大改变

## 十八、原文核心结语

> "Creative solutions require divergent thinking, causing us to think of one tool when we need others."

经典的蜡烛实验：被要求用一盒图钉把蜡烛固定在墙上。解决方案是把盒子作为底座——直接对蜡烛用图钉只会弄得一团糟。这很难，因为我们把盒子看作工具的容器，而不是工具本身。

这些工具中的许多可能允许你考虑从未想过的创造性解决方案。例如：
- 如果你是一个企业家，如果你像艺术家、教师或小说家的方式接近它，你的业务会是什么样子？
- 如果你是一个程序员，如果你采用销售人员或会计师的工具，你的代码会如何改进？
- 如果你是一个记者，如果你像科学家、经济学家或水管工一样行动，你的作品会有什么不同？
