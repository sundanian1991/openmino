01
痛点识别：找到重复操作的模式
动作
列出你每周重复 ≥ 3 次的后台操作。问自己：这个操作有没有固定的输入→点击→确认模式？
标准
找到至少 1 个纯 DOM 操作、无验证码、数据源固定的场景。如果涉及验证码或动态 token，先放弃——从最简单的开始。
风险
选错场景——一开始就挑战验证码或反爬机制，挫败感会阻止你继续。
02
环境准备：一条命令就能跑起来
动作
pip install playwright openpyxl → playwright install chromium。确保脚本和 Excel 数据源在同一目录。
标准
跑通 hello world：打开目标页面、截图保存。确认 浏览器能启动、网络能通、Cookie 有效。
风险
Playwright 版本不兼容、Chromium 下载失败。先解决环境问题，不要跳过直接上生产逻辑。
$ pip install playwright openpyxl && playwright install chromium
03
数据驱动：Excel 列名就是 DOM 字段名
动作
建立 Excel 列名 ↔ 页面字段的一对一映射表。通过 label 文本定位输入框，按行循环填充。
标准
用第一条数据跑通全字段填写，确认无遗漏字段，确认下拉选择/日期选择器/单选按钮三类特殊控件都能正确处理。
风险
变量名不匹配——Excel 的列名和页面的字段 label对不上。全链条中断，必须逐字段核对。
04
安全设防：三条红线，开工前就写好
动作
一、只点保存不点提交——给人工复核留窗口。二、出错立即暂停——不盲目 continue。三、先测 1 条——跑通全流程再批量。
标准
三条红线全部写进代码——不是注释，是 if-error-break、click-save-not-submit、batch-size=1-first。
风险
误点提交 = 不可撤回。批量中断后盲目 continue = 数据错乱。跳过单条测试直接批量 = 全量重做。
05
执行验证：每步截图，事后可追溯
动作
每条记录保存后截图，存到 screenshots/ 目录。全部完成后，逐条抽查：字段是否填对、截图是否完整。
标准
27 家全量截图 + 0 缺失字段。数据校验规则全部通过——供应商名称不为空、银行账号不为空、统一信用代码不为空。
风险
批量跑完才发现数据错乱——只能逐条人工修正，自动化白做。截图是最低成本的保险。
# 基础校验：跑之前先查数据
def validate(record):
    assert record["party_b_name"],     "供应商名称为空"
    assert record["account_number"],   "银行账号为空"
    assert record["uniform_credit_code"], "信用代码为空"
06
可迁移化：下次只需要换数据源
动作
提取通用模式：label 定位 → fill → select → screenshot → next。这个模式对任何后台表单类操作都适用。
标准
下次遇到新场景，只需换 Excel 列名映射表 + 页面 URL，核心循环不动。迁移到 ≥ 2 个不同场景验证通用性。
风险
过度抽象——为了"通用"写了一个框架，但实际只需要 50 行脚本。抽象成本 > 复用收益时，别抽象。
03 · 操作红线 · DO / DON'T
自动化不是放任，是更严格的自律。
DO · 该做

把安全写进代码

  • 先跑 1 条全流程，确认无误再批量
  • 出错立即暂停，不盲目 continue
  • 每一步操作截图留证
  • 只点保存，绝不点提交
  • 跑完后逐条人工抽查
DON'T · 别做

把信任写成放任

  • 跳过测试直接批量 27 条
  • 忽略错误继续执行
  • 盲目信任自动化不抽查
  • 误点提交按钮
  • 只在注释里写安全规则，代码里不设防
CLOSING · 方法论要义
自动化的边界不是代码能力，是你留给人工复核的窗口。只保存，不提交——不是保守，是把最终决策权留在人手里。三根支柱（环境准备、数据驱动、安全红线）撑起的不是一台机器，是一次可靠的人机协作。
