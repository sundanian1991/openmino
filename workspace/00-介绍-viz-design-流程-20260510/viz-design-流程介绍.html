viz-design 可视化设计器

9509 条 DNA 模式库 · 15 类意图索引 · 从需求到交付的完整流程

核心定位

不是画图工具，是可视化设计师。

它决定"用什么图、为什么、怎么讲"，但不自己动手画。画图交给渲染技能，容器交给容器技能。

输入

  • 抽象概念、非结构化数据
  • 文字描述、复杂关系
  • 供应商分析、业绩数据
  • 战略建议、咨询图表需求

输出

  • intent.md — 意图决策
  • storyboard.md — 视觉叙事
  • spec.md — 编译规格
  • checklist.md — 验收清单

核心原理

图表思维是认知压缩。 文字是线性的，图像是并行的。一张好图让大脑跳过解码阶段，直达模式识别。
选图本身就是洞察。 选漏斗说明你看到了衰减关系，选齿轮说明你看到了联动机制。图的选择先于图的绘制。
9509 条 DNA = 9509 个经验记录。 不需要从零想，从 DNA 索引中匹配最佳实践。

视觉层级铁律

视觉突出程度必须与重要性严格匹配。

最重要的元素 = 视觉上最突出。禁止多个元素同等突出相互抢夺注意力。

手段用法
大小/类型/标签调整关键元素的视觉权重
参考线/分割区块组织信息层级
字号/加粗节制使用，重要文本突出
颜色强度为关键数据着色
注释/说明重要事件添加文字解释
离群值标注或将双 Y 轴数据拆为两张图

全局色系（Birchline）

图表颜色不超出此体系，不自创任何颜色。用灰度 + 单重点色让关键信息跳出来。

四阶段工作流详解

Phase 0 — 全局风格锁定

多图场景先锁全局风格，后续每张图从全局继承。单图场景跳过。

风格学派： hash(主题名) → style-schools.md
全局色系： 1 套主色 + 1 套强调色 → color-themes.md
全局字体： 锁定 1 对 → typography-moods.md
多图衔接： 编号连续 / 视觉引导 / 配色统一
产出： global-style.md

Phase 1 — DNA 意图匹配

从 9509 条 DNA 中匹配最佳实践，不重新发明轮子。

1. 读 dna-index-ai.md — 意图速查表匹配
   "趋势变化" → 面积图/折线图/散点图
   "分类对比" → 柱状图/棒棒糖图/蝴蝶图
   "占比构成" → 柱状图/堆叠条/饼图
   "驱动因素" → 瀑布图/桥接图
2. 锁定图表 → 从索引取最佳案例 ID
3. 按 ID 查 TSV → 继承全策略
   chart_type / highlight / annotation / composition
   reading_path / color_palette
4. 未命中 → 自主推理，参考 concept-pattern-map
5. 自动选择器 → auto-selector.md
DNA 优先：锁定后按案例 ID 查完整 DNA，继承全策略。未命中时自主推理，不走独立决策树。
速查优先：DNA 匹配后立即查 dna-playbook.md。已有施工图模板的直接套用，只替换数据。

  • 读 narrative-arcs.md + visual-beats.md 定弧线和节拍

Phase 3 — spec.md

编译规格，生成 machine-readable 的渲染约束。

   可视化目的 / 想传达什么 / 结论 / 思路

   模式选择 / 标题(结论性) / 视觉编码
   数据组织 / 标注策略(高亮≤10%) / 布局

Phase 4 — checklist.md 验收

出图前自检 → 验收 → 不通过回退。

  • 出图前自检 — 读 anti-patterns.md
  • 验收 — 读 verification-standards.md
  • 写入 checklist.md，标注通过/不通过
  • 4 项以上不通过 → 回退 Phase 1
  • 同一修复策略连续失败 2 次 → 升级为架构级变更

轻量模式（单图快速场景）

  • Phase 1+2 合并为口头确认（不写文件）
  • Phase 3 必须写 spec.md
  • Phase 4 验收后，必须委托渲染技能执行，不得自己出图

SPEC 编写规范

SPEC 是视觉叙事剧本，不是渲染指令。

叙事意图（必须先写）

【可视化目的】 为什么要做这张图？认知缺口是什么？
【想传达什么】 核心信息（只有一句话）
【结论】 读者应该得出什么判断？
【思路】 为什么选这个图？视觉叙事路径是什么？

视觉执行

【模式选择】 编号 + 名称 + 匹配理由（1 句话）
【标题】 主标题：结论性（不是描述性）；副标题：时间范围 + 对象
【视觉编码】 X/Y 轴编码什么 / 颜色编码什么 / 大小编码什么
【数据组织】 字段清单 / 排序规则 / 聚合规则
【标注策略】 高亮 ≤10% / 标注写原因+幅度（不写数字）
【布局】 画布尺寸 / 配色参考 13-VISUALIZATION.md 或自选

渲染约束（machine-readable）

spec.md 内"渲染约束"章节，下游技能必须严格消费，不得自行翻译。

约束项内容
标注清单逐项枚举：type/target/text/position/offset/style
视觉权重hero/medium/light 的 target + 精确样式
颜色映射每个元素标识 → hex 色值
布局约束canvas/grid/legend/tooltip 精确值

设计原则映射

设计原则SPEC 中如何体现
标题写结论SPEC【标题】必须是结论性的
颜色是信号不是装饰SPEC【视觉编码】说明颜色编码什么
高亮 ≤10%SPEC【标注策略】限定高亮范围
色系 ≤2 rampSPEC 注明色系选择理由
一个图一个核心洞察SPEC【想传达什么】只写一句话

三层架构

viz-design 处于中间层，上游接收策略决策，下游驱动渲染和容器。

策略层 viz-data-story 大纲 → 几张图 → 叙事顺序
设计层 viz-design 每张图 → 选模式 → 写 SPEC → 选容器
渲染层 viz-echarts / viz-svg-flow / viz-chart / viz-infographic 实际绘制图表
容器层 ppt-html / fe-mino / mini-pdf / ppt-deck-builder / fe-frontend 组装、翻页、仪表盘等

技能边界

本技能负责不负责
概念 → 模式匹配（选什么图）实际画图（绝不自己写渲染代码）
SPEC 编写（标题/标注/布局/叙事）配色设计（各渲染技能自带配色体系）
视觉叙事策略（先讲什么后讲什么）数据清洗（交给数据技能）

反模式（禁止）

  • 自己写 ECharts option / SVG path / matplotlib 代码
  • 猜下游技能的 JSON 输入格式
  • 多图各自选色，不锁全局
  • 自己组装多图 HTML（应委派容器技能）
  • 出图后不验收即交付
  • 为了走流程而走流程（每个 Phase 都有明确产出价值）
