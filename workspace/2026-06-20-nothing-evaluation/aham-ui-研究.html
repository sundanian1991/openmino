01 · 概念

设计系统不是给设计师看的，
是给 AI 读的。

Aham 系列（Voice — 录音转写、Survey — 现场调研、PPT — 幻灯片制作）共享同一套本地优先 macOS AI 工具的设计地基。每个工具各自做一件事，但看起来像同一家公司的产品——靠的不是设计师手调，是 AI 消费同一套规范。

规范 = 一把尺子（tokens.json）+ 七层规则（DESIGN.md）+ 参考实现（aham-ui.css）。AI 从 token 取值、按规则组装、对照成品确认，不需要"问设计师这个间距用多少"。

02 · 一条铁律

比八条规则有效

AGENTS.md 没有写 8 条规则、没有 13 项校验、没有 Step 1-2-3。只有一句话。因为最常见的产出不一致不是 AI 违背了某条具体规则，是 AI 在不确定时发明了一个新值——新颜色、新字号、新间距。一条铁律覆盖所有情况。

Iron Rule
值只从 tokens.json 取，颜色只用调色板，字体只用两族（Inter / Inter Display + JetBrains Mono），间距/圆角/控件尺寸只用既定档位——绝不自创值、自创组件、加装饰。拿不准就回到 tokens.json 看一眼，不要猜。
这样做
token 取值
var(--blue) = #336EE8
不这样做
自由发挥
color: #4A90D9

04 · 线性消费

AI 的消费顺序，
不是"如果就"，是"先→再→然后"

Aham UI 没有条件分支。AI 始终按同一个顺序消费：先取值，再读规则，再看成品。不需要做路径选择。

对比 nian-design 的 Step 2（2A 命中变体 / 2B 组件替换 / 2C 注入参数 / 2D 灵感嫁接）——AI 需要不断判断"当前在哪条路径"。Aham 的线性顺序意味着 AI 从不需要问"我该走哪条路"，只需要按顺序执行。

06 · 设计决策

四个为什么

为什么是单蓝？
克制系统的极致测试——"如果只能用一个颜色，你的设计还成立吗？" 单蓝意味着你无法靠颜色区分信息层级，必须靠字号/间距/位置。这在单色体系中是一种纪律训练。
为什么不用 Apple 的取值？
"框架/方法/比例参照 Apple HIG，但取值全部自定。" 版权是表层原因，深层原因是 Apple 的 SF 字体和 8pt 基线网格不适合 Inter 和 4px 基自定体系。学方法不抄答案。
为什么 AGENTS.md 只有一页？
AI 的阅读成本很高。如果入口文件太长，AI 会选择性跳过。一页 + 一条铁律 + 四步顺序，AI 消化完再深入 DESIGN.md 或 css 文件。分层消费比全量加载高效。
为什么状态是"6px 点+文字"不是徽章？
pill/徽章/色块/红黄绿灯是多色系统的表达方式。单蓝体系中状态只需要"存在/不存在"和"正常/异常"两个信号。6px 点 + 文字是目前最小的视觉负担。
07 · 启发

三样可以拿走的东西

Aham 的做法nian 能借什么优先级
线性消费顺序
不是条件分支，是一条路走到底		 Step 2 的 2A-2D 四段分支合并为一段。换组件和嫁接灵感降级为例外处理，不出现主流程中。80% 场景下取变体→填内容→校验即可。 P0
一条铁律覆盖最常犯的错
"值只从 tokens.json 取"		 加一条"值只从 token-root.css 的 :root 取，不自创新值。拿不准就打开 token-root.css 看一眼。" 覆盖 AI 不确定时自由发挥颜色的 80% 场景。 P1
继承说明说清楚框架来源
"框架参考 Apple HIG，取值自定"		 在 CRAFT-RULES 开头写：规则框架借鉴 Nothing Design（工业审美+瑞士排版），但所有值（7 色 DNA、几何间距、8:1 字号）为原创。AI 知道边界后不会回去抄 Nothing 的值。 P2
核心差异总结
Aham 和 nian 都是"供 AI 消费的设计系统"，但解决的问题域不同：Aham 是控件级（按钮/表格/输入框），nian 是整页级（报告/品牌/数据页/策展）。所以 nian 不能照搬 Aham 的"一条铁律就够"方案——但你刚才看到的三个借用点（线性顺序、铁律兜底、继承说明）是领域无关的，可以直接拿。
