# 把团队工作沉淀为 AI 可用的知识库

> 卡帕西 LLM Wiki 范式、三个落地技能与 KaaS 团队共享
> 面向团队内部分享 · 2026 年 6 月
> 演示版：按"文档 → wiki → 健康检查 → 查询 → 团队共享"闭环组织

---

## 开场（1 分钟）

今天不讲概念，直接跑一遍完整链路。

**核心观点**：团队知识量级不大时（< 5-10 万 token），把 wiki 全塞进大模型上下文，比 RAG 更轻、更准、零基建。

**演示目标**：拿一份真实工作文档，现场走完"整理 → 体检 → 查询 → 共享"四步。

---

## 第一部分：我们在解决什么问题（2 分钟）

AI 转型需要知识底座。但常见的 RAG/向量检索方案有三个硬伤：

| 问题 | 具体影响 |
|------|---------|
| 搭建成本高 | 要向量库、embedding 服务、检索调优 |
| 分块切断语义 | 制度文档被切成 chunk，引用关系丢了 |
| 检索有损 | 向量是"大概相关"，关键阈值可能召回不到 |

**我们的判断**：供应商管理知识库提炼层约 5300 行，远低于 RAG 阈值。上 RAG 是过度设计。

---

## 第二部分：卡帕西 LLM Wiki 是什么（3 分钟）

Andrej Karpathy 今年提出的范式：用 LLM 维护个人/团队知识库。

### 三层结构

| 层 | 目录 | 职责 | 可变性 |
|---|------|------|--------|
| 原文层 | `raw/` | 完整原文归档，source of truth | 只读，永不修改 |
| 提炼层 | `distill/` | AI 直接调用的结构化知识，互相链接 | 可重新编译、可改 |
| Schema 层 | `CLAUDE.md` | 告诉 AI 怎么组织 wiki、按什么规则维护 | 随团队规范演进 |

**铁律**：`distill/` 每条规则必须能追溯到 `raw/`。追溯不上的，写进 `notes/待验证假设.md`，不硬塞。

### 三个核心操作

| 操作 | 做什么 | 一句话 |
|------|--------|--------|
| **Ingest 摄取** | 读新文档 → 写摘要 → 更新实体页 → 更新索引 → 记日志 | 把新材料"消化"进知识库 |
| **Query 查询** | 对知识库提问 → 定位 → 综合回答带引用 → 好答案回填 | 查 wiki，而不是重读原文 |
| **Lint 健康检查** | 定期体检：找矛盾、过期、孤儿页、缺失链接 | 防止知识库腐烂 |

## 第三部分：闭环演示（15-20 分钟，核心）

**演示材料**：`LLM-Wiki知识库方案-团队分享.docx`（就是今天这份分享稿本身）

任何同事电脑里都有的东西：

- 操作手册（PDF）
- 培训材料（PPT）
- 数据表格（Excel）
- 会议纪要（Markdown）

今天用这份 Word 分享稿做演示。

---

### Step 1：wiki:ingest —— 把文档"消化"进知识库

**触发语**：
> "把这些文档整理进知识库"
> "wiki 化这个文件夹"

**AI 自动做的事**：

```
扫描文件夹
    ↓
Word/Excel/PPT/PDF → Markdown（markitdown）
    ↓
归档到 raw/（一字不改）
    ↓
提炼到 distill/（只留可复用方法论 + 关键参数）
    ↓
更新交叉链接
    ↓
更新 00_索引与导航.md
    ↓
记录 ingest 日志
```

**实际演示结果**：

| 动作 | 结果 |
|------|------|
| 转换 | `LLM-Wiki知识库方案-团队分享.docx` → Markdown |
| 归档 | `wiki/raw/2026-06-16_LLM-Wiki知识库方案-团队分享/` |
| 提炼 | `wiki/distill/10_AI转型与知识库.md` 新增 §9 |
| 索引 | 分册数 10→11，raw 文件数更新 |
| 日志 | `wiki/notes/ingest日志.md` 追加 |

**关键原则强调**：
- raw 只归档，不改一个字
- distill 是结构化知识，不是原文复制
- 每段关键规则必须标 `[来源]`

---

### Step 2：wiki:health —— 给知识库做体检

**触发语**：
> "检查一下知识库"
> "跑个体检"

**检查项**：

| 检查项 | 目标 | 说明 |
|--------|------|------|
| 来源覆盖率 | ≥95% | distill 关键段落是否都标了 `[来源]` |
| raw 完整性 | 0 缺失 | distill 引用的 raw 文件是否都存在 |
| 冲突状态 | 逐个记录 | 不同文档对同一规则说法冲突 |
| 孤儿页 | 无 | 没有任何其他页链接到的"死页" |
| 过期声明 | 无 | raw 已更新但 distill 没改 |

**实际演示结果**：

```
提炼层文件: 13个
原文层文件: 52个
提炼层总行数: 5730行
来源标注覆盖率: 359/365 (98.4%)
未解决冲突: 14个
缺失raw文件: 2个
健康度得分: 40/100
```

**要解释的点**：
- 新增内容全部通过
- 剩余 8 个问题来自 11/12 分册历史遗留，不是本次引入
- 健康度分数低是因为历史遗留，新跑的内容是干净的
- 这就是 Lint 的价值：不是追求完美，而是知道哪里有问题

---

### Step 3：wiki:query —— 基于知识库回答问题

**触发语**：
> "查一下 wiki 里有没有 X"
> "本地 wiki 和 KaaS 是什么关系？"

**AI 回答流程**：

1. 先读 `00_索引与导航.md` 定位分册
2. 读 `10_AI转型与知识库.md` §9
3. 必要时回到 raw/ 看原文上下文
4. 引用来源回答

**示例问答**：

**问**：本地 wiki 和 KaaS 知识库是什么关系？

**答**：

| 系统 | 角色 | 何时用 |
|------|------|--------|
| 本地 wiki | 知识的"生产与编译" | 个人沉淀、提炼、迭代 |
| KaaS 知识库中台 | 知识的"共享与检索" | 跨项目查询、写 PRD、团队协作 |
| JoySpace（飞书） | 知识的"发布入口" | 把确认的 wiki 内容录入团队库 |

**落地路径**：个人整理 → 人工确认哪些适合共享 → 提炼并确认准确性 → 上传到 KaaS → 团队成员通过 `kaas-kb` 技能查询。

> 来源：`wiki/distill/10_AI转型与知识库.md` §9.5；原文见 `wiki/raw/2026-06-16_LLM-Wiki知识库方案-团队分享/`

---

### Step 4：上传到 KaaS —— 从个人沉淀到团队共享

**关键定位**：
- 本地 wiki = 生产车间（私密、迭代快、零基建）
- KaaS = 共享仓库（跨人检索、权限管理、版本控制）
- 两者是流水线上下游，不是二选一

**上传方式**：

| 方式 | 适合谁 | 步骤 |
|------|--------|------|
| Git 提交 | 有 Git 权限的人 | clone → 复制到 domains/一级域/项目/ → commit push → 管理台点同步 |
| JoySpace 批量导入 | 普通同事 | 文档传到 JoySpace 目录 → 管理台"文档批量录入" → 粘贴目录 URL |

**KaaS 管理台**：http://ai-kaas.pre-apps.jd.com

## 附录：KaaS 调用铁律

写 PRD / 改代码前必走：

1. `kb_search(query="需求关键词")` → 看命中哪个项目
2. `kb_pack(topic, role, intent, query)` → 一次性拿模板 + 角色 wiki + 历史全文
3. 基于 `pack` 返回的 template 章节顺序起草
4. 有缺口 → `kb_get_artifacts(paths=[...])` 补齐

**铁律**：写 PRD 前必须先 `kb_pack` 拿全量，不能只用 `kb_search` 片段就动笔。
