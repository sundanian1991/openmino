date: 2025-06-18

每天，数百万用户将敏感信息[托付](https://trust.anthropic.com/)给 Claude——从专有代码到机密商业策略，不一而足。在 Anthropic，我们正在研究和构建新技术，以确保用户的信任有据可依——事实上，是确保这种信任得到密码学层面的保障。

"密码学保障"意味着什么？在我们与 [Pattern Labs](https://www.patternlabs.co/) 合作发布的[一份新报告](https://assets.anthropic.com/m/c52125297b85a42/original/Confidential_Inference_Paper.pdf)中，我们描述了机密推理的机制。机密推理是一套工具集，可用于处理加密数据，并证明这些数据仅在能够证明自身可信的服务器中才可被读取。采用这套工具主要有两个原因：

- 模型权重安全：我们可以将机密推理作为整体安全工作的组成部分，保护 Claude 等前沿模型免受日益强大的威胁行为者的攻击，例如 RAND 近期报告《[保护 AI 模型权重](https://www.rand.org/pubs/research_reports/RRA2849-1.html)》中所述的那些威胁；
- 用户安全：我们可以使用机密推理来证明敏感用户数据确实被保密。

我们发布这篇文章及随附的[报告](https://assets.anthropic.com/m/c52125297b85a42/original/Confidential_Inference_Paper.pdf)，旨在解释什么是机密推理以及它能为用户带来的好处。我们也想分享我们对相关系统安全性的思考。这仅仅是研究的初步勾勒，旨在开启对话；我们仍处于这项工作的早期阶段，尚无法预测它会如何演化为未来可能提供的具体设计或功能。

以下章节将介绍机密推理实现的部分技术细节。核心要点是：我们正在构建的系统，旨在确保你的敏感数据除了在必须被处理的那一刻之外始终保持加密——即便处理时，也仅在一个高度受限、可验证的环境中进行。

## 推理服务

机密推理的指导原则是：敏感数据在到达处理点之前应始终保持加密。为强制执行这一点，我们采用了机密计算领域已有的成熟方法。这意味着我们构建一条信任链来证明软件的安全性，然后用该证明来强制执行关于哪些软件有权使用加密密钥的规则。

对于用户数据，有两个环节需要操作敏感的明文（即未被加密或以任何方式混淆的文本）：

- API 服务器。该服务器处理提示词，将其转换为 token，并执行 Claude API 请求背后的大部分逻辑；
- 推理服务器。该服务器在硬件加速器上运行 Claude 的"大脑"，从提示词生成补全 token。

对于模型权重，只有推理服务器会接收到敏感数据。

本文将聚焦推理服务器——API 服务器的安全性同样重要，但超出了本文想要描述的范围。由于并非所有加速器都已完全支持机密计算，我们正在探索一种推理服务器的实现方式：在一个小型的、安全的"模型加载器和调用器"之上构建，该组件可在可信环境中运行。这个加载器程序执行几项简单任务：

- 接收加密数据，解密后发送至加速器；
- 调用加速器执行推理，并将加密结果返回给调用方。

只有"可信"加载器能够访问解密数据。系统其余部分是"不可信"的，但可以向加载器发送请求。

我们正在基于这一设计构建自己的实现。在该实现中，推理服务器的大部分运行在"不可信"一侧——该侧可能会频繁变更，但变更不会影响系统的整体安全性。我们有一个小型可信加载器，运行在由 Hypervisor 隔离的独立虚拟机上。加载器向推理服务器呈现为一个"虚拟加速器"，不关心模型架构细节。该"虚拟加速器"仅接受经我们安全持续集成服务器签名的程序，这确保了任何运行的代码都经过了多位工程师的审查。

最终目的是确保：只要加载器正确运行，无论系统其余部分如何行为，我们的机密性要求都能得到满足。因此，证明加载器正确运行至关重要。

## 可信环境

报告描述了加载器运行在具有以下特定功能集的机密计算环境中：

- 由硬件隔离的内存，与其他工作负载隔开；
- 禁用调试功能；
- 密码学证明：正在运行的是正确的代码。

(1) 可抵御某些形式的物理攻击和恶意 Hypervisor，但将加密的主机内存与加速器共享所需的功能目前尚未完全成熟。我们将继续努力缩小这一差距，但同时将依赖我们的计算提供商在物理数据中心和 Hypervisor 软件层面维护安全。

(2) 和 (3) 可通过广泛支持的机密计算实践实现，使用可信平台模块 (TPM) 作为信任根。TPM 测量启动过程的每个阶段，并报告代表最终结果的哈希值。该哈希值构成了一份证明：加载器服务器按预期方式隔离、运行经过我们签名和审查的代码、并已配置为禁用相关调试功能。密钥服务器可以检查这份证明，仅当接收方已证明自身安全时才释放解密密钥。

一个环境是否"可信"，最终取决于密钥服务器的判断。我们也在探索其他机密计算模型，由其他方验证可信代码并管理独立的密钥服务器。这可能使我们能够为每份数据提供更强的机密性保证。

## 未来方向

随着前沿模型能力不断增强，我们可能需要在安全加载器层引入进一步的防护措施。这可能包括在持有明文模型权重的服务器上增加一层[出口带宽限制](https://www.anthropic.com/news/activating-asl3-protections)，或要求获得安全分类器的签名才能执行推理。我们希望提出这一机密推理模型能够激发讨论，探讨还有哪些额外功能值得探索，以确保 Anthropic 环境的持续安全性和用户数据的机密性。

## 结论

这项研究将推进我们在保护模型权重和用户数据方面的持续努力。使用该模型保护用户请求的设计目标是确保客户数据仅在有增强硬件安全控制的上下文中才被解密：

- 请求在到达 Anthropic 服务器之前的节点即已加密；
- 请求到达 API 服务器时，先解密、处理，然后在传递之前重新加密；
- 推理服务器以加密形式处理请求，请求仅在发送到可信加载器时才解密；
- 生成的补全结果在离开加载器之前即被加密，通过 API 服务器传回调用方。

模型权重的情况则更为简单：它们可以加密存储，在加载器处解密，且永不离开该环境。

硬件设计者（尚未[做到这一点](https://developer.nvidia.com/blog/confidential-computing-on-h100-gpus-for-secure-and-trustworthy-ai/)的）应考虑将机密计算融入他们的芯片中。如果加速器附有硬件信任根，那么此类系统的信任边界可以显著缩小。
