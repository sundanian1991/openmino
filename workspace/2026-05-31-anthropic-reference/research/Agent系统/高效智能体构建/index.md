![高效智能体构建](images/building-effective-agents_492ce5b057.svg)

过去一年，我们与数十个跨行业团队合作，共同构建大语言模型（LLM）智能体。我们观察到一个一致的规律：最成功的实现并非使用复杂的框架或专用库，而是采用简单、可组合的模式。

本文分享我们从客户合作与自身实践中总结的经验，为开发者提供构建高效智能体的实操建议。

## 什么是智能体？

"智能体"有多种定义方式。一些客户将其定义为在较长时间内完全自主运行、使用多种工具完成复杂任务的系统。另一些客户则用这个术语描述更规约化的实现——遵循预定义工作流的系统。在 Anthropic，我们将所有这些变体统称为**智能体系统（agentic systems）**，但在架构上做了一个重要区分——**工作流（workflows）**与**智能体（agents）**：

- **工作流**：LLM 和工具通过预定义的代码路径编排执行。
- **智能体**：LLM 动态主导自身流程和工具使用，自主控制任务完成方式。

下文将详细探讨这两类智能体系统。附录一（"实践中的智能体"）描述了客户在这类系统中发现显著价值的两个领域。

## 何时（以及何时不）使用智能体

用 LLM 构建应用时，我们建议先找出最简单的可行方案，只在必要时增加复杂度。这意味着有时根本不需要构建智能体系统。智能体系统通常以延迟和成本换取更好的任务表现，你需要判断这种取舍是否值得。

当复杂度确实有必要时，工作流为定义清晰的任务提供可预测性和一致性；而智能体则更适合需要灵活性和模型驱动决策的大规模场景。不过对于许多应用来说，通过检索和上下文示例优化单次 LLM 调用通常就足够了。

## 何时以及如何使用框架

有许多框架可以降低智能体系统的实现门槛，包括：

- LangChain 的 LangGraph；
- Amazon Bedrock 的 AI Agent 框架；
- Rivet，一个拖拽式 GUI LLM 工作流构建器；
- Vellum，另一个用于构建和测试复杂工作流的 GUI 工具。

这些框架通过简化调用 LLM、定义和解析工具、串联调用等底层标准任务，让开发者快速上手。但它们常常创建额外的抽象层，遮蔽了底层的提示和响应，使调试更加困难。它们还可能诱使开发者在简单方案足够时增加不必要的复杂度。

我们建议开发者从直接使用 LLM API 开始：许多模式只需几行代码即可实现。如果使用框架，务必理解其底层代码。对底层行为的错误假设是客户出错的常见根源。

参考我们的[cookbook](https://platform.claude.com/cookbook/patterns-agents-basic-workflows)查看示例实现。

## 构建块、工作流与智能体

本节探索我们在生产环境中看到的常见智能体系统模式。我们从基础构建块——增强型 LLM——开始，逐步增加复杂度，从简单的组合式工作流到自主智能体。

### 构建块：增强型 LLM

智能体系统的基本构建块是一个经过检索、工具和记忆等增强的 LLM。我们当前的模型可以主动使用这些能力——生成自己的搜索查询、选择合适的工具、决定保留哪些信息。

我们建议关注实现的两个关键方面：针对具体用例定制这些能力，以及确保它们为 LLM 提供简洁、文档完善的接口。实现这些增强的方式有很多，其中一种是通过我们最近发布的[模型上下文协议（Model Context Protocol）](https://www.anthropic.com/news/model-context-protocol)，开发者可以通过简单的[客户端实现](https://modelcontextprotocol.io/tutorials/building-a-client#building-mcp-clients)与日益增长的第三方工具生态集成。

在本文的其余部分，我们假设每次 LLM 调用都可以访问这些增强能力。

### 工作流模式：提示链（Prompt Chaining）

提示链将任务分解为一系列步骤，每次 LLM 调用处理前一步的输出。你可以在任意中间步骤添加程序化检查（见下图中的"gate"），确保流程仍在正轨上。

**适用场景**：任务可以干净利落地分解为固定子任务的场景。核心目标是通过让每次 LLM 调用处理更简单的任务，以延迟换取更高的准确率。

**典型用例**：

- 生成营销文案后翻译成其他语言。
- 先写文档大纲，检查大纲是否满足特定标准，再基于大纲撰写文档。

### 工作流模式：路由（Routing）

路由将输入分类并导向专门的后续任务。此工作流实现了关注点分离，可以构建更专门的提示。没有这个工作流，针对某一类输入的优化可能损害其他类输入的表現。

**适用场景**：存在明显不同类别、分开处理效果更好的复杂任务，且分类可以通过 LLM 或传统分类模型/算法准确完成。

**典型用例**：

- 将不同类型的客服查询（一般问题、退款请求、技术支持）导向不同的下游流程、提示和工具。
- 将简单/常见问题路由到较小模型（如 Claude 3.5 Haiku），将复杂/罕见问题路由到更强模型（如 Claude 3.5 Sonnet），以优化成本和速度。

### 工作流模式：并行化（Parallelization）

LLM 有时可以同时处理一个任务的不同部分，然后通过程序化方式聚合输出。此工作流有两种关键变体：

- **分段（Sectioning）**：将任务拆分为独立子任务并行运行。
- **投票（Voting）**：多次运行同一任务以获得多样化输出。

**适用场景**：拆分的子任务可以并行化以提升速度，或者需要多个视角/多次尝试以获取更高置信度结果。对于涉及多重考虑的复杂任务，LLM 通常在每次调用专注处理一个方面时表现更好，使注意力集中在每个特定维度上。

**典型用例**：

- **分段**：
  - 实施护栏机制：一个模型调用处理用户查询，同时另一个调用筛查不当内容或请求。这通常比让同一个 LLM 调用同时处理护栏和核心回复效果更好。
  - 自动化评估 LLM 性能：每次 LLM 调用评估模型在给定提示上的不同表现维度。
- **投票**：
  - 审查代码漏洞：多个不同的提示审查代码，发现问题时标记。
  - 评估内容是否不当：多个提示评估不同方面，或要求不同的投票阈值以平衡误报和漏报。

### 工作流模式：编排器-工作者（Orchestrator-Workers）

在编排器-工作者工作流中，一个中央 LLM 动态分解任务，委派给工作者 LLM，然后综合它们的结果。

**适用场景**：无法预测需要哪些子任务的复杂场景（例如编程任务中，需要修改的文件数量和每个文件的修改性质取决于具体任务）。虽然在结构上与并行化相似，但关键区别在于灵活性——子任务不是预先定义的，而是由编排器根据具体输入动态决定。

**典型用例**：

- 每次需要对多个文件进行复杂修改的编程产品。
- 需要从多个来源收集和分析信息以寻找相关信息的搜索任务。

### 工作流模式：评估器-优化器（Evaluator-Optimizer）

在评估器-优化器工作流中，一次 LLM 调用生成响应，另一次调用在循环中提供评估和反馈。

**适用场景**：有明确评估标准，且迭代优化能带来可衡量价值的场景。两个适用信号：第一，当人类明确提出反馈时，LLM 的响应可以得到可证明的改进；第二，LLM 能够提供这样的反馈。这类似于人类作者在打磨一篇精致文档时经历的迭代写作过程。

**典型用例**：

- 文学翻译：翻译 LLM 最初可能无法捕捉某些微妙之处，但评估 LLM 可以提供有用的批评。
- 复杂搜索任务：第一轮收集的信息需要多轮搜索和分析才能获得全面信息。

### 智能体（Agents）

随着 LLM 在关键能力上的成熟——理解复杂输入、进行推理和规划、可靠使用工具、从错误中恢复——智能体正在生产环境中崭露头角。智能体从人类用户的指令或交互讨论开始工作。任务明确后，智能体独立规划和运行，可能在需要更多信息或判断时返回给人类。在执行过程中，智能体必须在每一步从环境中获取"真实反馈"（如工具调用结果或代码执行结果），以评估进展。智能体可以在检查点或遇到障碍时暂停，等待人类反馈。任务通常在完成后终止，但常见的做法是包含停止条件（如最大迭代次数）以保持控制。

智能体可以处理复杂任务，但它们的实现往往是简单直接的。它们通常只是在循环中基于环境反馈使用工具的 LLM。因此，清晰、深思熟虑地设计工具集及其文档至关重要。我们在附录二（"工具的提示工程"）中展开了工具开发的最佳实践。

**适用场景**：难以或无法预测所需步骤数的开放式问题，且无法硬编码固定路径。LLM 可能需要运行多个轮次，你必须对其决策有一定程度的信任。智能体的自主性使其非常适合在可信环境中规模化任务。

智能体的自主性意味着更高的成本和错误累积的风险。我们建议在沙箱环境中进行充分测试，并配备适当的护栏。

**典型用例**（以下来自我们自己的实现）：

- 解决 SWE-bench 任务的编程智能体：根据任务描述对多个文件进行复杂编辑。
- 我们的"计算机使用"参考实现：Claude 使用计算机完成任务。

## 组合与定制这些模式

这些构建块并非刚性规定。它们是开发者可以根据不同用例塑造和组合的通用模式。与所有 LLM 功能一样，成功的关键在于衡量性能并迭代实现。再次强调：你应当*仅*在复杂性能带来可证明的改进时才增加它。

## 总结

LLM 领域的成功不在于构建最复杂的系统，而在于构建*适合你需求*的系统。从简单的提示开始，通过全面评估优化它们，仅在简单方案不足时添加多步骤智能体系统。

在实现智能体时，我们努力遵循三个核心原则：

1. **保持智能体设计简洁。**
2. **通过显式展示智能体的规划步骤来优先保证透明度。**
3. **通过充分的工具文档和测试，精心设计你的智能体-计算机接口（ACI）。**

框架可以帮助你快速起步，但在进入生产环境时，不要犹豫减少抽象层，用基本组件构建。遵循这些原则，你可以创建不仅功能强大，而且可靠、可维护、受用户信任的智能体。

## 附录一：实践中的智能体

我们与客户的合作揭示了两类特别有前景的 AI 智能体应用，它们展示了上述模式的实际价值。这两类应用都说明，智能体在需要对话与行动结合、有明确成功标准、支持反馈循环、融入有意义的人工监督的任务中价值最大。

### A. 客户支持

客户支持结合了熟悉的聊天机器人界面与通过工具集成增强的能力。这天然适合更开放式的智能体，原因如下：

- 支持交互自然遵循对话流程，同时需要访问外部信息和执行操作；
- 可集成工具以获取客户订单数据、知识库文章，以及执行退款或更新工单等程序化操作；
- 成功可以通过用户定义的解决标准和客户满意度评分进行可衡量跟踪。

多家公司通过按成功解决收费的用量定价模式证明了这种方法的可行性，显示出对其智能体有效性的信心。

### B. 编程智能体

软件开发领域对 LLM 功能展现出显著潜力，能力从代码补全演进到自主问题解决。智能体在此特别有效，原因如下：

- 代码解决方案可以通过自动化测试验证；
- 智能体可以使用测试结果作为反馈迭代解决方案；
- 问题空间定义明确且结构化。

在我们自己的实现中，智能体现在可以仅凭 PR 描述解决[SWE-bench Verified](https://www.anthropic.com/research/swe-bench-sonnet)基准中的真实 GitHub 问题。然而，虽然自动化测试有助于验证功能，但人工评审仍然是确保解决方案符合更广泛系统需求的关键。

## 附录二：工具的提示工程

无论你构建哪种智能体系统，工具都可能是你的智能体的重要组成部分。[工具](https://www.anthropic.com/news/tool-use-ga)使 Claude 能够通过在 API 中指定其精确结构和定义，与外部服务和 API 交互。当 Claude 响应时，如果计划调用工具，它会在 API 响应中包含一个[工具使用块](https://docs.anthropic.com/en/docs/build-with-claude/tool-use#example-api-response-with-a-tool-use-content-block)。工具定义和规范应该获得与整体提示同等程度的提示工程关注。在这个简短的附录中，我们描述如何对你的工具进行提示工程。

通常有多种方式指定相同的操作。例如，你可以通过写 diff 来指定文件编辑，也可以通过重写整个文件来指定。对于结构化输出，你可以在 markdown 或 JSON 中返回代码。在软件工程中，这类差异是表面的，可以从一种形式无损转换为另一种。然而，某些格式对 LLM 来说比其他格式难写得多。写 diff 需要在新代码写出来之前就知道块头中有多少行在变化。在 JSON 中写代码（相比 markdown）需要额外转义换行符和引号。

我们关于选择工具格式的建议如下：

- 给模型足够的 token 用于输出，避免它在写之前就有用完的风险。
- 保持格式与模型在自然文本（如互联网）中遇到的格式一致。
- 避免格式开销，使模型不必在写之前预先计数或不必要地转义字符。

一个经验法则是想想在人机界面（HCI）上投入了多少精力，然后计划投入同样多的精力来创建好的*智能体*-计算机界面（ACI）。以下是一些关于如何做到这一点的思考：

- 站在模型的角度思考。基于描述和参数，使用工具的方式是否清晰，还是需要进一步思考？如果需要，模型大概率也需要，而且精确的参数可能比你想象的更重要。
- 如何优化参数名称或描述使事情更加直观？把这想象成为你的工具写一份优秀的文档页面。这对于常用工具尤其重要。
- 测试模型如何使用你的工具：在你的工作台中运行大量示例输入，观察模型犯什么错误，然后迭代改进。
- **防错设计（Poka-yoke）**你的工具。调整参数，使犯错更难。

在构建我们的 [SWE-bench](https://www.anthropic.com/research/swe-bench-sonnet) 智能体时，我们实际在优化工具上花的时间比优化整体提示还多。例如，我们发现当智能体离开根目录后，使用相对文件路径的工具会导致模型出错。为了解决这个问题，我们将工具改为始终要求绝对文件路径——结果发现模型完美地使用了这种方法。
