viz-design v1.0

可视化设计器

你说想表达什么，我决定用什么图表、怎么讲故事，然后交给专业工具画出来。

不自己写画图代码。是脑，不是手。

它做什么？

三件事：选图、讲故事、写规格

1

选什么图

根据你想表达的内容（趋势/对比/占比/机制/流程），从 9509 条案例库中匹配最佳图表类型，不是从零想。

2

怎么讲故事

设计阅读路径：眼睛先看什么、再看什么、最后落在哪。标题写结论，不是写描述。高亮关键数据，淡化非关键信息。

3

写渲染规格

产出完整的可视化 SPEC 文档，包含标注清单、颜色映射、布局约束、视觉权重。下游渲染技能直接消费，不需要猜。

一句话总结：你告诉我想表达什么，我产出完整的可视化设计方案（什么图 + 为什么 + 怎么突出关键信息 + 怎么讲故事），然后委托 viz-echarts 或 viz-svg-flow 把图画出来。

趋势变化（某个指标随时间涨跌）折线图 面积图 散点图
分类对比（谁大谁小、谁好谁坏）柱状图 棒棒糖图 蝴蝶图

技能边界

明确负责什么、不负责什么，避免期望错位

viz-design 负责

  • ✓ 概念 → 模式匹配（选图表类型）
  • ✓ SPEC 编写（标题 / 标注 / 布局 / 叙事）
  • ✓ 视觉叙事策略（先讲什么后讲什么）
  • ✓ 版式选择 + 解读策略
  • ✓ 出图后质量验收

viz-design 不负责

  • ✗ 实际画图（不写 ECharts / SVG 代码）
  • ✗ 配色设计（渲染技能自带配色体系）
  • ✗ 数据清洗（交给数据技能）
  • ✗ 多图 HTML 组装（委派容器技能）
  • ✗ 猜下游渲染技能的输入格式

适合 vs 不适合

不是所有信息都适合可视化——viz-design 会直接劝退不需要的请求

适合使用

  • 有分布、趋势、关联关系需要呈现
  • 多层级数据需要一张图讲清全貌
  • 抽象概念需要视觉化（机制 / 架构 / 流程）
  • 管理层汇报需要结论性图表
  • 多图需要统一风格叙事
  • 有对比 / 排名 / 差距需要突出
  • 复杂关系需要降维表达

不适合使用

  • 信息一句话说得清 → 直接文字
  • 精确数值 / 系数 / 清单 → 用表格
  • 只有单一数据点，无比较维度
  • 需要实时数据流 / 动态刷新
  • 需要用户交互编辑数据 → 用 BI 工具
  • 纯装饰性图片，不传达数据关系

设计原则

所有设计决策都遵循的铁律

标题写结论

不是"供应商业绩对比"，而是"B 组领先 F 组 62%"。

颜色是信号

颜色编码重要性，不是装饰。高亮 ≤10%，色系 ≤2 种。

一图一洞察

每张图只传达一个核心判断。多余的拆到下一张图。

视觉层级

最重要的元素 = 视觉上最突出。禁止多个元素同等突出。

删除非数据墨水

删除所有不影响表意的元素。网格线要淡，边框要去。

色盲友好

禁用红绿对比，使用色盲安全色板。颜色不是唯一的信息编码手段。

完整工作流详情

下面是给想了解内部机制的人看的。简单需求不需要读完。

R0 · 必要性门禁 — 真的需要画图吗？

不是所有信息都适合可视化。先做判断：

  • 信息能一句话说清 → 不做图，文字表达
  • 精确数值 / 系数 / 清单 → 用表格，不走后续流程
  • 涉及分布 / 趋势 / 关联 / 复杂比较？→ 继续
P0 · 全局风格锁定 — 多图场景先统一风格

仅多图场景执行。单图跳过此步。

  • 风格学派：McKinsey / FT / Bloomberg / Excalidraw 等
  • 全局色系：1 套主色 + 1 套强调色
  • 全局字体：锁定 1 对字体配对
  • 多图衔接协议：编号连续、视觉引导、配色统一

产出：global-style.md（多图场景）

P1 · DNA 意图匹配 — 选什么图

核心决策：根据表达意图匹配最佳图表类型。

  • 读 15 类意图索引，每类 3-6 种推荐图表
  • 命中 DNA 案例 → 继承全策略：chart_type / highlight / annotation / composition / reading_path / color_palette
  • 命中施工图模板 → 直接套用，只替换数据
  • 未命中 → fallback 链：
    • 第一步：查 concept-pattern-map.md（纯概念场景）
    • 第二步：查 consulting-archetypes.md（逻辑关系场景）
    • 第三步：自主推理，在 intent.md 中写明推理依据

产出：intent.md

P1.5 · 版式 + 解读策略

不写文件，口头决策。根据受众和场景调整复杂度：

  • 供应商月报（高频 / 低互动）→ 一页一结论，直接标注关键数字
  • 季度汇报（中频 / 中互动）→ 保留核心分布 + 趋势，精简细节
  • 领导 Briefing（低频 / 高互动）→ 突出结论，保留数据支撑细节备讨论
  • 公众 / 跨部门（低互动）→ 减少术语，直接标注结果

版式：40 种模板，按场景自动匹配（核心图表 → 单图居中 / 需解读 → L 型 / 多指标 → 顶部 KPI 等）

P2 · 视觉叙事设计 — 怎么讲故事

设计阅读路径和叙事弧线。

  • 叙事弧线：12 种之一（问题解决型 / 对比型 / 趋势型等）
  • 一句话灵魂：这张图的核心判断
  • 签名视觉元素：什么最突出
  • 克制声明：什么不放进去

产出：storyboard.md

P3 · SPEC 编译 — 渲染约束

将设计决策编译为渲染技能可消费的规格。

  • 数据组织：字段清单 / 排序规则 / 聚合规则
  • 视觉编码：X/Y 轴编码什么 / 颜色编码什么 / 大小编码什么
  • 标注策略：高亮 ≤10% / 标注写原因 + 幅度
  • 渲染约束代码块：标注清单 / 视觉权重 / 颜色映射 / 布局约束

产出：spec.md → 委托 viz-echarts / viz-svg-flow 渲染

嗅探 · 渲染结果快速扫描

委派后、验收前的快速扫描。

快速检查常见问题：内容截断 / 代码泄露 / 中文乱码 / 元素重叠。发现问题自动处理或告知，无问题则进入验收。

P4 · 验收 — 逐项检查

逐项检查，不通过 = 打回重做。

  • 出图前自检：读 anti-patterns.md
  • 按 checklist 逐项验收，标注通过 / 不通过
  • 4 项以上不通过 → 回退 Phase 1 重新设计
  • 同一修复策略连续失败 2 次 → 升级为架构级变更（换图表类型 / 拆分多图 / 重写叙事）

产出：checklist.md

轻量模式提醒：单图明确需求（如"画个柱状图对比 5 个供应商业绩"）→ 跳过 P0/P2，不写中间文件，口头确认后直接写 spec.md 委托渲染。迭代时只改表现层（风格 / 配色 / 版式），内容层（数据 / 结论 / 叙事）全程复用。

常见问题

viz-design 能直接画图吗？
不能。它产出四个 md 文档（intent / storyboard / spec / checklist），然后委托 viz-echarts 或 viz-svg-flow 执行渲染。它是脑，渲染技能是手。
工作流一定要走完所有 Phase 吗？
不一定。单图明确需求走轻量模式：跳过 P0/P2，口头确认直接写 spec.md。只有多图复杂场景才走完整流程。
改个配色要重跑全部流程吗？
不需要。资产分层机制：内容层（数据 / 结论 / 叙事）不重写，全程复用；表现层（风格 / 配色 / 版式）每次迭代可调。迭代时只改表现层。
DNA 库里找不到匹配怎么办？
三级 fallback：先查概念模式库（纯概念场景）→ 再查咨询架构库（逻辑关系场景）→ 最后自主推理，但必须在文档中写明推理依据。
这个技能自己会写渲染代码吗？
不会。这是铁律：viz-design 绝不自己写 ECharts option、SVG path 或 matplotlib 代码。所有渲染委派给下游技能执行。
