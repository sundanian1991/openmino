自己画图，自己验收。是脑，也是手。

它怎么做？

不是走流程，是做减法

1

初稿 — 先有

从 9509 条 DNA 案例库匹配最佳图表类型，写 intent.md / storyboard.md / spec.md 记录设计决策，出第一版完整图表。第一版可以包含所有元素。

2

减法迭代 — 再做精

逐项追问：这个元素有价值吗？能删吗？会让重点更突出吗？最多 3 轮。减到每个留下的元素都在为核心判断服务。

3

验收交付 — 最后检查

按 checklist 逐项检查：截断/乱码/色盲友好/高亮≤10%/标题是结论。不通过打回重做，通过则交付。

核心区别：不是"为了走流程而走流程"，是"为了做出一张清晰的图而反复减法"。好图不是走完 7 个 Phase 做出来的，是减掉不必要的东西做出来的。

减法迭代：逐项追问

出第一版后，对每个元素问这 7 个问题

元素追问判断标准
标题是结论还是描述？结论就留（"B 组领先 62%"），描述就改
图例必须吗？能用颜色+文字替代吗？单系列数据不需要图例
网格线帮助读图还是干扰？多数情况可以淡化或去掉
边框有结构价值吗？几乎总是可以去掉
标注写原因还是写数字？写原因+幅度，不写纯数字
参考线帮助理解基准吗？有明确基准时保留
色块/背景帮助区分阶段吗？有明确阶段划分时保留

意图速查：你想表达什么？

找到你想做的事，右边就是推荐的图表类型

你想表达…推荐图表
趋势变化（某个指标随时间涨跌）折线图 面积图 散点图
分类对比（谁大谁小、谁好谁坏）柱状图 棒棒糖图 蝴蝶图
占比构成（各部分占总体的比例）堆叠柱状图 饼图 堆叠条
驱动因素（什么导致了变化）瀑布图 桥接图
排名 / 差距（谁领先、差多少）水平条形图 棒棒糖图
分布（数据集中在哪、有没有离群）直方图 箱线图
关联关系（两个变量是否相关）散点图 气泡图
机制 / 联动（几个环节如何协作）齿轮图 循环图
层级结构（上下级、分类关系）树图 旭日图 组织架构图
流程 / 阶段（步骤先后顺序）流程图 时间线 桑基图

技能边界

明确负责什么、不负责什么

viz-design 负责

  • ✓ DNA 意图匹配（选图表类型）
  • ✓ 图表设计与渲染（ECharts / SVG）
  • ✓ SPEC 记录（标题 / 标注 / 布局 / 叙事）
  • ✓ 减法迭代（逐元素追问价值）
  • ✓ 出图后质量验收

viz-design 不负责

  • ✗ 数据清洗（交给数据技能）
  • ✗ 复杂多图容器/组装（可委派 html-interactive）

适合 vs 不适合

不是所有信息都适合可视化

适合使用

  • 有分布、趋势、关联关系需要呈现
  • 多层级数据需要一张图讲清全貌
  • 抽象概念需要视觉化（机制 / 架构 / 流程）
  • 管理层汇报需要结论性图表
  • 多图需要统一风格叙事
  • 有对比 / 排名 / 差距需要突出
  • 复杂关系需要降维表达

不适合使用

  • 信息一句话说得清 → 直接文字
  • 精确数值 / 系数 / 清单 → 用表格
  • 只有单一数据点，无比较维度
  • 需要实时数据流 / 动态刷新
  • 需要用户交互编辑数据 → 用 BI 工具
  • 纯装饰性图片，不传达数据关系

设计原则

所有设计决策都遵循的铁律

标题写结论

不是"供应商业绩对比"，而是"B 组领先 F 组 62%"。

颜色是信号

颜色编码重要性，不是装饰。高亮 ≤10%，色系 ≤2 种。

一图一洞察

每张图只传达一个核心判断。多余的拆到下一张图。

视觉层级

最重要的元素 = 视觉上最突出。禁止多个元素同等突出。

删除非数据墨水

删除所有不影响表意的元素。网格线要淡，边框要去。

色盲友好

禁用红绿对比，使用色盲安全色板。颜色不是唯一的信息编码手段。

工作流详情

三循环：初稿 → 减法 → 验收

R0 · 必要性门禁 — 真的需要画图吗？
  • 信息能一句话说清 → 不做图，文字表达
  • 精确数值 / 系数 / 清单 → 用表格
  • 涉及分布 / 趋势 / 关联 / 复杂比较？→ 继续
循环 1 · 初稿 — 先有
  1. DNA 意图匹配 — 读 dna-index-ai.md，按意图选模式
  2. 查 TSV 继承全策略（highlight / annotation / composition / reading_path / 色板）
  3. 命中施工图模板 → 直接套用，只替换数据
  4. 未命中 → fallback：concept-pattern-map → consulting-archetypes → 自主推理（写依据）
  5. 版式 + 解读策略 — 读 layout-selector.md
  6. 写中间文件：intent.md / storyboard.md / spec.md
  7. 出第一版图表（ECharts HTML / SVG）
循环 2 · 减法迭代 — 再做精

逐项追问每个元素（见上方"减法迭代"表格）。最多 3 轮迭代。用户说"够了"则提前终止。中间文件同步更新。

循环 3 · 验收 — 最后检查
  • 内容截断 / 代码泄露 / 中文乱码 / 元素重叠
  • 色盲友好（禁用红绿对比）
  • 高亮 ≤10%
  • 标题是结论性的
  • 一图一核心判断

4 项以上不通过 → 回退循环 1

常见问题

viz-design 能直接画图吗？
能。它使用 references 下的 ECharts 配方和 DNA playbook，直接写 ECharts HTML 或 SVG。不委派给其他渲染技能（除非遇到极端兜底场景）。
中间文件是干什么用的？
不是给下游消费的，是给自己看——记录意图决策、叙事设计、规格参数。做减法和验收时有据可查，迭代时同步更新。
一定要做减法迭代吗？
是的。这是 viz-design 的核心价值。第一版可以包含所有元素，但至少要减一轮，直到"每个留下的元素都在为核心判断服务"。
DNA 库里找不到匹配怎么办？
三级 fallback：先查概念模式库 → 再查咨询架构库 → 最后自主推理，但必须在文档中写明推理依据。
多图怎么保证风格统一？
先锁全局风格（色系 / 字体 / 风格学派），后续每张图从全局继承，不得各自选色。
