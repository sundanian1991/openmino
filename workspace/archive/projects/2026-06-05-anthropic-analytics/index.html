The Core Challenge

Data Is Not Software

LLM 的分析能力是一把双刃剑：能创造性解决问题，也能虚构错误答案。代码有测试作为防护栏，而数据分析往往只有一个正确答案、一个正确来源，却没有确定性的方法证明其正确性。

Generative Coding
开放性解决空间 - 模型的创造力是优势。文档和测试提供自然的防幻觉护栏。编译时报错、测试失败给出了明确的反馈信号。
Analytics Querying
单一正确空间 - 往往只有一个正确答案和一个正确来源。没有编译时验证。用户不具备验证底层正确性的能力。幻觉无声无息。
The Three Failure Modes

为什么数据分析代理会出错

Anthropic 将绝大多数不准确响应追溯为三类模式 - 每一类指向数据基础设施中的一个具体缺口。

01 · Ambiguity

同名不同物

同一个术语在不同上下文中含义不同。"收入"可能指营收、毛利、净收入或合同金额 - 如果底层数据模型没有统一口径，代理永远选不对。

02 · Search

同物不同名

同一个实体在不同系统、不同表中叫法不同。"用户"在 user 表中叫 customer_id，在 order 表中叫 buyer_id - 代理找不到正确的字段。

03 · Drift

概念过时

数据模型在变，但描述数据模型的文档没有跟着变。字段改名、口径调整、表废弃 - 等文档更新时，代理已经在用过时的定义回答问题。

The Solution Stack

四层治理架构

Anthropic 的核心应对 - 每一层针对一类失败模式。越底层越靠近基础设施，越上层越靠近最终用户。

Layer 1 Data Foundations 数据模型、转换、测试。用维度建模和 shift-left 测试消除口径歧义。
Layer 2 Sources of Truth 语义层、指标文档、表文档。代理查这些，不直接查原始数据仓库。
Layer 3 Skills 过程知识编码。告诉代理"先做什么、再做什么、注意什么"。
Layer 4 Validation 离线评测 / 消融实验 / 在线验证。持续量化代理准确性。
The Key Innovation

Pairwise Skills

没有 Skills 时，Claude 回答分析问题的准确率不超过 21%。加入 Skills 后，聚合准确率超过 95%，某些领域达到 ~99%。关键是拆为一对。

Skill 1 · Router

Knowledge Skill

轻量顶级路由器。先试语义层（受控路径），语义层覆盖不了才走 ~30 个参考文件（长尾路径）。在代理搜索百万字段的数据仓库之前，先缩小搜索空间。
  • 优先走语义层 - 口径固化，数字与 BI 工具一致
  • 参考文件描述表结构、粒度、排除条件、陷阱
  • 每个领域 ~30 份 Markdown 文件，人工精选
Skill 2 · Executor

Unbook Skill

将高级分析师的工作流程编码为可执行的步骤序列。问什么、去查哪、怎么查、结果怎么审 - 每一步都有明确规则。
  • 澄清问题→找源→跑查询→对抗式审查
  • 内建十几套可复用分析模式（留存曲线、率分解、漏斗）
  • 每个答案附带 provenance footer（来源 / 置信度 / 新鲜度）
Warning · Skill Decay

从 95% 到 65%，只需一个月

如果没有主动维护，Skill 准确率会在数周内大幅衰减。Anthropic 的应对：将 Skill Markdown 文件与数据模型放在同一个代码仓库，任何改变模型的 PR 都必须同步修改 Skill 文件。Code Review Hook 自动检查 - 如果 reporting-model 变更没有触及 Skill 文件，直接 Flag。大约 90% 的数据模型 PR 都包含配套的 Skill 修改。
Maintenance as Infrastructure

维护即基础设施

三条机制确保文档不会变成"一写就丢"的静态资产。

01 · 同仓共存
Skill 文档与 dbt 转换模型在同一个 repo 里。改模型的 PR 就是改文档的 PR，两者永远同时变更。
02 · 自动同步
Merge 后自动同步到三条路径：Plugin Marketplace（IDE）、Cloud Storage（托管App）、MCP（Agent 运行时）。
03 · 定期修剪
数据模型改进后，之前依赖这些模型的 Skill Scaffolding 不再需要 - 定期清理，不积累技术债。
Quality Assurance

三层验证体系

没有评测就没有改进。Anthropic 部署了三层验证，分别覆盖"知道有没有问题"、"找到问题在哪"、"上线后不出问题"。

01 · Offline Evals

离线评测

两套 QA 数据集：Dashboard-based 覆盖最常见问题，由 Claude 生成、人工验证；Long-tail 覆盖领域边缘场景。每次结构变更后重跑，比较 pass rate。

02 · Ablation

消融实验

每次改一个组件（加一个表、拆分一个 Skill、调整子 agent 调度），保持 QA 集不变，看 pass rate 变化。一次只变一个变量，每次跑一个小时。

03 · Online

在线验证

Provenance Footer（每个答案附带来源/置信度/新鲜度）、领导层报告需要人工签收、每个领域核心 KPI 每天与官方面板核对。

Getting Started

从零开始

如果从零开始，几个规范数据集 + 几十条离线评测 + 一个薄薄的 Knowledge Skill，就能覆盖绝大部分收益。文章中其他所有内容都是在这套基础上逐步追加的。

01
确立数据基础 选 3-5 个最核心的数据集，做维度建模、shift-left 测试、新鲜度监控。在一个受控空间中把口径彻底定死。
02
建立参考文档 为每个数据集写 Grain / Scope / Exclusions / Gotchas - 用 Markdown，面向 LLM 阅读优化。几十条参考文件足够覆盖 80% 的问题。
03
创建离线评测 Claude 自动生成最常见的问题+答案对，人工验证后作为 fixed eval set。每次改结构后重跑，对比 pass rate。
04
迭代，不完美 从 95% 到 65% 是正常衰减。把文档维护当工程问题处理 - 放到工作流里、写成 CI 规则。然后重复。
