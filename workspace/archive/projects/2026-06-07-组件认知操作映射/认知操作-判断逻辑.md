# 认知操作 → 数据形态 → 组件：判断逻辑

> 不是分类，是判断链。每一步回答"为什么"。

---

## 0. 前置：9 种认知操作是否穷尽？

这 9 种来自一个事实：**人看一页内容时，只执行一种认知操作——把视觉信息转化为判断。**

转化方式只有这些：

| 操作 | 本质 | 举例 |
|------|------|------|
| 扫 | 识别整体状态 | "这个月业绩怎么样" |
| 比 | 发现两个对象的差异 | "毛毛虫和毅航哪个好" |
| 排 | 建立多个对象的秩序 | "6 家供应商谁排第一" |
| 读 | 理解因果或顺序 | "问题是怎么发生的" |
| 量 | 判断一个指标的好坏 | "达成率算高还是低" |
| 解 | 理解论证或说明 | "为什么这么说" |
| 信 | 确认信息有依据 | "凭什么相信这个结论" |
| 展 | 揭示隐藏的细节 | "背后还有什么" |
| 感 | 不判断，只感受 | "这个品牌什么气质" |

**为什么是 9 个不是更多？** 因为认知心理学对"信息加工"的分类就是这些。你可以加第 10 个——但加出来的一定是这 9 个的子集或组合。

**验证方法**：拿你做过的所有案例，逐个 section 问"用户在这里执行什么操作"。如果都能归进这 9 种，就够了。

## 1. 扫（概览·SCAN）

**用户在做什么**：不分析，只建立整体印象。看一眼就走。

**数据形态判断**：扫的特征是**不深入、不比较、只看数量级**。所以数据必须是**并列的、同质的、一眼能扫完的**。

| 数据形态 | 为什么是这个形态 | 为什么不是其他 |
|---------|---------------|--------------|
| 并列指标 | 3-5 个数字并排，扫一眼就知道数量级 | 不是对比（没有 A vs B），不是排序（没有先后） |
| 数字矩阵 | 6+ 个数字排成方阵，扫一眼知道分布 | 比并列指标信息量更大，但仍然是"扫"不是"看" |

**组件选择逻辑**：

| 组件 | 为什么选它 | 什么时候不选它 |
|------|----------|--------------|
| **stat-block** | 大数字 + 小标签，扫的效率最高 | 需要看趋势时 → 换 metric-card |
| **metric-card** | 指标 + 迷你趋势线，比 stat-block 多一层"变化方向" | 只看数量级不需要趋势 → stat-block 更干净 |
| **kpi-card** | 带完成度进度，扫的时候同时知道"做了多少" | 不涉及完成度 → stat-block |
| **numeral-grid** | 数字方阵，同质数据的空间排列 | 数据不是同质的 → 用 bento-grid |
| **dashboard-grid** | 多指标仪表盘，扫的密度最高 | 指标 ≤5 个 → stat-block 够了 |
| **data-spotlight** | 极简单数字内联，嵌入文字流 | 需要多个指标 → stat-block |
| **summary-capsule** | 紧凑胶囊组，空间极度有限时 | 空间够 → stat-block 更清晰 |
| **hero-numeral** | Hero 内嵌数字，首屏就要扫 | 不是 Hero 位置 → stat-block |
| **nameplate-label** | 工业标签，等宽文字 | 不是技术/工业场景 → tag |

## 2. 比（对比·COMPARE）

**用户在做什么**：A 和 B 放在一起，找差异、判断优劣。

**数据形态判断**：比的特征是**两个对象同时呈现，差异一目了然**。所以数据必须是**对称的、可对照的**。

| 数据形态 | 为什么是这个形态 | 为什么不是其他 |
|---------|---------------|--------------|
| 左右对照 | 两个对象并排，视觉上直接比 | 不是排序（没有多个对象），不是度量（不是单个指标） |
| 表格对比 | 多维度逐项比，精确但需要阅读 | 左右对照是"扫着比"，表格是"读着比" |

**组件选择逻辑**：

| 组件 | 为什么选它 | 什么时候不选它 |
|------|----------|--------------|
| **comparison** | 双栏非对称，左 sticky "VS" 标签，比的意图最明确 | 维度 >5 个 → 用 data-table |
| **tension-grid** | 2 列 1px gap，Heritage vs Future，比的哲学性更强 | 不是哲学对比 → comparison 更直接 |
| **light-dark-switch** | 同一对象的两面，"before/after" 型对比 | 不是同一对象 → comparison |
| **do-dont** | 二元规则对比，AVOID/PREFER，比的结论最明确 | 不是规则对比 → tension-grid |
| **spec-table** | 技术参数逐项对比，精确但需要阅读 | 不是技术参数 → comparison |
| **data-table** | 多维度交叉对比，信息密度最高 | 维度 ≤3 个 → comparison 更直观 |
| **principle-card** | 单卡内行对比（原 vs 新），比的粒度最细 | 对比对象 >2 个 → data-table |

## 3. 排（排序·RANK）

**用户在做什么**：多个对象建立秩序——谁第一、谁最后、权重怎么分。

**数据形态判断**：排的特征是**有明确的序号或位置关系**，高低/先后/权重一目了然。

| 数据形态 | 为什么是这个形态 | 为什么不是其他 |
|---------|---------------|--------------|
| 条形 + 序号 | 长度=大小，序号=位置，排的意图最直接 | 不是概览（有顺序），不是对比（有多个对象） |
| 分段条 | 长度=权重，段=组成，排的是比例关系 | 不是排名（没有序号），是份额 |
| 方阵 | 空间位置=排序，方阵比条形信息密度更高 | 条形比方阵更容易"扫着排" |

**组件选择逻辑**：

| 组件 | 为什么选它 | 什么时候不选它 |
|------|----------|--------------|
| **rank-bar** | 水平排名 + 进度 + delta，排的意图最明确 | 不涉及进度 → rank-panel |
| **rank-panel** | 带标题的排名容器，排的包装感更强 | 需要精确数值 → ranking-table |
| **ranking-table** | 排名表格，精确数值 + 排名，排的精度最高 | 需要视觉冲击 → rank-bar |
| **seg-bar** | 分段条，排的是比例/权重，不是排名 | 不是比例关系 → rank-bar |
| **weight-bar** | 权重条，排的是各项占比 | 不是占比 → seg-bar |
| **stacked-bar** | 堆叠条，排的是组成结构 | 不是组成 → weight-bar |
| **progress-bar** | 进度条，排的是完成度 | 不是完成度 → seg-bar |
| **numeral-grid** | 数字方阵排，方阵比条形更适合"同质数据" | 数据异质 → rank-bar |
| **card-quad** | 四卡方阵，排的是 4 个同级对象 | 对象 >4 个 → numeral-grid |

## 4. 读（流程·FLOW）

**用户在做什么**：理解顺序——先做什么、后做什么、经过几个阶段。

**数据形态判断**：读的特征是**有明确的时间或逻辑顺序**，前后关系不可颠倒。

| 数据形态 | 为什么是这个形态 | 为什么不是其他 |
|---------|---------------|--------------|
| 节点 + 连线 | 节点=阶段，连线=顺序，流的意图最直接 | 不是排序（不是比大小），是比先后 |
| 编号步骤 | 编号=顺序，步骤=操作，流的是"怎么做" | 不是"经过什么阶段"，是"执行什么动作" |

**组件选择逻辑**：

| 组件 | 为什么选它 | 什么时候不选它 |
|------|----------|--------------|
| **timeline-track** | 链式节点，时间线，流的是"时间上的先后" | 不是时间线 → flow-pipeline |
| **flow-pipeline** | 横向流程线，done/active/pending 三态，流的是"状态推进" | 不涉及状态 → step-sequence |
| **progress-trail** | 进度路径，流的是"阶段 + 完成度" | 不涉及完成度 → timeline-track |
| **level-track** | 等级轨道，流的是"等级晋升" | 不是等级 → flow-pipeline |
| **phase-bar** | 阶段条，流的是"项目阶段" | 不是项目管理 → timeline-track |
| **step-sequence** | 编号步骤，流的是"怎么做 SOP" | 不是操作流程 → flow-pipeline |

## 5. 量（度量·MEASURE）

**用户在做什么**：判断一个指标的好坏——高还是低、达标还是没达标。

**数据形态判断**：量的特征是**单个指标 + 判断标准**，不是比两个对象，是判断一个对象。

| 数据形态 | 为什么是这个形态 | 为什么不是其他 |
|---------|---------------|--------------|
| 仪表 | 一个指针/弧形 + 一个阈值，量的意图最直接 | 不是概览（只有一个指标），不是对比（没有 A vs B） |
| 趋势线 | 一条线 + 时间轴，量的是"变化方向" | 不是当前值的好坏，是"在变好还是变坏" |
| 评分 | 一段条 + 一个等级，量的是"在什么水平" | 不是百分比，是等级划分 |

**组件选择逻辑**：

| 组件 | 为什么选它 | 什么时候不选它 |
|------|----------|--------------|
| **gauge-arc** | SVG 弧形仪表，量的视觉冲击最强 | 不需要视觉冲击 → block-bar |
| **gauge-segmented** | 分段仪表，量的是"在哪个等级" | 不需要等级划分 → gauge-arc |
| **bullet-chart** | 实际值 vs 目标 vs 范围带，量的是"对比基准" | 没有基准目标 → gauge-arc |
| **sparkline** | 迷你趋势线，量的是"变化趋势" | 需要看当前值好坏 → gauge-arc |
| **block-bar** | 段状评分，量的是"在什么水平" | 需要精确百分比 → progress-row |
| **progress-row** | 进度条 + 数值，量的是"达成率" | 不是达成率 → block-bar |

## 6. 解（解读·READ）

**用户在做什么**：理解一段文字/数据的含义——为什么这么说、背后的逻辑是什么。

**数据形态判断**：解的特征是**有论证结构**——观点 + 依据、结论 + 原因。

| 数据形态 | 为什么是这个形态 | 为什么不是其他 |
|---------|---------------|--------------|
| 边注 + 引用 | 边注=重点标注，引用=权威来源，解的意图最直接 | 不是度量（不是数字），不是对比（不是两个对象） |
| 标注 + 清单 | 标注=逐段解读，清单=该做什么，解的是"怎么做" | 不是"为什么这么说"，是"具体怎么做" |

**组件选择逻辑**：

| 组件 | 为什么选它 | 什么时候不选它 |
|------|----------|--------------|
| **callout** | 边注框，解的最常用形态——一个框标注重点 | 需要引用原文 → pull-quote |
| **pull-quote** | 引用块，解的是"权威说了什么" | 不是引用 → callout |
| **blockquote** | 内联引言，解的粒度最细——嵌入文字流 | 需要独立展示 → callout |
| **brand-statement** | 全宽宣言卡，解的是"品牌立场" | 不是品牌立场 → callout |
| **annotated-highlight** | 标注高亮，解的是"图表/图片里的重点" | 不是图表解读 → callout |
| **action-list** | 动作清单，解的是"该做什么" | 不是行动建议 → callout |

## 7. 信（背书·PROVE）

**用户在做什么**：确认信息有依据——数据从哪来、谁说的、凭什么信。

**数据形态判断**：信的特征是**有来源标注**——Logo、出处、代码，不是内容本身，是内容的"凭证"。

| 数据形态 | 为什么是这个形态 | 为什么不是其他 |
|---------|---------------|--------------|
| 来源 + Logo | Logo=谁在用，来源=数据出处，信的意图最直接 | 不是解读（不是论证），是"凭证展示" |
| 数字即证据 | 数字本身就是证据，不需要额外来源 | 不是"大数字概览"，是"用数据证明" |
| 代码即证据 | 代码本身就是证据，技术场景的背书 | 不是代码展示，是"证明这件事能跑" |

**组件选择逻辑**：

| 组件 | 为什么选它 | 什么时候不选它 |
|------|----------|--------------|
| **logo-wall** | Logo 墙，信的视觉冲击最强——一堆 logo 比一段文字更可信 | 不是合作方背书 → source-row |
| **source-row** | 数据出处行，信的精度最高——精确到来源+日期 | 不是数据来源 → tag |
| **tag** | 标签，信的粒度最细——标注来源/分类 | 不是来源标注 → callout |
| **stat-block** | 数字即证据，用数据证明 | 不是用数据证明 → logo-wall |
| **code-panel** | 代码即证据，技术场景 | 不是技术场景 → stat-block |

## 8. 展（深入·DIG）

**用户在做什么**：想看更多——点一下展开、切换视图、看详情。

**数据形态判断**：展的特征是**有交互**——点击/悬停/切换，信息分层揭示。

| 数据形态 | 为什么是这个形态 | 为什么不是其他 |
|---------|---------------|--------------|
| 折叠/切换 | 点一下展开/收起，展的意图最直接 | 不是静态展示（需要交互），不是概览（需要深入） |
| 清单/跟踪 | 勾选=完成，清单=任务，展的是"进度" | 不是"展开详情"，是"跟踪状态" |

**组件选择逻辑**：

| 组件 | 为什么选它 | 什么时候不选它 |
|------|----------|--------------|
| **accordion** | 折叠面板，展的最常用形态——FAQ、多层详情 | 需要并排切换 → tab-switcher |
| **tab-switcher** | 选项卡，展的是"同一空间多视图" | 不需要多视图 → accordion |
| **flip-card** | 翻转卡片，展的是"正反两面" | 不是正反 → accordion |
| **hover-reveal** | 悬停揭示，展的是"渐进式信息密度" | 不是渐进 → detail-panel |
| **detail-panel** | 全屏详情面板，展的是"完整信息" | 不需要全屏 → accordion |
| **interactive-checklist** | 交互清单，展的是"任务跟踪" | 不是任务 → accordion |
| **segmented-control** | 分段选择器，展的是"切换数据视图" | 不是数据视图 → tab-switcher |

## 9. 感（氛围·FEEL）

**用户在做什么**：不判断，只感受。建立品牌印象、空间节奏、视觉冲击。

**数据形态判断**：感的特征是**不传递具体信息，传递情绪和气质**。

| 数据形态 | 为什么是这个形态 | 为什么不是其他 |
|---------|---------------|--------------|
| Hero（开场） | 全屏、大字号、视觉冲击，感的意图最直接 | 不是内容区（不需要阅读），是"第一印象" |
| 装饰元素 | 幽灵文字、点阵、纹理，感的是"空间节奏" | 不是 Hero（没有冲击力），是"氛围填充" |
| 品牌系统 | 色板、字体、标志，感的是"品牌 identity" | 不是装饰（有系统性），是"品牌体系展示" |

**组件选择逻辑**：

| 组件 | 为什么选它 | 什么时候不选它 |
|------|----------|--------------|
| **V1-V6 Hero** | 6 种切割变体，感的冲击力最强——首屏决定气质 | 不是首屏位置 → 装饰元素 |
| **Marquee** | 品牌宣言 Hero，感的是"品牌态度" | 不是宣言 → V1-V6 |
| **Split Diptych** | 产品+风景对屏，感的是"产品与自然的关系" | 不是产品 → V1-V6 |
| **Photographic** | 全屏风光，感的是"自然气质" | 不是自然场景 → Marquee |
| **ghost-text** | 幽灵水印，感的是"空间深度" | 不是空间感 → dot-pattern |
| **seal** | 印章，感的是"仪式感" | 不是仪式感 → ghost-text |
| **dot-pattern** | 点阵背景，感的是"科技感/网格感" | 不是科技感 → texture |
| **dot-matrix** | 前景点阵字符，感的是"技术与手工并存" | 不是技术感 → ghost-text |
| **color-swatch** | 单个色板，感的是"颜色本身" | 需要展示色板体系 → color-swatch-grid |
| **typography-showcase** | 字体展示，感的是"字体性格" | 不是字体 → color-swatch |
| **brand-signature** | 品牌签名收尾，感的是"结束感" | 不是收尾 → Hero |

## 判断链总结

```
内容进来
  ↓
判断：用户执行什么认知操作？（9 选 1-3）
  ↓
判断：数据是什么形态？（每种操作有 2-3 种形态）
  ↓
选择：这个形态下哪个组件最合适？（读"为什么选它"列）
  ↓
读组件代码 → 写
```

**每一步都是判断，不是搜索。** AI 不需要浏览 160 个组件，只需要：
1. 判断认知操作（1 步）
2. 判断数据形态（1 步）
3. 在 3-5 个候选组件中选最合适的（1 步）

3 步完成，不需要翻文件。
