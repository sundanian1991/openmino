*本文由 Discovery 团队研究员 Siddharth Mishra-Sharma 撰写，解释了如何将多日智能体编程工作流——测试预言、持久记忆和编排模式——应用于科学计算任务，即使超出自身领域也适用。*

目前大多数使用 AI 智能体的科学家都在对话式循环中工作，紧盯着流程中的每一步。随着过去一年左右模型在[长周期任务](https://metr.org/time-horizons/)上的能力显著提升，一种新的工作方式出现了：不再介入每个细节，我们可以指定高层次目标，然后释放一组智能体自主工作。这使得在数小时内完成原本需要数天、数周甚至数月的工作成为可能。某些类型的科学任务非常适合这种模式，例如：重新实现数值求解器、将用旧版 Fortran 方言编写的遗留科学软件转换为现代语言，以及对照参考实现调试大型代码库。这些任务工作范围明确、成功标准清晰，人工监督可以间歇性而非持续。

Anthropic 的 [C 编译器项目](https://www.anthropic.com/engineering/building-c-compiler)展示了这种模式的一个版本，Claude 通过大约 2,000 个会话构建了一个能够编译 Linux 内核的 C 编译器。本文描述了如何针对典型的学术实验室，使用 Claude Code 为科学计算任务搭建类似的模式。作为具体示例，我将演示如何使用 Claude Opus 4.6 [实现一个可微分的宇宙学玻尔兹曼求解器](https://github.com/smsharma/clax)。这是一个预测大爆炸余辉——宇宙微波背景辐射（CMB）统计性质的数值代码，通过演化早期宇宙中光子、重子、中微子和暗物质的耦合方程组来实现。

类似 [CLASS](http://class-code.net/) 和 [CAMB](https://camb.info/) 这样的玻尔兹曼求解器是宇宙学的核心科学基础设施，使我们能够利用 *Planck* 和 *Simons Observatory* 等巡天项目的数据约束宇宙学模型。可微分版本——能够在整个求解器中传播梯度——使得基于梯度的推断方法成为可能，极大地加速了参数估计。用 JAX 来写非常自然，因为它几乎免费提供自动微分和加速器兼容性（例如 GPU）。

值得注意的是，这个任务不属于我的核心科学领域——我对相关工具和科学有高层次了解，但不具备在任何合理时间框架内自己完成的专业能力。那些*确实*具备这种专业能力的团队已经用 JAX 构建了[可微分](https://arxiv.org/abs/2311.03291)[求解器](https://arxiv.org/abs/2602.15104)，实现了 CLASS 的部分功能。这些工作通常代表数月至数年的研究人员时间。这里的重点在于，看看智能体能否在非领域专家的最小指导下走得更远。

这类任务在结构上与 C 编译器项目不同，后者可以分配给大量并行智能体。与此相反，玻尔兹曼求解器是一个深度耦合的流水线——一个微小的数值误差或对早期宇宙复合过程建模的不良近似，都可能在所有下游产生微妙偏移。因此，它需要一套不同的智能体技能。调试需要沿整个因果链条追溯并运用领域知识，这可能更适合单个智能体顺序工作，按需派生子智能体，并利用参考实现来二分定位差异。

我们将使用运行 SLURM 作业调度器的高性能计算集群作为计算环境，但核心思想——进度文件、测试预言、包含明确规则的智能体提示——无论你在哪里运行 Claude Code 都适用。

在向管理自主研究智能体团队的转变中，你应该花大部分时间（与 Claude 协作）来制定一组清晰表达项目交付物和相关背景的指令。这些指令应放在根目录下的 CLAUDE.md 文件中。Claude 会将此文件特殊对待，保持在上下文中并在整体计划中参考它。关键是，Claude 可以在工作过程中编辑这些指令，在遇到问题时为后续工作更新它们。

[这里](https://github.com/smsharma/clax/blob/6a6b2330cf25edded1bb31ec57a0091aa794a5d3/CLAUDE.md)是宇宙学玻尔兹曼求解器项目早期的 CLAUDE.md，展示了在初次尝试编写求解器后编纂的总体计划和设计决策。为了达成这个文件，我指定了项目的高层目标——实现对参考 CLASS 实现的完全功能对等，同时完全可微分，并在主要科学交付物上达到与 CLASS 相比 0.1% 的精度目标——并与 Claude 反复迭代，直到计划看起来满意。鉴于 0.1% 是两个经典玻尔兹曼代码 CLASS 和 CAMB 之间的一致程度，这似乎是一个合理的科学目标。

进度文件（按照约定我们在此称为 CHANGELOG.md）是智能体的便携式长期记忆，充当实验室笔记的角色。在 CLAUDE.md 中，Claude 被指示在此文件中跟踪进度。

一个好的进度文件可能跟踪当前状态、已完成任务、失败的方法及其失败原因、关键检查点的精度表以及已知局限。失败的方法很重要——没有它们，后续会话会重新尝试相同的死胡同。一条记录可能类似："尝试用 Tsit5 求解扰动常微分方程，系统过于刚性。改用 Kvaerno5。"[这里](https://github.com/smsharma/clax/blob/main/CHANGELOG.md)是运行示例的更新日志，展示了这些要素。

虽然通过智能体进行更开放的科学研究无疑已在地平线上，但今天长期运行为自主科学工作关键取决于智能体是否有办法知道它是否在取得进展。对于科学代码，这可以是参考实现、明确可量化的目标或已有的测试套件。指示智能体扩展测试套件并在工作时运行测试，以防止回归，也很有帮助。在我的示例任务中，Claude 被指示使用 [CLASS C 源码](https://github.com/lesgourg/class_public)作为参考实现来构建并持续运行单元测试。

Git 是监控和协调智能体进展的一种好方式，且不干扰工作。智能体应在每个有意义的工作单元后提交并推送。这为你提供了可恢复的历史记录以防出错，使进展在本地可见，并防止工作在计算配额中途耗尽时丢失。

在实践中，这可以是 CLAUDE.md 中的一组指令，例如："在每个有意义的工作单元后提交并推送。每次提交前运行 `pytest tests/ -x -q`。绝不提交会破坏现有通过测试的代码。"

为了引导智能体，你可以随时通过 SSH 进入集群并手动重新提示和/或更新其指令。更便捷的做法通常是用本地的 Claude Code 实例通过 SSH 进入并为你运行命令；这也适用于下文所描述的每项操作。

如上所述，通常有用的做法是在本地首先迭代计划，直到有一个看起来合理且编纂在 CLAUDE.md 中的版本。然后，在计算节点上的终端多路复用器（如 tmux）内启动 Claude Code 会话，告诉智能体在哪里找到你的代码库，然后让它工作。因为会话运行在 tmux 内，你可以分离、合上笔记本电脑，偶尔查看进展（在玻尔兹曼求解器这个例子中，我是在排队买咖啡时通过手机查看 GitHub）。

在高性能计算集群上，你可以通过 SLURM 调度器请求一个节点，一个在 tmux 会话中启动 Claude Code 的示例作业脚本大致如下：

```
#!/bin/bash
#SBATCH --job-name=claude-agent
#SBATCH --partition=GPU-shared
#SBATCH --gres=gpu:h100-32:1
#SBATCH --time=48:00:00
#SBATCH --output=agent_%j.log
cd $PROJECT/my-solver
source .venv/bin/activate
export TERM=xterm-256color
tmux new-session -d -s claude "claude; exec bash"
tmux wait-for claude
```

作业启动后，你连接到 tmux 会话，给 Claude Code 指示（例如，"阅读 CHANGELOG.md，然后继续下一个任务"），当认为它走在正确轨道上时分离。你可以随时重新连接以检查进度、引导或启动新任务，使用类似：

`srun --jobid=JOBID --overlap --pty tmux attach -t claude`

**Ralph 循环：** 随着模型能力越来越强，它们需要的定制编排（如提示工程、RAG 或上下文填充）会减少。但在特定时间点，提供一定程度的脚手架作为能力提升也可能有用。例如，当前模型可能会出现*智能体惰性*——当被要求完成一个复杂的多部分任务时，它们有时会找到借口在完成整个任务之前停下来（"时间不早了，明天再继续？"）。

为了避免这个问题，一个有用的编排模式是 [Ralph 循环](https://ghuntley.com/loop/)，它是一个简洁的循环控制机制。

Ralph 可以通过 /plugin 安装。Claude Code 中典型的调用提示可能如下：

`/ralph-loop:ralph-loop "请继续执行任务，直到在整个参数范围内达到 0.1% 精度的成功标准。" --max-iterations 20 --completion-promise "DONE"`

这里，Claude 将迭代最多 20 次，直到用 "DONE" 咒语确保任务完成。

Claude 在几天内从零开始构建了[该项目](https://github.com/smsharma/clax)，到结束时其各项输出与参考 CLASS 实现达成了亚百分比的一致性。我让 Claude 重构了代码主要输出——各个 CMB 角功率谱——在整个项目过程中的精度演变，同时标注了开发过程中的里程碑。它生成了下图，展示了达到亚百分比精度的路径。

智能体的开发轨迹有些笨拙。例如，它的测试覆盖率存在明显漏洞——有一段时间它只在一个（基准）参数点上测试代码，极大地减少了发现 bug 的表面积。它也会犯低级错误，比如搞混规范约定，或者花数小时追逐一个宇宙学家一眼就能看出的 bug，但它持续朝着亚百分比精度的既定目标前进。

这个项目的一个附带好处是，我通过观察 git 提交历史学到了大量关于玻尔兹曼求解器及其建模的物理知识。这个项目并非出自我核心的科学领域，但跟踪 Claude 的逐步进展并查阅我不认识的内容，被证明是渗透学习科学知识的有效方式。[提交日志](https://github.com/smsharma/clax/commits/main/)读起来像一个高效、极度按字面执行工作的博士后的实验笔记。

虽然最终得到的求解器并非生产级产品（例如，它在每个工况下都无法以可接受的精度匹配参考 CLASS 实现），但它证明了智能体驱动开发可以将数月至数年的研究人员工作压缩到数天。

这种压缩改变了什么是"闲置时间"的定义。AI 研究的一个普遍体验是，晚上启动一个实验（例如一次训练运行），然后早上看到结果时感到满足。不运行实验意味着机会成本。如今，不运行智能体也感觉有同样机会成本。如果你有计算资源，有定义明确的成功标准的项目，每一个你*没有*智能体为你工作的夜晚，都是留在桌面上的潜在进展。
