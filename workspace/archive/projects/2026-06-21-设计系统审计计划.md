## 二、组件完整度对比

### 2.1 nian-design（最完整）

**56 个权威组件**，分 7 类：

| 类 | 数量 | 包含 | 优先级 |
|----|:---:|------|:------:|
| A · 基础容器 | 8 | Card, List, Table, Bento Grid, Three-Column, Detail Panel, Flip Card, Stacked Cards | P0-P2 |
| B · 数据与进度 | 10 | Segmented Progress, Mini Bar, Sparkline, Gauge, Dot Grid, Quarter Bar, Pyramid, Radial, Waffle, Bullet | P0-P2 |
| C · 流程与导航 | 7 | Flow Pipeline, Vertical Timeline, Tree, Navigation, Tag, Segmented Control, Date Nav | P0-P2 |
| D · 对比与交互 | 10 | Do/Don't, Asymmetric Table, Tab Panel, Accordion, Checklist, Button, Input, Toggle, Widget KPI, Overlay | P0-P2 |
| E · 状态与反馈 | 3 | State(三态合并), Toast, Decorative Number | P1-P2 |
| F · 内容排版 | 14 | Prose, Article List, Quote(4变体), Code Panel, Tension Grid, Cluster Grid, Newspaper, Dot Table, Numbered Grid, Dark Quote, Magazine Cards, Compare Table, Pull Quote, Split Q&A | P0-P1 |
| G · 内容工具 | 5 | Chat, Filter Bar, H-Scroll, Calendar, Carousel | P1-P2 |

**已吸收**：nothing 的 26 组件全部纳入（标注 [N]），mino 的 8 个内容组件已吸收进 F/G 类。

### 2.2 nothing-design（精炼，但风格独特）

26 个组件 families，与 nian 的 A-D 类高度重叠（1-26 编号）。但 nothing 独特之处在于：
- **纯单色系统**（无 7 色）
- **字体不同**：Space Grotesk + Space Mono + Doto（nian 是 Georgia + Inter + JetBrains Mono）
- **无变体系统**，无 base templates
- 部分组件有 nothing 专属视觉细节（如点阵背景、工业感边框）

### 2.3 mino-design-system（内容型组件丰富）

45 个组件/变体，独特价值在于：
- 大量**内容排版组件**（引用块 5 变体、代码面板 4 变体、对话气泡、书卡等）
- **杂志/手绘风**变体（报纸多栏、明信片邮票风、胶片字幕风）
- **个人品牌**导向（头像集群、技能卡片、情绪日历等）
- nian 已经吸收了其中 8 个进 F/G 类

### 2.4 aham-ui（概念级，尚未实现为技能）

~65 组件（8 大类），核心创新：
- **机读优先**：tokens.json 是唯一事实源
- **线性执行流程**（无分支决策）
- **三层 Token 体系**：Primitive → Semantic → Component
- **铁律驱动**：值只从 tokens.json 取

**状态**：目前只有 2 个研究性 HTML 文档，没有 SKILL.md、没有可执行工作流。

## 三、质量对比

### 质量维度评分（1-5）

| 维度 | nian | nothing | mino | aham |
|------|:----:|:-------:|:----:|:----:|
| **组件实现完整度** | 5 | 3 | 4 | 1（概念） |
| **代码质量/一致性** | 5 | 4 | 3 | 2 |
| **Token 系统严谨性** | 5 | 4 | 3 | 3 |
| **亮暗双模式** | 5（完整） | 4（完整） | 2（部分） | 1 |
| **响应式** | 4（clamp） | 3 | 3 | 1 |
| **变体/模板丰富度** | 5（24 变体） | 1（无变体） | 3（4 模板） | 1 |
| **文档清晰度** | 5 | 4 | 3 | 2 |
| **可维护性** | 4（单一真相源） | 3 | 2 | 3（概念） |
| **生产验证** | 5（26+ 输出） | 3 | 3 | 1（无产出） |

**nian 的质量优势**：
- 单一 `:root` 真相源，所有产物内联引用
- 5 条硬规则校验 → 脚本自动验证
- 变体系统（24 个高质起点）+ 组件系统（56 个原子零件）+ showcase 案例库
- 已吸收 nothing 和 mino 的最佳实现
- 亮暗双模式每组件内联（不是全局覆写）
- 有 CI 级别的校验脚本

**nothing 的质量优势**：
- 设计哲学更纯粹（减法原则，工业感）
- 字体纪律最严格（3 尺寸 2 字重限制）
- 文档质量高，但组件实现不如 nian 完整

**mino 的质量问题**：
- Token 引用不规范（部分组件硬编码色值）
- 丰富度有余，一致性不足
- 部分组件缺少亮暗模式

## 四、气质对比

### nian-design：建筑感 · 块面感 · 结构化

> "Landscape and signal. 页面是一片风景，引导目光到信号。"
> "结构即是装饰。不靠阴影、渐变、毛玻璃。"

**关键信号**：
- Georgia（默认衬线）+ Inter（无衬线）+ JetBrains Mono（数据）
- 7 色体系（自然低饱和主色 + 高亮 accent + 场景色）
- 三层金字塔（Answer → Argument → Evidence）
- 8:1 底线 + 一处打破
- 圆形半径小（2-6px），无多余装饰

### nothing-design：工业单色 · 瑞士排版 · 减法

> "Subtract, don't add. Monochrome is the canvas."
> "Structure is ornament. Expose the grid."

**关键信号**：
- Space Grotesk + Space Mono + Doto（点阵感）
- 严格单色（白色为事件，不是默认）
- 三种工具体系（日式机械感）
- 无蓝色链接（Links 用 underline 替代颜色）
- 所有标签 ALL CAPS 大写 + 大间距

### mino-design-system：个人 · 温暖 · 手工

**关键信号**：
- Tiempos（衬线）+ Styrene（无衬线）+ Fira Code（等宽）
- 杂志排版感，大量引用块和装饰元素
- 手绘元素（虚线框、手写批注、用章边框）
- 更像"内容排版系统"而不是"UI 设计系统"
- 无统一 token 约束，风格偏自由

### aham-ui：AI 宪法 · 机读 · 铁律

**关键信号**：
- Inter Display + Inter + JetBrains Mono（类似 nian）
- 蓝色 accent（#336EE8），与 nian 的橄榄色不同
- "值只从 tokens.json 取"是核心铁律
- 7 层规则，线性执行，无分支决策
- Blue accent（科技感）vs nian 的 natural/outdoor 气质

### 气质对立关系

```
nian（建筑·自然） ←→ nothing（工业·机械）
                   ↕
       aham（科技·AI）   mino（手工·温暖）
```

nian 和 nothing 在某些设计原则上相通（结构即装饰、三层级、信息密度），但色彩体系和字体选择的方向完全不同。mino 和 aham 在两个极端：手工 vs 机读。

nian 的 COMPONENTS-MASTER.md 已经做了三体系合并：
- **体系 A**（nian 原始 32-34 族）→ 全部保留
- **体系 B**（nothing 26 + nian 独特 + mino 10 = 54 件）→ 否，同类择优

**aham 未吸收**：三层 Token 体系（Primitive→Semantic→Component）、线性执行流程、机读优先设计。

## 六、本地保留方案

### 核心判断

**nian-design 是主系统，应该作为默认设计系统保留和继续发展。** 理由：
1. 组件最完整（56 个，已吸收 nothing+mino 精华）
2. 质量最高（单一真相源、硬规则校验、26+ 真实产出）
3. 工业化程度最高（变体系统、自动化回溯、CI 脚本）
4. 正在活跃迭代（CHANGELOG 持续更新）

### 保留策略

#### 方案一：三系统并存（推荐初始状态）

| 技能 | 角色 | 触发条件 |
|------|------|---------|
| **nian-design** | 主设计系统 | 默认施工 |
| **nothing-design** | 风格参考/评审工具 | 用户指定 "nothing 风格" |
| **mino-design-system** | 个人品牌/内容工具 | 个人品牌相关页面 |
| **aham-ui** | 研究参考（不保留为独立技能） | 设计系统架构参考 |

**不建议合并为一个系统**。三个系统的气质差异太大，合并会导致：
- 选择混乱（每次要问用户走哪条路）
- 文件膨胀（全部载入 prompt 预算过高）
- 设计一致性下降（同一个系统内出现 Georgia 和 Space Grotesk 是矛盾的）

**保持独立比合并更好。**

#### 方案二：选择性裁剪（优化 prompt 预算）

nian 当前 538 个文件，考虑裁剪策略：

**必须保留的路径**：
- `SKILL.md`（~18KB）
- `references/COMPONENTS-MASTER.md`（1864 行，核心组件库）
- `references/templates/base/token-root.css`（唯一 token 真相源）
- `references/templates/variants/*.html`（24 个变体）
- `references/CRAFT-RULES.md`（硬规则）
- `references/philosophy.md`（设计哲学）
- `references/READING-ORDER.md`（加载策略）
- `references/EXTENSIONS-charts.md`（图表嵌入契约）
- `references/EXTENSIONS-interactive.md`（交互组件契约）

**可归档到 _archive 的路径**：
- `references/components.md`（已弃用，COMPONENTS-MASTER 替代）
- `references/components-core.md`（已弃用）
- `references/components-extended.md`（已弃用）
- `references/tokens.md`（已迁移到 token-root.css）
- `tokens/_archived-*` 旧 token 文件
- `templates/_deprecated/` 废弃模板
- `references/showcase-archive/`（灵感库，按需查，不自动加载）

#### 方案三：研究吸收 aham 的理念

aham 的核心价值不在组件而在架构理念：
1. **三层 Token 体系**：nian 目前只有一层 `:root` token。考虑引入 Primitive (raw values) → Semantic (intent-based) → Component (scope-specific) 分层
2. **线性执行流程**：nian 的 Step 2 有 2A/2B/2C/2D 四条分支路，aham 全部降为线性。考虑简化
3. **铁律优先**：aham 的"值只从 tokens.json 取"是核心防 AI 自由发挥手段。nian 可以在 SKILL.md 中强化类似规则

**但不生成独立 aham 技能**——这些理念直接吸收进 nian 的下一个版本。

## 八、最终建议

**简单版本**：保持三系统并存。nian 做主施工，nothing 做风格参考，mino 做个人品牌。aham 作为理念参考吸收进 nian。

**激进版本**：让 nian 成为唯一的前端设计技能。将 nothing 和 mino 的未吸收价值（nothing 的字体纪律、mino 的手绘变体）作为 nian 的 EXTENSIONS 并入。废弃 nothing 和 mino 独立技能。

**我推荐简单版本**——分开比合并干净。三系统各有清晰边界和触发条件，不需要用户每次决策。prompt 预算也不是问题（SKILL.md 在触发时才加载，不冲突）。
