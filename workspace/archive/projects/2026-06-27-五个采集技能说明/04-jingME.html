组合层 · 承诺追踪

jingME

编排 jdme-local-tools（京ME消息）与 read-joymeeting（慧记会议）两个只读数据源，产出一张可追溯的跟进事项表——谁答应做什么、什么时候交付、来源证据、当前状态、建议提醒文案。

版本 0.1.0 (local-draft) 状态 本地草稿 性质 只读编排 依赖 jdme-local-tools + read-joymeeting

五个采集技能里的第一个组合层。它本身不直接读数据，而是调度下面两个采集技能，把 IM 消息和会议逐字稿汇到一起，专门抽取"承诺/交付"信号——这是 personal-insight-summary 做日报级汇总、jingME 做事项级追踪的分工。

02安全边界

只读硬约束
  • 不直接发京ME消息
  • 不修改 JD ME 聊天数据库或日志
  • 不注入 JD ME 客户端或转储进程内存
  • 不提取凭据、cookie 或私钥
  • 不把未验证的推断当作事实呈现
用户要发消息时 生成草稿文案，并明确告知必须手动发送（除非存在官方授权的发送工具）。

03工作流（五步）

从澄清范围到产出跟进表。

STEP 1 · 澄清范围

问最小必要信息

缺失时问：人 ERP/姓名、群名或 session id、日期范围、关键词、是否含会议记录。默认范围：昨天=上一日；本周=周一到今天；最近=7 天。

STEP 2 · 采集京ME证据

jdme-local-tools 调度

已知 ERP 时先试私聊 session id 模式 <other>:ee:<self>:ee。未知时用 chat-search --auto --keyword <erp>。按时间戳在 Python 里过滤；原始 JSON 只放临时文件；用户没明确要求时不把私聊导出存进项目。

STEP 3 · 采集慧记证据

read-joymeeting 调度

取范围内会议列表，只对与主题相关的会议取详情。用 title / members / transcript / attachments。transcript 缺失时不编造结论。

STEP 4 · 抽取承诺

承诺线索识别与分类

见下一节的线索词表。每条分类：负责人、事项、截止、来源、证据引用、状态、建议提醒文案。

STEP 5 · 产出

结论 + 跟进表 + 提醒文案

用简洁中文。敏感内容用摘要而非原始逐字稿。

04承诺线索词表

三类线索，从 IM 与会议全文中匹配承诺信号。

① 负责人线索（谁承诺）

我来 我跟 我负责 我推进 我今天弄 已收到 会同步

② 截止线索（何时交付）

今天 明天 周五 下周 7月 上线前 明天发你 下周给

③ 交付线索（交付什么）

发你看 给谁过目 宣讲 提报 排期 上线 确认 补充数据
分类字段取值
负责人姓名 / ERP
事项承诺要做的具体任务
截止时间点或"待补"
来源京ME / 慧记
证据原话引用（短）
状态promised / done / blocked / unclear
建议提醒文案给该负责人的礼貌催办话术

05输出格式

结论 + 跟进表 + 建议提醒文案，简洁中文。

结论段

本次共识别 N 个跟进事项，其中 A 个有明确负责人， B 个有明确时间点，C 个需要补确认。
