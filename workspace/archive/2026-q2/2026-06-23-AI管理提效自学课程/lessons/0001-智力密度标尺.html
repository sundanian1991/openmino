第 1 课 · 标尺层（贯穿全课程的尺子）

智力密度：抛弃"用得多"的直觉

这是整门课的标尺。后面六层每一层的好坏，都用它来衡量。

课程：AI 管理与提效 · 第 1/7 层　|　预计 12 分钟　|　使命 · 资源

为什么先学这个

你已经有了强直觉："思维链应依据内容动态生成，而非套固定流程模板"。这节课把这个直觉升级成一把可量化的尺子，并给你一个让前沿研究背书的答案：到底怎么判断"用更多的 token"是值得的。

不立这把尺子，后面六层每一层都会陷入"好像有用但说不清值不值"的泥潭。先立尺子，每一层才有评判依据。

核心命题 Raw token 数是推理质量的劣质代理。 前沿共识正从"堆 token"转向"每个 token 承载了多少有效判断"——也就是智力密度。

三个前沿发现，同一个结论

过去一年的研究从三个独立方向收敛到同一个洞察：

研究一句话发现对你的意义
OckBench
arXiv 2511.05722		 提出 Per-Token Intelligence：好模型不只准，还要每个 token 都准。 建立"每个 token 的智力密度"作为标尺。
Densing Law
Nature Machine Intelligence		 模型正变得更聪明/单位算力——capability density 在涨。 别只看模型绝对能力，看性价比。
Deep-Thinking Tokens
Google Research		 "深度思考 token" 占比比总量更能预测准确率。 你的直觉是对的：该多想时多想，不该想时别空转。

三个研究用了不同的词（per-token intelligence / capability density / deep-thinking tokens），但指向同一件事：密度比总量重要。

你的直觉，已被前沿背书 你 06-04 写下的"思维链应依据内容动态生成，而非套固定流程模板"，正是 Deep-Thinking Tokens 的实操版。这不是巧合——你在一线踩出来的直觉，和实验室用数据推出来的结论是一致的。

把直觉变成可操作的标尺

智力密度可以拆成两个正交维度，组成一张象限图：

　高密度（每个 token 都在判断）低密度（token 在空转）
多 token ✓ 理想区：深度推理，
该花的钱花到位		 ✗ 最差：烧钱没产出，
典型"堆 token"陷阱
少 token ✓ 经济区：精准直击，
简单任务该有的样子		 ⚠ 危险区：看起来省，
实则判断不足

关键洞察："用多少 token" 不是判断标准，"密度高不高"才是。 同样的钱，高密度多 token（深度推理）和 高密度少 token（精准直击）都对；唯一要避开的是低密度多 token（烧钱空转）。

本课的标尺（请记住）

问自己：这个动作是在堆 token，还是在提升每个 token 的判断密度？

用你自己的数据验证

你 06-04 记录过 token 消耗量级：轻量日几百~几千、产研日 30-80 万。用标尺重审这两个数字：

  • 轻量日几百~千 token：如果这些 token 每个都在承担有效判断（检索、决策、产出）——高密度，值得。
  • 产研日 30-80 万 token：如果有相当比例是重复读同样的 context、空转的思维链、没被接住的工具调用——低密度，是优化靶点。

这也是为什么后面的 Harness 层（工具调用修复）和 Loop 层（回路设计）对你的 ROI 最高：它们直接把"低密度空转"转成"高密度判断"。

本课练习 · 5 分钟

拿出你最近一次产研日的 AI 协作（任何一个都行），用标尺给它打分：

  1. 那次协作总共烧了多少 token？（粗估量级即可）
  2. 其中有多少比例是"空转"？（重复读 context、失败重试、套模板的思维链）
  3. 如果要把空转比例砍一半，该动哪一层？（Context？Harness？Loop？）

不用写下来，脑子里过一遍即可。目的不是精确计算，是建立"用标尺看自己的协作"的习惯。

小测一下

检查理解

下面哪个动作，最可能是"低密度多 token"陷阱？

  • 让模型对复杂方案做多轮深度推理
  • 每次对话都把全部历史重新塞进 context
  • 给一个判断题用 test-time compute 多想一轮
  • 让 subagent 并行扇出探索多个方向
再想想。提示：哪个动作会重复消耗 token 却不增加新判断？
✓ 对。"把全部历史重新塞进 context"是典型的低密度空转——同样的信息反复读，不产生新判断。这正是后面 Context 层要解决的核心问题。

这一层为后面六层定了什么规矩

记住这把尺子，后面每层的好坏都用它量：

层用智力密度怎么量
Context窗口里每段信息都在为当前任务贡献判断密度吗？
Prompt这段 prompt 降低了多少理解损耗（= 提升单位 token 产出）？
Harness工具调用修复 / 校验，减少了多少低密度重试？
Loop每一圈是在堆 token，还是在提升判断密度？
Skill复用一次资产，省下了多少低密度重复劳动？
Eval评估本身是不是高密度（少 token 出准确判断）？
Cost花在刀刃上：哪步该用强模型，哪步该省？
