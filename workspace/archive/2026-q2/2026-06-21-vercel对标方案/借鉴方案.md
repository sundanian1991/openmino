# Vercel 三文件借鉴方案 — 落到 nian-design

> 基于 Vercel AGENTS / DESIGN / command 三文件分析，提炼可借鉴模式，映射到 nian-design 现有架构。
> 不是照搬，是把 Vercel 做对的事填进 nian 的 5 层体系。

---

## 核心判断

Vercel 做对了三件事，nian-design 缺了对应的三块：

| Vercel 做对的事 | nian-design 现状 | 缺口 |
|---|---|---|
| **三级约束**（MUST/SHOULD/NEVER） | 二元（P0 红线 / P1-P4 推荐） | AI 执行时无法区分"违反就挂"和"违反可接受" |
| **Token frontmatter**（DESIGN 文件头部 token 声明块） | tokens.css 有值但组件引用松散 | 组件定义和底层值的绑定关系不显式 |
| **规则→审核 command 闭环** | validate-nian-deck.mjs 有但不成体系；无 Claude Code slash command | 规则写给人看，不是写给 AI 消费的 |

---

## 方案一：CRAFT-RULES 升级为三级约束

### 现状

CRAFT-RULES.md 5 条硬规则 + checklist.md P0-P4 分级。P0 是红线，P1 是必查，P2-P4 是推荐。但执行时只有"通过/不通过"。

### 改法

在 CRAFT-RULES.md 每条规则前加**强制力标记**，与 checklist 的 P0-P4 对齐：

```
[MUST]   = P0 红线，违反不输出
[SHOULD] = P1-P2，违反标注但可输出（附理由）
[NEVER]  = P0 红线的反面表述，绝对禁止
```

#### 具体改写示例

**规则 1 · Three-Layer**

```markdown
## 规则 1 · Three-Layer（三层金字塔）

> [MUST] 每个 `<section>` 恰好三层重要性。不多不少。

| 层级 | 角色 | 字体 | 字号 | 位置 |
|---|---|---|---|---|
| **Primary** | 唯一结论 | Playfair Display | 36-96px | 视觉中心 |
| **Secondary** | 支撑上下文 | Inter | 14-18px | 紧贴 Primary |
| **Tertiary** | 元数据 / 来源 | JetBrains Mono | 10-12px | 边缘 / 底部 |

### 强制力

- [MUST] 眯眼测试通过——缩到 20% 能一眼看出最重要的东西
- [NEVER] 两个元素同字号竞争视觉中心
- [SHOULD] Primary 字号 ≥ Secondary 的 2 倍（差距越大，层级越清晰）
```

**规则 4 · One-Break**

```markdown
## 规则 4 · One-Break（一处打破）

> [MUST] 每屏恰好 1 处不守规则。

### 允许的打破形态（SHOULD 优先级排序）

1. [SHOULD] Ghost 大字（Playfair italic，opacity 0.03-0.06，推荐 0.045）
2. [SHOULD] 出血图片（1 张，跨过 section 边界）
3. [SHOULD] 异色强调（--yellow 或 --orange 一处使用）
4. [SHOULD] 异形元素（圆/斜/异尺寸）

### 强制力

- [MUST] 打破数量 = 1
- [NEVER] 0 处打破（"没看头"）→ 不通过
- [NEVER] ≥ 2 处打破（"失控"）→ 不通过
```

### 改动范围

| 文件 | 改动 |
|---|---|
| `references/CRAFT-RULES.md` | 每条规则加 `[MUST]`/`[SHOULD]`/`[NEVER]` 标记 |
| `references/checklist.md` | P0 项加 `[NEVER]`，P1 加 `[MUST]`，P2 加 `[SHOULD]`，P3-P4 不加 |
| `SKILL.md` Step 4 | 自检流程说明更新，解释三级含义 |

### 验证方式

- 改完后 grep `CRAFT-RULES.md` 确认每条规则都有强制力标记
- 改完后 grep `checklist.md` 确认 P0 项全部带 `[NEVER]`

### 不改什么

- 不改规则内容本身（三层金字塔、非对称等语义不变）
- 不改 P0-P4 分级逻辑（只是给每个分级贴上 MUST/SHOULD/NEVER 标签）
- 不改 checklist 的检查方式（grep/目测不变）

## 方案二：组件 Token 引用规范化

### 现状

components.md 的 CSS 示例用 `var(--surface)`、`var(--border)` 等引用 tokens，但有三个问题：

1. **色值表重复**：组件头部有 Light/Dark 两张手写色值表（约 40 行），与 colors.css 完全重复
2. **radius token 缺口**：组件定义卡片用 6px、按钮用 4px，但 spacing.css 里 `--radius-sm: 4px` / `--radius-md: 8px`，没有 6px 的语义 token
3. **无依赖声明**：AI 不知道组装某个组件需要哪些 token

### 改法

#### 2a. spacing.css 补充语义 radius token

在 `tokens/spacing.css` 的 Border Radius 区块末尾追加语义别名：

```css
/* ─── Semantic Radius Aliases（组件语义映射）───────────── */
--radius-card:     var(--radius-sm);    /* 4px → 组件卡片（原 6px，向 4px 对齐，统一规范） */
--radius-btn:      var(--radius-sm);    /* 4px → 按钮 */
--radius-technical: var(--radius-sm);   /* 4px → 技术元素（代码块、标签） */
```

> **决策点**：组件定义里卡片是 6px，但 6px 在 token 体系里没有对应值。两个选择：
> - **A. 加 `--radius-card: 6px`**：保留原视觉效果，但引入非标准值（不是 4/8/12 的倍数）
> - **B. 统一到 `--radius-sm: 4px`**：干净，但卡片视觉微调（从 6px → 4px）
>
> **建议选 B**。6px 和 4px 的视觉差异极小，统一到 4px 让 token 体系更干净。如果年老师坚持 6px，再加 `--radius-card: 6px`。

#### 2b. 组件头部加 token 依赖声明

每个组件族开头加一行：

```markdown
## 01. CARDS / SURFACES（卡片）

> **Token 依赖**: `--surface`, `--border`, `--border-strong`, `--text-disabled`, `--font-data`, `--radius-card`
```

这行告诉 AI：组装这个组件时，必须确保这些 token 已在 `:root` 中定义。

#### 2c. 删除重复色值表

components.md 头部的 Light Mode / Dark Mode 色值表（约 40 行）与 colors.css 完全重复。改为：

```markdown
## 色彩系统

> 色板**唯一信源**为 `tokens/colors.css`。组件 CSS 中用 `var(--xxx)` 引用，不硬编码值。
> 深色模式值（-d 后缀）已在 colors.css 的 `.dark / [data-mode="dark"]` 块中定义，组件加 `.dark` 类即生效。
> 完整色值查阅 → `tokens/colors.css`
```

删掉两张手写色值表。AI 生成时直接读 colors.css，不需要在 components.md 里再看一遍。

#### 2d. CSS 示例中用 token 替换硬编码值

当前示例（01. CARDS 的 CSS）：
```css
.card {
  background: var(--surface);
  border: 1px solid var(--border);
  border-radius: 6px;
  padding: 20px;
  transition: border-color 150ms ease-out;
}
.card__tag {
  font-family: var(--font-data);
  font-size: 9px;
  letter-spacing: 0.08em;
  text-transform: uppercase;
  color: var(--text-disabled);
  margin-bottom: 4px;
}
.card__num {
  font-family: var(--font-data);
  font-size: 32px;
  font-weight: 500;
```

改为：
```css
.card {
  background: var(--surface);
  border: 1px solid var(--border);
  border-radius: var(--radius-card);
  padding: var(--space-lg);
  transition: border-color 150ms ease-out;
}
.card:hover { border-color: var(--border-strong); }
.card__tag {
  font-family: var(--font-data);
  font-size: var(--text-xs);
  letter-spacing: 0.08em;
  text-transform: uppercase;
  color: var(--text-disabled);
  margin-bottom: var(--space-xs);
}
.card__num {
  font-family: var(--font-data);
  font-size: var(--text-display);
  font-weight: 500;
```

**完整映射表**（基于 tokens/spacing.css 实际存在的 token）：

| 硬编码值 | Token 替换 | 依据 |
|---|---|---|
| `border-radius: 6px` | `var(--radius-card)` | 新增语义别名，指向 `--radius-sm` |
| `border-radius: 4px` | `var(--radius-btn)` | 新增语义别名，指向 `--radius-sm` |
| `border-radius: 999px` | `var(--radius-full)` | 已存在 |
| `padding: 20px` | `var(--space-lg)` | 已存在（24px，微调） |
| `padding: 12px` | `var(--space-md)` | 已存在（16px，微调） |
| `padding: 8px` | `var(--space-sm)` | 已存在 |
| `margin-bottom: 4px` | `var(--space-xs)` | 已存在 |
| `gap: 16px` | `var(--space-md)` | 已存在 |
| `font-size: 9px` | `var(--text-xs)` | 需确认 typography.css 是否有 |
| `font-size: 12px` | `var(--text-sm)` | 需确认 typography.css 是否有 |
| `font-size: 14px` | `var(--text-base)` | 需确认 typography.css 是否有 |
| `font-size: 32px` | `var(--text-display)` | 需确认 typography.css 是否有 |

> **注意**：`padding: 20px` → `var(--space-lg)` 是 24px，差 4px。如果视觉敏感，可在 spacing.css 加 `--space-lg-card: 20px`。但建议先统一到 24px，看效果再决定。

### 改动范围

| 文件 | 改动 |
|---|---|
| `tokens/spacing.css` | 追加 3 个语义 radius 别名（`--radius-card` / `--radius-btn` / `--radius-technical`） |
| `references/components.md` | 头部色值表删除 → 改为一行引用；每个组件族加 token 依赖声明；CSS 示例中硬编码值替换为 token |

### 验证方式

- 改完后 grep components.md 确认无硬编码 hex（`#[0-9a-fA-F]{6}` 除非在注释中）
- 改完后 grep components.md 确认无硬编码 `border-radius: Npx`（全部走 `var(--radius-xxx)`）
- 改完后逐个检查 38 个组件族都有 Token 依赖声明

### 不改什么

- 不改组件的 HTML 结构
- 不改组件的视觉效果（token 替换后值尽量一致，允许 ±4px 微调）
- 不改组件的数量和分类
- 不改 `--radius-sm: 4px` 等现有 token 的值

## 方案三：`/nian-review` 审核命令

### 现状

- `validate-nian-deck.mjs`：Node.js 脚本，检查 data-layout、Hero 比例、禁用 CSS、颜色 token 等。能跑，但需要手动 `node scripts/xxx.mjs path/to/file.html`
- `validate-nian-html.mjs`：另一个校验脚本
- **没有 Claude Code slash command**：AI 生成 HTML 后，没有一个"一键审核"的入口

### 改法

创建 `.claude/commands/nian-review.md`，仿照 Vercel 的 command.md 结构。

```markdown
---
description: Review HTML for nian-design compliance
argument-hint: <file-or-pattern>
---

# nian-design Review

Review these files for compliance: $ARGUMENTS

Read files, check against rules below. Output concise—sacrifice grammar for brevity. High signal-to-noise.

## Rules

### [MUST] 三层金字塔 (Rule 1)

- 每个 `<section>` 恰好三层：Primary (Playfair 36-96px) / Secondary (Inter 14-18px) / Tertiary (Mono 10-12px)
- 眯眼测试：缩到 20% 能否一眼看出最重要的东西
- [NEVER] 两个元素同字号竞争视觉中心

### [MUST] 字体预算 (Rule 2)

- 字号层级 ≤ 3（display/body/meta）
- 字重 ≤ 2（400 + 600）
- 字体家族 ≤ 2（Playfair + Inter/Mono）
- [NEVER] Doto 出现在非 Hero/ghost 位置
- [NEVER] Playfair Display 出现在 `<p>` / body 正文中

### [MUST] 非对称 (Rule 3)

- [NEVER] `text-align: center` 在 section 级别
- [NEVER] `place-items: center` 在 section 级别
- [SHOULD] 主元素 7/12 + 辅元素 5/12

### [MUST] 一处打破 (Rule 4)

- [MUST] 恰好 1 处打破（ghost 水印 / 出血图 / 异色 / 异形）
- [NEVER] 0 处打破
- [NEVER] ≥ 2 处打破

### [MUST] 视觉多样化 (Rule 5)

- [MUST] ≥ 3 数据段时用 ≥ 2 种视觉形态

### [MUST] Token 合规

- [NEVER] 硬编码 hex 颜色（`#[0-9a-fA-F]{6}` 除非在 `var(--` 内）
- [NEVER] 硬编码 font-family（必须用 `var(--font-display)` / `var(--font-body)` / `var(--font-data)`）
- [MUST] 所有 section 有 `data-layout="S01"`~`"S28"`

### [NEVER] 禁止项

- 渐变（`linear-gradient`）
- 阴影（`box-shadow`）
- 模糊（`blur` / `backdrop-filter`）
- 圆角 > 8px（pill 999px 除外）
- emoji
- 斑马纹
- 骨架屏
- 视差 / 滚动劫持 / 弹跳缓动

### [SHOULD] 推荐项

- 深浅交替 ≤ 2 个连续浅色
- 间距优先于分割线
- 灰度先于颜色建立层级
- 图片容器直角无阴影
- 同族卡片统一样式（不混用 surface/cream）

## Output Format

Group by file. Use `file:line` format (VS Code clickable). Terse findings.

```text
## path/to/file.html

path/to/file.html:42 - [MUST] section 缺少 data-layout
path/to/file.html:67 - [NEVER] 硬编码 hex #2C2C2C → 改用 var(--text-display)
path/to/file.html:89 - [MUST] 两个同字号元素竞争视觉中心

## path/to/file2.html

path/to/file2.html:12 - [NEVER] Playfair Display 出现在 <p> 中
✓ pass
```

State issue + location. Skip explanation unless fix non-obvious. No preamble.
```

### 改动范围

| 文件 | 改动 |
|---|---|
| `.claude/commands/nian-review.md` | **新建**，仿 Vercel command.md 结构 |
| `SKILL.md` Step 5 | 输出后追加提示："可用 `/nian-review path/to/output.html` 做合规审核" |

### 验证方式

- 创建后立即跑一次：`/nian-review output/supplier-health-v4.html`，看输出是否合理
- 对比 validate-nian-deck.mjs 的检查项，确认 command 覆盖了核心规则

### 不改什么

- 不改 validate-nian-deck.mjs（脚本仍可用，command 是另一条路径）
- 不改校验逻辑（command 复用 CRAFT-RULES + checklist 的规则）

## 方案四：Do's & Don'ts 记忆锚点

### 现状

CRAFT-RULES.md 结尾有"与 Nothing Design 的对应"表，但没有精炼的 Do/Don't 锚点。checklist.md 有 P0-P4 但太长，AI 记不住。

### 改法

在 CRAFT-RULES.md 末尾（自检流程之后、与 Nothing 对应之前）加一个 **Do's & Don'ts** 块：

```markdown
## Do's & Don'ts（记忆锚点）

> 读完上面的规则，记住这 7+7 条就够了。

### Do

1. 用 token 引用颜色，不硬编码 hex
2. 每个 section 三层分明——眯眼看能分辨 Primary
3. 非对称布局——7/5 或 2/3+1/3
4. 恰好一处打破——Ghost 水印最省事
5. 数据段 ≥ 3 时换形态——大数字 + 进度条 + sparkline
6. 深浅交替——不连续超过 2 个浅色 section
7. Playfair 只做标题，Inter 做正文，Mono 做标签

### Don't

1. 不用渐变、阴影、模糊、毛玻璃
2. 不居中——`text-align: center` 是 AI 感的标志
3. 不堆字号——3 层级够了，层内靠间距和颜色区分
4. 不发明新组件——从 components.md 38 族里选
5. 不跳过 Step 4 自检——5 条全过才能输出
6. 不在同一页用两种场景色
7. 不用 Doto 做正文——Doto 只在 Hero/ghost/点阵中出现
```

### 改动范围

| 文件 | 改动 |
|---|---|
| `references/CRAFT-RULES.md` | 自检流程后、Nothing 对应表前，加 Do's & Don'ts 块 |

### 验证方式

- 加完后读一遍，确认 7+7 条每条都能对应到 CRAFT-RULES 的某条规则

## 实施优先级

| 优先级 | 方案 | 工作量 | 收益 | 前置依赖 |
|---|---|---|---|---|
| **P0** | 方案三：`/nian-review` 命令 | 0.5 天 | 最高——AI 生成后一键审核，闭环 | 方案一（命令引用三级标记） |
| **P1** | 方案一：三级约束标记 | 0.5 天 | 高——AI 执行时能分优先级 | 无 |
| **P2** | 方案四：Do's & Don'ts 锚点 | 0.5 小时 | 中——精炼记忆锚点，长期收益 | 方案一（锚点内容来自三级标记） |
| **P3** | 方案二：组件 token 引用规范化 | 1-2 天 | 中——工程规范提升，但需改 38 个组件的 CSS 示例 | 无 |

**建议执行顺序**：P1 → P2 → P0 → P3

先做方案一（三级约束），因为它是方案二（command）和方案四（锚点）的基础——command 里的 `[MUST]`/`[NEVER]` 标记需要先在 CRAFT-RULES 里定义好。然后方案四顺手加。方案三（command）在方案一完成后立刻做。方案二（组件 token 规范化）工作量最大，可以分批做——先改前 5 个核心组件族（Cards / Buttons / Inputs / Lists / Tables），验证流程通了再铺开。

---

## 不做的事

- **不改 CRAFT-RULES 的规则语义**——只加标记，不改"三层金字塔"、"非对称"等核心概念
- **不改 checklist 的 P0-P4 分级逻辑**——只是给每个分级贴 MUST/SHOULD/NEVER 标签
- **不删 validate-nian-deck.mjs**——command 和脚本是两条路径，互补不互斥
- **不改组件的 HTML 结构**——只改 CSS 示例中的值引用方式
- **不引入新 token 文件**——在现有 6 个 tokens/*.css 内补充 radius 等缺失 token
