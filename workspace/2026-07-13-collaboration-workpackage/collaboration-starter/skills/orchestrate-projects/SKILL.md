# Project Orchestrator

协调长时间运行项目的系统化方法。适用于项目可能涉及新发现、多项独立工作、或需要来自多个环境的证据时。

核心循环：对话 → 路线图 → 激活里程碑 → 委派工作 → 验证结果 → 更新路线图 → 下一个里程碑。

---

## 1. 初始规划：采访式启动

不要从计划模式开始。以对话方式启动，采访用户直到有足够上下文。

**采访维度**（按顺序问，一次问 2-3 个，不要一次全抛）：

### 第一轮：现状 + 动机
- 当前系统/流程的规模和状态
- 最核心的痛点是什么
- 触发事件（故障、事故、增长瓶颈）

### 第二轮：团队 + 目标态
- 团队分工和能力，决策者是谁
- 目标架构的初步设想
- 参考案例

### 第三轮：约束
- 不可中断的时间窗口
- 预算和人力硬约束
- 上层期望和容忍度

**输出**：初始计划草稿 + GOALS.md（用 `references/goals-template.md` 模板）

---

## 2. 路线图：GOALS.md

创建 `GOALS.md` 作为共享路线图。围绕里程碑组织，而非扁平任务列表。

**每个里程碑记录**：
```markdown
## Milestone X: [名称]

### 预期结果
[完成时应该是什么样子]

### 当前范围内的工作
- [ ] 任务 1
- [ ] 任务 2

### 重要决策
- 决策 1：[原因]

### 已知阻塞项
- 阻塞 1：[当前状态]

### 继续前进前所需的证据
- [ ] 证据 1：[如何验证]
```

**维护规则**：
- 主代理随项目发展维护这个文件
- 新发现可能改变里程碑范围、引入新里程碑、或改变完成所需的证据
- 同一时间只有一个活跃目标处于激活状态

---

## 3. 目标模式：单一焦点

一次设定并维护一个活跃目标。活跃目标对应当前里程碑。

**设置方式**：
- 用户自行设定（如果运行环境支持目标/任务模式命令）
- 或让主代理来设定（推荐）

**完成流程**：
1. 里程碑完成
2. 主代理更新 GOALS.md
3. 激活下一个目标
4. 发送系统通知提醒用户

---

## 4. 协调模式：主线程聚焦

主线程专注于：
- 目标和约束
- 决策和项目状态
- 委派有边界的任务
- 评估结果

主线程不需要承载每个实现细节。

**Worker 返回格式**（参考 `references/worker-report-template.md`）：
```
## 完成内容
[做了什么]

## 关键发现
[学到了什么、什么发生了变化]

## 证据
[支撑发现的数据/截图/日志]

## 下一步建议
[基于发现应该做什么]
```

---

## 5. 委派策略

### 并行模式（独立任务同时发）

```
主线程同时启动：
  worker A: 实现登录模块
  worker B: 实现支付模块
  worker C: 编写测试用例

主线程继续协调，不等结果
worker A/B/C 完成时通知主线程
主线程合并结果，更新 GOALS.md
```

**启动方式**：在一条消息中发起多个子代理调用，不要分开发。

### 串行模式（有依赖关系）

```
主线程启动 worker A: 实现登录模块
等待 worker A 返回结果
基于结果启动 worker B: 实现基于登录的权限检查
等待 worker B 返回结果
基于结果启动 worker C: 集成测试
```

**启动方式**：等待前一个子代理完成，再启动下一个。

### 选择依据

| 场景 | 委派方式 | 原因 |
|------|----------|------|
| 独立可并行的任务 | 多个子代理同时发 | 不阻塞主线程 |
| 串行依赖（改完再测） | 同步单个子代理 | 需要前一步结果 |
| 可能以后会重新访问的工作 | 独立线程 | 保留历史，易于检查 |
| 需要本地环境的测试 | 本地线程 | 浏览器、凭证、设备 |
| 单文件读取、简单搜索 | 主代理直做 | 不绕路 |

**硬约束**：
- 子代理 prompt 必须是自包含任务描述
- 禁止套娃：子代理不再派子代理
- Worker 返回结论和证据，而非完整记录

## 6. 里程碑审计

每个里程碑完成后，在激活下一个目标之前（参考 `references/audit-checklist.md`）：

**审计检查清单**：
1. 这个里程碑真的完成了吗？
2. 下一个目标还是对的吗？
3. 有没有遗漏的里程碑？
4. 新证据是否改变了工作顺序？
5. "完成"的定义还成立吗？

**运行代码审查**：
- 在实现上运行代码审查
- 检查测试覆盖
- 验证边界情况

**审计输出**：更新 GOALS.md，反映当前真实状态。

---

## 7. 本地线程：跨环境验证

对于依赖本地环境的测试，使用本地线程：

**适用场景**：
- 已登录的浏览器
- 本地凭证
- 设备模拟器（移动端、硬件等）
- 操作系统权限
- 桌面应用

**工作循环**：
```
远程 worker 实现功能
    ↓
远程测试通过
    ↓
本地线程执行本地环境测试
    ↓
发现问题 → 远程 worker 修复
    ↓
本地线程重复检查
    ↓
通过 → 继续下一个里程碑
```

**断线重连**：
- 本地线程断开（合上笔记本等）→ 远程 worker 继续工作
- 重新打开笔记本 → 检查本地线程是否重新可用
- 如果不可用 → 创建新的本地线程，检出最新代码继续测试

**项目默认保持远程状态**。本地机器只在需要时才加入。

## 8. 进度报告

### 简洁更新格式
项目状态变化时，只报告三部分：

```
### 已完成
- [具体的完成项]

### 下一步
- [接下来要做的]

### 阻塞项
- [需要用户协助的]
```

### 系统通知
里程碑完成或遇到阻塞时，发送系统通知提醒用户。

**macOS 示例**：
```bash
osascript -e 'display notification "Milestone 1 完成，准备启动 Milestone 2" with title "项目进展"'
```

**Linux 示例**（需安装 `libnotify`）：
```bash
notify-send "项目进展" "Milestone 1 完成，准备启动 Milestone 2"
```

### 进度仪表盘

对于多里程碑或并行工作流的项目，维护 `progress-dashboard.html`（用 `references/progress-dashboard-template.html` 模板）：

**显示内容**：
- 活跃目标
- 已完成的里程碑
- 证据链接
- 阻塞项
- 决策记录
- 最近更新

**查看方式**：本地打开 `progress-dashboard.html`（macOS: `open progress-dashboard.html`，Linux: `xdg-open progress-dashboard.html`）

**优势**：几小时后回来，无需阅读每个 worker 线程就能理解项目状态。

## 9. Side 线程：不打断的检查

如果你的运行环境支持多线程/side 线程，可使用它来查看情况，而不打断主线程：

**操作流程**：
1. 从 side 线程获取摘要："当前完成了什么？"
2. 让 side 线程把指令传达给主线程："帮我告诉主线程，优先处理 X"
3. 澄清指令、解决阻塞、补充上下文

**适用场景**：
- 回到项目时快速了解状态
- 需要手动点击、审批等操作
- 检查 worker 是否重新可用
- 不想打断主线程正在做的事

**示例对话**：
```
用户（side 线程）：主线程现在在做什么？
Side 线程：主线程正在等 worker 完成登录模块实现。
用户（side 线程）：告诉主线程，先处理支付模块，登录可以之后再做。
Side 线程：已传达。主线程会调整优先级。
```

> 注：如果你的环境不支持多线程，可以在主线程对话中直接询问"当前状态如何？"，效果类似。

## 10. 计划修订

当新发现改变项目方向时，标准操作：

**触发条件**：
- 实现过程中发现原方案不可行
- 技术选型需要调整
- 外部依赖发生变化
- 用户需求变更

**修订流程**：
1. 暂停当前目标
2. 在 GOALS.md 的"变更记录"中记录原因
3. 重新评估里程碑顺序
4. 更新受影响的里程碑范围
5. 激活新的目标（或恢复原目标）

**不要默默修改**：每次修订都要在 GOALS.md 中留下记录，说明为什么改。

---

## 11. 完成标准

**不要在约定的证据出现之前宣布完成**。

每个里程碑的完成需要：
- 预期结果已实现
- 所需证据已收集
- 代码审查通过
- 测试通过（如有）

---

## 快速启动模板

把以下内容粘贴到一个新的代理线程中：

```
把这作为长时间项目进行协调。

首先采访我，了解我想构建什么。让我以对话方式描述粗略的想法和约束，然后把那次讨论变成初始计划。

创建 GOALS.md 作为共享路线图。围绕里程碑组织，每个包含预期结果、范围、决策、阻塞项和所需证据。

一次设定并维护一个活跃目标。活跃目标应该对应当前里程碑。

让这个线程聚焦于目标、约束、决策和项目状态。把实现委派给子代理或新线程。

要求 worker 返回结论、变化和证据，而非完整记录。

当我可能想检查、继续或审查工作时，使用独立线程。对于不需要自己可见历史的有边界工作，使用子代理。

每个里程碑后，用一个独立线程把 GOALS.md 与当前状态对比审计，并在实现上运行代码审查。在激活下一个目标之前更新路线图。

对于依赖我的浏览器、桌面应用、权限、凭证或设备状态的测试，使用本地线程。

当项目状态变化时，只报告：

已完成
下一步
阻塞项

里程碑完成时发送系统通知提醒我。当项目跨越多个里程碑或 worker 时，维护 progress-dashboard.html。

在约定的证据出现之前，不要宣布完成。
```
