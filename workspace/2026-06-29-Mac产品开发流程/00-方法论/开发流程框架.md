# Mac 桌面产品开发流程框架（AI 时代版）

> 适用场景：个人开发者用 AI 辅助构建 Mac 桌面产品
> 创建日期：2026-06-29
> 核心理念：**判断力 > 计划，验证 > 文档，速度 > 完美**

---

## 设计原则

1. **探索先于计划**：不知道要什么就别急着做，先低成本试
2. **验证先于构建**：写代码之前先证明方向对
3. **AI 做执行，你做决策**：代码生成不是瓶颈，判断力才是
4. **迭代是常态**：没有完美第一版，只有越来越好的版本

---

## 核心循环

```
    ┌─────────────────────────────────────┐
    │                                     │
    ▼                                     │
 Explore ──→ Validate ──→ Build ──→ Iterate
   探索         验证       构建      迭代
   │                         │         │
   │    "这是个好问题吗？"    │         │
   │    "方向对不对？"        │         │
   │                         │         │
   └─────────────────────────┘         │
        用真实反馈驱动                   │
        └───────────────────────────────┘
```

## Phase 1: Explore（探索）

**目标**：搞清楚问题是什么，找到值得做的事

**时间预算**：1-3 天（不是 1-2 周）

**做什么**：

1. **写一页纸的问题陈述**
   - 我要解决什么问题？
   - 谁会遇到这个问题？
   - 他们现在怎么解决？
   - 现有方案哪里不好？

2. **做 3 个"试一下"的原型**
   - 每个原型只验证一个核心假设
   - 用最快的方式做：HTML 页面、命令行工具、甚至手绘草图
   - 不需要能跑，能说清楚"大概是这么回事"就行

3. **列出你已知的和未知的**
   - 已知：我已经知道用户需要 X
   - 未知：我不知道用户会不会用 Y
   - 猜测：我猜 Z 会是关键功能（待验证）

**产出**：
- 一页纸问题陈述（`01-探索/problem-statement.md`）
- 3 个原型的截图/描述（`01-探索/prototypes/`）
- 已知/未知/猜测清单

**门**：能不能用一句话说清楚"我要做什么，为谁做，为什么"？说不清楚就继续探索。

---

## Phase 2: Validate（验证）

**目标**：用最低成本证明方向对不对

**时间预算**：1-3 天（不是 1-2 周）

**做什么**：

1. **选一个核心假设，做最小验证**
   - 从 Phase 1 的 3 个原型中选最有感觉的一个
   - 做成能交互的原型（不是静态页面，是要能点的）
   - 自己用 10 分钟，模拟真实场景

2. **问自己 3 个问题**
   - 这个东西解决了我真正的问题吗？
   - 我会每天都用它吗？
   - 用它的时候我爽吗？

3. **决定：继续 / 调整 / 放弃**
   - 继续：方向对，进入构建阶段
   - 调整：核心想法对但细节不对，回 Explore 调整
   - 放弃：方向不对，重新探索

**产出**：
- 验证记录（`02-验证/validation-log.md`）：试了什么、结果如何、结论是什么
- 选定的方向（一句话描述）

**门**：能不能回答"我确定这个方向值得投入 2-4 周做出来"？回答不了就别进构建。

---

## Phase 3: Build（构建）

**目标**：把验证过的方向做成可运行的产品

**时间预算**：1-4 周（按迭代拆分）

**关键区别**：不是一次性做完，而是**每个迭代只做一件事**。

### 迭代结构

```
迭代 0：骨架（1-2天）
  → 能启动、能跑通主路径、UI 可以丑
  → 目标：证明技术方案可行

迭代 1：核心功能（3-5天）
  → 把最有价值的那个功能做完
  → 目标：自己能用

迭代 2：体验打磨（2-3天）
  → UI 细节、快捷键、错误处理
  → 目标：用起来顺手

迭代 3+：按需扩展
  → 根据实际使用决定加什么
  → 目标：解决真实遇到的问题
```

### 每个迭代的节奏

```
早上：想清楚今天要做什么（5分钟）
  ↓
白天：让 AI 帮你写代码，你做决策
  ↓
晚上：自己用一遍，记下哪里不对
  ↓
第二天：先修昨晚发现的问题
```

**产出**：
- 可运行的产品（每个迭代结束时）
- 迭代日志（`03-构建/iteration-N.md`）：做了什么、遇到什么问题、怎么解决的
- bug/体验问题清单（`03-构建/issues.md`）

**门**：每个迭代结束时问——"如果明天就要给朋友用，我敢不敢？"不敢就继续打磨。

---

## Phase 4: Iterate（迭代）

**目标**：用真实反馈驱动产品演进

**时间预算**：持续（不是一次性阶段）

**做什么**：

1. **自己用一周**
   - 每天都用，当正常工具用
   - 记录：哪里卡、哪里爽、哪里想要但没有

2. **给 1-3 个朋友试用**
   - 不要解释怎么用，看他们能不能自己搞懂
   - 观察他们在哪里卡住
   - 问："你会用这个吗？为什么？"

3. **根据反馈决定下一步**
   - 高频痛点 → 立即修
   - 偶发问题 → 记录，攒一批再修
   - "要是有 XX 功能就好了" → 先验证这个需求是否真实

**产出**：
- 使用日志（`04-迭代/usage-log.md`）
- 反馈收集（`04-迭代/feedback.md`）
- 下一步计划（`04-迭代/next.md`）

## 关于传统文档

PRD、架构文档、UI 规范不是不要了，而是**变成活文档**：

| 传统做法 | AI 时代做法 |
|----------|------------|
| 先写完整 PRD | 先写一页纸问题陈述，边做边补 |
| 先画完整架构图 | 先搭能跑的骨架，架构自然浮现 |
| 先定完整 UI 规范 | 先做能用的页面，审美边做边统一 |
| 先写测试用例 | 先自己用，发现问题再写测试 |

**原则**：文档服务于决策，不是决策服务于文档。如果一个文档不能帮你想清楚下一步做什么，就别写。

## 工具链建议

| 阶段 | 工具 | 说明 |
|------|------|------|
| 探索 | Markdown + HTML 原型 | 轻量，够用 |
| 验证 | 可交互原型（Figma/Excalidraw/HTML） | 能点就行 |
| 构建 | VS Code + Claude Code | AI 辅助写代码，你做决策 |
| 迭代 | 手动测试 + 使用日志 | 个人项目不需要复杂工具 |

## 快速检查清单

在每个阶段结束时问自己：

**Explore 结束**：
- [ ] 我能用一句话说清楚要做什么吗？
- [ ] 我做了至少 1 个原型吗？
- [ ] 我知道自己的未知项吗？

**Validate 结束**：
- [ ] 我自己试用过原型吗？
- [ ] 我能回答"为什么这个方向值得做"吗？
- [ ] 我决定了继续/调整/放弃吗？

**Build 每个迭代结束**：
- [ ] 产品能跑吗？
- [ ] 我自己用了吗？
- [ ] 我知道下一步要做什么吗？

**Iterate 结束**：
- [ ] 我用了至少一周吗？
- [ ] 我收集了至少 1 个外部反馈吗？
- [ ] 我知道下一步要修什么吗？
