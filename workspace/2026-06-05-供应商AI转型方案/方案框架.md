# 供应商管理 AI 转型方案框架

> 电销 AI 转型全员执行方案 · 供应商管理部分
> 范围：32+ 家供应商，3200+ 人，4 大产线（金条/企金/财富/信用卡）、17 条细分业务
> 初期金条 12 家试跑
> 2026-06-06（v2 — 整合 AI 提效岗位分析）

---

## 一、定位

本方案是"电销 AI 转型全员执行方案"中供应商管理板块的完整框架。

供应商管理岗的 AI 转型目标：**将日常运营操作→异常判断→管理决策的全链路系统化，构建一个可被动咨询、可主动监控、可群聊协作的供应商管理数字人，释放管理精力到策略和商务层面。**

月度工作总量约 196 小时，AI 可提效约 83.5 小时（43%），释放时间投向供应商寻源、站点布局、BPO 联盟、深度关系建设。

### 3.2 日常工作（每天循环）

| # | 工作内容 | 日均耗时 | AI 提效方案 | AI 不能做的 |

## 四、全生命周期：引入 → 管理 → 清退

### 4.1 引入阶段

#### 流程

```
供应商提交资料 → 资料初评 → 商务交叉验证 → 准入评估报告 → 你确认 → 建档入库 → 分初始量
```

准入评估 9 维度：过往业绩、同类型对标、管理方式、人员数量、公司背景、历史交付风格、配合商务、团队架构、资质合规。

审批周期：之前 30 天，现在 15 天，AI 辅助后目标压缩到 5-7 天。

#### 技能化

| Skill | 功能 | 输入 | 输出 |
|-------|------|------|------|
| `/供应商资料初评` | 按标准维度对提交材料打分 | 供应商提交的材料包 | 各维度分数 + 缺漏清单 + 建议补件 |
| `/商务交叉验证` | 查询资质状态/涉诉/关联方/异常名录 | 供应商名称 / 社会信用代码 | 商务风险报告 |
| `/评分体系完整性检查` | 检核当前评估维度的覆盖度 | 评估维度表 | 覆盖度分析 + 缺失项提示 |
| `/准入评估报告` | 整合初评+交叉验证输出结论 | 初评分数 + 验证结果 | 准入/待考察/拒绝 + 理由 |

**复用说明**：年老师之前做过供应商资料初评的项目，评估标准和评分逻辑可直接复用。

#### 知识库需求

- 评估标准说明（各维度评分细则）
- 历史准入案例（什么情况拒了、什么情况过了、原因）
- 储备池供应商信息（保持 10-15 家）

---

### 4.2 管理阶段

管理阶段的核心是**日常监控 + 周期性考核 + 动态调整 + 合规管理**。

#### 日常监控维度

| 维度 | 数据来源 | 监控频率 | 技能化 |
|------|---------|---------|--------|
| 业绩表现 | 数仓（data-agent） | 日 | `/业绩异常诊断` `/日简报` |
| 配合表现 | 群聊 + 人工反馈（joyclaw） | 周 | 规则判定 |
| 人员变动 | 数据表 + 群聊 | 周 | `/人员异动跟踪` |
| 品质合规 | 质检系统（待重构） | 月（后置） | 对接后自动化 |
| 月度考核 | 综合评分 | 月 | `/月度考核评分` |
| 合规问题 | 客诉 + 内部反馈 + 巡检 | 持续 | `/合规问题处理` |

#### 考核评分（5 维打分）

月度考核是管理阶段的核心输出，支撑后续的分量调整和清退决策。

**评分模型（已确认的权重）：**

| 维度 | 权重 | 指标 | 数据来源 |
|------|------|------|---------|
| 业绩 | 30% | 达标率、完成率、增长率 | 数仓 |
| 质检客诉 | 25% | 质检通过率、客诉次数/率 | 质检系统 |
| 上人 | 15% | 人员到位率、出勤率、稳定性 | 数据表 |
| 配合 | 15% | 响应时效、整改效率、商务支持 | 群聊+人工 |
| 战略 | 15% | 与业务方向匹配度、发展潜力 | 你判断 |

各维度加权 → 月度总分 → ABCD 评级 → 红黄牌判定

#### 合规管理

合规问题包含：客诉、招聘违规（用京东 logo）、坐席不当行为等。

处理流程：判断严重程度（轻微→口头警告；严重→发邮件）→ 要求 3-7 天整改 → 跟踪闭环 → 从个案中提炼可制度化的规则。

#### 巡检管理

出差巡检 12 项标准：
1. 空间布置 2. 人员状态 3. 薪资情况 4. 招聘情况 5. 品牌使用 6. 管理水平
7. 活动激励 8. 作息时间 9. 培训方式 10. 招聘方式 11. 品牌滥用 12. 质检合规

巡检流程：出发前准备 → 现场检查（参考历史问题）→ 记录 → 巡检报告 → 整改跟踪

### 4.3 清退阶段

#### 流程

```
红黄牌累积 / 触发清退条件 → 清退评估 → 切量方案 → 退出执行 → 档案归档 + 复盘
```

#### 技能化

| Skill | 功能 | 输出 |
|-------|------|------|
| `/清退评估` | 综合考核/配合/品质等多维数据评估清退必要性 | 清退建议 + 理由 + 风险提示 |

## 五、分量管理（核心管理动作）

分量是贯穿全生命周期的管理动作——引入分初始量，管理做增减，清退需切量。

### 5.1 三种量

| 类型 | 场景 | 频率 |
|------|------|------|
| **增量** | 有新需求/新业务线，分配给谁、分多少 | 按需 |
| **减量** | 某供应商表现下滑，减少其业务量 | 月度/按需 |
| **切量** | 清退/替换时，量从 A 转移到 B | 清退时 |

日常管理中"增量和减量"是最常说的口径。

### 5.2 分量算法（决策模型）

```
分量建议 = f(业绩30%, 质检客诉25%, 上人15%, 配合15%, 战略15%)
```

结合供应商标签（业务类型/能力/配合度/稳定性/潜力），匹配需求预测 → 建议分量范围。

动态调整原则：不都给、不一直不给，让供应商保持期待。

**输出形态**：系统给出建议分量表 + 依据说明 → 你确认或微调 → 执行。

### 5.3 分量调整触发机制

- **例行调整**：每月考核后，基于 5 维评分+产能滚动计算下月分量
- **事件触发**：异常告警 → 临时减量；新需求 → 增量分配
- **战略调整**：商务谈判需要、新供应商试跑期、清退切量

### 5.4 分量与全生命周期的关系

```
引入 → 分初始量（试跑期保护量）
  ↓
管理 → 月度调量（增量/减量，基于 5 维考核+产能）
  ↓         ↕
      异常事件 → 临时减量
  ↓
清退 → 切量（量转移给其他供应商）
```

## 六、数据底座

### 6.1 数据分类与路径

#### 结果数据（数仓有表，维度丰富）

- 内容：业绩数据、达标率、排名、订单量
- 现状：数仓有表，之前手工做表成本高，只实现了核心一小部分
- 方案：**data-agent** 数据工作流平台 → 自动搭建看板 + 定时输出
- 自动化程度：高，可全自动

#### 过程数据（群聊+人工反馈+商务沟通）

- 内容：配合表现、响应情况、沟通记录、异常反馈
- 现状：群多、语义歧义大、不稳定
- 方案：**joyclaw**（OpenClaw 京东版）→ 群聊接入 + 推送 + 记录
- 自动化程度：中，依赖人工校验兜底
- 关键挑战：聊天歧义判定，需要规则匹配 + 人工确认

#### 质检合规数据

- 现状：延迟后置，现有规范和指标需要重构
- 方案：先对齐"系统指标"再接入，暂不纳入 Phase1

### 6.2 口径问题（前置条件）

**人员数据存在多个口径，需先行拉齐，否则自动分析会失真：**

| 口径 | 使用场景 | 差异原因 |
|------|---------|---------|
| 数据人力 | 产能观察 | 按系统记录的在岗人数 |
| 出勤人力 | 日常出勤 | 按实际打卡人数 |
| 排班人力 | 排班管理 | 按排班表人数 |
| 结算人力 | 结算逻辑 | 按结算规则人数 |

**建议**：Phase1 先确定"产能观察口径"作为管理统一口径，各场景的转换规则写清楚。

## 七、规则引擎

### 7.1 分层设计

| 层级 | 内容 | 变动频率 | 谁改 |
|------|------|----------|------|
| 固定层 | 考核指标体系、5 维评分算法、权重（30/25/15/15/15） | 年度/半年度 | 你+领导确认 |
| 配置层 | 告警阈值、红黄牌条件、异常定义 | 按需调整 | 你自己改 |
| 动态层 | SPC 控制基线（基于历史自计算） | 自动调整 | 系统算，你确认 |

### 7.2 规则配置方式

**用业务语言写规则**，不写代码。示例：

> "如果毅航连续 3 天达标率 < 80%，黄牌预警，推送到金条管理群 @ 所有人"
> "如果某供应商月度考核评分 < 70 分，自动进入观察期，下月分量减半"
> "如果群聊中出现 '配合不好/响应慢/不配合' 关键词，标记该供应商一次负面记录"
> "如果同一供应商连续 2 次巡检出现相同问题，升级为黄牌"

遇到新情况 → 直接加一条新规则，不需要重新开发。

### 7.3 规则与 Skill 的关系

```
规则引擎 ≠ 一个 Skill

规则定义层 → `/供应商规则管理` Skill（你新增/修改/关闭规则的前台界面）
规则执行层 → 后台自动化 job（规则写好就自动跑，不依赖你调 Skill）
```

## 九、知识库体系

### 9.1 五库结构

#### 库1：运营 SOP 库
- 五步复盘模板
- 整改追踪 SOP
- 考核操作指南（5 维打分标准）
- 准入/清退流程
- 巡检 12 项标准及操作指南
- AB 实验设计规范
- 邮件/报告模板库（处罚/巡检/项目/汇报 4 类）
- 合规问题处理分级标准

#### 库2：供应商档案库
- 32+ 家供应商基础信息（资质、合同、分工范围）
- 各职场出勤规律、历史表现
- 负责人信息、配合风格
- 人员规模、坐席结构、站点分布
- 供应商标签（业务类型/能力/配合度/稳定性/潜力）
- 储备池供应商信息（10-15 家）
- 供应商老板画像（沟通风格、关注点、历史承诺）

#### 库3：经验案例库
- 真实波动案例 + 你的处置逻辑
- 各职场"脾气"（非量化的软性判断）
- 商务谈判关键节点和策略
- 坐席投诉 / 红线案例
- 合规问题典型案例 + 处理方式 + 制度化建议
- 巡检复发问题记录
- 降价谈判经验（什么条件用什么策略）
- AI 心法 7 条

#### 库4：规则库
- 所有规则的原文 + 触发条件 + 生效时间
- 历史变更记录
- 考核指标定义 + 5 维权重（30/25/15/15/15）说明
- 各口径定义与转换规则（数据人力/出勤/排班/结算）
- 红黄牌机制定义
- 分量调整规则
- 合规问题分级标准（轻微/严重判定规则）

#### 库5：商务沟通库
- 供应商老板沟通纪要
- 降价谈判记录和结果
- 重大商务决策记录
- 供应商诉求和承诺跟踪
- 内部协调记录（策略组/服务组/业务方）
- 向上汇报记录和反馈

### 9.2 使用方式

- 5 个知识库在 GetNote 中各自独立 topic
- 数字人按场景自动选择检索范围（查 SOP → 库 1，查档案 → 库 2，查先例 → 库 3）
- 也支持全库全局搜索
- 新增信息后自动归入对应库

### 9.3 维护机制

参照 Anthropic 经验：系统会在一个月内从 95% 衰变到 65%。维护是工程任务。

| 维护内容 | 触发条件 | 维护方式 |
|---------|---------|---------|
| 供应商档案更新 | 人员变动/站点变更/合同更新 | 自动触发提醒 + 你确认 |
| 规则更新 | 口径调整/新合规要求/考核标准变化 | 你主动修改 |
| SOP 更新 | 流程优化/巡检标准变化/新增制度 | 你主动修改 |
| 案例库更新 | 新异常事件/新处置方式/新谈判 | 事件后追加 |
| 商务沟通记录更新 | 每次重要沟通后 | 你录入或 AI 整理 |
| 验证评估 | 定期 | 每月拿历史问题测试系统准确率 |

## 十、数字人交互形态

### 10.1 三种工作模式

#### 被动响应
- 你随时提问："XX 供应商最近配合怎么样" "毅航上月评分多少" "站点布局有什么缺口"
- 数字人检索知识库 + 实时数据 → 回答
- 沟通前主动推送：来电/开会前推送供应商画像

#### 主动监控

| 类型 | 触发方式 | 推送路径 |
|------|---------|---------|
| 定时巡检 | 每日/每周固定时间 | 推送到你 + 管理群 |
| 异常告警 | 规则命中 | 推送到你 + 相关群 |
| 月报 | 每月考核完成后 | 生成报告推送到你 |
| 整改到期 | 到期日前 1 天 | 推送提醒 |

#### 群聊协作
- 监听关键群聊消息 → 提取关键信息 → 结构化入库
- 定时向群推送简报/异常提醒
- 群内可 @ 数字人查询信息
- 标注供应商回复及时性（如某供应商 24 小时未回复）
- **注意**：群聊语义歧义大，关键信息需人工校验

### 10.2 数字人"人设"

> 身份：供应商管理专家
> 能力：检索档案、分析数据、生成报告、执行规则、巡检支持、谈判参谋
> 风格：数据说话，判断在前，理由在后

### 10.3 AI 能做的 vs 不能做的

| 维度 | AI 能做的 | AI 不能做的 |
|------|---------|-----------|
| 数据 | 采集和汇总（排名、KPI、站点分布） | 解读数据背后的原因 |
| 文档 | 初稿生成（邮件、报告、巡检记录） | 审核修改、判断分寸和语气 |
| 规则 | 执行（5 维打分、分量范围、风险信号） | 战略打分、动态调整策略 |
| 信息 | 监控和分类（群消息、异常推送） | 判断哪些是真问题、怎么解决 |
| 商务 | 准备材料（画像、数据包、谈判建议） | 现场谈判、拿捏力度、关系感知 |
| 决策 | 提供数据和建议 | 最终决策、战略方向 |

## 十三、关键原则

### 来自 Anthropic 内部实践的经验

1. **结构优先于数据量**：不要堆积数据。正确的结构（分层知识库 + 模块化 Skill）比海量数据有效得多。Anthropic 喂了几千条历史 SQL，准确率提升不到 1%。

2. **Skill 是最大杠杆**：不加 Skills 准确率 21%，加了超过 95%。不是更强的模型，是结构化的工作流程。

3. **系统会衰变**：一个月内从 95% 掉到 65%。维护是严肃的工程任务，不是可选项。

4. **95% 够了**：最后几个百分点（对抗审查）成本是 +32% token +72% 延迟，只换 6% 准确率。性价比太低，不要追完美。

5. **最难的不是写 SQL（或配置规则），而是让系统找到正确的数据来源**：口径问题必须先解决。
