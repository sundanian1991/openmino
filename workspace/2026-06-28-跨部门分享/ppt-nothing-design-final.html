01 / PROBLEM
现有文档管理的三个痛点
团队每天都在生产信息——会议纪要、操作手册、复盘总结、数据报表。但这些信息散落在不同地方，AI根本用不起来。
01
散落各处
同一份制度在JoySpace、京me群、个人笔记里各有一份。AI不知道哪个是最新版，检索时返回矛盾信息。找不到 = 不存在。
02
格式混乱
Word、Excel、PPT、Markdown、截图混在一起。AI需要纯文本才能有效检索，格式噪声直接降低召回质量。
03
更新滞后
制度改了，文档没跟上。AI引用过期信息给出错误建议，比没有知识库更危险——因为它看起来很权威。
02 / SOLUTION
一套可复制的方法论
不是买一个工具，而是建立三层架构。任何团队都能照搬，从个人先行到团队共享。
01
原文层 RAW
02
提炼层 DISTILL
03
Schema层
原文层 RAW
原始文档的忠实副本。保留所有细节，不做任何删减。AI遇到具体问题时回到这里找原始依据。
内容
制度原文、操作手册、会议纪要、数据报表
格式
统一转为 Markdown，保留结构标题
命名
raw/制度名称.md
价值
可追溯、可审计、AI有据可查
03 / OPERATIONS
三个核心操作
整个知识库的生命周期，浓缩为三个动作。每个动作对应一个独立技能，可以单独使用，也可以串联。
I
摄取 INGEST
把原始文档喂进知识库。自动转格式、提取元数据、生成AI友好的摘要。一次摄取，永久可用。
Q
查询 QUERY
用自然语言问知识库问题。AI从结构化知识中精准召回，给出有出处、有版本的答案。不再靠记忆。
H
体检 HEALTH
定期扫描知识库健康度。发现过期内容、缺失文档、孤立节点。知识库和人一样，需要定期体检。
04 / SKILLS
三个技能对比
每个技能有明确的输入、处理、输出。知道什么时候用哪个，比知道怎么用更重要。
维度 wiki:ingest wiki:query wiki:health
触发场景 有新文档要入库 要查某个制度/流程/数据 定期维护 / 推广前检查
输入 文档文件或URL 自然语言问题 知识库目录
处理 格式转换 + 元数据提取 + 双层写入 语义检索 + 上下文组装 + 来源标注 完整性扫描 + 时效检查 + 质量评分
输出 raw/distill/ + 摘要 有出处的答案 + 引用链接 健康度报告 + 修复建议
频率 按需（新文档入库时） 随时（日常查询） 每周/每月定期
前置条件 知识库已初始化 知识库已有内容 知识库已运行一段时间
05 / CASE STUDY
整理前 vs 整理后
以供应商管理准入制度为例，看同一份文档在整理前后，AI检索效果的差异。
BEFORE / 整理前

原始状态：制度文档散落在3个地方

邮件附件 准入制度v2.1.docx（2024年版本）

JoySpace文档 准入管理办法.pdf（2025年更新）

个人笔记 我整理的准入流程要点.md

AI检索时：

- 返回3个版本，不知道哪个是最新的

- docx格式无法直接解析，丢失表格数据

- 个人笔记有主观解读，与原文矛盾

- 无法回答"最新准入门槛是什么"

AFTER / 整理后

结构化后：知识库中只有一份权威版本

raw/供应商准入管理办法-v3.0.md

来源：2026-06更新版 | 格式：Markdown | 142行

distill/供应商准入管理.md

提炼：核心流程 + 关键阈值 + 版本历史 | 58行

AI检索效果：

- 准确返回最新版本的准入门槛

- 表格数据完整保留，可回答数值问题

- 标注来源版本，可追溯到原文

- 健康度检查发现：缺少2025年Q3的补充说明

关键差异
不是AI变聪明了，是信息变干净了。结构化输入 = 结构化输出。散乱文档喂给AI，得到的只能是散乱回答。
06 / WRITING GUIDELINES
AI提炼 vs 人写
知识库最常见的问题是"信息量浅"——只写了摘要，丢失了关键细节。解决方法：让AI提炼，而不是让人重写。
人写的常见问题
人写摘要时会不自觉地"概括"——丢掉数字、省略条件、简化流程。结果是：看起来整洁，但AI拿到的信息量不够，回答时只能给出笼统建议。
X "准入门槛有要求"（什么数字？）
X "流程大概是这样"（哪几步？）
X "需要注意合规"（哪些红线？）
3x
信息密度提升
100%
数据保留率
0
主观解读
原则
AI提炼 = 忠实压缩，不是重新创作。保留所有数字、条件、例外。丢掉的是格式噪声，不是信息本身。
07 / WORKFLOW
详细流程
WIKI:INGEST / 六步
1
识别来源 — 确认文档位置（本地/URL/知识库）
2
格式转换 — docx/pdf/ppt 统一转 Markdown
3
元数据提取 — 标题/版本/日期/责任人/适用范围
4
双层写入 — raw/ 存原文，distill/ 存提炼
5
关联建立 — 与已有文档建立引用关系
6
入库确认 — 生成摘要，人工确认后生效
WIKI:QUERY / 五步
1
理解问题 — 解析用户意图，拆解为检索关键词
2
语义检索 — 在 distill/ 和 raw/ 中并行搜索
3
上下文组装 — 按相关度排序，组装上下文窗口
4
生成回答 — 基于检索结果生成有出处的答案
5
来源标注 — 每条信息标注文档名+版本+行号
WIKI:HEALTH / 六项检查
01
完整性
是否有缺失的文档？raw和distill是否配对？
02
时效性
文档是否过期？最后更新距今多久？
03
一致性
同一主题是否有矛盾信息？版本是否对齐？
04
可达性
引用关系是否完整？能否从摘要追溯到原文？
05
质量
distill是否保留了关键数据？信息量是否足够？
06
覆盖率
团队核心业务文档有多少已入库？缺口在哪？

09 / PATH
落地路径
不需要一步到位。三阶段渐进，每一步都能产生价值。
01
个人先行
一个人先用起来。把自己的文档整理进知识库，跑通 ingest → query → health 全流程。验证方法论可行，积累实操经验。
第 1-2 周
02
人工确认
关键节点保留人工确认。AI提炼的摘要由人审核，确保信息准确无遗漏。建立信任，收集反馈，迭代流程。
第 3-4 周
03
团队共享

11 / CORE CONCEPTS
KaaS 核心概念
三层结构：一级域 → 项目 → 知识库。理解这个层级关系，就知道文档该放哪、AI 去哪找。
L1
一级域 Domain
事业群下的业务大块。按业务线划分，每个域下有多个项目。
payment enterprise-finance credit-card
domains/
T
项目 Topic
一级域下的具体业务主题。每个项目有独立的 template、wiki 和 docs 目录。
voiceprint aipay rental-3c
domains/payment/voiceprint/
KB
知识库 KB
项目下可独立配置召回策略的库。按角色和意图组织，支持模板驱动的结构化输出。
template/ wiki/ docs/ artifacts/
仓库目录结构
目录结构
domains/一级域/项目/template/ — PRD 模板章节定义
domains/一级域/项目/wiki/ — 角色知识（用户心智、技术规范）
domains/一级域/项目/docs/ — 历史文档（PRD、实现、测试用例）
domains/一级域/项目/artifacts/ — 补充材料（架构说明、ADR）

四步流程
1

BE
后端开发 — kb_pack(intent="backend")
QA
写测试用例 — kb_pack(intent="test")
铁律
写 PRD 前必须先 kb_pack 拿全量。仅靠 kb_search 片段撰写方案是已知反模式 —— 会漏入口、漏用户心智、漏卖点字段。
