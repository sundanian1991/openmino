● 一个真实现象

AI 写得很快。
但经常写错。

你让 AI 代理帮你建一个功能。它 30 秒就输出了一堆代码。看起来能跑——但方向偏了、边界没处理、没有测试、没有文档。你花了一整天修。

01
跳过需求直接动手
AI 按最简理解实现，大概率不是你想要的。没有 spec = 没有合同 = 各说各话。
02
一口气塞太大需求
上下文溢出，AI 开始遗忘、猜测、自相矛盾。质量从第 3 轮对话开始崩塌。
03
"我觉得可以了"
没有验证标准就等于没完成。AI 说"做完了"≠ 真的做完了。你需要证据。
CHAPTER 01

问题不在 AI 的能力，
在它没有纪律。

高级工程师和初级工程师的差距从来不是知识量——是工作习惯。

Agent = LLM + Context + Tools
THE STANDARD FORMULA
Discipline = Process + Gates + Proof
WHAT AGENT SKILLS ADDS

高级工程师写代码之前先想清楚。他们拆任务、写测试、做 code review、检查安全——不是因为流程要求，是因为这些习惯让他们少犯错。

AI 代理没有这些习惯。它很聪明，但没有纪律。它会直接跳到写代码，会跳过测试，会在你没要求的情况下"觉得可以了"。

Agent Skills 做的事：把高级工程师的工作习惯编码成 AI 必须遵循的工作流。不是教 AI 新能力——是约束它的行为模式。

核心洞察
Skills 不是让 AI 更聪明。是让 AI 更可靠。
就像给一个天才实习生配了一套检查清单——他依然天才，但不会再忘记写测试。

CHAPTER 03

这台仪器
的三条铁律

理解了这三条，你就理解了为什么它有效。

01
Process, Not Prose
Skills 不是参考文档，是分步工作流。每个 skill 都是一套可执行的操作序列。
✗ "注意写测试"
✓ "先写失败测试 → 运行确认失败 → 写最小通过代码 → 运行确认通过 → 重构"
02
Anti-Rationalization
每个 skill 列出了常见借口和反驳。防止 AI（和你自己）跳过关键步骤。
✗ "这个功能太小了，不用写测试"
✓ "反驳：小功能也会出错。测试是证明，不是负担。"
03
Verification = Evidence
每个 skill 以证据要求结束。不是"做完了"，是"证明它工作了"。
✗ "代码写好了" → "好的，下一步"
✓ "测试通过、性能指标截图、review 完成 → 下一步"
CHAPTER 04

一次完整的
使用旅程

从"我想做一个 XX"到"它上线了"——AI 代理在 skill 约束下的真实工作流。

01
你有一个模糊的想法
Define

"我想做一个用户注册功能。" —— 停。不要直接让 AI 写代码。先用 /interview 让它问你 5 个关键问题：谁用？什么场景？成功标准是什么？边界在哪？

$/interview → AI 开始结构化访谈
02
把想法变成规格书
Define

访谈完成后，用 /spec 生成一份规格书。这不是 PRD 文档——是 AI 可以执行的工作合同：功能边界、接口契约、验收标准。

$/spec → 输出可验证的规格书
03
拆成原子任务
Plan

用 /plan 把规格书拆成 5-8 个原子任务。每个任务 = 一次 AI 会话可独立完成。有依赖关系：任务 3 依赖任务 1 完成。

$/plan → 输出任务列表 + 依赖图
04
增量实现 + TDD
Build

进入 /build。每个任务内：先写失败测试 → 写最小通过代码 → 重构。完成一个任务，验证通过，再做下一个。拒绝大爆炸。

$/build → 红 → 绿 → 重构 → 下一个切片
05
质量门禁
Review

代码写完了？用 /review 走审查清单：正确性、安全性、性能、可维护性。逐项打钩。不通过就打回重做。

$/review → 审查清单逐项验证
06
发布就绪
Ship

用 /ship 做最终检查：监控就位了吗？回滚方案有了吗？文档写了吗？全部通过——按发布按钮。

$/ship → 发布就绪确认 → 🚀
CHAPTER 05

四个专家
在你耳边说话

Persona = 特定场景下自动切换的审查视角。

CR
code-reviewer
Senior Staff Engineer

审查代码的正确性、可维护性、安全性。不做风格警察，关注真正的问题。

"这段逻辑在并发下会出问题。第 47 行的竞态条件——加个锁或者换个数据结构。"

TE
test-engineer
QA Specialist

设计测试策略、编写测试用例、验证边界条件。测试是证明，不是走过场。

"你测了 happy path，但空输入、超长字符串、并发调用——这三个边界你还没覆盖。"

SA
security-auditor
Security Engineer

安全审计。注入攻击、认证缺陷、数据暴露面、依赖漏洞——逐项排查。

"这个 API 没有速率限制。攻击者可以暴力枚举用户 ID。加 rate limiter。"

WP
web-performance-auditor
Web Performance Engineer

Web 性能审计。Core Web Vitals、资源加载、渲染性能。数据驱动，拒绝没有测量的"优化"。

"LCP 是 4.2 秒，超过 2.5 秒阈值。首屏图片没加 lazy loading，这是主因。"

CHAPTER 06

怎么用好它？

最后是一些真实的建议。

✓
应该做的
→ 把 skill 当约束，不是扩展。它让 AI 更可靠，不是更聪明。
→ 每个阶段有明确的"完成定义"，不模糊推进。
→ 验证是证据要求——测试通过截图、性能指标数据。
→ 把 AI 当成需要纪律的初级工程师，而非全知全能的助手。
→ 遇到"借口"时翻看反理性化清单——那就是你翻车的预兆。
✗
避免的坑
→ 跳过 spec 直接写代码——AI 会按最简理解实现，大概率偏了。
→ 一次性给 AI 太大需求——上下文溢出，质量崩塌。
→ 把 skill 当万能钥匙——它解决"怎么做对"，不是"做什么"。
→ 忽视反理性化清单——那些"借口"不是建议，是警告。
→ 用 skill 替代思考——skill 是纪律，不是思想替代品。
聪明是天赋，
纪律是选择。
Agent Skills 不教 AI 新东西。
它让 AI 每次都做对的事。
