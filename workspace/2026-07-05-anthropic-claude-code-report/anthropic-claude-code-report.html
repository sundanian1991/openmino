Introduction · 视角锚定

这可能预览了知识工作的走向。

Agentic coding 已经起飞。GitHub 项目中有 coding agent 活动的比例自 2025 年底以来翻了一倍多，Claude Code 用户现在平均每周花 20 小时使用这个工具。

没有正式编程经验的人，能不能成功指挥 Agent 完成复杂的技术工作？这些工具的快速采用和改进，对知识工作意味着什么？Anthropic 的这份报告从 Claude Code 使用数据中寻找早期信号。

"What happens on Claude Code may be a preview of where knowledge work is headed." —— 这是报告的开场判断。不是定论，是预览。

◆ 核心论点

Coding agents 不是在替代领域专业知识。

工作者带来的理解越多，Agent 能产出的高质量工作越多。成功取决于人对问题的理解程度，不是是否受过编程训练。如果这些模式在整个经济中成立，意味着 agentic coding 工具在吸收部分实现密集型工作的同时，也在奖励那些对解决的问题有扎实理解的人。

Section 2 · 分工

人决定做什么，Agent 决定怎么做。

报告构建了一个隐私保护的决策归因分类器——列出会话中所有有意义的决策，分为规划（做什么/走哪条路线/怎样算完成）和执行（改哪些文件/写什么代码/跑哪些命令），然后将每个决策归因于 Claude 或用户。

典型的 Claude Code 会话约有 4 轮交互。用户每发一条 prompt，Claude 平均执行约 10 个动作（有时超过 100 个），每轮输出平均 2,400 字——读文件、改代码、跑命令。

Claude 在两轮检查之间做多少事，很大程度上取决于谁在做决策。当用户保持执行控制（做 >80% 执行决策），Claude 每轮约 8 个动作。当 Claude 接管规划（做 >80% 规划决策），每轮动作数最高，约 16 个。

Section 3 · 专业度

回报来自领域知识，不是编程技能。

Claude 从每个转录中用五分制（novice 到 expert）评估用户在该任务上的专业度。分类器寻找三个信号：用户指令的精确程度、用户让 Claude 验证什么、以及用户纠正 Claude 还是 Claude 纠正用户。

关键：专业度是任务特定的，不是职位或通用能力。一个资深工程师第一次问 Rust 问题，在 Rust 上就是新手。一个从没用过 Python 的会计，如果能准确告诉 Claude 对账脚本必须执行哪些规则，并抓住月末结账时它处理错的边界情况——在这个任务上就是专家。

— 新手会话 · 每条 prompt
5 动作
~600 字输出
— 专家会话 · 每条 prompt
12 动作
~3,200 字输出

专家的每条 prompt 设触发的动作链是新手的两倍多（12 vs 5），输出是新手的五倍（3,200 vs 600 字）。这个差距在每一种工作类型、每一个任务价值区间都存在。

回归分析（控制工作模式、任务价值、月份、职业、模型家族）后，专业度每升一级，动作数增加 +9%，输出增加 +13%（p < 0.001，标准误按用户聚类）。

所有成功率指标都指向同一个模式：专业度越高，成功率越高。但大部分增益集中在量表低端——新手到中级的差距，比中级到专家的差距大得多。差距是 modest 的——在一个领域有足够的熟练度，就能用得几乎和深度精通者一样好。不必是大师，懂行就够了。

◆ "verified success" 严格定义

报告的"成功"有两层：judged success（分类器读完整转录，判断用户是否做到了设定的目标）+ verified success（judged success + 至少一个硬信号）。硬信号包括：git 提交/PR、测试套件通过、用户明确确认。两者同时满足才算 verified success。约 7.7% 的会话被判定为"无明确目标"，不纳入成功率分析。

Section 4 · 翻车恢复

翻车之后，专家救得回来。

"hit trouble" 定义：失败信号记录到验证过的失败证据——报错、测试失败、多次重试同一件事、用户表达不满或沮丧。复杂任务里，Agent 出错是常态。区别在出错之后。

◆ 关键发现

专家会话达到 verified success 的频率是新手的两倍多。翻车时新手放弃的比率是其他人的数倍。

专业度的一部分价值，似乎正是引导 Agent 走对方向的能力——知道补充什么约束、要求验证什么、在哪个方向纠偏。优势不在不翻车，在翻车后的恢复力。

Section 5 · 跨职业对比

软件工程师不独占鳌头。

报告从会话转录推断用户职业，映射到美国劳工统计局 SOC 分类的 23 个主要组。分类器明确指令：不要把"在编程"本身当作编程职业的证据。约 70% 的会话成功推断出职业。

软件相关职业在产代码会话中的 verified success 约 34%，其他职业约 29%——5 个百分点的差距，且在 7 个月内既没有扩大也没有缩小。在更宽松的"至少部分成功"指标下，两组差距几乎消失（89% vs 88%）。

十大职业每一个与软件/数学类用户的差距都在 7 个百分点以内。管理类在 verified success 上最高，略高于软件工程类。

Section 3.3 · 七个月变化

不再只是修修补补。

2025.10 → 2026.04，工作构成发生了实质性变化。最明显的是修 bug 的会话占比从 33% 降到 19%——取而代之的是围绕代码的工作在增长。

任务本身也变得更有价值。报告用自由职业市场的报价估算每个会话的经济价值——7 个月间，平均会话价值上升约 27%。这个上升跨多种工作类型：building 约 +43%，operating 约 +34%，fixing 约 +32%。报告说这些估价是粗略的，主要用于任务间的相对比较，不作为字面美元值。

◆ 方向信号

AI 编程工具的进化方向不是"更会补全代码"——用户开始让它干代码之外的活。

Looking Ahead · 未来观察指标

如果回报下降，意味着什么？

报告最深的一层不在发现，在未来观察指标。它给了一套可以持续追踪的尺子。

能力来自胜任，不是精通。——原文："competence, not mastery."。对领域有工作中的把握，就能获得大部分收益；深度专精只在此基础上增加一点点。

有这种把握的人，在任何领域，现在都可能做到以前做不到的技术工作。没有这种专业知识的人，从同一个工具得到的要少得多。

◆ 核心观察框架

如果专业度回报开始下降，意味着模型正在补上人类现在带的判断力。

那时受益群体会从领域专家扩展到所有人。同样，如果非软件职业的会话成功率持续上升，意味着软件开发正变成各行各业的普通工作能力，而不是单一职业的产物。这些转变会改变谁从 agentic coding 中受益、受益多少。

◆ 局限性

报告不能测量真实世界结果（代码是否被使用或丢弃、是否产出经济价值）。非交互式使用被排除。所有会话分类依赖模型对转录的阅读——分类器在规模上验证仍然困难。

◆ 这不是结论，是观察框架

这份报告留给你的是一套追踪指标——分工比例、专业度回报、错误恢复率、跨职业成功率。用它自己观察趋势，不要把它当定论。
