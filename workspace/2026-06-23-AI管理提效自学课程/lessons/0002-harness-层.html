你 06-22 写过一个修复器，解决了一个具体问题：开源模型（DeepSeek / Qwen / GLM）调用工具时反复失败。你的 tool-call-repair.md 开头有一句关键判断——

这句话之所以重要，是因为它把"模型笨"的诊断，重定向到了"harness 没接住"。这正是整个 Harness 层的核心心智模型，也是前沿正名运动的方向。

核心命题 Model + Harness = Agent。 模型负责提议，harness 负责执行、校验、修复。多数"模型不行"其实是"harness 没搭好"。Anthropic 正推动用 harness 替代被滥用的 agent 一词。

Harness 的九个构件（前沿共识）

综合 Anthropic 论文、Augment Code 指南、Claude Code 模式提炼，一个完整的 harness 有九个职能构件。我用你在 lib/tool-input-repair.ts 里已有的代码，逐个标注你已经覆盖了哪几个：

#构件职能你的现状
1 工具设计 工具的颗粒度、返回结构 部分（AGENTS.md 工具配置表）
2 工具调用修复 修复格式噪声，避免无谓重试 ✓ 完整（Repair Layer）
3 质量门 硬规则校验，拦截明显错误 ✓ Git/可视化规范
4 PEV 校验回路 Propose→Execute→Verify 循环 部分（主代理决策）
5 短期记忆 当前任务的上下文状态 靠 context window
6 长期记忆 跨会话的知识沉淀 ✓ memory/ 三级
7 权限护栏 危险操作确认 ✓ subagent 原则
8 成本约束 token 预算控制 靠直觉（第 7 层课会深讲）
9 反馈结构化 工具返回结构化结果，供下一轮判断 部分

数一下：你已经完整覆盖 4 个（第 2/3/6/7），部分覆盖 3 个。剩下需要补强的主要是 第 4 个 PEV 校验回路 和 第 9 个反馈结构化。

这不是新手课 你不是在学一个新概念——你是在给已有实践命名、归位、找缺口。你 06-22 写下的那句"修复而非拒绝，对可恢复噪声保持宽容"，就是 harness 工程的设计哲学原话。前沿工业界正在追上你半年前的判断。

PEV Loop：harness 的动态骨架

九个构件里，最值得你现在补的是 PEV Loop（Propose → Execute → Verify）。它是把九个静态构件串成动态系统的那根线：

阶段谁做你的对应
Propose（提议） 模型 模型输出工具调用 / 方案
Execute（执行） harness 你的 repairToolInput 在这步修复后执行
Verify（校验） harness（独立于模型） 这是你当前的缺口

关键洞察：校验必须独立于提议的模型。如果让同一个模型既提议又校验，它会倾向于认为自己对——这是 harness 设计的常见反模式。

你的 Repair Layer 是 E（执行）阶段的杰出实践。但 V（校验）阶段，你目前主要靠"主代理决策"的软规则，缺一个结构化的、独立于模型的校验层。

用智力密度重审你的 Repair Layer

现在用第 1 课的标尺，量一下你的 Repair Layer 到底值不值：

维度没有 Repair Layer 时有了之后
token 消耗 失败 → 重试 → 再失败 → 再重试
低密度多 token（最差象限）		 失败 → 修复 → 成功
高密度（该花的钱花到位）
失败模式 4 类格式噪声反复触发 结构性错误在入口处归零
ROI 每次重试 = 整段 context 重新过模型 一次轻量修复，省下整次模型重跑
用标尺得出的结论 你的 Repair Layer 不是"锦上添花"，而是把"低密度多 token"陷阱直接堵死的高 ROI 构件。这是 harness 层存在的根本理由——它是 token 经济的第一道防线。

你的代码里藏着的三个设计原则

回读你的 lib/tool-input-repair.ts，里面有三个选择，每一个都暗合前沿共识：

① 宽容而非严格（第 81 行）

你在 schema 校验之前修复，而不是校验失败后拒绝。这对应前沿的 "model proposes — harness executes" 原则：harness 的职责是让模型的提议能落地，而不是当裁判判它错。

② 有 Schema 时更精准（第 82 行）

你传 schema 时只修复已知类型的字段，无 schema 时全局扫描。这暗合 progressive disclosure——给 harness 的信息越完整，它能帮的越精准。

③ 执行顺序有讲究（第 83 行）

你的 REPAIRS 数组按固定顺序 reduce，JSON 字符串解析在裸字符串包装之前。这是修复器的依赖管理——先还原结构，再补形状，避免后序修复破坏前序结果。
