# haglofs-paradigm 技能测试计划

> 把零散的"每轮手动验证"升级为系统化、可回归的测试套件。
> 目标：每次版本升级后能快速重跑，确认质量不回归 + 发现新边界。
> 创建：2026-07-03 · 对应技能版本 v1.6+

---

## 一、测试目标

### 核心问题
目前技能经过 5 轮迭代验证，质感分从 7.5 升到 8.5，但存在三个隐患：
1. **无回归保护**——修了 C3 对比度矛盾，会不会影响之前 Fjällräven 页？不知道，因为没重跑。
2. **覆盖有盲区**——5 轮验证是"碰到什么测什么"，不是"系统覆盖"，可能遗漏未测场景。
3. **不可比**——每轮验证的输入素材、评判尺度不同，质感分 7.5 vs 8.5 难以横向对比。

### 测试计划要解决的
- **标准化测试矩阵**：覆盖全部页面类型 × 调性 × 边界条件
- **标准化输入素材**：每个测试用例有固定输入，可重复
- **标准化评判**：统一的 craft-checklist + 质感分 + 缺陷数三维度
- **回归基线**：每个版本跑同一套测试，对比指标趋势

## 二、测试矩阵

### 维度 1：页面类型（5 类预设 + 边界）
- P1 品牌宣言页（Statement Hero 主导）
- P2 品牌展示页（多组件 + 产品卡 + 传承）
- P3 品牌数据简报（混合页，Pulse Hero + 数据+品牌组件共存）
- P4 设计系统文档页（sidebar + 系统展示组件）
- P5 技术内容页（代码块密集 + 技术调性适配）

### 维度 2：内容调性（3 种）
- T1 户外/自然品牌（Haglöfs 同范式，调性最匹配）
- T2 科技/工程内容（调性有张力，测适配能力）
- T3 时尚/奢华品牌（高对比调性，测色彩弹性）

### 维度 3：边界条件（压测用）
- E1 标准场景（素材充足，7-8 section）
- E2 深段配额极限（Hero深+2内容深段+Footer深，逼近 N4 红线）
- E3 组件-深浅冲突（深段满 + 需深底组件，测 v1.6 冲突预检）
- E4 内容残缺（素材不足/主题模糊，测 Step 0 采集能力）
- E5 跨技能链路（nian-brand 决策卡 → haglofs-paradigm Step 0，测交接）
- E6 整页深色模式（整页 `.dark`，token-root 覆写已就绪但从未端到端测过）

## 四、评判标准（三维）

每个测试用例跑完，按三维度评判：

### 维度 1：craft-checklist 通过率
- 33 项逐条 ✓/✗
- 文档页例外（C2/C7/CP1/CP4）自动生效
- 目标：≥30/33（≥91%）

### 维度 2：质感分（1-10）
| 分段 | 含义 |
|------|------|
| 9-10 | 博物馆级，超越 H061 水准 |
| 8-8.5 | 范式内高质量交付（当前上限） |
| 7-7.5 | 可交付但有明显差距（当前下限） |
| <7 | 不合格，需返工 |

### 维度 3：缺陷数
- 统计新发现的缺陷（按类别：Step 0/组件/母版/规则/检查歧义）
- 目标：≤3 个 P1+ 缺陷（P2 不计入阻断，记录到 ROADMAP）

### 通过判定
- **绿灯**：checklist ≥目标 + 质感 ≥目标 + P1 缺陷 ≤目标 → 版本可发布
- **黄灯**：一项未达目标 → 标记问题，可发布但记入 ROADMAP
- **红灯**：两项以上未达 → 必须修复后重测

## 五、执行方式

### 方式 A：批量子代理并行（推荐，快）
**适用**：完整跑一轮测试矩阵（全部用例）
**做法**：
```
主代理一次性派 N 个子代理（一条消息多个 Agent tool call）
每个子代理跑一个测试用例：
  1. 读技能文件
  2. 执行 Step 0 蓝图（用测试用例的固定输入）
  3. 施工产出页面
  4. 跑 craft-checklist
  5. 产出测试报告
主代理等待全部返回，汇总成测试报告
```
**耗时**：约 8-15 分钟（并行）
**算力**：高（N 个完整页面生成）

### 方式 B：关键用例快速回归（最省）
**适用**：版本升级后的快速回归检查
**做法**：只跑 T1（基线）+ T3（混合）+ T5（技术）三个核心用例
**耗时**：约 15 分钟（3 个并行）

## 六、测试记录规范

### 每次测试产出 3 类文件

1. **测试页面**：`stage-outputs/test-T{N}-{brand}.html`
2. **测试报告**：`stage-outputs/test-T{N}-report.md`（含三维评判）
3. **测试汇总**：`stage-outputs/test-summary-{version}.md`（本轮全部用例汇总）
4. **TEST-STATE.md（可选）**：如果想可视化测试进度（特别是 9 个用例并行时），可在 stage-outputs/ 维护一个进度表：
```
| 用例 | 状态(pending/running/done/blocked) | 质感分 | checklist | 缺陷数 | 报告路径 |
```
这是可选的进度追踪文件，不是流程强制要求。并行模式下每个子代理独立，断了不影响其他。

## 七、回归基线（目标值）

> 每个版本跑完测试后，对比基线值。趋势应该是"缺陷数下降，质感分上升"。

| 用例 | v1.6 基线目标 | v1.7 目标 | v2.0 目标 |
|------|-------------|-----------|-----------|
| T1 品牌宣言 | ≥8.5 / ≤2 缺陷 | ≥8.5 / ≤1 | ≥9.0 / 0 |
| T2 品牌展示 | ≥8.0 / ≤3 | ≥8.0 / ≤2 | ≥8.5 / ≤1 |
| T3 混合页 | ≥8.0 / ≤3 | ≥8.0 / ≤2 | ≥8.5 / ≤1 |
| T4 文档页 | ≥8.5 / ≤2 | ≥8.5 / ≤1 | ≥9.0 / 0 |
| T5 技术内容 | ≥7.5 / ≤3 | ≥8.0 / ≤2 | ≥8.5 / ≤1 |
| T6 品牌传承 | ≥8.0 / ≤3 | ≥8.0 / ≤2 | ≥8.5 / ≤1 |
| T7 内容残缺 | 关键决策正确 + 填充≥80% / ≤4 | 关键决策正确 + 填充≥90% / ≤3 | 关键决策正确 + 填充≥95% / ≤2 |
| T8 组件冲突 | 预检触发 / ≤3 | 预检触发 / ≤2 | 预检触发 / ≤1 |
| T9 深色模式 | 首跑无基线 / ≤5 | 层次可感知 / ≤3 | 稳定≥8.5 / ≤1 |

**回归红线**（必须**同时满足**两个条件才判定回归）：
1. 质感分比上一版本低 ≥1.0 分，**且**
2. 缺陷数增加 ≥2 或 checklist 通过项减少 ≥2

> 质感分人工打分正常波动 0.3-0.5，单条件触发 = 噪声。双条件同时满足 = 真退化，必须排查。

## 八、首次执行建议

### 推荐范围
**首次跑全套**（方式 A，9 用例并行），原因：
- 建立完整基线（v1.6 的 9 个数据点）
- 后续版本只需对比，不用重新建基线
- 一次性暴露所有未测场景的缺陷

### 如果算力敏感
先跑**方式 B**（T1+T3+T5 三个核心），确认无严重回归后再补跑其余 6 个。

### 执行前准备
1. 确认技能版本（当前 v1.6）
2. 确认每个用例的输入素材已固定（本文件第三节）
3. 创建 `stage-outputs/` 下的测试目录

### 执行后
1. 汇总测试报告到 `test-summary-v1.6.md`
2. 对比基线目标，标记绿/黄/灯
3. 新缺陷追加到 ROADMAP.md
4. 决定是否需要 v1.7 修复
