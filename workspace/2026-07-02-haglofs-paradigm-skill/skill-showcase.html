§1 · Design Constitution

四条第一性 法则

v2.1 组件设计宪法。所有组件设计必须遵守——视觉形态不由"好看"决定，由信息本质决定。

Law 01

信息本质驱动 形态

视觉形态由信息本质决定。财报数字密集 → mono 右对齐；术语对照 → dot leader 点线。先问"这是什么信息"，再决定长什么样。

设计感
Law 02

每个元素挣到自己的 像素

减法优先。删了损失 ≤5% 就删。没有"以后可能用得上"的元素——存在就必须有理由。

干净
Law 03

字体翻译信息 性质

Playfair 扛权威（标题/装饰），JetBrains Mono 扛精度（标签/数据），Inter 扛功能（正文）。角色不混，一个字体的每个用法都对应它翻译的信息性质。

质感
Law 04

层次靠留白 + 色调 + 1px 线

不靠加东西。不加分隔条、不加斑马纹、不加阴影。层次来自留白节奏、色调微差、1px 发丝线——这是哑光克制的工程实现。

质感 + 干净

§3 · Typography

四字体 角色分工

字体角色是信息性质的翻译器（宪法法则 3）。Playfair 权威、Inter 功能、Mono 精度、Doto 装饰——角色不混。

Aa
Playfair Display
温度 / 权威
衬线 Display 字体。扛标题、扛装饰、扛温度。weight 300 细衬线最显高级，权威感来自细 + 巨幅，不来自粗。
Aa
Inter
功能 / 清晰
无衬线 body 字体。扛正文、扛功能信息。绝不做大标题，绝不承担温度。
Aa
JetBrains Mono
精度 / 刻度
等宽 data 字体。扛标签、扛数据、扛测绘基准线的"工程"视觉隐喻。全大写 + 加宽字距 + 前缀线。
Aa
Doto
装饰预留
点阵 decorative 字体。装饰预留位，用于特殊场景的年份/编号点阵效果。不承担信息载体职责。

Variant 3 · Spec
OAuth 2.0 端点规格 · key-value 双列
authorize
GET /oauth2/authorize · response_type=code · 需用户代理重定向
token
POST /oauth2/token · grant_type=authorization_code · client_secret + code
refresh
POST /oauth2/token · grant_type=refresh_token · 返回新 access_token
revoke
POST /oauth2/revoke · RFC 7009 · 吊销 access/refresh token
introspect
POST /oauth2/introspect · RFC 7662 · 查询 token 活跃状态
userinfo
GET /oauth2/userinfo · OIDC · Bearer token 鉴权
jwks
GET /.well-known/jwks.json · 公钥集 · 验证 JWT 签名
Variant 4 · Reference
OAuth 术语表 · dot leader 点线
  • Access Token授权服务器签发的访问凭证，附在 API 请求 Bearer 头
  • Refresh Token用于换取新 access token 的长效凭证，不直接访问资源
  • PKCEProof Key for Code Exchange，防止授权码截获攻击
  • Scope授权范围的字符串标识，限定 token 可访问的资源
  • Client Secret机密客户端的共享密钥，服务端存储不可泄露
  • Grant Type获取 token 的方式（授权码/客户端凭证/刷新等）
  • Redirect URI授权完成后用户代理跳转回客户端的地址，需预注册

§8 · Radius Convention
