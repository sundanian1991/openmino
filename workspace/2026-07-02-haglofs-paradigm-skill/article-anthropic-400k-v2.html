上周，Anthropic 发了一份报告，本来没太当回事。因为最近这家公司身上的新闻实在太多了——一会儿搞实名认证，一会儿顶级模型给看不给用，还有跟美国政府的对线。现在一看到他们家的名字，都快有点儿免疫了。

但当看完这份报告的详细内容后，还是忍不住想分享给你。因为这不是那种「我们觉得未来会怎样」的趋势预测报告，而是实打实的数据报告。

Claude Code 是过去一年全球知识工作者广泛使用的 AI 编程产品之一。说「之一」只是为了避免太绝对，但对于 AI 专业领域的人来说，它几乎就是必选项。甚至连它的竞争对手谷歌的工程师，自己家也在做类似的产品，还公开称赞 Claude Code 好用。

注意，这已经不是一个「写代码的工具」，而是一个「完成复杂任务」的 AI 产品。你给一个指令，它会自己读文件、改代码、跑命令、检查结果，然后再回到你这里。报告里有个小数据：一条指令下去，它平均触发大约 10 个动作，输出约 2400 字。

这种数据，在 AI 行业里，之前几乎没有。大多数 AI 研究报告，要么是跑测试题，要么是做问卷调查，要么是分析公开代码仓库。但真实的人机协作过程到底长什么样——人说了什么、AI 做了什么、最后成没成——这些细节，只有像 Anthropic 这样既做模型、又能看到真实使用数据的公司，才拿得到。
核心发现 01

人管判断，
AI 管执行

这份报告最有价值的发现，是第一次用真实使用数据证明了一件事。

~70%
人类决策
规划：做什么 / 走哪条路线 / 怎样算完成
~80%
AI 决策
执行：改哪些文件 / 写什么代码 / 跑哪些命令

70% 和 80%，这两个数字不是猜测，不是推演，是 40 万次对话里真实存在的分工比例。人管 what，AI 管 how，这句话以前很多人在说，但从来没有人拿数据证明过。现在有了。

举个小例子：你跟 Claude Code 说一句「帮我修复这个登录页面的 bug」，然后它自己读了三个配置文件，改了两处代码，跑了一遍测试，测试没过，又回去改了一处，再跑，过了，最后把改动列出来给你看。你全程只说了一句话，Claude Code 做了十几个动作。

而且这个分工还在往更深的方向走。报告追踪了七个月的变化：

修 bug
33% → 19%
运行软件
14% → 21%
写作 + 分析
~10% → ~20%

七个月下来，用户开始把更接近最终交付的任务交给它了。任务平均估算价值上升了 27%。AI 编程工具的进化方向，不是「更会补全代码」，而是让它干代码之外的活了。

核心发现 02

在 AI 面前，
人的资历归零了

报告里最锋利的发现来了，连定义本身都很有意思。

Anthropic 给每个会话里的用户打了一个「专业度」评分，从新手到专家，五个级别。但这个「专家」的定义，跟我们以为的完全不一样。一提「专家」，我们脑海里经常浮现的是老教授和名医，衡量标准主要是从业资历。

但在这个报告里，它不看职位头衔，不看从业年限，不看学历证书。它只看你在这一次具体任务里的具体表现——三个信号：

信号 1
指令够不够精确——不是说多少字，而是目标、边界、验收标准有没有说清楚
信号 2
知不知道该让 AI 验证什么——哪些地方需要检查、哪些边界情况要覆盖
信号 3
AI 理解错了能不能发现并纠正——在关键节点上拉回来

一个不会 Python 的会计，从来没写过代码，但她能清清楚楚地告诉 Claude Code 对账脚本必须执行哪些规则，而且能抓住月末结账时的边界错误。在这个任务上，她就是专家。

反过来，一个资深软件工程师，如果他对 Rust 不熟悉，那他在 Rust 相关任务上给出的指令就不精确，也不知道该让 AI 验证什么。在 AI 眼里，他就是新手。

"一大批只会管人的中高层领导，在 AI 面前，就要被归为「职场新手」了。做了 20 年产品经理，但如果只是跟 AI 说「你帮我做一下某个功能的产品需求文档」——在 AI 眼里就是一个产品经理新手，因为指令质量太低。"
核心发现 03

专家一条指令，
顶新手五倍

这个「指令质量」的差距，到底有多大？这份报告给出了数据：

5→12
每条指令触发动作
新手每条 5 个动作；专家 12 个。近 2.5 倍。
600→3200
每条指令输出字数
按每条指令的产出来算，专家从 AI 那里拿回来的，是新手的五倍还多

还没完。报告做了控制变量，把工作类型、任务价值、时间、职业、模型家族全部拉平之后，专业度每升一级，Claude Code 的动作数增加 9%，输出增加 13%。趋势依然显著。

这说明什么？好指令不是微操，不是事无巨细地告诉 AI 每一步该干什么。好指令是高质量授权——你把目标、边界和验收标准说得足够清楚，AI 拿到之后能自己跑很远。

甩手掌柜型
帮我写个脚本
AI 就给你写个脚本。对不对账、规则对不对，它不管。结果不能用。
高质量授权型
帮我写个对账脚本，必须校验这三条规则，月末结账的边界情况要单独处理
AI 拿到就能自己跑出一条完整的路来。一次通过。

差别不在于你说了多少字，而在于你有没有把 AI 需要的东西给到。

核心发现 04

管理者的验证成功率
比软件工程师还高

说到指令质量，还有一个让人意外的发现。在确实新增或修改了代码的会话里，Anthropic 把用户按职业分类，比较他们的「验证成功」率——要么代码提交了，要么测试通过了，要么用户明确确认。

职业 验证成功率 vs 工程师
管理类37%+3pp
软件工程师 / 数学34%基准
法律类33%-1pp
商业金融29%-5pp
医疗 / 设计 / 销售 / 教育 / 科学27-28%-6-7pp

管理职业的验证成功率比软件工程师还高。报告特别说明，这可能与管理者更习惯明确确认结果有关——毕竟当领导的习惯了说「对对对就是这」。

但即便打个折，这个数据依然给出了一个很强的信号：拆任务、说清目标、委派、检查结果——这些原本是管理动作，现在也成了操作 AI 的技术动作。你不需要先把自己训练成工程师，但你需要能说清业务规则、边界条件和验收标准。

核心发现 05

翻车后，
才是真正分水岭

所谓「遇到麻烦」，就是出现了报错、测试失败、反复重试、用户表达不满。在复杂任务里，AI 出错是常态，这不是什么丢人的事。但在遇到麻烦之后，不同专业度的用户，结局完全不同。

100 次翻车里

专家遇到麻烦后
15%
是新手近 4 倍
翻车不是终点
19%
新手放弃率
5-7%
专家放弃率

新手放弃率高，不是因为 AI 帮不了他们——恰恰相反，AI 就在那里等着，只要你给对方向它就能跑。但新手给不出方向，于是得出「AI 不好用」的结论。而专家遇到同样的麻烦，会说「你这里理解错了，应该是这样的」，然后继续推进。同一个工具，同一个困境，结局完全不同。

写在最后

生产软件，
可能正在变成普通工作能力

"生产软件，可能正在从一种程序员专属的职业能力，变成一种普通工作能力。"
—— Anthropic 报告原文

这不是说非程序员可以替代软件工程师——工程师的价值从来不只是写代码，还有系统架构、风险控制、长期质量维护。但方向是对的：未来不是人人都成为程序员，而是各行业懂问题的人，可以直接生产脚本、自动化流程、内部工具和技术文档。

律师会写合同检查脚本，还是一名律师；会计指挥 Python 对账，还是一名会计。编程不再是程序员的专利，而是各行各业解决自己问题的工具。

AI 面前，你的资历归零了，但你的专业能力没有归零——它只是换了一种接口和表现形式。
