## 二、结构性优化方向

> 超出单 bug 的系统性改进。这些是"技能形态升级"，不是补漏。

### 1. Step 0 智能化升级（deckify 式自动侦察）
**现状**：Step 0 靠提问采集，agent 要问 7 个问题。
**优化**：学 deckify 的自动侦察——如果用户给了品牌官网 URL，自动抓取 logo/品牌色/字体/文案素材，自动填蓝图采集区，只问推断不出的。
**价值**：减少提问摩擦，从"7 问"降到"2-3 问"。
**优先级**：中期

### 2. Hero 选型可视化（frontend 式 show don't tell）
**现状**：Hero 不确定时在蓝图标注"需用户选择"，但还是文字描述。
**优化**：生成 2-3 个 Hero 变体的真实 HTML 预览（mini 版，只有 Hero 区块），让用户视觉对比选。
**价值**：审美偏好不该用文字问，用真东西对比。
**代价**：耗算力（每个预览是一次小型生成）。建议条件触发——只在多候选时才生成。
**优先级**：中期

### 3. 跨技能端到端链路验证
**现状**：nian-brand（叙事）→ haglofs-paradigm（施工）的链路从未端到端测过。技能各自完善，但交接点没验证。
**优化**：设计一个测试——用 nian-brand 产出一份品牌决策卡，直接喂给 haglofs-paradigm 的 Step 0，验证"决策卡字段"是否能无缝填入"蓝图采集区"。
**价值**：确保技能群的协作链路通畅，不只是单技能可用。
**优先级**：短期（应纳入测试计划）

### 4. 回归测试基线建立
**现状**：每次修 bug 后没重跑之前的验证，无法确认"修了 C3 会不会影响之前的 Fjällräven 页"。
**优化**：建立标准测试集（见 TEST-PLAN.md），每次版本升级后重跑，对比缺陷数/质感分是否回归。
**价值**：防止"修一个引入一个"的隐性退化。
**优先级**：短期（TEST-PLAN.md 已设计）

### 5. 测试覆盖面扩展
**现状**：5 轮验证覆盖了宣言/展示/混合/文档/技术 5 类，但还有未测场景。
**未测场景清单**：
- 电商产品落地页（强 CTA + 产品卡密集）
- 招聘/团队页（人物卡 + 价值观）
- 活动页（时间敏感 + 报名流程）
- 纯深色模式页（token-root 有 .dark 覆写但从未测过纯深色页）
- 多语言（中英混排的排版规则）
**优先级**：中期

## 三、新能力扩展

> 这些是"如果要做"的方向，不是"应该做"的修补。

### 1. 技术内容变体（潜在独立技能）
**洞察来源**：Loop Engineering 验证（v1.6）发现"8.0 是技术内容用品牌范式的合理上限"。
**方案**：如果技术内容是高频场景，开发一个独立技能或强变体——slate/steel 冷调主导 + Mono 主导代码区 + Playfair 退到 section 标题。本质上是"Haglöfs 范式的技术分支"。
**判断时机**：如果后续技术内容需求 ≥30%，值得做；偶发则用 rules-content-types.md 适配够了。
**优先级**：长期（看需求）

### 2. 可访问性（a11y）维度
**现状**：craft-checklist 33 项全是视觉/结构，没有 a11y。
**缺口**：alt 文本、ARIA 标签、键盘导航、焦点管理、色彩对比度自动化检查（目前对比度靠人工判断）。
**方案**：新增 a11y 检查清单节，或独立 a11y-checklist.md。
**优先级**：中期

### 3. 性能维度
**现状**：未测加载性能。
**缺口**：单文件 HTML 体积（855 行是否过大？）、图片优化、字体加载策略（Google Fonts 的 FOUT）、首屏渲染。
**方案**：加性能基线（如"交付页 ≤1500 行 / 字体 ≤4 个 family / 无未压缩图片"）。
**优先级**：长期

### 4. 深色模式原生支持
**现状**：token-root 有 `.dark` 覆写（5 色提亮），但技能从未产出过纯深色页。
**方案**：新增"深色模式页"预设（第 6 类），用 `.dark` 作用域做整页深底。Hero/section/footer 都深色但靠层次区分。
**风险**：深色页的深段配额规则完全不同（全是深色=没有节奏断点），需重新设计。
**优先级**：长期
