概念建立 / Foundation

授权，而非认证

"

OAuth 2.0 是一个授权框架，让第三方应用在资源所有者授权下，以受限方式访问其 HTTP 服务——它解决的是"能做什么"，而不是"你是谁"。

— RFC 6749, The OAuth 2.0 Authorization Framework

一次授权码流程以一个重定向请求开始：客户端把用户引导到授权服务器的 /authorize 端点，附带客户端凭证与请求范围。下面的请求展示了一次最小化的授权请求。

authorize-request.http 
GET /authorize?response_type=code
  &client_id=s6BhdRkqt3
  &state=xyz123
  &redirect_uri=https://app.example.com/cb
  &scope=read write HTTP/1.1
Host: auth.example.com
流程选型 / Grant Selection

四种授权流程对比

RFC 6749 定义了四种授权类型，各自面向不同的客户端类型与信任边界。选型不当是 OAuth 实现中最常见的安全根因。

授权流程 安全性 适用场景 状态
Authorization Code code 高 · 双步骤隔离凭证 Web 应用、移动端、SPA（配 PKCE） 推荐
Implicit token 低 · Token 经前端返回 历史上面向纯前端 SPA 废弃
Password password 中 · 客户端接触密码 自有可信应用、迁移期 受限
Client Credentials client_credentials 高 · 机器对机器 服务间调用、后台任务 推荐
授权码流程 / Authorization Code Flow

端到端五步实现

授权码流程通过"前端拿码、后端换令"的双步骤设计，把访问令牌隔离在不暴露给浏览器的后端通道中，是当前所有新实现的事实标准。

01

用户重定向

客户端把用户浏览器重定向到授权端点 /authorize，携带 client_id、redirect_uri、state 与请求的 scope。

02

用户授权

授权服务器向用户展示同意页，用户登录并明确批准客户端请求的访问范围。授权服务器记录该授权决定。

03

回调携带 code

授权服务器把用户重定向回 redirect_uri，并在 query 中附带短时效的 code 与原样回传的 state（防 CSRF）。

04

用 code 换 token

客户端后端用 code + client_secret（机密客户端）向 /token 端点换取访问令牌。此步在后端完成，secret 不暴露给前端。

05

用 token 访问资源

客户端携带 Bearer 令牌访问受保护资源服务器，资源服务器校验令牌的有效性与作用域后返回数据。

第四步是整个流程的安全核心——后端到后端的令牌交换请求如下：

token-exchange.http 
POST /oauth/token HTTP/1.1
Host: auth.example.com
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=SplxlOBeZQQYbYS6WxSbIA
&redirect_uri=https://app.example.com/cb
&client_id=s6BhdRkqt3
&client_secret=dCvG9Kx3rF2nQp8 # 后端持有
安全加固 / Hardening

七条工程纪律

以下清单覆盖 OAuth 2.0 实现中最高频的安全失分点。每一条都对应一个真实出现过的事故模式。

交付前自检清单
Do 始终校验 state 参数，用不可预测的随机值绑定会话，防 CSRF 攻击。
Do 公开客户端强制启用 PKCE（RFC 7636），用 code_verifier / code_challenge 替代 client_secret。
Do 令牌存储在后端 + HttpOnly Cookie，绝不在 localStorage 或可被 JS 读取的位置持久化访问令牌。
Do 严格匹配 redirect_uri，逐字节比较而非前缀/子串匹配，防止开放重定向劫持。
Don't 不要使用 Implicit 流，令牌经 URL 片段返回会泄露到浏览器历史与 Referer 头，OAuth 2.1 已废弃该流程。
Don't 不要把 client_secret 放进前端，SPA / 移动端属于"公开客户端"，无法安全保管机密。
Don't 不要请求过宽的 scope，遵循最小权限原则——过宽的授权范围会放大令牌泄露后的爆炸半径。
常见陷阱 / Pitfalls

三个高频事故模式

以下陷阱都不违反协议本身，但实现疏忽会让一个"合规"的 OAuth 集成变成攻击面。

⚠ 陷阱 01 · Redirect URI 松散匹配
用 startsWith 或正则匹配 redirect_uri 是经典漏洞。攻击者注册 https://app.example.com.attacker.com/cb 即可接收授权码。必须全等比较，并维护已注册 URI 的精确白名单。
⚠ 陷阱 02 · Token 经 URL 泄露
Implicit 流把令牌放在 URL 片段（#access_token=...），会写入浏览器历史、通过 Referer 头外泄给第三方资源、被浏览器扩展读取。这正是它被 OAuth 2.1 废弃的根本原因——改用授权码 + PKCE。
⚠ 陷阱 03 · 把 OAuth 当登录认证
OAuth 解决的是授权（能做什么），不是认证（你是谁）。把 access_token 当作登录凭证会导致令牌窃取即账号接管。需要"登录"语义时使用 OpenID Connect，它在上层提供 id_token 承载用户身份断言。
