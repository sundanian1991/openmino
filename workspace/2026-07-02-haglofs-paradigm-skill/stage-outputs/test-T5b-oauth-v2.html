§ 01 · Foundation

委托授权，而非凭据共享。

OAuth 2.0 是一种授权协议：资源所有者向第三方应用授予对受保护资源的有限访问权限，全程不暴露自身凭据。它取代了"把密码交给客户端"这种危险的旧模式。

"The access token is a bearer instrument——持有即有效，签发即责任。"
RFC 6749 · §1.4 Access Token
01
四个角色

资源所有者、客户端、授权服务器、资源服务器。职责分离是安全的根基。

Resource Owner · Client · AS · RS
02
令牌即范围

每个 access_token 绑定 scope 与 expiry。最小权限原则由协议层强制，而非靠开发者自觉。

scope=read:profile&expires_in=3600
03
机密分级

Confidential 客户端持有 client_secret，Public 客户端用 PKCE 替代。机密性决定流程选型。

confidential | public
§ 02 · Grant Types

四种许可类型，各司其职。

选择错误的授权流程是 OAuth 实施中最常见的安全漏洞。下表按场景给出选型决策。

Grant Type Comparison
4 Flows · RFC 6749
许可类型 适用场景 客户端类型 推荐度
Authorization Code Web 应用、SPA（配合 PKCE）。服务端可安全存储 secret，授权码经前端中转但 token 直达后端。 Confidential + Public(PKCE) 推荐 · 首选
Client Credentials 机器对机器（M2M）。无最终用户参与，服务直接以自身身份获取 token。 Confidential 推荐 · M2M
PKCE Extension 移动端、SPA 等无法安全存储 secret 的公开客户端。用 code_verifier/code_challenge 替代 secret。 Public 推荐 · 公开客户端
Implicit / Password 已废弃。Implicit 易受 token 泄露，Password 暴露用户凭据。OAuth 2.1 移除。 — 废弃 · 勿用
§ 03 · Authorization Code Flow

五步握手，token 永不暴露给浏览器。

授权码流程的核心安全价值：前端只拿到一次性 code，真正的 access_token 在后端与授权服务器之间直接交换，前端历史记录与日志中均无 token 痕迹。

authorize-request.http HTTP · Step 1-2 
# ① 重定向用户到授权端点（浏览器跳转）
GET /authorize?response_type=code&
            client_id=s6BhdRkqt3&
            redirect_uri=https://app.example.com/cb&
            scope=read:profile write:posts&
            state=xyzABC123&
            code_challenge=E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM
Host: auth.example.com

# ② 用户同意后，授权服务器重定向回 redirect_uri，附带 code
302 Location: https://app.example.com/cb?code=SplxlOBeZQQYbYS6WxSbIA&state=xyzABC123
01
客户端发起 授权请求

客户端将用户重定向到授权服务器的 /authorize 端点，携带 client_id、scope、state 与 PKCE 的 code_challenge。

GET /authorize
02
用户 认证并同意

授权服务器展示同意页，用户登录并显式批准请求的 scope。state 参数在此回传，防 CSRF。

302 → redirect_uri?code=
03
客户端后端 交换 token

客户端后端用一次性 code、client_secret（或 PKCE 的 code_verifier）向 token 端点换取 access_token。此步在前端不可见。

POST /token · server-to-server
04
授权服务器 签发令牌

校验 code、secret/verifier 通过后，返回 access_token（短期）与可选 refresh_token（长期）。token 带 scope 与 expires_in。

200 {access_token, expires_in:3600}
05
客户端 访问资源

客户端携带 access_token 请求资源服务器。token 过期后用 refresh_token 静默续期，无需重新打扰用户。

GET /api · Authorization: Bearer
token-exchange.http HTTP · Step 3-4 
# ③ 后端用 code 换 token（机密请求，永不在前端执行）
POST /token HTTP/1.1
Host: auth.example.com
Content-Type: application/x-www-form-urlencoded

§ 06 · Best Practices

七条纪律，逐条可验收。

这是上线前的安全 checklist。每一条都对应一个可观测、可验收的工程动作。

OAuth 2.0 上线自检
公开客户端强制 PKCE。SPA/移动端一律用 code_challenge/code_verifier，禁止存储任何 secret。
access_token 有效期 ≤ 15 分钟。配合 refresh_token 静默续期；token 越短，泄露窗口越小。
state 参数必须是不可预测随机值。授权回调时严格比对，防 CSRF。
refresh_token 启用轮换（rotation）。每次使用后失效并签发新值，配合重用检测告警。
scope 遵循最小权限。按需申请，禁止 "all scopes" 兜底；用户可逐项撤销。
redirect_uri 精确白名单匹配。禁用通配符，禁用开放重定向；只允许 HTTPS（localhost 除外）。
client_secret 仅存后端/密钥管理服务。永不进前端代码、环境变量明文、日志或监控面板。
§ 07 · Common Pitfalls

三个高频陷阱，审计时必查。

这些不是理论问题——它们是真实生产事故的根因。

⚠ 陷阱 01 · 高危
redirect_uri 通配符匹配

允许 *.example.com 或前缀匹配会让攻击者用子域名劫持授权码。必须精确字符串匹配，每次新增回调地址走白名单审批流程。

⚠ 陷阱 02 · 高危
token 存 localStorage

localStorage 对任意 JS 可读，一次 XSS 即可窃取全量 token。SPA 应走 BFF（后端 for 前端）模式，token 存 HttpOnly Cookie。

◆ 陷阱 03 · 中危
忽略 token_type 校验

未校验 token_type=Bearer 与 alg 的实现可能接受任意格式令牌。资源服务器必须严格验证签名与类型。

◆ 陷阱 04 · 中危
refresh_token 永不过期

不设绝对过期时间的 refresh_token 等于永久凭据。应设 7-30 天绝对上限 + 滑动续期，超时强制重新认证。
