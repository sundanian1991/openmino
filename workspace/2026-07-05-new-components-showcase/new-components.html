Segmented Progress · 离散分段条

信息本质：把连续比例离散化成 N 格——"7/10"不是"70%"而是"7 格亮了"。读者一眼数格子，不需要读刻度。与 Seam Benchmark 的区别：Seam 回答"达到多少"，Segmented 回答"过了几格"。

元原则 · 当没有合适组件时

怎么自己造组件？

60 个登记组件不可能覆盖所有信息形态。当遇到没有合适组件的内容时——先问三个问题，再动手。这三个问题是从组件设计宪法推导出来的"自造准入门槛"。

01

这块信息要求读者做什么运算？

如果读者要在脑子里对比、排序、推算趋势、估算比例——画出来。如果只是读一个数字或一句话——不画，用文字。这是"要不要造组件"的第一道闸。

02

60 个组件里真的没有能做的吗？

先查选型决策树。editorial 组件（Stat Grid / Principle Cards / rank list）能完成的不用 chart。现有组件的变体或组合能解决的不用造新组件。只有当信息本质确实不在 60 个组件的覆盖范围内——才造。

03

自造组件守宪法的 4 条

如果决定造——必须守组件设计宪法的 4 条法则：信息本质驱动形态（从信息推导视觉，不是从好看推导）、每个元素挣到像素（减法优先）、字体角色是信息翻译器（Playfair/Mono/Inter 各司其职）、层次靠留白+1px线（不靠阴影/渐变/圆角）。自造组件的 class 名用 BEM kebab-case 自描述，radius 归入三级公约（结构件 0 / 内容卡 4 / 交互 8），深底继承 .section--dark 通用反相。

组件库不是封闭集合。60 个登记组件是 60 种已被验证的"信息→视觉"翻译方案——但它们不是唯一的翻译方式。当你理解了翻译的原理，你可以在没有现成组件时，自己推导出合适的视觉形态。这才是组件系统的真正价值。

"组件不是 UI 部件，是信息本质的翻译器。60 个登记组件是已验证的翻译方案——但不是唯一的翻译方式。"
