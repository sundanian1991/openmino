## 一、核心哲学层

### 1.1 Boil the Lake（把湖水煮干）

**本质**：AI 让"完整性"的边际成本趋近于零

| 传统思维 | gstack 思维 |
|----------|-------------|
| 80%覆盖就够了 | 剩下的20%只花几秒 |
| 先跳过测试 | 测试是最容易煮干的湖 |
| 后续PR再补 | 完整实现 = 几十行代码的差异 |

**压缩比参考表**：
```
脚手架/样板代码：100x（2天 → 15分钟）
测试编写：        50x（1天 → 15分钟）
功能实现：        30x（1周 → 30分钟）
Bug修复+回归：    20x（4小时 → 15分钟）
架构/设计：       5x（2天 → 4小时）
研究/探索：       3x（1天 → 3小时）
```

**关键洞察**：当你估算是"2周"，应该说"2周人力 / ~1小时 AI辅助"

### 1.2 Search Before Building（先搜后造）

**三层知识框架**：

| 层级 | 内容 | 价值 | 风险 |
|------|------|------|------|
| Layer 1 | 已验证的模式（distribution内） | 不要重复造轮子 | 偶尔质疑常规会有突破 |
| Layer 2 | 新流行/热门（需搜索） | 了解当前趋势 | 人类容易狂热，需批判 |
| Layer 3 | 第一性原理（原创推理） | 最有价值 | 需要深度思考 |

**Eureka 时刻**：
> 当第一性原理推理发现"常规做法是错的"
> 格式："EUREKA: 大家都做X是因为[假设]。但[证据]显示这是错的。[Y]更好因为[推理]"

## 二、核心方法层

### 2.1 YC Office Hours — 六步管线

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 1: Context Gathering                                 │
│  读CLAUDE.md/TODOS.md + git log + grep代码 + 问目标          │
├─────────────────────────────────────────────────────────────┤
│  Phase 2: Diagnostic (6个逼迫性问题)                         │
│  Q1需求真实性 → Q2现状 → Q3具体性 → Q4最小版本 → Q5观察 → Q6未来│
├─────────────────────────────────────────────────────────────┤
│  Phase 3: Premise Challenge                                 │
│  这是对的题目吗？如果什么都不做会怎样？已有代码能复用吗？       │
├─────────────────────────────────────────────────────────────┤
│  Phase 4: Alternatives Generation (强制)                    │
│  至少2-3个方案：最小可行版 / 理想架构版 / 创意侧向版           │
├─────────────────────────────────────────────────────────────┤
│  Phase 5: Design Doc Output                                 │
│  写文档存档，可供后续review使用                               │
├─────────────────────────────────────────────────────────────┤
│  Phase 6: Signal Synthesis                                  │
│  创始人信号评级 → 决定收尾力度                                │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 六个逼迫性问题（The Six Forcing Questions）

**核心原则**：
- 一次只问一个
- 推到答案具体、有证据、让人不舒服为止
- 舒适 = 没挖够深

| 问题 | 目的 | 好答案的特征 | 红旗信号 |
|------|------|-------------|----------|
| **Q1: 需求真实性** | 验证是否真有人要 | 具体行为、付费、扩展使用 | "很多人感兴趣"、waitlist数据 |
| **Q2: 现状** | 了解用户当前怎么解决 | 具体工作流、耗时、成本 | "没有解决方案" |
| **Q3: 绝望的具体性** | 找到最需要的人 | 姓名、职位、痛点、后果 | "中小企业"、"营销团队" |
| **Q4: 最小切入版本** | 找到最窄的楔子 | 几天能交付、有人愿付费 | "需要完整平台" |
| **Q5: 观察与惊讶** | 验证是否真的看过用户 | 具体的意外发现 | "问卷调查"、"按预期进行" |
| **Q6: 未来适配** | 判断长期价值 | 具体的世界变化+为什么更值钱 | "市场增长率20%" |

**智能路由**：
- 无产品阶段 → Q1, Q2, Q3
- 有用户 → Q2, Q4, Q5
- 有付费客户 → Q4, Q5, Q6
- 纯工程/基建 → Q2, Q4

### 2.3 18条CEO认知模式

**这些不是检查清单，是思维本能**

#### 决策类
1. **分类本能**（Bezos）：每个决策按"可逆性 × 量级"分类。大多数是双向门，快速决定。
2. **速度校准**（Bezos）：默认快。只对"不可逆+高量级"慢下来。70%信息足够决策。
3. **逆向反射**（Munger）：每个"怎么赢"都要问"什么会让我们失败"。
4. **聚焦即减法**（Jobs）：主要价值在于决定*不做什么*。从350产品减到10。

#### 战略类
5. **偏执扫描**（Grove）：持续扫描战略转折点、文化漂移、人才流失、流程代理病。
6. **时间深度**（Bezos）：以5-10年为弧度思考。用"80岁遗憾最小化"评估重大决策。
7. **创始人模式偏见**（Chesky/Graham）：深度参与不是微观管理——如果它扩展而非约束团队的思考。
8. **战时意识**（Horowitz）：正确诊断和平期vs战时。和平习惯杀死战时公司。
9. **意志力即战略**（Altman）：有意识地倔强。世界会让步于在一个方向足够久地推进的人。

#### 组织类
10. **人才优先序列**（Horowitz）：人才、产品、利润——永远是这个顺序。
11. **杠杆执念**（Altman）：找到小投入产生大产出的地方。技术是终极杠杆。
12. **等级即服务**：每个界面决策回答"用户应该先看到什么，第二，第三？"

#### 产品/设计类
13. **代理怀疑论**（Bezos Day 1）：我们的指标还在服务用户，还是已经变成自我指涉？
14. **叙事连贯性**：艰难决策需要清晰的框架。让"为什么"清晰可见，而非让所有人开心。
15. **勇气积累**：信心来自做出艰难决定之后，而非之前。"挣扎就是工作本身。"
16. **边缘案例偏执（设计）**：如果名字47字符？零结果？网络中途失败？
17. **减法默认**（Rams）："尽可能少的设计"。如果UI元素不挣得它的像素，砍掉。
18. **为信任设计**：每个界面决策要么建立要么侵蚀用户信任。

## 三、技术实现层

### 3.1 Markdown即代码

**核心设计**：用自然语言写逻辑，而非代码

```markdown
<!-- 示例：循环逻辑 -->
If the user disagrees with a premise:
  1. Ask why
  2. Update understanding
  3. Loop back to premise confirmation
```

**优势**：
- 可读性高
- 易于修改
- 不需要编程知识
- AI直接执行

### 3.2 智能体隔离

**只对两个skill使用独立Agent**：
- office-hours（避免情感参与影响客观性）
- plan-ceo-review（确保独立审查）

**其他skill直接在主线程执行**，避免上下文切换开销。

### 3.3 防幻觉设计

| 防护层 | 具体措施 |
|--------|----------|
| 爆炸半径限制 | 只输出文档，不直接改代码 |
| 人工校验 | 关键节点用 AskUserQuestion 强制确认 |
| 二元判断 | 用"是/否"替代模糊评分 |
| 状态持久化 | 写入文件，不依赖记忆 |

## 四、可迁移的模式

### 4.1 管线式设计

**gstack 模式**：Phase 1 → Phase 2 → Phase 3 → ...

**可迁移场景**：
- 供应商准入评估
- 供应商业绩诊断
- 项目可行性分析
- 人物观察建档

### 4.2 逼迫性问题集

**gstack 模式**：6个问题，智能路由

**可迁移场景**：
- 供应商沟通（替换为供应商相关的问题）
- 决策分析（替换为决策相关的问题）
- 人物观察（替换为行为分析问题）

### 4.3 信号采集与评级

**gstack 模式**：
- 每个问题采集特定信号
- 信号强弱决定后续处理方式
- 最终综合评级

**可迁移场景**：
- 供应商风险信号采集
- 人物行为信号采集
- 项目健康度信号采集
