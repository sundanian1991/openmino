## 1. 全局交互模型

### 1.1 主流程骨架

整个应用只有一条主线：

```
打开应用 → 顶部 Tab 选模式 → 录制 → 结果
                                        │
                                        └→ 保存 / 导出 / 再录一次
```

没有二级导航、没有复杂菜单。所有功能围绕"录一次"展开。这是自用工具，不需要登录、设置中心、帮助系统这类 SaaS 包袱。

### 1.2 模式 Tab 栏

顶部一排 Tab，横向排列：

```
┌──────────────────────────────────────────────────────────┐
│  语音工作站                                                │
│                                                          │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐           │
│  │ 训练  │ │ 报告  │ │ 纪要  │ │ 邮件  │ │ 笔记  │           │
│  └──────┘ └──────┘ └──────┘ └──────┘ └──────┘           │
│   ↑激活    ↑激活    灰色    灰色    灰色                   │
│   (MVP)   (MVP)   (后续)   (后续)   (后续)                │
└──────────────────────────────────────────────────────────┘
```

**Tab 状态规则：**
- **MVP 激活的模式**（训练、报告）：正常颜色，可点击，点击有过渡动画。
- **未激活的模式**（纪要、邮件、笔记）：灰色，可见但不可点击。悬停显示 tooltip：「即将推出」。让用户看到产品全貌，知道未来形态，但当前不开放。
- **切换模式**：横向滑动过渡（250ms），内容区整体平移。如果当前正在录制，切换 Tab 弹确认「正在录制，确定切换吗？丢弃当前内容」。

### 1.3 全局元素

无论在哪个模式、哪个状态，窗口顶部固定三块：

```
┌──────────────────────────────────────────────────────────┐
│ [应用名]        [模式 Tab 栏]              [AI 后端状态] [设置]│
└──────────────────────────────────────────────────────────┘
```

- **应用名**：左上角，应用 logo + 名称。
- **模式 Tab 栏**：居中或居左。
- **AI 后端状态**：右上角，显示当前用的是「云端」还是「本地 Ollama」，用一个圆点 + 文字。绿色 = 已连接，灰色 = 未配置/断开。点击进入设置面板。
- **设置**：齿轮图标，点开抽屉式设置面板（AI 配置、词库路径、模型路径）。

## 2. 训练模式完整用户旅程

### 步骤 1：进入界面（idle 状态）

**界面长什么样：**

```
┌──────────────────────────────────────────────────────────┐
│ 语音工作站    [训练] 报告  纪要  邮件  笔记      云端·已连接 ⚙│
├──────────────────────────────────────────────────────────┤
│                                                          │
│                    训练模式                                │
│            开口练习，即时看到你的表达习惯                     │
│                                                          │
│                                                          │
│                      ╭────────╮                           │
│                      │        │                           │
│                      │  ▶ 录  │   ← 大圆形录制按钮         │
│                      │        │                           │
│                      ╰────────╯                           │
│                                                          │
│                  点击开始录制（空格键）                      │
│                                                          │
│         最近 3 次训练：                                    │
│         · 7/13 晚 · 综合分 72                              │
│         · 7/12 早 · 综合分 68                              │
│         · 7/11 晚 · 综合分 75                              │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**元素说明：**
- 中央大圆形录制按钮（直径 120px），陶土橙底色，白色 ▶ 图标。
- 按钮下方一行辅助文字「点击开始录制（空格键）」。
- 底部「最近 3 次训练」：最近记录的入口，点击可回看历史报告。自用工具，历史就是简单列表。
- 空格键 = 全局快捷键，任何状态下空格都能开始/停止录制。

**用户动作：** 点击录制按钮，或按空格。

### 步骤 2：点击录制（initializing 状态）

**发生什么：**
1. 录制按钮立即变成「加载中」态（转圈），UI 即时反馈，不等 ASR。
2. 后台 ASR Engine.init()（首次）+ startStream()。
3. 请求麦克风权限（首次，系统弹窗）。
4. 初始化完成（通常 < 1s），进入 recording 状态。

**界面过渡：**
```
按钮: 大圆 ▶  →  大圆 转圈  →  界面切换到录制布局
                          ↑
                    通常 < 1s
```

**如果初始化失败**（无麦克风权限 / 模型缺失）：
- 按钮回到 idle 态。
- 顶部出一条 toast：「无法启动：xxx。请在设置中检查 xxx。」
- 不阻塞，用户可修复后再试。

### 步骤 3：录制中（recording 状态）

这是信息密度最高的界面，分四个区域：

```
┌──────────────────────────────────────────────────────────────┐
│ 语音工作站    [训练] 报告 ...                     ● REC 01:23 ⏹│
├──────────────────────────────────────────────────────────────┤
│ ┌──────────────────────────────────────────┐ ┌──────────────┐│
│ │ 统计条                                     │ │ AI 实时反馈   ││
│ │ 填充词 12  犹豫 5  笼统 3  有力 8           │ │              ││
│ └──────────────────────────────────────────┘ │ 你前面连续...  ││
│ ┌──────────────────────────────────────────┐ │ 嗯了三次,    ││
│ │                                            │ │ 建议在转折处  ││
│ │ 字幕区（大字）                              │ │ 直接停顿...  ││
│ │                                            │ │              ││
│ │ ...我觉得这个方案[可能]还不错，因为          │ │ (流式输出,   ││
│ │ 它[很好]地解决了[那个]问题，而且我们[大概]    │ │  逐字浮现)   ││
│ │ 可以在下个季度...                            │ │              ││
│ │  ↑灰色=partial    ↑高亮=词库命中            │ │              ││
│ │                                            │ │              ││
│ └──────────────────────────────────────────┘ │              ││
│                                              └──────────────┘│
│                                                            ■ │
│                                                            ■ │ ← 音量电平
└──────────────────────────────────────────────────────────────┘
```

**区域 1 — 顶部状态条：**
- `● REC` 红点 + 闪烁，表示正在录。
- `01:23` 录制时长，秒级跳动。
- `⏹` 停止按钮（也可点大圆/空格停止）。

**区域 2 — 统计条（左上）：**
四个数字：填充词、犹豫词、笼统词、有力表达。来自 Lexicon.analyze 的 stats。
- 数字变化时有滚动动画（300ms，见微交互章节）。
- 「有力表达」用鼠尾草绿数字，其余三类用对应高亮色。
- 这是训练模式的特征信息——让用户边说边意识到自己的习惯。

**区域 3 — 字幕区（主体，占最大面积）：**
- 大字号（3.5rem），高行高（1.8），居中。
- **partial 文本**（中间识别结果）：灰色（#B0AAA6），表示还没定稿，随识别实时变化。
- **final 文本**：主文字色（#2C2825），已确定。
- **词库高亮**：final 文本里的命中词按类型染色（annotated 字幕样式）：
  - 填充词 → 珊瑚红底（#D4847A，低透明）
  - 犹豫词 → 暖琥珀底（#E0A86B，低透明）
  - 笼统词 → 沙金底（#D4B860，低透明）
  - 有力表达 → 鼠尾草绿底（#7A9E7E，低透明）
- 高亮是 inline 的，不换行打断。新句子出现时淡入 + 微微上移。

**区域 4 — AI 实时反馈（右侧栏）：**
- 来自 AIBackend.streamFeedback()，逐 token 流式输出。
- 卡片样式，标题「实时反馈」。
- 文字逐字浮现（打字机效果但用透明度淡入更柔和）。
- 反馈内容是"教练式"提示，如「注意：连续 3 个'嗯'，试着在句末停顿替代」。
- 录制中持续累积，可滚动查看。

**区域 5 — 音量电平（右下）：**
- 小竖条，实时反映麦克风音量。让用户确认麦克风在收音。过小会提示「麦克风音量偏低」。

### 步骤 4：停止录制（processing 状态）

**用户动作：** 点 ⏹ / 点大圆 / 按空格。

**发生什么：**
1. ASR.stopStream()，拿到最终完整文本。
2. 界面切换到 processing 态：字幕区变成「正在分析你的表达…」，大圆变加载动画。
3. 后台：Lexicon.analyze(完整文本) → AIBackend.generateReport(text, stats)。
4. AI 生成需要几秒到十几秒（取决于云/本地）。
5. 完成后进入 result 状态。

**界面：**
```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│                    正在生成报告…                            │
│                                                          │
│                      ╭────────╮                           │
│                      │  ◐     │   ← 加载动画               │
│                      ╰────────╯                           │
│                                                          │
│              已识别 423 字 · 正在分析表达质量                │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**如果失败**（AI 接口报错）：
- 降级：仍显示基于词库的本地分析结果（统计 + 高亮原文）。
- toast 提示「AI 反馈暂时不可用，已显示本地分析。可在设置检查 AI 配置。」
- 自用工具的容错原则：本地能给的先给，不因为云端挂了就全黑。

### 步骤 5：结果视图（result 状态）

训练报告，6 个维度：

```
┌──────────────────────────────────────────────────────────────┐
│ 训练报告 · 7/14 19:42 · 423字 · 3分21秒      [再录一次] [保存] │
├──────────────────────────────────────────────────────────────┤
│ ┌──────────────────┐ ┌─────────────────────────────────────┐│
│ │                  │ │ 总评                                 ││
│ │     雷达图        │ │ 你的表达整体清晰，逻辑通顺，但填充词   ││
│ │  清晰度  ●85      │ │ 偏多（12 次），尤其在句首。犹豫表达    ││
│ │  简洁度  ●70      │ │ 集中在结论部分…                      ││
│ │  自信度  ●72      │ │                                     ││
│ │  结构性  ●80      │ │ 做得好的地方                          ││
│ │  情绪稳定 ●88     │ │ ✓ 开门见山，前 15 秒就点明主题        ││
│ │  说服力  ●75      │ │ ✓ 用了具体数据支撑（"增长 30%"）       ││
│ │                  │ │ ✓ 结论明确                            ││
│ │   综合 78         │ │                                     ││
│ │                  │ │ 可以改进                              ││
│ │                  │ │ △ 填充词 12 次（嗯 7 / 那个 5）        ││
│ │                  │ │ △ 结论用了"可能""大概"，削弱确定性     ││
│ │                  │ │ △ "很好""不错"等笼统评价过多           ││
│ └──────────────────┘ │                                     ││
│ ┌──────────────────────────────────────────────────────────┐│
│ │ 原文回顾（带标注）                              [展开/收起] ││
│ │ ...我觉得这个方案<可能>还不错，因为它<很好>地解决了<那个>... ││
│ └──────────────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────────────┘
```

**区域说明：**
- **顶部**：报告元信息（时间、字数、时长）+ 操作按钮（再录一次、保存）。
- **左侧雷达图**：6 维度得分（清晰度、简洁度、自信度、结构性、情绪稳定、说服力），中心显示综合分。雷达图用 SVG，维度线用分割线色，得分区域用陶土橙低透明填充。
- **右侧文字**：
  - 总评：一段话总结。
  - 做得好的地方：绿色 ✓ 列表。
  - 可以改进：琥珀色 △ 列表，每条关联到词库数据（具体词、次数）。
- **底部原文回顾**：完整转录文本，词库命中处带颜色标注，可展开收起。让用户对照"具体哪句话有问题"。

### 步骤 6：保存

**用户动作：** 点击「保存」。

**发生什么：**
- 报告以 JSON + 渲染后的 Markdown 双份存到本地数据目录（Electron app.getPath('userData')）。
- toast：「已保存到历史」。底部历史列表多一条。
- 「再录一次」：回到 idle 状态，清空当前内容。

## 3. 文档模式（正式报告）完整用户旅程

### 步骤 1：进入界面（idle 状态）

**与训练模式的差异：更干净，没有统计和反馈预告。**

```
┌──────────────────────────────────────────────────────────┐
│ 语音工作站   训练 [报告] 纪要  邮件  笔记        云端·已连接 ⚙│
├──────────────────────────────────────────────────────────┤
│                                                          │
│                    正式报告                                │
│            把你的口头汇报变成可交付的文档                     │
│                                                          │
│                                                          │
│                      ╭────────╮                           │
│                      │  ▶ 录  │                           │
│                      ╰────────╯                           │
│                                                          │
│           讲清楚：背景、进展、数据、结论、下一步              │
│                                                          │
│              最近报告：                                    │
│              · 7/13 · 季度进展汇报（820 字）                │
│              · 7/10 · 项目复盘（1150 字）                  │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**元素说明：**
- 大圆录制按钮同训练模式。
- 按钮下方是"提示讲什么"的引导（正式报告的结构要点），帮助用户讲全。
- 最近报告列表，点击可回看。

### 步骤 2：点击录制（initializing）

同训练模式。按钮转圈 → ASR 初始化 → 进入录制。

### 步骤 3：录制中（recording 状态）

**关键差异：干净字幕，无任何分析干扰。** 用户专注"把内容讲清楚、讲完整"，不需要分心看自己的填充词。

```
┌──────────────────────────────────────────────────────────────┐
│ 语音工作站   训练 [报告] ...                      ● REC 02:45 ⏹│
├──────────────────────────────────────────────────────────────┤
│ ┌──────────────────────────────────────────────────────────┐ │
│ │                                                            │ │
│ │                                                            │ │
│ │              这季度我们主要推进了三个方向，                  │ │
│ │              第一是用户增长，第二是产品迭代，                │ │
│ │              第三是团队建设...                               │ │
│ │                                                            │ │
│ │              (纯文本字幕, 无高亮, 大字, 居中)               │ │
│ │                                                            │ │
│ │                                                            │ │
│ └──────────────────────────────────────────────────────────┘ │
│                                                            ■ │
└──────────────────────────────────────────────────────────────┘
```

**元素说明：**
- **只保留字幕区**，占满中央。没有统计条、没有反馈栏。
- 字幕是 clean 样式：纯文字，partial 灰、final 黑，**无词库高亮**。
- 顶部仍有 REC + 时长 + 停止按钮。
- 右下音量电平保留。

**为什么这么干净：** 文档模式的用户心智是"我在口述一份文档"，不是"我在练习表达"。任何额外的分析信息都是噪音。让 ASR 忠实记录，剩下交给 AI 后处理。

### 步骤 4：停止 → AI 处理（processing 状态）

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│                 正在生成正式报告…                            │
│                                                          │
│                      ╭────────╮                           │
│                      │  ◐     │                           │
│                      ╰────────╯                           │
│                                                          │
│           已识别 820 字 · 正在转换为正式文档                  │
│              · 理清结构                                     │
│              · 规范用语                                     │
│              · 标注改动                                     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

后台：AIBackend.generateDocument(text, 'formal-report') → { content, changes, summary }。

### 步骤 5：三栏对比结果视图（result 状态）

**这是文档模式的核心，也是产品差异化的载体。**

```
┌──────────────────────────────────────────────────────────────┐
│ 正式报告 · 7/14 19:50 · 820字→780字          [再录] [导出 ▼]  │
├──────────────────────────────────────────────────────────────┤
│ ┌──────────────┬──────────────┬──────────────────────────┐  │
│ │ 原文标注      │ 正式文档      │ 变更说明                  │  │
│ │ (你说了什么)  │ (改成什么样)  │ (为什么改)  [8 处]        │  │
│ ├──────────────┼──────────────┼──────────────────────────┤  │
│ │              │              │                          │  │
│ │ 这季度我们    │ 一、本季度    │ ① 结构化                 │  │
│ │ 主要推进了    │ 重点工作进展  │ 原文: "这季度我们主要     │  │
│ ─[①]────────  │              │   推进了三个方向…"        │  │
│ │ 三个方向，    │ (一) 用户    │ 改为: 标题+分点           │  │
│ │ 第一是用户    │ 增长...      │ 原因: 口语无层次,         │  │
│ │ 增长，第二    │              │       加标题便于阅读       │  │
│ │ 是产品...    │              │                          │  │
│ │              │              │ ② 删冗余                  │  │
│ │ ┊(高亮块)    │ ┊(高亮块)    │ 原文: "...然后就是说..."  │  │
│ │              │              │ 改为: 删除                │  │
│ │              │              │ 原因: 连续填充词...        │  │
│ │              │              │                          │  │
│ └──────────────┴──────────────┴──────────────────────────┘  │
│                                                              │
│ ┌──────────────────────────────────────────────────────────┐│
│ │ 一句话总结: 将口述调整为结构化季度报告, 补充标题与分点      ││
│ └──────────────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────────────┘
```

**三栏联动（核心交互）：**
- 点击右栏「变更说明」的任意一条（如 ①）：
  - 左栏「原文标注」对应的句子高亮（加粗 + 背景色块）。
  - 中栏「正式文档」对应的段落高亮。
  - 视图自动滚动让两边都可见。
- 反向也可：点击左栏某句，右栏自动定位到相关的变更条目（如有）。
- 这种联动让用户清晰看到"AI 改了哪里、为什么"，把文档生成变成一次教学。

**左栏 — 原文标注：**
- 完整转录文本。
- 被 changes 涉及的句子用背景色块标出（低透明陶土橙），对应右栏的变更编号。
- 非改动部分正常显示。

**中栏 — 正式文档：**
- AI 生成的 content，按 Markdown 渲染（标题、列表、加粗）。
- 可直接编辑（点击进入编辑态，光标定位）。
- 对应 changes 的段落同样用色块标注，编号一致。

**右栏 — 变更说明：**
- changes 数组逐条展示，编号 ①②③…
- 每条：类型标签（结构化/删冗余/规范用语/清晰化/去填充）+ 原文片段 + 改后片段 + 原因。
- 类型标签用不同颜色（结构化=陶土橙，删冗余=珊瑚红，规范用语=鼠尾草绿，清晰化=暖琥珀，去填充=沙金）。
- 标题旁显示总数「[8 处]」。

**底部：**
- 一句话总结（summary），浅色卡片。

### 步骤 6：编辑 + 导出

**编辑：**
- 中栏「正式文档」可点击编辑，所见即所得改文字。
- 编辑后 changes 联动失效（因为用户改过了），右栏标注「已手动编辑，变更说明仅供参考」。
- 左栏原文不可编辑（它是事实记录）。

**导出（点 [导出 ▼] 下拉）：**
- Markdown（.md）
- 纯文本（.txt）
- 复制到剪贴板
- 导出包含三栏的「对比包」（一个 .md，含原文、正式文档、变更说明三段）—— 适合复盘存档。

## 4. 状态机

每个模式内部是一个状态机。状态名统一，转换条件清晰。

### 4.1 通用状态机

```
                 ┌──────────────────────────────────────┐
                 │                                      ▼
   ┌────────┐  开始   ┌──────────────┐  初始化完成  ┌──────────┐
   │  idle  │────────▶│ initializing │────────────▶│ recording │
   └────────┘         └──────────────┘             └─────┬────┘
       ▲                    │                            │ 停止
       │           初始化失败│(回到idle+toast)            ▼
       │                    │                     ┌────────────┐
       │                    │                     │ processing │
       │                    │                     └─────┬──────┘
       │                    │                           │ 完成
       │                    │                           ▼
       │                    │                     ┌──────────┐
       └─── 再录一次 ──────────────────────────────│  result  │
                             │                           │
                             │                  失败(降级) │
                             │                     ┌───────┴────┐
                             │                     │ result     │
                             │                     │ (本地降级) │
                             │                     └────────────┘
                             ▼
                        错误态(回到idle)
```

### 4.2 状态定义表

| 状态 | 含义 | 可执行操作 | UI 关键元素 |
|------|------|-----------|------------|
| `idle` | 空闲，等待录制 | 开始录制（点按钮/空格） | 大圆录制按钮、历史列表 |
| `initializing` | ASR 初始化 + 麦克风建立 | 取消（可选） | 按钮加载动画 |
| `recording` | 正在录制，实时识别 | 停止（点按钮/空格/⏹） | REC 指示、字幕、统计/反馈（按模式） |
| `processing` | 录制后处理（AI 分析/生成） | 取消（可选，回 idle） | 加载动画、进度提示 |
| `result` | 展示结果 | 再录一次、保存、导出、编辑 | 报告 / 三栏对比 |

### 4.3 转换条件明细

| 从 → 到 | 触发 | 前置条件 | 失败处理 |
|---------|------|---------|---------|
| idle → initializing | 点录制按钮 / 空格 | AI 后端已配置（否则提示去设置） | — |
| initializing → recording | ASR init + stream 就绪 | 麦克风权限已授予 | 失败→idle + toast |
| recording → processing | 点停止 / 空格 / ⏹ | — | — |
| processing → result | AI 返回 | — | AI 失败→result(降级,仅本地分析) |
| result → idle | 点「再录一次」 | 确认丢弃（如已编辑） | — |
| 任意 → idle | 错误恢复 | — | toast 说明 |

### 4.4 跨模式切换的约束

- **recording 状态下切换模式**：弹确认对话框「正在录制，切换将丢弃当前内容。确定？」确认→停止并丢弃→切到新模式 idle；取消→停留。
- **processing 状态下切换模式**：禁止切换，Tab 暂时禁用，提示「正在处理，请稍候」。
- **result 状态下切换模式**：允许，但若结果未保存，弹确认。

## 6. 微交互

### 6.1 录制按钮动画（三态）

**idle 态：**
- 圆形，陶土橙底，白色 ▶。
- hover：scale(1.03)，250ms ease-out，阴影加深。
- active（按下）：scale(0.97)，150ms。

**recording 态（脉冲）：**
- 点击后先快速变形：圆形 → 内部 ▶ 变成白色方块（停止图标）。
- 外圈出现脉冲光环：
  - 周期 2s，无限循环。
  - 光环从按钮边缘向外扩散，scale 1.0 → 1.4，opacity 0.6 → 0。
  - 光环颜色：珊瑚红（#D4847A）低透明。
- 按钮本体轻微呼吸：scale 1.0 → 1.05 → 1.0，2s 周期，与光环同步。
- 配合顶部 `● REC` 红点闪烁（1s 周期，opacity 1 → 0.3）。

**processing 态：**
- 按钮区域显示圆形加载动画：边框旋转（800ms 一圈），陶土橙。

### 6.2 字幕出现动画

每段新 final 文本进入字幕区：
- 淡入 + 上移：`opacity 0 → 1`，`translateY(4px → 0)`，200ms ease-out。
- 不用打字机逐字（太碎），整段淡入更柔和。
- partial → final 的转换：灰色文字颜色过渡到主色（150ms）。

### 6.3 数字变化动画（统计条）

统计数字从 N 变到 N+1：
- 旧数字向上滑出 + 淡出，新数字从下方滑入 + 淡入，300ms ease-out。
- 配合数字颜色的瞬时高亮（变深一下再恢复，150ms），强化"刚变化"的感知。

### 6.4 模式切换过渡

切换 Tab 时内容区过渡：
- 旧模式内容向左滑出 + 淡出（150ms），新模式内容从右滑入 + 淡入（250ms）。
- 顶部 Tab 激活态指示器（下划线或背景块）横向滑动到新 Tab（250ms ease-out）。
- 录制中切换（确认后）同样适用。

### 6.5 三栏联动高亮

点击右栏变更条目：
- 左栏、中栏对应处背景色块淡入（200ms），色块是陶土橙低透明（rgba(196,149,106,0.15)）。
- 被高亮的内容自动滚动到视口中央（平滑滚动，400ms）。
- 取消选中（再点一次或点空白）：色块淡出（150ms）。

### 6.6 其他细节

- **按钮 hover**：所有可点击元素 hover 有轻微背景色变化（150ms），不用突兀的颜色跳变。
- **卡片 hover**：阴影从 shadow-sm 过渡到 shadow-md（200ms）。
- **toast 出现**：从顶部下滑 + 淡入（250ms），4 秒后上滑 + 淡出（200ms）。
- **加载占位**：AI 生成期间，结果区可显示骨架屏（灰色块脉动，1.5s 周期），而非纯转圈，让用户预知结果布局。
- **录音电平**：右下竖条高度随音量实时变化（无延迟，直接映射），颜色从鼠尾草绿（正常）→ 暖琥珀（偏大）→ 珊瑚红（爆音）。

### 6.7 动效原则

1. **ease-out 为主**：元素进入用 ease-out（快速出现再缓缓停），符合自然预期。
2. **短促优先**：过渡默认 150-250ms，超过 400ms 的只用于大区域切换。
3. **克制**：不加无意义弹跳、旋转。这是工具，不是玩具。动效服务于"反馈发生了什么"。
4. **可关闭**：设置里提供「减少动效」开关（respect `prefers-reduced-motion`），无障碍友好。
