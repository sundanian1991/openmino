# 定价委员会机制建设汇报

**汇报人**：供应商管理岗
**汇报对象**：数据科技业务部负责人 C2
**时间**：2026-02-27

---

## 一、为什么要建这个机制

### 当前痛点

| 问题 | 具体表现 | 影响 |
|------|---------|------|
| **定价随意** | 策略/量化单方面决策，无供应商参与 | 供应商抵触，执行困难 |
| **降幅失控** | 单次调整曾达50%+ | 坐席收入骤降，离职率飙升 |
| **事后救火** | 供应商管理岗只有安抚权 | 问题重复发生，成本高 |
| **无复盘** | 调完就算，不追踪效果 | 无法积累经验 |

### 建设目标

从**事后救火**转向**事前防火**：
- ✅ 有约束的决策（单次≤30%、次月生效）
- ✅ 有代表的委员会（业务+策略+量化+督导）
- ✅ 有复盘的闭环（每次追踪30天）

---

## 二、机制核心设计

### 组织架构

```
5人委员会 + C2终审
├── 秘书长：供应商管理岗（组织、追踪、归档）
├── 委员：业务督导（坐席收入影响）
├── 委员：量化代表（数据测算）
├── 委员：服务组C3（职场管理）
├── 委员：策略组C3（定价策略）
└── 终审：C2（每项调整最终审批）
```

### 决策权限

| 调整类型 | 审批流程 | 通过条件 |
|---------|---------|---------|
| 常规调整（≤15%） | 委员会 → C2 | 3/5通过 + C2终审 |
| 涨价调整 | 委员会 → C2 | 3/5通过 + C2终审 |
| 重大调整（>15%） | 委员会 → C2 | 4/5通过 + C2终审 |

### 三条红线

| 红线 | 标准 | 例外处理 |
|------|------|---------|
| **单次降幅** | ≤30% | 超过分2次执行 |
| **生效时间** | 次月1日 | 避免突发冲击 |
| **调整频次** | 3个月/次 | 特殊调整单独申请 |

### 完整流程

```
提案（T-5）→ 评审（T-2）→ 终审（T-1）→ 生效（次月1日）→ 宣贯 → 复盘（+30天）
```

## 三、机制亮点

### 1. 风险前置防控

**早期预警信号**（出现任一立即暂停）：
- 投诉率上升 >20%
- 供应商利润 <5%
- 坐席收入不达标占比 >10%
- 离职率 >行业平均2倍

**应急预案**：
群体性事件、投诉率>50%、离职率>20% → 24小时内紧急会议

### 2. 供应商反馈通道

正式渠道让供应商对定价调整提出异议：
- 时机：方案确定后、生效前
- 流程：接收→审查→评估→讨论→反馈
- 结果：采纳/部分采纳/维持原案/延期执行

### 3. 每次必有复盘

调整生效30天后出复盘报告：
- 执行效果 vs 预期
- 供应商/坐席反馈汇总
- 改进建议纳入下次参考

- ✅ **风险可控**：前置预警机制，降低意外发生

### 业务收益

| 收益 | 说明 |
|------|------|
| **决策质量** | 有充分数据支撑的科学决策 |
| **业务稳定** | 减少因定价问题导致的波动 |
| **关系改善** | 供应商满意度提升，合作更稳定 |
| **成本可控** | 避免因剧烈调整导致的隐性成本 |

## 八、Q&A

### Q1: "这个机制是否会削弱我的决策权？"

**A**: 不会。您仍掌握最终决策权，机制提供更科学的前置评估。每项调整都需要您终审，但您将获得更充分的信息支撑。

### Q2: "如何确保机制不会影响业务灵活性？"

**A**:
- 保留您的紧急决策权，可绕过常规流程
- 设置"临时会议"处理紧急需求
- 3个月调整频次已给予足够灵活性

### Q3: "如何衡量机制效果？"

**A**:
- 量化指标：投诉率、离职率、供应商利润、业务产能
- 时间节点：每次调整后30天复盘
- 闭环反馈：复盘结果纳入下次调整参考
