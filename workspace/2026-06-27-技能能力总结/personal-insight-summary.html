Prompt Skill · v1.0.0

personal-insight
summary

多源信息融合洞察——把聊天、会议、邮件串成完整故事线，输出结构化信息简报。

核心突破

跨源关联分析——不是简单汇总，而是将京ME消息、慧记会议、Outlook 邮件三个数据源交叉关联：按项目聚类、识别相关事项、判断优先级、提取决策与分歧、生成待办。最终输出只保留证据最强的要点。

数据源配置

用户通过 JSON 配置指定要汇总的数据源。未提供配置时，默认采集今天的京ME、邮件、慧记三类数据。

{
  "sources": [
    {
      "skillId": "jdme-local-tools",
      "prompt": "只读取群号 10230120223、10228173601，
                 以及群名为'体验技术 - 产品群'的今日京ME消息"
    },
    {
      "skillId": "read-joymeeting",
      "prompt": "读取会议记录"
    },
    {
      "skillId": "read-email",
      "prompt": "读取工作邮件"
    }
  ]
}

时间范围从各 source 的 prompt 中识别。未指定时默认使用昨天 0:00-24:00。

工作流程

01

第一步：收集各数据源信息

并行调用所有数据源（提高效率），某个失败不影响其他。失败时标注「该数据源未获取，原因：xxx」。

京ME 采集边界
prompt 含「只读取」「指定群」「群号」等限定时进入精确会话模式——只从指定群号/群名读取，不扩大到其他群或私聊。未命中标注未获取。
邮件筛选原则
保留：直接工作邮件、项目讨论、领导/关键干系人、需回复处理、影响行动的系统邮件。
过滤：CI/Pipeline 通知、营销、newsletter 等无待办信息。
02

第二步：整合分析

采集完原始数据后，执行以下分析流水线：

  1. 时间过滤：按配置的时间范围过滤所有数据
  2. 京ME指定群聊过滤：按指定 sessionId/会话名二次过滤
  3. 人名映射表：从消息中提取 ERP → 中文名 映射
  4. 身份解析：从 sessionId（格式 erp1:ee:erp2:ee）提取对方 ERP
  5. 相关性分级：
强相关 用户发言/被@/私聊/直接邮件/参会 中相关 项目群相关讨论 弱相关 大群普通讨论
  1. 项目聚类：按项目维度归类（从群名/邮件主题/会议标题提取）
  2. 优先级判断：有人@ → 高优；要求回复/审批 → 高优；风险/阻塞 → 高优
  3. 决策提取：从会议和群聊提取已达成的决策
  4. 分歧识别：提取尚未达成一致的讨论点
  5. 风险识别：提取技术/进度/资源风险
  6. 待办生成：综合所有源提炼优先处理事项
03

第三步：输出

严格按固定 6 个一级小标题输出。每要点采用「结论 + 摘要上下文」两层结构。只输出证据强度为「强」的信息。

输出模板（固定 6 板块）

昨天你主要参与了 {N} 个项目：

每项含：项目名、摘要上下文（时间/人名/来源/内容/你的参与证据）

今天建议优先处理 {N} 件事：

每项含：具体行动+原因、摘要上下文（人/来源/当前状态）。不超过 5 条。

重点关注：

值得持续关注的事项/进展/协作动态。每项含摘要上下文。

风险提醒：

风险描述+建议处理方式、来源/原始表述/影响范围。

已定决策：

决策内容+来源/提出者/最终结论。

待定问题：

未解决问题/分歧/待确认事项，含各方观点和当前状态。

每个板块不超过 7 条；无内容写「暂无」。不输出额外一级标题。条目数用「及其他 N 条」收尾。

摘要上下文撰写规则

规则说明
忠实原文人名/项目名/数字/结论必须来自原始数据，不推测不补全
对话脉络优先保留关键轮次「谁说了什么→谁回应→结论」脉络
保留用户回复时间过滤后原始数据中用户本人的发言/回复必须保留
标注来源每段上下文末尾标注「——来源：{渠道}{具体位置}」
长度控制每个要点上下文不超过 5 条 bullet
时间标注上下文首条标注时间（如「6/5 晚」「昨天 14:30」）

核心约束

不编造人名、项目名、事件
项目名用原始叫法，不自行重命名
人名用中文姓名（有则），否则用 ERP
每个板块不超过 7 条；「今天建议优先处理」不超过 5 条
只输出固定 6 个一级标题，不新增/删除/改名
最终输出只保留「强」证据要点；中/弱仅作内部分析参考
输出中不得出现「证据强度」「强/中/弱」等内部字段
不输出 token/cookie/密码等敏感信息
摘要上下文对话保留原话关键词和语气，不过度书面化

无信息兜底

所有数据源均无目标时间范围内的有效工作信息时（全部为空、仅系统通知、采集全失败），不输出空板块模板，改用精简兜底输出，列出可能原因和各数据源状态。

  • 周末/节假日 → 优先提示「目标时间是{周末/节假日}」
  • 工作日无数据 → 提示「可能休假/出差/配置范围过窄」
  • 少量非工作消息 → 仍判为无有效信息，注明「有 N 条，均为非工作内容」
