## 一、13个SVG全貌

### 1.1 按复杂度分类

| 复杂度 | Path数量 | 数量 | 类型 | 手工难度 |
|--------|----------|------|------|----------|
| **超简单** | 1 | 5个 | UI图标（箭头、搜索、关闭） | ✅ 极易 |
| **简单** | 9-29 | 3个 | 餐厅、景点、扇子图标 | ✅ 容易 |
| **中等** | 26-29 | 1个 | 扇子细节版 | ✅ 可行 |
| **复杂** | 61-88 | 4个 | 猫咪主体图 | ⚠️ 需要参考 |
| **总计** | - | 13个 | - | - |

### 1.2 关键发现

**1. 超简单图标（1个path）不值得用API**
- 它们只是普通线条，手工写2行代码就行
- 示例：`<path d="M12 5v14M12 5l-6-6" stroke="currentColor" stroke-width="1.5"/>`
- **结论**：这些应该建立代码片段库，不是API调用

**2. 简单图标（9-29个path）手工完全可行**
- 结构清晰，技法单一
- 主要是：填充层 + 简单描边
- **结论**：这些应该建立参数化模板

**3. 复杂猫咪图（61-88个path）才是API的价值所在**
- 贝塞尔曲线复杂，手工调参成本高
- **但**：如果我们提取了它们的结构模式，手工复制是可行的
- **结论**：这些需要API，或者建立高级模板库

## 二、手工vsAPI的界限

### 2.1 手工可以做到的

**简单图标**（9-29个path）：
```xml
<!-- 筷子图 - 26个path -->
<svg viewBox="0 0 250 250">
  <!-- 主体填充 -->
  <path d="..." fill="#FEFFFE" stroke="#140A02" stroke-width="1.4"/>
  <!-- 细节描边 -->
  <path d="..." stroke="#140A02" stroke-width="0.7"/>
  <!-- 点缀 -->
  <path d="..." fill="#B93921"/>
</svg>
```

**手工成本**：
- 基础形状：5-10分钟
- 细节调整：10-15分钟
- **总计：20-30分钟/个**

### 2.2 API更有价值的

**复杂猫咪**（61-88个path）：
- 每个path有20-40个控制点
- 需要精确的手绘感（stroke-linecap, stroke-linejoin）
- 多层叠加，需要艺术判断

**但如果**：
- 我们建立了猫咪的"结构模板"（头部、身体、四肢的相对位置）
- 建立了"贝塞尔曲线库"（常用的弧线、S形）
- **那么**手工复制也是可行的，只是时间成本高（1-2小时）

## 三、模板化策略

### 3.1 三层模板体系

**Layer 1：代码片段库（超简单）**
```javascript
// 1个path的UI图标
const icons = {
  arrow: `<path d="M12 5v14M12 5l-6-6" stroke="currentColor" stroke-width="1.5"/>`,
  search: `<circle cx="11" cy="11" r="8"/><path d="m21 21-4.35-4.35"/>`,
  close: `<path d="M18 6L6 18M6 6l12 12"/>`
};
```

**Layer 2：参数化模板（简单-中等）**
```xml
<!-- 餐厅图标模板 - 9个path -->
<svg viewBox="0 0 250 250">
  <!-- {{主体路径}} -->
  <path d="{{主体贝塞尔}}" fill="{{色彩}}" stroke="{{线条色}}"/>
  <!-- {{细节路径}} -->
  <path d="{{细节贝塞尔}}" stroke="{{线条色}}" stroke-width="{{粗细}}"/>
</svg>
```

**参数**：
- `主体贝塞尔`：描述形状（圆形、方形、不规则）
- `色彩`：预定义色卡
- `线条色`：#0B0800（墨黑）或#140A02（深褐）
- `粗细`：0.7-1.4px

**Layer 3：结构模板（复杂）**
```xml
<!-- 猫咪模板 - 60+个path -->
<svg viewBox="0 0 200 200">
  <!-- 头部区域 -->
  <g id="head">
    <path d="{{耳朵左}}" fill="{{色彩}}"/>
    <path d="{{耳朵右}}" fill="{{色彩}}"/>
    <path d="{{脸部}}" fill="#FEFFFA" stroke="#0B0800"/>
    <!-- 眼睛 -->
    <circle cx="{{左眼x}}" cy="{{左眼y}}" r="{{大小}}" fill="#0B0800"/>
    <circle cx="{{右眼x}}" cy="{{右眼y}}" r="{{大小}}" fill="#0B0800"/>
    <!-- 鼻子 -->
    <path d="{{鼻子}}" fill="#BA3420"/>
    <!-- 嘴巴 -->
    <path d="{{嘴巴}}" stroke="#BA3420"/>
  </g>

  <!-- 身体区域 -->
  <g id="body">
    <path d="{{主体}}" fill="#FEFFFA" stroke="#0B0800" stroke-width="1.5"/>
    <path d="{{花纹}}" fill="#BA3420" fill-opacity="0.1"/>
  </g>

  <!-- 四肢区域 -->
  <g id="limbs">
    <!-- 左前腿 -->
    <path d="{{左前}}" fill="#FEFFFA"/>
    <!-- 右前腿（抬起的爪子） -->
    <path d="{{右前}}" fill="#FEFFFA"/>
    <!-- 后腿 -->
    <path d="{{左后}}" fill="#FEFFFA"/>
    <path d="{{右后}}" fill="#FEFFFA"/>
  </g>
</svg>
```

### 3.2 关键洞察

**复杂度不是binary的**

即使是88个path的猫咪图，也可以分解为：
- 5个主要部分（头、身、四肢、尾巴、装饰）
- 每个部分10-20个path
- **总计**：25-30个"组件"，不是88个独立手工path

**如果建立组件库**：
- 头部组件（10-15个path）
- 身体组件（8-12个path）
- 腿部组件（每个3-5个path）
- **那么**手工组装是可行的，只是需要时间

### 6.2 评估标准（binary 检查清单）

```
□ 双层分离（填充层 + 描边层）
□ stroke-linecap="round" 全局应用
□ 色彩系统（东京猫：#FEFFFA/#BA3420/#0B0800；李诞虾：#D6654B/#1A1612）
□ path 数量合理（简单10-30，复杂60-80）
□ 结构清晰（头部/身体/四肢分组）
□ 中文字符清晰可读（如果需要）
□ 视觉平衡（左右对称、重心稳定）

通过率 7/7 = 100分
每少1项扣15分
```
