汇总层 · 个人工作日报

personal-insight-summary

并行调度京ME、邮件、慧记三个数据源采集全文，做项目聚类、相关性分级（强/中/弱）、决策·风险·待办抽取，产出一份固定6段结构的个人工作日报——基于全文证据，不编造。

版本 1.0.0 类型 Prompt 技能（编排上层技能） 并行调用 3 个数据源 输出 固定 6 段日报

五个采集技能里最上层的汇总器。它不直接读文件，而是用一个 prompt 编排下面三个采集技能并行采集，然后做前置过滤（时间·指定群·人名映射·相关性分级·证据强度）→ 整合分析（项目聚类·优先级·决策·分歧·风险·待办）→ 固定6段输出。关键设计：最终输出只保留强证据要点，中/弱只作内部分析，不输出。

03三步工作流

第一步：收集各数据源信息

对每个 source：确认技能可用 → 合成执行指令（prompt + 时间范围）→ 调用技能 → 记录结果与失败原因。尽可能并行调用，某源失败不影响其他源，失败时标注"该数据源未获取，原因：xxx"。

京ME精确会话模式 prompt 出现"只读取/指定群/群号/群名"等限定时进入精确会话模式：边界只能来自用户指定群，不扩大到其他群/私聊；群名需 jdme-local-tools 解析成候选会话，仅当会话名完全一致或明显同一群才用；摘要前必须按目标群号/群名二次过滤。

第二步：整合分析（核心）

前置处理必须在分析前完成，见下一节"前置处理五项"。然后做：项目聚类（一条只归一个项目）→ 相关事项判断（只围绕用户后续需关注/处理/决策/同步）→ 优先级判断（@我/要回复/涉风险=高优）→ 决策提取 → 分歧识别 → 风险识别 → 待办生成。

第三步：输出

严格按固定6个一级小标题输出。无内容写"暂无"而非省略。直接从"昨天你主要参与了 N 个项目："开始，不输出日期行。每个要点采用"结论 + 摘要上下文"两层结构。

04前置处理（五项，必须在分析前完成）

① 时间过滤

优先用户 timeRange；无则从 prompt 识别；都没有则默认昨天 0:00–24:00。京ME timestamp 是毫秒级 Unix 时间戳。

② 京ME指定群聊过滤

prompt 指定群号/群名时，先按指定 sessionId/会话名过滤再做任何摘要。未命中指定群的消息视为不存在。

③ 人名映射表

从所有消息提取 @中文名(ERP)、中文名（英文名）、邮件发件人，生成 ERP→中文名映射。

④ 身份解析

私聊 sessionName 常为空，从 sessionId（erp1:ee:erp2:ee）提取对方 ERP，再用映射表匹配中文姓名，找不到则保留 ERP，不强行翻译。

⑤ 相关性分级 + 证据强度

每条信息先判断相关性并标注证据强度。最终输出只列证据强度为"强"的要点。

05相关性分级与证据强度

决定哪些信息进入最终输出。

强相关 · 强证据

进入最终输出

用户本人发言、被 @、私聊用户、邮件直接发给用户且需处理

中相关 · 中证据

仅内部分析参考

用户所在项目群中与负责项目相关的讨论、会议中用户为参会人

弱相关 · 弱证据

只作背景/舍弃

大群普通讨论、值班记录、无明确指向用户的信息

硬约束 最终输出只允许出现"强"证据的信息；"中""弱"不得输出，也不得作为独立条目。弱相关动态不能单独成章，只能作为强证据条目的摘要上下文；无对应强证据事项则直接舍弃。

06固定6段输出结构

一级小标题严格使用以下6个，不能新增/删除/改名/调序。

1

昨天你主要参与了 N 个项目：

项目名 + 摘要上下文（谁在哪个来源做了什么 + 用户参与证据）

2

今天建议优先处理 N 件事：

具体行动+原因 + 摘要上下文（当前状态）

3

重点关注：

值得持续关注的事项、项目进展或协作动态

4

风险提醒：

风险描述 + 处理建议 + 影响范围

5

已定决策：

来源 + 谁提出什么 + 最终结论

6

待定问题：

未解决的问题/分歧，各方观点 + 当前状态（卡在哪/待谁确认）

禁止出现的一级标题 关注动态 · 会议记录 · 邮件摘要 · 需要你处理 · 项目风险 · 关键决策 · 未解决分歧 · 可流转产物 · 日期行。会议/邮件/弱相关动态只能作为上述6段中某要点的上下文来源。

07要点结构：结论 + 摘要上下文

结论

一句话提炼，让用户 3 秒内理解发生了什么、需要做什么。

摘要上下文（证据链）

按时间顺序列出关键对话/邮件/会议片段。必须说明：谁、在什么来源、做了什么/说了什么、当前状态、为什么需用户关注。

摘要上下文撰写规则
  • 忠实原文：人名/项目名/数字/结论必须来自原始数据
  • 对话脉络优先：保留"谁说了什么→谁回应→结论"的关键轮次
  • 保留用户本人发言（用户需看到自己说了什么来回忆上下文）
  • 标注来源：——来源：{渠道}{具体位置}
  • 长度控制：每要点上下文不超过 5 条 bullet
  • 时间标注：首条标注时间（如"6/5 晚""昨天 14:30"）

08各数据源进入日报的筛选原则

京ME消息

采集重点 所有 @ 用户本人的消息（最高优先级）· 项目群中的讨论/决策/待办 · 私聊中的工作请求。忽略：闲聊、表情包、系统通知、机器人消息。

邮件

筛选原则（黑名单法） 保留：直接发给用户的工作邮件、项目相关讨论、领导/关键干系人邮件、需回复处理的邮件、权限申请/审批结单等可能影响行动的系统邮件。可过滤：无关 Code Review、批量通知、营销广告、newsletter。不要一刀切过滤所有 Coding/审批邮件——与当天工作强相关就保留。

会议

提取重点 会议主题、参会人、关键讨论内容、明确决策、待办事项、分歧和未解决问题。数据为空或失败时按返回状态记录原因，不编造会议内容。

09无信息场景兜底

所有源采集完成后，若全部无数据/全失败/过滤后无工作内容，判定"无有效信息"。

判定细则
  • 目标是周末/节假日且无数据 → 提示"目标是{周六/周日/节假日}"
  • 工作日无数据 → 提示"可能你在休假、出差，或数据源范围过窄"
  • 部分源有少量非工作消息 → 仍判无有效信息，状态注明"有 N 条消息，均为非工作内容"
  • 某些源失败而其他确实无数据 → 同时列失败原因和无数据状态，不混为一谈
不要 输出空板块模板。改用兜底输出：目标时间范围内没有采集到工作相关的信息 + 逐项列出每个 source 的状态。

10完整约束

类别约束
内容来源必须来自实际采集数据，不得编造任何人名/项目名/事件
命名项目名用原始叫法不重命名；人名用中文（无则 ERP）
数量每板块≤7 条，超出保留高优+末尾"及其他 N 条"；"今天优先处理"≤5 条
输出结构固定6段，不允许新增/改名/调序
证据强度只输出"强"；"中""弱"不输出不作独立条目
内部字段不输出"证据强度""强/中/弱""强证据"等内部判断字段
敏感信息不输出 token/cookie/密码；失败源末尾加"⚠️ {源} 数据未获取：{原因}"
原文保留摘要上下文保留原话关键词和语气，不过度书面化改写
