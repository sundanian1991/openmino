## 二、核心问题诊断

### 2.1 当前痛点

1. **模板覆盖不全**：当前 20 个 nian 模板 + 6 皮肤，无法覆盖 9气质×6场景=54 种组合
2. **组件使用不均**：Hero 类组件（C74-C78）使用频率远超数据图表类（C20-C24）
3. **场景偏向**：感知层和沉淀层模板多，执行层和计划层模板少
4. **R/H 割裂**：R 的深度场景和 H 的质感没有在模板层面融合
5. **模板"套感"**：同一模板换内容，输出缺乏内容自适应的变化

### 2.2 根因

- 模板是"一个成品 HTML"，不是"一套可组合的模块"
- 决策卡体系已定义（nian-decision-card），但模板层仍以"整页"为单位
- 组件 26 族未在模板层形成"配额系统"

## 三、目标模板矩阵

### 3.1 核心思路

**不追求 9×6×29 的全排列（1512 种不可能维护），而是：**

1. 以 **6 场景 × 3 深度（L2/L3/L4）** 为骨架 = 18 个「基础模板」
2. 每个基础模板 × **2 气质变体**（场景默认气质 + 备选气质） = 36 个「场景模板」
3. 每个场景模板内 **嵌 5-9 个 section**，每个 section 必选不同组件族
4. 保证 **26 组件族至少在 36 模板中出现 2 次以上**

### 3.2 矩阵：6 场景 × 3 深度 = 18 基础 × 2 气质 = 36 目标

```
                     L2(轻量)        L3(标准)        L4(深度)
感知层(Perception)   T01  + T07      T02  + T08      T03  + T09
决策层(Decision)     T04  + T10      T05  + T11      T06  + T12
计划层(Planning)     T13  + T19      T14  + T20      T15  + T21
执行层(Execution)    T16  + T22      T17  + T23      T18  + T24
沉淀层(Precipitate)  T25  + T31      T26  + T32      T27  + T33
对外层(External)     T28  + T34      T29  + T35      T30  + T36
```

- 每个 T 是一个完整的 HTML 页面
- L2 = 3-4 sections（快速出稿，适合周报/会议纪要）
- L3 = 5-7 sections（标准深度，适合方案选型/SOP）
- L4 = 7-9 sections（深度报告，适合行业研究/品牌展示）

### 3.3 每个模板的固定结构

```
<section data-layout="Sxx">  ← 从 29 骨架库选
  <!-- 使用 2-3 个组件族，真实内容占位 -->
</section>
```

每个模板保证：
- 至少 1 个 Hero section（S01-S04）
- 至少 1 个数据/指标 section（S05/S06/S14）
- 至少 1 个叙事/展开 section（S07/S09/S11/S19）
- 至少 1 个收束 section（S18/S21）
- 恰好 1 处「打破」元素（ghost 水印/斜切/数字装饰）

## 四、组件配额系统（核心创新）

### 4.1 26 组件族按使用场景分类

| 类别 | 组件族 | 使用频率目标 |
|------|--------|:----------:|
| **高频**（每模板必用） | C01 Cards, C04 Lists, C75/C77 Hero, C21 Footer | 36 次 / 36 模板 |
| **中频**（场景决定） | C02 Buttons, C05 Tables, C06 Nav, C08 Tags, C10 Toggles, C11 Progress, C12 Bar/Sparkline, C13 Widgets, C15 State, C16 Flow, C18 Tab, C19 Accordion, C20 对比表, C22 Detail, C23 Checklist, C24 数字标题, C25 三列网格, C26 堆叠图卡 | 12-24 次 |
| **特色**（特定场景） | C03 Inputs, C07 Segmented, C09 Date, C14 Overlays, C17 Do-Don't | 6-12 次 |

### 4.2 配额追踪机制

在生成过程中维护 `component-quota.json`：

```json
{
  "C01-cards": { "target": 36, "used": 0 },
  "C02-buttons": { "target": 18, "used": 0 },
  ...
}
```

每生成一个模板，更新使用计数。当某组件使用不足时，优先在后续模板中选用。

### 4.3 每个模板内的组件多样性规则

- 同一个 section 内最多用 2 个组件族（避免堆砌）
- 同一模板内同一组件族不重复出现（除非是 Cards/Footer）
- 相邻模板（如 T01 和 T07）用不同组件组合（确保气质差异）

## 八、架构升级：Section-as-unit（关键决策 ✅）

### 8.1 核心变化

**旧思路**：36 个模板 = 36 个独立 HTML 文件，每个是完整页面。

**新思路**（你指出的）：
- 36 个模板 = 36 个 **section 序列组合**，不是 36 个孤立文件
- 每个 section 是独立可复用单元，标注 `data-layout`、`data-temperament`、`data-scene`
- 技能调用时：可以跨模板取 section → 混搭组合 → 生成新页面

```
模板 A（感知·L3·Statement） = [S03, S05, S06, S07, S15, S18, S21]
模板 B（决策·L3·Diagonal）= [S01, S05, S09, S07, S15, S18, S21]

技能调用时：
  用户要"数据报告+品牌质感" →
    从模板 A 取 S03 Hero 数据
    从模板 B 取 S18 收束宣言
    从模板 C 取 S12 图片网格
    → 混搭生成
```

### 8.2 Section 统一接口

每个 section 标注：

```html
<section
  data-layout="S05"
  data-temperament="statement|diagonal|split|numeral|entrance|pulse|dashboard"
  data-role="hero|data|transition|evidence|closing"
  data-components="01,05,12"
  data-scene="perception|decision|planning|execution|precipitate|external"
>
```

### 8.3 对 Phase 的影响

| Phase | 变化 |
|-------|------|
| Phase 1 | 盘点时需要提取每个 section（而非整页）的模式 |
| Phase 2 | 规格卡定义到 section 级别（每个模板 = 5-9 个 section 规格） |
| Phase 3 | 生成引擎核心 = section 拼接器 + 跨模板混搭规则 |
| Phase 4 | 每批生成后验证 section 间过渡 + 跨模板可组合性 |
| Phase 5 | 技能接入时，模板选择 = section 序列匹配 + 跨模板 section 推荐 |

## 九、风险与约束

1. **Section 互操作性**：不同气质的 section 拼接时需过渡 section（S15 金句页天然适合做过渡）
2. **组件配额跨 section 统计**：每个组件的使用次数按 section 计算，再汇总到模板
3. **模板仍保留完整形态**：36 个模板仍是完整页面，但同时每个 section 也可独立引用
4. **5 条硬规则是底线**：每个模板必须通过自检
5. **R 系列内容场景不丢失**：供应商管理、SOP、数据报告等深度场景必须在新模板中有对应
