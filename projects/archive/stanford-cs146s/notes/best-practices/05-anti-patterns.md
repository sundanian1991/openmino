# 05 - 反模式与避坑

> **核心认知**：知道什么不该做，比知道什么该做更重要

---

## 🎯 反模式分类

### 分类体系

| 类别 | 说明 | 严重性 |
|------|------|--------|
| **Context 反模式** | Context 管理不当 | 🔴 高 |
| **规划反模式** | 跳过规划或规划不当 | 🔴 高 |
| **代码反模式** | 编码过程中的错误模式 | 🟠 中 |
| **工具反模式** | 工具使用不当 | 🟠 中 |
| **心理反模式** | 思维模式导致的错误 | 🟡 低 |

---

## 📊 Context 反模式

### 反模式 1: Context 过载

**描述**：不主动清理，让 Context 持续增长

**症状**：
```
会话开始 → 快速响应
    ↓
30 分钟后 → 响应变慢
    ↓
1 小时后 → 开始忽略早期指令
    ↓
2 小时后 → 质量严重退化
```

**原因**：
- 不知道清理阈值（60k tokens, 30%）
- 害怕丢失中间状态
- 习惯性"继续"而不清理

**修复**：
```
1. 设置清理阈值提醒
   - 60k tokens 或 30% 时主动提醒

2. 使用 Document & Clear
   - 保存中间状态到文件
   - /clear 清理上下文
   - 新会话读取文件继续

3. 定期重启（每小时）
   - 简单任务：/clear + /catchup
   - 复杂任务：Document & Clear
```

**示例**：

```bash
# ❌ 反模式：一直续，不清理
# 会话继续... 100k tokens... 质量下降

# ✅ 正确做法：定期清理
# 30 分钟后
echo "当前 Context: 45k tokens，即将达到阈值"
./scripts/save-session-state.md
/clear
# 新会话读取 session-state.md 继续
```

---

### 反模式 2: 过度使用 /compact

**描述**：依赖 /compact 压缩上下文

**问题**：
```
/compact
    ↓
自动压缩（黑盒）
    ↓
你不知道保留了什么，丢弃了什么
    ↓
压缩后的内容可能不准确
    ↓
后续决策基于错误信息
```

**替代方案**：
```
1. Document & Clear（推荐）
   - 你控制保留什么
   - 准确的中间状态
   - 新会话快速接续

2. 简单重启
   - /clear + /catchup
   - 适合无状态任务
```

**对比**：

| 维度 | /compact | Document & Clear |
|------|----------|-----------------|
| **透明度** | 低（黑盒） | 高（你控制） |
| **准确性** | 可能不准确 | 你保证准确 |
| **可控性** | 低 | 高 |
| **适用场景** | 几乎不推荐 | 复杂任务 |

## 📋 规划反模式

### 反模式 3: 跳过规划直接编码

**描述**：收到任务直接开始编码

**症状**：
```
收到任务："添加用户设置页面"
    ↓
立即开始：创建组件、写逻辑
    ↓
中途发现：需求理解错误
    ↓
返工：删除代码，重新理解
    ↓
中途又发现：架构冲突
    ↓
返工：重构已有代码
    ↓
...
```

**原因**：
- 低估任务复杂度
- 过度自信
- 急于"产出代码"

**修复**：
```
1. Plan First 机制
   - 所有 ≥3 步任务必须规划
   - 创建 plan/context/tasks 三文件
   - 用户批准后才开始编码

2. Explore → Plan → Code → Commit
   - Explore: 先研究，不编码
   - Plan: 设计方案
   - Code: 机械执行
   - Commit: 规范提交
```

**示例**：

```bash
# ❌ 反模式：直接编码
> 添加用户设置页面
> 好的，开始创建 UserSettings.tsx...

# ✅ 正确做法：先规划
> 添加用户设置页面
> 让我先研究现有代码和需求...
> [创建规划文档]
> 请审阅 plan.md，批准后我开始实施
```

### 反模式 4: 规划过于详细

**描述**：规划变成"伪编码"，过度细节

**症状**：
```
规划文档（5000 行）
    ↓
每行代码都规划了
    ↓
没有灵活性
    ↓
实际执行时发现规划有误
    ↓
但已经花了太多时间规划
    ↓
不愿意改，强行执行
    ↓
结果不佳
```

**原因**：
- 误解"详细规划"
- 过度控制欲
- 不信任实施过程

**修复**：
```
1. 规划正确层次
   - 高层：目标、范围、约束
   - 中层：架构、步骤、验证
   - 低层：留给实施时决定

2. 迭代规划
   - 阶段 1：总体规划
   - 阶段 2：详细设计（实施前）
   - 阶段 3：调整（发现新问题时）
```

**对比**：

| 层次 | 过于详细 | 正确详细度 |
|------|---------|-----------|
| **高层** | 包含实现细节 | 目标、范围、约束 |
| **中层** | 包含每行代码 | 架构、步骤、验证 |
| **低层** | - | 留给实施时决定 |

## 💻 代码反模式

### 反模式 5: 不写测试

**描述**：AI 生成代码后直接使用，不测试

**症状**：
```
AI 生成代码
    ↓
手动测试几个正常用例
    ↓
"看起来没问题"
    ↓
部署
    ↓
生产环境出现边界情况 bug
    ↓
紧急修复
```

**原因**：
- 过度信任 AI
- "看起来没问题"的错觉
- 测试"太麻烦"

**修复**：
```
1. TDD 流程
   - 先写测试
   - 确认测试失败
   - 实现直到通过
   - 不在实施中修改测试

2. 自动化测试
   - 单元测试：每个函数
   - 集成测试：API 调用
   - E2E 测试：关键流程

3. 测试覆盖
   - 正常情况
   - 边界情况
   - 错误情况
```

---

### 反模式 6: 过度工程化

**描述**：简单问题复杂化

**症状**：
```
需求："读取配置文件"
    ↓
过度设计：
- 创建配置管理框架
- 支持多种格式（JSON/YAML/TOML/INI）
- 支持配置继承
- 支持环境变量覆盖
- 支持配置验证 Schema
- 支持配置热更新
    ↓
实际只用：读取 JSON
```

**原因**：
- "为未来做准备"
- 展示技术能力
- 过度抽象

**修复**：
```
1. YAGNI 原则（You Aren't Gonna Need It）
   - 只实现当前需要
   - 不为未来可能的需求编码

2. 简单优先
   - 能用文件就用文件
   - 能用函数就不用类
   - 能用本地就不用远程

3. 需要时再重构
   - 确实需要时再扩展
   - 那时有更清晰的需求
```

**对比**：

| 需求 | 过度工程化 | 简单方案 |
|------|-----------|---------|
| 读取配置 | 配置管理框架 | `JSON.parse(readFileSync('config.json'))` |
| 日志 | 日志框架（分级、格式化、多输出） | `console.log()` |
| 状态管理 | Redux + Middleware | `useState()` |

### 反模式 7: 复制粘贴理解错误

**描述**：Stack Overflow/GPT 代码直接复制，不理解原理

**症状**：
```
遇到问题 → 搜索 → 复制代码
    ↓
代码"工作"了
    ↓
不知道为什么工作
    ↓
遇到类似问题
    ↓
不会举一反三
    ↓
再次搜索复制
```

**原因**：
- 急于解决问题
- 不想深入理解
- "能用就行"

**修复**：
```
1. 复制前先理解
   - 这段代码做什么？
   - 为什么能解决问题？
   - 有什么副作用？

2. 注释解释
   - 在代码中添加注释
   - 解释关键逻辑

3. 自己重写
   - 理解后自己实现
   - 确保真正理解
```

## 🔧 工具反模式

### 反模式 8: 滥用 Subagents

**描述**：能用简单工具解决的问题，非要调用 Subagent

**症状**：
```
任务："读取文件内容"
    ↓
调用 Explore Subagent
    ↓
Explore 调用 Read 工具
    ↓
返回结果
    ↓
绕了一大圈
```

**原因**：
- 觉得 Subagent "更高级"
- 不熟悉直接工具
- 习惯性"外包"

**修复**：
```
决策树：
  已知文件路径？
    ├─ 是 → 直接 Read
    └─ 否 ↓
  按名称找文件？
    ├─ 是 → Glob
    └─ 否 ↓
  按内容搜索？
    ├─ 是 → Grep
    └─ 否 ↓
  广泛探索（3+ 位置）？
    ├─ 是 → Explore Subagent
    └─ 否 → 直接工具
```

**对比**：

| 任务 | 直接工具 | Subagent |
|------|---------|----------|
| 读取已知文件 | `Read` | ❌ 过度 |
| 搜索文件名 | `Glob` | ❌ 过度 |
| 搜索代码内容 | `Grep` | ❌ 过度 |
| 探索代码库（3+ 位置） | - | ✅ Explore |
| 复杂研究（多步） | - | ✅ general-purpose |

### 反模式 9: Skills 不自动激活

**描述**：创建了 Skills，但 Claude 总是不用

**症状**：
```
创建了 security-review Skill
    ↓
用户："请审查这段代码的安全性"
    ↓
Claude：直接分析，不用 Skill
    ↓
结果：质量不稳定
```

**原因**：
- 手动 Skills 约 90% 被忽略
- Claude 优先用原生工具
- 没有"强制"机制

**修复**：
```
1. 基于 Hook 的自动激活
   - UserPromptSubmit Hook
   - 分析关键词/意图
   - 注入激活提醒

2. skill-rules.json
   - 定义触发条件
   - 自动匹配相关 Skills
```

**示例配置**：

```javascript
// skills-auto-activate.js
module.exports = {
  async UserPromptSubmit(context) {
    const { prompt } = context;

    const keywords = {
      'security-review': ['security', '安全', '漏洞'],
    };

    for (const [skill, words] of Object.entries(keywords)) {
      if (words.some(w => prompt.toLowerCase().includes(w))) {
        context.prompt = `🎯 SKILL ACTIVATION CHECK - Use ${skill} skill\n\n${prompt}`;
        break;
      }
    }

    return context;
  }
};
```

## 🧠 心理反模式

### 反模式 10: "AI 说没问题"

**描述**：过度信任 AI，不验证

**症状**：
```
让 AI 写代码
    ↓
问："有 bug 吗？"
    ↓
AI："没有，代码正确"
    ↓
直接部署
    ↓
生产环境 bug
```

**原因**：
- AI 说"没问题" = "我没看到明显问题"
- 不是"我验证了所有情况"
- 不是"我保证没有 bug"

**修复**：
```
1. 验证优先
   - AI 说"没问题" ≠ 真的没问题
   - 必须自己验证

2. 测试验证
   - 写测试
   - 运行测试
   - 边界情况测试

3. Code Review
   - 人工审查
   - 或用新 Claude 实例审查
```

**核心认知**：
> **AI 是助手，不是责任承担者。你提交的代码，你负责。**

---

### 反模式 11: "这次先这样"

**描述**：每次都"先这样"，从不优化

**症状**：
```
任务完成 → "这次先这样"
    ↓
下次类似任务 → "这次先这样"
    ↓
积累技术债务
    ↓
越来越难维护
    ↓
"下次一定要重构"
    ↓
继续"这次先这样"
```

**原因**：
- 短期压力
- "以后再说"
- 没有明确"下次"

**修复**：
```
1. 明确"下次"
   - 记录到 tasks.md
   - 设置明确时间点

2. 债务追踪
   - 技术债务清单
   - 优先级排序
   - 定期偿还

3. 防止积累
   - 第一次就做对
   - 不接受"先这样"
   - 及时重构
```
