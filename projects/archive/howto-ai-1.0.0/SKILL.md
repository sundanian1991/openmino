# How to AI — 综合运营 AI 转型通道

帮助 100+ 综合运营同事，根据自身工作内容、技术能力和真实痛点，生成一份**可立刻动手执行**的 AI 转型方案。
不做通用趋势科普，不堆砌工具清单。每一份输出都是量身定制。

---

## 核心改动（相比原版）

| 维度 | 原版 | 改版 |
|------|------|------|
| 用户画像 | 手动填写 3 题问卷 | Excel 数据预填 + 补充确认 |
| 交互流程 | 7 步线性 | 4 轮对话（含小白/有基础双版本措辞） |
| Skill 推荐 | 一一对应 | 整合覆盖（理解全部工作 → 用 2-3 个 Skill 覆盖大部分痛点） |
| 方案分流 | 仅推荐已有 Skill | Buy/Build 双线（现有 Skill + 自建引导） |
| 报告样式 | 固定奶白模板 | 7 种风格模板可选 |
| 进度追踪 | 无 | state.json 跨周追踪（W1→W2→W3→月度） |
| 定时任务 | 依赖 automation_update | 平台无关 cron 脚本 |
| 平台命名 | WorkBuddy / OpenCode | JoyClaw（小龙虾） |
| 非 Excel 用户 | 无 | 兜底纯对话流程 |

## 四轮对话流程

### 第一轮：身份确认 + 工作全貌（ERP 预填）

**目标**：确认身份，展示工作全貌，让用户补充 Excel 未覆盖的工作

#### 有 ERP 的用户

用 AskUserQuestion 提问：

> 请输入你的 ERP 编号，我会根据系统记录生成你的工作画像。
> - 输入 ERP 编号
> - 我不在系统里 / 我没有 ERP

用户输入 ERP 后：
1. 从 Excel 查找该员工的全部工作模块
2. 展示预填画像表（姓名、部门、岗位、工作模块列表）
3. 追问：

> 以上是你当前系统记录的工作模块。除此之外，你日常还有哪些工作是系统没记录的？
> - 例如：临时数据查询、跨部门协调、沟通对接等
> - 可多选，也可自行输入

#### 无 ERP 的用户

> 请告诉我以下信息，我会为你生成个性化方案：
> 1. 你的部门和大致岗位是什么？
> 2. 你日常最常做的工作有哪几类？（选 3-5 个）

🔴 **STOP — 第一轮结束。必须等用户回复后再进入第二轮。禁止连续发问。**

---

### 第二轮：技术背景 + 优先级 + 时间预算

**目标**：确定技术底子和可投入时间

用 AskUserQuestion 同时收集（最多 3 题）：

#### 小白版本

> 关于你的技术背景和时间：
>
> **1. 你目前用过哪些 AI 工具？**
> - 只用过豆包/Kimi 聊天
> - 用过 ChatGPT/Claude 处理过工作
> - 用过 AI 写代码或搭建自动化流程
>
> **2. 你最想先解决哪个工作？**
> - [列出工作模块，按 AI 化潜力排序]
>
> **3. 每周能花多少时间学新工具？**
> - 不到 1 小时
> - 1-3 小时
> - 3 小时以上

#### 有基础版本

> 关于你的技术背景和时间：
>
> **1. 你目前的技术背景属于哪类？**
> - 小白 — 偶尔用 AI 对话
> - 低代码 — 熟练使用 AI 工具，能配置工作流
> - 技术 — 能写代码或搭建自动化
>
> **2. 你最想先改造哪个工作模块？**
> - [列出工作模块，按 AI 化潜力排序]
>
> **3. 每周能投入多少时间？**
> - 1 小时以内
> - 1-3 小时
> - 3 小时以上

🔴 **STOP — 第二轮结束。必须等用户回复后再进入第三轮。禁止连续发问。**

---

### 第三轮：深度了解 + 切换容忍度

**目标**：理解具体痛点，确定推荐策略的过滤标准

先根据第二轮选定的优先模块，追问 1 个深度问题：

> 关于「[用户选定的工作模块]」：
> 描述一下你目前是怎么做这个工作的？哪个环节最花时间？

然后评估切换容忍度：

> 你对新工具/新技术的接受程度是？
> - **保守型**：除非效率提升 3 倍以上，否则不换
> - **平衡型**：效率提升 50% 以上，愿意花 1 小时迁移
> - **激进型**：有新工具都想试

拿到容忍度后，**记住它** — 它决定后续 Buy/Build 推荐策略和自动化频率。

🔴 **STOP — 第三轮结束。必须等用户回复后再进入第四轮。**

---

### 第四轮：Buy/Build 扫描 + 方案确认

**目标**：更广范围搜索 Skill，展示匹配结果，让用户确认后再生成报告

#### 4A. Skill 市场扫描（Buy 路线）

**搜索方式**（按优先级）：

1. **JoyClaw 环境有 find-skills**：用 `find-skills` 按关键词搜索已安装和市场可用的 Skill
2. **无 find-skills**：根据 `references/scenario_translation.md` 中的场景对照表，匹配用户工作模块对应的 AI 场景
3. **更广范围搜索**：在 Skill 市场中搜索能解决用户痛点的工具
4. **用户的 Excel 数据**：检查「使用AI工具」列，看用户已在用什么，避免重复推荐

**匹配逻辑**（整合覆盖，非一一对应）：
- 理解用户全部工作模块 → 分析共性（数据处理类、文档处理类、沟通协调类）
- 找到能覆盖多个模块的 Skill（一个 Skill 解决一类问题）
- 给出最少的 Skill 组合（2-3 个，覆盖大部分痛点）
- 已知常用 Skill 映射（不限于这些）：

| AI 场景 | 推荐 Skill |
|---------|-----------|
| 文档理解 + 合规核验 | grill-with-docs |
| Excel 数据提取比对 | doc-xlsx |
| 报告自动生成 | sn-research-report |
| 会议纪要整理 | 可用通用 AI 对话 |
| 数据查询分析 | 可用 AI + CSV 上传 |

**容忍度过滤**：

| 容忍度 | 推荐策略 | 推荐数量 |
|--------|---------|---------|
| 保守型 | 只推零迁移成本的 Skill | ≤ 3 个 |
| 平衡型 | 推 ROI 明确的 Skill，附量化对比 | ≤ 6 个 |
| 激进型 | 推所有匹配的 Skill | 不限 |

#### 4B. 展示匹配结果

> 我在更广范围内搜索了能解决这些痛点的 Skill 和工具：
>
> | 痛点 | 匹配 Skill | 能做什么 | 安装方式 | 状态 |
> |------|-----------|---------|---------|------|
> | [痛点1] | [skill-name] | [一句话说明] | [安装方法] | 已安装/需安装 |
> | [痛点1] | [skill-name-2] | [一句话说明] | [安装方法] | 已安装/需安装 |
> | [痛点2] | [skill-name-3] | [一句话说明] | [安装方法] | 已安装/需安装 |
>
> **可复制安装**：[skill-name-1], [skill-name-2]

Skill 名称旁必须有 copy 按钮。安装引导格式：

> 安装方法：打开 JoyClaw（小龙虾）→ Skill 商店 → 搜索「[skill-name]」→ 点击安装

#### 4C. 自建引导（Build 路线）

对于没有现成 Skill 匹配的工作模块：

> 以下痛点目前没有现成 Skill 能解决，需要自建：
>
> | 痛点 | 自建方案 | 难度 | 建议路径 |
> |------|---------|------|---------|
> | [痛点3] | [方案描述] | 低/中/高 | 完全自建 / 基于 [skill-name] 组合 |
>
> **自建路径**：
> - **完全自建**：用 skill-creator 从零创建
> - **组合自建**：基于 [已有 skill] 扩展功能
>
> 标注在 W3 阶段动手（前两周先熟悉已有工具）

#### 4D. 方案确认

用 AskUserQuestion 确认：

> 以上是我的方案预览。确认后我将生成完整报告。
>
> - 确认，生成报告
> - 我想调整优先级
> - 我想换一个工作模块
> - 某个 Skill 我想了解更多

🔴 **STOP — 第四轮结束。用户确认前禁止生成报告。禁止连续发问。**

## 报告生成

### 模板选择

**路径**：`assets/templates/`
**使用说明**：`assets/templates/TEMPLATE-GUIDE.md`（必读，说明如何从模板提取样式并生成报告）

| 模板文件 | 风格 | 优先级 |
|---------|------|--------|
| `template-original.html` | 奶白底 #faf8f5，衬线字体，卡片布局 | **默认（第一优先级）** |
| `template-muller.html` | 瑞士网格，零装饰 | 数据/分析岗偏好 |
| `template-loreal.html` | 黑+红，章节结构 | 高级感/管理岗偏好 |
| `template-nothing.html` | Nothing 工业风，深色 | 科技感/年轻用户 |
| `template-hallmark.html` | Hallmark 编辑风，橙色 | 内容/运营岗偏好 |
| `template-pentagram.html` | 粗体机构风，黑+钴蓝 | 正式/汇报场景 |
| `template-apple.html` | Apple HIG 产品页 | 简洁/产品岗偏好 |

**匹配逻辑**：

1. **默认**：使用 `template-original.html`
2. **用户选择**：第四轮确认方案后，用 AskUserQuestion 展示 2-3 个风格的简短描述让用户选
3. **不自动匹配**：不根据部门/岗位自动分配模板，因为偏好不可预测

**模板用法**：

- **结构骨架**：使用 `assets/report_template.html`（有 `{{占位符}}`，是干净的模板）
- **样式风格**：从选定模板的 CSS 变量（`:root`）和布局类中提取颜色、字体、间距
- **生成流程**：读 report_template.html 的结构 → 用选定模板的 CSS 变量替换 → 填入用户真实数据 → 输出完整 HTML
- **不要**复制 demo 模板中的示例数据（张明、供应商管理等）

### 报告必须包含的章节

1. **用户画像** — 姓名、部门、岗位、技术背景、容忍度、时间预算
2. **工作模块全貌** — 来源标记（系统记录 vs 用户补充）、频次、月工时、AI 化潜力
3. **优先改造项** — Buy/Build 标记，瓶颈分析，月节省预估
4. **AgentAI vs 手动对照** — 价值对比表
5. **Buy/Build 分流明细** — 横向标签布局，Skill 名称可复制
6. **技术栈清单** — 当前已安装/需安装的 Skill 和工具
7. **Skill 扫描结果** — 匹配度、功能说明、安装状态
8. **工具迭代建议** — 按容忍度过滤
9. **定时任务配置** — 平台无关 cron 脚本
10. **周度执行清单（W1-W3）** — 可打印，交互式复选框
11. **Build 自建引导**（如有 Build 项）

### 报告关键要素

- **来源标签**：横向排列，区分「系统」和「补充」
- **Buy/Build 状态**：横向标签，非表格竖排
- **Skill 名称**：必须有 copy 按钮
- **平台命名**：统一使用「JoyClaw（小龙虾）」
- **周度清单**：W1 打基础 → W2 串联工作流 → W3 自建 Skill
- **打印友好**：`@media print` 样式，隐藏操作按钮

## 跨周进度追踪（state.json）

### 机制

每次生成报告后，在用户工作目录创建 `state.json`：

```json
{
  "erp": "ZM001",
  "name": "张明",
  "department": "服务组-供应商管理",
  "tech_level": "novice",
  "tolerance": "balanced",
  "weekly_hours": 2,
  "current_phase": "W1",
  "phases": {
    "W1": { "status": "in_progress", "start_date": "2026-05-29", "completed_tasks": [] },
    "W2": { "status": "pending" },
    "W3": { "status": "pending" }
  },
  "buy_skills": ["doc-xlsx", "grill-with-docs"],
  "build_skills": ["临时数据查询与汇总"],
  "report_template": "template-loreal.html",
  "last_report": "output/report-2026-05-29.html"
}
```

### 跨周恢复

**触发**：用户再次触发此技能时，先执行以下检查：

1. 检查当前工作目录下是否存在 `output/state.json`
2. 如果存在 → 读取，判断 current_phase，进入对应阶段
3. 如果不存在 → 视为新用户，从第一轮开始

**操作**：
- 写入：报告生成后，用 Write 工具将 state.json 写入 `output/state.json`
- 更新：用户完成每周任务后再次对话时，用 Edit 工具更新对应 phase 的 status 和 completed_tasks
- 路径：始终使用 `output/state.json`（相对于技能目录）

**恢复逻辑**：

| current_phase | 读到后的行为 |
|---------------|------------|
| W1 (in_progress) | 展示 W1 清单，询问完成情况，标记已完成项 |
| W1 → W2 过渡 | W1 必做项全部完成 → 生成 W2 方案，更新 state.json |
| W2 (in_progress) | 展示 W2 清单，引导串联工作流 |
| W2 → W3 过渡 | W2 必做项全部完成 → 引导 Build 自建 |
| W3 (in_progress) | 引导用 skill-creator 创建 Skill |
| completed | 切换到月度迭代检查模式 |

**注意**：如果用户换了 workspace 或新对话找不到 state.json，不要报错，直接当新用户处理。

## 定时任务（平台无关）

### cron 脚本模板

**路径**：`assets/templates/cron-template.sh`

```bash
#!/bin/bash
# JoyClaw（小龙虾）AI 工具迭代检查 — 定时任务模板
# 适配平台：MyAgents / JoyClaw / crontab

# === MyAgents ===
# myagents cron add --name "[用户名]的AI工具迭代检查" \
```

## 核心约束

### 1. 业务语言翻译

参考 `references/scenario_translation.md` 中的对照表，把用户的工作描述翻译成 AI 技术场景。

### 2. 难度门槛严格匹配

| 用户级别 | 允许 | 禁止 |
|---------|------|------|
| 小白 | JoyClaw Skill 商店安装、cron 配置、AI 对话 | Python 脚本、命令行、API Key 配置、本地部署 |
| 有基础 | Skill 自定义、MCP 配置、cron 编排 | 深度学习框架、微调、GPU 操作 |
| 技术 | 全部功能 | 无限制 |

### 3. 输出节奏

- 四轮对话，每轮完成后等用户确认
- 第三轮（容忍度）必须在第四轮（扫描）之前
- 每步输出模块化、表格优先
- 不输出 AI 行业趋势、市场分析

### 4. 追问节制

- 第一轮：1-2 题（ERP + 补充工作）
- 第二轮：3 题（技术 + 优先级 + 时间）
- 第三轮：2 题（深度痛点 + 容忍度）
- 第四轮：1 题（方案确认）
- 总共最多 7-8 题

### 5. Buy/Build 分流原则

- **Buy**：市场有现成 Skill → 直接推荐安装
- **Build**：无现成 Skill → 引导自建，标注 W3 阶段动手
- **Hold**：人际互动性高 / ROI 不明确 → 标注观望，不推荐

### 6. 报告生成规范

- 使用 `assets/templates/` 中的模板
- Buy/Build 分流使用横向标签布局
- Skill 名称必须有 copy 按钮
- 统一使用「JoyClaw（小龙虾）」命名
- 周度清单：W1 安装 Skill → W2 串联工作流 → W3 自建 Skill
- 工具迭代建议按容忍度过滤

## 反例黑名单

> 以下是本技能的"不要做什么"。每条都来自真实踩坑或用户负面反馈。

| # | 禁止行为 | 为什么 | 替代做法 |
|---|---------|--------|---------|
| 1 | 一次跑完四轮对话不暂停 | 用户被信息淹没，无法消化和调整 | 每轮必须 🔴 STOP 等用户回复 |
| 2 | 推荐未验证存在的 Skill 或工具 | 用户搜索后找不到，信任崩塌 | 先用 find-skills 搜索确认存在再推荐；找不到则推荐通用工具 |
| 3 | 给小白用户推荐 Python/API/命令行 | 学习成本远超时间预算，直接劝退 | 严格按难度门槛表匹配：小白只推网页端和 Skill 商店安装 |
| 4 | 输出 AI 行业趋势/市场分析报告 | 用户问的是"我能用什么"，不是"行业在往哪走" | 聚焦个人场景，每个推荐必须对应用户的一个具体工作模块 |
| 5 | 夸大 AI 效果（"效率提升 10 倍"） | 无法兑现的承诺损害后续采纳意愿 | 用保守估算（"预计节省 XX 分钟/天"），附条件说明 |
| 6 | 不经确认直接生成报告 | 报告方向可能偏了，白费算力 | 第四轮方案确认后才能进报告生成 |
| 7 | 报告中复制 demo 模板的示例数据 | 用户看到"张明/供应商管理"以为发错了人 | 必须用 report_template.html 的结构 + 用户真实数据 |
| 8 | 自动匹配模板风格（按部门/岗位） | 偏好不可预测，自动匹配常翻车 | 让用户在 2-3 个风格中自选 |
| 9 | 跳过方案确认直接出报告 | 用户不知道方案是怎么来的，无法判断可信度 | 必须先展示 skill 匹配 + 自建建议，用户确认后再出报告 |
| 10 | 一一对应推荐 Skill | 12 个痛点推荐 12 个 Skill，用户无法消化 | 整合覆盖：理解全部工作 → 分析共性 → 用 2-3 个 Skill 覆盖大部分痛点 |

## 使用示例

**用户输入：**
> "我做供应商管理，审核材料最头疼。AI 能帮我做什么？"

**Agent 行为：**
1. 第一轮：询问 ERP → 查找 Excel 数据 → 展示预填画像 → 追问补充工作
2. 第二轮：评估技术背景（小白）+ 确认优先模块（准入材料审核）+ 时间预算（每周 1-2h）
3. 第三轮：深度了解审核流程痛点 → 评估切换容忍度（平衡型）
4. 第四轮：扫描 Skill（doc-xlsx + grill-with-docs = Buy）→ 引导临时数据查询自建（Build）→ 展示匹配结果 → 用户确认 → 生成报告

## 失败模式与 fallback

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| 用户不在 Excel 中 | 进入纯对话兜底流程，手动收集基础信息 | 以用户口述为准，标注「无系统数据」 |
| 描述太模糊（"我就是想了解下 AI"） | 追问一个具体工作场景："你上周最花时间的一件事是什么？" | 给出 3 个常见场景让用户选 |
| 同时涉及多个场景 | 请用户先选最迫切的一个 | 按工时降序排列，选工时最高的 |
| Excel 数据与用户实际工作不符 | 以用户口述为准，标注差异 | 忽略该条 Excel 数据，用户补充为准 |
| CSV 查询无结果（ERP 存在但查不到行） | 检查是否输入了完整 ERP 编号（含/不含前导零） | 降级为无 ERP 纯对话流程 |
| 报告生成中途写入失败 | 重试一次（文件可能被占用） | 输出纯文本版报告，提示用户稍后重试 HTML |
| find-skills 工具不可用 | 使用 scenario_translation.md 中的场景对照表手动匹配 | 只推荐已验证存在的通用工具（ChatGPT/Claude/DeepL） |
| 用户拒绝所有推荐方案 | 询问具体顾虑（隐私/复杂度/信任） | 退回 Quick Win 模板给出一个今天就能用的 prompt |

🔴 **绝对禁止**：不夸大承诺、不做 AI 行业研究报告、不推荐未验证存在的工具或 Skill。
