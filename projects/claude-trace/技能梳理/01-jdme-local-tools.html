iii.

命令清单

单入口 scripts/jd_me_tool.py，转发到 core 逻辑。所有命令输出 JSON Lines。

discover发现路径与账户

报告 app 根、ME 二进制、support roots、Electron roots、log roots、SQLCipher 候选、DLL 诊断、数据账户，以及 --auto 推荐账户（含评分明细）。最先跑，便宜且平台感知。

terminal
python3 scripts/jd_me_tool.py discover
sessions列出会话

从 session_info 表列出会话，支持按名称、类型过滤。用于解析 session_id。

terminal
python3 scripts/jd_me_tool.py sessions --auto --name-contains Relay --kind 512
chat-historySQLCipher 直读历史

按 session / sender / msg_type / 时间范围过滤，从 chat.data 直读。拿完整 msg_json。

terminal
python3 scripts/jd_me_tool.py chat-history --auto \
  --session erp1:ee:erp2:ee --limit 50
chat-search跨会话关键词检索

在 msg_json 和 search 列做子串搜索，多关键词 OR。一次扫所有会话。

terminal
python3 scripts/jd_me_tool.py chat-search --auto --keyword 上线 --limit 20
chat-watch数据库轮询监听

轮询 chat.data，按关键词 / @我 / sender / session 实时过滤新消息。常驻进程。

terminal
python3 scripts/jd_me_tool.py chat-watch --auto \
  --keyword 上线 --at-me --timeout 60
messages日志明文恢复（降级）

SQLCipher 不可用时使用。从 sdk-event.log 和 renderer.log 恢复明文，容错解析损坏的 JSON 转义。Windows 默认扫 renderer.log。

terminal
python3 scripts/jd_me_tool.py messages --auto --contact fanbaolin3 \
  --rotations 3 --limit 100
python3 scripts/jd_me_tool.py messages --auto --keyword 上线 --at-me
summary为 LLM 摘要设计

join chat_message 与 session_info，标记 private / named_group / at_me_other 三类，默认包含 Relay|Zero|基础服务 群、排除告警/robot 群。直接喂 LLM 做日/周报。

terminal
python3 scripts/jd_me_tool.py summary --auto --since 2026-05-14
watch · history · query辅助诊断

watch tail renderer.log + sdk-event.log 实时解析事件流为稳定 JSON，支持 --keyword / --at-me / --session；history 报告导出文件与数据库状态；query 在导出文本/日志中搜索 ERP（凭据自动 redact）。

iv.

平台覆盖

macOS

  • /Applications/ME.app
  • ~/Library/Application Support/com.jd.me
  • ~/Library/Application Support/ME
  • ~/Library/Logs/ME
  • SQLCipher: JD ME 自带二进制 Contents/Resources/app/.../dd_sqlite3/bin/sqlcipher

Windows

  • ~/Documents/JD/ME（数据）
  • %APPDATA%/ME/logs（日志）
  • %APPDATA%/ME（Electron 用户数据）
  • %APPDATA%/com.jd.me（备用 support）
  • SQLCipher: 自带 sqlcipher.exe 或 sqlcipher64.dll（ctypes ABI fallback）
Windows DLL fallback CLI 不可用时，用 ctypes 直接调用 sqlite3_open_v2 / sqlite3_exec / sqlite3_prepare_v2 等 ABI，只读模式打开加密库。不需要 pysqlcipher3。
v.

技术实现要点

密钥派生

SQLCipher 密钥 = md5(account_pin).hexdigest()，与 JD ME 自身的 db-helper 一致。pin 通常等于 ERP 账号但不绝对——代码注释明确写的是 account_pin。

降级路径

数据库访问失败（如 file is not a database）时，不重试 dump 进程内存或提取凭据，而是报告失败路径 + 所用 pin + SQLCipher 候选，fallback 到 messages 日志恢复。

--auto 账户选择

先读 ~/.jd_me_account，再按 chat.data / session.data / 日志 / 最新修改时间加权推荐。仅在推荐错误或歧义时才传 --account <pin>。

常用过滤器

filters · messages / chat-history 通用
# messages / chat-history 通用
--contact <erp-or-name>      # 匹配 sessionId / senderPin / receiverPin / 内容
--session <session-id>       # 精确会话过滤，可重复
--keyword <text>             # 内容子串，可重复
--at-me --self-pin <pin>     # @我 过滤
--since <date> --until <date> # 时间窗口
--limit <N>                  # 保留最近 N 条
--no-renderer                # Windows：不扫 renderer 日志
vi.

边界与局限

硬边界（绝对不做） 修改 JD ME 应用包/安装目录 · 注入 Electron · 从运行进程内存 dump 密钥 · 写入聊天数据库 · 提取凭据。这是本地只读助手，只操作用户已有访问权限的文件。

读不到什么

  • 超出 JD ME 本地保留期的旧消息——只能读本地缓存，服务器端已清理的拿不到
  • 邮件、会议、日程、通讯录、文件附件内容——本工具只覆盖 IM 聊天
  • 附件文件本身——只看到 msg_json 里的引用，不下载附件二进制
  • 非 JD ME 平台的供应商沟通（微信、企微、电话）不在能力范围

风险点

风险
说明
密钥派生算法公开
md5(pin) 一旦 pin 泄露，任何拿到 chat.data 的人都能解密。加密强度依赖 pin 保密
凭据 redaction 不彻底
REDACTION_PATTERNS 只在 snippet/导出文本做正则替换，chat-history 返回的完整 msg_json 不过滤——可能含密码、token、合同金额
合规边界
虽是用户本地数据，但供应商沟通涉及多方隐私，喂云端 LLM 做摘要时需注意数据外溢
SQL 注入（已缓解）
session_id/sender 用 replace("'","''") 转义，非参数化查询
