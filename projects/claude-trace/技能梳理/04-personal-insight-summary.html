i.

三步工作流程

采集 → 整合分析 → 固定格式输出。

01

收集各数据源

对配置中每个 source：确认技能可用 → 合成执行指令 → 调用技能 → 记录结果与失败原因。尽可能并行调用，某源失败不影响其他。

失败时标注「[该数据源未获取，原因：xxx]」

02

整合分析

前置处理（时间过滤、指定群过滤、人名映射、身份解析、相关性分级、证据强度标注）后，做项目聚类、相关事项判断、优先级判断、决策/分歧/风险/待办提取。

证据强度 强/中/弱 三级，只输出「强」

03

固定 6 板块输出

严格按固定 6 个一级小标题输出，无内容写「暂无」。每个要点采用「结论 + 摘要上下文」两层结构，所有信息必须基于实际采集数据，严禁编造。

会议/邮件/弱相关动态只能作上下文

ii.

三大数据源 · 采集规则

底层命令由各采集技能自身负责，本 prompt 只规定「进入日报的边界与筛选原则」。

jdme-local-tools

京 ME 消息

采集重点：@本人（最高优先）、项目群讨论/决策/待办、私聊工作请求。忽略闲聊、表情包、系统通知、机器人。

精确会话模式：prompt 出现「只读取/指定群/群号」时，边界只能来自指定群，不得扩大、不得因消息少而补私聊。

read-email

工作邮件

优先保留：直接发给用户的工作邮件、项目讨论、领导/关键干系人邮件、需回复处理的邮件。

黑名单过滤：权限申请/审批结单/加入项目等影响行动的系统邮件保留；CI、无关 CR、营销 newsletter 可过滤。不一刀切。

read-joymeeting

慧记会议

重点提取：会议主题、参会人、关键讨论、明确决策、待办、分歧和未解决问题。

不编造：会议数据为空或获取失败时，按 read-joymeeting 返回状态记录原因，不捏造会议内容。

iii.

证据强度分级

前置处理中每个候选要点先标注证据强度，最终只输出「强」——这是质量控制的核心阀门。

强度
对应相关性
输出处理
强
本人发言 / 被 @ / 私聊用户 / 邮件直接发给用户且需处理
✓ 进入最终输出
中
所在项目群中与负责项目相关的讨论、会议中本人为参会人
仅内部分析参考
弱
所在大群普通讨论、值班记录、无明确指向用户的信息
仅作背景上下文 / 舍弃
最终输出过滤 最终结果只列出证据强度为「强」的要点。中/弱不得进入最终输出，也不得作为独立条目；弱相关动态只能作为强证据条目的摘要上下文，无对应强证据事项则直接舍弃。
iv.

固定 6 板块输出模板

一级小标题必须严格使用以下 6 个，不能新增、删除、改名或调整顺序。无内容写「暂无」而非省略。

1
昨天你主要参与了 {N} 个项目
项目聚类，每条带「你参与的证据」（发言/被@/私聊/直接邮件/参会）
2
今天建议优先处理 {N} 件事
不超过 5 条，具体行动 + 原因，带当前状态
3
重点关注
值得持续关注的事项、项目进展、协作动态
4
风险提醒
技术/进度/资源风险，带影响范围与建议处理方式
5
已定决策
明确的结论性表述，谁提出、最终结论
6
待定问题
尚未解决的分歧/需进一步确认的事项，多方观点
两层结构 · 结论 + 摘要上下文 结论：一句话提炼，3 秒内理解发生了什么、需要做什么。摘要上下文：证据链，按时间顺序列出关键对话/邮件/会议片段，说明谁在什么来源做了什么、为什么需要关注。每条不超过 5 个 bullet。
v.

时间规则与硬约束

时间范围确定规则

JSON 配置不提供 timeRange，只能从各 source 的 prompt 识别：

  • 从 prompt 识别时间关键词（今日/昨天/本周/最近 N 条）
  • 无明确范围 → 默认「昨天 0:00–24:00」（昨天 0 点 ≤ timestamp < 今天 0 点）
  • 京 ME timestamp 是毫秒级 Unix 时间戳
  • 无 JSON 配置 → 默认采集 3 类数据源（京 ME / 邮件 / 慧记）

核心硬约束

  • 所有内容必须来自实际采集数据，不得编造人名/项目名/事件
  • 项目名用原始叫法，不自行重命名
  • 人名用中文姓名，否则用 ERP 账号
  • 每板块条目 ≤7 条，「今天优先处理」≤5 条
  • 不输出 token/cookie/密码等敏感信息
  • 某源采集失败 → 末尾加「⚠️ {源} 数据未获取：{原因}」
兜底 · 无信息场景 所有源采集完成后若满足任一条件（均无目标时间数据 / 全为系统通知机器人消息 / 均失败），不输出空板块模板，改用兜底输出说明原因（周末/节假日/休假/出差/配置过窄），并按源逐项列出状态。
◈
vi.

在 AI 转型中的角色

它是三个采集技能的大脑。单独看京 ME、邮件、会议，每一条都只是片段；这个技能把它们关联起来——一个会议上的承诺，对应群里第二天的追问，对应邮件里的确认，串成一条完整的故事线，让用户每天只需读一份简报就能掌握全局。
