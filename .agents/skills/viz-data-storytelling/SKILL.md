# Data Storytelling — 数据沟通策略与编排

> 整合自《Storytelling with Data》(Cole Nussbaumer Knaflic) 全书 | 策略层，只产出叙事大纲

---

## 定位

**不画图、不做 PPT、不写渲染代码**。唯一产出是一份 Markdown 叙事大纲，包含：

- 受众画像与 Big Idea（洞察 + 行动建议）
- 图表清单：每张图的叙事功能、推荐类型、视觉策略（高亮/灰化/排序）
- 三幕叙事顺序（铺垫 → 冲突 → 决议）
- 原始数据引用

大纲完成后 → **viz-design 做视觉设计 → viz-echarts / viz-svg-flow 渲染**。

---

## 六步工作流

```
拿到数据
  ↓
Step 1-2：理解上下文 + 选择图表（策略层）
  ↓
Step 3-4：消除杂乱 + 引导注意力（视觉策略层）
  ↓
Step 5：像设计师思考（整合为大纲）
  ↓
Step 6：讲述故事（三幕叙事编排）
  ↓
独立审查：子代理盲审大纲（找盲点）
  ↓
下游：viz-design → 渲染技能
```

## Step 1：理解上下文

### 1.1 理解数据

先回答三个问题：

| 问题 | 检查点 |
|------|--------|
| 数据**有什么** | 指标、维度、时间范围、样本量 |
| 数据**质量如何** | 缺失值、异常值、口径一致性 |
| **缺什么** | 关键维度缺失 → 先列出，不假装没有 |

### 1.2 定义受众

**禁止**用"所有利益相关者"、"领导"等模糊定义。精确到具体角色。

| 维度 | 回答 |
|------|------|
| **决策者是谁** | 具体人和角色 |
| **信任度** | 权威（他信你）/ 中立（需要证据）/ 怀疑（需要更多证明） |
| **受众是否多元** | 是 → 分开做，不要拼成一个谁都不满意的 |
| **受众动机** | 他们为什么要在意？这个数据对他们意味着什么？ |

### 1.3 Big Idea

一句话，包含 **洞察 + 推理依据 + 建议行动**，用行动动词开头。

> 好的 Big Idea（供应商场景）："伽玛本季度履约率下滑 15pp（从 93% 降至 78%），建议启动备选供应商引入流程"
> 好的 Big Idea（通用场景）："Q3 复购率下滑 5pp 集中在首单后 7 天窗口，建议启动新客留存触达计划"
> 差的 Big Idea："供应商绩效数据分析"
>
> **数据表达精度**：描述率的变化时必须标注 pp（百分点）或%（相对变化），不可省略单位。"履约率下滑 15%"是模糊的——是绝对下降 15pp 还是相对下降 15%？必须写清楚。

所有后续设计选择都必须服务于这个 Big Idea。不是 → 去掉。

**行动导向目标策略**：
- **直接建议**：适用于受众预期你给出方向（"建议立即启动备选供应商引入流程"）
- **催化决策的开放问题**：适用于不宜直接建议（"我们是否应该在 Q3 启动备选供应商试点？"）
- **明确决策节点**：在 Big Idea 中嵌入决策点（"请决定：A) 立即引入、B) 启动评估、C) 暂不行动"）
- **不完美的建议优于被动总结**：即使信息不完整，方向性建议也比被动陈列有价值。标注置信度，但不要回避判断

### 1.4 探索性 vs 解释性

| | 探索性（内部发现） | 解释性（对外沟通） |
|--|-------------------|-------------------|
| 目的 | 自己发现规律 | 让受众理解并采取行动 |
| 做法 | 多试，全保留 | 只展示"珍珠"（关键洞察） |
| 图表数 | 不限 | 精简（通常 3-7 张） |

**默认走解释性路线**，除非明确说是探索。

### 1.5 数据推理

理解数据之后、选择图表之前，先做一轮**计算和推理**。这是区分"格式正确的模板"和"有穿透力的分析"的关键。

必须回答以下问题（写到大纲的"数据推理"段）：

| 推理类型 | 问题 | 示例 |
|---------|------|------|
| **趋势推算** | 如果当前趋势延续，下一个周期会怎样？ | "Q2→Q3 降 10pp，若同速 Q4 将跌至 68%" |
| **相对量级** | 关键差异的倍数关系？ | "伽玛降幅是赛维斯的 14 倍" |
| **排名变化** | 排名在上升还是下降？谁在追赶？ | "毛毛虫 Q3 首次超越伽玛" |
| **业务后果** | 数据变化对业务意味着什么？ | "78% 低于 85% 合同底线，触发违约条款" |
| **分母陷阱** | 率的变化是否因为分母（业务量）变化？ | "如果伽玛 Q3 订单量翻倍，履约率下降可能是产能问题而非管理问题" |
| **基线校准** | 当前值相对历史基线/行业基准在哪？ | "78% 远低于行业平均 87%" |
| **季节性因素** | 数据波动是否受季节/周期规律影响？ | "伽玛 Q3 履约率下降与行业淡季吻合，需排除季节性因素后再下结论" |

每条推理必须标注**置信度**：高（有数据直接支持）/ 中（有间接证据）/ 探索性（需要额外验证）。

**推理结果直接写进 Big Idea 和图表标题**——空洞的标题会被推理出的结论替换。

> 好的 Big Idea（推理后）："伽玛 Q2→Q3 加速恶化，按当前速率 Q4 将跌破 70% 违约线，建议立即启动备选引入"
> 差的 Big Idea（无推理）："伽玛履约率连续下滑，建议引入备选供应商"

**Big Idea 与数据推理段的分工**：Big Idea 引用推理结果形成简短结论+行动建议（一句话）；数据推理段展开推理过程（分条列出）。两者内容交叉但不冗余。

## Step 2：选择图表

> 功能决定形式。先问"这张图要表达什么"，再选图表类型。

### 2.1 图表选择决策树

| 你要表达什么 | 推荐图表 | 禁忌 |
|-------------|---------|------|
| 随时间变化的趋势 | 折线图 | 不要用条形图 |
| 类别间的比较 | 水平条形图 | 不要用饼图 |
| 变量间的关系 | 散点图 | 不要用双轴折线 |
| 一个或几个关键数字 | 大号文字（不用图表） | 不要为了"有图"而做图 |
| 表格数据的模式识别 | 热力图 | 不要用彩虹色 |
| 两个时间点的变化幅度 | 斜率图 | 数据点多时用折线图 |
| 构成占比（仅 2-3 类） | 水平堆叠条形图 | 不要用饼图/甜甜圈 |
| 正负贡献 | 正负堆叠条形图 | |

### 2.2 四条绝对禁忌

| 禁忌 | 原因 | 替代方案 |
|------|------|---------|
| **饼图/甜甜圈图** | 人眼不擅长比较角度面积 | 默认禁用；仅在仅展示 2 个组成部分且目的是强调构成关系时可考虑，其余用水平条形图或直接标注数字 |
| **3D 效果** | 扭曲视觉感知，制造图表垃圾 | 永远用 2D |
| **次坐标轴** | 两条线在不同刻度上，视觉比较无意义 | 双图垂直对齐共享 X 轴，或直接标注数据值 |
| **条形图非零基线** | 人通过条形长度做比较，基线非零则比较不成立 | 条形图基线必须从零开始 |

### 2.3 进阶图表技巧

- **意大利面图解构**：多系列折线图（>5 条线）杂乱时，四步处理：
  - **先减后画**：评估是否真的需要全部系列，不需要就减
  - **强调一线**：彩色加粗关键线，其余灰化 — 适合演讲逐步揭示
  - **小倍数图-垂直**：子图上下堆叠共享 X 轴 — 适合比较各系列**趋势方向**
  - **小倍数图-水平**：子图左右排列共享 Y 轴 — 适合比较各系列**量级大小**
  - **组合式**：小倍数 + 每面板内强调一条主线 — 适合传阅报告，信息密度高
  - 无论哪种，所有子图必须使用**相同的坐标轴范围**

- **堆叠条形图排序**：关注焦点类别放在一致的基线旁（顶部或底部），用对比色突出

- **正负堆叠条形图**：零基线上方正贡献、下方负贡献，空心框高亮缺口
  - 适用场景：净变化由正负两部分驱动（人员流动、预算增减、gap analysis）
  - 执行要点：水平零基线 = 当前状态；正贡献堆在上方用正向色；负贡献堆在下方用低饱和色；缺口用纯轮廓框标注

- **区分预测与实际**：必须让受众一眼分清"已发生的"和"估计的"
  - 实际数据：实线、粗线、实心标记
  - 预测数据：虚线、细线、更小的实心标记
  - **锚点**（最后实际数据点）：粗体数值标注 + 特殊标记（白底彩色轮廓）— 这是连接已知和未知的桥梁，必须一眼看到
  - X 轴标签用 ACTUAL / FORECAST 全大写区隔
  - 预测区域加微妙背景阴影
  - 禁止合并为无区分线

- **分类数据排序**：
  - **默认规则**：按数值大小排序（从大到小或反之均可），或按有意义的标准（如地区、层级），不要按字母排序
  - **多分析焦点**：当同一批数据需服务多个分析焦点时，选择最主要维度的排序；其他维度的故事用颜色和标注突出，不为每个焦点重新排序
  - **跨图一致性**：多张图涉及同一分类变量时，分类顺序必须保持一致（除非不同图表达完全不同的故事）
  - **书面报告**：分组排列 + 描述框解释每组共性，而非完全依赖排序
  - **例外**：分类数据有自然顺序（如月份、教育程度、职级）时尊重自然顺序，不按数值重排

- **水平堆叠条形（调研排名数据）**：受访者对选项排名次（1st/2nd/3rd）时用
  - 水平方向，左侧放类别名，条形从左到右按排名深浅排列
  - 按总选择率从高到低排序类别； omit x-axis，直接标注数值
  - 最多 4 个排名层级，超过则不适合此图表

- **比例变化的替代方案（双饼图场景）**：当需要比较两个时间点的构成变化时：
  - **简单分组条形图**：比较各分类的绝对值变化 — 适合关注具体数值的场景
  - **100% 堆叠水平条形**：比较各分类的相对占比变化 — 适合关注份额分配的场景
  - **斜率图**：展示各分类在两个时间点之间的增减方向和幅度 — 适合关注变化方向（注意：斜率图会按变化幅度重新排序，不保留原始分类顺序）
  - **选择判断**：先问"受众关注绝对值还是占比" → 绝对值用分组条形，占比用堆叠水平条形；再问"是否有明确的比较对象（如 2 个时间点）" → 有则斜率图可选

### 2.4 热力图执行要点

当数据为**同行同单位的数值矩阵**（如绩效矩阵、调研交叉表）时：

- 用单调渐变色（如浅蓝→深蓝）映射值大小，色饱和度 = 值幅度
- 保留单元格内原始数值，支持模式识别 + 精确读取
- 必须配色图/副标题说明映射方向（"低 → 高"）
- 使用色盲友好配色（viridis、ColorBrewer 序列色板）

## Step 3：消除杂乱

> 每个视觉元素都有认知成本。只保留"信息价值 > 认知成本"的元素。

### 3.1 格式塔原则应用

用感知分组代替显性边框和分隔线：

| 原则 | 做法 | 替代的冗余元素 |
|------|------|--------------|
| **接近性** | 相关元素靠近放置 | 去掉分隔线 |
| **相似性** | 一致的色彩、形状、大小表示关联 | 去掉图例（用直接标注） |
| **包围** | 用微妙背景色区分区域 | 去掉粗重边框 |
| **闭合** | 去掉外框，受众自动感知完整性 | 去掉图表外框 |
| **连续性** | 沿平滑路径对齐元素 | 去掉不必要的引导线 |
| **连接** | 线条强关联元素 | 代替颜色区分 |

### 3.2 杂乱审计清单

对图表中的每个元素问："它带来的信息价值值得它的认知成本吗？"

**删除（不必要）**：
- [ ] 去掉装饰性边框和网格线（保留必要参考线）
- [ ] 去掉多余颜色（没有信息含义的颜色 = 噪音）
- [ ] 去掉冗余标签（能用直接标注代替图例的就去掉图例）
- [ ] 数值标注代替次坐标轴
- [ ] 表格：
  - 去掉粗重边框，用白色空间分隔
  - 数值右对齐、文字左对齐
  - **避免背景色和阴影**（白空间是更好的分隔方式）
  - **载体适配**：报告/印刷品 → 可含更多行列精确值；仪表盘 → 摘要 + 提取关键洞察；演讲 → 逐行阅读（用动画/标注引导视线）
  - **混合单位**：表格适合呈现混合单位数据（不同单位在同一表格中排版优于拼合成图）

**确保存在（必要）**：
- [ ] 所有轴有清晰标签（含单位）
- [ ] 有数据来源脚注
- [ ] 缩略语已定义
- [ ] 关键异常有上下文注释（如"峰值因春节"、"Q3 数据不含 9 月"）
- [ ] 15 秒无领域知识可理解主要结论

## Step 4：引导注意力

> 用前注意属性（preattentive attributes）让关键信息在一秒内跳出来。

### 4.1 前注意属性工具箱

| 属性 | 最佳场景 | 注意 |
|------|---------|------|
| **颜色** | 高亮关键数据点 | 关键用彩色，其余灰化 |
| **大小** | 放大信号重要性 | 不要超过 2 级大小差异 |
| **加粗** | 文字中的关键数字 | 最适合标题和标注 |
| **位置** | 放在序列首位或焦点位置 | 左上角和顶部最优先被看到 |
| **形状标记** | 标注异常点 | 少量使用，不喧宾夺主 |

### 4.2 黄金法则

**突出不超过 10% 的视觉元素，其余全部弱化。**

操作化定义：
- 条形图/折线图（数据点 ≤20）：高亮 ≤ 2 个关键数据点
- 散点图：高亮 ≤ 1 个关键簇或象限
- 热力图：高亮 ≤ 10% 的单元格，且为连续区域
- 验证：闭眼睁眼看图 1 秒，第一眼落点必须是你要传达的关键信息

做法：关键的 1-2 个数据点用彩色/加粗，所有其他数据用浅灰色。不要试图让所有东西都"重要"——全突出 = 没突出。

### 4.3 属性联动规则

前注意属性不是独立工具，需要**协同设计**：

1. **数据点-标题-标注同色联动**：高亮的数据点、对应标题中的关键词、对应标注线/文字必须用同一颜色。这是视觉闭环——读者从标题看到颜色，视线自然落到同色的数据点
2. **极端关键元素可叠加 2-3 个属性**（大 + 粗 + 彩色），一般关键点只用 1 个属性（颜色或加粗）
3. **验证标准**：给没有上下文的人看 1 秒，问他第一眼看到什么——答案必须和你的 Big Idea 一致

### 4.4 标题策略

- **行动导向标题**：直接说出结论，不是描述内容
- 好："伽玛履约率连续 3 个月下滑至 78%"
- 差："供应商履约率趋势图"

## Step 5：像设计师思考

> 把上述所有策略整合为一份连贯的大纲。

### 5.1 迭代方法

| 方法 | 做法 | 适用场景 |
|------|------|---------|
| **纸笔草图** | 先画草图，关注布局和信息层级，不关注精确度 | 任何图表设计 |
| **验光师方法** | (1) 从服务于 Big Idea 的最关键视觉元素开始测试；(2) 版本 A → 复制为版本 B → 只改一个元素 → 并排对比；(3) 判断标准 = 哪个版本让 Big Idea 更一目了然；(4) 停止条件 = 连续两轮改动未带来明显提升 | 优化已有图表 |
| **外部反馈** | 完整流程：(1) 找代表目标受众的同事（非设计团队）；(2) 不给任何解释，直接展示；(3) 用 4 条标准引导问题："你首先注意到什么？"、"你认为这张图在说什么？"、"有什么让你困惑的？"、"你会怎么改？"；(4) **不辩护、不纠正**——仔细听反馈，不解释你的设计意图；(5) 分析"意图-接收差距"：你希望受众看到什么 vs 他们实际看到了什么；(6) 根据差距决定修改方案，然后进入下一轮反馈循环 | 验证视觉策略 |

### 5.2 时间预算

数据叙事的时间应与分析时间相当。拒绝"画完图就走"（graph-and-go）的陷阱——默认图表设置缺乏叙事意图，永远不够。

### 5.3 情境适配

**载体规则**：

| 载体 | 规则 |
|------|------|
| **演讲** | 一页一图，精简，视觉强化口头观点，可用动画逐步揭示 |
| **报告/邮件** | 多图一页可接受，确保自包含，用完整标注的静态版 |
| **仪表盘** | 表格可扫视，提取关键洞察，避免完整大表格 |
| **混合体（幻灯文档）** | 慎用。需分别检查演示可读性和报告自包含性，任一不通过则拆分为两个版本 |

**视觉色调选择**：
- **urgent（紧迫型）**：高对比色、粗线条、强调异常和下降趋势 — 适合问题诊断、风险预警
- **analytical（分析型）**：中性色调、细致标注、包含上下文参考 — 适合战略评审、月度汇报
- **inspirational（激励型）**：品牌色 + 积极基调、强调增长和成果 — 适合启动会、愿景对齐
- 选择后，标题措辞、颜色系统、图表风格须保持一致

**深色背景处理**：
- 深色背景 ≠ 反转颜色。关键内容（数据、标题）用高明度色突出，次要元素（网格、轴）用低明度色弱化
- 保持高对比：浅色系列文字和数据点 vs 深色背景
- 色盲友好配色在深色背景下依然有效
- 首选浅色背景，除非演示环境明确要求深色（如暗场演讲）

**动画 vs 静态双版本策略**：
- 演讲版常见误区是把静态图表"飞入"就完事。真正的动画版：逐步揭示数据点、标注逐步出现、结论最后呈现
- **动画→最终帧 = 静态版**：动画的最后一帧就是你需要的静态自包含版本
- **多页动画仅传最终页**：如果做了多页逐步揭示的动画，只把最终页（信息完整的那一帧）作为报告/附件
- **演示者独立性**：即使没有你引导，静态版也必须能独立传达核心信息
- **叙述同步**：每步动画对应一句话叙述，不要让动画和讲述脱节

## Step 6：讲述故事（三幕叙事）

> 在确定图表清单后、编写叙事之前，先用**故事板**方法：把每张图的核心结论写在卡片上，在桌面排列叙事顺序，调整直到形成流畅的逻辑链条。这是低成本测试叙事结构的最佳方法。

### 6.1 三幕结构

| 幕 | 内容 | 图表编号 | 核心问题 |
|----|------|----------|---------|
| **第一幕（铺垫）** | 设定场景、指出失衡（什么变了/为什么现在要行动）、定义期望平衡态（理想未来是什么）、**受众即主角**（为什么你是解决这个问题的人） | 图 1-2 | "我为什么要在意？" |
| **第二幕（冲突）** | 用证据展开问题、对比案例、不行动的后果，**将受众动机贯穿始终** | 图 3-5 | "所以呢？" |
| **第三幕（决议）** | Bing Bang Bongo 收口 + 明确行动号召 | 图 6-7 | "你想让我做什么？" |

**Bing Bang Bongo 收口策略**：
1. **Bing（预告）**：开场时预告核心结论 — "今天我想让你们看到 X，并批准 Y"
2. **Bang（展开）**：第二幕用证据展开（已在第二幕完成）
3. **Bongo（回顾收口）**：收尾时行动导向回顾 — "我们看到了 X，它意味着 Y，所以下一步是 Z"

### 6.2 多图表排序规则

1. **先全局后细节**：先整体趋势，再拆解分项
2. **先问题后方案**：先"问题图"，再"解决方案图"
3. **先事实后推断**：先实际数据，再预测/建议
4. **每张图只讲一件事**

### 6.3 数据伦理

论证类沟通（汇报、提案、建议）：
- **不得只呈现支持结论的数据**，必须提及已知反对证据或局限性
- 把反对点当作分析全面性的证据，不是弱点
- 单方面呈现只支持结论的数据会损害可信度

**披露分级矩阵**：

| 场景 | 披露深度 |
|------|---------|
| 高利害 + 低信任 + 专家受众 | 完整披露反对证据，逐条回应 |
| 低利害 + 高信任 + 一般受众 | 概括性提及局限性 |
| 中间地带 | 按实际情况取舍，但至少提及关键局限 |

**证据分类**：supporting（支持结论的）、opposing（反对结论的）、contextual（提供上下文的）— 三类都必须被审视，opposing 和 contextual 按分级矩阵决定披露深度。

## 独立审查

> 大纲写完后，**不要自己审自己**。派一个子代理，只给 Markdown 大纲，不给任何背景上下文。

### 审查输入约束

| 给什么 | 不给什么 |
|--------|----------|
| Markdown 大纲全文 | 数据原始表格 |
| 图表策略描述 | 背景上下文（如"这是给供应商季度汇报"） |
| Big Idea | 业务细节 |

**为什么不给上下文**：审查者知道背景会"自动补全"逻辑缺口，失去发现问题的意义。

### 审查维度

| 维度 | 检查点 |
|------|--------|
| **Big Idea 清晰度** | 一句话能否说清？是否有洞察 + 建议？ |
| **图表必要性** | 每张图是否服务于 Big Idea？有没有"为了画图而画图"？ |
| **高亮合规** | 每张图的高亮是否 ≤10%？是否只高亮一个关键点？ |
| **排序一致性** | 多张图讲同一分类时，顺序是否一致？ |
| **叙事连贯性** | 上一张到下一张是否有逻辑递进？ |
| **标题质量** | 标题是否行动导向（直接说结论），还是空洞描述？ |
| **数据伦理** | 是否提及已知局限性或反对数据？ |
| **图表类型合理性** | 图表类型是否匹配数据关系（不是"什么都用折线图"）？ |
| **数据推理** | 推理有实质计算而非空洞描述？每条标注了置信度？排名变化和季节性因素被纳入考虑？ |

### 审查输出

```
## P0（必须修复，阻塞进入渲染阶段）
- [ ] 问题描述 + 修复建议

## P1（强烈建议修复）
- [ ] 问题描述 + 修复建议

## P2（可选优化）
- [ ] 问题描述
```

### 审查后行动

- **有 P0**：修复后重新审查
- **无 P0，有 P1**：修 P1 后传递给下游
- **无 P0 无 P1**：直接传递给下游

### 跳过审查的条件

- 简单场景（1-2 张图的快速展示）
- 紧急汇报（时间窗口 < 30 分钟）
- 探索性分析（非对外沟通）
