# 08-经验沉淀

## 概述

策展做完了不等于结束了。每次策展都是一次学习，经验如果不沉淀，下次还会踩同样的坑。这个阶段做的事情是：回顾过程，提炼经验，更新资源库。

---

## 什么时候做

- 策展项目完成（原型已交付）
- 原型第一版出来后（中期复盘）
- 收到重要反馈后
- 发现了可复用的方法

不是每次都要做。但如果做了策展，值得花 10 分钟回顾一下。

---

## 回顾问题

### 关于定题

- 策展地图的结构是否准确反映了材料的本质？
- 规模判断是否正确？（小型/中型/大型）
- 有没有遗漏的主题？
- 策展目的是否从头到尾保持一致？

### 关于材料

- 有没有材料缺口在后期才发现？
- 有没有材料被过度使用或完全没用到？
- 材料的呈现方式是否合适？

### 关于叙事

- 叙事结构是否服务了策展目的？
- 信息层级是否清晰？
- 有没有"讲不清楚"的地方？原因是什么？

### 关于视觉和实现

- 视觉语言和可视化技能之间有没有冲突？
- 标准单元是否成功扩展到同类单元？
- 有没有用 CSS 掩盖结构性问题？

### 关于流程

- 哪个阶段花的时间最多？为什么？
- 有没有跳阶段或回退？原因是什么？
- 下次可以优化什么？

---

## 提炼内容

回顾后，提炼以下内容：

### 1. 可复用的方法

如果发现了好的做法，写成简洁的方法说明：

```
## {方法名称}

**场景**：什么时候用
**做法**：怎么做
**注意**：什么情况下不适用
```

追加到 `references/方法库.md`（如不存在则创建）。

### 2. 踩过的坑

如果踩了坑，写成警示：

```
## {坑的名称}

**现象**：发生了什么
**原因**：为什么发生
**解法**：怎么解决的
**预防**：下次怎么避免
```

追加到 `references/踩坑记录.md`（如不存在则创建）。

### 3. 案例沉淀

如果这次策展有值得参考的产出物，整理到 `references/` 下：

- 定题说明 → `references/案例-{项目名}-定题.md`
- 页面结构 → `references/案例-{项目名}-页面结构.md`
- 视觉语言 → `references/案例-{项目名}-视觉语言.md`
- 原型文件 → 保留原始路径，在 references 中记录链接

### 4. 技能改进建议

如果发现策展工作台的某个阶段有问题，记录改进建议：

```
## {阶段名} 改进建议

**问题**：当前有什么问题
**建议**：怎么改
**优先级**：高/中/低
```

---

## 输出

- 更新 `references/方法库.md`（如有新方法）
- 更新 `references/踩坑记录.md`（如有新坑）
- 添加案例文件到 `references/`
- 记录改进建议（如有）

---

## 自检

经验沉淀完成后，检查：

- [ ] 提炼了 ≥1 条可复用的方法
- [ ] 记录了踩过的坑（如果有）
- [ ] 案例文件已整理到 references/
- [ ] 改进建议已记录（如果有）
