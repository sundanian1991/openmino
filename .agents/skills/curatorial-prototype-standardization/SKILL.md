# 原型实现与标准单元扩展

## 概述

在结构和视觉语言稳定之后使用这个 skill。先做一个完整的标准单元，验证它能承载内容和交互，再扩展到同类单元。

## 前置条件

编码或扩展之前，确认：

- 页面或单元角色已经明确。
- 内容字段已经可用，或可以从策展文本中稳定提取。
- 导航和交互层级已经定义。
- 视觉语言和材料处理规则已经存在。
- 被选为标准单元的内容足够成熟，能测试整个系统。

如果这些条件缺失，应回到 `curatorial-narrative-modules` 或 `curatorial-visual-language`。

## 实现流程

1. 检查现有前端工程栈和代码风格。
2. **选择视觉技能路由**：根据项目特点选择合适的视觉技能来执行具体设计实现：

| 项目特点 | 推荐技能 | 适用场景 |
|---------|---------|---------|
| 需要电影级叙事节奏、沉浸式体验 | `fe-cinematic-ui` | 重叙事的策展项目、纪录片式展览 |
| 需要快速迭代、截图驱动调整 | `waza-design` | 需要频繁反馈循环的项目 |
| 需要高保真原型、交互验证、动画 | `huashu-design` | 复杂交互、需要演示流程的项目 |
| 极简工业设计、数据驱动、结构即装饰 | `nothing-design` | 数据密集型策展、技术/档案类展览 |
| 通用Web制品、需要全面设计系统 | `web-design-engineer` | 通用性强的策展网站、多类型页面 |

选择后，加载对应技能的SKILL.md和references，按其流程执行视觉实现。策展文本、页面结构和视觉语言规则作为输入，视觉技能负责将其转化为可运行的代码。

3. 为标准单元定义结构化内容字段。
4. 围绕页面角色和模块字段构建或重构组件。
5. 实现与策展结构对应的交互状态。
6. 先测试内容最丰富的标准单元。
7. 通过替换数据和校准差异扩展到同类单元。
8. 检查桌面端和移动端的阅读顺序、文字溢出、导航和材料渲染。

## 标准单元

标准单元可以是专题页、案例页、物件页、地点页、时间单元页、档案条目页、访谈页或分析节点。它不一定是传统意义上的标准页面，关键是它代表一种可重复的内容结构。

## 输出

输出可运行的代码修改、组件和数据结构说明，以及验证结果。如果实现时暴露结构缺口，应记录缺口，而不是用 CSS 或一次性代码掩盖。

## 参考范式

《汴京十二时辰》使用一个内容最成熟的高光单元校准同组页面。未来项目可以使用其他重复单元：展品、路线、访谈、文档、地点、案例或片段。
