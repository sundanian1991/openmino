# Design Process

> 从理念到实现的完整路径。不是"选组件填内容"，而是"发现张力，用组件表达"。

---

## 核心洞察

**现有流程的问题：**

```
扫描代码 → 问受众 → 选宏观结构 → 选主题 → 加载规则 → 选组件 → 构建
```

这是**组件驱动**的：你已经在选组件了，但还不知道内容的核心张力是什么。

**应该的流程：**

```
研究材料 → 发现张力 → 视觉表达 → 组件选择 → 克制使用 → 细节精度
```

这是**理念驱动**的：组件是"从内容长出来的"，不是"套上去的"。

---

## 为什么这次 Haglöfs 不同

| 步骤 | 现有流程 | 这次 Haglöfs |
|------|---------|-------------|
| **起点** | 扫描代码 | 研究品牌设计系统 |
| **理解** | 问受众/用例/基调 | 发现核心张力（Nature vs Tech） |
| **组件选择** | 选好看的 | 选能表达张力的 |
| **颜色** | 好看就行 | 每个色号都在说 Nature vs Tech |
| **布局** | 标准网格 | 斜切打破稳定，创造张力 |
| **细节** | 够用就行 | 1px 都在思考 |

## 完整流程

### Phase 1: 研究（Before Design）

**目标：** 理解内容的核心张力，而不是直接选组件。

**步骤：**

1. **材料扫描**
   - 有 URL → WebFetch 研究品牌设计系统
   - 有截图 → 提取设计 DNA
   - 有文字 → 理解核心信息
   - 什么都没有 → 从 brief 推断

2. **发现张力**
   - 问：这个品牌/内容的核心二元对立是什么？
   - 常见张力：
     - Heritage ↔ Innovation
     - Organic ↔ Engineered
     - Nature ↔ Technology
     - Tradition ↔ Future
     - Simplicity ↔ Complexity
     - Human ↔ Machine

3. **一句话表达**
   - 用一句话说清楚：这个内容要表达什么？
   - 例：Haglöfs = "瑞典户外品牌，110年传承，自然与科技的张力"

### Phase 2: 视觉表达（Design Decisions）

**目标：** 用视觉元素表达核心张力。

**步骤：**

4. **色彩表达**
   - 问：这个张力用什么颜色表达？
   - 例：Nature ↔ Tech → 自然灰调（#4A6741）+ 技术亮色（#E8453C）
   - 原则：灰调色比纯色更高级（"说"而不是"喊"）

5. **字体表达**
   - 问：这个张力用什么字体表达？
   - 例：Heritage ↔ Future → 传统 serif + 现代 mono
   - 原则：2+1 规则（Display + Body + Technical companion）

6. **布局表达**
   - 问：这个张力用什么布局表达？
   - 例：Stable ↔ Dynamic → 斜切打破矩形
   - 原则：斜线创造张力，矩形创造稳定

7. **节奏表达**
   - 问：这个张力用什么节奏表达？
   - 例：Heavy ↔ Light → 深色段落（重音）+ 浅色段落（呼吸）
   - 原则：有变化的节奏，不是简单交替

### Phase 3: 组件选择（Component Selection）

**目标：** 选择能表达张力的组件，而不是"好看的"组件。

**步骤：**

8. **组件适配**
   - 问：这个内容需要什么样的组件来表达张力？
   - 不是"我有这个组件就用"，而是"这个内容需要什么"
   - 例：Haglöfs 需要色卡展示（不是标准卡片）

9. **克制使用**
   - 问：哪些组件我不用？
   - 不是"能用就用"，而是"只用能表达张力的"
   - 例：Haglöfs 没有用 Testimonials，因为品牌设计系统不需要

### Phase 4: 执行（Execution）

**目标：** 精确执行，1px 都在思考。

**步骤：**

10. **细节精度**
    - 问：这个 1px 重要吗？
    - 答案：重要
    - 例：箭头是 1px 线 + 8px 三角，不是粗箭头

11. **留白呼吸**
    - 问：这个空间给内容呼吸了吗？
    - 例：section padding 120px，不是 40px

12. **色彩克制**
    - 问：这个颜色是"喊"还是"说"？
    - 例：#E8453C（略带橙）比 #FF0000（纯红）更高级

## 设计语言 vs 组件

**核心洞察：** 高级感来自"设计语言"，不是"组件"。

**组件** = 具体的 UI 元素（按钮、卡片、导航）
**设计语言** = 共享的 token、原则、节奏

**例子：**

```
showcase 和 article 共享：
- Token 系统（--color-offwhite, --color-cream...）
- Hero 斜切（clip-path polygon）
- Section 结构（120px padding, section-label）
- 字体（Georgia + Helvetica Neue）
- 留白（120px）
- 响应式断点
- 动画（fadeUp）

但它们是不同的页面类型：
- showcase — 品牌设计系统展示
- article — 文章阅读
```

**如何建立设计语言：**

1. **Token 系统** — 基于核心张力选择颜色
   - 例：Nature ↔ Tech → 自然灰调（#4A6741）+ 技术亮色（#E8453C）
   
2. **设计原则** — 明确的规则
   - 留白：120px section padding
   - 克制：只用能表达张力的组件
   - 精度：1px 都在思考
   
3. **视觉节奏** — 有变化的呼吸
   - 浅色 → 深色 → 浅色 → 深色
   - 不是简单交替，而是有变化的节奏

4. **变种能力** — 基于语言创造变种
   - showcase：品牌设计系统展示
   - article：文章阅读
   - landing page：落地页
   - documentation：文档

**检查清单**

在交付前，问自己：

- [ ] 我研究过这个材料吗？
- [ ] 我发现了核心张力吗？
- [ ] 我用一句话说清楚了吗？
- [ ] 每个颜色都在表达张力吗？
- [ ] 每个字体都在表达张力吗？
- [ ] 布局在表达张力吗？
- [ ] 我克制使用组件了吗？
- [ ] 细节精度够吗？（1px 都在思考）
- [ ] 留白给内容呼吸了吗？
- [ ] 色彩是"说"而不是"喊"吗？

---

## 反模式

1. **组件驱动** — "我有这个组件，我来填充内容"
2. **跳过研究** — 直接选组件，不理解内容
3. **过度使用** — 能用就用，不克制
4. **纯色呐喊** — 用 #FF0000 而不是 #E8453C
5. **填满空间** — 40px padding，不给呼吸
6. **忽略细节** — "够用就行"
