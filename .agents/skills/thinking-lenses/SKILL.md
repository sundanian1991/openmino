# 思维透镜 (Thinking Lenses)

## 核心理念

> "More mental models means you have more ways to solve more problems." — Charlie Munger

人的最大思维陷阱，是误以为自己看到的就是全部。25个思维透镜 = 25把看问题的钥匙。

### 浅层做法 ❌

```
Doctor: 诊断是什么？
Engineer: 能建模吗？
Hacker: 底层在运转什么？
```

→ 三个问题换了皮，本质都是"你觉得问题在哪？"

### 深层做法 ✅

每个职业加载一套完整的**认知协议**——不是一句话问题，是 3-6 步微型思维算法。
- 每步强制输出中间推理
- 使用该职业的专业语言约束
- 如果两个透镜的核心判断用了相同思考框架 → 判定切换失败，重新加载

> 完整 25 个认知协议见 `references/cognitive_protocols.md`

## 动态思考旅程（内部运作流程）

每次调用遵循统一的5阶段旅程。**前3个阶段是强制门——不跳过。**

🛑 **CHECKPOINT**：Situation → Check-In → Input Gate 三个阶段必须按顺序完成，不跳过任何一步。跳过 = 输出质量不可控。

```
1. Situation（锚定问题）
   └─ 接收用户输入 → 隐性识别问题类型/情绪/约束
   └─ 如果输入含糊 → 轻量Check-In（最多1-2个追问）
   └─ 如果带具体内容 → 进入内容审查模式
   └─ 🛑 1a. 结构化摘要（CHECKPOINT）：把用户输入拆成5个要素——背景、目标、约束、痛点、模糊点
        - 每个要素用一句话概括，包含用户原话中的具体数字/人名/资源限制
        - 如果某个要素缺失（比如没有明确目标），标注"用户未明确"而不是自己补
        - 这5个要素是后续所有步骤的锚点，不跳过

2. Check-In（用户适配）
   └─ 系统内部完成：问题分类 + 情绪识别 + 约束判断
   └─ 不增加额外交互轮次，除非必须确认关键信息

3. 🛑 Input Gate（CHECKPOINT — 选透镜前必须完成）
   └─ 3a. 事实/观点校验（Journalist精神）：扫一遍输入，区分事实/推测/情绪。推测不丢弃但标"待核实"
   └─ 3b. 内部判定（一次性完成，不单独输出）：
        - 认知偏差扫描：对照8种偏差信号（见 usage_guide.md），有则标记
        - 场景匹配：扫9大预设场景，命中的以场景推荐为起点
        - 跨类型判定：翻译为映射表中的问题类型 → 主类型 + 次类型，输出时告知用户
   └─ 3c. 查表交叉（强制）：对每个已判定的类型，查映射表（cognitive_principles.md）获取推荐透镜 → 交叉筛选 → 初选集合（4-7个候选）
        - 交叉筛选时优先保留在多个类型中重复出现的透镜

4. Tool Thinking（视角调用）
   └─ 基于 3c 的初选集合，结合跨类别覆盖检查（≥3个认知原理类别）确定最终透镜组合（3-5个）
   └─ 🔴 4a. 回环验证（CHECKPOINT — 产生通透感的关键步骤）：
        - 选完透镜后，回到用户原话中标注的"痛点"要素
        - 问自己：这个组合真的对准了用户最痛的点吗？
        - 如果没对准 → 回到 3e 重新筛选
        - 如果对准了 → 继续下一步
        - 回环最多做1次，不无限循环
   └─ 按下方「透镜输出协议」逐个输出分析

### 透镜输出协议（每个透镜必须遵守）

#### 铁律 1：加载完整认知协议

每个透镜必须加载完整认知协议（见 `references/cognitive_protocols.md`），按步骤逐一执行，每步输出中间推理。

**协议执行要求**：
- 使用该职业的语言约束词汇（如 Doctor 用"主诉""鉴别诊断"，不用"问题""原因"）
- 每步必须输出中间推理，不能跳过
- 完成全部步骤后，输出一句话核心判断
- 如果两个透镜的核心判断使用了相同思考框架 → 判定切换失败，重新加载

**输出原则**（不是具体格式，是指导原则）：
- **透镜名称格式**：英文在前，中文括号跟后。如 "Engineer（工程师）""Plumber（水管工）""Chess Master（象棋大师）"。不只写中文（"工程师"），也不只写英文（"Engineer"）——用户可能不熟悉职业对应的英文名
- **不显示步骤标签**：不要输出 "Step 1"、"第一步" 这样的标签，用内容本身组织
- **不显示语言约束标签**：不要输出 "语言约束：用XX思考"，语言约束在内部执行
- **用"核心问题"统领**：每个透镜开头用一句话点破核心，统领整个分析
- **让分析自然流动**：像在讲一个分析故事，不是在执行一个清单
- **关键问题锚定**：每个透镜必须有一个关键问题，锚定用户输入中的具体事实
- **行动优先级用时间层**：用 "立即→本周→决定→准备" 或类似时间维度，比 🔴🟡🟢 更具体
- **必须有"一句话总结"**：收尾用一句话概括核心判断，干脆利落

> 具体格式参考 `references/output_examples.md` 中的高质量示例

#### 铁律 2：永不重复上一轮的角度

输出当前透镜前，检查上一个透镜的核心判断：
- 如果当前分析结构和上一个同构（只是换了职业标签），立即停止
- 换一个异质性框架重新切入：上一个拆结构 → 这一个追问矛盾；上一个对比张力 → 这一个推演后果
- 这个检查在输出前内部完成，不暴露给用户

#### 冲突矩阵（主动制造张力）

故意让对立视角碰撞，暴露单一视角看不到的张力：

| 冲突对 | 张力 | 适用场景 |
|--------|------|---------|
| Artist × Accountant | 创造力 vs 数字纪律 | 制度设计、创新评估 |
| Soldier × Entrepreneur | 流程 vs 试错 | 体系搭建、风险决策 |
| Journalist × Novelist | 事实 vs 叙事 | 向上汇报、晋升述职 |
| Engineer × Hacker | 建模 vs 拆解 | 问题诊断、系统设计 |
| Politician × Mathematician | 可信 vs 可证 | 跨部门推动、方案说服 |
| Doctor × Philosopher | 诊断 vs 追问后果 | 业绩下滑根因分析 |

#### 反例：这样不算完成认知呼吸

| 错误 | 示例 | 应该 |
|------|------|------|
| 锚定用了抽象描述 | "从激励角度看，供应商管理的核心是利益分配" | "你面对的30家供应商利润收窄，激励设计的前提已经变了" |
| 展开只有信息搬运 | "激励机制包括：1.物质激励 2.精神激励 3.竞争激励" | "利润收窄时物质激励空间为零，但你还在用利润分成做杠杆——这个杠杆已经空转了" |
| 验证用了通用提问 | "你觉得这个方向可行吗？" | "30家里有几家是真的没钱赚，还是在用'没钱赚'当谈判筹码？" |
| 收尾带导游话术 | "通过这个视角，你会发现问题根源在于..." | "问题根源是激励工具和利润环境不匹配，不是供应商不配合" |
| 两个透镜同构 | A(Economist)："激励不匹配→不愿配合" B(Salesperson)："动机没对准→缺乏动力" | 一个讲激励结构，一个讲关系信任——不同维度 |
| 跳过协议步骤 | 直接输出结论，没有中间推理 | 严格按协议步骤执行，每步输出中间推理 |
| 不用语言约束 | 用通用语言描述，不用职业术语 | Doctor 用"主诉""鉴别诊断"，Hacker 用"表面声明""信任边界" |

5. Synthesis（综合洞察）
   └─ 5a. 冲突焊接（CHECKPOINT — 综合阶段的质量基石）：
        - 列出所有透镜的核心判断（一句话每个）
        - 识别矛盾：哪些互相冲突？哪些是同一发现的不同表述？
        - 对每个冲突：
          · 是事实矛盾还是视角差异？
          · 如果事实矛盾 → 哪个透镜的证据标准更高？
          · 如果视角差异 → 综合后你看到了什么单一视角看不到的？
        - 识别汇聚：多个透镜指向同一个发现 → 这是最高信度的信号
        - ⚠️ 冲突焊接不是"列一遍结论"，是找碰撞和汇聚。如果找不到任何冲突或汇聚，说明透镜结果还没真正交叉 → 回到 4a 重新验证
   └─ 5b. 涌现检查（基于 5a 的冲突焊接结果）：
        - 冲突焊接后，是否涌现出了单个透镜看不到的新判断？
        - 如果是 → 这就是综合洞察，放在输出最前面
        - 如果否 → 说明冲突焊接还没到位，不是透镜不够多而是碰撞不够深
   └─ 5c. 结尾设计（强制——决定对话能不能继续）：
        - ❌ 交作业式收尾：平铺行动清单 + "需要我帮你深入展开某个透镜吗？" → 用户拿到清单就走了
        - ✅ 持续对话式收尾：在行动优先级之后，留一个"关键不确定性"——这次分析发现了什么还没想清楚的东西，用一个具体的问题抛给用户
        - 具体做法：从 5b 的涌现判断中，提取一个最值得继续深挖的矛盾或假设，用一句话抛出来
        - 示例："三方激励交集为零"就是这样一个结尾——它不是答案，是让用户想继续想的东西
        - 不要用封闭式提问（"选A还是B？"），用开放式（"这里最大的不确定性是X，你怎么看？"）
   └─ 跑一遍"执行自检清单"（见下方）——输出前最后一道质量门
   └─ 输出"行动优先级"汇总
   └─ 使用记录附在末尾（附录角色，不是收尾角色）
```

## 模糊输入识别与隐形Check-In

用户经常表达含糊——情绪化的抱怨、零散的想法、没有明确问题陈述。**系统应在内部完成识别和适配，不增加额外交互轮次。**

### 识别信号
- 输入以情绪词开头（"烦死了""感觉不对""事多且杂"）
- 只有症状描述，没有明确问题（"供应商不配合""方案写不下去"）
- 输入长度短且跳跃，缺乏上下文
- 用户说"帮我看看"但没有指定看什么

### 隐形Check-In流程（系统内部执行）

**Step 1：问题分类**（自动识别，不问用户）
| 输入特征 | 判定类型 | 进入模式 |
|---------|---------|---------|
| "帮我看看这段内容/方案/邮件" | 内容审查 | 模式E |
| "全面审视/有没有漏掉" | 全面扫描 | 模式B |
| 具体问题描述（有主体、有症状） | 精准诊断 | 模式A |
| "卡住了/想不出" | 创意突破 | 模式C |
| "用XX视角看看" | 单透镜深入 | 模式D |

**跨类型问题处理**：
现实问题往往是跨类型的（如"供应商不配合"既涉及人际又涉及系统）。此时：
- 判定**主类型**（最突出的那个维度）
- 标注**次类型**（隐藏的维度）
- 输出时明确告知用户："这个问题涉及[主类型]和[次类型]，我先从[主类型]切入，同时补充[次类型]的视角"
- 主类型给2个透镜，次类型给1个透镜，避免透镜数量爆炸

**Step 2：轻量补充**（只在必要时问，最多1个追问）
- 如果无法判定目标（决策/方案/说服/梳理）→ 问："你希望通过这次思考达成什么结果？"
- 如果带情绪但无事实 → 问："能描述一下最近发生的具体事件吗？"

**Step 3：约束判断**（内部推断，不询问）
- 从上下文推断时间、资源、权力边界等约束
- 如果推断不确定，在输出中标注"假设你有X权限，如果没有请告诉我"

## 事实/观点区分与校验协议

用户的输入往往混合了事实、观点、假设和情绪。**每次使用透镜前，先用 Journalist 精神做一次快速扫描**：

- **事实** → 直接接受，标注'待核实' if 无法即时验证
- **观点/假设** → 明确标记，用 Scientist 或 Philosopher 透镜检验
- **情绪** → 提取背后的具体事实，剥离情绪外壳

> 详细校验流程与表达模板见 `references/usage_guide.md` → '事实/观点校验' 章节

## 五种使用模式

根据用户情境选择最合适的模式。模式选择规则见下方「快捷指令表」。

### 模式A：精准诊断（默认）
**触发**：用户描述了一个具体问题、困惑或决策（不含具体文档内容）
**流程**：
1. 分析用户问题的性质（决策类/诊断类/创意类/沟通类/系统类）
2. 从25个工具中推荐**3-5个最相关**的透镜
3. **盲区覆盖裁决（停止信号）**：基于用户问题的特征，判定核心盲区和补充盲区
   - **核心盲区（必看）**：Top 2 透镜，覆盖约80%的高频盲区
   - **补充盲区（建议看）**：第3个透镜，覆盖特定场景下的遗漏风险
   - **深度盲区（可选）**：第4-5个透镜，仅在决策重大或时间充裕时展开
4. 对每个推荐透镜，以分析为主体输出具体判断（指出盲区/假设/矛盾），需要验证时用锚定提问收尾
5. 请用户选择想深入探索的1-2个透镜

#### 模式A-深挖版：二次补充协议
**触发**：用户对第一轮结果追问、要求"更全面""更深入"、或系统在Synthesis阶段发现覆盖不足
**前提**：已完成一轮模式A（第一轮），产出了一组透镜分析和行动优先级
**核心逻辑**：第一轮看机制层（系统/激励/流程），第二轮补人和体验层（文化/叙事/心理/感知）。第二轮不是重复扫描，是基于第一轮盲区的定向补充。

**流程**：
1. **盲区盘点**：检查第一轮用了哪些认知类别 → 列出没碰的类别（通常是人际类、创意类、推演类中的2-3个）
2. **定向补选**：从没碰的类别中选3个透镜，优先选能回答"第一轮没解释清楚的人的维度"的透镜
   - 常见补充组合：Anthropologist（沉浸对方世界）+ Novelist（叙事构建）+ Designer（体验设计）
   - 或：Salesperson（深层动机）+ Psychologist（认知偏差）+ Chess Master（多步推演）
3. **锚定**：每个补充透镜的提问必须引用第一轮的发现，形成"第二轮追问"而非"平行分析"
4. **整合输出**：标注"第二轮补充"，与第一轮结论合并为更完整的行动方案

**两轮覆盖效果**：
| 轮次 | 透镜数 | 覆盖类别 | 关注维度 |
|------|--------|---------|---------|
| 第一轮 | 3-5个 | 3-4个认知类别 | 机制层（系统/激励/流程） |
| 第二轮 | 3个 | 补充2-3个未碰类别 | 人和体验层（文化/叙事/心理） |
| 合计 | 6-8个 | 6-7个认知类别 | 机制+人+体验+叙事+风险 |

#### 模式A-快速版：一句话洞察
**触发**：用户说"快速看看""简单说说""一句话"，或输入很短（<30字），或明显需要快速反馈的场景
**流程**：
1. 只推荐**Top 2**最相关的透镜
2. 每个透镜只给**一句话为什么匹配** + **一个核心问题**
3. 不给引导问题列表，直接给结论性洞察
4. 输出控制在200字以内

**快速版 vs 完整版对比**：
| 维度 | 快速版 | 完整版 |
|------|--------|--------|
| 透镜数量 | 2个 | 3-5个 |
| 每个透镜输出 | 一句话+一个问题 | 为什么匹配+2个引导问题 |
| 输出长度 | <200字 | 不限 |
| 使用场景 | 快速咨询、碎片时间 | 深度分析、重要决策 |
| 下一步 | 用户追问才深入 | 主动引导选择深入 |

### 模式B：全视角扫描
**触发**：用户说"帮我全面看看这个问题"或"有没有我漏掉的角度"
**流程**：
1. 将25个透镜按问题性质分成高相关/中相关/低相关三组
2. 每组各选2-3个最有洞察力的提问
3. 用结构化格式输出（表格或卡片），让用户快速浏览
4. **标记"高风险盲区"**：你最可能漏掉的几类透镜

### 模式C：强制跨界组合
**触发**：用户陷入深度思维定势，常规分析无效，或明确说"来个离谱的"
**流程**：
1. 识别用户当前最常用的1-2个思维习惯
2. 故意推荐**最不相关/最反直觉**的2个透镜组合
3. 引导用户思考："如果用一个艺术家+一个水管工的方式处理这个问题，会看到什么？"
4. 强制制造认知冲突，打破框架

### 模式D：单透镜深入
**触发**：用户主动说"我想用XX的视角看看"或"如果从XX角度呢"
**流程**：
1. 确认用户选择的透镜
2. 加载该透镜的完整解释（见 `references/25_tools.md`）
3. 提出3-4个层层递进的引导问题
4. 帮助用户将洞察转化为可执行结论

### 模式E：内容审查
**触发**：用户说"帮我审一下这段内容/方案/邮件/PPT/制度"，或直接粘贴文档内容
**流程**：
1. 识别内容类型（方案/邮件/纪要/制度/PPT大纲）
2. 自动选取**沟通类透镜组合**：
   - **Journalist（记者）**：事实核查——数字、日期、人名是否准确？有无情绪化表达被当作事实？
   - **Novelist（小说家）**：叙事结构——起承转合是否清晰？故事讲得通吗？受众能follow吗？
   - **Designer（设计师）**：信息设计——内容是否在"自动引导"读者到正确结论？排版/结构是否自解释？
   - **Politician（政治家）**：受众感知——读者会怎么理解这段话？有没有 unintended 的负面解读？
   - **Critic（评论家）**：深度解读——有没有遗漏的角度？能在别人基础上构建更有洞察的版本吗？
3. 逐条输出审查发现，每条标注严重度（🔴 必须改 / 🟡 建议优化 / 🟢 可考虑）
4. 输出修订建议（可选：直接输出修订版）

### 认知原理七分类（透镜选择时的跨类别覆盖依据）

25 个透镜按底层认知机制分为 7 类。选择透镜组合时，**必须确保覆盖至少 3 个不同类别**，避免所有透镜集中在同一维度（如全是诊断类，没有执行类）。

| 类别 | 核心机制 | 透镜 | 一句话 |
|------|---------|------|--------|
| **诊断类** | 从症状到根因 | Doctor, Accountant, Engineer, Hacker | 找到问题在哪 |
| **建模类** | 抽象、量化、预测 | Engineer, Architect, Programmer, Mathematician | 把问题结构化 |
| **人际类** | 动机推断、认知弥合 | Salesperson, Psychologist, Teacher, Anthropologist, Economist | 理解人 |
| **推演类** | 多步预判、极限检验 | Chess Master, Philosopher, Politician | 看未来会怎样 |
| **创意类** | 约束释放、快速试错 | Artist, Entrepreneur, Critic, Novelist | 打破定势 |
| **执行类** | 纪律、自动化、设计 | Soldier, Programmer, Designer, Plumber | 保落地 |
| **求真类** | 事实核查、假设检验 | Journalist, Scientist, Mathematician | 去噪音 |

> 注：Engineer、Programmer、Mathematician 跨两个类别——选择时可计入其中任一类别以满足覆盖要求。

## 停止信号（认知负荷控制）

### 核心问题

> 5 个透镜 = 5 次认知切换 = 每次使用都在付认知税。
>
> 用户的时间有限，**不是每个盲区都值得看**。

### 经济学原理

用 **Economist（边际收益）** 的视角控制输出量：

- **第 1-2 个透镜**：覆盖约 **80%** 的核心盲区（边际收益最高）
- **第 3 个透镜**：覆盖特定场景下的遗漏风险（边际收益递减）
- **第 4-5 个透镜**：仅在决策重大、时间充裕或用户明确要求时展开（边际收益最低）

### 裁决规则

每次输出透镜组合时，**必须**附带盲区覆盖裁决表。**盲区列必须用第二人称描述具体盲区**（"你最容易忽略的X"），不能写透镜名或抽象概念：

| 优先级 | 透镜 | 覆盖的盲区（第二人称） | 建议投入时间 |
|--------|------|---------------------|-------------|
| 🔴 核心（必看） | [职业A] + [职业B] | 你最容易忽略的[具体盲区A]和[具体盲区B] | 2-3分钟 |
| 🟡 补充（建议看） | [职业C] | 特定场景下的[具体风险] | 1-2分钟 |
| 🟢 深度（可选） | [职业D] + [职业E] | 重大决策时需补充的[具体维度] | 3-5分钟 |

**错误示例**：
- ❌ "覆盖目标：80%关键盲区"（这是百分比，不是盲区描述）
- ❌ "覆盖的盲区：Economist + Architect"（这是透镜名，不是盲区描述）
- ✅ "覆盖的盲区：你最容易忽略的'供应商在没钱赚时的真实需求'和'联盟一年后到底长什么样'"

### 快速版的本质

模式 A-快速版（2 个透镜）不是"偷工减料"，而是**用 20% 的时间获取 80% 的价值**。

## 🛑 执行自检清单（CHECKPOINT — 输出前最后一道质量门）

**在输出任何透镜分析结果之前，逐项过一遍。不跳过。**

| # | 检查点 | 不通过的表现 | 修复 |
|---|--------|------------|------|
| 1 | **1a 结构化摘要做了吗？** 输入拆成背景/目标/约束/痛点/模糊点5要素？ | 直接跳到分析 | 回到 1a 补拆解 |
| 2 | **3c 查表交叉做了吗？** 透镜选择基于映射表推导还是直觉？ | 直接列出透镜，无推导路径 | 回到 3c 补推导 |
| 3 | **4a 回环验证做了吗？** 选完透镜后回头看用户痛点？ | 透镜"通用正确但不锋利" | 回到 4a 对准检查 |
| 4 | **每个透镜加载了完整认知协议？** 按步骤执行？每步输出中间推理？用了语言约束词汇？ | 跳过步骤、不用职业术语、直接输出结论 | 加载 `references/cognitive_protocols.md`，重新执行 |
| 5 | **铁律 2 检查了吗？** 当前透镜和上一个透镜是否同构？ | 两个透镜只是换了职业标签在说同一件事 | 换异质性框架重新切入 |
| 6 | **冲突矩阵利用了吗？** 有没有主动制造对立视角碰撞？ | 所有透镜都在同一维度分析 | 从冲突矩阵中选一个冲突对，强制对撞 |
| 7 | **停止信号表格式完整？** 4列：优先级、透镜、覆盖的盲区（第二人称）、时间 | 盲区列写的是透镜名或百分比 | 改为"你最容易忽略的X" |
| 8 | **结尾留了关键不确定性？** | 结尾是平铺清单 + "需要展开吗？" | 从涌现判断中提取矛盾，开放式抛出 |
| 9 | **5a 冲突焊接做了吗？** 列出了冲突和汇聚信号？ | Synthesis只是"各透镜结论拼接"，没有涌现 | 回到 5a 跑冲突焊接流程 |

> **如果时间只够做 3 项：做 #4（认知协议执行）、#5（异质性检查）、#9（冲突焊接）。这三项直接决定输出是深度对话还是浅层分析。**

## 输出格式规范与排版标准

### 设计原则

- **标题即结论**：每个标题都是一句话结论，不是名词短语
- **高信息密度**：目标 7-8/10
- **无阴影无渐变**：扁平化区块，顶部细线分隔
- **结构即导航**：用户扫一眼标题和表格就能定位关键信息
- **停止信号优先**：每个输出必须先给盲区覆盖裁决表（🔴核心/🟡补充/🟢深度），让用户根据时间选择深度
- **每段只讲一件事**：长段落（>3句）必须拆——并列2-4项用列表，5项以上或有对比维度用表格，有步骤用有序列表，判断+论据用加粗+列表。宁可多用表格和列表让结构清晰，不堆长段落让读者自己找重点

### 各模式输出要点

| 模式 | 输出必须包含 | 控制项 |
|------|-------------|--------|
| A 精准诊断 | 盲区覆盖裁决表 + 2-5个透镜提问 + 行动优先级 | 🔴≤3项 🟡≤5项 🟢≤5项 |
| B 全视角扫描 | 扫描收敛裁决表 + 高/中/低分组 + 行动优先级 | 标记'高风险盲区' |
| E 内容审查 | 审查优先级裁决表 + 逐条标注（🔴必须改/🟡建议优化/🟢可考虑）| 优先过🔴核心项 |
| 快速版 | 只推Top2透镜 + 一句话匹配 + 一个核心问题 | 输出<200字 |

> 完整输出模板示例与排版细则见 `references/output_style.md`

## 反例与黑名单（不要做什么）

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|------------|---------|
| 1 | **一次性抛出 25 个透镜让用户选** | 用户认知负荷爆炸，等于没推荐 | 先分析再精准推荐 3-5 个 |
| 2 | **每个透镜只提问不分析** | 输出变成"问题清单"，用户没有得到判断 | 分析是主体，提问是收尾验证 |
| 3 | **透镜间换标签不换角度** | Economist 说"激励不匹配"，Salesperson 说"动机没对准"——同一件事 | 铁律 2：异质性检查，换框架切入 |
| 4 | **锚定用抽象描述** | "从激励角度看..."——没有刺穿问题 | 引用用户输入中的具体数字/人名/约束 |
| 5 | **收尾用"你会发现"** | 导游话术，降低判断力 | 直接给判断，句号结尾 |
| 6 | **跳过 Input Gate 直接选透镜** | 透镜选择基于直觉而非推导 | 3c 查表交叉是强制门 |
| 7 | **验证用通用提问** | "你觉得可行吗？"——不包含用户具体事实 | 锚定问题 = 包含用户描述中的数字/人名/约束 |
| 8 | **模式 A 输出没有盲区覆盖裁决表** | 用户无法根据时间选择深度 | 每次输出必须先给 🔴核心/🟡补充/🟢深度 |

## 使用原则

1. **不做简单罗列**：不要一次性抛出25个工具让用户自己选。先分析，再精准推荐。
2. **引导而非灌输**：提出好问题比给答案更重要。每个透镜的核心价值在于那个"核心问题"本身。
3. **鼓励远迁移**：主动提示用户"这个工具来自XX领域，用到你的问题上会看到什么？"
4. **承认边界**：有些组合确实不适用（如小说家诊断医疗问题），不要强行套用。
5. **中文为主**：用户偏好中文沟通。职业名称保留英文以便溯源，解释和引导用中文。
6. **每次输出使用记录模板**：帮助用户积累个人透镜档案。
7. **输出最后必须有"行动优先级"**：不让思考停在洞察层面，必须落地为可执行项。

## 失败模式与 Fallback

流程假设输入清晰、透镜匹配准确，但实操常遇异常。以下预定义 fallback：

| 场景 | 触发条件 | 一线修复 | 仍失败兜底 |
|------|---------|---------|-----------|
| 输入太模糊 | 1a 拆不出 5 要素，3 个以上标"用户未明确" | 追问 1 个关键问题（目标或约束） | 用"症状描述"模式，只做快速版（2 透镜） |
| 透镜匹配不准 | 4a 回环验证发现没对准痛点 | 回到 3c 重新查表交叉 | 降级为模式 B（全视角扫描），让用户自己选 |
| 透镜间同构 | 铁律 2 检查发现当前透镜和上一个重复 | 换异质性框架重新切入 | 砍掉重复透镜，减少到 2 个但保证角度不同 |
| 用户无响应 | 输出后用户沉默或只说"嗯" | 用关键不确定性追问一次 | 停止推进，输出当前结论 + "需要深入哪个方向告诉我" |
| 跨类型无法判定 | 主类型和次类型都模糊 | 默认"精准诊断"模式，选最通用的 3 个透镜 | 直接问用户"这个问题你最想从哪个角度切入" |
