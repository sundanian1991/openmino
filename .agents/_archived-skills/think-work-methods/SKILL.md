# Work Methods — 工作方法与执行体系

> WBS 拆结构，八大方法保执行。结构决定范围，行为决定质量。

---

## 触发信号

| 年老师说 | 触发 |
|----------|------|
| 这个项目怎么做 / 帮我拆一下 | WBS 拆解 |
| 接下来怎么推进 / 分阶段 | WBS + 里程碑 |
| 这个活怎么接 / 怎么干 | 八大方法 |
| 怎么汇报 / 怎么写周报 | STAR / 周报三段式 |
| 出了问题怎么报 | 风险预警法 |
| 做完怎么复盘 | PDCA |

### 与相邻技能区分

| 技能 | 区别 |
|------|------|
| **think-structured** | 认知拆解——理解问题本身（信息太多理不清） |
| **think-task-dispatcher** | 任务调度——模糊需求→澄清→派活执行 |
| **think-work-methods** | **结构+执行**——大项目拆任务树，每个任务怎么接怎么做怎么报 |
| **think-work** | 供应商管理日常——22个场景模板（业绩异常/约谈/巡检等） |

---

## 两层结构

```
第一层：WBS — 项目级拆解
  大目标 → 一级模块 → 二级任务（可分配、可检查）
  回答：谁在什么时候做什么，交付什么

第二层：八大方法 — 个人执行闭环
  接任务 → 拆任务 → 定目标 → 执行协作 → 汇报同步 → 风险预警 → 复盘沉淀
  回答：接到每个任务后怎么干靠谱
```

**WBS 告诉你做什么，八大方法保证你做得靠谱。**

---

## 第一层：WBS 项目拆解

### 核心原则

| 原则 | 说明 |
|------|------|
| 不是清单是结构 | 建立任务间的逻辑关系和依赖 |
| 逐层拆解到可执行 | 底层任务：一个人能独立负责、有明确交付物、能检查 |
| 自上而下 + 自下而上校准 | 先目标分解，再执行层验证是否合理 |
| 设检查节点 | 没有里程碑的 WBS = 待办清单 |

### 执行流程

**Step 1：定义总目标**
```
目标：[动词 + 可衡量的结果 + 时间边界]
例：6月底前完成34家供应商SABC分级，产出差化管理策略
```

**Step 2：一级模块拆解（3-8 个）**
按时间阶段或功能模块拆，每个模块是完成总目标的必要条件。

**Step 3：二级任务细化**
- 一个任务只对应一个交付物
- 一个任务只对应一个负责人
- 任务时长不超过 2 小时（超过就继续拆）

**Step 4：标注依赖与风险**
- 前后依赖关系
- 阶段检查点（里程碑）
- 风险最高的 1-2 个任务

**Step 5：输出任务树**
```
# [项目名称] WBS

**总目标**：[一句话]
**时间边界**：[起止日期]
**负责人**：[总负责人]

## 1. [一级模块A]
- 1.1 [任务] — 负责人：__ 截止：__ 交付物：__
- 1.2 [任务] — 负责人：__ 截止：__ 交付物：__

## 里程碑与检查点
| 节点 | 时间 | 检查内容 | 通过标准 |

## 风险与依赖
- [最高风险]：[为什么] → [缓解措施]
```

### 验收标准

- [ ] 一级模块 MECE（不重叠、不遗漏）
- [ ] 每个叶子任务能直接分配给一个人
- [ ] 每个叶子任务有明确交付物
- [ ] 依赖关系已标注
- [ ] 至少标注 1 个最高风险

## 第二层：八大方法（个人执行闭环）

> 普通员工和靠谱员工的区别不在能力，在行为模式。

### 阶段对比

| 工作阶段 | 普通员工 | 靠谱员工 | 对应方法 |
|----------|----------|----------|----------|
| 接任务 | 直接说"好的" | 先确认目标、标准、边界 | 五问法 + 5W2H |
| 定目标 | "做好一点" | 具体、可衡量、有时限 | SMART |
| 执行协作 | 埋头干，很少同步 | 主动同步，责任清晰 | RACI + 风险预警 |
| 交付结果 | "我做完了" | 讲清结果、数据、影响 | STAR + 周报三段式 |
| 工作结束 | 交完就结束 | 复盘总结，沉淀方法论 | PDCA |

### 一、接任务五问法

接到任何任务（无论来自 WBS 还是临时指派），先问清楚再动手：

| 问题 | 目的 |
|------|------|
| **目标是什么？** | 解决什么问题，避免跑偏 |
| **交付物是什么？** | 最终提交什么结果 |
| **标准是什么？** | 做到什么程度算完成 |
| **时间和优先级？** | 何时需要，与其他任务的紧急程度对比 |
| **资源和边界？** | 可调用什么资源，什么不能做 |

### 二、5W2H 任务拆解

把模糊任务变成可执行计划：

| 维度 | 问题 | 示例 |
|------|------|------|
| **Why** | 为什么做 | 提升供应商分级准确性 |
| **What** | 做什么 | 对34家供应商逐一评分 |
| **Who** | 谁来做 | 我负责，业管提供数据 |
| **When** | 何时完成 | 6月7日前 |
| **Where** | 哪个范围 | 全量34家 |
| **How** | 怎么做 | 按7维度评分表逐项打分 |
| **How much** | 多少资源 | 需要业绩+合规数据各一份 |

### 三、SMART 目标确认

把抽象目标变具体：

| 要素 | 要求 | 对比 |
|------|------|------|
| **S 具体** | 不说"做好一点" | ✘ 提升数据 → ✔ 收藏率到8% |
| **M 可衡量** | 有数据、有数量 | ✘ 多做几家 → ✔ 完成30家 |
| **A 可达成** | 不脱离资源条件 | ✘ 一周评完34家 → ✔ 分两周 |
| **R 相关性** | 与业务目标挂钩 | 分级结果直接影响资源分配 |
| **T 有时限** | 明确截止日期 | 6月7日前完成 |

### 四、RACI 协作分工

多人协作时明确责任边界：

| 角色 | 定义 | 规则 |
|------|------|------|
| **R 执行人** | 动手做的人 | 一个任务可以有多个 R |
| **A 负责人** | 对结果负责的人 | **一个任务只能有一个 A** |
| **C 咨询人** | 需要提前征求意见的人 | 决策前先问 |
| **I 知会人** | 需要同步进展的人 | 做完后告诉他 |

**应用**：WBS 任务树中每个叶子任务标注 RACI，避免"人人参与、没人负责"。

### 五、风险预警法

提前暴露问题，不拖到最后：

**五步框架**：
1. 风险是什么？
2. 影响是什么？（进度/结果/成本）
3. 原因是什么？
4. 方案是什么？（解决思路+措施）
5. 需要什么支持？

**话术模板**：
> "目前任务有一个风险：[风险]。可能影响：[影响]。原因是：[原因]。我准备先[措施]。希望您帮忙协调[支持]。"

**核心原则**：报风险时必须带方案，不能只抛问题。

### 六、STAR 结果汇报

结构化呈现工作价值，不写流水账：

| 要素 | 内容 | 示例 |
|------|------|------|
| **S 背景** | 当时情况 | Q1 绩效面谈指出缺乏体系化 |
| **T 任务** | 要完成的目标 | 建立供应商SABC四级评价体系 |
| **A 行动** | 具体做了什么 | 定7维度权重→试评5家→全量34家评分 |
| **R 结果** | 带来的成果 | S级3家、A级5家、B级20家、C级6家，差异化管理策略已落地 |

**汇报模板**："在 [S] 背景下，我的任务是 [T]。我采取了 [A] 动作，最终实现了 [R] 结果。"

### 七、周报三段式

标准化结构保持透明度：

1. **本周完成**：关键事项及结果（数据支撑）
2. **遇到问题**：风险、阻塞和待解决事项
3. **下周计划**：下一步行动及所需支持

**升级**：增加"关键结果"和"需要支持"两项，提升周报价值。

### 八、PDCA 复盘法

从经验中学习，单次变方法论：

| 阶段 | 问题 | 产出 |
|------|------|------|
| **P 计划** | 原本目标和计划是什么？ | 计划文档 |
| **D 执行** | 实际做了哪些？ | 执行记录 |
| **C 检查** | 结果与目标的差距？ | 差距分析 |
| **A 改进** | 下次保留什么？调整什么？ | 改进清单 |

**复盘四问**：目标是什么？结果怎么样？原因在哪？下次怎么改？

## 两层联动

WBS 拆解 + 八大方法执行的完整流程：

```
WBS 输出任务树
    ↓
每个叶子任务 → 接任务五问法确认边界
    ↓
执行中 → RACI 明确谁做谁负责 + SMART 定具体标准
    ↓
遇风险 → 风险预警法（带方案报风险）
    ↓
交付时 → STAR 汇报结果 + 周报三段式同步进展
    ↓
完成后 → PDCA 复盘沉淀方法论
```

**WBS 负责"不遗漏"，八大方法负责"不跑偏"。**

## 实战案例：供应商绩效评价体系 WBS

```
# 供应商绩效评价体系 WBS

**总目标**：6月底前完成34家供应商SABC分级，产出差化管理策略
**时间边界**：2026-05-10 至 2026-06-30

## 1. 定标准（5月中旬）
- 1.1 确认7维度权重 — 负责人：年老师 截止：5/15 交付物：权重确认表
- 1.2 评分标准量化（1-5分描述） — 截止：5/17 交付物：评分标准文档
- 1.3 数据来源对齐（与业管确认口径） — 截止：5/17 交付物：数据对齐记录

## 2. 试评（5月下旬）
- 2.1 选3-5家试评 — 截止：5/22 交付物：试评结果表
- 2.2 调整权重和标准 — 截止：5/24 交付物：v1.1版标准

## 3. 全量评级（6月上旬）
- 3.1 34家逐一评分 — 截止：6/7 交付物：评分结果表
- 3.2 SABC分级结果 — 截止：6/9 交付物：分级清单
- 3.3 向易人汇报确认 — 截止：6/11 交付物：确认记录

## 里程碑
| 节点 | 时间 | 检查内容 | 通过标准 |
|------|------|----------|----------|
| 标准定稿 | 5/17 | 7维度+权重+评分标准 | 能说清每家打分逻辑 |
| 试评验证 | 5/24 | 3-5家试评能区分等级 | 不挤在同一个等级 |
| 全量完成 | 6/11 | 34家分级，易人确认 | 易人认可分类 |

## 风险标注
- 最高风险：标准定得太慢，连锁延误后续 → 先做粗糙版本再迭代
```

每个叶子任务执行时按八大方法：接任务先五问、执行中按 RACI、交付用 STAR、遇问题用风险预警、完成后 PDCA 复盘。

---

## 边界

**不要用的场景**：
- 目标本身不清晰（先用 think-framework 澄清）
- 只需要列几条 todo（不需要结构）
- 紧急场景需要立刻行动（先做再拆）

**失败模式**：
- 拆成清单而非结构——没有层级和依赖
- 颗粒度不对——太大无法执行或太小失去意义
- 没有检查节点——拆完了无法追踪

---

*结构决定范围，行为决定质量。先拆清楚，再干靠谱。*
