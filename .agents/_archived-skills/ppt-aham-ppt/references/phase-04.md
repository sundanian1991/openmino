## 执行步骤

### 步骤1：金字塔论证树构建

**金字塔结构（三层）：**

```
第一层（顶）：灵魂句（唯一，来自Phase 3）
    ↓
第二层：支撑论点（2-4个，MECE）
    ↓
第三层：证据层（每个论点对应2-3个证据，来自Phase 2信息清单）
```

**第二层论点构建原则：**

优先从Phase 3压力测试步骤中识别的反对意见反推论点：
每个反对意见对应一个需要在PPT里消除的顾虑，也就是一个论点。

常见论点组合模式（供参考，不强制）：
- 必要性 + 可行性 + 紧迫性
- 问题定性 + 解决方案 + 实施路径
- 现状分析 + 目标定义 + 行动建议

**MECE检验（互不重叠、合计完整）：**
- 互不重叠：任意两个论点是否在说同一件事？有则合并
- 合计完整：三个论点合在一起，是否足够证明灵魂句？有漏洞则补充

---

### 步骤2：Action Title（结论式标题）写法

**定义：** 每页标题是结论性陈述，而不是话题描述。读完标题，受众已知道这页的核心信息。

**写法规则：**
- 必须是完整陈述句：【主体】+【判断/结论】
- 字数：≤20个汉字
- 行数：≤2行
- 必须包含动词（是/有/导致/实现/证明等）
- 禁止：话题词、问句、「关于…」句式

**好坏示例：**

| 话题标题（差）| Action Title（好）|
|-------------|-----------------|
| 成本比较分析 | TCO测算显示：继续维护3年总成本超过新系统建设成本2.1倍 |
| 迁移方案介绍 | 三阶段迁移方案已将业务中断风险控制在4小时以内 |
| 时机分析 | Q3是唯一可行窗口：Q4业务高峰、Q1预算锁定、供应商协议Q3到期 |

---

### 步骤3：Ghost Deck页面结构

**标准页面顺序：**

```
页1   封面          PPT主标题（非Action Title，是项目名/汇报名）
页2   执行摘要      SCQA的A精华，第一个让受众看到结论的地方
页3   目录          章节结构概览
页4   章节过渡页    第一部分：章节Action Title
页5+  内容页        内容页Action Title（每页≤20字结论）
...   章节过渡页    第二部分：章节Action Title
...   内容页        内容页Action Title
...   结尾行动页    受众预期行动的明确呼吁
```

**页数分配：**
```
内容页数 = 总页数 - 4 - 章节数
（封面1 + 执行摘要1 + 目录1 + 结尾1 + 各章节过渡页各1）
```

**narrative_purpose（叙事目的）标注：**
每页必须标注以下6类之一：
- impact（震撼）：震撼性数字、关键结论
- structure（结构）：目录、章节过渡、框架说明
- explain（说明）：流程、方案、方法论解释
- evidence（证据）：数据、案例、验证
- transition（过渡）：章节衔接、节奏调剂
- action（行动）：结尾行动页、建议清单

**节奏要求：**
- 10页滑动窗口内，版式种类≥6种
- 每10页至少1页impact类
- 每8-10页至少1页transition类

---

### 步骤4：Ghost Deck四项测试

生成完整标题序列后，执行以下四项测试：

**测试1 · 只读标题测试**
将所有页面标题连起来读，不看正文。
通过标准：读完标题序列，受众已能理解「为什么要做这件事、做什么、怎么做、结果如何」。
不通过特征：读完只知道「讲了哪些话题」。

**测试2 · MECE检验**
同一章节内的各内容页标题检验：
- 互斥性：任意两页是否在说同一件事？有则合并
- 完备性：合在一起是否完整证明章节Action Title？

**测试3 · 灵魂句回溯**
对每一页标题问：「这页在支撑灵魂句吗？」
如果这页的内容无论对错，都不影响灵魂句成立 → 冗余页，删除或移入附录。

**测试4 · Action Title检验**
对每一页标题检验：
- 是结论陈述，还是话题描述？
- 读完标题，是否已知道这页的核心信息？
- 标注需要改写的标题，给出修改建议。

**测试处理规则：**
- 全部通过 → 生成完成卡，放行
- 有问题 → AI自行修改问题标题，重新测试，最多迭代3次
- 3次后仍有问题 → 标注问题，在完成卡中列出，让用户决策
