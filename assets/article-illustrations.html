01 无 Skill vs 有 Skill — 21% → 95% 前后对比
"在没有 Skills 引导的情况下，Claude 回答数据分析问题的准确率无法超过 21%。加上 Skills 后，聚合准确率稳定在 95% 以上，某些领域达到 99%。" — 这是整篇文章最核心的数据冲击点
02 概念 ⇔ 实体歧义 概念隐喻
文章定义的三个失败模式之首：用户问"活跃用户有多少？"但数据模型里有几十种不同的"活跃用户"。规范数据集（Canonical Dataset）解决的就是这个问题——让每个概念只对应一个受治理的数据集。
03 强制走语义层（默认路径优先） 概念隐喻
Agent 不是自由搜索，而是结构性引导。语义层是必经的第一路径——如果一个问题能匹配到已定义的指标，Agent 调用函数返回一个数字，这个数字跟公司所有其他 BI 工具输出的完全一致。
04 改模型必改 Skill — 同一 Repo 维护 概念隐喻
文章最重要的实操经验：Skill 文档和数据模型放在同一仓库。当数据模型发生变更时，PR 必须同时更新对应的 Skill 文档。CI 中有一个代码审查钩子，任何报表模型的变更如果没有同时修改 Skill 文件，就无法通过。这使得维护成本变得可控。
05 对抗验证与纠正收割闭环 小漫画分镜
"我们使用一个 Claude Skill 对潜在的最终答案进行激进地审查，所有底层假设都会被质疑。这带来了 6% 的准确率提升，但代价是 32% 的额外 token 消耗和 72% 的更高延迟。" — 验证闭环还包括：自动扫描频道中的纠正信号 → 生成修复性 PR → 同步到所有 Surface。
06 三个失败模式 · 三道关 地图路线
文章的框架性问题定义。三个失败模式构成了整个解决方案的出发点。每个模式对应四层方案中的一层或多层。其中"检索失败"最微妙——文章通过消融实验发现，即使给 Agent 数千条历史 SQL 的 grep 权限，准确率也只提升了不到 1 个百分点，真正有效的是结构性引导。
