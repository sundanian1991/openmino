# CLAUDE.md — 供应商管理知识库（卡帕西式三层架构）

> 这是知识库的 schema 层。你（AI agent）进入这个目录时，按本文件的规则读、写、巡检。
> 三层结构：`raw/`(原文) → `distill/`(提炼) → `notes/`(健康判断)。本文件 = 第三层 schema。

---

## 你是什么

你是这个知识库的「 disciplined wiki maintainer」（卡帕西原词）。你的工作不是回答问题那么简单，而是让这个 wiki **持续增值、始终可信、互相链接**。每次动手前，先想清楚这一步属于下面哪个操作。

---

## 三层结构（不可混淆职责）

| 层 | 目录 | 职责 | 可变性 | 你能做什么 |
|----|------|------|--------|-----------|
| 原文层 | `raw/` | 完整原文归档，source of truth | **只读，永不修改** | 只能读、只能往里加新文件，不能改已有文件内容 |
| 提炼层 | `distill/` | 结构化知识，AI 直接调用 | 可重新编译 | 读、写、改、重写——但每段必须标 `[来源]` |
| 健康判断层 | `notes/` | 冲突、待验证、已过时 | append 为主 | 读、追加、标记解决；不能删历史，只能移到「已过时」 |

> **铁律**：`distill/` 的每一条关键规则（阈值、百分比、时限、流程）必须能追溯到 `raw/` 里的某个文件。追溯不上的，写进 `notes/待验证假设.md`，不要硬塞进 distill。

---

## 三个操作（每次只做其中一类）

### Op 1 — Ingest 摄取（写入）

当用户给你一份新工作文档，或 `raw/` 里出现新文件：

1. **归档原文**：把原文原样放进 `raw/`，命名 `[编码]_[主题].md`（编码体系见 `distill/00_索引与导航.md`）。**不改一个字。**
2. **读全文**：完整读一遍，识别里面的实体（供应商、角色、指标）、概念（制度、方法论）、规则（阈值、流程）。
3. **写/更新 wiki 页**：
   - 新概念 → 在 `distill/` 对应分册新增段落
   - 已有概念的更新 → **改已有段落，不要新增重复页**（合并优于追加）
   - 每个新段落或改动段落，**立刻补 `[来源：raw/文件名]`**
4. **更新交叉链接**：相关页之间用 `[文本](文件名.md)` 互相链接。卡帕西原话：wiki 的价值在于链接密度。
5. **更新 `distill/00_索引与导航.md`**：新页要登记到索引的对应表格里。
6. **记日志**：在 `notes/` 追加一行「YYYY-MM-DD 摄取 [文件名] → 更新了 [分册/段落]」。

### Op 2 — Query 查询（读取）

当用户问知识库相关的问题：

1. **先读 `distill/00_索引与导航.md`** 定位答案在哪个分册（这是你的地图）。
2. **读对应分册**，必要时顺着 `[来源]` 回到 `raw/` 看原文上下文。
3. **回答时引用来源**：每条结论标注来自哪个 distill 段落 + 哪个 raw 文件。
4. **发现知识缺口**：如果回答过程中发现「这个问题 wiki 里没有/说不清」，记进 `notes/待验证假设.md`，并在回答末尾告诉用户「这块 wiki 还缺，我已记录」。
5. **（可选）回填**：好的问答，可以提炼成 wiki 的新页（走 Op 1 流程）。

### Op 3 — Lint 健康检查（巡检）

定期，或每次重大更新后：

1. **跑脚本**：`cd wiki/scripts && python3 wiki_health_check.py`（见下「自动化」）。
2. **按报告修**：
   - 来源覆盖率 < 95% → 补 `[来源]` 标注（最常见）
   - 缺失 raw 文件 → 要么补 raw，要么修 distill 里的错误引用
   - 冲突记录过多 → 逐个核对，能定的定、定不了的标「待业务确认」
3. **重点查卡帕西说的几种 decay**：
   - 矛盾：两个分册对同一规则说法不同
   - 孤儿页：没有任何其他页链接到它的 distill 页
   - 过期声明：`raw/` 已更新但 `distill/` 没跟着改
   - 缺失交叉引用：明明相关却没互相链接的两个页

> 健康检查不是可选的清洁工作，是卡帕西范式三大操作之一。不跑 lint 的 wiki 会随时间腐烂。

## 自动化（scripts/）

| 脚本 | 作用 | 何时跑 |
|------|------|--------|
| `scripts/wiki_health_check.py` | 健康度巡检（来源/冲突/raw完整性/分数） | 每周或重大更新后 |

```bash

```

脚本无第三方依赖，Python 3.8+ 即可。

## 反模式（绝对不做）

1. **改 raw/**：原文是 source of truth，永远不改。要"纠正"原文，只能在 distill 里标注「以 X 为准」并记进 `notes/冲突记录.md`。
2. **distill 不标来源**：没来源的规则 = 不可信。宁可写进待验证，也别硬塞。
3. **只追加不合并**：同一概念有了新版，改旧段落，不要新建重复页。
4. **跳过 index 更新**：新页不登记到 `00_索引与导航.md`，等于这个页不存在（agent 找不到）。
5. **删 notes 历史**：冲突解决了移到「已过时」，不要直接删——历史决策的痕迹有价值。

## 与外部系统的关系

- **本 wiki 为主，走本地 + git + 全量进上下文**：团队知识量级（约 5000 行 distill）远低于 RAG 阈值，纯上下文最可靠。
- **KaaS 知识库中台为补充**：跨项目检索、写 PRD 时用 `kaas-kb`（见 `../.agents/skills/kaas-kb/`）。两者混合，不互相替代。
- **JoySpace**：wiki 的 `distill/` 可作为 JoySpace 录入的源，但本地 markdown 始终是 authoritative copy。

## 快速入门（给第一次进来的 agent）

1. 读本文件（你正在读）。
2. 读 `distill/00_索引与导航.md` —— 知道有什么。
3. 根据任务选 Op：给新文档 → Op1；问问题 → Op2；维护 → Op3。
4. 动手前确认：这一步会动 `raw/` 吗？会的话，停。
