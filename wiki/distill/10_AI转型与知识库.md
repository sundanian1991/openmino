> [来源：方案介绍.md / 方案框架.md / Task3-供应商管理数字化.md]

```
L4 验证层 ─────── 人工抽样/离线评估/衰变监控/规则校准
     ↑
L3 模型层 ─────── 目标理解/任务拆解/上下文管理/多轮推理/报告生成
     ↑
L2 工具层 ─────── 数据采集/知识检索/报表生成/规则计算/档案管理
     ↑
L1 数据层 ─────── 5库知识库 + 实时数据接口 + 档案数据
```

### Anthropic经验（5条关键原则）

> [来源：方案介绍.md / 方案框架.md / Task3-供应商管理数字化.md]

| # | 原则 | 含义 |
|---|------|------|
| 1 | **结构优先于数据量** | 正确结构比海量数据有效。喂了几千条SQL，准确率提升<1% |
| 2 | **Skill是最大杠杆** | 不加Skill准确率21%，加了>95%。不是更强模型，是结构化工作流 |
| 3 | **系统会衰变** | 一个月内95%→65%。维护是严肃工程任务 |
| 4 | **95%够了** | 最后几个百分点成本+32%token+72%延迟，只换6%准确率，性价比太低 |
| 5 | **最难的是找正确数据源** | 口径问题必须先解决 |

## 五、数字人运行模式

> [来源：方案介绍.md / 方案框架.md / Task3-供应商管理数字化.md]

### 5.1 三种模式

> [来源：方案介绍.md / 方案框架.md / Task3-供应商管理数字化.md]

| 模式 | 触发方式 | 功能 |
|------|---------|------|
| **被动模式** | 用户发起提问 | 查询档案、检索规则、生成报告、分析数据 |
| **主动模式** | 定时/事件触发 | 日简报推送、异常告警、到期提醒、考核评分 |
| **群聊模式** | 接入关键群（内部+供应商） | 采集群消息、分类异常、推送提醒 |

### 5.2 人设

> [来源：方案介绍.md / 方案框架.md / Task3-供应商管理数字化.md]

- **身份**：供应商管理专家
- **能力**：检索档案、分析数据、生成报告、执行规则、巡检支持、谈判参谋
- **风格**：数据说话，判断在前，理由在后

## 六、AI能做的 vs 不能做的

> [来源：方案介绍.md / 方案框架.md / Task3-供应商管理数字化.md]

| 维度 | AI能做的 | AI不能做的 |
|------|---------|-----------|
| 数据 | 采集和汇总（排名、KPI、站点分布） | 解读数据背后的原因 |
| 文档 | 初稿生成（邮件、报告、巡检记录） | 审核修改、判断分寸和语气 |
| 规则 | 执行（5维打分、分量范围、风险信号） | 战略打分、动态调整策略 |
| 信息 | 监控和分类（群消息、异常推送） | 判断哪些是真问题、怎么解决 |
| 商务 | 准备材料（画像、数据包、谈判建议） | 现场谈判、拿捏力度、关系感知 |
| 决策 | 提供数据和建议 | 最终决策、战略方向 |

> [来源：方案介绍.md / 方案框架.md / Task3-供应商管理数字化.md]

### 现状 vs 目标

> [来源：方案介绍.md / 方案框架.md / Task3-供应商管理数字化.md]

| 类别 | 当前占比 | 目标占比 | 核心变化 |
|------|---------|---------|---------|
| 日常工作 | 47% | 30%（-17%） | 确认关键节点，异常你来判断 |
| 周期性工作 | 19% | 15%（-4%） | 巡检现场判断、谈判现场、决策拍板 |
| 战略性工作 | 22% | **38%（+16%）** | 寻源储备、站点布局、BPO联盟、深绑、向上管理 |
| 支撑性工作 | 12% | 17%（+5%） | 高质量汇报、制度体系化 |

### 释放时间投向

> [来源：方案介绍.md / 方案框架.md / Task3-供应商管理数字化.md]

| # | 投向 | 增加时间 | 为什么高价值 |
|---|------|---------|-------------|
| 1 | 供应商寻源与储备 | +5h/周 | 人才储备是生态健康基础 |
| 2 | 站点合理布局 | +4h/周 | 生态韧性、竞争优势 |
| 3 | BPO联盟机制 | +4h/周 | 从价格压降到价值共创 |
| 4 | 深度关系建设 | +4h/周 | 跳过中间层获取真实信息 |
| 5 | 向上管理与资源争取 | +2.5h/周 | 讲洞察+给方案 |
| 6 | 制度与体系化 | +1.5h/周 | 从人治到机制 |

### 角色本质变化

> [来源：方案介绍.md / 方案框架.md / Task3-供应商管理数字化.md]

**不是被替代，是升级**：从一线操作工到指挥官。

- 数字人处理80%日常运营
- 你处理20%异常 + 持续优化系统
- 最核心新工作：看规则跑得对不对，遇到新情况加新规则，让数字人越来越聪明
- 最不可替代的：**商务沟通、现场谈判、关系维护、战略判断**

## 九、卡帕西 LLM Wiki 范式与团队落地

> [来源：raw/2026-06-16_LLM-Wiki知识库方案-团队分享/LLM-Wiki知识库方案-团队分享.md]
> 补充说明：本节把知识库本身的维护方法论沉淀下来，让 AI 在整理团队资料时有一个可复用的框架。

### 9.1 为什么需要新的知识库范式

> [来源：raw/2026-06-16_LLM-Wiki知识库方案-团队分享/LLM-Wiki知识库方案-团队分享.md §一]

常见的 RAG/向量检索方案对团队量级有三个问题：

| 问题 | 影响 |
|------|------|
| 搭建成本高 | 需要向量库、embedding 服务、检索调优 |
| 分块切断语义 | 完整文档被切成 chunk，页间引用和链接关系丢失 |
| 检索有损 | 向量检索是"大概相关"，关键阈值或规则可能召回不到 |

在约 **5-10 万 token（约 150-200 页）以内**，把 wiki 直接放进大模型上下文，比 RAG 更轻、更准、更可靠。

### 9.2 三层结构

> [来源：raw/2026-06-16_LLM-Wiki知识库方案-团队分享/LLM-Wiki知识库方案-团队分享.md §二]

| 层 | 目录 | 职责 | 可变性 |
|---|------|------|--------|
| 原文层 | `raw/` | 完整原文归档，source of truth | 只读，永不修改 |
| 提炼层 | `distill/` | AI 直接调用的结构化知识，互相链接的 markdown | 可重新编译、可改 |
| Schema 层 | `CLAUDE.md` | 告诉 AI 怎么组织 wiki、按什么规则维护 | 随团队规范演进 |

**关键原则**：`distill/` 的每条规则都必须能追溯到 `raw/` 的某个文件。追溯不上的，写进 `notes/待验证假设.md`，不硬塞。

### 9.3 三个核心操作

> [来源：raw/2026-06-16_LLM-Wiki知识库方案-团队分享/LLM-Wiki知识库方案-团队分享.md §二]

| 操作 | 做什么 | 一句话 |
|------|--------|--------|
| **Ingest 摄取** | 读新文档 → 写摘要 → 更新实体页 → 更新索引 → 记日志 | 把新材料"消化"进知识库 |
| **Query 查询** | 对知识库提问 → 定位 → 综合回答带引用 → 好答案回填 | 查 wiki，而不是重读原文 |
| **Lint 健康检查** | 定期体检：找矛盾、过期、孤儿页、缺失链接 | 防止知识库随时间腐烂 |

### 9.4 三个落地技能

> [来源：raw/2026-06-16_LLM-Wiki知识库方案-团队分享/LLM-Wiki知识库方案-团队分享.md §四]

| 技能 | 对应操作 | 触发方式 | 作用 |
|------|---------|---------|------|
| `wiki:ingest` | Ingest | "把这些文档整理进知识库" | 扫描文件夹 → 转 Markdown → 归档 raw → 提炼 distill → 更新索引 |
| `wiki:query` | Query | "查一下 wiki 里有没有 X" | 先读索引 → 再读 distill → 必要时回 raw → 引用来源回答 |
| `wiki:health` | Lint | "检查一下知识库" | 跑 `wiki_health_check.py`，查来源覆盖、raw 完整、冲突、孤儿页 |

### 9.5 与 KaaS 的关系：本地是车间，KaaS 是仓库

> [来源：raw/2026-06-16_LLM-Wiki知识库方案-团队分享/LLM-Wiki知识库方案-团队分享.md §五、§六]

| 系统 | 角色 | 何时用 |
|------|------|--------|
| 本地 wiki（raw/、distill/、notes/ 与 CLAUDE.md） | 知识的"生产与编译" | 个人沉淀、提炼、迭代 |
| KaaS 知识库中台 | 知识的"共享与检索" | 跨项目查询、写 PRD、团队协作 |
| JoySpace（飞书） | 知识的"发布入口" | 把确认的 wiki 内容录入团队库 |

**落地路径**：个人整理 → 人工确认哪些适合共享 → 提炼并确认准确性 → 上传到 KaaS → 团队成员通过 `kaas-kb` 技能查询。

### 9.6 KaaS 日常调用工作流

> [来源：raw/2026-06-16_LLM-Wiki知识库方案-团队分享/LLM-Wiki知识库方案-团队分享.md §六]

写 PRD / 改代码前必走：

1. `kb_search(query="需求关键词")` → 看命中哪个项目
2. `kb_pack(topic, role, intent, query)` → 一次性拿模板 + 角色 wiki + 历史全文
3. 基于 `pack` 返回的 template 章节顺序起草
4. 有缺口 → `kb_get_artifacts(paths=[...])` 补齐

**铁律**：写 PRD 前必须先 `kb_pack` 拿全量，不能只用 `kb_search` 片段就动笔。
