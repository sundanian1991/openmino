# 02 - 规划与架构

> **核心认知**：Plan First 不可协商 — 生产代码必须先规划

---

## 🎯 为什么 Plan First 不可协商

### 问题本质

**没有规划的后果**：

```
收到任务 → 直接编码
    ↓
发现理解错误 → 返工
    ↓
发现架构冲突 → 重构
    ↓
发现边界情况 → 补丁
    ↓
代码质量下降 → 技术债务累积
```

**有规划的流程**：

```
收到任务 → Plan First
    ↓
充分理解需求
    ↓
设计解决方案
    ↓
识别风险和依赖
    ↓
用户确认规划
    ↓
编码实施（机械执行）
    ↓
质量有保障
```

### 数据支撑

> **"步骤 #1-#2 至关重要 — 没有它们，Claude 倾向于直接跳到编码"** — 社区共识

**统计发现**：
- 规划时间：10-15 分钟
- 返工时间节省：2-4 小时
- 代码质量提升：显著减少 bug

## 📋 Planning Mode 工作流

### 完整流程

```
Step 1: 初始规划
  1. 进入 Planning Mode（或使用 /plan 命令）
  2. 提供高层描述 + 现有代码指针
  3. 让 Claude 研究并提出方案
  4. 彻底审查（早期发现误解）

Step 2: 计划验证
  - 问澄清问题
  - 挑战假设
  - 请求 2-3 个替代方案（优劣对比）
  - 使用 "think", "think hard", "ultrathink" 深度分析

Step 3: 文档化
  - 退出 plan mode 并创建 dev docs
  - 或使用 /dev-docs slash 命令
  - 存储在版本控制位置

Step 4: 实施
  - 用计划启动新上下文
  - 分阶段实施（每次 1-2 部分）
  - 阶段之间审查
  - 边做边更新计划
```

### Step 1 详解：初始规划

#### 输入准备

**好的输入示例**：

```
我要添加用户设置页面。

当前状态：
- 用户模型已定义：src/models/User.ts
- 使用 Shadcn UI 组件库
- 已有 UserProfile 组件可复用

需求：
- 在 /settings 路由创建页面
- 包含：个人资料、通知偏好、隐私设置
- 使用现有的表单验证逻辑
```

**不好的输入示例**：

```
添加用户设置页面。
```

**为什么不好**：
- 没有当前状态上下文
- Claude 需要探索整个代码库
- 可能提出不兼容的方案

#### 让 Claude 研究

**提示词模板**：

```
请先研究现有代码：
1. 找到 User 模型定义
2. 找到现有的表单组件
3. 找到路由配置方式
4. 确认技术栈和依赖

然后提出 2-3 个实现方案，包括：
- 技术方案
- 需要修改的文件
- 预估工作量
- 潜在风险
```

#### 彻底审查

**审查清单**：

- [ ] Claude 理解需求正确吗？
- [ ] 技术方案可行吗？
- [ ] 有遗漏的边界情况吗？
- [ ] 有更好的替代方案吗？
- [ ] 工作量估算合理吗？

**早期发现误解的价值**：
- 在规划阶段纠错：成本 = 10分钟
- 在编码阶段纠错：成本 = 2小时+
- 在上线后纠错：成本 = 紧急修复

### Step 2 详解：计划验证

#### 问澄清问题

**示例**：

```
问题1：用户设置页面需要权限验证吗？
- A：所有用户可访问
- B：仅登录用户
- C：仅管理员

问题2：通知偏好包含哪些类型？
- 邮件通知？
- 短信通知？
- 应用内推送？
```

**为什么重要**：
- 消除歧义
- 明确边界
- 避免后期返工

#### 挑战假设

**示例**：

```
你提到使用现有的 UserProfile 组件。
但 UserProfile 是用于"展示"用户信息，
而设置页面需要"编辑"功能。
这两个组件的职责不同，强行复用可能导致：
- 组件职责不清
- 逻辑耦合
- 难以维护

建议：创建新的 UserSettings 组件，
提取可复用的逻辑到 hooks。
```

**价值**：
- 发现隐性假设
- 提前识别风险
- 推动更优方案

#### 请求替代方案

**示例**：

```
请提供 2-3 个实现方案：

方案A：单页面，Tab 切换
优点：用户体验连贯
缺点：首次加载慢

方案B：多页面，路由分离
优点：代码分离，易维护
缺点：用户体验割裂

方案C：折叠面板
优点：平衡体验和性能
缺点：移动端适配复杂

推荐：方案C，因为...
```

**价值**：
- 暴露权衡
- 明确优劣
- 支持决策

### Step 3 详解：文档化

#### Dev Docs 三文件模式

```
~/dev/active/[task-name]/
├── [task-name]-plan.md      # 被接受的计划
├── [task-name]-context.md   # 关键文件、决策
└── [task-name]-tasks.md     # 工作检查清单
```

**plan.md 模板**：

```markdown
# [任务名] - 实施计划

## 需求重述

【用户原话】
【理解后的需求】

## 技术方案

### 方案选择

选择了方案 X，理由：
- 理由1
- 理由2

### 架构设计

```
[图示或代码结构]
```

## 实施步骤

1. [ ] 步骤1：XXX — 预计 30 分钟
2. [ ] 步骤2：YYY — 预计 1 小时
3. [ ] 步骤3：ZZZ — 预计 45 分钟

## 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 风险1 | 高 | 措施1 |
| 风险2 | 中 | 措施2 |

## 验证方案

- [ ] 功能测试：XXX
- [ ] 性能测试：YYY
- [ ] 兼容性测试：ZZZ

## 决策记录

| 决策 | 选项 | 选择原因 |
|------|------|---------|
| 决策1 | A/B/C | 选择 B，因为... |
| 决策2 | X/Y | 选择 Y，因为... |
```

**context.md 模板**：

```markdown
# [任务名] - 上下文

## 相关文件

- `src/models/User.ts` — 用户数据模型
- `src/components/UserProfile.tsx` — 可参考的组件
- `src/hooks/useFormValidation.ts` — 表单验证逻辑

## 技术决策

### 为什么选择方案 B

- 原因1：XXX
- 原因2：YYY
- 放弃方案 A：因为 ZZZ

### 依赖关系

- 依赖：`package-name` 版本约束
- 外部 API：需要确认 XXX 接口可用

## 约束条件

- 必须使用 TypeScript
- 必须通过 ESLint 检查
- 兼容移动端（响应式）
```

**tasks.md 模板**：

```markdown
# [任务名] - 任务清单

## 实施步骤

- [ ] 步骤1：XXX
  - [ ] 子步骤 1.1
  - [ ] 子步骤 1.2
- [ ] 步骤2：YYY
- [ ] 步骤3：ZZZ

## 验证清单

- [ ] 功能测试：所有功能正常
- [ ] 边界测试：边界情况处理
- [ ] 性能测试：加载时间 < 2s
- [ ] 兼容性测试：移动端正常

## 完成标准

- [ ] 所有测试通过
- [ ] Code Review 通过
- [ ] 文档更新
```

### Step 4 详解：分阶段实施

#### 为什么分阶段

**一次性实施的风险**：

```
一次性写完 1000 行代码
    ↓
测试时发现基础架构错误
    ↓
需要重构 80% 的代码
    ↓
浪费大量时间
```

**分阶段实施的好处**：

```
阶段1：基础架构（200 行）
    ↓ 测试验证
阶段2：核心功能（300 行）
    ↓ 测试验证
阶段3：边缘情况（200 行）
    ↓ 测试验证
阶段4：优化和文档（100 行）
```

**每阶段后的审查**：
- 代码质量检查
- 功能验证
- 调整下一阶段计划

#### 实施中更新计划

**为什么需要更新**：

```
初始计划 → 发现新问题 → 调整方案 → 更新计划
```

**更新时机**：
- 发现技术障碍
- 需求发生变化
- 找到更优方案

**更新方式**：
- 在 plan.md 中添加"变更记录"章节
- 记录：变更原因、变更内容、影响范围
- 保持计划与实际同步

## 🔄 Explore, Plan, Code, Commit 工作流

### 四步流程

```
1. Explore：读相关文件、图片、URL（明确告知暂不编码）
2. Plan：用 subagents 验证细节，用 "think" 模式创建计划
3. Code：实施并明确验证步骤
4. Commit：更新 README/changelog，创建 PR
```

### Step 1: Explore（探索）

**目标**：充分理解现状

**提示词模板**：

```
请探索以下内容，但暂不编码：

1. 找到所有与 XXX 相关的文件
2. 分析当前架构
3. 识别潜在的依赖关系
4. 找到可复用的组件/函数

输出：
- 文件清单
- 架构分析
- 依赖关系图
- 可复用资源
```

**为什么"暂不编码"**：
- 防止 Claude 跳到编码
- 确保充分探索
- 避免基于不完整信息的决策

### Step 2: Plan（规划）

**目标**：设计解决方案

**提示词模板**：

```
基于探索结果，请设计实施方案：

1. 使用 "think" 模式深度思考
2. 考虑 2-3 个方案
3. 分析每个方案的优劣
4. 推荐一个方案并说明理由

输出：
- 方案对比
- 推荐方案
- 实施步骤
- 风险评估
```

**用 subagents 验证细节**：

```
【并行任务】验证方案
- Agent A: 验证技术可行性
- Agent B: 验证性能影响
- Agent C: 验证安全风险
```

### Step 3: Code（编码）

**目标**：机械执行规划

**关键原则**：
- 按计划执行，不偏离
- 每个步骤可验证
- 实时更新 tasks.md

**提示词模板**：

```
请按 plan.md 执行步骤1：

修改文件：src/components/UserSettings.tsx
变更：添加表单验证逻辑

验证：
- [ ] 表单验证生效
- [ ] 错误提示正确
- [ ] 边界情况处理
```

### Step 4: Commit（提交）

**目标**：规范提交变更

**Commit 格式**：

```
type(scope): subject

body（可选）

footer（可选）
```

**示例**：

```
feat(user): add settings page

- Add personal info section
- Add notification preferences
- Add privacy settings

Closes #123
```

**提交前检查**：
- [ ] 代码通过测试
- [ ] 文档已更新
- [ ] CHANGELOG 已更新
- [ ] Commit message 符合规范

## 🏗️ 简单控制循环

### 核心洞察

> **"Debuggability >>> 复杂的手调多 Agent lang-chain-graph-node 混合体"**

**Claude Code 架构**：
- 一个主线程（扁平消息列表）
- 最多一个分支（subagent 结果加入主历史）
- 无复杂多 Agent 系统
- 大多数任务用简单迭代工具调用

### 为什么简单胜过复杂

**复杂系统的问题**：

```
多 Agent 系统
    ↓
每个抽象层让调试指数级变难
    ↓
偏离通用模型改进轨迹
    ↓
LLM 是脆弱的；增加复杂性会不可预测地破坏
```

**简单系统的优势**：

```
单主线程 + Subagent
    ↓
清晰的执行路径
    ↓
易于调试
    ↓
随模型改进自然提升
```

### Claude Code 的设计哲学

> **"尽管多 Agent 系统很流行，Claude Code 只有一个主线程……我高度怀疑你的应用需要多 Agent 系统。"**

**三个设计原则**：

1. **扁平消息列表**：不需要复杂的路由
2. **最多一个分支**：subagent 结果合并回主线程
3. **简单迭代调用**：大多数任务用工具调用循环

**什么时候需要多 Agent**：
- ✅ 真正并行的独立任务
- ✅ 明确的模块边界
- ❌ 为了"看起来高级"
- ❌ 因为"别人都这么做"

## 🔍 LLM Search > RAG

### Claude Code 方法

**核心做法**：复杂的 ripgrep、jq、find 命令（无 RAG）

### 为什么不用 RAG

**RAG 引入隐藏失败模式**：

| 问题 | 说明 | 影响 |
|------|------|------|
| 相似度函数 | 用什么算法？ | 结果质量不稳定 |
| 重排器 | 怎么排序？ | 可能错过关键信息 |
| Chunk 代码 | 怎么分块？ | 上下文可能断裂 |
| 大 JSON/logs | 怎么处理？ | 格式依赖严重 |

**LLM Search 方法**：

```
看 10 行理解结构
    ↓
需要再看 10 行（就像人类）
    ↓
RL 可学习（BigLabs 已在工作）
    ↓
模型做重活（更少移动部分）
```

### 对比

| 维度 | RAG | LLM Search |
|------|-----|------------|
| **复杂度** | 高（多组件） | 低（直接调用） |
| **可调试性** | 难（黑盒组件） | 易（命令可见） |
| **准确性** | 依赖调参 | 模型能力 |
| **维护成本** | 高 | 低 |
| **随模型进化** | 需重新调参 | 自动提升 |

### 引用

> **"这是 LLM 时代的 Camera vs Lidar"**

**类比**：
- RAG = Lidar（复杂、昂贵、需融合）
- LLM Search = Camera（简单、便宜、算法驱动）
- 终极方向：算法 > 硬件/复杂度
