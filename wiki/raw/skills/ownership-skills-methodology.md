# OWNER Method 详细方法论

## Observe（观察）

停下来。列出所有已尝试的方案，找共同模式。如果你一直在做同一思路的微调（换参数、换措辞、改格式），你就是在原地打转。

**自检问题**：
- 我已尝试的方案有哪些？
- 这些方案有什么共同点？
- 我是否只是在同一思路上微调？

## Widen（拓宽）

按顺序执行这 5 个维度（跳过任何一个 = 不负责）：

### 1. 逐字读信号
错误信息、拒绝原因、空结果、用户的不满意——不是扫一眼，是逐字读。90% 的答案你直接忽略了。

### 2. 主动搜索
不要靠记忆和猜测——让工具告诉你答案：
- 代码场景 → 搜索完整报错信息
- 研究场景 → 搜索多个关键词角度
- API/工具场景 → 搜索官方文档 + Issues

### 3. 读原始材料
不是读摘要或你的记忆，是读原始来源：
- 代码场景 → 出错文件上下文 50 行
- API 场景 → 官方文档原文
- 研究场景 → 原始来源，不是二手引用

### 4. 验证前置假设
你假设成立的所有条件，哪个没有用工具验证过？全部确认：
- 代码 → 版本、路径、权限、依赖
- 数据 → 字段、格式、值域
- 逻辑 → 边界情况、异常路径

### 5. 反转假设
如果你一直假设"问题在 A"，现在假设"问题不在 A"，从对立方向重查。

维度 1-4 完成前不允许向用户提问（原则二）。

## Navigate（导航）

自检：
- 是否在重复同一思路的变体？（方向不变，只是参数不同）
- 是否只看了表面症状，没找根因？
- 是否该搜索却没搜？该读文件/文档却没读？
- 是否检查了最简单的可能性？（错别字、格式、前提条件）

## Execute（执行）

每个新方案必须满足三个条件：
- 和之前的方案**本质不同**（不是参数微调）
- 有明确的**验证标准**
- 失败时能产生**新信息**

## Reflect（复盘）

哪个方案解决了？为什么之前没想到？还剩什么未试？

**复盘四问**：
1. **回顾目标**：用户要的是什么？验收标准是什么？
2. **评估结果**：实际交付了什么？有差距吗？
3. **分析原因**：弯路的根因——信息不足、方案选错、还是执行偏差？
4. **沉淀规律**：可复用的经验是什么？好的复盘产出 SOP，不是"下次注意"

**复盘后的主动延伸**（原则三）：问题解决后不要停。检查同类问题是否存在、修复是否完整、是否有可以预防的措施。这是 Contributor 和 Owner 的区别。
