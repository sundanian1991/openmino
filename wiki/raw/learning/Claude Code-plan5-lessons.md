# /plan5 使用经验教训

> 2026-02-24 — 交互式学习网页项目复盘

---

## 项目回顾

**任务**：为 `docs/学习 Claude code 使用.md` 创建交互式学习网页

**第一次尝试**：❌ 失败
- 输出：深色主题 HTML，比例失调，审美极差
- 问题：技能没理解需求，直接输出代码

**第二次尝试**：✅ 成功
- 输出：明亮专业配色，比例正常，可用
- 关键：先明确要求，再让技能执行

---

## 核心教训

### 1. 计划文件要写清楚"不要什么"

**第一次计划的缺陷**：
```markdown
### 配色方案：
- 主色：Claude 品牌色（暖橙色/红色）← 模糊
- 辅色：中性灰、白色 ← 模糊
```

**问题**：只说了"要什么"，没说"不要什么"
- 技能默认理解为深色主题
- "暖橙色/红色"被解读成深色背景配橙色强调

**改进**：明确排除项
```markdown
### 配色方案（明确排除）：
- ❌ 不要深色主题（#1a1a2e 这种）
- ❌ 不要赛博朋克风格
- ✅ 要明亮专业：白底 + 浅灰 + 蓝色主调
```

---

### 2. 技能调用前要先对齐审美

**第一次**：直接调用 `frontend-design`
- 技能按默认理解执行（深色 = 专业开发者工具）
- 输出后发现问题，但已经浪费了

**第二次**：先调用 `web-design-expert` 对齐方向
- 明确要求："参考 Vercel、Linear、Raycast"
- 明确要求："明亮专业，不要深色强迫症"
- 然后再调用 `frontend-design` 执行

**经验**：
```
审美对齐技能 → 设计执行技能 → 代码审查技能
     ↓              ↓              ↓
 web-design    frontend-design  frontend-patterns
```

---

### 3. 验收标准要可检查

**第一次的验收标准**：
```markdown
- [ ] 七大组件内容完整呈现 ← 模糊
- [ ] 导航可快速跳转各章节 ← 模糊
- [ ] 代码块有高亮效果 ← 模糊
```

**问题**：这些都是"主观判断"，没有客观标准

**改进**：
```markdown
- [ ] 打开 HTML 文件，能看到 7 个组件卡片
- [ ] 点击卡片，有完成状态变化（绿色边框）
- [ ] 点击折叠标题，内容能展开/收起
- [ ] 代码块右上角有"复制"按钮
- [ ] 刷新页面，学习进度不丢失（LocalStorage）
- [ ] 提交测验，能看到分数和反馈
```

---

### 4. 文件写入前要先验证格式

**第一次的问题**：
- 技能输出的 HTML 包含行号前缀（`1→`、`2→`）
- 直接写入文件，导致乱码

**原因**：技能输出的是"带行号的展示格式"，不是纯 HTML

**改进**：
```bash
# 写入前先检查
head -5 file.html | od -c

# 或直接让技能输出到临时文件，检查后再移动
```

**或者**：明确要求技能
```
请输出纯净的 HTML 代码，不要行号、不要解释文字
```

---

### 5. 计划中的"执行步骤"要更细

**第一次的执行步骤**：
```markdown
1. 用 frontend-design 设计主页面
2. 用 frontend-patterns 审查组件
3. 用 web-design-expert 优化视觉
4. 本地验证
5. 用户确认
```

**问题**：
- 步骤 1 和 3 顺序反了（应该先审美对齐再执行）
- 没有"检查输出格式"的步骤
- "本地验证"太模糊

**改进**：
```markdown
1. 调用 web-design-expert：明确配色、字体、排版要求
2. 调用 frontend-design：基于设计要求生成 HTML
3. 检查输出格式：确认没有行号污染
4. 写入文件
5. 打开浏览器验证：
   - 比例是否正常
   - 配色是否明亮
   - 交互是否可用
6. 用户确认
```

## /plan5 正确使用流程

### 阶段 1：需求澄清（最重要）

**必填清单**：
- [ ] 要什么（具体描述）
- [ ] 不要什么（排除项）
- [ ] 参考案例（URL 或截图）
- [ ] 验收标准（可检查的清单）

**示例**：
```markdown
## 目标
创建一个明亮专业的学习网页，参考 Vercel 文档风格

## 排除项
- 不要深色主题
- 不要赛博朋克/游戏风格
- 不要固定侧边栏占超过 300px

## 交付物
- 单文件 HTML（内联 CSS/JS）
- 文件位置：docs/learning/claude-code-learning.html

## 验收标准
- 打开文件，Hero 区域背景是浅色渐变
- 侧边栏宽度 280px，白色背景
- 主内容区最大宽度 960px
- 代码块有深色背景和浅色文字
```

---

### 阶段 2：技能调用策略

**技能调用顺序**：
```
1. web-design-expert（审美对齐）
   → 输出：配色方案、字体系统、间距规范

2. frontend-design（设计执行）
   → 输出：完整 HTML 代码

3. frontend-patterns（代码审查）
   → 输出：改进建议

4. （可选）web-design-expert（视觉复核）
   → 确认符合设计方向
```

**每个技能的提示词要点**：

```markdown
// web-design-expert
"参考 [URL] 的设计语言，给出配色、字体、间距方案"
"明确排除：不要 XXX 风格"

// frontend-design
"基于以下设计要求生成 HTML：[粘贴设计方案]"
"输出纯净代码，不要行号、不要解释"

// frontend-patterns
"审查这个 HTML 的：语义化、可访问性、响应式"
"指出需要改进的地方"
```

---

### 阶段 3：验证与交付

**验证清单**：
```bash
# 1. 检查文件格式
head -10 file.html | od -c

# 2. 打开浏览器
open file.html

# 3. 交互测试
- 点击卡片是否标记完成
- 折叠内容是否展开
- 代码能否复制
- 测验能否提交
- 刷新后进度是否保存
```

**用户确认前准备**：
- 截图关键界面（Hero、组件卡片、测验）
- 列出已完成功能清单
- 说明已知问题（如果有）

```markdown

## 一、需求澄清

### 目标
[一句话描述要做什么]

- [URL 2]

- [ ] 检查项 1
- [ ] 检查项 2

## 二、设计方案

### 技能调用计划
| 技能 | 用途 | 提示词要点 |
|------|------|-----------|
| web-design-expert | 审美对齐 | 参考 XXX，排除 YYY |
| frontend-design | 设计执行 | 基于设计方案生成 |
| frontend-patterns | 代码审查 | 检查语义化、可访问性 |

### 输出文件
| 文件 | 位置 | 说明 |
|------|------|------|

## 三、执行步骤

1. 调用 web-design-expert

3. 检查输出格式（无行号污染）
4. 写入文件
5. 本地验证（打开浏览器测试交互）

| 风险 | 应对 |
|------|------|
| 技能输出格式错误 | 先用 od -c 检查 |
| 审美不符 | 先调用 web-design-expert 对齐 |
| 内容太多 | 增加折叠/分页机制 |
```

## 核心洞察

**plan5 不是"写了计划就能成功"**，是"想清楚再行动"。

**关键不是计划文件本身**，是逼自己回答三个问题：
1. 我到底要什么？（需求澄清）
2. 我怎么确保技能理解对了？（审美对齐）
3. 我怎么知道完成了？（验收标准）

**这三个问题想清楚，成功率至少提高 50%**。
