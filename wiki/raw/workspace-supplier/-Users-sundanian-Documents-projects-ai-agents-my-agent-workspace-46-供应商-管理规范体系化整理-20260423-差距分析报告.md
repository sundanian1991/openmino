## 二、缝隙内容（未被覆盖或覆盖不足的管理场景）

### 2.1 定价管理（缝隙等级：高）

**现状**：11个文件中无任何定价策略、定价模型、调价机制、定价委员会相关制度。

- F2指引第四节"结算与争议处理"只有结算周期和流程，没有定价依据
- F8准入模板有"定价建议"占位符，但无方法论
- F10方法论提到"降价谈判"（场景B周期性工作），但无具体规则

**缺失内容**：
- 定价模型（按通次/通时/转化率计价？阶梯定价？）
- 调价触发条件（业绩连续达标是否降价？低于成本是否调价？）
- 定价审批流程
- 与分量管理的联动（量大价优？量小溢价？）

### 2.2 分量管理细则（缝隙等级：中）

**现状**：F2和F5有赛马机制和分量规则，但缺少量化公式。

- F2指引第四节有"常规分量规则"和"突发增量分量规则"6因素，但无具体分配比例算法
- F5运营要求有"集中度管控"上限（30%/50%），但没有分量调整的计算方法
- F8/F9有集中度检查（≤40%），与F1总纲（单供应商产能≤20%、前三≤50%）**数字不一致**

**缺失内容**：
- 分量调整的量化公式（排名变化与分量变化的映射关系）
- 分量调整的审批权限（多大比例调整需谁审批）
- 临时分量调度的时效要求

### 2.3 合同管理（缝隙等级：中）

**现状**：仅F3引入SOP阶段6提到"确认合同模板"，无合同管理制度。

**缺失内容**：
- 标准合同模板清单
- 合同变更/补充协议流程
- 合同到期续约评估流程
- 违约责任与罚则（除清退外的中间态处罚）

### 2.4 供应商赋能/培训（缝隙等级：低-中）

**现状**：F2指引第三节有"最佳实践推广"，F10方法论有"HRBP式扶持"，但缺少制度化的赋能体系。

**缺失内容**：
- 供应商赋能计划（年度培训计划、技能提升项目）
- 标杆供应商经验分享机制的制度化
- 新供应商加速计划（除了6月里程碑外的具体支持措施）

### 2.5 供应商分级/差异化管控（缝隙等级：中）

**现状**：F1总纲提到"分级管控"原则，F7有ABCDE评级，但缺少各等级对应的差异化管控措施。

**缺失内容**：
- A级供应商：哪些简化管理？优先权如何落地？
- B级供应商：标准管理
- C级供应商：增加哪些管控频率？
- D/E级：除了整改和清退，有无中间态（如限制业务线、降级分量）？

### 2.6 信息安全与技术合规（缝隙等级：中）

**现状**：F1总纲第四章有4条红线，但缺少技术性安全标准。

**缺失内容**：
- 终端安全管理标准（杀毒软件版本、网络隔离要求、USB管控）
- 数据脱敏规范
- 应急响应预案（信息安全事件的处置SOP）
- 定期安全审计/渗透测试要求

## 四、清退SOP差距分析

### 4.1 基准：引入SOP的8阶段结构（F3）

F3引入SOP有完整的8阶段、每阶段含：触发条件、责任人、输入物、输出物、验收标准、审计检查点、模板、时效。标准周期T+15。

### 4.2 清退相关文件对比

| 维度 | F3引入SOP（基准） | F1总纲第七条 | F2指引第六章 | F5运营要求第六章 | F6 SLA 5.3 | F7评估方案5.2 |
|------|---|---|---|---|---|---|
| **阶段划分** | 8阶段清晰 | 3步（决策→通知→交接） | 3步骤（评估→交接→终止） | 6步骤（申请→评估→审批→通知→过渡→终止） | 6步（申请→受理→确认→过渡→终止→结算） | 仅结果应用，无流程 |
| **触发条件** | 需求发起（数据驱动） | 3类（即时/预警/整改） | 站点缩减触发线+退出条件（7条单业务线+4条全业务线） | 同上F2 | 红线响应流程 | 连续3月E级 |
| **责任人** | 每阶段明确 | 供管岗 | 供应商管理岗 | 供应商管理岗→C3→C2 | 供应商管理岗 | 未涉及 |
| **输入物** | 每阶段列明 | 量化依据 | 退出交接表 | 退出申请/清退申请 | 书面终止申请 | 评估结果 |
| **输出物** | 每阶段列明 | 清退决议（4项内容） | 数据删除确认函 | 退出通知 | 结算归档 | 清退评估 |
| **验收标准** | 每阶段有 | 交接完成+数据移交15工作日 | 保密延续2年 | 30天过渡期完成 | 30天过渡期 | 无 |
| **审计检查点** | 每阶段3-5个 | 无 | 无 | 无 | 无 | 无 |
| **时效** | 每阶段精确到天 | 30天通知+15工作日移交 | 30天过渡+3月禁入 | 30天过渡+3月禁入 | 30天过渡 | 无 |
| **模板** | 9个 | 无 | 2个（退出交接表、数据删除确认函） | 2个（同F2） | 无 | 无 |

### 4.3 差距总结

| 差距项 | 严重程度 | 说明 |
|--------|---------|------|
| **清退无阶段化SOP** | 🔴 高 | 引入有8阶段，清退最多只有6步且没有逐阶段的输入/输出/验收/审计标准 |
| **清退触发条件分散** | 🔴 高 | F1（3类）、F2（11条退出条件）、F5（同F2+站点缩减线）、F6（红线）、F7（连续E级）——5个文件各有定义，没有统一清退触发矩阵 |
| **清退后禁入期矛盾** | 🟡 中 | F1说1年，F2/F5说3个月，F7说全年末位不续约——3个不同标准 |
| **清退结算规则模糊** | 🟡 中 | F2有"最后合作业务月结算款压3个月再结"，但无争议处理细则；F6有主动终止流程但无清退结算特别说明 |
| **缺少清退模板** | 🟡 中 | 引入有9个模板，清退只有2个（退出交接表、数据删除确认函），缺少：清退通知书模板、清退决议模板、交接进度跟踪表、清退复盘报告 |
| **清退复盘缺失** | 🟢 低 | 引入有"首日质量报告"验证，清退后无复盘机制（清退原因归因、对其他供应商影响评估） |

### 4.4 建议：清退SOP应当对齐引入的结构

引入8阶段的镜像清退结构建议：
1. 清退触发确认（对应"需求发起"）
2. 清退评估与审批（对应"方案审批"）
3. 清退通知与沟通（对应"结果同步"）
4. 供应商交接准备（对应"供应商准备"）
5. 系统与数据回收（对应"内部操作"）
6. 业务交接与过渡（新增，引入无此阶段）
7. 结算与最终处理（对应"培训上线"的对端）
8. 清退复盘与归档（新增）

## 五、日常碎片化分析

### 5.1 涉及"日常运营"的5个文件

| 文件 | 日常内容占比 | 侧重面 | 与其他文件的重叠 |
|------|-------------|--------|----------------|
| F4 日常手册 | 100% | 操作层：谁做什么、什么时候做、怎么做 | 与F5高度重叠 |
| F5 运营管理要求 | ~50%（第四章） | 供应商须自建制度的要求+京东的检查标准 | 与F4重叠60%以上 |
| F2 全生命周期指引 | ~30%（第四章） | 运营要求+过程管理 | 与F5重叠70%以上（F5是F2的扩展版） |
| F10 思路梳理 | ~20% | 方法论：排名追踪、异常信号、产能诊断 | 与F1-F7不重叠，是分析层 |
| F11 可视化指南 | ~10% | 图表描述：场景分布、决策树 | 与F10高度一致，是F10的可视化版本 |

### 5.2 核心重叠区域

**F4 vs F5 重叠率估算：60-70%**

具体内容重叠：
- 出勤报送规范（时间、分类、格式）→ 完全一致
- 排班管理要求 → 完全一致
- 人员准入5项+试用期30天+2倍质检 → 完全一致
- 离职交接5步+报备时限 → 完全一致
- 异常分级+响应时限 → 完全一致
- 合规复训频率 → 完全一致
- 花名册维护 → 完全一致

**差异**：
- F4是京东侧操作规范（"我们怎么做"）
- F5是供应商侧管理要求（"供应商须建立什么制度"）+ 京东侧检查标准
- F5比F5多出：赛马分量规则、结算争议处理、变更报备、最佳实践推广

**结论**：F4和F5的分工定位理论上不同（操作层 vs 要求层），但内容高度重复。供应商看到的"要求"和京东执行的"操作"用的是同一套文字，说明没有做好视角分离。

### 5.3 日常监控模板重叠（F10 vs F11 vs 制度层）

| 监控维度 | F10方法论 | F11可视化指南 | F7评估方案 | F6 SLA |
|---------|----------|-------------|-----------|--------|
| 出勤率 | 异常信号：周度<85% | 图表11热力图 | 10分评分档 | 达标≥95%/预警<90%/红线<85% |
| 产能/业绩 | 双维度诊断（自身峰值比+头部差距） | 图表8四象限+15饼图 | 10分评分档 | 达标≥90%/预警<80%/红线<70% |
| 排名 | 5维度加权排名 | 图表14堆叠图 | ABCDE五级 | 无 |
| 合规 | 异常信号31个 | 图表11热力图 | 20分（信息安全8+持证6+报备3+录音3） | 4项合规指标 |
| 主管胜任 | 5项指标（流失/出勤/人效/响应/整改） | 图表9雷达图 | 无 | 无 |

**分析**：
- F10/F11的"日常监控"是方法论层（怎么判断、怎么诊断），与F7评估方案的评分体系（怎么打分）**不是一套逻辑**
- F7评估用线性评分（出勤率≥95%=10分），F10用状态机（排名后30%=首次预警），两者**同时生效但没有映射关系**
- F6 SLA的阈值与F7的得分档在5.1.1有对照表，但对照关系复杂，实际执行中容易混淆

### 5.4 碎片化根因

| 根因 | 表现 | 影响 |
|------|------|------|
| **缺乏单一事实来源** | 出勤规则在F1、F4、F5、F6、F7中各有描述 | 执行者不知道该以哪个为准 |
| **制度层与方法论层断裂** | F1-F7是"规定"，F10-F11是"分析方法"，两者没有引用关联 | 制度无法利用方法论的诊断能力 |
| **操作层与要求层未分离** | F4（京东怎么做）和F5（供应商怎么做）用同一套文字 | 供应商困惑哪些是自己的责任，哪些是京东的操作 |
| **预警体系多套并行** | F1三级预警、F6达标/预警/红线、F7五级评级、F10九种状态 | 同一指标可能同时触发多套体系，响应优先级不清 |
