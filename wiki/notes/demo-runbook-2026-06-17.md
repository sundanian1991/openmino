## 一、核心一句话（开场必背）

> "我们要做的不是又一个 RAG 知识库，是卡帕西（Andrej Karpathy）今年提出的 **LLM Wiki**——把团队的工作沉淀成三层结构的本地 markdown：原文不动、提炼成 wiki、配一个 schema 让 AI 学会自己维护。这套东西我们**已经跑起来了**，今天就现场演示。"

---

## 二、范式图（投屏用，2 分钟）

```
┌─────────────────────────────────────────────┐
│  CLAUDE.md   schema 层   ← 告诉 AI 怎么维护  │  ← 今天新建的，之前缺
├─────────────────────────────────────────────┤
│  raw/        原文层      ← 只读，永不改       │  44 个文件
├─────────────────────────────────────────────┤
│  distill/    提炼层      ← AI 调用，必须标来源 │  11 个分册，5353 行
├─────────────────────────────────────────────┤
│  notes/      健康判断层  ← 冲突/待验证/日志    │  8 个文件
└─────────────────────────────────────────────┘
```

三个操作（每次只做一类）：
- **Ingest 摄取**：新文档 → 写 raw → 提炼进 distill → 更新链接 → 更新 index → 记日志
- **Query 查询**：问问题 → 读 index 定位 → 读 distill（必要时回 raw）→ 引用来源回答
- **Lint 巡检**：跑脚本 → 按报告修 → 重点查矛盾/孤儿页/过期/缺链接

### Demo C：演示 Lint 循环（最有戏剧性，重点讲）

**真实故事线**（这是今天发生的，不是编的）：

1. 初始分数：**95/100**（1 个 WARN：场景速配表缺来源）
2. 修了那个 WARN + 做完 ingest → 重跑
3. 分数反而**掉到 40/100**！出现 4 个 ERROR：引用的 raw 文件不存在
4. 诊断根因：不是知识库错了，是**健康检查脚本的正则太宽**，把我引用块里"raw/原文层"的"原文"当文件名匹配了
5. 双修：① 文档改成规范引用格式 ② 脚本加防御（引用名必须有编码特征、排除中文括号）
6. 重跑 → **100/100** ✅

**话术**："看，连 AI 维护工具本身也是知识库的一部分——lint 不只是查文档，也查工具。这次就顺手把脚本的 bug 修了。这就是闭环。"

## 四、为什么是 Wiki 不是 RAG（3 分钟，应对必问）

> 一定会有人问"我们不是有知识库吗，为什么搞这套"。

**核心论点**（背下来）：
- 卡帕西原文 + 社区实践都指出：在 **~5-10 万 token（约 150-200 页）以内，把 wiki 直接塞进上下文，完胜 RAG**。
- 我们供应商管理知识库 distill 层 5353 行 ≈ 远低于阈值。
- RAG 的代价：要向量库、要分块、分块会**切断 wiki 页之间的链接语义**、检索有召回率损失。
- Wiki 的好处：100% 检索可靠、零基建、能全局推理、页之间互相链接涌现。
- **结论**：我们的量级，上 RAG 是过度设计。

**与现有系统的关系**（避免"取代"争议）：
- 本地 wiki 为主（团队知识，进上下文）
- KaaS 知识库中台为补充（跨项目检索、写 PRD 时用 `kaas-kb`）
- 两者混合，不互相替代

## 五、诚实讲缺口（2 分钟，争取信任）

> 别人会觉得你太完美反而怀疑。主动说短板。

1. **没有"一键批量 wiki 化整个文件夹"的技能**。目前 ingest 是半自动——AI 读原文 + 按规范写 distill，但分类、编码、提炼要人来定。这是下一步要补的（可能做成一个 `wiki-ingest` 技能）。
2. **lint 脚本还比较粗**：只查来源覆盖、raw 完整性、冲突数。卡帕西说的"孤儿页、缺失交叉引用、矛盾检测"还没完全实现。
3. **14 个未解决冲突**还在 notes 里，需要业务侧逐个确认。

## 七、Q&A 预案

| 可能的问题 | 回答要点 |
|-----------|---------|
| "这和 Obsidian/飞书文档有啥区别？" | 区别在 schema：CLAUDE.md 让 AI 成为" disciplined maintainer"，会自己 ingest/lint，不是被动存文件 |
| "为什么不直接让 AI 读原文？" | 读原文每次消耗全量 token；wiki 是"编译产物"，AI 读编译后的省 token 还能跨文档推理 |
| "怎么保证 wiki 不和原文脱节？" | lint 循环 + 来源标注 100% + 冲突记录机制。今天演示的脚本就是干这个的 |
| "团队不会写 markdown 怎么办？" | 不用他们写。AI 负责所有 bookkeeping，人只管给原文 + 审 distill |
| "能撑多大量？" | 社区实践有 3000 页哲学文献的案例。我们目前 44 个原文完全不是问题 |
