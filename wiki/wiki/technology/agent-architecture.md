## 概述

Agent 架构是 Mino 的核心执行引擎，围绕"主代理做裁判、子代理做运动员"的铁律设计。从 2026-02-12 上线至今，经历了从 5Agent 团队到纪年 8 合 1 的演进，核心思路是不自己做、派专家去、然后验证。

## 主代理与子代理分工

**主代理职责**：意图识别 → 任务拆分 → 委派子代理 → 聚合结果 → 交付。主代理不参与具体执行（读多文件、深度分析、写代码、搜索研究）。

**轻量直执行例外**（保证丝滑，不走 Agent 开销）：

| 操作 | 条件 |
|------|------|
| 读文件 | 仅 1 个已知路径 |
| 搜索 | 简单 grep/glob，结果 ≤5 条，不需深度分析 |
| 修改 | 单文件单行 |
| 其他 | 严格顺序依赖或年老师说"直接做" |

**上下文隔离三原则**：

| 原则 | 做法 |
|------|------|
| **探索归子代理，决策归主代理** | 多文件/多路径/试错 → 派子代理，只带回结论 |
| **主上下文只装决策信息** | 子代理返回结果，主代理只保留与决策相关的部分 |
| **宁派不读** | ≥2 个文件的读取 → 宁派子代理，不在主上下文逐个读 |

## 子代理体系演进

### 历史演进

| 阶段 | 时间 | 架构 | 核心变化 |
|------|------|------|----------|
| 5Agent 团队 | 2026-03-18 | 秘书官/笔杆子/情报官/数据官/合规官 | 参考傅盛体系建立分工 |
| 宙斯家族 8 角色 | 2026-03~04 | 文书/掌柜/判官/账房/师爷/管家/信使/情报官 | 8 角色独立分工 |
| 纪年 8 合 1 | 2026-04-19 | 整合 8 角色为单一 sub-agent | 简化路由，降低维护成本 |

### 纪年 8 合 1

2026-04-19 将原有的 8 个独立 Agent 角色整合为纪年 sub-agent。整合目标是减少路由复杂度，但实际体验暴露了问题：

- **技术问题**：每次调用出现"模型找不到"，自动路由未生效
- **体验问题**：大而全但不精，年老师仍习惯直接调用单一 skill

### 子代理数量约束

同一批次并行子代理 ≤5 个。超过则分批，主代理控制节奏。

## 技能类型与选用原则

| 类型 | 用法 |
|------|------|
| **Explore** | 搜文件、搜关键词、探索目录结构 |
| **Plan** | 设计方案、架构决策、多方案对比 |
| **general-purpose** | 深度研究、复杂多步骤任务 |
| **专家子代理** | 领域任务（判官→供应商、账房→数据、文书→公文） |

**选用原则**：专家 > general-purpose > Explore。能 Explore 解决的不用更重的。

## 上下文管理

> 上下文不可再生。满了就压缩，压缩就有损。

**对话总结规则**：只记异常/线索和当前卡点，禁止流水账。压缩时最容易丢失的就是异常信息。

**回退策略**：≥2 步错误 → 声明回退，用精准指令替代逐步纠正。

**触发条件**：任何可能大量占用上下文的操作（长文件读取、大量搜索结果、批量数据处理、多步骤分析）都必须用子代理。2026-04-22 年老师强调："每一次遇到有可能会撑爆上下文的时候，都应该多用子代理"。
