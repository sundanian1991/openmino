## 概述

2026 年 2 月，一个名为 OpenClaw 的开源 AI Agent 框架在 GitHub 上获得了 355K stars。这个看似技术社区的事件，直接催生了 AI 定价模式从 Coding Plan 向 Token Plan 的转变。OpenClaw 让 Agent 可以自动循环调用模型，Token 消耗增长 6-10 倍，最终导致"不限量"订阅模式不可持续。本文分析 OpenClaw 现象背后的商业逻辑，以及 Agent 框架如何从根本上改变 AI 定价的底层假设。

## OpenClaw 是什么

OpenClaw 是一个开源的 AI Agent 框架，允许用户创建可以自动执行任务的 AI 代理。与传统的手动调用 API 不同，Agent 框架让 AI 能够：

- 自动循环调用模型（一个任务 = 几十次模型调用）
- 读取文件、使用工具、重发上下文
- 持续运行直到任务完成

GitHub 355K stars 意味着什么？这意味着大量开发者在使用这个框架。大量开发者使用意味着什么？意味着 Token 消耗从"人工控制"变成了"自动控制"。

## 从手动到自动的消耗变化

### 手动调用模式

开发者手动写代码调用 API：
- 一次提问 → 一次调用
- 一次代码补全 → 一次调用
- 开发者可以控制调用频率和 Token 消耗

在这种模式下，包月订阅（Coding Plan）是可行的——因为用户的使用频率和 Token 消耗是有限的、可控的。

### Agent 自动调用模式

Agent 自动循环调用：
- 一个 Agent 任务 → 几十次模型调用
- MiniMax M2 的 token 用量：2 月 vs 12 月增长 6 倍
- Coding Plan 消耗增长 10 倍+

在这种模式下，包月订阅变得不可持续——因为用户的使用频率和 Token 消耗是无限的、不可控的。

## 实际数据印证

### 开发者案例

一个开发者 8 个月 Claude Code 消耗 100 亿 Token：
- 按 API 价格计算超过 15000 美元
- 但 Max plan 只花了 800 美元
- 节省了 93%

这意味着什么？订阅模式下，重度用户的实际消耗是订阅费的 18.75 倍（$15000 / $800）。如果所有用户都是重度用户，厂商的模型会瞬间崩溃。

### 厂商的反应

**阿里云算力告急**：Lite 档 2026-04-13 直接停售。这是 Agent 消耗爆发的直接后果。

**Anthropic 收缩**：2026-04 从标准订阅中移除 OpenClaw 访问。这是最标志性的事件——Anthropic 是 OpenClaw 生态的核心模型提供商，但因为成本撑不住，不得不限制访问。

**MiniMax 更名**：2026-03 将 Coding Plan 更名为 Token Plan。这不只是改名，是从"卖编程工具"转向"卖通用算力"，因为 Agent 消耗的已经不只是编程 Token，而是文件读取、工具调用、上下文重发。

## Token 消耗的结构变化

Agent 场景下，Token 消耗的构成与手动调用完全不同：

| 用途 | 占比 | 说明 |
|------|------|------|
| 文件读取 | 35-45% | Agent 需要读取大量文件作为上下文 |
| 工具输出 | 15-25% | Agent 调用工具的结果需要返回给模型 |
| 上下文重发 | 15-20% | 每轮对话需要重发之前的上下文 |
| 推理 | 10-15% | 模型实际推理生成的 Token |
| 代码生成 | 5-15% | 真正用于代码生成的 Token |

关键数据：只有 5-15% 的 Token 真正用于代码生成。85-95% 是 Agent 的上下文开销。

这意味着什么？意味着 Agent 的成本效率极低。你付的钱大部分不是在"用 AI"，而是在"维持 AI 的上下文"。

## 为什么"不限量"模式不可持续

Coding Plan 的"不限量"承诺基于一个假设：用户的使用量是可预测的、有限的。

Agent 框架打破这个假设的三个方式：

**频率爆炸**：从手动一次一次调用，变成 Agent 自动循环调用。频率增长 10 倍+。

**上下文膨胀**：Agent 需要大量上下文（文件读取 35-45%），每次调用都携带大量上下文 Token。

**不可控性**：开发者可以控制自己什么时候调用 API，但 Agent 一旦启动，调用频率由 Agent 的任务复杂度决定，不是由开发者控制。

在 Agent 时代，"不限量"订阅等于"无限风险"。对厂商来说，这是不可接受的商业模式。

## 对定价逻辑的改变

OpenClaw 现象从根本上改变了 AI 定价的底层假设：

**从"用户行为可预测"到"用户行为不可预测"**：Agent 框架让 Token 消耗从线性增长变成指数增长。

**从"编程工具"到"算力资源"**：85-95% 的 Token 是上下文开销，不是编程内容。定价不能再以"编程场景"为基础，必须以"算力消耗"为基础。

**从"包月制"到"按量制"**：不限量模式在 Agent 时代不可持续，所有厂商最终都会转向精细计量。

## 对未来趋势的预判

**趋势一**：Agent 框架会继续发展，Token 消耗会继续增长。推理需求 2026 年已达训练的 4-5 倍，IDC 预测 2028 年推理占比 73%。

**趋势二**：厂商会推出 Agent 专属定价。可能是按任务计费、按 Agent 运行时间计费，或者按上下文大小计费。

**趋势三**：开源 Agent 框架和闭源模型的博弈会继续。开源框架推动使用量增长，闭源模型需要控制成本。两者的张力决定了定价的未来走向。

## 对供应商管理的迁移

Agent 框架对定价的影响，在供应商管理中有一个对应的逻辑：**自动化对人力成本的影响**。

就像 Agent 框架让 Token 消耗从可控变成不可控，自动化流程（如 AI 辅助外呼、智能质检）让供应商的人力成本结构发生根本变化。

**应用方式**：
- 了解供应商的自动化程度，判断其真实成本结构
- 自动化程度高的供应商，人力成本下降，但技术投入增加
- 分量分配中要考虑"自动化红利"——供应商效率提升后，应该体现在分量增长还是成本下降上
