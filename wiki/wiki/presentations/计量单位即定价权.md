# 计量单位即定价权：AI定价案例的核心商业规律

## 概述

"计量单位即定价权"是 AI 定价商业案例中最核心的五条洞察之首。这条规律指出：谁能定义计量单位（Token、Prompt、API 请求次数、Credit），谁就掌握了定价的主动权。这不是 AI 行业的独特发现，而是所有"算力即服务"商业模式的底层规律。本文深入分析这一规律在 AI 行业的具体表现、背后的商业逻辑，以及它对供应商管理和其他商业场景的迁移价值。

## 五种计量单位并存的事实

2026 年 4 月，国内七家 AI 平台的定价使用了五种完全不同的计量单位：

**API 请求次数**：阿里百炼、火山方舟、腾讯云、百度千帆采用。1 次用户提问约等于 5-30 次 API 请求。数字看起来很大（百炼 Lite 给 18000 次/月），但实际消耗极不透明。

**Prompt 次数**：智谱 GLM、MiniMax（早期）采用。1 Prompt 约等于 15-20 次模型调用，约等于 1200-1600 API 请求。比 API 次数直观一些，但仍然不够透明。

**Token 计量**：Kimi（2026 年 1 月 28 日切换后）、MiniMax Token Plan 采用。直接按输入输出 Token 计费，仅统计未命中缓存的 Token。这是最透明的计量方式，但用户难以预估实际消耗。

**Credit 计量**：Cursor、小米 MiMo、阿里百炼 Token Plan 采用。统一 Credit 池，不同模型不同倍率。小米 MiMo 的倍率：Omni 模型 1x、Pro 256k 模型 2x、Pro 1M 模型 4x。抽象了一层但可预测。

**时间窗口配额**：智谱 GLM 采用 5 小时窗口、Claude Code 采用 5 小时+7 天双层。限制的是使用节奏而非总量。

**关键数据**：同样 40-50 元/月，阿里百炼给 18000 次 API 请求，MiniMax 给 40 prompts/5h（约等于 1200 API 请求/5h）。看起来百炼多了 15 倍，但因为计量单位完全不同，直接比数字毫无意义。这就是为什么 90% 的开发者容易踩坑。

## 计量单位的演变方向

从 2024 Q4 到 2026-04，计量单位的演变轨迹清晰可辨：

```
API 请求次数 → Prompt 次数 → Token/Credit → 统一 Credits
```

方向是越来越透明、越来越可预测。但这不意味着早期的计量单位是"错误"的——每种计量单位都服务于特定的商业阶段：

**API 请求次数**适合早期获客阶段。数字大、消耗不透明，用户对成本感知弱。百炼给 18000 次，用户觉得"好多"，但实际 18000 次 API 请求只约等于 600-3600 次提问。

**Prompt 次数**是中间态。比 API 次数直观（一个 Prompt 就是一个问题），但实际消耗仍然不透明（一个 Prompt 等于 1200-1600 次 API 请求）。

**Token 计量**是终态。最透明，直接告诉你用了多少"算力"。但用户需要学习如何预估 Token 消耗，这本身就是一个门槛。

**Credit 计量**是平衡态。抽象了一层（用户不用理解 Token），但通过倍率体系保持了可预测性。

## 谁在争夺计量单位的定义权

**MiniMax 的动作**：2026 年 3 月，官方将"Coding Plan"更名为"Token Plan"。这不是简单的改名，而是从"编程工具"升级为"通用算力"的战略转向。同时，从 Prompt 计量切换到 Token 计量——从模糊打包到透明计费。

**Kimi 的动作**：2026 年 1 月 28 日，全面从 Prompt 计量切换为 Token 计量。与 MiniMax 的逻辑一致——争夺"最透明"的心智定位。

**云平台的动作**：阿里百炼、火山方舟、腾讯云坚持用 API 请求次数或 Credit。因为云平台的定位是"超市"（聚合多模型），用统一的抽象单位（API 请求或 Credit）比用 Token 更灵活。

**DeepSeek 的动作**：不走订阅路线，只卖 API，直接用 Token 计费。定价极低（V4-Flash 输入 0.2 元/百万 Token），用极致低价和透明度重塑定价权。

## 计量单位背后的商业博弈

### 模糊计量利早期获客

早期的计量单位为什么模糊？因为模糊对用户有心理优势。18000 次 API 请求听起来比 3500 万 Token 直观（虽然实际消耗更多）。40 prompts/5h 听起来比 1200 API 请求/5h 少（虽然实际等效）。

模糊计量让用户难以横向比较，降低了对价格的敏感度。这是早期获客阶段的有意设计。

### 透明计量利长期信任

转向 Token 计量的厂商，核心诉求是建立长期信任。当用户能清楚看到"我用了多少 Token，花了多少钱"，信任感会增强。Cursor 2025 年 6 月突然从 request 改为 credit-based 引发退款风波，就是因为用户觉得"计费规则变了，我不信任你了"。

长期来看，透明计量是建立用户信任的必由之路。但转型需要过渡期，不能突然切换。

### 计量单位即信息壁垒

五种计量单位并存，最大的后果是信息壁垒。用户无法横向比较各平台的真实价格，这就给了厂商定价的自由度。谁率先统一计量单位，谁就可能成为行业标准的制定者。

MiniMax 将 Coding Plan 更名为 Token Plan，本质上是在争夺这个标准制定权。如果所有平台都叫"Token Plan"，用户就会直接比较 Token 价格——这对 MiniMax 有利，因为它的定价最激进。

## 对供应商管理的迁移价值

计量单位即定价权的规律，在供应商管理中有直接的应用：

### 分量分配的"计量单位"

在供应商分量分配中，"分量"就是计量单位。谁来定义什么是"分量"？是按拨打量、接通量、转化量，还是综合评分？

如果你（甲方）定义计量单位，你就掌握了分配的主动权。如果你让供应商自己定义，你就被他们的标准牵着走。

### 评分体系的"计量单位"

供应商评分体系也是一种计量单位。评分维度（合规、质量、效率、配合度）的权重分配，直接决定了供应商的排名和分量。

谁设计评分体系，谁就掌握了评价权。这比"给多少分量"更根本——因为评价决定了分量。

### 合作条件的"计量单位"

合作条件（响应时间、人员配置、培训投入）的量化方式，也是一种计量单位。用"30 分钟回复"还是"1 小时内回复"，用"10 人团队"还是"5 人团队"，这些量化标准的选择，直接影响合作的成本和预期。

## 对其他商业场景的迁移

**SaaS 产品**：按用户数收费 vs 按功能模块收费 vs 按使用量收费。计量单位决定了客户的成本感知和你的收入模型。

**云计算**：按 vCPU 收费 vs 按内存收费 vs 按综合实例收费。AWS 的实例类型定义就是一种计量单位。

**电信行业**：按通话分钟收费 vs 按流量 GB 收费 vs 按综合套餐收费。从话费套餐到流量套餐的演变，本质上就是计量单位的升级。

**共享经济**：按次收费 vs 按时间收费 vs 按里程收费。网约车的计费单位（时间+里程）是对传统出租车（里程）的升级。

## 核心启示

计量单位即定价权的核心启示是：在设计和提供任何"服务"时，不要只关注"多少钱"，更要关注"按什么计费"。

计量单位的选择，决定了：
1. 用户的成本感知（数字大还是小，直观还是不直观）
2. 你的利润空间（模糊还是透明，有利于你还是有利于用户）
3. 竞争的维度（可比还是不可比，同质化还是差异化）
4. 信任的建立（透明还是不透明，短期利益还是长期关系）

这四个维度，比单纯的"价格高低"更重要。
