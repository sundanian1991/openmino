# Agent时代成本结构剧变：85-95%的Token是上下文开销

> Sources: AI产品定价模式演变研究 - Agent成本分析 (2026-04-24)
> Raw: [头脑风暴源文件-Agent成本](../../raw/presentations/projects-AI产品定价模式演变-从CodingPlan到TokenPlan_ppt169_20260425-sources-00-brainstorm.md); 

## 概述

Agent 时代的成本结构剧变是 AI 定价演变中最被忽视但最重要的发现。DEV Community 的研究显示，70% 的 coding agent token 是浪费。一个开发者 8 个月 Claude Code 消耗 100 亿 Token，其中只有 5-15% 真正用于代码生成，85-95% 是上下文开销。这意味着什么？意味着 Agent 用户付的钱大部分不是在"用 AI 写代码"，而是在"维持 AI 的上下文"。本文深入分析这一成本结构的成因、影响和未来趋势。

## Token 消耗的五部分构成

### 文件读取（35-45%）

Agent 在执行任务时，需要读取大量文件作为上下文。比如让 Agent "修复这个 bug"，它需要：
1. 读取相关文件
2. 读取依赖文件
3. 读取测试文件
4. 读取配置文件

每一步都把文件内容作为输入 Token 发送给模型。这些 Token 不是"生成"的内容，是"读取"的内容。但它们同样计费。

### 工具输出（15-25%）

Agent 调用工具（如文件搜索、终端命令、API 调用）后，工具的输出需要返回给模型作为上下文。这些输出通常很长——一个文件搜索结果可能包含几百行文本。

### 上下文重发（15-20%）

这是最容易被忽视的部分。在多轮对话中，每一轮都需要重发之前的对话历史。因为 LLM 是无状态的——它不记得之前说过什么，需要每次都告诉它。

这意味着：第 10 轮对话的 Token 消耗 ≈ 前 9 轮的对话历史 + 第 10 轮的新内容。随着对话进行，上下文越来越长，消耗越来越大。

### 推理（10-15%）

模型实际进行推理、生成新内容的 Token。这才是真正"用 AI"的部分。

### 代码生成（5-15%）

真正用于代码生成的 Token。这是 Agent 的核心价值，但在总消耗中占比最小。

## 真实案例：100 亿 Token 的消耗

一个开发者 8 个月 Claude Code 消耗 100 亿 Token：
- 按 API 价格计算超过 15000 美元
- 但 Max plan 只花了 800 美元
- 节省了 93%

这个数字的含义是：如果没有订阅保护，这个开发者的 Token 费用是订阅费的 18.75 倍。

**Cursor 退款风波**：2025 年 6 月 Cursor 从 request 改为 credit-based 后，重度用户一周超支 350 美元。这是因为 credit-based 让实际消耗暴露了——用户才发现"原来我花了这么多"。

## 为什么上下文开销这么大

### 根本原因：LLM 的无状态性

LLM 不记住之前的对话。每一轮都需要重发完整的历史。这是架构决定的，不是厂商可以优化的。

### 直接原因：Agent 的循环调用

Agent 不是"问一次答一次"，而是"问→做→检查→再做→再检查"的循环。每一轮都携带大量上下文。

一个 Agent 任务 = 几十次模型调用 = 几十次上下文重发。

### 放大原因：文件读取的冗余

Agent 读取文件时，经常读取相同的文件。比如第一轮读了 A.py，第三轮又读了 A.py。这些重复读取消耗了额外的 Token。

## 对定价的影响

### 订阅费是地板不是天花板

在 Agent 时代，订阅费只是起步价。Token 超额、premium 模型加价、agent 循环消耗 → 实际账单是标价的 2-5 倍。

### "不限量"模式不可持续

Coding Plan 的"不限量"承诺在 Agent 时代是不可持续的。因为 Agent 的消耗是不可预测的、指数增长的。所有厂商最终都会转向精细计量——谁先转谁主动，谁后转谁被动（Cursor 的教训）。

### 厂商的成本压力

Agent 框架（如 OpenClaw）让 Token 消耗增长 6-10 倍。Anthropic 从标准订阅中移除 OpenClaw 访问，就是因为"成本撑不住了"。

### 用户对成本的感知

当用户发现"我付 49 元月费，但实际消耗相当于 1000 元"，他们会觉得不公平。这种不公平感会影响用户留存和厂商声誉。

## 未来的优化方向

### 上下文压缩

减少重发的上下文大小。比如只重发关键信息而不是完整历史。

### 缓存优化

提高 KV Cache 命中率，减少重复计算的 Token 消耗。DeepSeek V4 的 KV Cache 占用降至 10%，是这方面的突破。

### Agent 效率提升

减少不必要的工具调用和文件读取。让 Agent 更智能地选择需要的上下文。

### 定价模式创新

可能是按任务计费（不是按 Token）、按 Agent 运行时间计费、或者按上下文大小计费。

## 对供应商管理的迁移

### 隐性成本识别

Agent 成本结构的启示是：关注隐性成本，不只是显性成本。

在供应商管理中：
- 显性成本：报价、人力费用
- 隐性成本：沟通成本、培训成本、返工成本、管理成本

只看显性成本，你可能选择了"便宜"但"隐性成本高"的供应商。

### 效率优化的价值

就像 Agent 需要优化上下文开销来降低成本，供应商也需要优化隐性成本来提高效率。

**沟通成本优化**：建立标准化的沟通流程和模板，减少重复沟通。

**培训成本优化**：建立标准化的培训体系，减少重复培训。

**管理成本优化**：建立自动化的数据追踪和报告机制，减少人工管理。

### 成本透明度的重要性

Agent 成本结构不透明的后果是用户信任损失。供应商管理中，成本不透明同样会损害信任。

**正确做法**：主动透明化成本结构。让供应商知道分量分配的逻辑，让用户知道计费的逻辑。透明建立信任。
