## Overview

《Storytelling with Data》是数据沟通领域的经典著作，包含30个独立技能。孙大年在准备晋升材料期间，系统学习了全书内容并将其应用到供应商管理的数据呈现中。本文章将书中的30个技能整合为5个核心模块，并结合供应商管理的实际场景，形成一套可操作的数据讲故事实战指南。

## 一、数据沟通的完整流程

### 七步框架

```
理解受众 → 明确目标 → 选图表 → 去噪音 → 引注意力 → 讲故事 → 迭代打磨
   ↓           ↓         ↓        ↓        ↓         ↓        ↓
 context    big idea   选图    declutter   preattentive  story   iterate
```

这本书的核心观点是：**数据沟通不是"画完图就走"，而是一个完整的流程**。每一步都不可跳过。

### 七步详解

| 步骤 | 关键问题 | 常见错误 |
|------|----------|----------|
| 理解受众 | 谁在看？他们关心什么？ | "所有利益相关者"这种模糊定义 |
| 明确目标 | 你希望他们做什么？ | 被动总结，没有行动导向 |
| 选图表 | 什么图最适合这个信息？ | 用饼图展示占比 |
| 去噪音 | 哪些元素没有信息价值？ | 装饰性元素过多 |
| 引注意力 | 第一眼应该看到什么？ | 所有元素一样突出 |
| 讲故事 | 如何组织叙事结构？ | 按时间线罗列 |
| 迭代打磨 | 怎么持续改进？ | "画完就走" |

## 二、模块一：沟通策略层

### 定义受众——拒绝模糊

**核心规则**：找出真正的决策者，评估信任度，根据画像决定语气和深度。

| 受众类型 | 语气 | 深度 | 侧重 |
|----------|------|------|------|
| 信任你的上级 | 直接、简练 | 结论+关键数据 | 结果和下一步 |
| 需要证明的中立者 | 客观、数据密集 | 完整论证过程 | 证据和逻辑 |
| 有质疑的反对者 | 谨慎、包含反对证据 | 全面+自我批判 | 可信度和风险 |

**供应商管理中的应用**：

| 场景 | 受众 | 沟通方式 |
|------|------|----------|
| 晋升述职 | 评委（不熟悉你的日常） | 案例驱动，用数据证明能力 |
| 向上汇报 | 军哥（穿透式追问） | 结论先行，预判追问 |
| 供应商会议 | 供应商（关注自身利益） | 排名+分量+改进方向 |

### 明确目标——用强动词

| 弱目标 | 强目标 |
|--------|--------|
| "让评委了解我的工作" | "让评委批准我晋升P7" |
| "展示供应商数据" | "让军哥批准引入新供应商" |
| "说明巡检结果" | "让供应商在下周五前提交整改计划" |

### Big Idea——一句话概括

**结构**：`洞察 + 建议`

| 弱Big Idea | 强Big Idea |
|------------|------------|
| "供应商管理取得了很大成绩" | "通过自建团队对标，年省1200万，建议将此模式复制到其他外包场景" |
| "合规情况良好" | "巡检发现问题6-7个/家，建议增加巡检频次到每月一次" |

## 三、模块二：图表选择与形态

### 功能决定形式

先定义受众该做什么/理解什么，再选图表。

| 功能目标 | 图表形式 | 供应商管理案例 |
|----------|----------|---------------|
| 时间趋势 | 折线图 | 月度GMV变化趋势 |
| 分类对比 | 条形图 | 33家供应商产能排名 |
| 结构与组成 | 堆积图 | 各产线GMV占比 |
| 关键数字 | 大字 | "年省1200万"——不要用图 |
| 相关性 | 散点图 | 供应商配合度vs业绩 |

### 禁止清单

| 禁止 | 替代方案 |
|------|----------|
| 饼图/环形图 | 水平条形图 |
| 双Y轴 | 上下分图或直接标注 |
| 3D图表 | 永远不用 |
| 截断Y轴的条形图 | 零基线 |

### 供应商管理常用图表模板

**供应商排名——水平条形图**：

| 要点 | 做法 |
|------|------|
| 排序 | 按产能降序排列 |
| 高亮 | Top3用强调色，其余灰色 |
| 标注 | 直接在条形末端标注GMV数值 |

**月度趋势——折线图**：

| 要点 | 做法 |
|------|------|
| 核心线条 | 总GMV用强调色实线 |
| 辅助线条 | 各产线用灰色细线 |
| 轴标签 | 不需要每个月都标，每季标一次 |

**正负变化——正负堆叠条形图**：

| 要点 | 做法 |
|------|------|
| 零基线 | 水平零基线=当前状态 |
| 正贡献 | 堆在上方（产能增长） |
| 负贡献 | 堆在下方（产能下降） |
| 缺口 | 用空心条形标注（最关键信息） |

## 四、模块三：视觉清晰度和认知原则

### 前注意属性——≤10%高亮法则

人脑在200-250毫秒内就能感知以下属性：

| 属性 | 效果 | 供应商管理应用 |
|------|------|---------------|
| 颜色 | 最强 | 强调色只用于核心数据 |
| 大小 | 强 | 关键数字放大 |
| 加粗 | 强 | 标题和结论句加粗 |
| 位置 | 中 | 关键信息放左上或顶部 |

**铁律**：≤10%的元素要突出，其余灰化。永远不要同时高亮多个不相关的点。

### 去噪音清单

| 噪音元素 | 问题 | 修正 |
|----------|------|------|
| 图表背景填充 | 分散注意力 | 去掉 |
| 不必要的网格线 | 视觉干扰 | 去掉或淡化 |
| 3D效果 | 扭曲感知 | 去掉 |
| 多余的颜色 | 没有焦点 | 保留1个强调色 |
| 冗余标签 | 信息重复 | 合并或删除 |
| 边框 | 增加认知负荷 | 去掉，用留白替代 |

### 10秒测试

非专业人士能否在10秒内理解你的图表要点？如果不行，说明：
- 标题不够洞察导向
- 噪音太多
- 没有视觉焦点

## 五、模块四：三幕叙事结构

### 三幕结构在供应商管理中的应用

| 幕 | 内容 | 晋升述职案例 |
|----|------|-------------|
| **第一幕（铺垫）** | 设定场景、指出问题 | "老商躺平、新商活不了" |
| **第二幕（冲突）** | 用数据和问题展开 | "如何激活竞争？没有议价权怎么办？" |
| **第三幕（决议）** | 明确行动和结果 | "建立选汰机制+自建对标，年省1200万" |

### 每个幕的关键动作

**第一幕——回答"我为什么要关心"**：

| 动作 | 示例 |
|------|------|
| 设定场景 | "金科电销渠道全部外包，33家供应商，3239人" |
| 指出失衡 | "但老商进入舒适区，不再主动优化" |
| 预览方案 | "我做了两件事：激活竞争+建立标杆" |

**第二幕——用证据展开**：

| 动作 | 示例 |
|------|------|
| 数据对比 | "自建110%增速 vs BPO 45%" |
| 案例展示 | "引入8家新商，75%进TOP3" |
| 不行动的后果 | "如果不做，老商持续躺平，成本无法优化" |

**第三幕——Bing-Bang-Bongo技术**：

> "先告诉他们会说什么 → 再说 → 最后告诉他们你说了什么"

| 动作 | 示例 |
|------|------|
| 明确行动 | "建议将此模式复制到其他外包场景" |
| 回顾紧迫性 | "没有产能掌控力，就没有议价权" |
| 具体到行动 | "自建团队是第一步，下一步是数据对标" |

## 六、模块五：迭代与反馈

### 验光师迭代法

每次只改一个元素，左右对比选优：

| 轮次 | 改什么 | 对比 |
|------|--------|------|
| 1 | 标题（从描述性→洞察性） | "供应商绩效" vs "新商75%进TOP3" |
| 2 | 图表颜色（全彩→≤10%强调色） | 五彩斑斓 vs 灰色底+红色强调 |
| 3 | 图表类型（饼图→条形图） | 饼图 vs 水平条形图 |
| 4 | 数据标签（轴上→直接在元素上） | 看轴读数 vs 直接标注 |

### 外部反馈的三个标准问题

找代表受众的同事，不解释你的意图，让他们自己看：

| 问题 | 目的 |
|------|------|
| "你首先注意到什么？" | 验证视觉焦点是否正确 |
| "你得出了什么结论？" | 验证信息传达是否准确 |
| "怎样能让重点更清晰？" | 找到改进方向 |

**铁律**：只听不辩。如果对方关注对了、结论对了，说明有效。如果偏了或困惑，找到问题元素。

## 七、数据伦理：必须包含反对证据

### 原则

数据论证必须包含反对证据和上下文，否则损害可信度。

### 供应商管理中的应用

| 主张 | 支撑证据 | 反对证据/上下文 |
|------|----------|---------------|
| "新商质量高" | 75%进TOP3 | "但新商需要3个月保护期" |
| "自建团队效率高" | 110%增速 | "但前期招聘花了大量时间" |
| "降本效果显著" | 年省1200万 | "但自建团队也有成本投入" |

**把反对证据不是当弱点，而是当深度分析的证据**。

## 八、载体适配规则

| 载体 | 规则 | 供应商管理案例 |
|------|------|---------------|
| 晋升述职PPT | 页面精简，视觉强化口头观点 | 250字/案例+图表 |
| 向上汇报邮件 | 包含更多细节，确保自包含 | 数据+根因+时间线 |
| 供应商月度报告 | 完整数据+趋势分析 | 排名+分量+整改要求 |

## 总结

数据讲故事的核心不是"图表有多好看"，而是**"数据是否推动了决策和行动"**。理解受众、明确目标、选对图表、去掉噪音、引导注意力、讲好故事、持续迭代——七个步骤缺一不可。最关键的一步往往是最后一步：给讲故事留时间。分析完成后，**留出和分析阶段同等甚至更多的时间做沟通打磨**。
