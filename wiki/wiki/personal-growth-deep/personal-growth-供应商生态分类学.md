## Overview

供应商生态分类学是孙大年管理33家供应商时的核心认知框架。其核心观点是：**供应商不是一个个孤立的个体，而是网状结构中的节点**。通过分类、画像、匹配，实现"把合适的供应商放在合适的地方"，最大化生态整体产出。

## 一、为什么需要分类学？

### 传统管理方式的问题

| 问题 | 具体表现 | 后果 |
|------|---------|------|
| 一视同仁 | 所有供应商用同样的方式管理 | 浪费资源 |
| 只看业绩 | 只关注产能，忽略其他维度 | 错过潜力供应商 |
| 静态管理 | 分类之后不再调整 | 跟不上变化 |
| 孤视角 | 一个一个看供应商 | 看不到生态全貌 |

### 分类学的价值

| 价值 | 说明 | 效果 |
|------|------|------|
| 精准匹配 | 把合适的供应商放在合适的地方 | 产出最大化 |
| 资源倾斜 | 不同类别供应商不同管理力度 | 效率提升 |
| 风险预判 | 知道哪些供应商风险高 | 提前干预 |
| 生态健康 | 从全局看结构是否合理 | 系统性优化 |

## 二、业务分类体系

### 四类业务

| 业务类型 | 特点 | 管理关注点 |
|---------|------|-----------|
| 拉新型 | 需要大量新客，转化率要求高 | 拉新数量、新客质量、转化率 |
| 企微型 | 企业微信私域运营 | 私域转化、用户留存 |
| 大额型 | 高客单价、长周期 | 单客价值、周期管理 |
| 创新型 | 新模式、新流程、不确定因素多 | 适应速度、反馈质量 |

### 业务分类的管理意义

| 维度 | 拉新型 | 企微型 | 大额型 | 创新型 |
|------|--------|--------|--------|--------|
| 适合的供应商 | 擅长拉新、有名单资源 | 擅长私域运营 | 擅长大额、稳定型 | 学习能力强、配合度高 |
| 管理方式 | 关注拉新效率和转化率 | 关注留存和满意度 | 关注单客价值 | 关注适应速度 |
| 风险点 | 名单质量波动 | 用户隐私合规 | 长周期不确定性 | 模式验证失败 |

## 三、供应商画像体系

### 六维标签

| 标签维度 | 分类 | 说明 |
|---------|------|------|
| 业务类型 | 拉新型/企微型/大额型/创新型 | 擅长什么业务 |
| 能力层级 | 头部/中部/尾部 | 当前表现水平 |
| 配合度 | 高/中/低 | 响应速度、执行态度 |
| 稳定性 | 稳定/波动/不稳定 | 业绩是否持续 |
| 潜力 | 高/中/低 | 未来提升空间 |
| 上人速度 | 快/中/慢 | 人员到位速度 |

### 画像矩阵

```
        拉新型          企微型
           |              |
    头部：A、B        头部：E、F
    中部：C、D        中部：G、H
    尾部：I、J        尾部：K、L

        大额型          创新型
           |              |
    头部：M、N        头部：Q、R
    中部：O、P        中部：S、T
    尾部：U、V        尾部：W、X
```

### 画像的使用场景

| 场景 | 使用的标签 | 决策 |
|------|-----------|------|
| 新业务上线 | 业务类型 + 配合度 | 找谁承接 |
| 分量调整 | 能力层级 + 稳定性 | 给多还是给少 |
| 清退评估 | 稳定性 + 配合度 | 是否进入淘汰流程 |
| 培育方向 | 潜力 + 上人速度 | 怎么帮供应商成长 |

## 四、分量打分体系

### 5维权重模型

| 维度 | 权重 | 说明 |
|------|------|------|
| 业绩 | 30% | 产能、转化效率 |
| 质检客诉 | 25% | 合规表现、用户满意度 |
| 上人数字 | 15% | 人员到位率 |
| 配合度 | 15% | 响应速度、执行态度 |
| 战略打分 | 15% | 扶持潜力、学习能力、额外技能 |

### 权重设计的逻辑

| 维度 | 为什么这个权重 | 如果调整会怎样 |
|------|--------------|--------------|
| 业绩30% | 产能是硬指标 | 降低→供应商不重视产出 |
| 质检25% | 合规是底线 | 降低→风险增加 |
| 上人15% | 产能的基础 | 降低→人员不到位 |
| 配合度15% | 管理效率的关键 | 降低→沟通成本高 |
| 战略15% | 长期价值 | 降低→只看眼前 |

### 打分的动态性

| 维度 | 变化周期 | 调整依据 |
|------|---------|---------|
| 业绩 | 月度 | 月度排名数据 |
| 质检客诉 | 月度 | 质检报告、客诉数据 |
| 上人数字 | 周度/月度 | 人员到位情况 |
| 配合度 | 季度 | 综合评估 |
| 战略打分 | 季度 | 战略调整需要 |

## 五、精准匹配策略

### 匹配原则

> "把合适的供应商放在合适的地方，才能产出最大价值。"

### 匹配矩阵

| 业务需求 | 供应商特征 | 匹配示例 |
|---------|-----------|---------|
| 拉新业务要快速起量 | 拉新型 + 头部 + 上人快 | 供应商A |
| 创新型业务要试错 | 创新型 + 高配合 + 高潜力 | 供应商Q |
| 大额业务要稳定 | 大额型 + 稳定 + 头部 | 供应商M |
| 企微业务要精细化 | 企微型 + 高配合 + 稳定 | 供应商E |

### 不匹配的代价

> "没把合适供应商放在合适地方 → 潜在产出损失。"

| 不匹配类型 | 后果 | 案例 |
|-----------|------|------|
| 能力不匹配 | 做不好、出产能慢 | 创新型给尾部供应商 → 失败 |
| 配合度不匹配 | 沟通成本高、响应慢 | 紧急项目给低配合供应商 → 延误 |
| 稳定性不匹配 | 业绩波动、不可预测 | 核心业务给不稳定供应商 → 风险 |

## 六、动态管理

### 动态调整原则

> "不都给，也不一直不给。让供应商保持期待。头部和尾部重点管理。中间供应商可以激励成头部。"

### 供应商流动路径

```
尾部 → 培育 → 中部 → 赛马 → 头部
                              ↓
                          持续下滑 → 尾部 → 淘汰
```

| 流动方向 | 触发条件 | 管理动作 |
|---------|---------|---------|
| 尾部→中部 | 业绩改善、配合度提升 | 增加分量、给新项目 |
| 中部→头部 | 连续排名靠前 | 战略供应商认证 |
| 头部→中部 | 业绩下滑 | 约谈、减少分量 |
| 中部→尾部 | 持续不改善 | 限制新进 |
| 尾部→淘汰 | 严重违规或持续不改善 | 清退流程 |

### 生态健康度评估

| 指标 | 说明 | 健康标准 |
|------|------|---------|
| 头部占比 | 头部供应商的产能占比 | 40-60% |
| 尾部占比 | 尾部供应商的数量占比 | <20% |
| 流动性 | 供应商层级变化的频率 | 月度有变化 |
| 多样性 | 各业务类型的供应商覆盖 | 每类≥3家 |

## 七、分类学的未来

### 从人工分类到智能分类

| 阶段 | 方式 | 特点 |
|------|------|------|
| 当前 | 人工标签、手动打分 | 依赖经验和直觉 |
| 近期 | 系统辅助、数据驱动 | 系统自动计算分数 |
| 未来 | 智能匹配、自动推荐 | 系统根据业务需求推荐供应商 |

### 生态健康度评分模型

| 维度 | 说明 |
|------|------|
| 多样性 | 4-6家均衡配置，单点风险<10% |
| 抗风险能力 | 线路储备超配150%，极端情况下零中断 |
| 进化能力 | 新供应商季度Top3占比60%，供应商梯队持续升级 |

## 总结

供应商生态分类学的核心是：**供应商不是一个个孤立的个体，而是网状结构中的节点**。通过业务分类、画像标签、分量打分、精准匹配和动态管理，实现"把合适的供应商放在合适的地方"。这套分类学的价值不仅在于"管得更细"，更在于**识别潜在的产出损失**——没把合适的供应商放在合适的地方，就是浪费了生态的潜在价值。
