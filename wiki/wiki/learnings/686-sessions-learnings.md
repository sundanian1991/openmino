# 686 会话学习精华与纠错记录

> Sources: mino, 2026-04-28
> Raw: ../../raw/learnings/LEARNINGS.md; ../../raw/learnings/ERRORS.md; ../../raw/learnings/FEATURE_REQUESTS.md

## 概述

基于 686 个对话 session 文件（2026-02-01 至 2026-04-24）回溯提炼的学习、纠错、需求记录。覆盖纠正（correction）、洞察（insight）、知识缺口（knowledge_gap）、最佳实践（best_practice）四大类别，以及错误记录和特性需求。以下按主题聚合核心发现。

## 一、分析准确性纠错

### 问题范围理解

多次出现对问题范围和数据的理解偏差。典型案例：将"定价问题"误判为实际讨论的"开门红激励和三月激励"（时间节点差异）；将"达成率 60-70%"理解为"当前冲刺可达成的目标"，实际含义是"即使拿到影响也很小"。教训：分析问题前先确认问题范围，数据要确认是"当前状态"还是"假设场景"，不能基于假设做判断。

### 组织关系准确性

多次出现组织关系理解错误：将王易人误认为间接上下级（实际是直接老板）、将结算部门误认为"坤哥部门"。教训：组织关系要确认，不能假设，不要用昵称代替部门名。

### 人物分析标签

使用"人格问题"等固化标签导致后续分析先入为主，调整为"争议事件记录"。教训：人物观察用行为描述，不用人格定性标签。

## 二、设计与可视化

### 可视化质量标准

年老师对可视化质量要求极高且具体：文字穿过、箭头没有指向对象等像素级反馈，不是"差不多就行"。AI 只检查代码语法但年老师否决全部 HTML。教训：检查可视化质量必须用浏览器实际打开看渲染效果，不能只检查代码。

### 设计风格偏好

SVG 必须有手工感，Q 贝塞尔曲线优于直线/矩形，禁止 emoji。核心目标不是"学会画虾"，而是建立可识别的风格系统，别人一看就知道是"年老师风格"。每种风格应该有自己的一套图标集，不是一套图标多种颜色。设计流程应为：设计哲学 → 颜色配置 → API 生成。

### 可视化形式选择

不要文本符号模拟可视化，直接用彩色图表。速查表/模板都是底层原理的预计算结果，不是独立路径，方法论要统一。

### HTML 响应式设计

大屏幕被撑满像 PPT 是最不能容忍的问题。教训：设置最大宽度，大屏幕居中，响应式布局要覆盖不同屏幕尺寸。

### 内容均衡

设计时注意各板块内容均衡，避免视觉不协调。内容少的板块不应显得瘦长拉长。

## 三、供应商管理核心洞察

### 培养悖论与组织绑架

京东供应商培养的必然路径：培养 → 做大 → 外部扩张 → 有退路 → 议价/流失。这不是背叛，是模式必然结果。同时存在组织结构绑架：定价权在策略组，流失风险由年老师承担。新人流失率有结构性原因：结算低 → 培训投入不足 → 新人体验差 → 流失高。教训：供应商管理要考虑结构性矛盾，不是单点问题。

### 权责不对等保护

建立边界清单（8 类工作流）+ 留痕模板（7 套话术）+ 决策速查表。核心原则：不揽不该揽的责任，控制可控的，联盟不孤军。

## 四、工作方法与流程

### 代码调试三步法

修补失败 2 次 → 立刻停下来反思方向。架构问题不能靠修补解决，必须重构。问自己："是不是我对问题的理解就错了？"

### 透明工作流

复杂任务（≥7 步/架构修改）必须使用透明工作流：意图分类 → 委派声明 → 任务拆分 → 过程透明 → 验证总结。交互过程中必须有过程信息同步，不是只给"正在考虑下一步"。

### 上下文保护

年老师强调"每一次遇到有可能会撑爆上下文的时候，都应该多用子代理"。不仅是多文件阅读，任何可能大量占用上下文的操作（长文件读取、大量搜索结果、批量数据处理、多步骤分析）都必须用子代理，保护主上下文空间。

### 代码工作流

永远不要让 AI 直接写代码，直到审核通过一份书面计划。避免"直接让 AI 写代码 → 运行 → 改 bug → 越改越乱"。

## 五、技能与工具使用

### 技能评估

评估技能时先看脚本和材料，再看 SKILL.md。脚本内容比 SKILL 更重要，因为最终能否使用好，还是看具体内容和材料是否充分。技能设计要考虑跨平台兼容性。

### 技能使用原则

先按原样安装技能，不要过度分析。技能命名要贴近年老师的认知，不要过度细分（如从"Plan Think"改为"Work Think"）。

### 可视化质量自检

AI 需要自问"你是不是真的看到了这个图的实际渲染情况？"。不能只检查代码语法，必须实际渲染验证。

## 六、沟通与表达

### 语言偏好

强制中文回复，无论年老师输入什么语言。表达简洁，说问题就好，禁止客套话和过渡句。

### 表达层级

四层表达模型（纯感受 → 结构化定性 → 定量 → 定性 + 定量融合），目标用 Level 4 表达。向上汇报用"结果 → 根因 → 下一步 → 时间线"结构，适配军哥穿透式管理风格。

### 审美判断

不是"三选一"，而是在多维空间里定位（温度、密度、权威感、时间感等）。多维度综合评估，不是简单二选一/三选一。

## 七、关键人物观察

### 刘乾坤

"有话没好好说，爱用反着说的话"。年老师最反感的是在其不参与具体业务执行的情况下说"在你的带领下做大做强"这类话，感觉阴阳怪气。

## 八、错误模式汇总

高频错误领域：

> 前端/HTML：数据绑定不完整、响应式布局问题、渲染异常、内容不显示、筛选后数据丢失 — 占比最高的错误类别

> 设计/SVG：手绘 SVG 质量不达标、颜色分段问题、内容不均衡、通报模式过于固定

> 基础设施：API 额度不足、网络连接问题、模型报错、登录困难

> 技能/工具：技能移植时未去原项目特定文化元素、配置偏离核心目标、技能评估只看表面

核心教训：

> 修补失败 2 次 → 立刻停止反思方向

> 不要一出问题就全盘卸载/回滚，先定位具体故障点

> 技术问题先收集信息、分析根因、给出明确结论，不要基于假设

> 多源文档整理时，先列出所有源文件清单，逐一确认是否已对齐

## 九、需求与特性请求

高频需求方向：

> 供应商看板数据完整性（筛选后数据丢失是反复出现的问题）

> 手绘 SVG 达到精细艺术水平，建立风格系统

> 供应商集中度预警系统（总人力占比 >30% 或 ≤2 家供应商时预警）

> 多技能协同工作，共同解决复杂问题

> 可视化示意图生成，先看效果再决定如何使用
