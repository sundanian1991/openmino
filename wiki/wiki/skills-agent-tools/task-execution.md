## 概述

任务执行与技能工具体系由八大核心技能组成，覆盖从想法萌芽到任务交付的完整生命周期。Task Alignment 负责将模糊想法收敛为清晰目标并决定是否固化为独立任务；Task Dispatcher 将明确后的需求通过 IPOFA 五环流程调度给最匹配的专业技能执行；Task Implement 作为 UserProxy Agent 在人类离线时独立执行、验证并交付；Ownership Skills 提供端到端责任引擎，确保问题在 AI 侧终结而不甩锅；Person Observer 深度分析职场人物行为模式和潜台词；Arming Thought 在会话启动时建立"实事求是"总原则并自动选择适用的思想武器；Workspace Tidy 管理三区域文件生命周期；Darwin Skill 借鉴 Karpathy autoresearch 实现 Skills 的自主优化循环。八个技能共同构成"想清楚、派出去、跑到位、负责任、能复盘、会进化"的闭环体系。

## Task Alignment — 想法对齐与任务固化

Task Alignment 解决的核心问题是：一个想法到底该在当前会话里顺手做掉，还是应该固化成独立任务交给子代理去跑。它不是一个填表工具，而是一场对话。

**两条收敛路径**：如果用户希望实时参与、需要大量中间判断、工作范围小，就留在当前会话，内联声明一个微型合同（目标+完成标准）后直接执行。如果用户希望 fire-and-forget、需要定时执行、有明确的独立验证门、或者需要长时间隔离运行，就固化为 Task，写入四份文档（alignment.md 记录决策原因、task.md 作为执行北极星、verify.md 定义验收标准、progress.md 从执行计划变为动态状态文档），然后通过 `myagents task create-from-alignment` 创建任务。

**关键设计**：Alignment 要求先读代码再提问，先查资料再下判断，不猜测只验证。验证标准由 AI 主动提出而非反问用户，分为三层——自动化检查（命令级 pass/fail）、Agent 自查（需要判断的模式一致性）、集成验证（端到端场景）。当发现一个想法实际包含多件事时，强制拆分；当缺少上游信息时，暂停对齐建议先做探索。

## Task Dispatcher — 通用任务调度器

Task Dispatcher 是模糊需求到可复用 Prompt 的生产线，核心产出不是执行结果本身，而是"最终版 Prompt"——年老师可以复制到其他对话直接使用，不用重新解释上下文。

**IPOFA 五环流程**：Input（输入澄清）阶段用 7 元素框架（上下文/目标/思考方式/约束/交付物/缺失处理/示例）替代传统 5W2H，一次性问完不挤牙膏，Mino 再从 memory 和历史记录中补位上下文和风险预警。Process（处理）阶段根据需求类型调度最匹配的专业技能——PPT 调 ppt-master、前端页面调 mino-frontend、供应商分析调判官、制度 SOP 调师爷。遵循"能调专家不调通用、能调内部不调外部"的原则。Output（输出交付）阶段产出最终交付物 + 可复用 Prompt 模板 + 使用说明。Feedback（反馈闭环）追踪产出质量和 Prompt 模板的可用性。Asset（资产沉淀）将模板按场景分类存入 `plans/templates/`，统一索引便于下次复用。

**7 元素设计哲学**：用"思考方式"替代"角色"——定义怎么想而不是定义是谁，避免触发表演模式。"缺失处理"作为一等公民，给模型处理不确定性的出口，防止编造或过度谨慎。风格降级入约束——风格是约束的子项而非独立维度。

## Task Implement — 独立执行代理

Task Implement 扮演 UserProxy Agent 的角色——人类定义任务后离开，它负责执行到交付、验证到通过、全程无需人类在环。

**执行前**：通过 `myagents task get <taskId>` 获取任务文档路径，按序读取 alignment.md（理解决策原因）、task.md（执行北极星）、verify.md（完成标准）、progress.md（执行计划）。如果是 git 仓库则在独立分支上工作，每次变更可追溯。

**执行核心原则**：分解、委派、综合。小而直接的工作自己做，需要干净上下文或多路并行的工作委派给子代理。工作节奏是"计划一步 → 执行 → 检查 → 调整 → 计划下一步"，不是写一个宏大计划然后盲跑。目标不变，计划灵活——可以重排步骤、拆分、新增或跳过。

**验证必须独立**：写完代码的代理有确认偏差，必须由独立的子代理进行代码审查。验证分三层：自动化检查（自己先跑）、独立审查（子代理带着 verify.md 的准则做 pass/fail 评判）、集成验证（端到端场景）。反复失败时不自修，停下来思考方法论是否错误。

**中途重对齐**：发现任务文档与现实不符时停止执行，记录发现，评估影响。小偏差与用户沟通后调整继续，根本性问题需要用户重新决策。只有人类能改目标。

## Ownership Skills — 端到端责任引擎

Ownership Skills 是一套完整的责任体系，核心理念是"你不是执行者，你是 Owner。问题到你这里，就要在你这里终结。"

**五原则**：穷尽方案（穷尽前不说"无法解决"）、先查后问（带着证据提问）、主动闭环（改完→验证→检查同类→确认上下游→汇报）、质量守门（不合格不流出）、复盘沉淀（每个任务产出经验）。

**责任等级**：Observer（只看报错）、Contributor（查上下文+搜索+验证）、Owner（系统排查+发现隐藏问题+同类检查+风险汇报），目标是每个任务都以 Owner 级别执行。

**挑战升级机制**：L1 标准执行 → L2 切换本质不同方案 → L3 搜索+读原始材料+列假设 → L4 完成 7 项检查清单 → L5 最小复现+全新方法+跨界借鉴。每次升级必须有实质变化，不是参数微调。

**OWNER Method 五步法**：Observe（列已尝试方案找共同模式）、Widen（逐字读信号→主动搜索→读原始材料→验证假设→反转思路）、Navigate（自检是否原地打转）、Execute（新方案必须本质不同、有验证标准、失败能产生新信息）、Reflect（复盘四问：目标/结果/根因/经验）。

**反模式识别**：提前放弃、责任转移、虚假穷尽、未验证归因、原地打转、未验证交付、被动等待、粒度不足、信息惰性、知识借口——共 10 种反模式各有检测信号和强制动作。组合反模式（磨洋工+虚假穷尽、未验证交付+责任转移）有更严厉的处理流程。完成 7 项检查清单后仍未解决时，输出体面的失败报告——不是"我不行"，而是"问题的边界在这里"。

**情境方法选择器**：根据挑战类型自动选择最佳方法论——思路卡壳用 Musk 算法、直接放弃用 Netflix Keeper Test、质量不佳用 Jobs 标准、信息不足用百度模式、被动等待用阿里关怀型。

## Person Observer — 人物观察分析

Person Observer 专门分析职场人物的行为模式、潜台词和互动关系，建立人物档案并持续追踪。核心能力是识别"表演型管理"、"责任转移"、"向上展示"等职场微妙手法。

**四层听话模型**（每次分析强制校验）：第一层听对方说了什么，第二层思考他没说什么，第三层想他为什么这么说，第四层想他为什么这么对我说。第四层是强制检验——他为什么选择对我而不是对别人说？他预期我会怎么反应？他真正想让谁看到这个行为？

**行为模式库**：抛问题（只出数据不出结论）、表演型同步（全程观众意识）、公开锁定（群里@制造公开压力）、软硬施压（阴阳怪气+公开追问切换）、正面对冲（好消息平衡坏消息）、任务下放+CC展示（给下属任务同时向上展示培养）、责任转移（成了是"我们一起"，没成是"他们还没"）。

**工作流程**：自动读取索引文件和已有档案 → 意图确认 → 深度分析（表面行为拆解+潜台词解读+行为模式匹配+关系网络分析） → 更新索引和档案 → 输出洞察和行动建议。档案包含基本信息、核心行为模式、互动模式、风格对比、关键观察记录和待观察问题，按上级/同事/供应商分类存放。

## Arming Thought — 求是思想工具箱

Arming Thought 在每次顶层对话开始时自动调用，建立"实事求是"总原则并根据任务类型自动选择适用的思想武器。子代理处理单一具体任务时跳过。

**总原则**：先看事实再下判断，事实与预想冲突时修改判断而非事实。不空谈看事实，验证才算完成，承认不知道，遇阻探原因。

**九大思想武器**：矛盾分析法（复杂问题抓主要矛盾）、调查研究（信息不足时深入一线）、实践认识论（试错迭代螺旋上升）、集中兵力（多任务争夺时伤其十指不如断其一指）、持久战略（长期任务划分阶段稳步推进）、批评与自我批评（完成后查问题找根因定措施）、群众路线（收集多方意见系统化后返回验证）、星火燎原（从零起步找最小可行切入口）、统筹兼顾（多目标冲突时寻找系统平衡）。

**智能调度规则**：按任务特征自动匹配思想武器——复杂问题用矛盾分析法，信息不足用调查研究，需验证方案用实践认识论，多任务争夺用集中兵力。一次只用一个主武器，确有必要时才串联第二个。

## Workspace Tidy — 三区域文件生命周期管理

Workspace Tidy 管理 docs/、memory/、workspace/ 三个文件夹的文件生命周期，统一命名规范，归档散落文件。

**三区域架构**：workspace/ 是入口区（草稿、进行中、产出物，周度清理），docs/ 是归档区（正式文档、知识资产，长期保留），memory/ 是核心区（洞察、决策、人物画像，永久保留）。

**Workspace 内部结构**：inbox（新文件入口必须清零）→ drafts（草稿编辑中）→ active（进行中项目，`{主题}-{日期}/` 命名）→ output（已交付产出物）→ archive（按周期归档）。命名统一用 `{YYYY-MM-DD}-{主题}/` 格式，禁止"完整版""最终版"等命名。

**五阶段流程**：扫描诊断（找出问题文件）→ 分类整理（按区域目标结构移动）→ 生成移动计划（展示每个文件的源位置、目标位置、理由）→ 执行移动（先删除后移动最后验证）→ 验证与记录。

**智能价值判断**：包含"总结/报告/规范/机制"的文档价值高，归档到 docs/；包含"洞察/发现/经验/教训"的提炼到 memory/；包含"草稿/临时/test/TODO"的价值低，删除或放 drafts/。内容空洞/重复/过时的直接删除。定期提醒在 workspace/ 文件数超过 50 或距上次整理超过 7 天时触发。

## Darwin Skill — Skills 自主优化器

Darwin Skill 借鉴 Karpathy autoresearch 的自主实验循环，对 SKILL.md 文件进行持续优化。核心理念是"评估 → 改进 → 实测验证 → 人类确认 → 保留或回滚"。

**五条核心原则**：单一可编辑资产（每次只改一个 SKILL.md）、双重评估（结构评分+效果验证）、棘轮机制（只保留改进自动回滚退步）、独立评分（评分用子代理避免自己改自己评的偏差）、人在回路（每个 Skill 优化完暂停等用户确认）。

**8 维度评估体系**（满分 100）：结构维度 60 分——Frontmatter 质量（8）、工作流清晰度（15）、边界条件覆盖（10）、检查点设计（7）、指令具体性（15）、资源整合度（5）；效果维度 40 分——整体架构（15）、实测表现（25）。实测表现是最大特色：为每个 Skill 设计 2-3 个典型用户 prompt，用子代理分别带 skill 和不带 skill 跑，对比输出质量打分。

**自主优化循环**：Phase 0 初始化（确认范围、创建 git 分支）→ Phase 0.5 测试 Prompt 设计（覆盖典型场景和复杂场景）→ Phase 1 基线评估（结构评分主代理做，效果评分子代理独立做）→ Phase 2 优化循环（找出最低分维度 → 生成改进方案 → 编辑提交 → 子代理重评 → 新分>旧分保留否则 git revert → 展示改动摘要等用户确认）→ Phase 3 汇总报告。Hill-climbing 涨不动时可探索性重写（git stash 保存当前版本，从头重写，比 stash 版好才采用）。

**约束规则**：不改变 Skill 核心功能、不引入新依赖、每轮只改一个维度、优化后不超过原始大小 150%、所有改动在 git 分支上可回滚、效果维度必须独立评分。
