# Legacy Vault — 技能存档管理器

> Sources: MyAgents, 2026-04-27
> Raw: [legacy-vault](../../raw/skills/legacy-vault-SKILL.md)

## 概述

Legacy Vault 是一套技能存档管理机制，负责管理 `~/.claude/skills/_legacy/` 目录下的已归档技能。核心理念是"暂存而非删除"——当某个技能暂时不需要但未来可能复用时，将其移入存档目录使 Claude 不再自动加载，需要时再从存档恢复。该技能覆盖"列出、检视、恢复、存档、清理、整理建议"六大操作，并处理冲突、批量、二次确认等边界情况，确保技能目录始终保持精简高效。

## 存档位置与进出机制

存档目录与技能目录的关系如下：

- **存档目录**：`~/.claude/skills/_legacy/`，存放暂不使用的技能
- **技能目录**：`~/.claude/skills/`，存放当前生效的技能

移入存档操作将技能从技能目录移动至存档目录，该技能在后续会话中不再被 Claude 自动加载。移出存档操作则将技能从存档目录移回技能目录，恢复自动加载状态。这种"进出"管理机制避免了直接删除造成的不可逆损失，同时也防止了技能目录中堆积大量无用技能影响加载效率。

## 核心操作

### 1. 列出存档（list）

当用户表达"看看 legacy 里有什么"、"查看存档"、"有哪些存档技能"等意图时触发。通过 `ls -la ~/.claude/skills/_legacy/` 列出所有存档子目录，并对每个子目录进一步检查是否存在 SKILL.md 或 README.md 文件——有则显示文件大小，没有则标注为"空壳"。

输出采用结构化格式，包含总览统计（存档总数、有内容的数量、空壳数量），并按"有内容的技能（可恢复）"和"空壳技能（可清理）"两个分类分别列出。这使用户能一目了然地判断哪些存档值得保留、哪些可以直接清理。

### 2. 查看详情（inspect）

当用户说"看看 XX 里面有什么"或"XX 是什么"时触发。先通过 `ls -la` 查看指定存档目录的文件结构，若存在 SKILL.md 或 README.md 则显示文件前 10 行内容，让用户快速了解该技能的功能定位。若目录下无任何说明文件，则明确告知用户这是一个空壳存档。

### 3. 恢复技能（restore）

当用户表达"恢复 XX"、"把 XX 放回去"、"XX 我想再用"等意图时，执行 `mv` 操作将技能从 `_legacy/` 移回 `~/.claude/skills/`。

恢复前需检查两项前提条件：目标位置是否已有同名文件夹——若有则报错提示冲突，不执行移动操作；指定技能是否真的存在于存档目录中——若不存在则报错提示该技能不在存档列表中。操作成功后明确告知用户技能已恢复且下次会话生效，避免用户产生"改了但没生效"的困惑。

### 4. 存档技能（archive）

当用户说"把 XX 存档"或"XX 先不用了放到 legacy 里"时，执行反向的 `mv` 操作，将技能从 `~/.claude/skills/` 移入 `_legacy/`。

存档前同样需要检查冲突：若存档目录中已存在同名文件夹，则提示用户先清理再存档；若技能目录中根本不存在该技能，则提示不存在。操作成功后告知用户技能已移入存档。

### 5. 清理技能（clean）

当用户说"清理 XX"、"删除 XX"或"XX 不要了"时，执行 `rm -rf` 删除存档目录下的对应子目录。

清理操作有严格的安全约束：只有空壳技能或用户明确表示不要了的技能才能直接删除。对于有内容的技能（即包含 SKILL.md 的技能），必须执行二次确认——向用户展示该 SKILL.md 的核心描述内容，询问"确定要删吗？"。此外支持批量清理操作，如"清理所有空壳"可一次性处理全部无内容的空壳存档。

### 6. 整理建议（suggest）

当用户说"帮我看看哪些该存档"或"整理一下技能"时，执行全面的技能目录审计：

第一步扫描技能目录 `~/.claude/skills/` 下所有已安装技能，第二步扫描存档目录 `~/.claude/skills/_legacy/` 下所有存档，第三步对照技能清单文档 `docs/skills-inventory-2026-04.md`（若该文件存在）获取各技能的使用频率和功能分类信息。基于以上三步信息，给出具体建议：哪些低频技能建议移入存档、哪些存档可以安全清理。

## 边界情况处理

| 情况 | 处理方式 |
|------|----------|
| 恢复时目标位置已有同名文件夹 | 报错提示同名冲突，不执行移动操作 |
| 存档时存档目录中已存在同名文件夹 | 报错提示先清理再存档 |
| 对不存在的技能执行任何操作 | 报错提示该技能名不在列表中 |
| 批量清理操作 | "清理所有空壳"需逐个确认后执行，避免误删 |
| 用户使用"所有"、"全部"等全局指令 | 先列出受影响的具体列表，确认后再执行，不直接批量操作 |

边界情况处理的核心原则是：涉及删除的操作必须有确认环节，涉及移动的操作必须检查冲突，涉及全局的操作必须列出影响范围。这确保存档管理不会因误操作导致技能丢失。
