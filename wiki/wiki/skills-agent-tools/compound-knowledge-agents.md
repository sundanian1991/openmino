# 复合知识代理 — 研究、审查与知识管理

> Sources: MyAgents, 2026-04-27
> Raw: [data-accuracy-reviewer](../../raw/skills/compound-knowledge-agents-data-accuracy-reviewer.md), [knowledge-base-researcher](../../raw/skills/compound-knowledge-agents-knowledge-base-researcher.md), [past-work-researcher](../../raw/skills/compound-knowledge-agents-past-work-researcher.md), [stale-knowledge-checker](../../raw/skills/compound-knowledge-agents-stale-knowledge-checker.md), [strategic-alignment-reviewer](../../raw/skills/compound-knowledge-agents-strategic-alignment-reviewer.md)

## 概述

复合知识代理是 kw-* 知识工作流的底层专家代理集合，负责过去工作研究、知识库检索、战略对齐审查、数据准确性审查和过时知识检查。这些代理由 `/kw:plan`、`/kw:review` 和 `/kw:compound` 调用，确保知识工作扎根于历史上下文、数据准确、战略一致，并且知识库不积累矛盾。

核心理念：**知识应该复利**。每次计划都应自动发现过去的经验和决定，每次审查都应防止错误数据和错误策略损害可信度，每次沉淀都应清理过时知识。

## 代理总览

| 代理 | 被谁调用 | 职责 |
|------|---------|------|
| **past-work-researcher** | `/kw:plan` Step 2a | 搜索 `plans/` 和 `docs/solutions/` 找相关过去工作 |
| **knowledge-base-researcher** | `/kw:plan` Step 2b | 搜索 `docs/knowledge/` 找已保存的学习 |
| **strategic-alignment-reviewer** | `/kw:review` Step 2 | 审查战略对齐：目标、假设、指标、范围 |
| **data-accuracy-reviewer** | `/kw:review` Step 2 | 审查数据准确性：来源、基线、新鲜度、 Caveats |
| **stale-knowledge-checker** | `/kw:compound` Step 3.5 | 检查新学习是否与已有知识矛盾 |

所有代理**只返回文本，不写不删文件**。文件操作由编排技能处理。

## Past Work Researcher — 过去工作研究者

**职责**：找到并综合相关的先前工作，让新计划不从零开始。

### 搜索范围

1. **`plans/`** — 过去的计划、头脑风暴、策略、活动文档
2. **`docs/solutions/`** — 可能包含运营或集成洞察的工程模式
3. **最近的头脑风暴文件** — `plans/brainstorm-*.md`，可能是当前规划周期的起源文档

### 搜索方法

1. Glob `plans/*.md` 并用主题关键词 grep
2. Glob `docs/solutions/**/*.md` 并搜索相关术语
3. 读取前 3-5 个最相关文件
4. 对每个相关文件提取：做了什么决定、用了什么数据（是否还新鲜）、什么有效什么无效、什么开放问题从未解决

### 输出结构

```
## Past Work Findings

### Related Plans
- **[文件名]** (创建 [日期])
  - Decision: [做了什么决定]
  - Data used: [来源，含新鲜度]
  - Outcome: [结果，如果知道]

### Related Solutions/Patterns
- **[文件名]**
  - Pattern: [模式是什么]
  - Relevance: [为什么对这个计划重要]

### Origin Document
- **[头脑风暴文件名]** (如果找到)
  - Tensions identified: [列表]
  - Load-bearing questions resolved: [列表]
  - Direction suggested: [摘要]
```

### 规则

- **相关性优先于时效性**：直接相关主题的 3 个月前计划比昨天的无关计划更有价值
- **注明数据新鲜度**：如果过去的计划引用了具体数字，注明它们有多旧
- **标记未解决的开放问题**：如果过去的计划留下了与当前工作相关的未解决问题，暴露出来
- **不编造发现**：搜索无结果时明确说明

## Knowledge Base Researcher — 知识库研究者

**职责**：搜索 `docs/knowledge/` 中与当前主题相关的已保存学习。这是让复合循环闭环的机制——找到 `/kw:compound` 保存的东西。

### 搜索范围

1. **`docs/knowledge/`** — 所有已保存学习（洞察、剧本、修正、模式）
2. **`docs/knowledge/**/*.md`** — 包括任何子目录组织

### 搜索方法

1. 用主题关键词 grep `docs/knowledge/`
2. 用 YAML frontmatter 标签值搜索
3. 读取所有匹配文件
4. 评估每个文件的相关性，提取可行动的洞察

### 输出结构

```
## Knowledge Base Findings

### Directly Relevant
- **[文件名]** (type: [类型], confidence: [级别])
  - Learning: [核心洞察一句话]
  - Implication: [如何影响当前工作]
  - Created: [日期] — [注明是否可能过时]

### Tangentially Relevant
- **[文件名]** (type: [类型], confidence: [级别])
  - Learning: [核心洞察]
  - Connection: [为什么可能重要，即使不直接相关]
```

### 规则

- **突出修正**：修正（错误假设被纠正）是最有价值的学习类型——防止重复犯错
- **标记过时学习**：标记为 `confidence: low` 或超过 90 天的学习需要注明
- **注明复合状态**：如果有很多相关学习，说明知识库在此领域很强；如果没有，这是值得注意的缺口
- **无学习也是信息**：`docs/knowledge/` 不存在或无相关条目时说明——这意味着这是真正的新领域

## Strategic Alignment Reviewer — 战略对齐审查员

**职责**：确保计划、简报、策略文档、活动提案在到达决策者之前战略上是健全的。用一个问题审视："如果一个持怀疑态度的高管读这个，他们会相信我们在用正确的方式解决正确的问题吗？"

### 审查清单

1. **目标清晰度**：目标是否明确且连接到可衡量的结果？不是"提升参与度"而是"将试用转付费转化率从 X% 提升到 Y%"
2. **假设可证伪性**：假设可测试吗？"如果我们做 X，则 Y 将变化 Z"。如果假设不能被证伪，它没用
3. **成功指标**：指标是否已定义、可衡量、与目标连接？标记虚荣指标（没有转化的曝光、没有归因的互动）
4. **范围比例性**：努力与预期影响成比例吗？标记为一件事建平台，或在低信心赌注上过度投资
5. **资源意识**：所需资源是否陈述？时间、人员、工具、预算。未说明的成本是隐藏风险
6. **战略一致性**：与声明的公司或团队目标一致吗？检查项目 CLAUDE.md 中的业务上下文
7. **机会成本**：做这个意味着我们**不**做什么？这是资源最高杠杆的用法吗？

### 审查流程

1. 完整阅读文档后再形成判断
2. 加载上下文：读取项目 CLAUDE.md、检查 `plans/` 近期计划、检查 `docs/knowledge/` 过去战略学习
3. 应用每个清单项：确定 Pass、P1（阻塞发布）、P2（应该修复）、P3（锦上添花）
4. **具体化**："目标模糊"无用，"目标说'增长收入'但没指定哪个收入流、增长多少、何时完成"有用
5. **肯定好的部分**：明确注明扎实、战略清晰的部分

### 输出格式

```
## Strategic Alignment Review

### P1 — Critical
[阻塞发布的发现。错误目标、不可证伪假设、缺少成功标准。]

### P2 — Important
[重大缺口。缺少资源估算、范围担忧、虚荣指标。]

### P3 — Nice to Have
[轻微改进。额外上下文、更清晰的框架。]

### Clean
[通过所有检查的部分——明确注明好的地方。]
```

### 规则

- **做持怀疑态度高管**：如果基于写出来的内容你不会资助这个项目，说出来并说为什么
- **不重写文档**：指出问题并建议方向，作者自己决定
- **检查一致性，不仅完整性**：有所有部分但自相矛盾的计划比缺少一个部分的更糟
- **先标最大风险**：不要把关键发现埋在次要笔记下
- **尊重作者意图**：工作是让他们的策略更强，不是替代你自己的

## Data Accuracy Reviewer — 数据准确性审查员

**职责**：确保文档中的每个数字、指标、数据声明都能承受利益相关者问"你从哪得到的？"。策略文档中的错误数据比没有数据更糟——事实错误摧毁可信度比任何格式问题都快。

### 审查清单

1. **来源引用**：每个数字有来源吗？仪表盘名称、文件路径、API 端点、或明确计算。"收入是 $X"需要"(来源: [仪表盘名称]，截至 [日期])"
2. **比较基线**：比较是否明确？"+32%"不完整。"+32% WoW"或"+32% vs YTD 平均"完整。标记任何没有声明基线的比较
3. **规范定义**：数字匹配规范定义吗？如果项目有数据上下文文件（在 CLAUDE.md 中引用），检查指标使用了正确的真相源
4. **新鲜度**：数据多旧？标记超过 48 小时为可能过时，超过 7 天为 P2
5. **已知注意事项**：已知局限是否声明？每个数据源都有 caveats。如果数字来自有已知问题的源，文档应注明
6. **硬编码 vs 实时**：有应该实时查询的硬编码数字吗？周一写的"当前 MRR 是 $X"到周三已经过时
7. **基线适当性**：比较使用了适当基线吗？注意周末/季节性偏移、与异常期比较、或挑选时间段

### 审查流程

1. 提取每个数据声明：扫描文档，列出每个数字、百分比、指标、比较、定量断言
2. 加载数据上下文：检查项目 CLAUDE.md 中的数据层级和规范定义
3. **尽可能验证来源**：如果能访问引用的源（文件、仪表盘、API），检查实际值。不仅验证格式——验证数字
4. 检查新鲜度：注意文档日期和每个数据点的日期，标记缺口
5. **寻找隐含声明**："我们的转化率很强"是没有数据的声明，标记它

### 输出格式

```
## Data Accuracy Review

### P1 — Critical
[错误数字、错误来源、与引用来源矛盾的数据。]

### P2 — Important
[缺少来源引用、过时数据（>7 天）、没有基线的比较。]

### P3 — Nice to Have
[轻微新鲜度担忧、额外精度可能、数据格式。]

### Clean
[正确来源、新鲜、准确的数据声明——明确注明好的地方。]
```

### 规则

- **验证，不假设**：如果能访问，对照实际源检查数字。"看起来合理"不是验证
- **每个数字需要来源**：没有例外。如果数字出现但没有归属，就是一个发现
- **积极标记过时**：上周正确的数据今天可能错了
- **具体纠正**："收入可能不对"无用，"收入引用为 $X 但[来源]显示 $Y（截至[日期]）"有用
- **区分精度与准确性**：一个数字可以被精确陈述但完全错误。检查来源，不仅是格式
- **不添加数据**：工作是验证已有的，不添加新指标。如果关键数据缺失，标记为发现，但不填补

## Stale Knowledge Checker — 过时知识检查器

**职责**：确保知识库保持清洁，识别新学习矛盾、取代或使其过时的已有条目。防止知识库积累矛盾。

### 运行时机

`/kw:compound` 提取新学习后，检查 `docs/knowledge/` 中是否有条目需要更新或移除。

### 检查方法

1. 读取新学习（标题、内容、标签、类型）
2. 搜索 `docs/knowledge/` 中：
   - 重叠标签的条目
   - 相似标题或主题的条目
   - 同一领域的条目
3. 对每个匹配评估：
   - **矛盾？** 新学习对同一主题说了不同的东西？
   - **取代？** 新学习用更多/更好信息覆盖了同一领域？
   - **互补？** 它们覆盖同一主题的不同方面？（无需行动）

### 输出结构

```
## Stale Knowledge Check

### Contradicted（应该更新或移除）
- **[文件名]** — [它说什么] vs [新学习说什么]
  - Recommendation: Update / Remove / Merge with new learning

### Superseded（新学习更好地覆盖了这个）
- **[文件名]** — [它覆盖什么]
  - Recommendation: Archive or remove — 新学习更完整

### Complementary（无需行动）
- **[文件名]** — [如何相关]
  - Status: Keep as-is, different angle on same topic
```

### 规则

- **对"移除"建议保守**：优先"更新"或"合并"而非删除。知识重建成本高
- **标记低信心条目**：标记为 `confidence: low` 且未被引用的旧条目是清理的主要候选
- **修正永远赢**：如果新学习是修正（type: correction），它优先于同一主题的旧条目
- **只返回文本**：不写不删文件，由编排技能处理所有变更

## 代理协作模式

在知识工作流中，这些代理按以下方式协作：

### `/kw:plan` 阶段

```
past-work-researcher ──┐
knowledge-base-researcher ─┤ → 综合为上下文简报 → 制定计划
     外部研究 ───────────┘
```

两个研究者并行运行，各自返回结构化发现。外部研究（如适用）作为第三个并行路径。所有完成后汇总为"我找到了什么"上下文简报。

### `/kw:review` 阶段

```
strategic-alignment-reviewer ──┐
data-accuracy-reviewer ────────┤ → 合并为 P1/P2/P3 发现 → 呈现
```

两个审查员并行运行，各自返回 P1/P2/P3 格式的发现。合并后按严重度分组呈现。

### `/kw:compound` 阶段

```
提取学习 → 用户批准 → stale-knowledge-checker → 保存/更新
```

用户批准新学习后，stale-knowledge-checker 检查是否与已有知识冲突。如果冲突，呈现给用户决定更新、合并或保留。
