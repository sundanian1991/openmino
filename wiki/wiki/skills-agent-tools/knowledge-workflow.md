# 知识工作流 — 从头脑风暴到知识复利

> Sources: MyAgents, 2026-04-27
> Raw:[kw-brainstorm](../../raw/skills/kw-brainstorm-SKILL.md); [kw-compound](../../raw/skills/kw-compound-SKILL.md); [kw-confidence](../../raw/skills/kw-confidence-SKILL.md); [kw-plan](../../raw/skills/kw-plan-SKILL.md); [kw-review](../../raw/skills/kw-review-SKILL.md); [kw-work](../../raw/skills/kw-work-SKILL.md); [kw-workflow](../../raw/skills/kw-workflow-SKILL.md)

## 概述

kw-* 是一套完整的知识工作技能链，覆盖从问题理解、计划制定、信心评估、质量审查、执行交付到知识沉淀的全流程。七个子技能可以独立使用，也可通过 kw-workflow 一键串联。

核心哲学：**知识工作应该复利**。每次会话产生的洞察、修正、模式都应该保存为可检索的知识条目，被未来的计划自动发现。

## 工作流总览

```
brainstorm → plan → [confidence?] → review → work → compound
```

| 阶段 | 技能 | 功能 | 输出 |
|------|------|------|------|
| 头脑风暴 | kw:brainstorm | 捕获想法、识别张力、解决负载问题 | `plans/brainstorm-{name}.md` |
| 制定计划 | kw:plan | 研究过去工作、结构化计划 | `plans/{type}-{name}.md` |
| 信心检查 | kw:confidence | 诚实评估已知和未知 | `plans/confidence-{date}.md` |
| 质量审查 | kw:review | 战略对齐+数据准确性双重审查 | `03-review.md` |
| 执行工作 | kw:work | 分批次执行计划产出 | `02-workspace/` |
| 知识复利 | kw:compound | 提取洞察保存到知识库 | `docs/knowledge/{slug}.md` |

## Brainstorm — 头脑风暴

**触发场景**：会议后需要整理、新项目启动、"帮我理清这个问题"、有零散输入需要组织。

**核心定位**：把脑中的一切倒出来，理解问题空间，在承诺路径之前摸清全貌。**不要跳到解决方案。**

### 七步流程

**Step 1：捕获头脑倾倒**。接受任何输入形式：会议纪要、语音转文字、半成型想法、文档链接、"我在想……"。**不急于组织**，先确认输入类型。

**Step 2：提取核心元素**。从头脑倾倒中抽取：
- **关键决策**：需要做出的决定
- **开放问题**：尚无答案的问题
- **约束条件**：时间线、预算、依赖、障碍
- **利益相关者**：谁在乎什么
- **数据点**：提到的数字、指标、参考
- **想法和选项**：提出的方案（哪怕不成熟）

**Step 3：拉取参考资料（自动）**。并行搜索 `docs/knowledge/`、`plans/`、`docs/solutions/`，为每个找到结果标注"相关原因"和"关键收获"。

**Step 4：识别主题与张力**。
- **主题**：反复出现什么？真正的基础问题是什么？
- **张力**：哪些想法冲突？哪里有权衡？
- **缺口**：缺少什么？什么没被覆盖？

**Step 5：解决负载问题**。从开放问题中识别哪些是**负载性的**——即答案不同会导致计划结构完全不同的问题。常见的负载问题：
- **范围**：速赢还是多阶段？
- **受众**：为谁做的？
- **优先级**：X 和 Y 哪个更重要？
- **时间线**：天、周还是月？
- **决策人**：谁拍板？
- **资源上限**：有预算天花板吗？

用 AskUserQuestion 同时问 1-3 个负载问题（不超过 3 个）。如果所有问题都不负载，跳过并说明。

**Step 6：建议方向**。基于一切（包括已解决的问题），给出观点："核心问题是 [X]，主要张力在 [A] 和 [B] 之间，我建议 [方向]，因为 [理由]，但 [注意事项]。"

**Step 7：提供下一步**。选项：运行 `/kw:plan` 结构化计划、深入特定主题、推到 Proof 共享、保存为文件、继续补充。

**关键规则**：
- **不跳解决方案**：头脑风暴的目的是理解问题空间
- **反映，不重写**：用用户的语言，不要用企业话术清洗思考
- **早暴露张力，晚解决负载问题**：2-4 步命名冲突，第 5 步只解决改变计划形状的问题
- **拉，不推**：问参考可能在哪，而非猜测
- **输入量无上限**：30 分钟会议纪是好输入，不要让人先组织再头脑风暴

### Pipeline Mode

从自动化系统调用时：跳过所有确认，使用默认值，自动写文件并推进到下一技能。

## Confidence — 信心检查

**触发场景**：承诺计划前、感觉不确定但说不清哪里、研究后验证是否足够、任何 `/kw:` 工作流中途。

**核心定位**：像同事一样诚实地说清楚你知道什么、不知道什么，然后决定是否继续或深挖。

### 评估维度

内部思考四个区域（独立评估，互不 inflate）：
- **任务理解**：清楚被要求做什么吗？
- **信息充分性**：有做好这件事所需的一切吗？
- **方法确定性**：方法有 proven 还是在猜？
- **风险意识**：能看到什么可能出错吗？

### 输出结构

```
## Confidence Check

**Confident about:** 你知道的及原因（具体到读过的文件、识别的模式）
**Less confident about:** 不知道的及为什么重要（具体缺口：缺失数据、未验证假设）
**My recommendation:** 三选一："Proceed." / "Proceed, but [caveat]." / "Pause for [specific thing]."
```

**高信心时简短**：两句话即可。不要强求完整 breakdown。

### 提升信心

用户选择"Increase confidence"时，产出按影响力排序的可执行动作列表：
1. [做什么] — [为什么重要，关闭什么缺口]
2. [做什么] — [为什么重要]
3. [做什么—注明是否需要用户输入] — [为什么重要]

每个动作必须**立即可执行**："读 `data/q4-results.csv` 确认 $50K 基准"，而非"收集更多数据"。

### 重要规则

- **永远不给数字评分**：不用百分比、1-10 量表、字母等级，用散文体
- **具体**："缺少 Q4 数据"而非"有些信息缺口"
- **不掩饰已知**：信心剧场（处处掩饰以显得谨慎）比过度自信更糟
- **动作可执行**：列表中的每个项必须现在就能做
- **非破坏性中断**：在工作流中间调用后，精确恢复中断点
- **比例适当**：高信心=2 句话，混合信心=几段短话
- **这不是 `/kw:review`**：Confidence 评估**你**的认知状态，Review 评估**完成的产物**质量

## Plan — 计划制定

**触发场景**：头脑风暴后准备承诺方向、新策略文档/活动计划、"规划 X"、任何受益于历史上下文的知识工作。

**核心定位**：研究你已知道的一切，然后制定扎根于数据和过去经验的计划。**结论先行。**

### 工作类型自动分类

| 类型 | 信号 | 开头 |
|------|------|------|
| **Strategy** | 路线图、架构、长期、阶段 | 推荐 |
| **Campaign** | 发布、促销、时间线、渠道 | 时间线 |
| **Brief** | 给别人的指令、范围、交付物 | 推荐+范围 |
| **Research** | 调查、竞争、分析、综合 | 发现 |
| **Operations** | 手册、SOP、重复流程 | 触发+步骤 |

### 详细度三级

| 级别 | 信号 | 输出 |
|------|------|------|
| **Quick** | "要不要？"、快速检查、单问题、<30 分钟 | 标题+推荐+2-3 条理由+1 个成功指标 |
| **Standard** | 默认——大多数计划 | 该工作类型的完整模板 |
| **Deep** | "重构"、"战略"、"多季度"、"根本性" | 完整模板+研究附录+竞争分析+风险矩阵+分阶段时间线 |

### 研究阶段（并行）

1. **过去工作**：启动 `compound-knowledge:research:past-work-researcher`，返回相关计划、过往决策、起源头脑风暴文档
2. **知识库**：启动 `compound-knowledge:research:knowledge-base-researcher`，返回 `docs/knowledge/` 中的洞察、修正、剧本、模式
3. **外部研究**：如话题受益于外部上下文，搜索框架、最佳实践、竞争案例（内部运营可跳过）
4. **实时数据**：如果计划涉及数据，拉取当前指标
5. **起源文档检查**：搜索 `plans/brainstorm-*.md`，找到后作为起源文档，在计划中交叉引用

### 上下文简报

在写任何东西之前，呈现研究发现：
```
## What I Found
**Related plans:** [计划名] — [相关点]
**Past learnings:** [洞察标题] — [洞察]
**Current data:** [指标]: [值] ([来源, 日期])
**External research:** [发现] — [来源]
```

### 五种模板

**策略（Strategy）**：金字塔原理，结论先行
```markdown
# [标题]
## Recommendation → [做什么+为什么]
## Current State → [数据，每个数字标注来源和日期]
## Proposed Approach → [如何达成，分层或分阶段]
```

**活动（Campaign）**：时间线优先
```markdown
## Timeline → [日期|动作|渠道|负责人]
## Goal → [目标+成功标志]
## Audience → [目标人群]
## Assets Needed → [需要的产出]
```

**简报（Brief）**：指令优先
```markdown
## Recommendation → [做什么+为什么]
## Scope → [范围内/外]
## Deliverables → [具体产出]
## Constraints → [时间、预算、依赖]
```

**研究（Research）**：发现优先
```markdown
## Key Findings → [发现]+[数据+来源+日期]
## Implications → [对业务的意义]
## Methodology → [如何收集数据]
## Raw Data → [表格/图表/仪表盘链接]
```

**运营（Operations）**：触发优先
```markdown
## Trigger → [何时运行：定时/事件/请求]
## Steps → [步骤+细节+工具+命令]
## Edge Cases → [情况|对策]
## Owner → [谁运行+升级给谁]
```

### 公共尾部

每个计划末尾都包含：
```markdown
## Success Metrics → [指标|当前基线|目标|来源]
## Open Questions → [还不知道的]
## References → [使用的计划、知识条目、数据源]
```

### 重要规则

- **读者需要什么先给什么**：如果只读第一节，应该得到最重要的信息
- **引用一切**：每个数据点都要来源
- **暴露过去工作**：相关知识必须出现在上下文简报中
- **不要过度模板化**：跳过不适用的部分
- **优雅降级**：数据源失败时，用已有的东西继续，注明缺失

## Review — 质量审查

**触发场景**：`/kw:plan` 后验证计划、与利益相关者分享前、"审查这个计划"。

**核心定位**：两个自动审查员检查可能损害可信度的错误：错误策略和错误数据。

### 审查内容

| 情况 | 审查什么 |
|------|---------|
| `/kw:plan` 刚运行 | 它产生的计划文件 |
| 用户指向文件 | 那个文件 |
| 用户粘贴内容 | 那部分内容 |
| 不明确 | 问用户 |

### 双审查员并行

1. **战略对齐审查员**（`compound-knowledge:review:strategic-alignment-reviewer`）：检查目标清晰度、可证伪假设、成功指标、范围比例性、资源意识、战略一致性
2. **数据准确性审查员**（`compound-knowledge:review:data-accuracy-reviewer`）：检查来源引用、比较基线、规范定义、新鲜度、注意事项、硬编码数字

### 严重度分级

| 严重度 | 定义 | 示例 |
|--------|------|------|
| **P1 Critical** | 事实错误、错误数据源、缺少目标、不可证伪假设 | "指标引用了错误来源" |
| **P2 Important** | 缺少来源引用、过时数据、不清的成功指标 | "转化率没有比较基准" |
| **P3 Nice-to-have** | 轻微框架、额外上下文、格式 | "可指定这个指标的时间段" |

**P1 = 硬闸门**。策略文档中的事实错误比错别字更严重。

### 编辑检查（对外内容）

如果内容将公开发布：检查 AI 写作模式（通用措辞、套话过渡、模糊主张）、语调和声音一致性。内部内容跳过此步。

### 重要规则

- **验证，不假设**：如果引用了数字，尽可能对照实际来源检查
- **标记过时**：数据超过 48 小时给新鲜度警告，超过 7 天给 P2
- **具体**："数据可能不对"无用，"收入引用为 $X 但来源显示 $Y（截至 [日期]）"有用
- **肯定好的部分**：不要只标问题，也注明扎实、清晰的部分

## Work — 执行工作

**触发场景**：`/kw:plan` 或 `/kw:review` 后计划就绪、"开始做这个"、"执行计划"。

**核心定位**：有计划了，执行它。拆成任务，做出来，追踪发生了什么。

### 任务拆分

从计划中提取具体交付物，每个交付物创建一个任务。

**知识工作类型映射**：

| 交付物 | 执行方式 |
|--------|---------|
| 策略文档/简报 | 用计划结构写出来 |
| 邮件草稿 | 写、检查语气、提交审查 |
| 社交媒体文案 | 写变体、对照风格指南 |
| 数据分析 | 拉数据、分析、总结发现 |
| 演示文稿 | 结构化幻灯片、写内容 |
| 研究综合 | 收集来源、提取洞察、总结 |
| 会议议程 | 结构化话题、时间盒、加上下文 |
| 活动资产 | 写文案、简报、时间线 |

### 批次分组

按依赖关系将任务分组为批次：
- **Batch 1（并行）**：无依赖的独立任务
- **Batch 2（并行）**：依赖 Batch 1 产出
- **Batch 3（顺序）**：需要用户反馈

**分组规则**：
- 互相无依赖的任务放同批次
- 需要其他任务产出的放更后批次
- 需要用户反馈才能启动的单独批次

### 执行原则

- **用现有工具**：有 MCP 工具、API、技能就，不重建已有的
- **遵循项目约定**：检查 CLAUDE.md 中的风格指南、数据源、工具偏好
- **展示，不描述**：产出实际交付物，不是"我会创建一个……"
- **批次间审批，不逐个任务审批**：独立任务不需单独签字
- **不确定时回退顺序**：错误的并行比没有并行更糟

### 执行日志（关键要求）

每批次完成后**追加日志到计划文件**的 `## Execution Log` 部分：
```markdown
### [时间戳] Batch 1: [批次名]
- [Task 1] ✅ — 产出: [什么], 位置: [路径]
- [Task 2] ⏳ 阻塞 — [阻塞原因], 下一步: [需要什么]
```

这个日志为 `/kw:compound` 提供可提取学习成果的具体材料。

### 处理阻塞

- **缺少信息**：问用户，具体说明需要什么
- **缺少访问**：记下并继续下一个任务，解除阻塞后回来
- **范围蔓延**：比预期大时标记："这变成了独立项目，要继续还是单独做计划？"
- **质量担忧**：产出感觉不对时说出来

### 重要规则

- **产出，不计划**：执行模式，发现自己在写新计划就是走错了工作流
- **尊重范围**：计划定义了做什么，不在计划内的先问再加
- **追踪一切**：工作日志是知道发生了什么的方式，也喂给 compound
- **批次间反馈**：知识工作主观，每批次后检查而非全部产出后赌对

## Compound — 知识复利

**触发场景**：完成计划/活动/分析/策略会话后、"沉淀这个"、"保存学到的"、数据修正/流程修复/战略洞察后。

**核心定位**：闭环。提取学到的，保存到未来工作能自动找到的地方。

### 可沉淀的学习类型

| 类型 | 信号 |
|------|------|
| **Insight（洞察）** | "我们发现……"、意外发现、反直觉结果 |
| **Playbook（剧本）** | 有效的可重复流程、别人可遵循的步骤 |
| **Correction（修正）** | 错误假设修正、数据源澄清、定义更新 |
| **Pattern（模式）** | 反复出现的事物、系统性观察 |

**每会话最多 1-3 条学习**。质量胜于数量。

### 流程

**Step 1：识别学习**。扫描会话中可复合的洞察，为每条起草："学习：[一句话——我们现在知道什么]"、"类型"、"为什么重要"。

**Step 2：用户批准**。展示草稿，用户可以：批准、编辑措辞、跳过单条、补充你遗漏的。**未经批准不保存任何东西。**

**Step 3：查重**。用 grep 搜索 `docs/knowledge/` 和 `docs/solutions/`，找到相似条目时问"更新现有还是保存新的？"

**Step 3.5：过时知识检查**。启动 `compound-knowledge:research:stale-knowledge-checker`，传入新学习，返回可能被矛盾或取代的现有条目。

**关键要求**：代理只返回文本，不写不删文件。只有编排的 compound 技能写文件。

**Step 4：本地保存**。写入 `docs/knowledge/{descriptive-slug}.md`：
```markdown
---
type: [insight|playbook|correction|pattern]
tags: [关键词]
confidence: [high|medium|low]
created: [日期]
source: [来源描述]
---
# [标题]
[2-4 句话，具体到 3 个月后读的人能理解]
## Context → [发现时在做么]
## Implication → [如何改变未来工作，具体化]
```

### 重要规则

- **每会话 1-3 条上限**：保存 5 件事说明过滤不够
- **必须批准**：永远不自动保存
- **具体**："用正确的数据源"无用，"收入指标来自 [具体仪表盘]，不是 [其他来源] 后者多算约 $X" 有用
- **重复是浪费**：创建前永远检查，优先更新已有条目
- **信心标记**：基于一个数据点标 low，多次会话或数据验证标 high
- **标签为了检索**：选择 `/kw:plan` grep 搜索能匹配上的标签

## Workflow — 一键完整流程

kw:workflow 是编排器，串联全部子技能：

```
brainstorm → plan → [confidence?] → review → work → compound
```

**文件组织**：所有输出保存到 `plans/问题名-日期/` 下

| 步骤 | 技能 | 输出文件 |
|------|------|---------|
| 1 | Brainstorm | `00-brainstorm.md` |
| 2 | Plan | `01-research.md` |
| 3 | Confidence（可选） | `02-confidence.md` |
| 4 | Review | `03-review.md` |
| 5 | Work | `02-workspace/` |
| 6 | Compound | `docs/knowledge/` |

Confidence 仅在执行高风险决策或用户明确要求时触发。Workflow 是流程编排器，不自行执行子步骤，每步读取对应子技能的 SKILL.md 按指令执行。

### 使用场景对比

| 场景 | 用子技能 | 用 kw:workflow |
|------|---------|----------------|
| 快速头脑风暴 | `/kw:brainstorm` | 太重 |
| 只需要研究计划 | `/kw:plan` | 太重 |
| 完整分析复杂问题 | 需手动串联 | 一键完成 |
| 不确定从哪开始 | — | 自动推进 |

## Pipeline Mode 统一规范

所有 kw-* 技能在 `disable-model-invocation` 上下文（从编排器或自动化调用）下：
- 跳过所有 AskUserQuestion 提示
- 使用合理默认值
- 无需确认直接写文件
- 自动推进到下一建议技能
- 输出结构化结果供调用上下文解析
