## 概述

Mino 技能体系中的每个技能都经历了从发现、安装、探索到实战使用的完整生命周期。这些实践记录不仅反映了每个技能的实际使用效果，也揭示了技能管理的方法论——如何判断一个技能值得安装、安装后如何快速验证能力、如何在实际工作场景中使用和评估。本文从实战角度梳理技能安装与使用的全链路经验。

## 一、技能发现渠道

### 技能市场

技能安装前的第一步是搜索。主要通过技能市场（Tencent SkillHub、ClawHub、claw123.ai）获取候选技能列表，搜索流程：

1. curl 获取技能列表：`curl -s https://clawhub.ai/api/skills.zh.json`
2. 根据需求匹配 `name`、`description_zh`、`category_zh`
3. 获取 SKILL.md 源 URL

### 安装原则

- **按需安装，不批量装**：需要某个功能时才安装对应技能
- **推荐 1-3 个后确认**：搜索到候选后推荐给年老师，确认后再安装
- **安装后验证**：检查 SKILL.md 行数、文件数、文件大小是否正常

## 二、典型安装实战

### 案例 1：ppt-master（2026-04-09）

**安装动机**：需要一套完整的从文档到 PPT 的自动化管线。

**安装过程**：
- 从 GitHub 克隆完整仓库
- 安装 Python 3.10+ 依赖
- 安装可选依赖（Node.js、Pandoc）
- 运行 `pip install -r requirements.txt`

**安装规模**：
- 87 个相关文件（仓库根目录及 skills/ppt-master/）
- 7015 个文件总数（含图标库、模板库、示例项目）
- 30MB 总大小

**安装后验证**：
- 检查 SKILL.md 完整性（314 行）
- 验证 scripts/ 目录下的 11 个脚本是否可执行
- 确认 templates/ 目录下有布局模板、图表模板、图标库

**首次使用**：通过完整的 AI 定价研究项目验证了管线能力。

### 案例 2：presentation-skill（2026-04-16）

**安装动机**：需要为每日书信生成精华演示文稿。

**安装过程**：
- 将 SKILL.md 放入 `~/.claude/skills/presentation/`
- 重启 Claude Code 使技能生效

**安装规模**：
- 41 个文件
- 796K 总大小
- 452 行 SKILL.md

**首次使用**：将每日书信内容转化为可播放的 HTML 幻灯片，这是 presentation-skill 在实际工作场景中的典型应用——个人知识管理和分享，而非商业案例。

### 案例 3：mino-frontend（2026-04-01~02）

**安装动机**：需要年老师专属的前端设计技能，覆盖供应商管理等日常工作场景。

**安装后探索**：
- 安装了 54 个设计系统（11MB，2461 文件）
- 进行了风格情绪矩阵研究
- 建立了参数系统框架
- 制定了跨技能设计原则体系

**特点**：mino-frontend 不是一个"安装即用"的技能，而是需要深度定制和方法论建设的平台级技能。安装只是起点，后续的规则建设和方法论完善才是核心价值。

## 三、跨技能设计原则体系

在安装和使用多个前端设计类技能的过程中，发现了一个关键问题：同一个需求用不同技能会得到风格迥异的产出。为此建立了跨技能设计原则体系。

### 反 AI Slop 原则

拒绝 AI 生成内容的低质美学特征：

- 紫色渐变背景
- Emoji 作为图标
- 装饰性圆角卡片
- 无意义的装饰元素

### 克制优先原则

装饰是最后的手段，先用以下方式传达信息：

- 网格结构
- 排版对比
- 色彩对比
- 留白控制

### 品牌一致性原则

所有前端设计技能共享基础 token：

- 颜色：统一定义主色、辅助色、语义色
- 字体：统一定义标题、正文字体
- 间距：统一的间距层级

### 场景匹配原则

根据使用场景选择技能，而非随机：

| 场景 | 推荐技能 | 理由 |
|------|----------|------|
| 供应商管理看板 | mino-frontend | 企业级，信息密度高 |
| 营销落地页 | huashu-design | 高保真原型，交互 Demo |
| 技术架构图 | diagram-design | 专业图表，信息关系图 |
| 快速演示 | presentation-skill | 62 种品牌风格，一键生成 |
| 电影感界面 | cinematic-ui | 导演级艺术指导 |

## 四、技能使用的工作流集成

### kw-workflow 集成

kw-workflow（一键完整知识工作流）将技能使用嵌入到完整的知识工作流程中：

```
头脑风暴 → 研究 → 决策 → 故事板 → 设计规格 → 交付物 → 演讲脚本
```

在 AI 定价 PPT 项目中，kw-workflow 串联了多个技能：
- 头脑风暴 → kw-brainstorm
- 深度研究 → ultra-research
- 设计规格 → ppt-master Strategist
- PPT 生成 → ppt-master Executor / presentation-skill
- 演讲脚本 → presentation-skill

### 技能组合使用模式

实战中发现，单一技能很少独立完成复杂任务，更多时候是技能组合：

| 组合 | 使用场景 | 协同方式 |
|------|----------|----------|
| ppt-master + presentation-skill | 双版本演示文稿 | ppt-master 出 PPTX 版，presentation-skill 出 HTML 演讲版 |
| mino-frontend + impeccable | 高品质前端界面 | mino-frontend 生成，impeccable 做质量标准 |
| diagram-design + chart-visualization | 数据报告 | diagram-design 做关系图，chart-visualization 做数据图表 |
| kw-workflow + 多个技能 | 端到端知识工作 | kw-workflow 作为编排器，调度各领域技能 |

## 五、使用中的问题和经验

### 技能加载问题

在技能全量分析中发现了一些加载异常：

- **getnote**：SKILL.md 行数 0，可能存在安装问题或技能文件损坏
- **LearnFromHistory-skill**：268 文件/14M，体量异常，可能包含冗余数据

### 重叠技能的决策

当发现功能重叠时，决策流程：

1. 确认重叠程度（高/中/低）
2. 评估每个技能的独特定位
3. 检查实际使用频率
4. 基于数据决定保留、合并或清理

### 技能安装的注意事项

- **Python 依赖**：部分技能（如 ppt-master）需要 Python 环境和额外依赖，安装前确认
- **Node.js 版本**：部分脚本需要 Node.js 18+
- **存储空间**：大型技能（如 ppt-master 30M、mino-frontend 11M）占用空间较多
- **技能目录规范**：每个技能放在 `~/.claude/skills/{技能名}/` 下，SKILL.md 在根目录

## 六、技能生命周期管理

### 从安装到清理的完整周期

```
发现 → 评估 → 安装 → 验证 → 探索 → 实战 → 评估 → [保留/清理]
```

每个阶段都有明确的目标：

| 阶段 | 目标 | 检查项 |
|------|------|--------|
| 发现 | 找到候选技能 | 需求匹配度 |
| 评估 | 判断是否值得安装 | 与现有技能重叠度 |
| 安装 | 正确安装到技能目录 | SKILL.md 可加载 |
| 验证 | 确认基本功能正常 | 生成一个简单的测试输出 |
| 探索 | 了解能力边界 | 阅读 SKILL.md，查看示例 |
| 实战 | 在真实场景中使用 | 完成一个实际任务 |
| 评估 | 判断是否保留 | 使用频率、满意度、重叠度 |
| 保留/清理 | 决策执行 | 保留或卸载 |

### 维护建议

- **定期审计**：每季度检查一次技能列表，清理不再使用的技能
- **安装前查重**：安装新技能前先检查是否有功能重叠
- **记录使用日志**：每次使用技能都记录场景和满意度，为后续评估提供数据
- **异常监控**：关注技能的文件大小和文件数变化，异常增长时及时分析
