## 概述

2026-04-24 进行的设计技能大比武是 Mino 技能体系中最系统的一次能力对标活动。用同一份数据（8 家供应商、5 个指标的月度概览）、同一个场景（供应商管理月度汇报），让 5 个设计技能各自产出可视化结果，然后通过标准化的评估维度进行横向对比。这不仅是技术测试，更是技能体系自我进化的重要机制——通过客观对比，识别重叠、淘汰低效、保留优质能力。本文记录了测试方法论的完整框架，包括测试设计、评估维度、参与技能和分析方法。

## 一、测试设计原则

### 控制变量法

设计技能对比测试的核心方法论是控制变量——确保所有技能在相同条件下竞技：

| 变量 | 控制方式 |
|------|----------|
| 数据源 | 同一份供应商月度概览（8 家供应商，5 个指标） |
| 场景 | 供应商管理月度汇报 |
| Brief | 统一的测试需求和目标描述 |
| 输出格式 | 各技能各自推荐的格式（HTML/SVG/PPTX 等） |

**为什么要用同一份数据？** 因为如果每个技能用不同的数据，就无法区分"技能能力差异"和"数据复杂度差异"。同一份数据确保产出差异完全来自技能本身的能力差异。

### 测试就绪检查

在执行测试前，需要确认以下检查项全部通过：

| 检查项 | 内容 |
|--------|------|
| 测试 Brief 已定义？ | 供应商管理月度概览的具体需求描述 |
| 测试数据已准备？ | 8 家供应商，5 个指标的量化数据 |
| 参与技能已选定？ | 明确哪些技能参与及完整度评估 |
| Workspace 已创建？ | 独立的测试工作区，避免污染 |
| 评估维度已定义？ | 视觉/信息/实用/调整/主观满意度 |

## 三、评估维度体系

测试采用五维评估模型：

### 1. 视觉品质（Visual Quality）

评估视觉呈现的专业度和美观度：

- 色彩搭配是否和谐？有无"AI Slop"特征（紫色渐变、emoji 图标、装饰性圆角卡片）？
- 排版是否精致？有无信息层级？
- 整体视觉风格是否一致？
- 装饰元素是否服务于内容而非喧宾夺主？

### 2. 信息传达（Information Clarity）

评估信息传递的效率和准确度：

- 核心信息是否在 3 秒内可识别？
- 数据是否可读？关键数字是否突出？
- 供应商之间的对比是否清晰？
- 趋势和异常是否被有效标注？

### 3. 实用性（Practicality）

评估产出物在实际工作场景中的可用性：

- 能否直接用于供应商月度汇报？
- 是否支持交互（筛选、排序、钻取）？
- 导出能力如何（PDF/PPTX/图片）？
- 移动端是否可用？

### 4. 可调整性（Adjustability）

评估产出物的可定制和修改能力：

- 修改数据是否方便？
- 调整配色是否灵活？
- 增减内容是否破坏布局？
- 代码是否可读和可维护？

### 5. 主观满意度（Subjective Satisfaction）

评估使用体验的整体感受：

- 年老师的直观感受（满意/不满意/需要大改）？
- 与期望值的差距？
- 是否会复用到下一次？

## 四、执行方式

### 逐技能生成

测试采用逐技能生成模式：年老师每次说一个技能名，Mino 调用对应技能生成，产出保存到 `产出-{技能名}/` 目录。

这种执行方式的好处是：

1. **隔离性**：每个技能独立运行，不相互影响
2. **透明性**：每次生成过程可追溯
3. **可比性**：产出物结构一致，方便横向对比

### 评估记录表

每个技能生成后，按五维模型打分并记录：

| 技能 | 视觉 | 信息 | 实用 | 调整 | 主观满意度 | 备注 |
|------|------|------|------|------|-----------|------|
| diagram-design | - | - | - | - | - | 四象限定位图 |
| huashu-design | - | - | - | - | - | 待测 |
| presentation-skill | - | - | - | - | - | 待测 |
| mino-frontend | - | - | - | - | - | 待测 |
| html-ppt-skill | - | - | - | - | - | 待测 |

## 五、测试数据——供应商月度概览

测试使用的是真实的供应商管理月度数据，包含 8 家供应商和 5 个核心指标。选择真实数据而非模拟数据的原因是：

1. **真实性检验**：技能需要处理真实工作中的数据特征（缺失值、异常值、命名差异）
2. **场景贴近实际**：供应商月度汇报是年老师的日常工作场景，测试结果直接指导技能选择
3. **决策相关性**：评估结果会影响后续日常工作中技能的使用偏好

## 六、测试的方法论意义

### 技能体系的自我进化

设计技能对比测试不是"用完就丢"的一次性活动，而是技能体系自我进化的核心机制：

1. **识别重叠**：多个技能产出相似结果 → 考虑清理或合并
2. **发现缺口**：没有技能能做好某类呈现 → 需要新技能或扩展现有技能
3. **能力标定**：为每个技能建立"能力档案"，知道它在什么场景下表现最好
4. **使用决策支持**：下次遇到类似需求时，能直接推荐最合适的技能

### 从主观判断到客观评估

在没有标准化测试之前，技能选择依赖主观经验——"我觉得这个技能不错"。测试框架将评估过程从主观判断转为客观记录：

- 有据可查：每次评估都有记录
- 可回溯：可以对比不同时间点的评估结果
- 可迁移：评估维度可复用到其他技能对比测试

### 对后续技能清理的指导

2026-04-24 的技能大比武直接指导了后续的技能清理决策：

- PPT 类技能集群中，高度重叠的技能被标记为待清理
- 前端设计类技能通过对比明确了各自的独特定位
- 清理决策基于测试数据，而非主观感觉

## 七、测试框架的复用价值

这套评估框架不仅适用于设计技能，也可以扩展到其他技能领域的对比测试：

| 测试场景 | 评估维度调整 |
|----------|-------------|
| 文档处理技能对比 | 速度/准确/格式支持/编辑友好度/主观满意度 |
| 数据分析技能对比 | 图表类型/交互能力/数据密度/导出能力/主观满意度 |
| 浏览器自动化技能对比 | 稳定性/速度/错误处理/脚本灵活性/主观满意度 |

核心方法论不变——控制变量、多维度评估、逐技能执行、记录追踪——只需根据领域调整评估维度。
