## 概述

本文档整理了/plan5计划模式的完整最佳实践，基于交互式学习网页项目的真实复盘经验。文档涵盖了计划模式中最常见的五大失败原因及其解决方案，包括需求澄清不清、审美未对齐、验收标准不可检查、输出格式未验证、执行步骤不详细等。同时，文档整合了技能加载优化、技能集成计划、技能改进方案等相关内容，形成了从需求澄清到技能调用再到验证交付的完整方法论。核心洞察：plan5不是"写了计划就能成功"，而是"想清楚再行动"——关键不是计划文件本身，是逼自己回答三个问题：我到底要什么？我怎么确保理解对了？我怎么知道完成了？

## 一、项目复盘：两次尝试的对比

### 1.1 第一次尝试：失败

**输出结果**：深色主题HTML，比例失调，审美极差。

**失败原因**：
- 技能没理解需求，直接输出代码
- 计划文件中只说了"要什么"，没说"不要什么"
- "暖橙色/红色"被技能解读成深色背景配橙色强调
- 直接调用frontend-design技能，按默认理解执行（深色=专业开发者工具）

### 1.2 第二次尝试：成功

**输出结果**：明亮专业配色，比例正常，可用。

**成功关键**：
- 先明确要求，再让技能执行
- 先调用web-design-expert对齐方向
- 明确要求"参考Vercel、Linear、Raycast"
- 明确要求"明亮专业，不要深色强迫症"
- 然后再调用frontend-design执行

## 二、五大核心教训

### 2.1 教训一：计划文件要写清楚"不要什么"

**问题**：只说了"要什么"，没说"不要什么"。

**第一次计划的缺陷**：
```
配色方案：
- 主色：Claude品牌色（暖橙色/红色）← 模糊
- 辅色：中性灰、白色 ← 模糊
```

**改进**：明确排除项。
```
配色方案（明确排除）：
- ❌ 不要深色主题（#1a1a2e这种）
- ❌ 不要赛博朋克风格
- ✅ 要明亮专业：白底 + 浅灰 + 蓝色主调
```

**设计原则**：AI对"不要什么"的理解往往比"要什么"更准确。明确排除项是防止AI误解的最有效手段。

### 2.2 教训二：技能调用前要先对齐审美

**第一次**：直接调用frontend-design，技能按默认理解执行，输出后发现问题，已经浪费了。

**第二次**：先调用web-design-expert对齐方向，明确要求后再调用frontend-design执行。

**正确的技能调用顺序**：
```
审美对齐技能 → 设计执行技能 → 代码审查技能
     ↓              ↓              ↓
 web-design    frontend-design  frontend-patterns
```

**核心原则**：执行类技能需要明确的设计输入，而设计输入需要通过专门的审美对齐技能来生成。

### 2.3 教训三：验收标准要可检查

**第一次的验收标准**（模糊，主观判断）：
- 七大组件内容完整呈现
- 导航可快速跳转各章节
- 代码块有高亮效果

**改进后的验收标准**（可检查，客观标准）：
- 打开HTML文件，能看到7个组件卡片
- 点击卡片，有完成状态变化（绿色边框）
- 点击折叠标题，内容能展开/收起
- 代码块右上角有"复制"按钮
- 刷新页面，学习进度不丢失（LocalStorage）
- 提交测验，能看到分数和反馈

**设计原则**：验收标准必须是可以被第三方客观验证的，不能依赖主观判断。

### 2.4 教训四：文件写入前要先验证格式

**第一次的问题**：技能输出的HTML包含行号前缀（`1→`、`2→`），直接写入文件导致乱码。

**原因**：技能输出的是"带行号的展示格式"，不是纯HTML。

**改进方案**：
- 写入前先检查文件格式
- 或直接让技能输出到临时文件，检查后再移动
- 明确要求技能："请输出纯净的HTML代码，不要行号、不要解释文字"

### 2.5 教训五：计划中的"执行步骤"要更细

**第一次的执行步骤**（模糊，顺序错误）：
1. 用frontend-design设计主页面
2. 用frontend-patterns审查组件
3. 用web-design-expert优化视觉
4. 本地验证
5. 用户确认

**问题**：
- 步骤1和3顺序反了（应该先审美对齐再执行）
- 没有"检查输出格式"的步骤
- "本地验证"太模糊

**改进后的执行步骤**：
1. 调用web-design-expert：明确配色、字体、排版要求
2. 调用frontend-design：基于设计要求生成HTML
3. 检查输出格式：确认没有行号污染
4. 写入文件
5. 打开浏览器验证：比例是否正常、配色是否明亮、交互是否可用
6. 用户确认

## 三、/plan5正确使用流程

### 3.1 阶段一：需求澄清（最重要）

**必填清单**：
- 要什么（具体描述）
- 不要什么（排除项）
- 参考案例（URL或截图）
- 验收标准（可检查的清单）

**需求澄清模板**：
```
## 目标
创建一个明亮专业的学习网页，参考Vercel文档风格

## 排除项
- 不要深色主题
- 不要赛博朋克/游戏风格
- 不要固定侧边栏占超过300px

## 交付物
- 单文件HTML（内联CSS/JS）
- 文件位置：docs/learning/claude-code-learning.html

## 验收标准
- 打开文件，Hero区域背景是浅色渐变
- 侧边栏宽度280px，白色背景
- 主内容区最大宽度960px
- 代码块有深色背景和浅色文字
```

### 3.2 阶段二：技能调用策略

**技能调用顺序**：
1. web-design-expert（审美对齐）→ 输出：配色方案、字体系统、间距规范
2. frontend-design（设计执行）→ 输出：完整HTML代码
3. frontend-patterns（代码审查）→ 输出：改进建议
4. （可选）web-design-expert（视觉复核）→ 确认符合设计方向

**每个技能的提示词要点**：

web-design-expert：
- "参考[URL]的设计语言，给出配色、字体、间距方案"
- "明确排除：不要XXX风格"

frontend-design：
- "基于以下设计要求生成HTML：[粘贴设计方案]"
- "输出纯净代码，不要行号、不要解释"

frontend-patterns：
- "审查这个HTML的：语义化、可访问性、响应式"
- "指出需要改进的地方"

### 3.3 阶段三：验证与交付

**验证清单**：
```bash
# 1. 检查文件格式
head -10 file.html | od -c

# 2. 打开浏览器
open file.html

# 3. 交互测试
- 点击卡片是否标记完成
- 折叠内容是否展开
- 代码能否复制
- 测验能否提交
- 刷新后进度是否保存
```

**用户确认前准备**：
- 截图关键界面（Hero、组件卡片、测验）
- 列出已完成功能清单
- 说明已知问题（如果有）

## 四、技能加载优化与渐进式披露

### 4.1 技能加载的问题

大型技能文件（1500+行）每次加载全部内容，Token浪费严重。这是手动skills约90%时间被忽略的重要原因之一。

### 4.2 渐进式披露方案

**重构前**（1500+行单文件）：
```
SKILL.md
├── 核心概念（200行）
├── 基础用法（300行）
├── 进阶用法（400行）
├── 示例代码（500行）
└── FAQ（100行）
```

**重构后**（300-400行主文件 + 10-11个资源文件）：
```
SKILL.md（主文件，300-400行）
├── 核心概念 + 快速参考
└── 资源文件链接：
    ├── basics/（基础详解）
    ├── advanced/（进阶用法）
    ├── examples/（示例代码）
    └── faq.md（FAQ）
```

**效果**：Token效率提升40-60%。

### 4.3 主文件结构

```markdown
# [Skill Name]

> 一句话描述

## 快速参考

| 场景 | 命令/方法 | 说明 |
|------|----------|------|
| 场景1 | `xxx` | 说明1 |

## 核心概念（简洁）

1-2段话，不超过200字

## 基础用法（常见场景）

### 场景1
**提示词**：[模板]
**详细说明**：见 `resources/basics/scenario-1.md`

## 进阶用法（按需）
详见 `resources/advanced/`

## 示例代码
见 `resources/examples/`

## FAQ
见 `resources/faq.md`
```

## 五、技能集成计划

### 5.1 技能集成的挑战

多个技能之间可能存在冲突、重复或功能重叠。需要进行系统的集成规划。

### 5.2 集成策略

**第一步：功能映射**
- 列出所有已安装技能
- 标注每个技能的核心功能
- 识别功能重叠和冲突

**第二步：优先级排序**
- 按使用频率排序
- 按重要性排序
- 确定哪些技能是核心，哪些是辅助

**第三步：冲突解决**
- 功能重叠的技能，保留最适用的
- 配置冲突的技能，调整触发条件
- 不兼容的技能，考虑替换

**第四步：集成测试**
- 测试技能之间的交互
- 验证技能链的完整性
- 确保技能触发不冲突

## 六、技能改进方案

### 6.1 技能评估维度

| 维度 | 评估标准 |
|------|---------|
| 准确性 | 技能输出是否符合预期 |
| 效率 | 技能执行的Token消耗 |
| 可靠性 | 技能在不同场景下的稳定性 |
| 可用性 | 技能是否容易被正确使用 |

### 6.2 改进流程

1. 收集技能使用反馈
2. 分析失败案例
3. 识别改进点
4. 修改技能配置
5. 测试验证

### 6.3 持续优化

技能不是一次定义就完事的。需要：
- 定期审查技能的使用情况
- 根据实际使用反馈调整技能
- 删除不再使用的技能
- 添加新的技能来满足新需求

## 八、核心洞察总结

**plan5不是"写了计划就能成功"，是"想清楚再行动"。**

**关键不是计划文件本身，是逼自己回答三个问题**：
1. 我到底要什么？（需求澄清）
2. 我怎么确保技能理解对了？（审美对齐）
3. 我怎么知道完成了？（验收标准）

**这三个问题想清楚，成功率至少提高50%。**

**设计原则**：
- 明确排除项比描述期望更有效
- 审美对齐先于设计执行
- 验收标准必须可客观验证
- 输出格式验证先于文件写入
- 执行步骤要详细到每一步
