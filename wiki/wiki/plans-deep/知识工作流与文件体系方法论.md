## Overview

2026年4月下旬，年老师对`workspace/`、`archive/`、`docs/`、`plans/`、`memory/`五大目录的文件体系进行了系统性整理规划。这不仅是一次文件整理，更是一次知识管理体系的重构。核心决策：workspace是临时工作区（不入库），archive是归档区（分projects/ongoing/scattered三类），docs是长期知识资产，plans是计划与思考过程，memory是记忆与索引。整理过程遵循三区域文件生命周期管理原则，制定了R2（archive命名对齐）、R3（workspace内容提炼）、R4（docs重组）四大phase的迁移计划，并为供应商目录建立了"三层架构"文件体系。

---

## 一、五目录定位与生命周期

### 1.1 目录定位

| 目录 | 定位 | 生命周期 | 入库规则 |
|------|------|---------|---------|
| **workspace/** | 临时工作区 | 项目结束后清理 | 不入库，有价值的迁移到docs/memory |
| **archive/** | 归档区 | 永久保留 | 已完成的项目/持续追踪/散点讨论 |
| **docs/** | 长期知识资产 | 持续更新 | SOP、方法论、制度、规范 |
| **plans/** | 计划与思考过程 | 完结后归档或删除 | 正在进行中的计划和过程文档 |
| **memory/** | 记忆与索引 | 持续更新 | 核心记忆、洞察、索引、待办 |

### 1.2 文件流转路径

```
workspace/（临时工作）
    ↓ 有价值 → docs/（知识资产）或 memory/（记忆索引）
    ↓ 完成 → archive/（归档）

plans/（进行中的计划）
    ↓ 完成 → archive/ 或 删除
    ↓ 产出知识 → docs/

docs/（知识资产）
    ↓ 过时 → archive/（历史版本）

memory/（记忆索引）
    ↓ WAL协议持续更新
```

## 二、archive/ 归档分类体系

### 2.1 三类归档

| 类型 | 特点 | 存放位置 | 命名格式 |
|------|------|----------|----------|
| **项目型** | 有起点终点，做完归档 | `archive/projects/` | `YYYY-MM-项目名` |
| **持续型** | 长期追踪，不断更新 | `archive/ongoing/` | `项目名`（无日期） |
| **散点型** | 零散讨论，不一定有后续 | `archive/scattered/` | `YYYY-MM-DD-主题` |

### 2.2 合并规则

- 同主题跨周项目型合并，不保留周度层级
- 同主题散点合并，不保留周度层级
- 合并后使用最早的日期作为归档名

### 2.3 命名规范

| 规范 | 正确示例 | 错误示例 |
|------|---------|---------|
| 日期前缀 | `2026-03-供应商站点看板` | `供应商站点看板-2026-03` |
| 主题清晰 | `2026-04-权责不对等分析` | `2026-04-分析` |
| 内部编号 | `00-brainstorm.md`、`01-research.md` | `brainstorm.md` |

## 三、workspace/ 内容提炼策略

### 3.1 扫描分类法

对workspace/中的48个文件夹逐个扫描，标记内容类型：

| 内容类型 | 迁移目标 | 示例 |
|---------|---------|------|
| 文档类 | docs/ | SOP、模板、指南 |
| 洞察类 | memory/insights.md | 分析结论、行为模式 |
| 技能类 | .claude/skills/ | 技能配置、工具规则 |
| 临时类 | 删除 | 测试输出、调试文件 |

### 3.2 提炼清单（R3）

提炼清单定义了从workspace到docs/memory的映射规则：

- 文档类制定迁移到docs/的映射表
- 洞察/评价类制定迁移到memory/的映射表
- 执行迁移后验证：有价值内容已分流，workspace/只剩临时工作

## 五、docs/ 重组原则

### 5.1 设计原则

1. **一级目录≤6个**：避免层级过深
2. **无重复主题**：每个文档只属于一个目录
3. **命名统一**：中文命名，日期格式统一
4. **可发现性**：通过目录名就能判断内容类型

### 5.2 推荐的一级目录结构

| 目录 | 用途 | 示例内容 |
|------|------|---------|
| `SOP/` | 标准操作流程 | 供应商准入流程、清退流程 |
| `方法论/` | 分析方法论 | 评分模型、风险评估方法 |
| `制度/` | 制度规范 | 供应商管理制度、考核制度 |
| `模板/` | 常用模板 | 评估报告模板、约谈通知模板 |
| `培训/` | 培训材料 | 质检培训、合规培训 |
| `参考/` | 参考资料 | 行业报告、竞品信息 |

## 六、plans/ 目录规范

### 6.1 生命周期

| 阶段 | 处理方式 |
|------|---------|
| 进行中 | 保留在plans/ |
| 已完结 | 归档到archive/projects/或删除 |
| 无后续 | 直接删除 |

### 6.2 命名规范

**文件夹命名**：
- 格式：`YYYY-MM-DD-主题描述`
- 主题要具体清晰，一看就知道是什么
- 禁止模糊命名（如"工作计划"、"分析"、"设计"）

**内部文件命名**：
- `00-brainstorm.md` — 头脑风暴，自由探索
- `01-research.md` — 研究分析，收集信息
- `02-workspace/` — 工作产出，临时文件
- `03-plan.md` — 执行计划（可选）
- `04-context.md` — 上下文记录（可选）

## 七、方法论总结

### 7.1 知识管理体系的核心原则

1. **临时与核心分离**：workspace是临时的，其余是核心的。临时区定期清理，核心区持续维护。
2. **流转单向**：从临时到核心是单向的。一旦进入docs/memory，就获得了"长期资产"地位。
3. **归档有分类**：不是所有完成的东西都扔到一个目录。项目型、持续型、散点型各有归属。
4. **命名即索引**：好的命名本身就具备索引功能。`YYYY-MM-DD-主题`的格式让人一眼就能判断内容。

### 7.2 可迁移到个人知识管理的规律

- **三区分法**：临时/核心/归档的三分法适用于任何知识管理系统
- **流转规则**：定义清楚什么进什么出，否则系统会膨胀到无法维护
- **命名规范**：统一的命名规范比复杂的分类系统更有效
- **定期清理**：workspace的临时性意味着必须定期清理，否则就失去了"临时"的意义
