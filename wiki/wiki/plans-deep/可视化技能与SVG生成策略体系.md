## 概述

2026年4月，年老师完成了两项可视化技能的系统性升级。第一项是Quiver API（arrow-preview模型）的高质量SVG生成策略——在每周20次配额限制下，通过prompt结构优化、references参数使用技巧、批量vs单次生成策略，生成东京猫/李诞虾级别的精细手绘SVG。第二项是chart-visualization和echarts-visualization两大技能的"分析先行"工作流改造——从"选图→出图"的简单模式升级为"分析→确认→出图→注解"的完整工作流，包含14种分析目标、主题色palette传递、数据分析前摇等核心改进。这两项升级共同构建了一个从AI生成SVG到ECharts数据可视化的完整可视化技能矩阵。

## 一、Quiver API高质量SVG生成策略

### 1.1 API核心能力与约束

**核心约束**：
- 每周20次请求（硬约束）
- 每次1 credit（包括n>1的情况）
- Rate limit: 20 requests/60 seconds
- 每周约3次/天

**模型能力**：
- `arrow-preview` — Quiver旗舰SVG生成模型
- context_length: 131072
- 支持streaming（Server-Sent Events）
- 输入：prompt（必需）+ instructions（可选）+ references（可选，最多4张）+ n（1-16）

### 1.2 高质量Prompt的构成要素

**从东京猫/李诞虾反推的特征**：

| 特征 | 东京猫 | 李诞虾 |
|------|--------|--------|
| path元素数 | ~50个 | 更多 |
| 线条技法 | stroke-linecap="round" | 双层技法更明显 |
| 色彩系统 | 陶土#BA3420 + 墨黑#0B0800 | 陶土色点缀 |
| 图层 | 填充层+线条层 | fill layer + ink line layer |
| 特殊效果 | - | feTurbulence纸张纹理 |

**Prompt结构公式**：

```
prompt = [主体描述] + [风格描述] + [构图说明] + [细节要求]

示例：
"Japanese organizational chart showing power structure,
策略组 on left with pricing authority, 服务组 on right carrying risk,
供应商 in middle receiving conflicting signals.
Hand-drawn ink style with vermillion accents,
双层技法: fill layer for color weight + ink layer for fine outlines.
Use #BA3420 for accents, #0B0800 for lines, #F7F3ED for background.
约50个path elements, stroke-linecap round."
```

### 1.3 References参数使用技巧

| 策略 | 用法 | 适用场景 |
|------|------|---------|
| **风格参考** | 东京猫SVG + 李诞虾SVG | 需要统一手绘风格 |
| **内容参考** | 类似的组织架构图 | 需要特定构图 |
| **混合策略** | 风格参考 + 内容参考 + 色彩参考 | 最全面的质量控制 |

**关键待验证问题**：
- references接受URL还是base64？
- n>1时是变体还是重复？
- instructions对中文prompt支持如何？

### 1.4 配额分配策略

| 日期 | 用途 | 配额 |
|------|------|------|
| 周一 | 探索性测试（n=3，快速看方向） | 3次 |
| 周二-周四 | 精细生成（n=1，迭代优化） | 每天2次 |
| 周五 | 最终版本（n=1-2，确保质量） | 3次 |
| 预留 | 应急/重试 | 4次 |

**关键原则**：
- 每次调用前必须充分准备（prompt + references）
- 失败立即分析原因，不要盲目重试
- 记录每次参数和结果，形成知识库

### 1.5 色彩系统规范

**统一色彩**（适用于所有权责不对等研究内容的可视化）：

| 角色 | 色值 | 用途 |
|------|------|------|
| 陶土色（重点） | `#BA3420` | 强调、警示 |
| 陶土色（辅助） | `#E2725B` | 辅助高亮 |
| 米白（背景） | `#F7F3ED` | 页面背景 |
| 米白（深色） | `#F5F1EE` | 卡片背景 |
| 墨黑（线条） | `#0B0800` | 线条、轮廓 |
| 墨黑（文字） | `#111111` | 正文文字 |
| 灰度 | `#D4CFC7`、`#A8A098` | 辅助元素 |

## 二、图表技能改造：分析先行工作流

### 2.1 改造前 vs 改造后

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 工作流 | 选图→出图 | 分析→确认→出图→注解 |
| 分析前摇 | 无 | 步骤0：数据分析前摇 |
| 分析目标 | 不清晰 | 14种分析目标库 |
| 主题色传递 | 无 | palette从分析阶段传递到出图阶段 |
| 注解 | 无 | 自动生成图表注解 |

### 2.2 chart-visualization重写方案

**5步完整工作流**：

1. **步骤0：数据分析前摇**
   - 识别数据类型（时间序列/分类/关系/分布/组成）
   - 分析数据特征（极值、缺失、分布形态）
   - 确定分析目标（对比/趋势/占比/关联/异常）

2. **步骤1：图表推荐**
   - 根据数据特征和分析目标，推荐最佳图表类型
   - 提供2-3个备选方案及权衡分析

3. **步骤2：确认**
   - 与用户确认图表选择
   - 确认配色、标注方式

4. **步骤3：出图**
   - 生成图表
   - 传递分析阶段的palette

5. **步骤4：注解**
   - 自动生成图表注解（趋势说明、异常标注）
   - 提供解读建议

### 2.3 14种分析目标库

| 分析目标 | 推荐图表 | 关键特征 |
|---------|---------|---------|
| 时间趋势 | 折线图/面积图 | 连续数据、时间轴 |
| 类别对比 | 柱状图/条形图 | 离散数据、排序 |
| 占比分析 | 堆叠条形图/瀑布图 | 避免饼图 |
| 关系分析 | 散点图/气泡图 | 两个变量、相关性 |
| 分布分析 | 直方图/箱线图 | 数据分布、异常值 |
| 层级关系 | 树图/旭日图 | 层级结构、嵌套 |
| 流程分析 | 桑基图/漏斗图 | 流向、转化率 |
| 地理分析 | 地图/热力地图 | 地理维度、区域对比 |
| 矩阵分析 | 热力图 | 两个分类维度的交叉 |
| 异常检测 | 控制图/散点图 | 离群值识别 |
| 排名分析 | 水平条形图 | 排序、TOP N |
| 目标达成 | 子弹图/仪表盘 | 实际vs目标 |
| 多维对比 | 雷达图/平行坐标 | 多个维度同时对比 |
| 组成分析 | 堆叠面积图 | 随时间变化的组成 |

### 2.4 echarts-visualization手术刀编辑

**核心改动**：
- 保持原7步工作流不变
- 在前面加步骤0：数据分析前摇
- 增加主题推荐度标记（每种图表类型的推荐场景标记）
- 加固配色体系（与13-VISUALIZATION.md三色系统对齐）

### 2.5 改造核心决策

**合并 vs 各自增强**：选择不合并，各自增强。

| 理由 | 说明 |
|------|------|
| 定位不同 | chart-visualization侧重分析决策，echarts-visualization侧重技术实现 |
| 用户场景不同 | chart用于"选什么图"，echarts用于"怎么画好" |
| 互补关系 | 分析先行→ECharts实现，两个技能形成工作流上下游 |

## 三、可视化技能矩阵

### 3.1 技能定位与分工

| 技能 | 定位 | 适用场景 | 输出格式 |
|------|------|---------|---------|
| **Quiver API** | AI生成手绘SVG | 信息图、隐喻图、人物插画 | 单SVG文件 |
| **chart-visualization** | 分析先行选图 | 数据可视化方案推荐 | 图表+分析报告 |
| **echarts-visualization** | ECharts技术实现 | 桑基图/K线图/热力图等复杂图表 | 独立HTML |
| **generative-ui-widget** | 交互式可视化 | 流程/关系/对比/数据 | HTML fragment |
| **diagram-design** | 技术图表 | 架构图、流程图、序列图 | SVG/HTML |
| **mino-frontend** | 前端页面 | 数据看板、演示文稿、通报 | 单HTML文件 |

### 3.2 技能协作流程

```
数据分析 → chart-visualization（选图推荐）
    ↓
ECharts实现 → echarts-visualization（复杂图表）
    ↓
页面整合 → mino-frontend（整体页面+数据卡片+文本组件）
    ↓
装饰元素 → Quiver API（手绘SVG插图）/ diagram-design（技术图表）
```

### 3.3 统一视觉语言

所有可视化技能遵循共同的视觉语言（来自13-VISUALIZATION.md）：

- **三色核心**：主题红`#C13531` + 深蓝`#293C54` + 纯灰`#CDCECD`
- **禁止**：渐变、阴影、发光、3D效果、默认9色轮播
- **风格**：克制专业主义，信息密度适中，像精心排版的PDF

## 四、方法论总结

### 4.1 Quiver API使用的心智模型

把Quiver API当成"有限配额的高级设计师"：
- 每周20次 = 有限的设计资源
- 每次调用前必须充分准备 = 好的brief才能出好设计
- 失败立即分析 = 不盲目重试，要理解失败原因
- 记录参数和结果 = 形成知识库，持续优化

### 4.2 分析先行的核心价值

"分析先行"的本质是：**图表是分析的结果，不是分析的替代品**。

- 选图前先理解数据特征和分析目标
- 图表选择由分析需求驱动，而非视觉效果驱动
- 注解是分析的延伸，帮助读者理解图表背后的含义

### 4.3 可迁移的规律

- **配额约束催生质量意识**：有限的API配额逼迫每次调用都追求最高质量
- **分析先于可视化**：任何可视化都应该有明确的分析目标，否则就是装饰
- **技能分工优于技能合并**：chart和echarts不合并，各自增强，形成工作流上下游
