01

What it is.

task-handoff 解决单任务交接。Gabriel Chua 解决长周期路线管理。GDD 把两者合在一起，形成五层叠加架构。

L0 · Project 目标锚点
PROJECT.md · 整个项目唯一的真相源
Goal / Milestones / Decisions / Blockers。跨 session 不丢失。所有任务从这里继承上下文，不再手抄。
PROJECT.md
↓　引用

MILESTONE-AUDIT.md · 里程碑边界的停车检查
每个里程碑完成后停、看全局、做 5 项校验。合格才推进。不合格修正路线或回退。
MILESTONE-AUDIT.md

02

How it works.

四个阶段往复循环。每阶段有明确触发条件和产出，不靠 AI 记忆靠文件系统。

01
建项目
理解需求 → 定 Goal → 拆 Milestones → 写 PROJECT.md
→ PROJECT.md
02
下发任务
从 PROJECT.md 摘上下文 → 写 PROMPT.md → 吐 prompt → 给执行方
→ PROMPT.md
03
验收任务
读 DONE.md → 对照 PROMPT 验证方式逐条 → accepted / rejected
→ accepted ✓
04
里程审计
本里程碑全部 accepted → 停、5 项校验 → 推进 / 修正 / 回退
→ AUDIT.md
03

Four deliverables.

每种文件都有固定结构和字段。不是文档，是工程接口——人和 AI 之间的机器可读协议。

01
PROJECT.md
持久目标锚点
整个项目唯一的真相源。每次 session 开始时第一个读的文件。
Goal一句话，不超过 30 字
Milestones按依赖顺序，每个有「完成定义」
Active同一时刻只有一个活跃里程碑
Decisions关键取舍，防止重复讨论
Blockers显式声明，不让执行方猜
02
PROMPT.md
单任务下发约束
发给执行方的唯一指令文件。没有完成定义不下发。背景摘录进来，执行方不需要读任何外部文件。
scope允许改动的路径，结构化约束
完成定义逐条可勾选的客观标准
验证方式执行方提前知道验收标准
背景摘录从 PROJECT.md 抄仅需要的
pending → dispatched → submitted → accepted
03
DONE.md
执行方回写 / 证据回报
不是声称做了，而是证明做了。每条验证方式都要有执行结果。
success全部验证通过，产出完整
partial大部分完成，minor 缺陷已声明
failed未达到完成定义
验证过程每条的执行结果，不是声称
04
MILESTONE-AUDIT.md
路线图审计 / 5 项校验
里程碑边界的停车检查。阻止长期推进中的路线漂移。
①完成定义是否仍成立？
②下一个目标是否合理？
③有没有遗漏步骤？
④新证据是否要重排顺序？
⑤验证描述是否可靠？

--- task: extract-framework scope: output/m1-extract/ type: exploratory --- ## 目标 读原文 → 提取 5 习惯的结构化信息 + 三维归类 ## 完成定义 - [ ] 全覆盖，每项「核心动作 / 细节 / 亮点」 - [ ] 三维归类（锚定 / 验收 / 外置）互不重叠 - [ ] 输出可独立读，不回看原文 ## 验证方式 1. grep 5 项关键字 → 全部命中 2. 三维归类交集 → 空 3. 打开验证独立性 ## 背景 Gabriel Chua, OpenAI Codex DX 工程师。 5 个习惯让 AI 长任务不跑偏。 核心锚点：目标锚定 / 证据验收 / 状态外置。

07

Three anchors.

所有设计决策从这三条推导出来。

Anchor 01
目标锚定
AI 的对话上下文是会消亡的。PROJECT.md 是不随 session 切换而丢失的持久锚点。每个任务从这里继承上下文，不靠 AI「记住」项目。
Anchor 02
证据验收
DONE.md 不只是 status: success —— 它必须是可核验的证据链。验收时只看证据，不看声称。里程碑审计是项目级第二道证据关。
Anchor 03
状态外置
全局状态不应该藏在 thread 对话历史里。它必须在文件系统里、在仪表盘里——任何人任何时候打开都能零成本重建。
"AI 模型负责产出智能能力。人类开发者的核心价值是引导这份智能始终锚定统一目标，沿着验证过的证据稳步推进。"
— 改编自 Gabriel Chua, OpenAI Codex DX
