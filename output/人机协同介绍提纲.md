## 第一部分：真实案例（8 分钟）

### 案例背景

**项目**：优化 haglofs-paradigm 技能（从 451 行编辑器页面生成技能 → 可视化场景支持）

**痛点**：
- 当前技能只支持 editorial 页面（品牌页、长文、报告）
- 不支持纯数据可视化场景（dashboard、图表、infographic）
- 用户想把它用在更多可视化场景，但能力不够

**约束**：
- 不能破坏现有的 editorial 能力
- 需要保持 Haglöfs 范式的视觉一致性
- 需要与 nian-design 等其他技能配合

### 用 orchestrate-projects 怎么做的

**Step 1：采访式启动**（不是直接写代码）

```
采访维度：
1. 现状：haglofs-paradigm 451行，7种Hero×56组件，但只支持editorial
2. 动机：想用在可视化场景，但当前不具备
3. 团队：只有我一个人
4. 目标态：支持dashboard/图表/infographic，保持范式一致性
5. 约束：不能破坏现有能力
```

**Step 2：创建 GOALS.md 路线图**

```
Milestone 1: 能力诊断（2天）
- 预期结果：明确哪些能力缺失，哪些可以复用
- 证据：能力矩阵文档

Milestone 2: 组件扩展（5天）
- 预期结果：新增数据可视化组件集
- 证据：组件通过测试

Milestone 3: 集成验证（3天）
- 预期结果：用真实数据跑通一个dashboard
- 证据：dashboard页面可访问
```

**Step 3：委派与协调**

```
主线程（我）：协调、决策、验收
Worker 1：分析现有组件，输出能力矩阵
Worker 2：研究数据可视化最佳实践
Worker 3：实现新组件（串行依赖 Worker 1 的结果）
```

**Step 4：里程碑审计**

```
Milestone 1 完成后审计：
- 能力矩阵是否完整？→ 是
- 下一个目标是否正确？→ 需要调整（发现需要先解决数据源问题）
- 更新 GOALS.md，插入新的 Milestone 1.5
```

### 案例结果（预期）

- 技能从 451 行扩展到 ~600 行
- 新增 12 个数据可视化组件
- 成功生成一个 dashboard 页面
- 整个过程有完整的决策记录和证据链

### 3A 分诊法（现场演示）

```
接到一个任务 → 30秒判断：

Auto（自动化）：
- 重复性高、规则明确
- 例：格式转换、数据清洗

Augment（增强）：
- 需要人的判断，AI辅助
- 例：写文档、做分析

Agent（代理）：
- 复杂、长期、需要协调
- 例：项目重构、多模块开发
```

### orchestrate-projects 的核心循环

```
对话 → 路线图 → 激活里程碑 → 委派工作 → 验证结果 → 更新路线图 → 下一个里程碑
     ↑_____________________________↓
            （新发现时修订计划）
```

**三个防漂移锚点**：
1. **GOALS.md** — 共享路线图，所有决策都记录在这里
2. **Goal 模式** — 一次只有一个活跃目标，不分散
3. **里程碑审计** — 每个阶段结束都验证，不盲目推进
