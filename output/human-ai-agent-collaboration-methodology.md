## 第一部分：分工框架——哪些交给AI，哪些留给人

### ★ 3A分诊法：30秒决定每个任务怎么分配 【🆕anysearch新发现】

**来源**：人人都是产品经理 https://www.woshipm.com/ai/6412361.html

**核心思路**：每接到一个新任务，先扔进三个"抽屉"之一。这个动作熟练后，30秒就能决定。

| 模式 | 定义 | 人做什么 | AI做什么 | 典型场景 |
|------|------|---------|---------|---------|
| **Automation（自动化）** | AI独立完成确定性任务，你不用盯，结果好验证 | 验收结果 | 全流程执行 | 格式转换、数据清洗、日报汇总成周报 |
| **Augmentation（增强）** | AI给信息给建议，但拍板的是人 | 最终决策 | 提供选项和利弊分析 | 竞品调研摘要、要不要做某个需求 |
| **Agent（智能体）** | AI自主规划执行多步任务，你只设边界 | 设边界和约束 | 拆步骤+调工具+执行 | 竞品调研+出对比报告全流程 |

**三个分诊维度**：

1. **确定性**：规则有多确定？输出好不好验证？→ 越确定越往Automation推
2. **可逆性**：万一做错了，能不能撤回来？→ 越不可逆越要留在Augmentation，人保留决策权
3. **任务跨度**：单步还是多步串联？→ 跨度大的多步任务适合Agent模式

**一个精妙的例子**（发产品更新邮件）：
同一件事拆成三段——更新日志整理成几条人话（Automation，规则清楚好验证）→ 语气该热情还是克制、该不该提灰度功能（Augmentation，AI给两版建议我拍板）→ 真正点"发送"发给哪个分组（人自己来，AI碰都不碰）。**同一封邮件，三段活儿分属三个抽屉。**

> **关键洞察**：把整件事一股脑丢给全自动Agent，它大概率会用一封措辞得体、却把灰度功能也写进去的邮件把你坑了。

### 学术支撑：A²C框架 + 人类能动性量表 【🆕补充】

- **A²C框架**（学术界）：Automation、Augmentation、Collaboration 三级，与3A分诊法对应。
- **Human Agency Scale（人类能动性量表）**（斯坦福SALT Lab）：用五个级别刻画一件任务里人该保留多少话语权——"该让AI多自主"是可以分级的滑块，不是开关。
- **研究发现**：工人巴不得自动化掉的是低价值重复的体力活；坚决想自己留着的是高判断、高情感投入的部分。**人的直觉在做分诊：可逆的、低判断的推出去，不可逆的、要担责的攥在手里。**

## 第二部分：模式选择——Tool / Copilot / Agent 五阶段演进

### ★ 五阶段自主性光谱（完整版）【🆕anysearch提取全文】

**来源**：The Thinking Company https://thinking.inc/en/blue-ocean/ai-native/copilot-to-agent-transformation/

```
Stage 1  Autocomplete  → AI预测片段，人选择（延迟<200ms）
Stage 2  Copilot       → AI生成完整建议，人审查采纳
Stage 3  委托执行       → AI执行完整任务，人审批结果
Stage 4  监督自主       → AI在边界内自主运行，人监控异常
Stage 5  完全自主       → AI自主发现任务并执行，人设目标
```

**完整对比矩阵**（这张表是决策的核心工具）：

| 维度 | Stage 1 自动补全 | Stage 2 Copilot | Stage 3 委托执行 | Stage 4 监督自主 | Stage 5 完全自主 |
|------|-----------------|-----------------|-----------------|-----------------|-----------------|
| **AI角色** | 建议片段 | 建议完整动作 | 执行任务 | 在策略内运行 | 自主决策 |
| **人角色** | 做工作 | 审查草稿 | 审批结果 | 看仪表盘 | 设目标 |
| **生产力倍数** | 1.2-1.3x | 1.3-1.5x | **2-5x** | **5-20x** | 20-100x |
| **信任要求** | 低 | 中 | 高 | 很高 | 极高 |
| **单任务成本** | $0.001-0.01 | $0.01-0.05 | $0.10-1.00 | $0.05-0.50 | $0.10-1.00 |
| **错误容忍度** | 高（人纠正） | 高（人审查） | 中（人审批） | 低（有限监督） | 极低（无监督） |

**阶段跃迁的关键**：

- **Stage 2→3（最关键的跃迁）**：从"建议"到"执行"。需要：工具集成 + 迭代推理能力 + 质量自评 + 审批界面设计。"这个转变的产品价值主张不再是'AI帮你编码'，而是'AI编码，你审结果'。"
- **Stage 3→4（需要3-6个月积累）**：从逐任务审批到自主运行。需要：可靠性数据积累 + 策略框架 + 监控基础设施 + 事件响应流程 + 渐进式扩范围。
- **Stage 4→5（目前对大多数产品来说还不到）**：需要AI在判断力、长期规划、自我评估方面的突破。

**⚠️ 跳阶段的代价**：
跳过Stage 3直接部署Stage 4（监督自主）的组织，**事故率高出2.7倍**（McKinsey, 2025）。超过40%的agentic AI项目将在2027年前被取消，因为组织试图跳过成熟度阶段。

### Copilot的天花板效应 【🆕补充·关键数据】

- Copilot阶段的结构性限制：**每个AI产出都需要人审查，形成瓶颈，生产力上限被卡在30-50%**。
- 突破天花板必须进入Stage 3（委托执行）：AI处理所有中间步骤，人只在最终审批。
- SWE-bench数据：Claude Sonnet 4解决了72.7%的真实GitHub issue——意味着每10个委托任务约7个无需人工纠正就能完成。

## 第三部分：落地实操——怎么给AI下指令、怎么验收

### 3.1 AGENTS.md：给AI的"操作手册" 【🆕anysearch提取全文 + 📚已有笔记呼应】

**来源**：https://qubittool.com/zh/blog/agents-md-best-practices + 你笔记里的Codex AGENTS.md范式(7/12)

**核心定位**：README.md是项目的"门面"（给人看），AGENTS.md是项目的"操作手册"（给AI看）。它通过机器友好的Markdown格式，以极低Token成本让AI"瞬间入职"。

**四层结构**：

```
AGENTS.md
├── 核心上下文（技术栈快照 + 项目架构说明）
├── 运行指南（Build/Test命令 + 环境变量要求）
├── 质量约束
│   ├── Do's: 必须遵循的规则
│   └── Don'ts: 严格禁止的事项 ★最重要
└── 动态演进（随项目迭代更新）
```

**五条最佳实践**：

1. **机器友好的Markdown格式**：多用表格、列表、代码块。AI对结构化数据处理远优于散文。
2. **用强语气词**：`Must`/`Never`/`Always` 的效果远好于 `Should`/`Recommend`。
3. **提供具体的代码正误对比**：展示❌错误 vs ✅正确，是最高效的沟通方式。
4. **单一职责**：不要在AGENTS.md里写详细业务逻辑，只关注工程规范。业务逻辑让AI通过阅读代码获取。
5. **Monorepo用嵌套模式**：每个子package下放一个AGENTS.md，主流AI工具自动检索最邻近配置。

**防坑指南**：
- ❌ 文件超过200行 → AI注意力分散
- ❌ 信息过时（更新了依赖版本却没更新AGENTS.md）
- ❌ 含糊其辞（写"写高质量代码"是废话，应写"函数必须包含JSDoc注释并标注类型"）
- ❌ 忘记安全约束（必须明确写"禁止将API Key写在代码里"）

## 第四部分：Agent工程化——最小闭环到多Agent

### 4.1 Agent最小闭环 【🆕anysearch提取全文】

**来源**：https://www.promptnet.cn/2026/06/17/ai-agent-practical-guide-2026/

```
User Task → Plan → Tool Call → Observation → Revise Plan / Finish
```

**验收标准（demo能继续投入的四条底线）**：
1. 同一输入连续跑5次，关键步骤一致
2. 工具失败时不会无限重试
3. 每一步都有日志，能看出为什么走到当前结论
4. **人类可以在关键节点中断或接管**

> 达不到这四条，就先别加更多工具。

### 4.2 三种架构选择 【🆕补充】

| 架构 | 适合 | 关键原则 |
|------|------|---------|
| **单Agent** | 80%任务（表单处理、数据抽取、内容QA、代码小修） | 最稳，状态简单日志清晰 |
| **Planner-Executor** | 复杂任务（代码重构、多网页调研、批量内容） | 职责隔离：Planner不直接改文件，Executor不改目标 |
| **多Agent** | 天然可并行任务（各审一个模块/各查一组资料） | 最后再上！多个Agent频繁共享状态会复杂度飙升 |

### 4.3 工具调用设计原则 【🆕补充】

| 原则 | 说明 |
|------|------|
| **schema要窄** | 不要给AI一个泛化command接口，而是具体到file_path+old_string+new_string |
| **权限分层** | 只读（默认允许）/ 本地可逆（按规则允许）/ 外部可见（必须确认） |
| **幂等** | 同样参数跑两次不破坏系统（写临时文件 > 直接覆盖生产配置） |
| **超时** | 每个工具必须有超时，否则Agent会消耗完整上下文 |

### 4.4 Cursor Team Kit：生产级团队的真实工作流 【📚已有笔记·完整拆解】

**来源**：你笔记《Cursor Team Kit深度解析》(5/12)

Cursor团队自己怎么用Cursor写Cursor——17个skill、1个sub-agent、2条rule构成的工作循环：

```
开发阶段：new-branch-and-pr（起手）→ check-compiler-errors（边写边自查）
评审阶段：review-and-ship（自检）→ pr-review-canvas（reviewer看快）→ get-pr-comments（收反馈）
CI阶段：loop-on-ci → fix-ci（红的修到绿）
收尾阶段：verify-this（证明bug修好了）→ deslop（擦干净AI写脏的代码）
汇报阶段：what-did-i-get-done → weekly-review（输出对外状态）
```

> **最有价值的skill是verify-this**：很多agent写完代码会自信地说"已经修好了"，然后你跑一下发现没修好。verify-this强制AI自己跑验证。

## 第六部分：核心洞察综合

### 6.1 人机协同的"黄金法则"（综合所有资料）

1. **先分诊再动手**：用3A分诊法30秒判断每个任务是Automation/Augmentation/Agent
2. **规则先行**：AGENTS.md/规则文件是ROI最高的投入——决定AI使用上限的不是单句Prompt精巧度，而是能否把需求、习惯、评判标准沉淀为AI可反复调用的长期档案
3. **小步快跑+门禁**：不做大计划，把事逐步交给AI，但每个阶段都有验收标准和可验证的检查点
4. **人管判断，AI管执行**：70%的规划决策是人做的，80%的执行决策是AI做的——这个分工比例已被40万次真实对话验证
5. **不要跳阶段**：从Copilot到Agent是五阶段渐进过程，跳阶段的组织事故率高2.7倍

### 6.2 最危险的三种用法（综合失败教训）

| 危险模式 | 为什么危险 | 正确做法 |
|---------|----------|---------|
| **把不可逆决策交给Agent** | AI会用同样笃定的语气支持你做或不做 | 不可逆操作（发送/发布/删除）必须人来按 |
| **验收流于形式** | "假HOTL"——名义上人监督，实则注意力衰减失能 | 设计强制抽检、随机复核、对齐激励 |
| **一股脑丢给全自动** | 同一封邮件的三段活儿分属不同抽屉 | 同一任务拆段，不同段不同模式 |

### 6.3 从Cursor团队真实工作流学到的 【📚已有笔记·你最该深读的一篇】

Cursor Team Kit的价值不在于那17个skill本身，而在于它**完整展示了一支把AI编程跑到生产级的团队，工作流到底长什么样**：
- 每个环节都有对应的skill（起手→自查→评审→CI→收尾→汇报）
- 专门有deslop清理AI写脏的代码（说明他们深知AI产出的"坏味道"）
- verify-this强制AI自己跑验证（解决"AI说修好了其实没修好"的问题）
- workflow-from-chats把摸索到的协作模式沉淀为可复用skill（持续进化）
