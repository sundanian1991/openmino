# 第一章 本质：人机协同到底在协同什么

## 核心命题

> **人机协同的本质，不是静态地划"AI干什么、人干什么"，而是动态地把人的判断力，从脑子里传递到AI可执行的流程里。**

- 判断说不清楚 → AI只能做最浅层的事（写JD、润色邮件）
- 判断说得清楚 → AI能做深层的事（招聘漏斗分析、绩效质检、组织诊断）
- **判断说得越清楚，能交出去的越多，自己就能往越高的层次走**

这个命题有五条交叉验证：

| 来源 | 验证 |
|------|------|
| 40万次Claude Code对话分析 | 专业度 = 指令精准度 + 验证要求 + 纠偏能力（全是"把判断说清楚"的不同侧面） |
| 判断档案方法论 | 直觉→显式化→可复用，整个过程就是"把判断说清楚" |
| HRBP应用指南 | 核心竞争力 = "能不能把业务问题拆成清楚的数据、流程和动作" |
| Agent群聊协作方案 | Agent的价值 = "你配置的判断框架有多清晰，它就有多有用" |
| Cursor Team Kit | 17个skill的本质 = 把团队的隐性判断固化成可复用工具 |

## 哲学根基：增强还是替代

这是最底层的价值选择。源头是Licklider（1960）"Man-Computer Symbiosis"与Engelbart（1962）"Augmenting Human Intellect"。

| 哲学 | 目标 | 价值衡量 | 适用场景 |
|------|------|---------|---------|
| **Automation（替代）** | AI替代人做某事 | 省下的人力成本/时间 | 规则清晰、可标准化、错误可容忍 |
| **Augmentation（增强）** | 让人做到原本做不到的事 | 新产生的可能性与产出质量 | 需要判断、创造性的工作 |

两者不是非此即彼，而是一个谱系。**判断依据：任务的可结构化程度 + 错误可逆性 + 创造性需求。** 三者均高适合augmentation，反之适合automation。

**落地启示**：高价值岗位应优先augmentation——"让强者更强"的边际收益高于"替代弱者"。一个组织内部通常是混合策略：底层任务自动化（释放时间），高层任务增强化（提升产出）。

**头部人物的共识**：

| 人物 | 核心主张 | 人机关系定位 |
|------|---------|------------|
| **Sam Altman**（OpenAI） | "温和奇点"已开始；人类拥有不可替代优势——天生在乎他人 | 放大器，一人顶多人 |
| **Dario Amodei**（Anthropic） | "智能边际回报"框架；人类约束是智能的互补要素，人机协同是结构性必然 | "数据中心天才之国"+人类约束 |
| **Mira Murati**（Thinking Machines Lab） | "扩展人类的意志和判断"；知识分散性论证——默会知识无法被中央智能聚合 | "双人自行车"，永久回路 |

三人分歧最大的点：对AI自主性的态度。Altman和Amodei拥抱自主性但强调人类约束；Murati直接质疑自主性路线，主张"参与即能力"。

# 第二章 分工：谁做什么

## 2.1 四层递进模型

分工不是一张静态表，是四层递进结构。**越往上越核心，越往下越容易外包。**

```
第四层 · 进化分工（谁在变好）
  人：把隐性判断显式化 → 不断往上游走
  AI：按显式化判断执行 → 不断往下游接
       │
第三层 · 知识分工（谁记住什么）
  人：记住"意味着什么"（判断、风格、直觉）
  AI：记住"发生了什么"（事实、过程、上下文）
       │
第二层 · 判断分工（谁定标准）
  人：判断决策 + 结果验收（只握两个环节）
  AI：信息收集 + 信息整理 + 执行落地
       │
第一层 · 执行分工（谁做）
  人：低频高价 + 不可逆 + 创造性跳跃
  AI：高频重复 + 可验证 + 模式匹配
```

### 第一层：执行分工（谁做）

最表层。40万次真实对话验证：**人管"规划决策"(What) ≈ 70%，AI管"执行决策"(How) ≈ 80%。**

| 维度 | AI做 | 人做 |
|------|------|------|
| 可验证性 | 即时可验证（编程、格式转换、数据汇总） | 长周期难验证（战略、文化、关系） |
| 频率 | 高频重复（周报、质检、分类） | 低频高价（关键决策、危机处理） |
| 可逆性 | 可撤销（草稿、分析初稿） | 不可逆（发出的邮件、签的合同） |
| 认知类型 | 模式匹配（归类、对比、检索） | 创造性跳跃（从0到1的框架） |

### 第二层：判断分工（谁定标准）

多数人只看到第一层，忽略了关键判断：**判断本身不外包，但"判断之前的整理"和"判断之后的执行"都可以外包。**

一个完整的判断过程：

```
信息收集 → 信息整理 → 模式识别 → 判断决策 → 执行落地 → 结果验收
   AI做       AI做      人+AI       人做       AI做      人做
```

**人只握两个环节：判断决策 + 结果验收。**

**陷阱**：判断决策的质量取决于信息整理做得好不好。AI整理得不对，人的判断就建立在错误信息上。→ 这就是为什么"不能把AI当裁判"——**AI做的是"减少遗漏"，不是"做出裁决"。**

### 第三层：知识分工（谁记住什么）

传统工作里知识存在三个互不互通的地方：人的脑子里、文档里、聊天记录里。数字分身/Agent群聊的突破在于：让AI成为"统一知识层"——判断档案（脑子里的显式化）+ 知识库（文档结构化）+ 群聊记忆（协作上下文自动沉淀）。

**一句话：AI记住"发生了什么"，人记住"意味着什么"。**

### 第四层：进化分工（谁在变好）

40万次对话的关键趋势：7个月里，用户交给AI的任务价值层级持续上移（修bug 33%→19%，部署运维 14%→21%，写作分析 10%→20%）。**人在不断把低层任务"交出去"，自己往更高层走。**

判断力传递的过程：

| 阶段 | 人做什么 | AI做什么 |
|------|---------|---------|
| 第一次做 | 全程参与 | 辅助 → 人提取判断样本（5问） |
| 第二次做 | 给判断框架 | 按框架执行 → 人验收 |
| 第三次做 | 只验收异常 | 自己按框架执行，判断被"外化"了 |
| 第N次 | 去做更高层的新任务 | 这个任务完全由AI承担 |

## 2.2 实操工具：3A分诊法

四层模型是认知框架，3A分诊法是**日常操作工具**——每接到一个任务，30秒扔进三个"抽屉"之一。

| 模式 | 定义 | 人做什么 | 典型场景 |
|------|------|---------|---------|
| **Automation** | AI独立完成确定性任务，你不用盯，结果好验证 | 验收结果 | 格式转换、数据清洗、日报汇总 |
| **Augmentation** | AI给信息给建议，你拍板 | 最终决策 | 竞品调研摘要、要不要做某需求 |
| **Agent** | AI自主规划执行多步任务，你只设边界 | 设边界和约束 | 完整调研+出对比报告全流程 |

**三个分诊维度**：
1. **确定性**：规则有多确定？输出好不好验证？→ 越确定越往Automation推
2. **可逆性**：做错了能不能撤？→ 越不可逆越留在Augmentation
3. **任务跨度**：单步还是多步串联？→ 跨度大的适合Agent模式

**一个精妙例子**（发产品更新邮件）：同一件事拆三段——日志整理成几条（Automation）→ 语气判断、该不该提灰度功能（Augmentation，AI给两版我拍板）→ 真正点发送给哪个分组（人自己来）。**一股脑丢给Agent，它会把灰度功能也写进去。**

## 2.3 理论支撑

四层模型和3A分诊法并非凭空发明，背后有坚实的学术根基：

| 理论 | 提出者 | 核心观点 | 对应本框架 |
|------|--------|---------|-----------|
| **HCAI 2×2矩阵** | Ben Shneiderman (马里兰大学) | "高自动化+高人类控制"才是理想，不是权衡关系 | 第二层：保留有意义的人类控制 |
| **Parasuraman四阶段** | Parasuraman, Sheridan, Wickens (2000) | 信息获取→分析→决策→执行，前端可自动化，后端留人 | 第二层：判断过程的六个环节 |
| **Centaur模型** | Garry Kasparov | 人负责战略判断，AI负责战术执行，边界清晰 | 第一层+第二层 |
| **分布式认知** | Edwin Hutchins (1995) | 认知分布在人+工具+环境系统中 | 第三层：知识分工 |
| **Augmentation谱系** | Licklider / Engelbart | AI目标是扩展人不是替代人 | 哲学根基 |
| **Human Agency Scale** | 斯坦福SALT Lab | 人类能动性可分五级，是滑块不是开关 | 3A分诊的学术版本 |

# 第三章 演进：从Tool到Agent的五阶段光谱

## 五阶段自主性模型

```
Stage 1  自动补全     AI预测片段，人选择
Stage 2  Copilot     AI生成完整建议，人审查采纳
Stage 3  委托执行     AI执行完整任务，人审批结果
Stage 4  监督自主     AI在边界内自主运行，人监控异常
Stage 5  完全自主     AI自主发现任务并执行，人设目标
```

| 维度 | Stage 1 | Stage 2 | Stage 3 | Stage 4 | Stage 5 |
|------|---------|---------|---------|---------|---------|
| AI角色 | 建议片段 | 建议完整动作 | 执行任务 | 在策略内运行 | 自主决策 |
| 人角色 | 做工作 | 审查草稿 | 审批结果 | 看仪表盘 | 设目标 |
| **生产力倍数** | 1.2-1.3x | 1.3-1.5x | **2-5x** | **5-20x** | 20-100x |
| 信任要求 | 低 | 中 | 高 | 很高 | 极高 |
| 错误容忍 | 高(人纠正) | 高(人审查) | 中(人审批) | 低 | 极低 |

## 关键阶段跃迁

**Stage 2→3（最关键的跃迁）**：从"建议"到"执行"。产品价值主张从"AI帮你编码"变成"AI编码，你审结果"。需要：工具集成 + 迭代推理 + 质量自评 + 审批界面。

**Stage 3→4（需3-6个月积累）**：从逐任务审批到自主运行。需要：可靠性数据 + 策略框架 + 监控基础设施 + 渐进扩范围。

**⚠️ 跳阶段的代价**：跳过Stage 3直接部署Stage 4的组织，**事故率高2.7倍**（McKinsey, 2025）。超过40%的agentic AI项目将在2027年前被取消。

## Copilot的天花板

Copilot阶段的结构性限制：**每个AI产出都需要人审查，形成瓶颈，生产力上限被卡在30-50%**。突破天花板必须进入Stage 3（委托执行）。SWE-bench数据：Claude Sonnet 4解决了72.7%的真实GitHub issue——每10个委托任务约7个无需人工纠正。

# 第四章 方法：怎么做好

## 4.1 给AI的"操作手册"：AGENTS.md

README.md是给人看的项目门面，AGENTS.md是给AI看的项目操作手册。

**四层结构**：

```
AGENTS.md
├── 核心上下文（技术栈快照 + 项目架构说明）
├── 运行指南（Build/Test命令 + 环境变量要求）
├── 质量约束
│   ├── Do's: 必须遵循的规则
│   └── Don'ts: 严格禁止的事项 ★最重要
└── 动态演进（随项目迭代更新）
```

**五条最佳实践**：
1. **机器友好的格式**：多用表格、列表、代码块
2. **用强语气词**：`Must`/`Never`/`Always` 效果远好于 `Should`/`Recommend`
3. **提供正误对比**：展示❌错误 vs ✅正确
4. **单一职责**：只写工程规范，不写业务逻辑
5. **文件不超过200行**：超过会分散AI注意力

**防坑**：信息过时、含糊其辞（"写高质量代码"是废话，应写"函数必须包含JSDoc注释并标注类型"）、忘记安全约束。

## 4.2 Codex PMP五阶段法

把使用AI从"写代码做任务"升级为"AI辅助项目管理"：

| PMP阶段 | 操作 | 输出物 |
|---------|------|--------|
| ①启动 | "我们要做什么？不要做什么？验收标准是什么？" | PROJ.md |
| ②规划 | Plan Mode："不要写代码，先拆解任务，列风险点和验证方式" | PLAN.md |
| ③执行 | 小步快跑，按计划执行 | 代码/文档 |
| ④监控 | 边做边检查，防止偏离 | 门禁检查 |
| ⑤收尾 | 验收标准对照 | 交付物 |

**核心理念**："不做冗余计划，将任务逐步交付给AI，通过标准化流程实现最稳定的掌控。"

## 4.3 三段式工作流：人定方向→AI规划→AI执行

| 阶段 | 谁主导 | 核心动作 |
|------|--------|---------|
| 规划 | 人（可借助更强模型） | 产品脑暴、架构决策、任务拆解、验收标准定义 |
| 执行 | Agent | 进入代码库→修改→跑测试→生成diff |
| 审核 | 人 | 质量评估、迭代推进，不通过则调整重新执行 |

## 4.4 Agent工程化原则

**Agent最小闭环**：`User Task → Plan → Tool Call → Observation → Revise Plan / Finish`

**验收四条底线**（达不到就别加更多工具）：
1. 同一输入连续跑5次，关键步骤一致
2. 工具失败时不会无限重试
3. 每一步都有日志，能看出为什么走到当前结论
4. **人类可以在关键节点中断或接管**

**三种架构选择**：

| 架构 | 适合 | 原则 |
|------|------|------|
| 单Agent | 80%任务（文档处理、数据抽取、内容QA） | 最稳，状态简单 |
| Planner-Executor | 复杂任务（重构、调研、批量生产） | 职责隔离：Planner不改文件，Executor不改目标 |
| 多Agent | 天然可并行（各审一个模块） | 最后再上！频繁共享状态会复杂度飙升 |

**工具设计三原则**：schema要窄（不要给泛化接口）、权限分层（只读/本地可逆/外部可见必须确认）、工具必须幂等+有超时。

## 4.5 效果评估：四维体系

| 维度 | 核心指标 | 说明 |
|------|---------|------|
| **效率** | 任务完成时间、吞吐量、首次完成率 | 最易量化，也最易被厂商夸大 |
| **质量** | 缺陷率、返工率、一次通过率、修改深度 | AI产出被直接采纳的比例 |
| **创新** | 方案多样性、探索广度 | **最被低估**。AI辅助下想法更收敛但更窄——"创新陷阱" |
| **人因** | 认知负荷、技能退化指标、满意度 | 长期使用后脱离AI的表现衰减 |

# 第五章 验证：实证数据、案例与教训

## 5.1 重磅实证：40万次Claude Code对话分析

这是目前人机协同领域**数据量最大、最接近真实场景**的实证研究。

| 发现 | 数据 |
|------|------|
| 人管What，AI管How | 人做约70%规划决策，AI做约80%执行决策 |
| 资历归零 | 专业度仅由3项决定：指令精准度、验证要求明确度、纠偏能力 |
| 管理者反超工程师 | 管理类职业验证成功率37% > 软件工程师34% |
| 专家vs新手差距 | 专家1条指令触发12个动作/3200字；新手5个/600字——**5倍差距** |
| 任务价值上移 | 7个月里修bug类33%→19%，写作分析类10%→20% |

**专家协作三要素**（每次给AI发任务前确认）：
1. 是否清晰传递了**任务目标**
2. 是否明确告知AI需要验证的**规则**
3. 是否具备识别AI理解偏差并**纠正**的能力

## 5.2 跨行业实证

### 软件开发（数据分歧最大的领域）

| 研究 | 关键数据 | 备注 |
|------|---------|------|
| GitHub官方实验(2022) | Copilot组快55.8%，完成率78% vs 70% | 简单任务实验室上限 |
| METR(2025,真实代码库) | 熟悉代码库开发者用AI**反而慢19%** | 与厂商宣称强烈反差 |
| GitClear(2024,8亿行) | 代码churn增加7倍 | 质量风险信号 |
| Uplevel(2024) | Bug解决速度无显著提升，但burnout下降30% | 真实工程环境 |

> **判断**：55%是实验室上限；真实工程环境中净提速往往落在**0%–30%区间**。这是目前最被厂商夸大的领域。

### "人机协同未必1+1>2"的反例

- **医疗（Goh et al., JAMA 2024）**：对比LLM单独、LLM+医生、医生单独——**"LLM+医生"组合反而低于LLM单独**，揭示"锚定偏差"。
- **医疗（俄亥俄州立）**：AI正确时提升护士表现+67.4%；**AI错误时暴跌120.5%**。"贝恩布里奇悖论"：AI越可靠，错误越罕见，人越放松警惕。
- **创意（MIT, 2024）**：AI组速度快37%，但**表现最好的人没怎么受益，表现最差的人提升最大**——AI是"水平均衡器"非"上限突破器"。
- **BCG顾问实验(HBR, 2023)**：+12.2%任务量/+25.6%速度/+40%质量，但**"在不该用的任务上硬用反而比不用更糟"**（边界外表现下降23%）。

## 5.3 真实案例：Cursor Team Kit

Cursor团队自己怎么用Cursor写Cursor——17个skill、1个sub-agent、2条rule构成的工作循环：

```
开发：new-branch-and-pr → check-compiler-errors（边写边自查）
评审：review-and-ship → pr-review-canvas → get-pr-comments
CI：  loop-on-ci → fix-ci（红的修到绿）
收尾：verify-this（证明bug修好了）→ deslop（擦干净AI写脏的代码）
汇报：what-did-i-get-done → weekly-review
```

最有价值的skill是**verify-this**：解决"AI说修好了其实没修好"——强制AI自己跑验证。

## 5.4 HRBP场景验证

HRBP的6大场景是四层模型的完整验证场：

| 场景 | AI做（执行层） | 人做（判断层） | 工具化方向 |
|------|--------------|--------------|-----------|
| 招聘漏斗 | 分析+看板+排名 | 定"卡点怎么算异常" | 招聘进度看板 |
| 绩效质检 | 批量扫描6维度 | 定"什么算合格评价" | 绩效评价质检表 |
| 组织诊断 | 按维度分类纪要 | 定"问题怎么分类" | 组织诊断报告模板 |
| 员工关系 | 整理时间线+标风险 | 最终判断（不可外包） | 员工访谈归类工具 |
| 管理者赋能 | 生成工具包 | 定"什么场景需要什么" | 管理者沟通话术库 |

**关键发现**：8个工具中价值最高的不是任何一个单独工具，而是**一个配置了HRBP Skill+知识库+判断档案的数字同事**——等于8个工具的集合+持续记忆+可进化。

### 失败的根本模式

| 根本原因 | 表现 | 本框架定位 |
|---------|------|-----------|
| **默会知识丢失** | 裁一线员工丢失场景知识和纠错能力 | 第三层（知识分工）被破坏 |
| **自动化偏见与技能退化** | AI越可靠人越放松警惕，罕见出错时人已丧失判断力 | 第二层（判断分工）的陷阱 |
| **跳阶段** | 从Copilot直接到完全自主 | 违反第三、四层的渐进原则 |
| **责任真空** | 名义上人监督实则失能 | "假HOTL"——第二层的隐蔽失败 |

### 方法论自身的反噬风险

- **过度显式化陷阱**：把所有判断拆成规则 → 僵化checklist → 失去灵活性
- **判断污染**：早期错误判断被AI强化 → 正反馈循环
- **能力退化**：人把判断交给AI → 自己判断力退化（计算器效应）

## 6.2 最关键的起步动作：判断档案

**成本最低（每次5分钟），长期复利最大。**

每次做完一个判断后，用5问提取判断样本：
1. 我否定了什么
2. 为什么否定
3. 在AI的平行可能中保留了什么
4. 我真正关心的问题本质是什么
5. 下次类似任务应避免什么

积累2-3个月，就有一份不可替代的职业判断资产——这是判断力从脑子里传递到AI流程里的"桥梁"。

## 一页纸总结

**本质**：人机协同 = 动态地把判断力从脑子传递到AI流程。

**分工**：四层递进（执行→判断→知识→进化），越往上越核心。日常用3A分诊法30秒决定每个任务怎么分。

**演进**：五阶段（自动补全→Copilot→委托执行→监督自主→完全自主），生产力从1.2x到100x，但不能跳阶段。

**方法**：写好AGENTS.md（给AI的操作手册）、用PMP五阶段控场、保持Agent最小闭环可验收、用四维体系评估效果。

**验证**：40万次对话证明人管判断AI管执行；但医疗/创意的反例证明"1+1未必>2"；Klarna/福特的教训证明默会知识丢失最致命。

**进化**：从入门（会用AI生成）→工具化（做可复用工具）→判断力（把判断显式化）→分身化（数字分身）。**第一步是建判断档案——成本最低，复利最大。**

**未知**：最根本的——这套方法是在搬运判断力，还是在生长判断力？目前没有答案。
