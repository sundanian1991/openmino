03
知识分工
谁记住什么？
→
人记住
"意味着什么"——判断、风格、直觉、为什么这么定、下次遇到类似的怎么判断。
AI记住
"发生了什么"——事实、过程、上下文。什么文件在哪、上次怎么定的、谁说了什么。
突破：传统工作里知识散在三处（脑子/文档/聊天记录）互不互通。数字分身/Agent群聊让AI成为"统一知识层"——判断档案（脑子显式化）+知识库（文档结构化）+群聊记忆（协作上下文自动沉淀）。
02
判断分工
谁定标准？
→
人只握两个环节
判断决策（做出裁决）+ 结果验收（确认对不对）。这两个环节不可外包。
AI承接四个环节
信息收集 → 信息整理 → 模式识别（人+AI） → 执行落地。判断前后的"体力活"都可以外包。
陷阱：AI整理得不对，人的判断就建立在错误信息上。所以"不能把AI当裁判"——员工关系/绩效争议等场景，AI只做"减少遗漏"，不做"做出裁决"。

L1 · AGENTS.md：给AI的"操作手册" ⚡ 可做技能

README.md是给人看的项目门面，AGENTS.md是给AI看的项目操作手册（类似 Cursor 的 .cursorrules 或 Claude Code 的 CLAUDE.md）。通过机器友好的Markdown格式，以极低Token成本让AI"瞬间入职"。适用于日常单次任务——你给AI规则，它按规则执行，你验收结果。

01
用强语气词
用 Must / Never / Always，效果远好于 Should / Recommend。AI对强指令的遵从度显著更高。
02
提供正误对比
展示 ❌错误 vs ✅正确，是最高效的沟通方式。比写500字描述规则，不如给一对代码/文案的正反示例。
03
机器友好格式
多用表格、列表、代码块。AI处理结构化数据远优于散文。把命令、路径、参数用表格呈现。
04
单一职责
只写工程规范，不写业务逻辑。业务逻辑让AI通过阅读代码/文档获取。文件不超过200行——超过会分散AI注意力。
05
禁令清单最重要
明确写出 Don'ts。必须包含安全约束（如"禁止将API Key写在代码里"）。这是防止AI"好心办坏事"的关键。
!
防坑
最常见错误：①含糊其辞（"写高质量代码"是废话）②信息过时（依赖版本变了没更新）③文件太大 ④忘记安全约束

L3 · 长周期复杂项目

AI 长周期项目协作方法论

当项目跨越数天甚至数周、涉及多轮会话和多个子任务时，AI会在对话增长中丢失目标、偏离路线、遗忘决策。这套方法解决的核心问题是：怎么在长周期项目中保持AI不漂移。

三大锚点 · 防止AI漂移
ANCHOR 01
目标锚定
把目标外置到持久文件 GOALS.md，不随对话漂移。每个里程碑记录目标、范围、决策、阻塞、完成证据。
ANCHOR 02
证据验收
里程碑不是AI说了算，是证据说了算。双轨审计：路线图审计 + 代码审查，独立线程执行避免实现者偏差。
ANCHOR 03
状态外置
项目状态存在于文件中，不存在于对话上下文中。三行简报 + 仪表盘，任何人随时可重建全局认知。

四维效果评估

不管用哪个层次的方法，都需要评估效果。单纯"效率提升"不够，成熟体系应是四维。

⚡
效率
任务完成时间、吞吐量、首次完成率。最易量化，也最易被厂商夸大。
✓
质量
缺陷率、返工率、一次通过率（AI产出被直接采纳的比例）、修改深度。
✦
创新
方案多样性、探索广度。最被低估。AI辅助下想法更收敛但更窄——"创新陷阱"。
☺
人因
认知负荷、技能退化指标（长期使用后脱离AI的表现衰减）、满意度。
