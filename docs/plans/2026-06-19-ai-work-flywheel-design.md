# AI 工作飞轮体系设计

> 把 AI 从"手里的工具"变成"持续运转的系统"，实现百倍提效。
>
> 设计日期：2026-06-19 | 作者：年老师 + Mino

---

## 一、核心命题

### 1.1 百倍提效的真相

**百倍提效不是靠花更多 token，是靠改变 token 的运转方式。**

当前工作模式（线性）：

```
人发起请求 → AI 响应 → 产出 → 会话结束
```

线性模式的硬上限是**用户个人发起请求的带宽**——一天能跟 AI 对话几十次到顶。单次 token 消耗再大，乘以几十，量级上不去。瓶颈不是 token，是人。

目标模式（飞轮）：

```
事件/时间触发 → 多 agent 并行 → 主 agent 汇总 → 推送人审 → 产出回流知识库 → 下次复利
```

飞轮模式下，AI 不再等人说话，而是自己循环。token 消耗自然上 10-100 倍。

### 1.2 核心矛盾

用户底层武器齐全（300+ skills + 知识库 + 记忆体系），但缺**把武器变成军队的编排层**——触发机制、并行 agent、沉淀闭环。这三样建起来，飞轮自转。

---

## 二、五层全景

从下往上，①是地基（已有），②③④⑤是要新建的。每加一层，token 消耗和提效上一个台阶。

| 层 | 作用 | 现状 | 要做什么 | 提效杠杆 |
|----|------|------|---------|---------|
| **⑤ 反馈层** | 度量：哪些循环真省时间，该停哪些 | 无 | 建"循环效果账本" | token 花刀刃上 |
| **④ 沉淀层** | 每次产出回流知识库，下次复用 | 半个（只 ingest 输入，不回流产出） | 补产出端回流闭环 | **复利**——下次不重复花 |
| **③ 编排层** | 多 agent 并行调度，主 agent 做决策 | 无 | 建 agent 调度机制 | 串行→并行，同时干 N 件 |
| **② 触发层** | 事件/时间/条件触发，不等人说 | 无 | 建触发机制 | **去掉用户的发起频率上限** |
| **① 基础设施** | 知识库 + 记忆 + skills | 已有，需补全 | 补供应商档案库等缺口 | 地基 |

### 2.1 各层具体动作（结合供应商管理场景）

**② 触发层**

| 触发类型 | 场景 | 现状 | 改造后 |
|---------|------|------|--------|
| 定时 | 每月 5 号业绩排名+分析 | 手动发起 | 到点自动跑，推送结论给人审 |
| 事件 | 供应商数据更新→刷新经营画像 | 不刷新/手动 | 数据一变自动更新 |
| 阈值 | 业绩下滑>20%、新准入、制度异常 | 靠人看 | 触线自动告警+分析 |

**③ 编排层**

| 场景 | 现状 | 改造后 |
|------|------|--------|
| 6 家重点供应商月度分析 | 一家家串行看 | 6 个 agent 并行出报告，主 agent 汇总 |
| 准入评估（财务/合规/产能/团队） | 一个个维度过 | 多维度并行评估，主 agent 出结论 |
| 多份周报/汇总 | 逐份做 | 并行生成，统一汇总 |

**④ 沉淀层**

| 沉淀类型 | 现状 | 改造后 |
|---------|------|--------|
| 产出回流 | 用完即弃 | 每份报告→自动更新对应供应商的 entity 页 |
| 模式提炼 | 靠人记 | 同类分析做 3 次→自动抽象成 skill/模板 |
| 决策记录 | 无 | 每次准入/清退决策入库，可复盘 |

**⑤ 反馈层**

每个循环记一笔账：token 消耗 / 节省时间 / 产出质量。效果差的循环停掉，效果好的扩。循环多了才需要这层，先放着。

## 三、使用边界 / 场景

供应商管理在金融科技，边界必须清楚——**不是所有事都该交给 AI 自动跑**。

| 场景 | 模式 | AI 做 | 人做 |
|------|------|------|------|
| 汇总/排版/周报材料 | **全自动** | 跑完直接给人 | 扫一眼即用 |
| 业绩分析/经营画像 | **AI 做+人审** | 出完整分析 | 审结论，定动作 |
| 准入评估 | **AI 辅助** | 出多维度评估 | 人定结论 |
| 清退/分工决策 | **AI 建议** | 给依据和风险 | 人决策，AI 不拍板 |
| 对上级汇报/对外材料 | **人在场** | 起草润色 | 人审了再发 |

### 3.1 数据安全红线

- 涉客户/供应商敏感数据 → 本地处理或脱敏后再过 AI
- 对外材料 → 必过一道人审
- 合规敏感操作（准入/清退）→ AI 只给依据，决策权永远在人

**一句话边界**：执行层（汇总、分析、起草）尽量全自动；决策层（准入、清退、对外）AI 只到"建议"，人定。

## 四、第一个飞轮：月度业绩分析

选它做起点的原因：**每月固定发生 + 用户 MEMORY 里早就写了要它"主动行为" + 输入输出都清楚 + 最能尝到飞轮味道。** 从"每月手动发起"改成"到点自转"。

### 4.1 飞轮全貌

```
┌─────────────────────────────────────────────────────────┐
│ 触发：每月 5 号 09:00（定时）                            │
└──────────────┬──────────────────────────────────────────┘
               ▼
┌─────────────────────────────────────────────────────────┐
│ 并行编排：6 个子 agent 同时跑（③编排层）                 │
│   agent-毅航 / agent-毛毛虫 / agent-伽玛 / agent-赛维斯  │
│   agent-岐力 / agent-翰锐                                 │
│ 每个产出：单家经营画像 + 业绩排名数据 + 风险信号         │
└──────────────┬──────────────────────────────────────────┘
               ▼
┌─────────────────────────────────────────────────────────┐
│ 主 agent：汇总 + 排名 + 横向对比 + 出决策建议（③编排层）  │
│   产出：月度业绩分析报告 + 排名 + 异常告警               │
└──────────────┬──────────────────────────────────────────┘
               ▼
┌─────────────────────────────────────────────────────────┐
│ 推送 + 人审：结论先给人（②触发层）                       │
│   人只看：排名变化、异常、该不该有动作                    │
└──────────────┬──────────────────────────────────────────┘
               ▼
┌─────────────────────────────────────────────────────────┐
│ 沉淀：报告 → 自动更新每家供应商的 entity 页（④沉淀层）    │
│   下月起点不再是零——上月的分析直接作为基线              │
└──────────────┬──────────────────────────────────────────┘
               ▼
┌─────────────────────────────────────────────────────────┐
│ 反馈：记一笔账（⑤反馈层）                                │
│   token 消耗 / 节省时间 / 产出满意度                      │
└─────────────────────────────────────────────────────────┘
```

### 4.2 当前缺口（按是否卡住飞轮排序）

| 缺口 | 严重性 | 说明 |
|------|--------|------|
| **触发机制** | 🔴 卡死 | 现在没法定时跑，第一个要解 |
| **6 家供应商完整档案** | 🔴 卡死 | memory/ 下 18 个画像未入库，子 agent 无料 |
| **子 agent 输出格式契约** | 🟡 重要 | 6 个并行需统一格式，主 agent 才能汇总 |
| **月度数据来源** | 🟡 重要 | 业绩数据来源（用户已确认系统有接口，需进一步验证可用性） |
| **报告回流管道** | 🟢 后置 | 先转起来，回流闭环第二步再加 |

## 五、落地路径

环境前提：用户主要在 **CLI（命令行工具）** 跑 AI，**业务系统有接口**可拉数据。

不一次建完，按"能不能跑通"的风险排序。**先拿 1 家跑通，再加并行**——符合"先做出来，完美靠迭代"原则。

| 步骤 | 做什么 | 目的 | 风险 |
|------|--------|------|------|
| **0 验证** | 确认 CLI 支持 headless（能被定时调、传 prompt 自动跑）+ 确认系统接口能调 | 消灭最大技术不确定性 | 🔴 不验证就建，白干 |
| **1 补数据** | 把 memory/ 下 18 个供应商画像入库（库 2：供应商档案库） | 子 agent 有料可分析 | 🔴 没数据飞轮空转 |
| **2 最小飞轮** | 先拿 1 家（例如毅航），跑通：拉数据→AI 分析→出报告→给人审 | 验证全链路通 | 🟡 单家跑通是地基 |
| **3 加并行** | 单家跑通后扩到 6 家并行 + 主 agent 汇总 | 串行变并行，杠杆上来 | 🟢 模式已验证 |
| **4 加沉淀** | 报告自动回流每家 entity 页 | 复利——下月起点非零 | 🟢 锦上添花 |

**关键原则**：飞轮是"模式"，不是巨型系统。1 家跑通 = 模式验证完，扩到 6 家只是复制。直接上 6 家，一个环节卡住 6 个一起死。

## 六、日常角色转变

飞轮跑起来后，用户从"发起者"变"审核者"：

| 现在 | 跑通后 |
|------|--------|
| 每月 5 号人发起分析 | 到点自动跑，人不动 |
| 人逐家看数据 | 6 家并行出完，主 agent 汇总好 |
| 人从零写分析 | 上月分析已沉淀，AI 带基线继续 |
| 人决定怎么呈现 | 报告自动出，**人只看：排名变化、异常、该不该有动作** |

**一句话**：token 花在执行（拉数、分析、排版、汇总），人的精力只花在决策（审结论、定动作）。百倍提效的日常样子——不是人干得更快，是**人不再干那些事**。

## 七、范围与延展

### 7.1 本次范围

第一个飞轮（月度业绩分析）从设计到跑通。跑通后形成可复制模式。

### 7.2 后续可复制的飞轮（按优先级）

1. **供应商准入评估飞轮**（事件触发：新供应商申请 → 多维度并行评估）
2. **日常 BPO 沟通纪要飞轮**（高频，每日发生）
3. **季度经营分析飞轮**（周期性，价值高）

### 7.3 关键约束

- 不为一次性场景抽象（做减法）
- 不先搭空架子再填内容（先跑通一个真实飞轮）
- 决策权永远在人（合规红线）
