## 0. 一句话目标

**让每个概念只有一个家。** 文件该去哪儿，由"它现在处于什么生命阶段"决定，不由"它是什么格式 / 哪个工具生成"决定。

## 1. 诊断：为什么会乱（6 类结构性病灶）

| 病灶 | 表现 | 量化证据 |
|---|---|---|
| **① 概念多主** | 会话记录存 5 处、计划存 5 处、幻灯片 2 套、可视化知识 4 处 | workspace 7 个对话摘要 + memory/conversations(132,停更) + memory/daily(碎片) + memory/daily-letter(停更) + memory/sessions(纯噪声) |
| **② 业务域粉碎** | "供应商管理"散在 docs/projects/plans/workspace/myagents_files/wiki/memory 共 7 处 | 供应商画像在 memory/ 存了两份；金条Q3定价同时存在于 docs/供应商+workspace+根目录+myagents_files |
| **③ 生命周期倒挂** | 临时区堆长期产物，长期区装垃圾 | projects/ 7883 文件中 **6950 个(88.7%)是 node_modules**；plans/ 30 个文件夹中约 25 个已完结未归档 |
| **④ 内存系统半瘫** | memory/ 4 条 cron 流水线死 3 条 | conversations 截止 06-08 停更、daily-letter 06-08 停更、sessions 64/65 文件是巡检噪声 |

## 2. 第一性原理

> 一个文件唯一决定它该去哪儿的，是**"它对我现在处于什么生命阶段"**。

由此推出 **5 区模型**——每层职责互斥，一个文件只属于一区：

| 区 | 回答的问题 | 特征 | 目录 | 时间感 |
|---|---|---|---|---|
| **① Engine 引擎** | "这东西跑我的工具吗" | 代码/配置/运行时 | `config/ lib/ scripts/ .codex/ .claude/ .agents/ package.json` | 稳定 |
| **② Brain 大脑** | "这是我 agent 的内部记忆吗" | 认知契约/工作记忆/索引 | `memory/`(清理后) | 持续·agent 私有 |
| **③ Workbench 工作台** | "我现在正在做这个吗" | 进行中的会话/计划/草稿 | `workspace/ + plans/ + intake/` | 热·按周轮转 |
| **④ Knowledge 知识** | "这是我能复用的沉淀吗" | 方法论/SOP/原子知识 | `docs/ + wiki/ + wiki-viz/ + reference/` | 常青·按主题 |
| **⑤ Vault 金库** | "这是做完了想留底的吗" | 已完成项目记录 | `workspace/archive/ + projects/` | 冷·按时间归档 |

## 3. 目标目录结构

```
my-agent/
├── ① Engine（工程基础设施，保持现状）
│   ├── config/ lib/ scripts/ package.json reasonix.toml opencode.jsonc skills-lock.json
│   └── .codex/ .claude/ .agents/ .assistant/ .codegraph/ .joycode/ .next/ .venv/
│
├── ② Brain（清理后回归纯记忆）
│   └── memory/  ← 只留：认知契约 + 工作记忆 + 索引；业务文档迁出
│
├── ③ Workbench（热区，强制轮转）
│   ├── workspace/   ← YYYY-MM-DD-主题，2 周不更新即归档
│   ├── plans/       ← 只放进行中思考；完结即蒸馏到 docs/plans 或归档
│   └── intake/      ← 【改造自 myagents_files】正式收件箱，强制分流
│
├── ④ Knowledge（常青知识，按主题）
│   ├── docs/        ← 沉淀知识（合并可视化入口、清理临时件）
│   ├── wiki/        ← 供应商管理原子库（保留）
│   ├── wiki-viz/    ← 可视化原子库（保留·兄弟关系，加 README 说明）
│   └── reference/   ← 第三方参考资料 + HTML 范式库
│
├── ⑤ Vault（冷区，按时间归档）
│   ├── projects/    ← 长期业务项目（砍 node_modules、迁走第三方代码）
│   └── workspace/archive/  ← projects/ongoing/scattered 三类归档
│
└── ❌ 解散（不再作为概念存在）
    ├── myagents_files/  → 改造为 intake/（落客→正式收件箱）
    ├── my-slide/        → 模板入 reference/，实验残留删除
    └── designs/         → 内容归位后删除
```

## 6. intake/ 收件箱纪律（改造自 myagents_files）

**定位**：AI agent 执行任务时产出的**临时落客区**，但不再是"无限堆积的垃圾场"，而是**强制当天分流**的收件箱。

**规则**：
1. 所有 agent 运行时下载/生成的文件**先落 intake/**（agent 默认输出路径）
2. **每次会话结束前**，agent 必须执行分流：业务文件 → projects/docs、参考资料 → reference、垃圾 → 删除
3. intake/ **理想状态应保持空**（或仅含本次会话未完成项）
4. 周度巡检：intake/ 有文件 = 有未完成分流，列入待办

**实现**：把 myagents_files/ 重命名为 intake/，先做阶段 2 把 119 文件分流到各区，清空后即得干净的收件箱。

## 7. memory/ 重启机制

**现状**：4 条 cron 流水线死了 3 条，sessions 持续制造噪声。

**决策**：归档历史 + 重启核心机制。

| 流水线 | 处置 |
|---|---|
| `conversations/`（132 文件，截至 06-08 停更） | **归档**到 `memory/archive/conversations-pre-2026-06-09/`（只读），**重启 cron 写入** |
| `daily-letter/`（57 文件，截至 06-08 停更） | 归档到 `memory/archive/`，暂不重启（与 workspace 摘要重叠，待评估价值） |
| `events/`（10 文件） | 归档，暂不重启 |
| `sessions/`（64 噪声） | **删除**，目录改名 `gap-reports/` 或并入 `logs/` |
| `daily/`（碎片抽取） | **停掉 cron**（信息量远低于 workspace 摘要，重复劳动） |

**memory/ 清理后应只剩**：认知契约 + state.json + MEMORY/insights 索引 + thinking/(WAL缓冲) + context/(待办) + 重启后的 conversations/ + learnings/ + topics/(纯个人认知)。

**业务文档迁出**：projects/供应商画像 + 关键人画像（去重留一份）→ wiki/entities/；供应商管理工具箱/可视化体系 → docs/；topics/_archive/电销知识库 → docs/供应商/。

## 9. 长期维护机制（防止再次熵增）

1. **每周巡检**：intake/ 不空、workspace 超 2 周未归档、plans 有完结项 → 触发分流
2. **季度大整理**：执行 workspace→docs/archive 流转（约定已写明，需落实）
3. **引入工具时立规矩**：新 AI 工具/新 cron 必须先决定"它的产物落哪个区"，不另开新目录
4. **一概念一家**：发现同类内容散落多目录，立即合并，不留"并行体系"
