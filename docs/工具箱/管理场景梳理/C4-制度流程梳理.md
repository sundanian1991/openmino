# C4 制度/流程梳理

> 把脑子里的经验变成别人能照着做的制度文件，不写空话，只写能落地的。

## 触发信号

- 王易人说"你把XX流程整理一下"
- 发现某件事每次处理方式不一样，需要标准化
- 出了问题后发现"如果早有规定就不会出这事"
- 新同事入职，没有操作手册可看
- 现有制度过时了，和实际操作对不上
- 合规检查/审计要求补制度

## 核心问题

> 制度不是写出来的，是"把正在做的事说清楚"。核心动作：现状是什么 → 问题在哪 → 怎么改 → 怎么落地。

## 思考步骤

### 第1步：明确制度边界

**想什么**：这份制度要管什么？不管什么？

**检查什么**：管谁？（在营供应商，不含储备）管什么事？（日常运营，不含准入/清退）不管什么？（分量由策略组定）谁执行？（服务组供应商管理岗）谁审批？（服务组负责人审批，二级部门备案）

**判断什么**：边界说不清楚→先别写；边界太大→拆成多份；有重叠→先确认归属

**举例**：王易人说"整理一下供应商巡检流程"。边界：管什么=日常运营期间的现场巡检；不管什么=准入期的现场考察、合规专项检查；谁执行=年老师+质检团队；谁审批=王易人审批巡检计划。

### 第2步：还原现状（现在是怎么做的）

**想什么**：别从理想状态开始写，先还原现在实际在做的操作。

**检查什么**：

| 还原维度 | 查什么 | 怎么查 |
|---------|--------|--------|
| 触发条件 | 什么事触发？ | 回忆最近3次触发 |
| 操作步骤 | 具体做了什么？顺序？ | 回忆实际操作过程 |
| 参与角色 | 谁参与了？各做什么？ | 列出实际参与的人 |
| 产出物 | 最后出什么文档？ | 查最近的产出物 |
| 时间周期 | 从开始到结束多久？ | 按实际估算 |
| 判断标准 | 中间做了哪些判断？ | 回忆当时的决策依据 |

**判断什么**：
- "每次做法不一样" → 正是需要制度化的原因，列出来
- "某一步没人在做" → 可能是漏洞，标注
- "某一步多余但还在做" → 考虑删掉

**举例**：还原供应商约谈流程。触发：连续2个月后30%或A级骤降 → 操作：私下沟通→准备数据→发约谈邮件→约谈→整改通知→验收 → 时间：3-5工作日 → 产出物：约谈邮件、整改通知、验收记录。

### 第3步：找到现状中的问题

**想什么**：现状还原完了，哪里容易出错、遗漏、拖延？

**检查什么**：

用以下4类问题逐项检查：

| 问题类型 | 检查问题 | 典型问题 |
|---------|---------|---------|
| **标准模糊** | 有"看情况""酌情"的地方？ | "视情况决定是否约谈"——什么情况？ |
| **责任不清** | 有"大家觉得对方在做"的灰色地带？ | 整改验收谁发起？ |
| **时间缺失** | 有没有deadline的环节？ | "整改后跟进"——多久后跟？ |
| **信息断点** | 有没通知到该通知的人？ | 约谈后抄送军哥？有时抄有时不抄 |

**判断什么**：
- 标准模糊 → 写明量化标准
- 责任不清 → 每个环节指定唯一责任人
- 时间缺失 → 每个环节加deadline
- 信息断点 → 明确每个节点的通知对象

**举例**：约谈流程问题：(1) "视情况决定是否约谈" → 改成"连续2个月后30%必须约谈" (2) 验收责任不清 → 指定年老师发起 (3) "整改后跟进"无时间 → 加5工作日deadline (4) 抄送不统一 → 橙级以上抄送军哥。

### 第4步：设计改进方案

**想什么**：问题清楚了，怎么改？别设计完美但没人能执行的流程。

**检查什么**：

| 设计原则 | 检查方法 |
|---------|---------|
| **最小改动** | 只改有问题的环节，不动的别动 |
| **可执行性** | 每一步问：我现在能照着做吗？做不到就改 |
| **可检查性** | 每一步有没有明确的完成标志？ |
| **可追溯性** | 每一步有没有留痕？（文档/邮件/系统记录） |

**判断什么**：
- 涉及其他部门配合 → 先沟通确认再写
- 需要系统支持 → 标注"待实现"，先用手动流程
- 涉及审批权限变更 → 王易人确认，不能自己加

**举例**：约谈流程改进方案。

| 现状问题 | 改进方案 |
|---------|---------|
| "视情况约谈" | 明确触发：连续2个月后30%→必须约谈；A级骤降→48h内 |
| 验收责任不清 | 年老师发起验收，质检配合 |
| 跟进无时间 | 整改期限后5工作日内验收 |
| 抄送不统一 | 橙级以上抄送军哥，黄级不抄送 |

### 第5步：写成制度文件

**想什么**：前面的分析是给自己看的，现在要写成别人能看懂的文件。

**检查什么**（制度文件标准结构）：

| 章节 | 内容 | 注意事项 |
|------|------|---------|
| **目的** | 为什么要有这个制度 | 一句话，不写空话 |
| **适用范围** | 管谁、不管谁 | 必须明确 |
| **触发条件** | 什么情况下启动 | 量化标准 |
| **操作步骤** | 按顺序写每一步 | 每步：做什么→谁做→何时→产出什么 |
| **判断标准** | 中间需要决策的地方 | 给明确依据 |
| **权限说明** | 谁能决定什么 | 对照决策授权矩阵 |
| **附件/模板** | 用哪些模板 | 引用 official-doc 模板文件名 |

**判断什么**：
- 不了解情况的人能看懂吗？看不懂就是不清楚
- 有"加强管理""提高意识"这种废话？删
- 每个动词有没有明确主语？（谁整改？谁验收？）

### 第6步：确认落地路径

**想什么**：制度写了不等于落地了。怎么确保执行？

**检查什么**：

| 环节 | 做什么 | 频率 |
|------|--------|------|
| **发布** | 发给谁看？邮件还是会议宣贯？ | 一次性 |
| **培训** | 执行人是否清楚新流程？ | 发布时+人员变动时 |
| **试运行** | 先跑一遍，看有没有卡点 | 1-2周 |
| **反馈收集** | 执行中遇到什么问题？ | 每次执行后 |
| **迭代** | 根据反馈调整 | 发现问题时 |

**判断什么**：
- 涉及供应商配合 → 通知供应商（`supplier-operations.md`）
- 涉及审批流程变更 → 王易人确认
- 涉及系统改动 → 提需求，先用线下流程

## 关键变量

| 变量 | 判断标准 | 影响 |
|------|---------|------|
| **制度成熟度** | 首次制定/需更新/仅有口头惯例 | 首次从现状开始，更新从差异开始 |
| **涉及角色数** | 仅服务组/跨组/跨部门 | 角色越多确认流程越长 |
| **改动影响面** | 仅内部/影响供应商/涉及权限 | 影响供应商需通知，涉及权限需审批 |
| **紧急程度** | 审计合规（急）/常规优化（不急） | 急的先出最小可用版本 |
| **执行频率** | 高频（每周）/低频（每季度） | 高频写细，低频可粗 |

## 常见误区

| 误区 | 后果 | 正确做法 |
|------|------|---------|
| 从理想状态开始写 | 写出来的制度没人能执行 | 从现状出发，改有问题的部分 |
| 写成学术文章 | 没人看得懂 | 用大白话，用祈使句（"做X"，不是"应当做X"） |
| 覆盖所有corner case | 制度太复杂，正常流程都记不住 | 先覆盖80%常见场景，特殊情况事后处理 |
| 照搬其他公司模板 | 和实际操作脱节 | 基于自己的实际操作来写 |
| 写完不试运行 | 上线就出问题 | 新流程至少跑一遍再正式发布 |
| 制度写了不更新 | 半年后制度和实际操作又是两回事 | 每次执行发现问题就改，别攒着 |

## 决策输出

> **制度梳理报告模板**

| 维度 | 内容 |
|------|------|
| 制度名称 | 如"供应商日常巡检流程规范" |
| 适用范围 | 管谁、不管什么 |
| 现状描述 | 现在实际怎么做的 |
| 问题清单 | 标准模糊/责任不清/时间缺失/信息断点 |
| 改进方案 | 每条问题对应一个改进动作 |
| 落地计划 | 发布方式+培训+试运行+反馈 |
| 审批链 | 需谁审批、需谁知会 |

> **示例**：供应商预警约谈管理流程 | 适用：在营供应商日常预警约谈 | 问题：触发标准模糊+验收责任不清+无时间约束+抄送不统一 | 改进：量化触发+指定责任人+加deadline+分级抄送 | 审批：王易人审批→军哥备案
