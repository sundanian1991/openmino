# C4-制度流程梳理

> 需要更新制度或设计流程时，系统化梳理规则、权责和检查点

## 触发信号
- 发现现有规则有漏洞（供应商钻空子）
- 业务变化需新增规则（新项目/新产线）
- 流程效率低（多次重复工作）
- 审计检查发现缺失环节
- 领导要求"把制度细化一下"

## 核心问题
- 这个制度/流程要解决什么问题？
- 涉及哪些角色？权责如何划分？
- 关键决策点在哪里？判断标准是什么？
- 流程有几个阶段？每个阶段的输入输出是什么？
- 如何检查执行效果？（闭环验证）
- 需要哪些附件？（表格/模板/记录）

## 思考步骤

**Step 1：定义问题与目标（工具箱§十八）**
```
问题陈述：
当前：[描述现状，如：新供应商引入无标准流程]
问题：[具体问题，如：引入质量参差不齐，6个月达标率仅50%]
影响：[量化影响，如：浪费2-3个月时间+清退成本]

目标设定：
制度目标：[一句话，如：规范新供应商引入，提升6个月达标率至70%+]
流程目标：[具体可衡量，如：T+15标准周期，审计检查点100%覆盖]

范围界定：
适用范围：[哪些供应商/哪些业务线]
生效时间：[YYYY-MM-DD]
责任人：[年老师/易人哥/军哥]
```

**Step 2：角色与权责矩阵（RACI）**
```
角色定义：
- 年老师：发起人/执行者
- 王易人：审批人/监督者
- 军哥：重大事项审批人
- 供应商：执行配合方
- 刘伟佳/刘乾坤：相关方/协作者

RACI矩阵示例（新供应商准入）：
阶段         │ 年老师 │ 易人哥 │ 军哥 │ 供应商 │ 刘伟佳
─────────────┼────────┼────────┼──────┼────────┼──────
需求发起     │ R/A    │ —      │ —    │ I      │ C
需求拆解     │ R/A    │ —      │ —    │ I      │ I
寻源触达     │ R/A    │ —      │ —    │ C      │ I
方案审批     │ R      │ A      │ C    │ I      │ I
结果同步     │ R/A    │ —      │ —    │ I      │ C
供应商准备   │ I      │ —      │ —    │ R/A    │ I
内部操作     │ R      │ A      │ —    │ I      │ I
培训上线     │ R/A    │ —      │ —    │ R/A    │ I

符号：R=执行 Responsible / A=审批 Accountable / C=咨询 Consulted / I=知悉 Informed
```

**Step 3：流程阶段拆解（工具箱§三 + SOP参考）**
```
流程设计原则：
1. 阶段清晰：每个阶段有明确的起止标志
2. 输入输出：每阶段有输入物+输出物
3. 时效控制：每阶段有完成时限
4. 审计检查点：每阶段有可验证的完成标志

示例：新供应商准入8阶段（工具箱§三）
阶段1：需求发起（T+0）
  输入：需求说明书（5W2H）
  输出：寻源任务单
  动作：年老师填写任务单
  检查点：需求是否清晰？预算是否明确？
  时限：0.5天

阶段2：需求拆解（T+1）
  输入：寻源任务单
  输出：供应商画像（9维度）
  动作：年老师拆解需求为9维度要求
  检查点：画像是否可量化？是否覆盖全部维度？
  时限：1天

...（后续阶段同B1模板）

流程可视化（可选）：
用mcp-diagram生成流程图，标注：
- 阶段框（含输入/输出/时限）
- 决策 diamonds（是/否分支）
- 角色标注（谁负责）
```

**Step 4：关键决策点与判断标准（工具箱各章节）**
```
决策点清单：
决策点1：[供应商初筛通过/淘汰]
判断标准：
  ✓ 基础门槛4项全部满足
  ✓ 9维度总分≥70分
  ✗ 任一门槛不满足直接淘汰
输出：初筛结论报告

决策点2：[供应商引入/不引入]
判断标准：
  ✓ 初筛通过
  ✓ 有资源空余名额
  ✓ 集中度未超标
  ✗ 尾部供应商需特殊理由+上级审批
输出：准入审批意见

决策点3：[转正/延长/清退]
判断标准：
  ✓ 6个月排名前70% → 转正
  ✓ 接近达标 + 有改进 → 延长3个月
  ✗ 连续3个月后30%或诚信红线 → 清退
输出：TRN/EXT/CLR决策编号

每个决策点必须：
- 有明确的判断公式/标准
- 有通过的阈值
- 有输出物（审批记录/通知）
- 有负责人（谁决策）
```

**Step 5：闭环验证机制（工具箱§六）**
```
验证设计：
什么需要验证？
- 整改计划执行情况
- 新供应商里程碑达成
- 流程是否被遵守（审计检查点）

如何验证？
- 数据对比：目标vs实际
- 记录检查：是否有存档
- 现场核实：FCI巡检

验证标准：
- 目标达成≥80%
- 动作执行≥70%
- 持续性观察2个月

验证频率：
- 周度：整改进度（每周五）
- 月度：里程碑达成（每月初）
- 季度：流程合规审计（每季末）

验证输出：
- 通过：归档成功案例
- 部分通过：延长观察
- 未通过：升级处理
```

**Step 6：制度文件结构**
```
制度文件 = 规则 + 流程 + 模板 + 附件

结构：
1. 目的与范围（Why + Scope）
2. 定义与术语（名词解释）
3. 角色与权责（RACI矩阵）
4. 流程说明（阶段拆解 + 输入输出）
5. 判断标准（决策点 + 计算公式）
6. 闭环验证（如何检查 + 标准）
7. 附件（模板/表格/记录）

示例：供应商空间规范（工具箱§十八）
8个文件夹结构 + 6个填报模板 + 分步实施建议
每份模板：填报说明 + 截止时间 + 数据源
```

## 关键变量
- **问题类型**：规则缺失 / 流程低效 / 权责不清，决定解决方向
- **涉及角色**：RACI矩阵明确谁执行/谁审批/谁咨询/谁知悉
- **决策点数量**：关键决策3-5个，不宜过多
- **闭环验证**：每个流程阶段必须有检查点
- **时效控制**：每阶段明确时限，避免拖延

## 常见误区
- ❌ 只写"应该做什么"：不写"谁做+何时做+如何验证"
- ❌ 权责不清：多个R（都该做）或没有A（没人审批）
- ❌ 流程过长：超过5个阶段需简化，合并同类项
- ❌ 无检查点：流程走完不知道执行没执行
- ❌ 忽略例外场景：只设计正常流程，无异常处理
- ❌ 不配套模板：有流程无表格，执行不下去

## 决策输出

**制度设计卡**：
```
制度名称：[XXX管理制度]
要解决的问题：[具体问题+影响]
适用范围：[业务线/供应商类型/时间范围]

核心流程：[X个阶段]
关键决策点：[3-5个，每个含判断标准]
权责矩阵：[RACI表格]
闭环验证：[如何检查+标准+频率]

输出文件：
□ 制度正文（docx/md）
□ 流程附件（visio/diagram）
□ 模板附件（Excel/Word表单X个）
□ 说明文档（使用指南）

生效时间：[YYYY-MM-DD]
责任人：[年老师]
```

**流程设计检查清单**：
```
□ 流程阶段≤5个？（超过需合并）
□ 每阶段有明确起止标志？（输入/输出物）
□ 每阶段有时限？（T+X天）
□ 每阶段有审计检查点？（可验证的完成标志）
□ 决策点≤5个？（每个有判断公式）
□ RACI矩阵无多R或无A？
□ 有配套模板？（表格/记录）
□ 有闭环验证？（如何检查执行效果）
```

**制度文件示例结构**：
```
# 新供应商引入管理制度

## 1. 目的
规范新供应商引入流程，提升6个月达标率至70%以上。

## 2. 范围
适用于金条/企金/信用卡/财富四大产线新供应商引入。

## 3. 角色权责
[插入RACI矩阵]

## 4. 流程说明（8阶段T+15）
[阶段1-8表格，含输入/输出/时限/检查点]

## 5. 判断标准
- 初筛：4项门槛 + 9维度≥70分
- 引入：资源可用 + 集中度≤40%
- 转正：6个月前70% + 无合规问题
- 延长：接近达标 + 有改进计划
- 清退：连续3个月后30%或诚信红线

## 6. 闭环验证
- 试运行周报（供应商提交）
- 6月末评估（里程碑评判）
- 整改闭环率月度统计（≥70%）

## 附件
- 附件1：需求说明书模板
- 附件2：9维度评估表
- 附件3：供应商画像模板
- 附件4：SOP时间表（8阶段速查）
```

## 下一步衔接
- 新制度发布 → 内部邮件通知（【C2-邮件起草】）+ `internal.md`
- 流程培训 → 向相关同事讲解流程
- 配套工具 → 若需系统支持，提出需求给IT
- 执行跟踪 → 调用【A10-整改闭环跟踪】验证流程执行
- 定期复盘 → 调用【B4-月度复盘】评估制度效果
- 输出公文 → 调用 official-doc 的正式制度文件模板
