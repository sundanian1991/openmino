# 需求文档结构规范

> **来源**：供应商站点看板 V6 需求文档编写经验（2026-04-04）
> **类型**：playbook
> **标签**：#需求文档 #产品管理 #AI协作

---

## 核心洞察

需求文档要便于**AI或人工实现**，必须包含**四大支柱**：

1. **数据结构** — 数据从哪来、长什么样
2. **风险/功能定义** — 计算逻辑、阈值
3. **图表/组件清单** — 每个版本用什么图
4. **样式规范** — 色、字、距、角

缺少任何一环，实现都会卡住。

---

## 文档结构模板

```markdown
# [项目名] 需求文档

## 1. 概述
- 版本定位（谁用、什么时候用、为什么用）
- 共享组件（所有版本通用）
- 差异化特点（每个版本独有）

## 2. 数据结构
- RAW数据格式（字段说明+示例）
- 聚合引擎（merge函数说明）
- 筛选器逻辑（代码示例）

## 3. 风险/功能定义
- 每个维度的定义表
- 计算函数（可直接复制）
- 阈值说明

## 4. 各版本详细说明
- 版本A：图表组合清单
- 版本B：图表组合清单
- 版本C：图表组合清单

## 5. 样式规范
- 色彩方案（CSS Variables）
- 字体、间距、圆角

## 6. 交付物格式
- 技术要求（单文件/零依赖）
- 文件命名规范
- 验收标准（checklist）

## 7. 参考文件
- 已实现版本（提取共享组件）
- 风格参考（提取设计元素）
```

## 四大支柱详解

### 支柱1：数据结构

**目标**：回答"数据从哪来、长什么样"

**必须包含**：

| 内容 | 格式 | 示例 |
|------|------|------|
| RAW数据格式 | JSON对象+注释 | `{ id: 1, site: "颍州站点", ... }` |
| 聚合引擎 | 函数签名+返回值说明 | `mergeWorkplaces(data) → 职场数组` |
| 筛选器逻辑 | 可运行的代码 | `function getFiltered() { ... }` |

**最佳实践**：
- ✅ 提供真实数据示例（不是假的）
- ✅ 字段类型标注清楚（数字/字符串/数组）
- ✅ 必填字段标明

---

### 支柱2：风险/功能定义

**目标**：回答"怎么计算、阈值是多少"

**必须包含**：

| 内容 | 格式 | 示例 |
|------|------|------|
| 定义表 | Markdown表格 | \| # \| 风险维度 \| 对象 \| 定义 \| 阈值 \| |
| 计算函数 | 可运行的JavaScript代码 | `function calcTinySiteRisk(workplaces) { ... }` |
| 阈值说明 | 文字+代码注释 | `// 阈值：>30%` |

**最佳实践**：
- ✅ 提供可直接复制的代码（不要伪代码）
- ✅ 阈值在代码中标注清楚
- ✅ 返回值格式统一 `{ count, items, threshold }`

---

### 支柱3：图表/组件清单

**目标**：回答"每个版本用什么图、放哪里"

**必须包含**：

| 内容 | 格式 | 示例 |
|------|------|------|
| Tab结构 | 列表 | **Tab1 - 全局大盘**：KPI卡片+风险卡片+趋势图 |
| 图表清单 | 列表+说明 | 月度人力趋势图（折线图+面积图） |
| 核心特色 | 代码/截图 | 供应商风险评级表（HTML表格结构） |

**最佳实践**：
- ✅ 图表类型明确（折线图/饼图/热力图）
- ✅ 数据来源说明（用哪个聚合结果）
- ✅ 交互说明（点击下钻/筛选联动）

---

### 支柱4：样式规范

**目标**：回答"什么颜色、什么字体、多大间距"

**必须包含**：

| 内容 | 格式 | 示例 |
|------|------|------|
| 色彩方案 | CSS Variables | `:root { --bg: #f5f3f0; --accent: #c26d3a; }` |
| 字体 | CSS | `font-family: 'Inter', system-ui, sans-serif;` |
| 间距 | CSS | `padding: 1rem; gap: 0.5rem;` |
| 圆角 | CSS | `border-radius: 2px;` |

**最佳实践**：
- ✅ 用CSS Variables，方便统一修改
- ✅ 提供多版本对比表（V6-RP vs V6-Data）
- ✅ 标注核心设计Token（圆角2px、进度条3px）

## 交付物格式

### 技术要求

| 要求 | 说明 |
|------|------|
| 单文件HTML | 所有代码（HTML+CSS+JS）在一个文件中 |
| 零依赖 | 除ECharts外不依赖其他库 |
| 浏览器兼容 | Chrome 90+、Safari 14+、Edge 90+ |

### 文件命名

```
workspace/output/
├── [项目名]-v6-rp.html       （日常汇报版）
├── [项目名]-v6-data.html     （深度审计版）
├── [项目名]-v6-consult.html  （战略展示版）
└── [项目名]-v6-swiss.html    （精简版）
```

### 验收标准（checklist）

- [ ] 数据结构正确（RAW格式+聚合引擎）
- [ ] 风险维度计算正确
- [ ] 筛选器功能正常
- [ ] 各版本差异化图表正确实现
- [ ] 样式符合规范（色彩/字体/间距/圆角）
- [ ] 单文件HTML，零依赖
- [ ] 浏览器兼容性测试通过

## 避免的陷阱

| 陷阱 | 问题 | 解决方案 |
|------|------|----------|
| 只说"要做X" | 不清楚具体怎么做 | 提供计算函数代码 |
| 用"建议"、"可以考虑" | 模糊表述，无法执行 | 明确"必须实现"、"可选" |
| 缺少样式规范 | 实现风格不统一 | 提供CSS Variables |
| 缺少验收标准 | 不清楚怎么算完成 | 提供checklist |

---

## 易读性最佳实践

### ✅ 应该做

- 提供计算函数代码（直接可用）
- 提供样式规范（CSS Variables）
- 提供参考文件路径（可验证）
- 明确验收标准（可测试）
- 用表格呈现结构化信息

### ❌ 不应该做

- 用"建议"、"可以考虑"等模糊表述
- 只说"做一个好看的图表"
- 缺少代码示例（只给文字描述）
- 缺少参考文件（无法验证）

## 适用场景

- 任何需要交给AI实现的需求
- 任何需要多人协作的项目
- 任何需要保证实现质量的需求文档
