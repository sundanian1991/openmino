## 判断样本

### Q1 · 否定了什么

否定了**"用 tag 重叠度召回关系候选"**这个原方案。

具体特征：
1. 否定的是一个看起来合理的机制——两个资产如果有共同 tag，就认为它们有关系。这是关系图构建的常规做法
2. 否定的深层假设——"资产会被正确打上 tag，且 tag 能有效表达资产的主题相关性"
3. 实际症状：关系图为空，pickCandidates（候选选择函数）召回 0 个关系

### Q2 · 为什么否定

**机制层（停止信号命中）**：

原方案失败的根本原因是 **tag 字段在生产端就是空的**。这是一个数据质量传导问题：

1. 上游环节（LLM 提取）产出的资产，tags 字段大多为空——可能是因为提取 prompt 没强制要求 tag、或 LLM 判断"无明显 tag"时留空
2. 下游环节（relate 关系构建）依赖 tag 重叠召回候选——上游 tag 空，下游必然召回 0 个

AI 的默认偏好是**机制合理性偏向**——设计算法时倾向于选"看起来合理的机制"（tag 重叠是关系构建的经典方法），而不优先验证"这个机制依赖的前置数据是否存在"。这个偏好的机制：机制层面的"合理"比数据层面的"可用"更容易在编码时被验证。

**与用户标准的对齐**：用户的实际目标是"关系图要有内容"，不是"关系构建算法要合理"。tag 重叠机制再合理，产不出关系图就是失败。

**prompt 层**：如果最初设计 relate 模块时，先验证"现有资产的 tag 字段填充率"，会发现前置数据缺失，不会选 tag 重叠方案。AI 默认跳过"前置数据可用性检查"直接进入"机制设计"。

### Q3 · 保留了什么

**胜出方案**：embedding 相似度 top-K 召回 + 无 embedding 现场生成兜底。

关键判别特征：
1. **绕过 tag 依赖**——embedding 直接基于内容向量相似度，不需要 tag 这个中间字段
2. **兜底机制**——新资产没有 embedding 时现场调用 embedder 生成，保证流程不中断
3. 区别于 tag 重叠方案：embedding 不依赖上游字段填充质量，依赖内容本身
4. **降级路径清晰**——如果 embedding 也失效，可以回退到 tag 重叠（作为 fallback 而非主路径）

**让未来 AI 能据此生成的特征**：设计依赖型算法时，先验证依赖数据的存在性，再选机制；机制失败时，找绕过依赖字段的替代路径。

### Q4 · 真正关心的问题本质

- **表层需求**：让关系图有内容
- **实际校验**：**关系构建算法依赖的前置数据是否真实可用**。用户在校验的不是"算法是否合理"，而是"算法能否在这个数据现状下产出结果"

更深一层：这是一个关于**算法设计与数据现状的匹配判断**——再合理的算法，如果依赖的前置数据不存在，也是空转。判断算法可行性的标准不是"机制本身是否合理"，而是"机制依赖是否在当前数据现状下满足"。

### Q5 · 下次避免的误判

- **遇到设计依赖型算法（依赖某个字段/标记/标签），先实测该依赖数据的填充率**——判断标准：填充率低于 50% 时，不能选依赖该字段的算法，要找绕过依赖的替代路径
- **遇到"看起来合理"的算法方案，区分"机制合理性"和"数据可用性"**——AI 系统性偏好选机制合理的方案，但合理机制在数据现状下可能空转。验证顺序：先数据可用性，再机制合理性
- **遇到算法产出为空（关系图为空/召回 0 个/匹配失败），先诊断"是算法错还是依赖数据缺失"**——大多数情况下是后者。诊断方法：直接查依赖字段的实际填充情况
- **遇到主路径依赖缺失数据，设计降级链**——主路径（tag）→ 次路径（embedding）→ 兜底（现场生成）。不要只设计主路径不设计 fallback
