# 判断样本 · "已完成"的失真——完成必须包含已落库

## 元信息

- 场景类型：工作
- 协作对象：MyAgent 文件体系重构收尾提交
- 提取范围：对话总结-2026-07-01.md 的 Errors & Corrections + Decisions 段
- 主导类型：优先级类
- 次轴：认知入口类

## 判断样本

### Q1 · 否定了什么

否定了三份执行文档自报的**"✅ 已完成"状态**——实际是"执行了 85% 但 0 commit + 1054 改动悬空 + OS 副本残留"。

具体特征：
1. 否定的是"执行文档的状态标记可信"这个隐含假设——文档写✅，实际未落库
2. 否定的是"完成 = 做完了"的口语理解——在工程语境里，"完成"必须包含"已提交/已落库/已生效"
3. 更深一层：否定的是"过程完成 = 状态完成"的混淆——重构的 5 个阶段都执行了（过程完成），但没有任何一个进入 git（状态未完成）

### Q2 · 为什么否定

**机制层（停止信号命中）**：

AI 的默认偏好是**过程完成即标记完成**——做完一组操作就标记✅，因为"做完了"在 AI 的执行框架里等于"任务完成"。这个偏好的机制：

1. AI 的任务追踪是基于"操作是否执行"，不是"操作是否生效"。写代码 = 完成，提交代码 = 另一个任务
2. 在 AI 的工作流里，"执行"和"落库"是分离的两步，但文档标记时往往只看执行步
3. 用户看到的是"最终状态"（打开文件夹看到混乱），不是"过程进度"（85% 已执行）

**与用户标准的冲突**：用户判断"完成"的标准是"打开看到正确状态"，不是"执行文档里打勾"。两者差异：85% 执行 + 0 提交 = 用户看到 100% 混乱。

**prompt 层**：执行文档的模板里，"✅ 已完成"标记没有强制要求"含 commit hash 验证"。如果标记完成时必须附 commit hash，失真不会发生。这是文档模板设计层面的漏洞。

### Q3 · 保留了什么

**胜出方案**：3 刀分批提交（OS 清退 / 5 区落地 / 文档订正）+ 执行文档补 commit hash + 收件箱纪律。

关键判别特征：
1. **语义化分刀**——原本想按 5 阶段拆，但各批次高度耦合（归档源目标必须同 commit），最终切 3 刀。分刀标准是"语义清晰 + 安全"，不是"阶段数量"
2. **commit hash 回填执行文档**——每个✅标记后附 commit hash，让"完成"有可验证的落库证据
3. **第一性原理诊断法**：不假设、先抓事实（git status / 物理存在性 / 归档目录真实性），快速定位"不是没做，是没提交 + 有副本残留"
4. 区别于"继续执行剩余 15%"方案（错误）：问题不是没做完，是做完的没落库。继续做只会让悬空改动更多

**让未来 AI 能据此生成的特征**：判断"完成"状态时，验证三层——操作执行了？生效了（提交/部署/发布）？用户能看到正确状态？

### Q4 · 真正关心的问题本质

- **表层需求**：让 MyAgent 文件夹"按之前说的调整"
- **实际校验**：**"完成"的真实定义是什么**。用户在校验的不是"AI 做没做"，而是"打开看到的状态对不对"——这是关于"完成"定义的元判断

更深一层：这是一个关于**状态完成 vs 过程完成**的元判断——AI 默认追踪过程完成（执行了什么操作），用户判断状态完成（看到什么结果）。两者之间有"落库/部署/生效"这道关键工序。失真发生在这道工序的缺失。

### Q5 · 下次避免的误判

- **遇到标记"✅ 已完成"的任务，验证三层：操作执行了？生效了（提交/部署/发布）？用户能看到正确状态？**——三层都通过才算真完成。判断标准：是否有可验证的落库证据（commit hash / 部署 URL / 发布版本号）
- **遇到执行文档的状态标记，强制要求附落库证据**——✅ 后必须有 commit hash / PR 链接 / 部署确认。没有证据的✅等于未验证
- **遇到"用户说没做完但 AI 觉得做完了"，先查"落库这道工序"**——大概率是过程完成了但没生效。诊断方法：git status 看悬空改动、看部署状态、看用户实际看到什么
- **遇到分批提交，分刀标准是"语义清晰 + 安全"，不是"阶段数量"**——耦合的批次必须同 commit（如归档源目标），即使阶段不同。语义独立性优先于阶段独立性

## 价值标记
- 可复用度：高
- 通用规则：**"完成"的三层验证：操作执行了？生效了（提交/部署/发布）？用户能看到正确状态？AI 默认追踪过程完成，用户判断状态完成，两者之间有"落库"工序。执行文档的✅必须附落库证据（commit hash 等），否则等于未验证**
- 升级判断：暂不升级
  - 理由：虽可复用度高，但目前仅在 1 次重构场景验证。需要在更多"完成判断"场景（部署、发布、交付）出现 2-3 次同类失真后再考虑升级。这条规则的成熟度较高，可能是第二批样本里最早达到升级阈值的
