# 职场资料分类与知识可视化工作流

> 版本：v1.0 | 定位：把碎片信息变成可检索、可复用、可传播的组织资产

---

## 0. 核心认知

职场中有三类东西最容易混淆，导致资料越积越乱：

- **形态（Format）**：Wiki、Slide、信息图、表格——只是容器/外壳
- **场景（Scene）**：感知、决策、计划、执行、沉淀、对外——是目的
- **加工深度（Depth）**：从剪藏到产品化——是价值层级

**分类混乱的根因**：很多人按形态或来源分类（"这是PPT""这是论文"），而不是按"加工深度+使用场景"分类。

本文给出两套交叉坐标：
1. **职场6环节**：回答"这份资料在什么业务环节产生价值"
2. **加工深度L0-L5**：回答"这份资料被消化到什么程度、值不值得留"

---

## 1. 职场资料的 6 个场景环节

职场是一个**"感知 → 决策 → 计划 → 执行 → 沉淀 → 对外"**的循环。

### ① 感知层：外面发生了什么？

| 资料类型 | 场景 | 核心特征 |
|:---|:---|:---|
| 行业研究报告 | 进入新市场前判断赛道 | 宏观、第三方视角、数据密集、结论前置 |
| 竞品分析 | 产品设计前看对手怎么做 | 对标矩阵、功能逐项对比、差异化缺口 |
| 数据看板/仪表盘 | 日常监控业务健康度 | 实时、可视化、异常自动标红 |
| 用户调研报告 | 产品改版前理解用户痛点 | 定性+定量混合，有用户原话 |
| 政策法规扫描 | 合规驱动型行业预判风险 | 法条映射业务影响，带时间线 |
| 新闻情报剪报 | 追踪行业动态、对手动作 | 短平快、事实为主、需标注来源和日期 |

### ② 决策层：我们要不要做？怎么做？

| 资料类型 | 场景 | 核心特征 |
|:---|:---|:---|
| 立项报告/可行性研究 | 申请资源启动新项目 | 投入产出比、风险评估、Go/No-go |
| 方案选型评估 | 技术栈/供应商/路径选择 | 多维度打分表、推荐结论 |
| 预算/资源申请 | 向管理层要钱/要人 | 精确到科目、关联业务目标 |
| 商业计划书 BP | 融资或内部孵化新项目 | 故事线：痛点→方案→商业模式→财务预测 |
| **独立阅读型报告** | 让决策者不问你就能独立判断 | 信息密度高、逻辑自洽、附录详尽 |

> **关键区分**：独立阅读型报告的核心特征是"**闭环自足**"——不需要讲者补充， reader-alone 就能做出判断。

### ③ 计划层：谁、在什么时间、做到什么程度？

| 资料类型 | 场景 | 核心特征 |
|:---|:---|:---|
| 项目计划书/WBS | 项目启动会对齐范围和里程碑 | 时间线、责任人、交付物、依赖关系 |
| 产品需求文档 PRD | 产品经理向研发/设计传递需求 | 用户故事、验收标准、优先级、异常流程 |
| 技术方案/架构设计 | 研发确定怎么实现 | 模块划分、接口定义、技术选型、性能指标 |
| 标准操作程序 SOP | 重复性工作的最佳执行路径 | 步骤清单、判断分支、异常处理、输出模板 |

### ④ 执行层：现在到哪了？卡在哪了？

| 资料类型 | 场景 | 核心特征 |
|:---|:---|:---|
| 周报/月报/季报 | 例行向上和平级同步进展 | 结构化（进展/风险/下周计划）、数据量化 |
| 项目进度更新 | 非例行的关键节点汇报 | 红绿灯状态、阻塞项、需要的支持 |
| **会议纪要/Recap** | 会议结束后 24h 内同步结论 | 决策项、Action item（谁+做什么+何时）、待确认 |
| 日报/站会同步 | 敏捷团队每日快速对齐 | 极简（昨天/今天/阻塞），3分钟内阅读 |
| 任务工单/Ticket | 具体工作项的指派和追踪 | 可验收、可指派、有截止时间、状态流转 |

### ⑤ 沉淀层：这次学到了什么？下次怎么更快？

| 资料类型 | 场景 | 核心特征 |
|:---|:---|:---|
| 复盘报告/Post-mortem | 项目结束后总结成败 | 不问责只问因；Timeline 还原；可落地改进项 |
| 操作手册/使用指南 | 新人上手或系统上线后 | 步骤截图、FAQ、常见错误、联系谁 |
| 最佳实践/案例库 | 把个人经验变成组织能力 | 背景→挑战→动作→结果，可复用的方法论 |
| 培训课件 | 系统性能力提升 | 学习目标、案例、练习、考核 |
| **阅读笔记/知识卡片** | 个人或团队的知识积累 | 原文摘录+自己的理解+可关联的场景 |
| **知识库 Wiki 词条** | 半衰期较长的参考信息 | 可检索、可更新、可协作、结构化 |
| 信息图/知识模型图 | 把复杂关系压缩成一眼可读的形态 | 空间逻辑替代线性逻辑、可跨场景复用 |

### ⑥ 对外层：外人怎么看我们？怎么说服他们？

| 资料类型 | 场景 | 核心特征 |
|:---|:---|:---|
| 品牌宣传册/One-pager | 展会、官网、销售第一次见面 | 一句话价值主张、视觉锤、联系方式 |
| **销售 Deck/提案** | 向客户提案或竞标 | 客户痛点开头、方案匹配、ROI、成功案例 |
| 产品白皮书 | 技术型客户的深度说服 | 行业痛点、技术原理、架构图、性能数据 |
| 投标书/应标文件 | 响应客户招标 | 严格按标书结构、资质证明、点对点应答 |
| 新闻稿/公关声明 | 产品发布或危机回应 | 媒体视角、金句可引用、多渠道适配 |
| 招聘 JD | 吸引目标候选人 | 不是岗位说明书，是"我们为什么值得来" |
| **现场演示 Deck** | 配合讲者的实时说服 | 信息密度低、每页一个观点、依赖口语补充 |

> **关键区分**：现场演示 Deck ≠ 独立阅读型报告。演示 Deck 是"讲者工具"，阅读报告是"读者工具"。

## 2. 加工深度模型（L0-L5）

这条坐标回答：**"这份资料被消化到什么程度、值不值得留"**。

| 层级 | 名称 | 定义 | 典型形态 | 是否算沉淀 | 半衰期 |
|:---|:---|:---|:---|:---|:---|
| **L0** | 原始捕获 | 搬运，未经过任何加工 | 微信收藏、浏览器书签、PDF丢进文件夹、邮件归档 | ❌ 不算 | 极短 |
| **L1** | 初步标记 | 做了高亮/批注/标签，但未转述 | 论文划线、MarginNote批注、Readwise同步、标签分类 | ❌ 不算 | 短 |
| **L2** | 个人转述 | 用自己的话重述了核心观点 | 阅读笔记、文献卡片、双链笔记中的原子笔记、摘要 | ⚠️ 个人沉淀 | 中 |
| **L3** | 结构化整理 | 建立了分类/对比/时间线/归因框架 | 竞品矩阵、论文综述、行业演变时间线、数据对比表、脑图 | ✅ 算沉淀 | 较长 |
| **L4** | 知识重构 | 提炼出模型化、原理化的独立框架 | 自建分类体系、决策树、流程图、思维模型、检查清单 | ✅ 核心沉淀 | 长 |
| **L5** | 产品化输出 | 可独立交付/传播，脱离作者仍能运转 | 信息图、白皮书页、培训课件、标准Wiki词条、品牌物料、销售工具包 | ✅ 组织级沉淀 | 极长 |

### 如何判断自己处于哪一层？

问自己三个问题：

1. **如果半年后我忘了这件事，这份资料还能不能让我快速恢复认知？**
   - L0-L1：不能，你只会记得"我好像看过"
   - L2：勉强能，但可能读不懂当时的思路
   - L3-L4：能，因为结构在那里
   - L5：不仅能，别人也能

2. **这份资料离开了我（离职/转岗）还有价值吗？**
   - L0-L2：几乎没有
   - L3：有一定价值，但需要有人解释背景
   - L4-L5：独立有价值

3. **我下次遇到类似问题，能直接复用吗？**
   - L0-L1：不能，要重新读原始材料
   - L2：能复用观点，但不能复用框架
   - L3-L5：能复用框架，甚至直接复用产出物

## 3. 三维定位法（形态 × 场景 × 深度）

拿到任何资料，用这三个问题定位：

| 维度 | 问题 | 常见选项 |
|:---|:---|:---|
| **场景/目的** | 它是用来干嘛的？ | 感知 → 决策 → 计划 → 执行 → 沉淀 → 对外 |
| **加工深度** | 它是原始材料还是已被重构？ | L0 捕获 → L1 标记 → L2 笔记 → L3 整理 → L4 重构 → L5 产品 |
| **形态载体** | 它用什么容器装？ | 文档 / 表格 / Slide / Wiki / 信息图 / 视频 / 音频 / 数据库 |

### 举例定位

| 资料 | 场景 | 深度 | 形态 |
|:---|:---|:---|:---|
| 微信收藏了10篇AI论文链接 | ① 感知 | L0 原始捕获 | 链接列表 |
| 论文高亮同步到Readwise | ① 感知 | L1 初步标记 | 软件同步 |
| 写了500字论文摘要+自己的理解 | ① 感知 | L2 个人转述 | 笔记文档 |
| 做了5个AI模型的成本性能对比表 | ① 感知 | L3 结构化整理 | 表格 |
| 提炼出"AI选型决策树"并画成流程图 | ② 决策 | L4 知识重构 | 信息图 |
| 把决策树+对比表更新进团队Wiki | ② 决策 | L5 产品化输出 | Wiki词条 |
| 销售拿着Wiki内容去客户现场讲方案 | ⑥ 对外 | L5 产品化输出 | Slide |
| 项目结束后写的复盘文档 | ⑤ 沉淀 | L3 结构化整理 | 文档 |
| 把复盘提炼成"XX类项目避坑检查清单" | ⑤ 沉淀 | L4 知识重构 | 表格/信息图 |
| 把检查清单做成新人培训PPT | ⑤ 沉淀 | L5 产品化输出 | Slide |

## 4. "把一切信息可视化"完整工作流

这不是一个资料类型，而是一个**知识产品化工程**：把组织内散落在 L0-L2 的原始信息，通过标准化加工，提升到 L4-L5，形成可复用的认知资产。

### 4.1 工作流总览

```
输入层                加工层              重构层              输出层              维护层
┌─────────┐        ┌─────────┐        ┌─────────┐        ┌─────────┐        ┌─────────┐
│ L0 捕获  │  ──▶  │ L1 标记  │  ──▶  │ L2-L3   │  ──▶  │ L4-L5   │  ──▶  │ 更新   │
│         │        │         │        │ 整理    │        │ 可视化  │        │ 归档   │
└─────────┘        └─────────┘        └─────────┘        └─────────┘        └─────────┘
 原始信息            加标签/高亮         笔记+结构化         模型图/信息图        定期Review
 新闻/论文/数据      统一命名            分类/对比/时间线    Wiki/Slide          版本管理
```

### 4.2 输入层（L0-L1）：捕获规范

#### 捕获什么？

按**6场景×3形态**来定义你的捕获范围：

| 场景 | 捕获内容举例 | 常见来源 |
|:---|:---|:---|
| ① 感知 | 行业报告、竞品更新、政策文件、技术论文、数据发布 | 微信公众号、36氪、知乎、arXiv、政府网站、财报 |
| ② 决策 | 立项模板、ROI案例、预算范例、方案框架 | 内部历史项目、咨询公司模板、投资人反馈 |
| ③ 计划 | 优秀PRD、技术方案、项目计划范例 | 内部Wiki、GitHub、技术博客 |
| ④ 执行 | 高效会议纪要、周报结构、项目进度模板 | 内部最佳实践、管理书籍 |
| ⑤ 沉淀 | 复盘框架、SOP模板、知识模型 | 课程、书籍、咨询方法论 |
| ⑥ 对外 | 优秀销售Deck、品牌案例、白皮书结构 | 竞品官网、行业峰会资料 |

#### 怎么捕获？

建议采用**"单一入口+自动分发"**策略，避免散落在微信、邮件、浏览器、本地文件夹四处：

| 信息类型 | 推荐工具/方式 | 关键动作 |
|:---|:---|:---|
| 网页/文章 | 剪藏工具（Notion Web Clipper / Cubox / Matter） | 整页剪藏，保留原始URL和日期 |
| PDF/报告 | 统一丢进指定网盘文件夹（按命名规范） | 不修改原文件名，用文件夹结构分类 |
| 微信文章 | 发送到"文件传输助手"再批量剪藏，或直接用工具收藏 | 不要只点"收藏"，收藏=信息坟墓 |
| 邮件/聊天记录 | 关键结论截图或复制到笔记 | 保留上下文（谁说的、什么时候、关于什么） |
| 线下会议/讲座 | 录音转文字（飞书妙记/讯飞）+ 人工标重点 | 24小时内完成转写和初标 |
| 数据/表格 | 直接入库（飞书多维表格/Notion Database/Airtable） | 标注数据来源、更新频率、负责人 |

#### 命名规范（核心！）

命名规范是**知识管理的地基**。推荐采用以下格式：

**格式A：对外资料（报告/论文/行业文件）**

```
[日期]_[来源]_[主题]_[类型]_[版本]
```

示例：
- `20250605_艾瑞_中国AI大模型市场研究报告_行业报告_v1.pdf`
- `20250603_工信部_生成式AI服务管理暂行办法_政策法规_v1.pdf`
- `20250520_McKinsey_企业AI转型案例集_案例库_v2.pptx`

**格式B：内部资料（会议/项目/复盘）**

```
[日期]_[项目/部门]_[主题]_[类型]_[版本]_[作者]
```

示例：
- `20250605_客服外包项目_供应商选型评估_决策报告_v1_张三.pdf`
- `20250604_电销团队_5月业绩复盘_复盘报告_v1_李四.docx`
- `20250603_市场部_618品牌传播方案_计划书_v3_王五.pptx`

**格式C：原始数据/素材**

```
[日期]_[主题]_[数据类型]_[来源]_[版本]
```

示例：
- `20250605_27家供应商合同清单_表格_法务部_v1.xlsx`
- `20250601_竞品A产品功能截图_图片_官网_v1.png`
- `20250528_用户访谈录音_音频_第三方_v2.mp3`

**命名原则：**
1. **日期前置**：文件系统默认按名称排序时，时间线自然呈现
2. **来源明确**：知道是谁生产的，方便溯源和核实
3. **主题具体**：不要用"临时文件""资料"这种无意义词
4. **类型标注**：一眼知道这是什么（报告/表格/录音/截图）
5. **版本管理**：修改过一次的必须加v2，避免"最终版""最终最终版"乱象

### 4.3 加工层（L2-L3）：从笔记到结构化

#### L2：个人转述（原子笔记）

**核心动作**：用自己的话重述，不是复制粘贴。

每条原子笔记建议包含：

```markdown
## 原子笔记模板

**标题**：一句概括（不是原文标题）
**来源**：作者/机构/URL/日期
**核心观点**：1-3句话，用自己的话讲
**原文摘录**：关键金句或数据（标注页码/段落）
**我的理解**：这和我的工作/项目有什么关系？
**关联标签**：#感知/竞品 #场景/决策 #行业/AI
**关联笔记**：[[相关笔记A]] [[相关笔记B]]
```

#### L3：结构化整理（建立框架）

当积累了一定数量的原子笔记后，开始建立框架。常见的结构化形态：

| 结构化形态 | 适用场景 | 工具建议 |
|:---|:---|:---|
| **对比矩阵** | 竞品分析、方案选型、产品功能对比 | 表格（Excel/飞书/Notion） |
| **时间线** | 行业演变、项目历程、政策变迁 | 横向时间轴（Miro/FigJam/ProcessOn） |
| **分类树/脑图** | 知识分类、需求拆解、问题归因 | 思维导图（XMind/MindNode） |
| **流程图** | 业务流程、审批链路、用户旅程 | 流程图（ProcessOn/Draw.io） |
| **关系图谱** | 利益相关方、系统依赖、因果网络 | 关系图（Obsidian Graph View/Gephi） |
| **数据仪表盘** | 业务监控、KPI追踪、异常发现 | BI工具（飞书BI/Tableau/Metabase） |

**关键原则**：L3 的核心是"**发现模式**"——不是罗列信息，而是看出信息之间的关系（谁比谁快、谁先谁后、谁导致谁）。

### 4.4 重构层（L4）：模型化

L4 的标志是：**你创造了一个原本不存在的框架，它能解释多个案例**。

常见 L4 产出物：

| 产出物 | 定义 | 示例 |
|:---|:---|:---|
| **决策框架** | 面对某类选择时的判断标准 | "AI工具选型四象限：成本×合规×效果×实施难度" |
| **检查清单** | 避免重复踩坑的标准化提问 | "供应商合同签署前12项核查清单" |
| **分类体系** | 对混沌领域进行系统切分 | "客户服务外包合同的5种风险类型与对应条款" |
| **流程模型** | 把模糊经验变成可复制的步骤 | "从需求到上线的7步审查流程" |
| **概念模型** | 用隐喻或图形解释复杂机制 | "电销团队产能漏斗：流量→接通→意向→成交" |

**怎么判断到了L4？**
- 你的框架能否**预测**下一个案例的结果？
- 你的框架能否**指导**一个没有经验的人做出合格决策？
- 别人是否开始**引用**你的框架？

### 4.5 输出层（L5）：产品化与可视化

L5 的标志是：**脱离原始上下文，独立产生价值**。

#### 信息图的设计原则

信息图不是"把文字变成图"，而是**把线性逻辑变成空间逻辑**。

| 原则 | 说明 | 反例 |
|:---|:---|:---|
| **一图一模型** | 每张图只表达一个核心关系 | 一张图塞满所有信息，变成海报而非信息图 |
| **空间即语义** | 上下、左右、远近、包围都有含义 | 随意排列，观众无法从位置读取关系 |
| **视觉锚点** | 用颜色/大小/形状建立层次 | 所有元素一样大，没有焦点 |
| **数据有来源** | 每个数字标注出处和日期 | "据统计""行业平均"这种模糊表述 |
| **自解释性** | 不需要旁边有人讲解 | 必须配500字说明才能看懂 |

#### L5 产出的形态对照表

| 原始信息类型 | L3 结构化 | L4 重构 | L5 可视化产出 |
|:---|:---|:---|:---|
| 10篇行业报告 | 行业趋势时间线 | "行业演变三段论"模型 | **演变路径信息图** + Wiki词条 |
| 竞品功能对比表 | 功能矩阵 | "竞品战略定位四象限" | **定位矩阵图** + 销售话术卡 |
| 用户访谈记录 | 痛点分类树 | "用户决策旅程地图" | **旅程地图信息图** + 设计原则清单 |
| 项目复盘文档 | 成败因素归因表 | "XX项目成功公式" | **成功公式图** + 培训课件 |
| 数据仪表盘 | 关键指标关联分析 | "业务健康度诊断模型" | **诊断流程信息图** + 自动预警规则 |
| 政策法规文件 | 合规要求映射表 | "合规风险雷达图" | **雷达图** + 自检清单 |

#### 可视化风格建议

基于你的偏好（Claude博客/贝恩灰橙风格），统一一套视觉规范：

| 元素 | 规范 |
|:---|:---|
| 主色调 | 暖橙 `#CC6B1E` |
| 背景色 | 暖白 `#FAF8F5` 或 `#FFF8F0` |
| 辅助灰 | 中性灰 `#B8B8B8` |
| 字体 | 无衬线（Noto Sans SC / PingFang SC / Microsoft YaHei） |
| 节点样式 | 扁平矩形、圆角、无阴影 |
| 连线样式 | 实线/虚线区分关系强弱，箭头表达方向 |
| 信息密度 | 中高（拒绝大量留白） |
| 标题风格 | 标题即结论，去装饰化 |

### 4.6 维护层：更新节奏与归档

知识资产会腐烂。必须建立维护机制。

#### 更新频率建议

| 资产类型 | 更新频率 | 触发条件 | 负责人 |
|:---|:---|:---|:---|
| 行业数据看板 | 实时/每日 | 数据源更新 | 数据运营 |
| 竞品情报库 | 每周 | 竞品有重大发布/融资/改版 | 产品经理 |
| 决策框架/检查清单 | 每季度 | 实际使用中发现漏洞或新场景 | 业务负责人 |
| 培训课件 | 每半年 | 业务逻辑有重大变化 | L&D/业务专家 |
| 信息图/知识模型图 | 每半年 | 底层假设被颠覆 | 知识管理负责人 |
| Wiki 词条 | 按需 | 发现错误或过时时 | 全员的义务（谁发现谁修） |

#### 归档策略

不是所有东西都要永远保留。建立**退出机制**：

```
活跃库（Active）      参考库（Reference）     归档库（Archive）
   ↓ 每周Review          ↓ 每季度Review         ↓ 每年Review
  高频使用              偶尔查阅              历史记录
  当前项目              过往项目经验            已过时但可能有法律/审计价值
  正在迭代的框架         稳定的SOP和模板         旧版本文件（保留但标记deprecated）
```

**归档命名**：在文件名前加 `[ARCHIVED]` 或移入 Archive 文件夹，并在Wiki中标注"本页面已归档，最新版本见XXX"。

## 结语

这套体系的终极目标不是"分类整齐"，而是：**当组织中任何一个人需要某个信息时，他能在 30 秒内确定它是否存在、在哪里、值不值得信、能不能直接用。**

分类是手段，**降低认知摩擦**才是目的。
