# 供应商日常管理执行手册 — 04-供应商沟通

> 来电处理、日常对接、异常应对的全流程操作指引
>
> 核心目标：沟通有节奏、有记录、有闭环，不被动接球
>
> 工具箱引用：§十（沟通规范）、§十六（异常信号）、§十七（甲方责任场景）
>
> 版本：V1.0 | 2026-04-23

---

## 一、触发条件

供应商沟通贯穿日常管理全程，以下场景需要主动或被动沟通：

| 场景 | 触发信号 | 紧急程度 | 渠道 |
|------|----------|----------|------|
| 供应商来电/微信 | 供应商主动联系 | 按内容判断 | 电话/微信 |
| 群内@你 | 供应商在群内提问 | 30分钟内回复 | 群消息 |
| 异常报备 | 出勤/业绩/人员异常 | 按级别判断 | 电话+书面 |
| 分量调整通知 | 月度分量变动 | 常规 | 书面通知 |
| 月度排名沟通 | 排名结果需要传达 | 常规 | 约谈/书面 |
| 新项目需求 | 同事提出寻源需求 | 按优先级 | 内部沟通 |

## 二、日常对接规范

> 建立稳定的沟通节奏，不临时抱佛脚。

### 2.1 响应要求

| 场景 | 响应时间 | 渠道 | 说明 |
|------|----------|------|------|
| 群消息 | **30分钟内**回复 | 群消息 | 工作时间内的基本要求 |
| 重要事项 | **2小时内**书面+电话 | 电话+群内补发 | 挂断后群内补发书面说明 |
| 紧急事项 | **立即**电话 | 电话先行 | 书面后补 |
| 文件/资料 | 按要求时限 | 邮件/共享文档 | 按模板格式提交 |

### 2.2 日常沟通内容

| 事项 | 频率 | 渠道 | 责任方 |
|------|------|------|--------|
| 出勤数据确认 | 每日 | 群内/共享表格 | 供应商对接人 |
| 业绩数据确认 | 每日 | 群内 | 供应商对接人 |
| 排班信息 | 每日 | 共享表格 | 供应商对接人 |
| 异常事项报备 | 事件触发 | 电话+书面 | 供应商对接人/负责人 |
| 周报提交 | 每周五 | 邮件/共享文档 | 供应商对接人 |
| 月度总结提交 | 次月3日前 | 邮件 | 供应商负责人 |

### 2.3 群昵称规范

| 项目 | 要求 |
|------|------|
| 格式 | `职场-职务-姓名` |
| 示例 | `金条-站点负责人-张三` |
| 时限 | 首次要求后**24小时内**完成修改 |
| 未完成处理 | 清除出群，由站点负责人重新申请入群 |

## 三、来电处理

> 供应商来电是高频场景，需要有框架地应对。

### 3.1 常见来电类型

| 类型 | 典型内容 | 处理方式 |
|------|----------|----------|
| **要量** | "能不能多给点名单/量" | 用产能双维度判断，引导看自身问题 |
| **要支持** | "需要培训/系统/资源支持" | 记录需求，评估是否合理，协调资源 |
| **名单问题** | "名单质量不好/不够" | 区分甲方责任 vs 供应商问题 |
| **结算问题** | "结算有争议/延迟" | 记录问题，协调结算团队 |
| **异常报备** | "人员离职/系统故障" | 按异常分级处理 |
| **情绪表达** | "压力大/不公平" | 倾听+记录，不轻易承诺 |

### 3.2 处理框架

| 步骤 | 操作 | 要点 |
|------|------|------|
| 1 | 接听/了解诉求 | 让对方说完，不打断 |
| 2 | 分类判断 | 要量/要支持/异常/情绪 → 不同处理路径 |
| 3 | 区分责任归属 | 甲方责任 vs 供应商责任（§十七） |
| 4 | 给出回应 | 能承诺的承诺，需评估的约定回复时间 |
| 5 | 记录存档 | 沟通内容、承诺事项、跟进计划 |

### 3.3 甲方责任场景

> 以下场景属于甲方责任，供应商反馈时需正面回应：

| 场景 | 甲方责任表现 | 处理方式 |
|------|-------------|----------|
| 名单调整 | 你要求加人/调名单 | 承认甲方调整，给出恢复计划 |
| 产线切换 | 你调产线/产品 | 评估影响，给出过渡方案 |
| 系统问题 | 系统故障影响拨号 | 联系技术团队，给出修复时间 |
| 目标调整 | 你临时调高目标 | 说明原因，协商合理目标 |

### 3.4 供应商自身问题

> 以下场景属于供应商责任，需引导其正视：

| 场景 | 表现 | 回应方式 |
|------|------|----------|
| 管理松弛 | 主管松懈、排班不合理 | "你打算怎么调整？" |
| 人员流失 | 大面积离职 | "流失原因是什么？你准备怎么补？" |
| 转化率低 | 名单给了但转化不行 | "同类名单 others 转化率多少？" |
| 响应慢 | 群消息回复延迟 | "需要什么支持才能跟上？" |

### 3.5 常见认知误区

- 替供应商想解决办法 → 应引导式设问："你觉得最难的是哪一步？"
- 不区分甲方责任 vs 供应商责任 → 混淆导致处理不当
- 承诺超出权限的事项 → 不能承诺的明确说"需要评估"

## 四、异常应对

> 异常分级处理，不一刀切。

### 4.1 异常分级

| 级别 | 定义 | 典型场景 | 响应时间 |
|------|------|----------|----------|
| **一般异常** | 对业务影响有限 | 个别人迟到、系统偶发卡顿 | 当日处理 |
| **重要异常** | 对业务有明显影响 | 大面积请假、系统故障、批量投诉 | **2小时内** |
| **紧急异常** | 对业务/安全造成严重威胁 | 信息安全事件、安全事故 | **立即** |

### 4.2 异常处理流程

| 级别 | 处理流程 | 升级路径 |
|------|----------|----------|
| 一般异常 | 站点自行处理 → 群内报备 → 记录台账 | 站点自行处理 |
| 重要异常 | 立即报备（2h内）→ 供应商管理介入 → 制定方案 → 执行 → 闭环 | 升级至供应商管理岗 |
| 紧急异常 | 立即电话报备 → 启动应急响应 → 联合处理 → 事后复盘 | 升级至供应商管理岗+业务对接人 |

### 4.3 异常记录要求

每条异常须记录：

- 异常发生时间
- 异常类别与级别
- 影响范围（涉及人员/业务量）
- 处理措施
- 处理结果
- 根因分析与改进措施
- 闭环时间

### 4.4 关键异常信号

> 以下信号出现时，需主动联系供应商了解情况：

| 信号 | 异常标准 | 动作 |
|------|----------|------|
| 出勤异常 | 有效出勤率 < 85% | 问供应商出勤情况 |
| 业绩断崖 | 周GMV比近4周均值下降 > 20% | 按A级骤降逻辑立即约谈 |
| 流失加速 | 单周离职 > 5% | 了解离职原因 |
| 沟通频次下降 | 群内互动减少50%+ | 主动联系了解情况 |
| 响应延迟 | 群消息回复超过30分钟 | 提醒响应要求 |

## 五、信息报送管理

> 确保供应商按规范报送信息，数据及时准确。

### 5.1 报送体系

| 报送类型 | 频率 | 报送时间 | 报送人 | 格式 |
|----------|------|----------|--------|------|
| 每日出勤 | 每日 | 10:00前 | 对接人 | 出勤报送表 |
| 每日排班 | 每日 | 下班前 | 对接人 | 共享表格 |
| 每日业绩 | 每日 | 下线前 | 对接人 | 群内发送 |
| 周报 | 每周 | 周五下班前 | 对接人 | 周报模板 |
| 月度总结 | 每月 | 次月3日前 | 负责人 | 月度总结模板 |
| 人员花名册 | 实时 | 变动后3日内 | 对接人 | 共享表格 |

### 5.2 逾期处理

| 逾期次数 | 处理方式 |
|----------|----------|
| 第1次 | 群内提醒，要求当日补交 |
| 第2次 | 书面催办，抄送供应商负责人 |
| 连续3次 | 纳入配合度评分，约谈时提出 |

### 5.3 数据质量要求

- 所有报送数据须与考勤系统/业绩系统一致
- 不得瞒报虚报
- 异常缺勤须在备注栏说明原因
- 数据准确率达标线：100%（SLA）

## 六、会议机制

> 会议是沟通的重要载体，需要有准备、有产出。

### 6.1 周度业务复盘会

| 项目 | 要求 |
|------|------|
| 参与方 | 业务对接人、供应商管理岗、各站点对接人 |
| 频率 | 每周1次 |
| 议程 | 各站点周报同步、关键异常讨论、下周重点 |
| 会议纪要 | 由供应商管理岗记录并群内发送 |

### 6.2 月度评估沟通会

| 项目 | 要求 |
|------|------|
| 参与方 | 供应商管理岗、各站点负责人 |
| 频率 | 每月1次（次月第1周） |
| 议程 | 月度评估结果公布、排名公示、一对一沟通 |
| 输出 | 月度评估报告、排名公告 |

### 6.3 专项沟通会

| 触发条件 | 说明 |
|----------|------|
| 月度评估C级及以下 | 约谈站点负责人，讨论改进方案 |
| 触发预警 | 要求提交整改计划 |
| 重大业务调整 | 由业务方或供应商管理发起 |
