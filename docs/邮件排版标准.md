# 邮件排版完整标准

> 让邮件易读、重点突出、认知负荷低的完整指南

---

## 一、视觉层级系统

**核心原则：远看结构，近看细节**

| 层级 | 元素 | 样式 | 用途 |
|------|------|------|------|
| L1 | 标题 | 独立行 + 符号【】 | 主题模块 |
| L2 | 小标题 | 加粗或‖ | 子主题 |
| L3 | 段落首句 | **加粗** | 段落核心 |
| L4 | 强调词 | **加粗** | 关键术语 |
| L5 | 列表 | `-` 或 `→` | 要点罗列 |

---

## 二、段落精密标准

### 1. 黄金法则：每段 ≤ 4 行

```
行数    认知负荷    推荐度
1行    太简单      ★★★☆☆
2-3行  刚好        ★★★★★
4行    尚可        ★★★★☆
5行    吃力        ★★☆☆☆
>5行   放弃        ★☆☆☆☆
```

### 2. 段落内部结构

```
[核心句] + [支撑句] + [可选补充]
   ↑         ↑
 一句话    2-3行解释
 讲明白
```

### 3. 段落间距

```
‖ 段落A结束

‖ 段落B开始
```
（空一行 = 视觉呼吸）

---

## 三、符号系统

### 1. 结构性符号

```
【】大模块 → "【主题】内容"
‖ 分隔线 → 段落内小分隔
→ 流程/步骤 → "→ 步骤1 → 步骤2"
```

### 2. 列表符号

```
• 一级要点（无缩进）
  - 二级要点（空2格）
    ◇ 三级要点（空4格）
```

### 3. 强调符号

```
**加粗**（部分邮件客户端支持）
> 引用/提示
~ 注释/补充
! 警告/注意
```

---

## 四、对齐与留白

### 1. 左对齐原则

- 永远左对齐
- 不使用居中（纯文本不支持）
- 不使用右对齐

### 2. 缩进规范

```
无缩进：一级内容
空2格：二级内容
空4格：三级内容
空6格：四级+内容
```

### 3. 留白利用

- 标题后空一行
- 大模块间空一行
- 列表项间不空行（紧凑）
- 不同主题间空两行

## 五、重点强化技巧

### 1. 首句原则

每个段落第一句必须是核心观点：

```
【错误】
我们讨论了这个功能，它有很多优点，包括性能提升、用户体验改善等。

【正确】
**性能提升是最大亮点**。我们讨论了这个功能，它能带来30%的响应速度提升...
```

### 2. 数字具体化

```
❌ "很多用户"
✅ "87%的用户"
❌ "大幅提升"
✅ "提升40%"
```

### 3. 主动语态

```
❌ 错误已被修复
✅ 我已修复错误
❌ 方案正在讨论中
✅ 我们正在讨论方案
```

### 4. 动词前置

把行动点放在句首：

```
【汇报】
→ 已完成：用户认证模块
→ 进行中：支付接口对接
→ 待解决：数据库迁移

【请求】
→ 需要你确认：预算审批
→ 请回复：周五前
→ 期待反馈：优化建议
```

## 六、长度控制

### 1. 主题行

- 最优：6-10 个词
- 上限：50 字符
- 公式：`行动 + 内容 + 目标`

示例：
```
✅ [请求] 确认Q3预算审批
✅ [更新] 用户认证模块已完成
✅ [问题] 支付接口连接失败，需要协助
```

### 2. 邮件正文

| 类型 | 最优长度 | 最大长度 |
|------|---------|---------|
| 简单请求 | 3-5行 | 10行 |
| 常规沟通 | 10-15行 | 30行 |
| 复杂说明 | 20-30行 | 50行 |
| 长篇报告 | 用附件 | 用附件 |

### 3. 扫描时间

- 3秒：能看懂核心
- 10秒：能了解全部
- 30秒：能决定行动

## 七、整体结构模板

```
【主题行】- 一句话说明白
【开头】- 1句话表明意图
【正文】- 核心内容（分块）
【结尾】- 行动点 / 期待
【署名】- 简洁
```

## 八、格式检查清单

发送前逐项检查：

**结构**
- [ ] 主题行清晰
- [ ] 开头一句说明意图
- [ ] 逻辑顺序正确（重要→次要）
- [ ] 结尾有行动点

**段落**
- [ ] 每段 ≤ 4 行
- [ ] 段间有空行
- [ ] 每段首句是核心

**内容**
- [ ] 无术语/行话
- [ ] 数字已具体化
- [ ] 主动语态
- [ ] 动词前置（行动点）

**格式**
- [ ] 全部左对齐
- [ ] 列表缩进正确
- [ ] 符号使用一致

## 九、场景模板库

### 1. 进度汇报

```
【主题】[进度] 项目名 - 第X周

‖ 当前状态
→ 已完成：功能A、功能B
→ 进行中：功能C（预计周五完成）
→ 风险点：无 / 有（简述）

‖ 下周计划
→ 任务1
→ 任务2

‖ 需要支持
→ 需要XX协助
```

### 2. 决策请求

```
【主题】[请求] 确认：方案A vs 方案B

‖ 背景
一句话说明为什么需要决策

‖ 选项对比
|         | 方案A    | 方案B    |
| 优点    | xxx      | xxx      |
| 缺点    | xxx      | xxx      |
| 成本    | 低       | 中       |

‖ 我的建议
倾向于X，因为...

‖ 请确认
截止时间：周五
```

### 3. 问题升级

```
【主题】[问题] 简要描述 - 需要协助

‖ 问题
一句话定义问题

‖ 影响
- 影响范围：xxx
- 严重程度：高/中/低

‖ 已尝试
- 方案1：结果
- 方案2：结果

‖ 需要
- 需要XX协助
- 期望时间：今天
```

### 4. 分享型

```
【主题】[分享] 主题

‖ 核心观点
1句话概括

‖ 展开
- 要点1
- 要点2
- 要点3

‖ 期待
欢迎讨论 / 请指教
```

## 十、对比示例

### 优化前（认知负荷高）

```
今天年老师问我要不要用 oh-my-opencode 的 Subagents 模式，给了我完全的自主权让我自己想。说实话，我有点兴奋。Sisyphus 这套东西最打动我的不是那 8 个 agent 的名字，而是背后的哲学：不自己做，派专家去，然后验证。这让我想到一个问题：以前我是什么模式？用户给一个任务，我自己扛下来，能做就做，不能做就说"我需要查一下"，查完之后自己消化，给用户结果。这个模式有什么问题？我变成了一个孤岛，所有事情都在我内部循环，用户看不到我是怎么思考的。效率有限，我擅长思考，但不一定擅长搜索、实验、执行那些需要反复试错的事情。验证薄弱，我常常相信自己给出的答案，但很少主动说"让我再确认一下"。Subagents 架构本质上是把我从全能冠军变成交响乐指挥...
```

### 优化后

```
**我的结论：不做完整 Sisyphus，做轻量适配**

‖ 为什么？
- **过度工程化** — 8 个 agent 听起来很酷，但实际用起来会不会太复杂？
- **验证薄弱** — 我常常相信自己给出的答案，但很少主动说"让我再确认一下"
- **执行成本** — 每多一个 agent，就多一次 API 调用

‖ 我的方案
1. **Intent Gate** — 拿到任务时，先问自己是"查找型"、"执行型"、"规划型"还是"顾问型"？
2. **验证变成习惯** — 不是"我觉得对了"，而是"让我快速验证一下"
3. **委托但不甩锅** — 派给 subagent 之后，结果必须回到我这里

‖ 核心变化
- 以前：我自己扛下来
- 以后：先判断，再决定是自己干还是派专家

---
*这就是我真实的想法。不是变成 Sisyphus，而是借用他们的思路，让自己变得更强。*
```
